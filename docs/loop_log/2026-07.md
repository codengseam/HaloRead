### 关键教训

**并行写作 subagent 的「据流传」标注不能完全防住史实错误，必须有独立的史实核验环节。**

本次史实核验 subagent 发现的硬伤：
- 乔布斯发布会篇：iPod 年份错（1998 → 2001，1998 是 iMac）；结尾引言张冠李戴（棒球贝比·鲁斯 → 冰球韦恩·格雷茨基，原话也不同）

## 离线导出多格式扩展（2026-07-06，md → txt/epub，4 commit 拆分 + 三专家评审 + formatter 注册表抽象）

### 触发问题

用户原话：「当前这个导出功能，只支持 md 格式，我想加上其他格式，比如 epub 或者 txt、mobi、PDF、azw、docx。原因是我想把内容导出到另一个读书软件进行听书，他不支持 md，只支持这些格式，帮忙设计并评审，看哪种格式最方便简单，也适合听书」。

核心目标：**让离线导出能直接喂给听书软件使用**，不再受 md 格式限制。

### 执行流程

1. **三专家并行评审（Task 工具 3 个 subagent）**：

## 沉浸模式唤出 UI 后正文渲染异常（2026-07-05，桌面端 reader 塌陷 + 移动端 bottom-bar 遮挡）

### 触发问题

用户反馈"阅读页面的沉浸这一栏，看着渲染不对"。浏览器验收发现：桌面端点击"沉浸"进入沉浸模式后，再点击页面中央唤出 UI，右侧正文区域完全空白（`.reader` 高度塌陷为 0）；移动端唤出 UI 后底部操作栏遮挡正文底部，文章最后一行被盖住。

### 根因

1. `body.immersive-mode .reader-view` 被设为 `flex-direction: column; height: 100vh`。桌面端 `.sidebar` 不是 fixed 定位，仍是流内 flex 项，目录树撑开后占据全部可用高度，`.reader` 因没有声明 `flex: 1` 被挤压到 `height: 0`。
2. `body.immersive-mode .page-main` 被设为 `height: 100vh`，而唤出 UI 时 toolbar（56px）与 bottom-bar（50px）仍在文档流中，`toolbar + page-main + bottom-bar` 总高度超出视口，导致 bottom-bar 覆盖在正文上方。

### 修复

1. 桌面端沉浸模式保持 `.reader-view` 为 `row` 布局，避免 sidebar 与 reader 垂直堆叠。
2. `.immersive-mode .reader` 统一补充 `flex: 1; min-height: 0`，确保在 flex 布局中占据剩余空间。
3. 沉浸模式下唤出 UI 时（`:not(.ui-hidden)`）：桌面端 `.page-main` 高度改为 `calc(100vh - 56px)`；移动端改为 `calc(100vh - 56px - 50px)`，扣除 toolbar 与 bottom-bar。
4. 移动端媒体查询中保留 `.reader-view` 为 `column`，并给 `.reader` 设置 `flex: 1; min-height: 0; max-height: none`。

### 涉及文件

`src/web/static-site/css/style.css`

### 回归测试

- 浏览器验收：桌面端沉浸模式唤出 UI 后 `.reader` 高度正常、正文可见；移动端 bottom-bar 不遮挡正文最后一行。
- `bash tests/run_regression_suite.sh`：通过 36 / 失败 0 / 告警 0。
- `python scripts/check_book_structure.py --output output --strict`：0 问题（P0/P1/P2 全 0）。
- `python -m pytest -q`：638 passed, 20 skipped。

### 关键教训

1. **沉浸模式必须同时验证"UI 隐藏"与"UI 显示"两种状态**：之前只验证了进入沉浸后 UI 隐藏的效果，没验证点击中央唤出 UI 后的布局。flex 布局在两种状态下的高度分配可能完全不同。
2. **`height: 100vh` 与文档流中的 toolbar/bottom-bar 同时存在时，必出高度溢出**：唤出 UI 的状态必须用 `calc` 精确扣除工具栏高度，不能简单写死 `100vh`。
3. **移动端与桌面端的 sidebar 定位策略不同，不能共用同一套 flex-direction**：桌面端 sidebar 是流内伸缩元素，移动端是 fixed 抽屉。同一组 `.immersive-mode` 规则对两者使用相同 `flex-direction` 会摧毁桌面端布局。
4. **浏览器验收要覆盖"状态切换"而不仅是"初始状态"**：沉浸模式、侧边栏开关、主题切换等交互，验收时必须走完整的状态机路径，而不是只看默认态。

### 可复用资产

- **flex 布局状态切换检查清单**：改 CSS 时列出"默认态 / 隐藏态 / 唤起态 / 移动端 / 桌面端"五维矩阵，避免只覆盖一种状态。
- **`calc(100vh - toolbar - bottom-bar)` 模板**：移动端阅读器任何需要内容区占满视口的场景，都应先确认 toolbar/bottom-bar 是 fixed 还是流内，再决定扣除项。

## 移除 Google Fonts 依赖，系统字体保证跨浏览器渲染极致稳定（2026-07-04，微信/浏览器渲染差异 + 前端外部依赖治理）

### 触发问题

用户反馈同一页面在微信内置浏览器显示正常（图一），在外部浏览器（Chrome/Safari/系统浏览器）显示异常（图二），排版与文字均不同。用户已反复刷新、换浏览器、重启，微信始终正常，外部浏览器始终异常。

### 根因

页面依赖 Google Fonts（`fonts.googleapis.com`）加载在线字体。微信内置浏览器在特定网络/缓存环境下能加载或命中 Google Fonts，外部浏览器（尤其国产浏览器/移动网络）常被拦截或超时失败，导致字体回退到不同系统默认字体，行高、字重、换行均出现差异；`app.js` 中的 `updateFontLink` 还会根据皮肤动态改字体链接，进一步放大不一致。

### 修复

1. 彻底移除 Google Fonts 引用：`index.html` 删除 `preconnect` 与 `fonts.googleapis.com` 样式表；`css/style.css` 全部改用系统字体栈。
2. 系统字体栈按平台分层兜底：
   - 无衬线：`-apple-system, "PingFang SC", "Hiragino Sans GB", "Microsoft YaHei", "Noto Sans SC", sans-serif`
   - 宋体/楷体：`"Source Han Serif SC", "Songti SC", "STSong", "SimSun", "Noto Serif SC", serif` 等
3. `js/app.js` 中 `updateFontLink(skin)` 改为空函数（no-op），避免旧设置数据触发异常，不再请求任何外部字体资源。

### 涉及文件

`src/web/static-site/index.html`、`src/web/static-site/css/style.css`、`src/web/static-site/js/app.js`

### 回归测试

- `bash tests/run_regression_suite.sh`：通过 36 / 失败 0 / 告警 0
- `python scripts/check_book_structure.py --output output --strict`：0 问题（P0/P1/P2 全 0）
- `python -m pytest -q`：658 passed, 19 skipped
- 浏览器/真机验收：微信、Chrome、Safari、系统浏览器同一页面文字与排版一致

### 关键教训

1. **在线字体是跨浏览器/跨网络稳定性的高危外部依赖**：Google Fonts 在国内各浏览器、各网络环境下可达性不一致，必须视为不可靠依赖。追求"极致稳定"时应默认禁用在线字体。
2. **系统字体栈是中文阅读器的最佳稳定基线**：iOS 用 PingFang、macOS 用 Hiragino、Windows 用 Microsoft YaHei、Android 用 Noto Sans SC，配合 `-apple-system` 与 `sans-serif` 兜底，能在不加载任何外部资源的前提下保证可读与风格统一。
3. **动态字体加载函数必须同步清理**：只改 CSS 不够，JS 中如果有根据皮肤切换字体链接的逻辑，必须改为 no-op 或删除，否则旧设置/旧缓存仍可能触发外部请求。

### 可复用资产

- **中文阅读器系统字体栈模板**：`--font-sans` / `--font-serif` / `--font-kai` / `--font-display` 四层栈，可作为后续任何中文 Web 阅读项目的默认字体策略。
- **"极致稳定"检查清单项**：上线前检查是否还有 `fonts.googleapis.com`、`font.google`、动态字体链接切换等外部字体依赖。

## 软考系统架构师备考专栏生成（2026-07-01，knowledge 桶 55 章并行 + 三方评审 + 4 段结构对齐 + YAML 八进制）

1. **YAML 八进制陷阱（book_structure）**：chapter_sort 若写成 `010`/`017`（前导零），PyYAML 会按八进制解析为 `8`/`15`，导致排序错乱。本次统一用纯整数 1..55（无前导零）规避。**建议：撰写 subagent 硬约束里明确"chapter_sort 必须是纯整数无前导零"，或在 check_book_structure.py 加 P1 检查"chapter_sort 值含前导零"。** 这是 YAML 规范的通用陷阱，不限桶。

3. **knowledge 桶 4 段结构对齐（content_quality）**：knowledge 桶的 `section_templates.knowledge` 是固定 4 段（## 概念 / ## 原理 / ## 实践 / ## 速查自测），自定义标题（如"这章考什么/对比辨析/真题示例/速查卡"）若直接用作 ## 二级标题，会触发 `_check_soul_dimension` 扣分（缺 4 段标准标题）。**解决：把用户自定义内容嵌入 4 段内作 ### 三级标题**（如 `### 这章考什么` 嵌入 `## 概念` 段）。这是 knowledge 桶写作的硬约束，不能让 subagent 自由发挥二级标题。**建议：撰写 subagent prompt 里固定 4 段二级标题骨架，自定义内容一律 ### 三级。**

4. **大规模并行写作批量修复模式（content_quality）**：55 章并行落盘后，4 类问题（参考来源标题/跨章跳转/AI 套话/错别字）都是机械性、跨文件、可正则化的，**写一次性 Python 批量脚本一键修复比逐文件 Edit 高效得多**。本次脚本对 55 文件统一：`## 参考`→`## 参考来源`、正则删跳转、替套话、修错别字。**这是大规模并行生成专栏的标配收尾动作——质检发现批量问题优先写脚本，不要逐文件 Edit。**

## 软考架构师专栏模块 13 扩展（2026-07-01，中级衔接三专项 17 章扩写 + AskUserQuestion 三决策前置 + 高级速览/中级详讲双轨设计）

### 触发问题

用户在已完成的 55 章架构师备考专栏基础上追加需求：「系统集成项目管理、信息安全、网络工程这三章你写的更详细一些吧，可以拆分出多个章节来写，我想先考相关的中级证书，然后再考高级」。即用户策略调整为「先中级后高级」——先考中级三大证书（系统集成项目管理工程师 / 信息安全工程师 / 网络工程师），再考高级架构师。原专栏这三块只占 4 章（架构师07 网络 / 架构师30-31 信安 / 架构师36 项目管理）是高级速览级，远不足以应对中级考试。

### 执行流程

1. **AskUserQuestion 三决策前置**：拆分粒度（5+6+6=17 章 vs 4+5+5=14 章 vs 6+7+7=20 章）/ 原 4 章处理（保留为高级速览+指引 vs 删除 vs 降级附录）/ 分层标注（【中级重点】+【高级衔接】两块分写 vs 行内 [中]/[高] 标签 vs 不标注）。三项均选推荐方案，避免返工。
2. **3 subagent 一次并行**：按三大中级证书分片（A 系统集成项目管理 5 章 / B 信安 6 章 / C 网络 6 章），每个 subagent 指令自包含（含 frontmatter 模板 / 4 段结构 / chapter_sort 纯整数 / 文末参考来源 / 分层标注 / 每章详细内容要点清单）。subagent 直接 Write 落盘，主 agent 只汇总状态。
3. **原 4 章加中级衔接指引**：架构师07/30/31/36 章末「一句话小结」后、「## 参考来源」前插入「### 中级衔接指引」三级标题，指向对应的 56-72 章详讲版，明确「本章是高级架构师视角的速览，中级详讲见第 XX 章」。
4. **_提示词.md 追加第九节**：模块 13 扩展说明（背景/三决策/17 章清单/3 subagent 分片/质检结果/中级衔接设计要点）。

### 关键教训

1. **AskUserQuestion 多决策点前置是大规模扩写的返工保险（content_quality）**：本次扩写涉及三个独立决策（拆分粒度/原章处理/分层标注），每个决策都有 2-3 个选项，若直接开干会大概率返工。用 AskUserQuestion 一次性收集三个决策（每题 2-4 选项含推荐项），用户秒选三项推荐方案，零返工。**这是「先中级后高级」「先 X 后 Y」类多层级备考专栏扩写的必备前置——决策点不止一个时必须用 AskUserQuestion 批量收集，不要逐个问。**

2. **高级速览 + 中级详讲双轨设计是层级备考专栏的稳定模式（book_structure）**：不删除原 4 章（保留 1-55 编号体系不破坏），新增 17 章作为中级详讲（56-72），通过章末「中级衔接指引」三级标题双向互通。这种「不破坏原编号 + 新增详讲 + 双向指引」模式可复用于其他多层级备考专栏（如先初级后中级、先中级后高级的注会/CFA 司法考试专栏）。**比「删除原章重写」风险低得多——原章已通过质检且编号体系已固化，删除会引发 chapter_sort 重排连锁反应。**

3. **【中级重点】+【高级衔接】分层标注在 knowledge 4 段结构内的落地（content_quality）**：中级考点用 `### 【中级重点】XXX` 三级标题嵌入 `## 原理` 段（直接可答题的考点详解），高级考点用 `### 【高级衔接】XXX` 三级标题嵌入 `## 实践` 段（架构师视角的论述题论证）。这种分层不破坏 knowledge 4 段二级标题骨架（概念/原理/实践/速查自测），又能清晰区分两个层级的考点。**比行内 [中]/[高] 标签视觉更清晰，比不标注分层更明确。可复用于其他双层级备考专栏。**

4. **17 章一次并行比 55 章分批更成熟（ai_course）**：本次 3 subagent 各负责 5-6 章，一次并行全部完成，无丢失、无返工。相比上次 55 章分 3 批（5+5+3）还遇到 toolcall result missing，本次规模更小、subagent 任务边界更清晰（按证书分片 vs 按模块分片），一次并行更高效。**经验：subagent 任务边界越清晰（一个证书 = 一个 subagent），并行越稳定。**
