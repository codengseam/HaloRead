# 算法课·26｜滑动窗口与双指针：扫描问题的三模板

> 前置知识：学完第 15 章左右（数组、哈希表基础），尤其要懂"双层 for 枚举"为什么会超时
> 学完你能：①给小白讲清"滑动窗口为什么是 $O(n)$ 不是 $O(n^2)$"②默写三模板应对 90% 的扫描题 ③手撕 LC 3/76/15/11 四道高频题 ④判断一道题该用定长窗口、可变窗口还是双指针 ⑤用对数器对比暴力 $O(n^2)$ 与窗口 $O(n)$

## 一、问题与思路：从"两层 for 循环"到"两个指针滑过去"

### 1. 一个具体场景：最长无重复子串

给你字符串 `abcabcbb`，找出不含重复字符的最长子串长度，答案是 `abc`，长度 3。

直觉做法是枚举所有子串起点 `i`，再枚举终点 `j`，检查 `[i, j]` 是否有重复字符——三重循环 $O(n^3)$。优化一步：固定 `i`，让 `j` 往右走，用哈希表记录字符出现次数，遇到重复就停，外层 `i`、内层 `j` 各扫一遍，看起来仍是 $O(n^2)$。

但这里有个反直觉的事实：**这题的最优解是 $O(n)$**。关键观察——当 `j` 遇到重复字符时，`i` 没必要退回 `i+1` 重新枚举，可以直接跳到"造成重复的那个字符的下一个位置"，因为 `[i+1, j]` 一定还包含那个重复字符，重新枚举全是浪费。

这就是滑动窗口（Sliding Window）的起点：**用两个指针圈出一段区间，右端扩张、左端收缩，全程只前进不回退，总操作数被压缩到 $O(n)$**。

### 2. 暴力做法的两个痛点，窗口与双指针怎么破

**痛点一：枚举了大量"明显不优"的区间。** 暴力做法把所有 $O(n^2)$ 个区间都试一遍，但真正可能是答案的只有 $O(n)$ 个——窗口每一步要么右扩、要么左缩，产生的"有意义区间"数量是线性的。窗口的破法：只在这些"有意义区间"上停留。

**痛点二：每次区间变化都从头算。** 暴力做法每换一个起点，哈希表、计数器全部推倒重来。窗口的破法：区间变化是"增一个元素"或"删一个元素"的增量操作，用一个哈希表/计数器维护当前区间的状态，每次 $O(1)$ 更新，不重算。

那双指针（Two Pointers）和滑动窗口是什么关系？严格说，滑动窗口是双指针的一种特例——左右指针同向移动、维护一段连续区间。广义的双指针还包括"对撞双指针"（一左一右向中间靠）和"同向双指针"（一快一慢同向走）。本章把扫描类问题的解法归纳成**三模板**：

1. **定长滑动窗口**：窗口大小固定为 `k`，整体右移。
2. **可变滑动窗口**：窗口大小可变，按条件伸缩，求最长/最短。
3. **双指针**：含对撞（左右夹逼）与同向（快慢）两种 flavor，不要求维护连续区间状态。

**核心思想**：这三模板的共同根基是「用空间换时间 + 利用单调性避免重复计算」——用哈希表/计数器等 $O(\Sigma)$ 空间维护窗口状态（空间换时间），让指针只朝一个方向走、已排除的答案不再回头（单调性避免重复计算），于是 $O(n^2)$ 的枚举被压成 $O(n)$ 的扫描。这是它们能突破暴力复杂度的关键。

### 3. 为什么这三模板值得独立成章

很多算法书把滑动窗口塞进"双指针"一章带过，但面试中这三模板的判题套路、代码骨架、坑点完全不同：定长窗口考"维护窗口内聚合值"，可变窗口考"何时收缩左端"，对撞双指针考"单调性证明"。把它们合并讲，新手只会背一个"双指针"名词，遇到题分不清该用哪种。独立成章的目的，是让你拿到一道扫描题能在 30 秒内判定模板——这恰恰是面试高频考点。

## 二、原理与实现：三模板的代码骨架与复杂度

### 1. 模板一：定长滑动窗口

适用场景：窗口大小固定为 `k`，求窗口内的最大值/最小值/和/特定计数。

经典题：长度为 `k` 的子数组最大和。

```python
def max_sum_fixed_window(nums, k):
    n = len(nums)
    if n < k:
        return 0
    window_sum = sum(nums[:k])          # 第一个窗口的初始和
    ans = window_sum
    for i in range(k, n):               # 窗口整体右移：进一个、出一个
        window_sum += nums[i]           # 右端进入
        window_sum -= nums[i - k]       # 左端离开
        ans = max(ans, window_sum)
    return ans
```

**逐行讲透**：

- **`window_sum = sum(nums[:k])`**：第一个窗口的初始和，$O(k)$。后续不再重算。
- **`for i in range(k, n)`**：`i` 是新进入窗口的右端下标。从 `k` 开始，因为前 `k` 个已覆盖第一个窗口。
- **`window_sum += nums[i]`**：右端进入，累加。
- **`window_sum -= nums[i - k]`**：左端离开，减去。`i - k` 正是离开的那个下标——窗口是 `[i-k+1, i]`。
- **`ans = max(ans, window_sum)`**：滚动更新答案。

**复杂度**：每个元素被加一次、减一次，共 $2n$ 次操作，$O(n)$。空间 $O(1)$。

**关键陷阱**：如果题要的是"窗口内最大值"而非"和"，定长窗口就不能简单 $O(1)$ 维护了——需要单调队列（Monotone Queue）把"窗口最大值"也压到均摊 $O(1)$。这是 LC 239 滑动窗口最大值的考点，留到栈与单调结构章细讲，这里只需知道"定长窗口求和是 $O(1)$ 更新，求最值要额外结构"。

### 2. 模板二：可变滑动窗口

适用场景：窗口大小不固定，按某个条件伸缩，求满足条件的最长/最短子数组或子串。

代码骨架（求"满足条件的最长"）：

```python
def variable_window_longest(nums, condition):
    n = len(nums)
    left = 0
    state = {}                         # 维护窗口状态，如字符计数
    ans = 0
    for right in range(n):             # 右端逐步扩张
        add(state, nums[right])        # 1. nums[right] 加入窗口，更新 state
        while not valid(state, condition):  # 2. 窗口非法时收缩左端
            remove(state, nums[left])
            left += 1
        ans = max(ans, right - left + 1)    # 3. 此时合法，更新答案
    return ans
```

**这套骨架有两种变体，区别在 `while` 收缩的条件**：

- **求最长**（如 LC 3 无重复字符的最长子串）：`while` 条件是"窗口非法"，收缩到"刚好合法"就停，答案在合法时取。窗口只扩张、不主动缩小，最长。
- **求最短**（如 LC 76 最小覆盖子串）：`while` 条件是"窗口合法"，合法就继续收缩左端以找更短，直到"刚好不合法"才停，答案在合法时取。窗口尽量缩小。

两种变体的代码几乎一样，只是 `while` 的判断方向反过来。记住一句话：**求最长，"不合法才缩"；求最短，"合法就缩"**。

**为什么是 $O(n)$ 而不是 $O(n^2)$**：乍看 `for` 套 `while` 像两层循环。但关键在于——`left` 和 `right` 都只单调右移，全程各走最多 $n$ 步，所以 `while` 累计执行次数不超过 $n$，总操作数 $O(n)$。这就是摊还分析（Amortized Analysis）的经典例子：单看某一步 `while` 可能执行很多次，但均摊到全部步骤，每步均摊 $O(1)$。

### 3. 模板三：双指针（对撞 + 同向）

#### 3.1 对撞双指针

适用场景：数组有序（或问题本身具有单调性），左右两端各放一个指针，向中间靠拢。

经典题：LC 11 盛最多水的容器、LC 15 三数之和（排序后）。

```python
def container_most_water(height):
    n = len(height)
    left, right = 0, n - 1
    ans = 0
    while left < right:
        area = min(height[left], height[right]) * (right - left)  # 短板×宽度
        ans = max(ans, area)
        if height[left] < height[right]:   # 移动较短的一端
            left += 1
        else:
            right -= 1
    return ans
```

**为什么移动较短的一端**：面积由"短板 × 宽度"决定。宽度每一步都在缩小，唯一可能让面积变大的是"换一个更高的板"。如果移动较长的一端，短板没变、宽度还变小，面积只会更小——这种解一定不优，直接排除。移动较短的一端，才有机会遇到更高的板。这个"排除法"是 $O(n)$ 成立的根基：每一步都排除掉一个不可能优的解，无需回头。

#### 3.2 同向双指针（快慢指针）

适用场景：数组（不一定有序），用一个指针遍历、另一个指针记录"有效位置"。常见于原地修改、去重、分区。

经典题：LC 26 删除有序数组中的重复项、LC 27 移除元素。

```python
def remove_duplicates(nums):
    if not nums:
        return 0
    slow = 0                            # slow 指向"已写入的最后一个有效位置"
    for fast in range(1, len(nums)):    # fast 负责扫描
        if nums[fast] != nums[slow]:
            slow += 1
            nums[slow] = nums[fast]
    return slow + 1                     # 新长度
```

`fast` 负责扫描，`slow` 负责落位，两者同向前进，`fast` 永远 ≥ `slow`。同向双指针的关键是"**用一个指针读、一个指针写，原地完成筛选**"，空间 $O(1)$。

### 4. 三模板的复杂度与适用场景对照

| 模板 | 时间 | 空间 | 适用场景 | 典型题 |
|---|---|---|---|---|
| 定长滑动窗口 | $O(n)$ | $O(1)$ 或 $O(k)$ | 窗口大小固定，求聚合值 | LC 239、定长子串 |
| 可变滑动窗口 | $O(n)$ | $O(\Sigma)$ | 窗口大小可变，按条件伸缩 | LC 3、LC 76 |
| 对撞双指针 | $O(n)$ | $O(1)$ | 有序数组、夹逼求最优 | LC 11、LC 15 |
| 同向双指针 | $O(n)$ | $O(1)$ | 原地筛选/去重/分区 | LC 26、LC 27 |

注：$\Sigma$ 表示字符集大小，ASCII 是 128，LC 3 的哈希表最多存 26 个字母。

### 5. 为什么这三模板能突破 $O(n^2)$——单调性是根基

三模板都是 $O(n)$，背后的统一根基是**单调性**：

- **滑动窗口**的单调性：`right` 永远右移，`left` 永远右移，不回退。已经判断过"窗口非法"的 `left` 位置不会再次成为合法窗口的左端（因为合法窗口只会更靠右）。
- **对撞双指针**的单调性：`left` 只增、`right` 只减。已经排除的较短端不会再被考虑。
- **同向双指针**的单调性：`fast` 和 `slow` 都只右移，已写入的位置不被覆盖回退。

没有单调性，三模板都不成立——这也是为什么 LC 15 三数之和必须先排序：排序制造了"指针移动的单调性"，才能用对撞双指针。如果数组无序，对撞双指针无法保证"移动较短端能排除不优解"。可见"用空间换时间"换来的是 $O(\Sigma)$ 的状态存储，"利用单调性"换来的才是从 $O(n^2)$ 到 $O(n)$ 的复杂度跨越，两者缺一不可。

## 三、实践与面试：四道高频题、对数器、判题套路

### 1. LC 3：无重复字符的最长子串（可变窗口·求最长）

**题意**：给定字符串 `s`，找不含重复字符的最长子串长度。

**思路**：可变窗口，`while` 收缩条件是"窗口内有重复字符"。用哈希表记录每个字符最近出现的下标，遇到重复时把 `left` 跳到"上次出现位置 + 1"。

```python
def length_of_longest_substring(s):
    last = {}                           # char -> 最近一次出现的下标
    left = 0
    ans = 0
    for right, ch in enumerate(s):
        if ch in last and last[ch] >= left:
            left = last[ch] + 1         # 跳过重复字符
        last[ch] = right
        ans = max(ans, right - left + 1)
    return ans
```

**讲透一行**：`last[ch] >= left` 这个判断是为了避免"陈旧记录"——如果 `last[ch]` 在 `left` 之前，说明那个重复字符已被收缩出窗口，不算重复。这是这题最容易踩的坑，漏掉就会在 `abba` 这类输入上出错。

**复杂度**：$O(n)$，每个字符进出窗口各一次。

**用一个例子走一遍**。字符串 `abcabcbb`，看 `left` 怎么跳：

```
right=0 'a': 无记录,          left=0, last={'a':0},        ans=1
right=1 'b': 无 b,            left=0, last={...,'b':1},    ans=2
right=2 'c': 无 c,            left=0, last={...,'c':2},    ans=3
right=3 'a': last['a']=0 ≥ 0, left=1, last['a']=3,         ans=3  (窗口 bca)
right=4 'b': last['b']=1 ≥ 1, left=2, last['b']=4,         ans=3  (窗口 cab)
right=5 'c': last['c']=2 ≥ 2, left=3, last['c']=5,         ans=3  (窗口 abc)
right=6 'b': last['b']=4 ≥ 3, left=5, last['b']=6,         ans=3  (窗口 cb)
right=7 'b': last['b']=6 ≥ 5, left=7, last['b']=7,         ans=3  (窗口 b)
最终 ans=3
```

观察：`left` 从 0 单调跳到 7，全程不回退——这就是 $O(n)$ 的来源。`abc` 之后的每个新字符都在窗口内出现过，所以 `ans` 一直停在 3。

### 2. LC 76：最小覆盖子串（可变窗口·求最短）

**题意**：给定字符串 `s` 和 `t`，找 `s` 中包含 `t` 所有字符的最短子串。

**思路**：可变窗口，`while` 收缩条件是"窗口已覆盖 `t`"，合法就继续缩左端找更短。用一个 `need` 字典记 `t` 的需求，一个 `have` 字典记窗口已有，再用一个 `valid` 计数器记"已满足几种字符"避免每次遍历字典。

```python
from collections import Counter

def min_window(s, t):
    need = Counter(t)
    have = {}
    valid = 0                           # 已满足的字符种类数
    left = 0
    ans_len = float('inf')
    ans_start = 0
    for right, ch in enumerate(s):
        have[ch] = have.get(ch, 0) + 1
        if ch in need and have[ch] == need[ch]:
            valid += 1                  # 这个字符刚好够数
        while valid == len(need):       # 窗口合法，尝试缩左
            if right - left + 1 < ans_len:
                ans_len = right - left + 1
                ans_start = left
            lc = s[left]
            have[lc] -= 1
            if lc in need and have[lc] < need[lc]:
                valid -= 1              # 这个字符不够数了
            left += 1
    return "" if ans_len == float('inf') else s[ans_start:ans_start + ans_len]
```

**关键设计**：`valid` 计数器把"判断窗口是否覆盖 `t`"从 $O(|\Sigma|)$ 降到 $O(1)$。这是可变窗口处理"多约束条件"的通用技巧——为每个约束维护一个布尔/计数信号，再聚合一个总信号。

**复杂度**：$O(n + m)$，`n = len(s)`，`m = len(t)`。

### 3. LC 11：盛最多水的容器（对撞双指针）

**题意**：数组 `height`，每条竖线高度 `height[i]`，找两条线与 x 轴围成的容器能盛最多水。

**思路**：见 §2.3.1，对撞双指针，每次移动较短端。

```python
def max_area(height):
    left, right = 0, len(height) - 1
    ans = 0
    while left < right:
        h = min(height[left], height[right])
        ans = max(ans, h * (right - left))
        if height[left] < height[right]:
            left += 1
        else:
            right -= 1
    return ans
```

**面试加分点**：能讲清"为什么移动较短端"的排除法论证（见 §2.3.1），而不是只背"双指针模板"。这题面试官最爱追问的就是正确性证明。

**用一个例子走一遍**。`height = [1,8,6,2,5,4,8,3,7]`，`left=0, right=8`：

```
left=0,right=8: h=min(1,7)=1, area=8,  1<7 移左 → left=1
left=1,right=8: h=min(8,7)=7, area=49, 8≥7 移右 → right=7   ★当前最优
left=1,right=7: h=min(8,3)=3, area=18, 移右 → right=6
left=1,right=6: h=min(8,8)=8, area=40, 移右 → right=5
left=1,right=5: h=min(8,4)=4, area=16, 移右 → right=4
left=1,right=4: h=min(8,5)=5, area=15, 移右 → right=3
left=1,right=3: h=min(8,2)=2, area=4,  移右 → right=2
left=1,right=2: h=min(8,6)=6, area=6,  移右 → right=1, 停
ans=49
```

观察：第一步之后 `left` 就固定在 1（因为 `height[1]=8` 是全局最高，移动它只会让短板更短），后续全靠 `right` 左移试探。每一步都排除了"移动较长端"那一支不可能更优的解，8 步内定最优。

### 4. LC 15：三数之和（排序 + 对撞双指针）

**题意**：数组中找所有不重复的三元组，使其和为 0。

**思路**：先排序 $O(n\log n)$。固定一个数 `nums[i]`，剩下两个用对撞双指针在 `[i+1, n-1]` 里找 `=-nums[i]`。去重在指针移动时做——遇到相同的 `nums[i]` 跳过，双指针找到解后也跳过相同值。

```python
def three_sum(nums):
    nums.sort()
    n = len(nums)
    ans = []
    for i in range(n - 2):
        if i > 0 and nums[i] == nums[i - 1]:       # i 去重
            continue
        left, right = i + 1, n - 1
        while left < right:
            total = nums[i] + nums[left] + nums[right]
            if total < 0:
                left += 1
            elif total > 0:
                right -= 1
            else:
                ans.append([nums[i], nums[left], nums[right]])
                while left < right and nums[left] == nums[left + 1]:
                    left += 1                       # left 去重
                while left < right and nums[right] == nums[right - 1]:
                    right -= 1                      # right 去重
                left += 1
                right -= 1
    return ans
```

**复杂度**：排序 $O(n\log n)$ + 双指针 $O(n^2)$，总 $O(n^2)$。比暴力的 $O(n^3)$ 优一个量级。

**关键坑**：去重不能漏——`i` 层、`left` 层、`right` 层都要去，否则会出重复解。这是这题面试最容易丢分的地方。

### 5. 对数器：暴力 $O(n^2)$ 对比窗口 $O(n)$

左神反复强调：写完算法要用对数器对照"绝对正确的暴力解"验证。下面用 LC 3 举例，暴力解 $O(n^2)$、窗口解 $O(n)$，跑随机数据对比。

```python
import random

def brute_longest(s):
    """暴力 O(n^2)：枚举所有子串起点，遇到重复就停。"""
    n = len(s)
    ans = 0
    for i in range(n):
        seen = set()
        for j in range(i, n):
            if s[j] in seen:
                break
            seen.add(s[j])
            ans = max(ans, j - i + 1)
    return ans

def window_longest(s):
    """窗口 O(n)。"""
    last = {}
    left = ans = 0
    for right, ch in enumerate(s):
        if ch in last and last[ch] >= left:
            left = last[ch] + 1
        last[ch] = right
        ans = max(ans, right - left + 1)
    return ans

def checker(times=2000, max_n=50, chars="abcde"):
    for _ in range(times):
        n = random.randint(0, max_n)               # 含 0 和 1 的边界
        s = "".join(random.choice(chars) for _ in range(n))
        if brute_longest(s) != window_longest(s):
            print(f"出错！输入: {s}, 暴力: {brute_longest(s)}, 窗口: {window_longest(s)}")
            return False
    print("对数器验证通过！")
    return True

checker()
```

**对数器的价值**：①覆盖边界（空串、单字符、全相同字符）②不依赖人工肉眼 ③能在写错时立刻定位。把 `window_longest` 故意写错（如漏掉 `last[ch] >= left` 判断），对数器会在几次内报错——这就是它的威力。同样的对照思路可用于 LC 11（暴力 $O(n^2)$ 枚举两两组合 vs 对撞 $O(n)$）、LC 15（暴力 $O(n^3)$ vs 排序+对撞 $O(n^2)$）。

### 6. 判题套路：30 秒选定模板

| 题目特征 | 选哪个模板 |
|---|---|
| "固定大小 k 的子数组/子串" | 定长滑动窗口 |
| "最长/最短的子数组，满足某条件" | 可变滑动窗口 |
| "有序数组 + 两数/三数关系" | 对撞双指针 |
| "原地删除/去重/分区，要求 $O(1)$ 空间" | 同向双指针 |
| "无序 + 两数关系" | 先排序转对撞，或用哈希表 |

**一句话判题法**：看到"子数组/子串 + 连续区间"，先想滑动窗口；看到"有序 + 两端"，想对撞双指针；看到"原地修改 + $O(1)$ 空间"，想同向双指针。

### 7. 三个真实工程坑

**坑一：可变窗口的 `while` 写成 `if`。** 求最长时，"窗口非法才缩"应该用 `while`——可能要缩多次才合法。写成 `if` 只缩一次，答案错。这是新手最常犯的错，在 `pwwkew` 这类输入上立刻暴露。

**坑二：对撞双指针忘了排序。** LC 15 不排序直接对撞，正确性立刻崩——单调性不成立。记住：对撞双指针的前提是有序（或问题本身单调）。

**坑三：可变窗口求最短时，答案在"收缩后"取而非"合法时"取。** 收缩后窗口可能已经不合法，这时取的长度是错的。正确做法是"合法时更新答案，再缩"。看 LC 76 代码，`ans` 更新在 `while valid == len(need)` 内部、`left += 1` 之前。

**坑四：滑动窗口下标法漏掉"陈旧记录"判断。** 用 `last[ch]` 记录字符最近下标时，必须加 `last[ch] >= left` 判断，否则会把"已收缩出窗口的旧记录"误判为重复。以 `abba` 为例：

```
right=0 'a': left=0, last={'a':0},            ans=1
right=1 'b': left=0, last={'a':0,'b':1},      ans=2
right=2 'b': last['b']=1 ≥ 0 重复, left=2,     ans=2  (窗口 b)
right=3 'a': last['a']=0, 但 0 < left=2 陈旧!  不跳, left=2, ans=2  (窗口 ba)
```

正确答案 2（"ab" 或 "ba"）。若漏掉 `>= left`，`right=3` 时会把 `left` 错跳到 1，得到窗口 "bba"（含重复 b），`ans` 错算成 3。这个坑在 `abba`、`tmmzuxt` 这类"重复字符间隔较远"的输入上必现。

## 四、速查与自测

### 速查表：三模板五指标

| 模板 | 时间 | 空间 | 单调性来源 | 典型题 |
|---|---|---|---|---|
| 定长滑动窗口 | $O(n)$ | $O(1)$ | 窗口固定大小右移 | LC 239 |
| 可变滑动窗口 | $O(n)$ | $O(\Sigma)$ | `left`/`right` 只右移 | LC 3、LC 76 |
| 对撞双指针 | $O(n)$ | $O(1)$ | 有序数组、两端夹逼 | LC 11、LC 15 |
| 同向双指针 | $O(n)$ | $O(1)$ | `fast`/`slow` 只右移 | LC 26、LC 27 |

**判题一句话**：连续区间想窗口，有序两端想对撞，原地筛选想同向。

### 自测三问

**问题一：** 滑动窗口的 `for` 套 `while` 看起来像两层循环，为什么是 $O(n)$ 不是 $O(n^2)$？

**参考答案：** 关键在指针的单调性。`right` 在外层 `for` 里只右移，最多走 $n$ 步；`left` 在内层 `while` 里也只右移，全程累计最多走 $n$ 步。虽然某一步 `while` 可能执行多次，但 `left` 的总移动次数被 $n$ 封顶。把 `while` 的执行次数均摊到所有 `for` 步骤上，每步均摊 $O(1)$，总 $O(n)$。这是摊还分析（Amortized Analysis）的经典应用——单步最坏不是 $O(1)$，但均摊是。

**问题二：** LC 11 盛最多水的容器，为什么移动较短的一端而不是较长的一端？请用"排除法"论证。

**参考答案：** 面积 = 短板高度 × 宽度。假设 `height[left] < height[right]`，当前面积由 `height[left]` 决定。如果移动较长端（`right -= 1`），宽度变小，而新的短板高度 ≤ `height[left]`（因为短板还是 `height[left]` 或更小），面积只会更小——这种解一定不优于当前，可以直接排除。移动较短端（`left += 1`），才有可能遇到更高的板，让短板高度提升。每一步都排除掉一个不可能更优的解，所以 $O(n)$ 内必能找到最优。这就是对撞双指针的单调性根基——**用排除法替代枚举**。

**问题三：** 给你一道新题"找数组中和为 target 的两个数，数组无序"，该用哪个模板？如果允许预处理呢？

**参考答案：** 无序数组不适合直接对撞双指针（缺单调性）。两条路：①哈希表法，$O(n)$ 时间 $O(n)$ 空间，一次遍历边查边存；②先排序 $O(n\log n)$ 再对撞双指针 $O(n)$，总 $O(n\log n)$ 时间 $O(1)$ 空间（不计排序的栈空间）。判题时——若要求 $O(1)$ 空间且允许改原数组，选排序 + 对撞；若要求保持原数组、或要返回下标，选哈希表。这题（LC 1 两数之和）的考点正是"无序时如何获得单调性"——要么用哈希表绕过排序，要么用排序制造单调性。

### 算法思想 × 生活迁移

三模板的共同根基是"利用单调性避免重复计算"——已经排除的不再回头。

**迁移一：超市排队选最短队。** 你要选最短的收银队。暴力做法是每条队从头数到尾 $O(n)$，共 $m$ 条队 $O(mn)$。窗口式做法：站在固定位置，眼睛从左扫到右，只记住"当前最短"——扫一遍 $O(m)$ 就够。前提是"你只关心最短，不关心每条队的细节"，这就是单调性——已经排除的更长的队不再回头看。

**迁移二：双指针式查书找引用。** 在一本 500 页书里找"第 100 页提到的概念在第几页定义"。暴力做法：对第 100 页每个概念，从头翻到尾找定义 $O(n^2)$。对撞双指针式：先建索引（排序），再用两个书签一个从前往后、一个从后往前夹逼——前提是索引有序。无序就只能一页页翻（哈希表式：建目录）。

**迁移三：代码 review 找重复实现。** 一个大项目里找"重复实现的函数"。同向双指针式：把所有函数按签名排序，`fast` 扫描、`slow` 记录"上一个不重复的位置"，遇到与 `slow` 不同的就落位——这就是 LC 26 删除有序数组重复项的工程版本。前提是先排序（建函数签名索引）。

**为什么这些迁移成立：** 三模板成立的前提是"问题有单调性可用"——要么数据本身有序，要么可以通过预处理（排序/索引）制造单调性，要么问题的解空间天然单调（如"最长子串"的左右端只增不减）。生活中大多数"找最优"的问题，只要能转化成"按某维度有序后扫描"，就能套这三模板。如果问题本质无序且无法预处理（如真正的随机匹配），就只能用哈希表或暴力枚举——这也是为什么 LC 1 两数之和（无序、要下标）用哈希表而非双指针。
