# 算法课·08｜堆排序：用数据结构保证最坏情况

> 前置知识：学完第 06-07 章（快排与归并），理解"分治突破 $O(n^2)$"和"最坏情况"的概念
> 学完你能：①讲清堆的数据结构（完全二叉树 + 数组实现）②默写堆排序的"建堆 + 取顶调整"流程 ③理解堆排为什么最坏 $O(n\log n)$ 但常数大 ④知道堆的真正用武之地是 Top-K 和优先队列，不是通用排序 ⑤用对数器验证自己写的堆排对不对 ⑥按"技术的水有多深"八层面给面试官讲透堆排

## 一、问题与思路：堆的发明——为"部分排序"而生

### 1. 一个生活场景：急诊室的分诊台

医院急诊室同时来了 30 个病人，护士要根据病情严重程度决定谁先看医生。

**笨办法（全排序）**：把所有病人按病情从重到轻全部排好，最重的第一个看。每来一个新病人，要重新排序——30 个病人重排一次要 400 多次比较，新来的越多越慢，而且护士根本不关心第 15 名是谁。

**聪明办法（堆）**：护士只关心"现在最重的是谁"，不需要把所有人排好。她维护一个"最重者在最前"的结构——

- 看完最重的病人后，从剩下的病人里**快速找到下一个最重的**（取顶 + 下沉）
- 新来一个病人，**快速把他放到合适位置**（上浮）
- 任何时候，最重的那个**永远在第一个**（堆顶）

这就是堆的核心思想——**不追求全局有序，只维护"当前最值"随时可取**。

**堆的本质是什么**？急诊室护士的"分诊台"——不把所有病人排好，只保证"最重的随时可取、新来的快速归位"。1964 年 J. W. J. Williams 发明堆排序时，想的正是这件事：**为"部分排序"而生**。

### 2. 堆排序解决了一个不同的问题

前面讲的快排、归并、简单排序，都是"把整个数组排好"。但 Williams 当年想的不是"怎么排好整个数组"，而是一个更具体的问题：**"我只需要最大（或最小）的那个元素，能不能不全排序？"**

这个问题就是 Top-K 问题的雏形。比如"100 万简历里找前 50 名"，不需要把 100 万全部排好（$O(n\log n)$），只需要维护一个"前 50 名"的结构，每次来一个新元素和当前最小比较——这就是堆的用武之地。

**本质**：堆排序的真正价值不是"通用排序"（这个领域被 Timsort 占据），而是**"部分排序"**——只需要 Top K、只需要最大/最小、需要动态维护优先级的场景。堆是这些场景的最优解。

### 3. 堆是什么：完全二叉树 + 堆序性

堆（Heap）是一种特殊的二叉树，有两个性质：

1. **完全二叉树**（Complete Binary Tree）：除了最后一层，其他层都是满的；最后一层从左到右连续填充
2. **堆序性**（Heap Property）：父节点总是 ≤（或 ≥）子节点

满足"父 ≤ 子"的叫**最小堆**（Min-Heap），堆顶是最小值。满足"父 ≥ 子"的叫**最大堆**（Max-Heap），堆顶是最大值。

```
最大堆示例：          数组表示：
       9              [9, 8, 7, 6, 5, 4, 3]
      / \
     8   7            父节点 i 的子节点：2i+1, 2i+2
    / \ / \           子节点 j 的父节点：(j-1)//2
   6  5 4  3
```

**完全二叉树的核心思想**：它可以用数组紧凑表示，不需要指针。父节点 i 的子节点是 `2i+1` 和 `2i+2`，子节点 j 的父节点是 `(j-1)//2`。这个映射是纯数学的，访问 $O(1)$。这是堆能高效的关键——**用数组下标代替指针，省空间且缓存友好**。

### 4. 堆的核心思想：部分有序，不全排序

堆不保证整个数组有序，只保证"堆顶是最值"。这是堆和排序的核心区别：

- 排序：整个数组有序，任意位置元素都就位
- 堆：只有堆顶就位，其他元素"部分有序"（父比子大/小，但兄弟之间无序）

这种"部分有序"是堆的优势——**维护成本远低于全排序**。插入一个元素到有序数组要 $O(n)$（要移动后续元素），但插入堆只要 $O(\log n)$（沿着一条路径调整）。这就是堆适合"动态维护 Top-K"的原因——动态场景下堆完胜排序。

## 二、原理与实现：建堆、取顶、调整

### 1. 堆的两个核心操作

**上浮（sift up）**：子节点比父大（最大堆），交换上去，递归到根。用于插入。

```python
def sift_up(heap, i):
    """最大堆：把位置 i 的元素上浮到正确位置。"""
    while i > 0:
        parent = (i - 1) // 2
        if heap[i] <= heap[parent]:
            break
        heap[i], heap[parent] = heap[parent], heap[i]
        i = parent
```

逐行讲解：

- **`while i > 0`**：只要还没到根节点（根没有父亲），就继续往上找
- **`parent = (i - 1) // 2`**：用纯数学算父节点下标，**不需要指针**
- **`if heap[i] <= heap[parent]: break`**：已经满足"父≥子"的堆序性，**提前终止是上浮的关键**——很多插入只需 1-2 步
- **交换 + `i = parent`**：把当前节点"提升"到父亲位置，**继续在父亲位置上比**

**下沉（sift down）**：父比某个子小（最大堆），和较大的子交换，递归到叶。用于取顶后调整。

```python
def sift_down(heap, i, n):
    """最大堆：把位置 i 的元素下沉到正确位置，n 是堆大小。"""
    while True:
        left = 2 * i + 1
        right = 2 * i + 2
        largest = i
        if left < n and heap[left] > heap[largest]:
            largest = left
        if right < n and heap[right] > heap[largest]:
            largest = right
        if largest == i:
            break
        heap[i], heap[largest] = heap[largest], heap[i]
        i = largest
```

逐行讲解（**这是堆排序最核心的函数，面试必考**）：

- **`left = 2*i+1` / `right = 2*i+2`**：算左右子下标，**完全二叉树的数学映射**
- **`largest = i`**：先假设自己最大，再和左右子比——**这是 sift_down 的标准三选一框架**
- **`if left < n`**：`n` 是当前堆大小（不是数组长度），**这个边界不能漏**——取顶后末尾已不属于堆，下沉不能踩进去
- **`if left < n and heap[left] > heap[largest]`**：左子存在且比自己大，**largest 指向左子**
- **`if largest == i: break`**：自己已经是三者最大，**提前终止是下沉性能的关键**
- **交换 + `i = largest`**：把自己"压下去"，**继续在新位置上比**

**为什么有两个操作**？上浮用于"新元素插入末尾后调整"，下沉用于"堆顶被取走后用末尾元素填补再调整"。两者方向相反，分别处理"自底向上"和"自顶向下"的调整需求。

### 2. 建堆：O(n) 而非 O(n log n)

建堆有个反直觉的结论：**从空堆逐个插入是 $O(n\log n)$，但"Floyd 建堆法"是 $O(n)$**。

Floyd 建堆法：从最后一个非叶节点开始，从右到左、从下到上，对每个节点做下沉。

```python
def build_heap(nums):
    """Floyd 建堆：O(n)"""
    n = len(nums)
    # 最后一个非叶节点是 (n//2) - 1
    for i in range(n // 2 - 1, -1, -1):
        sift_down(nums, i, n)
```

逐行讲解：

- **`n // 2 - 1`**：最后一个非叶节点的下标。**为什么是这个**？因为叶节点（共 $\lceil n/2 \rceil$ 个）天然满足堆序性，不需要下沉；只有非叶节点需要调整
- **`range(..., -1, -1)`**：从后往前遍历，**自底向上调整**——这样保证下沉到一个节点时，它的子树已经是堆
- **`sift_down(nums, i, n)`**：每个非叶节点做一次下沉，**n 是完整堆大小**

**为什么是 $O(n)$ 而不是 $O(n\log n)$**？直觉上每个 sift_down 是 $O(\log n)$，共 n/2 个节点，应该是 $O(n\log n)$。但实际上：

- 大部分节点是叶节点或接近叶，下沉深度很小
- 只有少数节点（靠近根）下沉深度大
- 严格计算：$\sum_{i} \text{下沉深度}(i) = O(n)$，因为深度 h 的节点最多 $n/2^{h+1}$ 个，每个下沉 h 步，总 $\sum h \cdot n/2^{h+1} = O(n)$

这个 $O(n)$ 建堆是 Floyd 1964 年的发现，是堆排序"建堆阶段"快的关键。

### 3. 堆排序：建堆 + 反复取顶

堆排序的流程：

1. **建最大堆**：$O(n)$
2. **反复取顶 + 调整**：把堆顶（最大值）和末尾交换，堆大小减 1，对新的堆顶做 sift_down。重复 n-1 次。

```python
def heap_sort(nums):
    n = len(nums)
    # 阶段 1：建最大堆
    for i in range(n // 2 - 1, -1, -1):
        sift_down(nums, i, n)
    # 阶段 2：反复取顶
    for i in range(n - 1, 0, -1):
        nums[0], nums[i] = nums[i], nums[0]   # 最大值放到末尾
        sift_down(nums, 0, i)                  # 调整剩余堆
    return nums
```

逐行讲解（**这是面试手撕的完整模板**）：

- **阶段 1：建最大堆**——和 `build_heap` 一模一样，从 `n//2-1` 倒着下沉
- **阶段 2：`for i in range(n-1, 0, -1)`**：从末尾往前扫到下标 1（下标 0 不用动，单个元素天然有序）
- **`nums[0], nums[i] = nums[i], nums[0]`**：**把当前堆顶（最大值）和位置 i 交换**——最大值就此就位，末尾逐渐填满"从大到小"
- **`sift_down(nums, 0, i)`**：对新的堆顶做下沉，**第三个参数是 i 而不是 n**——堆大小每次减 1，已排好的末尾不再参与调整

**关键细节**：

- **建最大堆排升序**：每次取最大放末尾，末尾逐渐填满从大到小，最终升序
- **取顶后堆大小减 1**：`sift_down(nums, 0, i)` 的第三个参数是当前堆大小，不包括已排好的末尾
- **原地排序**：堆用数组本身表示，不需要额外空间（除几个临时变量）

### 4. 复杂度分析

**时间复杂度**：

- 建堆：$O(n)$
- 取顶 + 调整：n-1 次，每次 $O(\log n)$，共 $O(n\log n)$
- 总：$O(n\log n)$，且**最好、平均、最坏都是 $O(n\log n)$**

**为什么堆排最坏也是 $O(n\log n)$**？因为堆的调整路径长度是 $\log n$（完全二叉树高度），与数据分布无关。不像快排可能切偏退化 $O(n^2)$，堆的"树高"恒定 $\log n$。

**空间复杂度**：$O(1)$，原地排序。

### 5. 五指标评价

| 指标 | 堆排 | 说明 |
|---|---|---|
| 最好 | $O(n\log n)$ | 不自适应 |
| 平均 | $O(n\log n)$ | 树高恒定 |
| 最坏 | $O(n\log n)$ | **无 $O(n^2)$ 陷阱** |
| 空间 | $O(1)$ | 原地 |
| 稳定 | 否 | 取顶时跨距离交换 |
| 自适应 | 否 | 不识别有序 |
| 缓存 | **差** | 父子节点跨度大，缓存不友好 |

**堆排的"致命伤"是缓存不友好**。父子节点在数组里跨度是 2 倍（父 i，子 2i+1），不是顺序访问。CPU 缓存预取失效，实际性能比快排慢 2-3 倍。这就是为什么"堆排最坏 $O(n\log n)$ 比快排好，但生产还是用快排"——常数和缓存决定实际速度。

### 6. 堆排 vs 快排 vs 归并

| 维度 | 快排 | 归并 | 堆排 |
|---|---|---|---|
| 平均 | $O(n\log n)$ | $O(n\log n)$ | $O(n\log n)$ |
| 最坏 | $O(n^2)$ | $O(n\log n)$ | $O(n\log n)$ |
| 空间 | $O(\log n)$ | $O(n)$ | $O(1)$ |
| 稳定 | 否 | 是 | 否 |
| 缓存 | 极好 | 中 | 差 |
| 实际速度 | 最快 | 中 | 最慢 |
| Top-K | 不适合 | 不适合 | **最优** |
| 优先队列 | 不适合 | 不适合 | **最优** |

**堆排的定位**：不是通用排序（被快排/Timsort 压制），而是 Top-K 和优先队列的最优解。这是堆的真正用武之地。

## 三、实践与面试：堆的真正价值是 Top-K 和优先队列

### 1. 教科书做法 vs 生产做法

| 场景 | 教科书做法 | 生产做法 | 原因 |
|---|---|---|---|
| 通用排序 | 手写堆排 | `sorted()`（Timsort） | 堆排缓存不友好，慢 |
| Top-K | 手写堆 | `heapq.nlargest/nsmallest` | 库已优化 |
| 优先队列 | 手写堆 | `heapq` / `PriorityQueue` | 库已实现 |
| 流式数据 Top-K | 手写小顶堆 | `heapq` 维护大小 K 的堆 | 堆是流式 Top-K 最优解 |
| 实时中位数 | 手写双堆 | 两个 heapq | 双堆法是标准解 |

**生产里堆排序几乎不用，但堆数据结构无处不在**：Python 的 `heapq`、Java 的 `PriorityQueue`、C++ 的 `std::priority_queue`、Go 的 `container/heap`——所有这些库底层都是堆。堆是"动态维护优先级"的标准数据结构。

### 2. 三个真实工程坑

**坑一：用堆排排通用数据，明显比 Timsort 慢。**

```python
import heapq, time
data = list(range(10**6))
import random; random.shuffle(data)

# 堆排
start = time.time()
heapq.heapify(data[:])
result = [heapq.heappop(data[:]) for _ in range(len(data))]
print(f"heap: {time.time()-start:.2f}s")

# Timsort
start = time.time()
sorted(data)
print(f"tim: {time.time()-start:.2f}s")
```

症状：堆排比 Timsort 慢 2-3 倍。根因：堆排的父子节点跨度大，缓存不友好。修复：通用排序用 `sorted()`，堆排只在 Top-K 和优先队列场景用。

**坑二：Top-K 用全排序而不是堆。**

```python
# 错误：100 万数据求 Top 10，用全排序
data = list(range(10**6))
random.shuffle(data)
top10 = sorted(data, reverse=True)[:10]   # O(n log n)，浪费

# 正确：用大小 10 的小顶堆
import heapq
top10 = heapq.nsmallest(10, data)   # O(n log k)，k=10
```

症状：Top-K 用全排序，$O(n\log n)$，数据量大时慢。根因：只需要前 K 个，不需要全排序。修复：用 `heapq.nlargest/nsmallest`，复杂度 $O(n\log k)$，k 远小于 n 时快几个数量级。

**坑三：流式 Top-K 用列表 + 每次排序。**

```python
# 错误：每来一个新数据，把列表排序后取 Top K
class TopKBad:
    def __init__(self, k):
        self.k = k
        self.data = []
    def add(self, x):
        self.data.append(x)
        self.data.sort(reverse=True)   # 每次 O(n log n)
        self.data = self.data[:self.k]

# 正确：用大小 K 的小顶堆
class TopKGood:
    def __init__(self, k):
        self.k = k
        self.heap = []
    def add(self, x):
        if len(self.heap) < self.k:
            heapq.heappush(self.heap, x)
        elif x > self.heap[0]:
            heapq.heapreplace(self.heap, x)   # O(log k)
```

症状：流式数据每次排序，$O(n \cdot n\log n)$ 灾难。根因：没有利用"只维护 Top K"的结构。修复：用大小 K 的堆，每次插入 $O(\log k)$。

### 3. 一个生产级例子：实时 Top-K 与双堆求中位数

```python
import heapq

class TopKTracker:
    """实时维护数据流中的 Top K 大元素。"""
    def __init__(self, k):
        self.k = k
        self.heap = []   # 小顶堆，堆顶是 Top K 中最小的

    def add(self, num):
        if len(self.heap) < self.k:
            heapq.heappush(self.heap, num)
        elif num > self.heap[0]:
            heapq.heapreplace(self.heap, num)

    def topk(self):
        return sorted(self.heap, reverse=True)


class MedianFinder:
    """双堆法求流式数据中位数。"""
    def __init__(self):
        self.max_heap = []   # 前半部分（最大堆，存较小的一半）
        self.min_heap = []   # 后半部分（最小堆，存较大的一半）

    def add(self, num):
        # 先加到 max_heap（取负实现最大堆）
        heapq.heappush(self.max_heap, -num)
        # 平衡：max_heap 堆顶 ≤ min_heap 堆顶
        if self.max_heap and self.min_heap and \
           -self.max_heap[0] > self.min_heap[0]:
            heapq.heappush(self.min_heap, -heapq.heappop(self.max_heap))
        # 大小平衡：max_heap 可以比 min_heap 多 1
        if len(self.max_heap) > len(self.min_heap) + 1:
            heapq.heappush(self.min_heap, -heapq.heappop(self.max_heap))
        if len(self.min_heap) > len(self.max_heap):
            heapq.heappush(self.max_heap, -heapq.heappop(self.min_heap))

    def median(self):
        if len(self.max_heap) > len(self.min_heap):
            return -self.max_heap[0]
        return (-self.max_heap[0] + self.min_heap[0]) / 2
```

这两个例子体现了堆的真正价值：**动态维护"部分有序"结构**。Top-K 维护前 K 大，中位数维护前一半和后一半——都不需要全排序，堆是这些场景的最优解。

### 4. 对数器：左神风格的验证方法

左神反复强调：**写完排序不要靠肉眼检查，要用对数器验证**。对数器的思路是——生成随机数据，用你的堆排和标准库排序（绝对正确）对比，跑足够多次，只要有一次不一致就说明你的堆排有 bug。

```python
import random

def sift_down(heap, i, n):
    """最大堆：把位置 i 的元素下沉到正确位置，n 是堆大小。"""
    while True:
        left = 2 * i + 1
        right = 2 * i + 2
        largest = i
        if left < n and heap[left] > heap[largest]:
            largest = left
        if right < n and heap[right] > heap[largest]:
            largest = right
        if largest == i:
            break
        heap[i], heap[largest] = heap[largest], heap[i]
        i = largest

def heap_sort(nums):
    """你的堆排实现。"""
    n = len(nums)
    # 阶段 1：建最大堆
    for i in range(n // 2 - 1, -1, -1):
        sift_down(nums, i, n)
    # 阶段 2：反复取顶
    for i in range(n - 1, 0, -1):
        nums[0], nums[i] = nums[i], nums[0]
        sift_down(nums, 0, i)
    return nums

def checker(test_times=5000, max_n=100, max_val=1000):
    """对数器：跑 test_times 次，每次随机数据对比。"""
    for _ in range(test_times):
        n = random.randint(0, max_n)         # 随机长度（含 0 和 1）
        arr1 = [random.randint(0, max_val) for _ in range(n)]
        arr2 = arr1[:]                        # 深拷贝
        heap_sort(arr1)                       # 你的堆排
        arr2.sort()                           # 标准库（绝对正确）
        if arr1 != arr2:
            print(f"出错！输入: {arr2}, 你的结果: {arr1}")
            return False
    print("对数器验证通过！")
    return True

checker()
```

**对数器的价值**：

- **覆盖边界**：随机长度包含 0 和 1（空数组和单元素数组），随机值包含重复值，能覆盖手写时容易遗漏的边界——比如 `n // 2 - 1` 在 n=1 时是 -1（循环不执行），`sift_down` 的 `left < n` 判断在叶子节点能否正确退出
- **不依赖人工**：跑 5000 次只要几秒，比手动构造测试用例全面得多
- **左神原话**："对数器是算法工程师的基本功。写完一个算法，第一件事就是写对数器，而不是跑几个例子看看对不对。"

**对高手的启发**：对数器思想不止用于排序——任何"有标准答案可对照"的算法都能用对数器。写完红黑树？对数器对比 `std::map`。写完哈希表？对数器对比 `dict`。这是"工程化验证"的思维，比"理论上推导正确"更可靠。

### 5. 面试手撕模板（按"技术的水有多深"八层面）

**面试手撕的完整模板**就是上面的 `sift_down` + `heap_sort`，背到能默写。**面试时怎么讲**（按"技术的水有多深"八层面，挑重点讲）：

1. **问题层**（核心思想）：通用排序要"全排好"，但很多场景只需要"最值或前 K"——Top-K、动态中位数、优先队列。堆就是为"部分排序"而生：用完全二叉树 + 数组下标，$O(1)$ 拿最值，$O(\log n)$ 插入/取顶。
2. **原理层**：堆 = 完全二叉树 + 堆序性（父 ≥ 子或父 ≤ 子）。用数组实现，父 i 的子是 `2i+1`/`2i+2`，子 j 的父是 `(j-1)//2`。两个核心操作：上浮（插入用）、下沉（取顶用）。堆排序 = 建最大堆 + 反复取顶放末尾。
3. **实现层**：`sift_down` 是核心——比较 i、left、right 三者，最大的放 i，递归向下（**这是必背函数**）。建堆从 `n//2-1` 倒着下沉（Floyd 法）。堆排主体：建堆后，每次 `swap(0, i)` 把最大放末尾，再 `sift_down(nums, 0, i)` 调整剩余堆。
4. **优劣层**：最好 = 平均 = 最坏 = $O(n\log n)$，空间 $O(1)$ 原地；但不稳定（取顶跨距离交换）、缓存不友好（父子跨度 2 倍）、不自适应。**最坏 $O(n\log n)$ 是堆排相对快排的最大优势**。
5. **边界层**：$n=0/1$ 直接返回；建堆循环从 `n//2-1` 开始（最后一个非叶节点）；`sift_down` 的第三个参数 `i` 是当前堆大小（不含已排好的末尾）；空堆的 `left < n` 判断不能漏——少了就会越界访问已排好区段。
6. **演进层**：Williams 1964 原始堆排（先建堆 + 反复取顶）→ Floyd 1964 同年提出 $O(n)$ 建堆法（取代逐个插入的 $O(n\log n)$）→ 后续派生 Top-K、优先队列、双堆中位数等应用。**Floyd 的 $O(n)$ 建堆是堆排效率的关键贡献**。
7. **应用层**：生产里堆排几乎不用作通用排序（被 Timsort/快排压制），但堆数据结构无处不在：Python `heapq`、Java `PriorityQueue`、C++ `std::priority_queue`、Go `container/heap`。Top-K、流式中位数、优先队列、Dijkstra 最短路都用堆。
8. **扩展层**：Top-K 流式（小顶堆维护大小 K）→ 双堆求中位数（前半最大堆 + 后半最小堆）→ 优先队列调度（OS 进程/任务调度）→ k 路归并（多个有序流合并）→ 索引堆（带删除的优先队列，Dijkstra 优化）。

**面试加分点**：主动提"Floyd 建堆是 $O(n)$ 不是 $O(n\log n)$"，并说出"大部分节点是叶节点，下沉深度小，级数 $\sum h/2^h$ 收敛"——这能立刻区分你和"只会背模板"的候选人。再补一句"堆排最坏 $O(n\log n)$ 比快排好，但生产通用排序还是用快排/Timsort，因为缓存不友好、不稳定"——直接留下深度印象。

## 四、速查与自测

### 速查表：堆排序五指标

| 指标 | 堆排 | 说明 |
|---|---|---|
| 最好 | $O(n\log n)$ | 不自适应 |
| 平均 | $O(n\log n)$ | 树高恒定 $\log n$ |
| 最坏 | $O(n\log n)$ | **无 $O(n^2)$ 陷阱** |
| 空间 | $O(1)$ | 原地 |
| 稳定 | 否 | 取顶跨距离交换 |
| 自适应 | 否 | 不识别有序 |
| 缓存 | **差** | 父子跨度大 |

**堆的核心操作复杂度**：

| 操作 | 复杂度 |
|---|---|
| 建堆（Floyd） | $O(n)$ |
| 插入（push） | $O(\log n)$ |
| 取顶（pop） | $O(\log n)$ |
| 查顶（peek） | $O(1)$ |
| Top-K 维护 | $O(n\log k)$ |

### 自测三问

**问题一：** 建堆是 $O(n)$ 还是 $O(n\log n)$？请解释 Floyd 建堆法为什么是 $O(n)$。

**参考答案：** Floyd 建堆是 $O(n)$。直觉上"n/2 个节点每个 sift down $O(\log n)$"应该是 $O(n\log n)$，但实际是 $O(n)$。原因是大部分节点是叶节点或接近叶，下沉深度很小——深度 h 的节点最多 $n/2^{h+1}$ 个，每个下沉 h 步，总工作量 $\sum_{h=0}^{\log n} h \cdot n/2^{h+1}$。这个级数收敛到 $O(n)$（因为 $h/2^h$ 是收敛级数）。只有少数靠近根的节点下沉深度大，但它们数量少。所以总 $O(n)$。这是堆排序"建堆阶段"快的关键。

**问题二：** 堆排最坏 $O(n\log n)$ 比快排好，为什么生产通用排序不用堆排而用快排（或 Timsort）？

**参考答案：** 两个原因。①缓存不友好：堆的父子节点在数组里跨度是 2 倍（父 i，子 2i+1），不是顺序访问，CPU 缓存预取失效，实际比快排慢 2-3 倍。快排的分区是顺序扫描，缓存命中率极高。②不稳定：堆排取顶时跨距离交换，破坏相等元素相对顺序，对象排序会出 bug。Timsort 既稳定又自适应（对基本有序数据接近 $O(n)$），最坏也是 $O(n\log n)$，综合更优。**$O$ 符号隐藏了常数和缓存，工程选型不能只看 $O$**——堆排的 $O(n\log n)$ 比快排的 $O(n\log n)$ 慢几倍，因为常数和缓存。

**问题三：** 堆排的真正用武之地不是通用排序，而是 Top-K 和优先队列。请用一个流式 Top-K 的例子说明堆为什么是这些场景的最优解。

**参考答案：** 流式 Top-K：数据一个一个来，要实时维护前 K 大。用排序？不行——每来一个数据重新排序 $O(n\log n)$，数据量大时灾难。用列表 + 每次找最小替换？$O(nK)$，也慢。用大小 K 的小顶堆？每次来新数据，和堆顶（当前 Top K 中最小）比较，大于堆顶就替换并调整。每次插入 $O(\log k)$，总共 $O(n\log k)$。k 远小于 n 时（如 100 万数据求 Top 10），$O(n\log 10) \approx O(n)$，比排序 $O(n\log n)$ 快几个数量级。堆是最优解因为：①动态维护"部分有序"——只保证堆顶是最值，不全排序，维护成本低 $O(\log k)$。②流式友好——来一个处理一个，不需要等全部数据到齐。这是堆"部分排序"特性的用武之地。

### 算法思想 × 生活迁移

堆的核心思想是"**部分有序 + 动态维护最值**"——不追求全局有序，只维护"当前最值"随时可取。

**迁移一：管理一个"候选池"。** 招聘时维护一个"前 10 候选人"的池子，来一个新简历就评估，比池里最差的强就替换。这是 Top-K 堆的思路——不追求把所有候选人都排好，只维护"当前 Top 10"随时可面试。比"全部面完再选"高效，且适合"边来边选"的流式场景。

**迁移二：优先级队列管理任务。** 待办事项按优先级排，但优先级会变（紧急任务插队）。用堆维护——堆顶永远是最高优先级，随时可取；新任务插入 $O(\log n)$，比"每次重排"高效。这是 OS 进程调度、消息队列、任务调度器的底层思路。

**迁移三：动态中位数监控。** 监控系统指标的中位数（比平均数抗异常值），数据是流式的。用双堆（前半最大堆 + 后半最小堆），中位数恒在两堆顶之间，每次新数据 $O(\log n)$ 调整。比"每次重算中位数"高效。

**为什么这些迁移成立：** 堆成立的前提是"只需要最值或部分有序，不需要全排序"。这个前提在候选池管理、任务调度、监控中位数里都成立——都是"动态场景、只需要最值或 Top K"。迁移成立的关键是"识别场景是动态的、只需要部分结果"——如果是静态全排序场景，用排序更直接；如果是动态部分有序场景，堆是唯一选择。**判断"静态全排序 vs 动态部分有序"是选排序还是选堆的关键。**
