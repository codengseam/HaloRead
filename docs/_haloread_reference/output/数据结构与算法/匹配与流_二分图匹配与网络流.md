# 算法课·33｜匹配与流：二分图匹配与网络流的对偶之美

> 学完你能：①给小白讲清二分图匹配与匈牙利算法 ②默写 Edmonds-Karp 求最大流 ③说清 Dinic 分层为何更快 ④解释"最大流=最小割"对偶定理 ⑤用对数器验证最大流正确性 ⑥看清二分图匹配与最大流的统一性

## 一、问题与思路：从分配难题到"流"的统一视角

### 1. 一个反直觉起点：分配问题能用水管求解

公司有 5 名员工、5 项任务，每名员工只擅长其中几项，每人至多做一项、每项至多一人做。问最多能安排几对"员工-任务"？这是经典的二分图匹配（Bipartite Matching）。反直觉的地方在于：这个"配对"问题，可以转化成"从水管一头打水到另一头，最多能流多少"——而水管视角还顺手奉上一个定理：能流过的最大水量，恰好等于"最细的一刀割断所有管道"的容量。这就是网络流（Network Flow）的对偶之美。

为什么要绕这道弯？因为匹配问题靠匈牙利算法（Hungarian Algorithm）单独求解时，每次都要重新搜一遍增广路；把它装进"流"的框架后，统一的求解器（Ford-Fulkerson 系列）一次就能处理匹配、项目调度、二分图最小点覆盖等一大类问题。流的视角是"图论 + 优化"的集大成，对偶定理则是这套框架的数学之花。

### 2. 二分图匹配：定义与"增广路"直觉

二分图（Bipartite Graph）指顶点能分成左右两半、所有边都跨左右、同侧无边的图。匹配（Matching）是一组"两两无公共端点"的边。最大匹配就是边数最多的那个匹配。

理解匈牙利算法靠一个动作——**找增广路（Augmenting Path）**。增广路是一条从左侧未匹配点出发、交替走"非匹配边 / 匹配边 / 非匹配边 …"最终到达右侧未匹配点的路径。把这条路上的匹配边和非匹配边"翻转"（匹配变非匹配、非匹配变匹配），匹配数就 +1。

这里有个反直觉点：增广路里"翻转匹配边"意味着已经配好的一对会被拆开。但拆开不是为了减少匹配，而是为了让被拆的左点去抢另一个右点——一次翻转净增 1 条匹配，总数不降反升。**增广路 = 调整已有匹配，换取更大匹配**。这就是匈牙利算法的核心思想。

### 3. 网络流：把匹配问题"装进管道"

二分图匹配靠"找增广路"求解。而增广路这个概念，在另一个问题里也出现了——最大流（Maximum Flow）。最大流问题：给定带源点 s、汇点 t 的有向图，每条边有容量（Capacity），问从 s 到 t 最多能"压"多少流量。

求解最大流的通用方法叫 Ford-Fulkerson 方法（注意是"方法"不是"算法"——它给框架，找增广路的具体策略可替换）。框架极简：只要还能找到一条 s→t 的增广路，就把流量压过去，直到找不到为止。增广路在这里指"还能加流量的路径"，每次压一个"瓶颈容量"的流量。

二分图匹配就是最大流的特例：左点连超级源 s（容量 1），右点连超级汇 t（容量 1），原匹配边容量设为 1。这套图的最大流值，就是二分图的最大匹配数。所以匈牙利算法其实是 Ford-Fulkerson 在"01 容量二分图"上的特例——这是统一性的第一个证据。

### 4. 历史地位与对偶之美

匹配理论始于 1931 年 König，最大流的 Ford-Fulkerson 方法 1956 年提出，Edmonds-Karp（BFS 优化）1972 年，Dinic 分层算法 1970 年。这些算法不只是"求一个数"，它们共同托起了一个对偶定理——**最大流 = 最小割（Max-Flow Min-Cut）**。

割（Cut）是把顶点分成 S 和 T=V−S（s∈S, t∈T）的一种切法，割的容量是"从 S 指向 T 的所有边容量之和"。最小割就是容量最小的切法。对偶定理说：网络能通过的最大流量，恰好等于"切断 s 到 t 所需的最小代价"。

这是数学之美：一个"最大化"问题和一个"最小化"问题，看似方向相反，最优值却恒等。这种"对偶"在后续线性规划、拉格朗日对偶、博弈论的 minimax 里反复出现——网络流是认识"对偶"思想最直观的入口。本章核心论点也在此：网络流是图论 + 优化的集大成，对偶定理是它的数学之美。

## 二、原理与实现：匈牙利算法与最大流三连

### 1. 匈牙利算法：用增广路找最大匹配

```python
def hungarian(adj, n_left, n_right):
    """二分图最大匹配 - 匈牙利算法。adj[i] 是左点 i 能连到的右点列表。"""
    match_r = [-1] * n_right   # 右点当前匹配的左点，-1 表示未匹配

    def try_kuhn(u, visited):
        for v in adj[u]:                # 遍历左点 u 能连的所有右点
            if visited[v]:
                continue
            visited[v] = True
            # 右点 v 未匹配，或 v 的原配能腾位 → 增广成功
            if match_r[v] == -1 or try_kuhn(match_r[v], visited):
                match_r[v] = u
                return True
        return False

    matching = 0
    for u in range(n_left):
        visited = [False] * n_right     # 每个左点重新找增广路时重置访问标记
        if try_kuhn(u, visited):
            matching += 1
    return matching, match_r
```

逐行讲透：

- **`match_r`**：记录每个右点当前匹配的左点。用右点索引是因为"翻转"操作要快速定位某右点的现配。
- **`try_kuhn(u, visited)`**：为左点 u 找增广路。返回 True 表示找到（u 能配上某个右点）。
- **`visited`**：每轮重置，防止同一条增广路里重复访问同一右点导致死循环。
- **`if match_r[v] == -1 or try_kuhn(match_r[v], visited)`**：这是增广路翻转的关键。若 v 未匹配，直接配；若 v 已配给 u'，就递归问 u' 能不能换一个右点——能换则 v 让给 u，整个匹配数 +1。
- **`matching += 1`**：每次成功配对左点 u，匹配数 +1。注意这里"配对 u"可能拆散了别的左点，但净增 1。

**复杂度**：n 个左点，每个最多遍历 m 条边，最坏 $O(V \cdot E)$，其中 V=n_left+n_right，E=边数。这是匈牙利算法的标准上界。

**用一个例子走一遍**。左点 4 个（u0-u3）、右点 4 个（v0-v3），邻接：u0→{v0,v1}，u1→{v0,v2}，u2→{v1}，u3→{v2,v3}。

```
处理 u0：增广路 u0→v0，v0 未匹配，配对。       match_r=[0,-1,-1,-1]  匹配=1
处理 u1：试 v0（已配 u0）→ 递归 u0 试 v1（未匹配）
         → u0 换到 v1，v0 让给 u1。            match_r=[1,0,-1,-1]  匹配=2
处理 u2：试 v1（已配 u0）→ 递归 u0 试 v0（已配 u1）
         → 递归 u1 试 v2（未匹配）
         → 链式翻转：u1→v2, u0→v0, u2→v1。     match_r=[2,1,2,-1]   匹配=3
处理 u3：试 v2（已配 u1）→ 递归 u1 试 v0（已配 u0）
         → 递归 u0 试 v1（已配 u2）→ u2 无他选，回溯失败
         再试 v3（未匹配），直接配对。           match_r=[2,1,2,3]    匹配=4
最终最大匹配 = 4
```

**关键观察**：u1 处理时"拆散 u0-v0"不是损失，而是让 u0 去占 v1、腾出 v0 给 u1，净增 1。u3 处理时链式翻转失败（u2 卡死，无路可换），但 v3 直接可用，照样 +1。增广路的"翻转"是有方向的——只在能净增时才成功，失败则原匹配不动、换一条路试。这就是"调整已有匹配换更大匹配"的具体形态。

### 2. Ford-Fulkerson 与 Edmonds-Karp：BFS 找增广路

Ford-Fulkerson 方法框架不规定"怎么找增广路"，只要找得到就行。但"怎么找"决定了复杂度：

- 用 DFS 随便找：最坏 $O(E \cdot |f^*|)$，其中 $|f^*|$ 是最大流值。容量是整数时还能跑，若是浮点或大容量，可能不收敛。
- Edmonds-Karp：固定用 BFS 找最短增广路，最坏 $O(V \cdot E^2)$，且与流值无关。这是工程首选。

Edmonds-Karp 实现：

```python
from collections import deque

def edmonds_karp(cap, s, t, n):
    """最大流 - Edmonds-Karp。cap[u][v] 是容量，返回最大流值。"""
    flow = 0
    while True:
        # BFS 找一条 s->t 的最短增广路
        parent = [-1] * n
        parent[s] = s
        q = deque([s])
        while q and parent[t] == -1:
            u = q.popleft()
            for v in range(n):
                # 只走还有剩余容量的边
                if parent[v] == -1 and cap[u][v] > 0:
                    parent[v] = u
                    q.append(v)
        if parent[t] == -1:        # 找不到增广路，结束
            break
        # 求瓶颈容量
        bottleneck = float('inf')
        v = t
        while v != s:
            u = parent[v]
            bottleneck = min(bottleneck, cap[u][v])
            v = u
        # 沿增广路更新容量（正向减、反向加）
        v = t
        while v != s:
            u = parent[v]
            cap[u][v] -= bottleneck
            cap[v][u] += bottleneck   # 反向边，支持"反悔"
            v = u
        flow += bottleneck
    return flow
```

逐行讲透重点：

- **BFS 找最短增广路**：`parent` 数组同时承担"是否访问过"和"回溯路径"两个职责。BFS 保证找到的是边数最少的增广路，这是 Edmonds-Karp 复杂度可证明的关键。
- **`cap[u][v] > 0`**：只走还有剩余容量的边。已满的边跳过。
- **瓶颈容量**：增广路上剩余容量最小的那条边决定本次能压多少流量。
- **反向边加容量**：这是网络流最反直觉、也最关键的一步。给反向边加容量等于"允许后续增广路把这次压过去的流量退回来"。没有反向边，算法会卡在局部最优；有了反向边，每条增广路都能"纠错"，最终保证全局最优。

**反向边为何能纠错**——一个微缩例子。图：s→a(1), s→b(1), a→b(1), a→t(1), b→t(1)。若第一次走了 s→a→b→t，压 1 单位，则 a→t 和 s→b 还空着但已无法直接利用。Edmonds-Karp 下一轮 BFS 会走 s→b→a（走 a→b 的反向边）→t，再压 1 单位。两次合计 2，正是最大流。反向边让"先压错的那次"被纠正——这是流的本质特性：流量可回退。

**复杂度**：Edmonds-Karp 最坏 $O(V \cdot E^2)$。每条边成为瓶颈的次数有上界，BFS 保证增广路条数 $O(V \cdot E)$，每条增广路 BFS 是 $O(E)$，相乘即得。

### 3. Dinic：分层图 + 阻塞流

Edmonds-Karp 每次只找一条增广路就重跑 BFS，BFS 的成果没被复用。Dinic（1970）的优化：一次 BFS 建好"分层图"，之后用 DFS 在分层图里一口气找完所有增广路（阻塞流），再重新分层。

```python
from collections import deque

def dinic(cap, s, t, n):
    """最大流 - Dinic。cap[u][v] 是容量，返回最大流值。"""
    flow = 0
    level = [0] * n

    def bfs():
        # 建 s 到各点的最短层数
        for i in range(n):
            level[i] = -1
        level[s] = 0
        q = deque([s])
        while q:
            u = q.popleft()
            for v in range(n):
                if level[v] == -1 and cap[u][v] > 0:
                    level[v] = level[u] + 1
                    q.append(v)
        return level[t] != -1       # t 是否可达

    def dfs(u, pushed):
        if u == t:
            return pushed
        while ptr[u] < n:
            v = ptr[u]
            if level[v] == level[u] + 1 and cap[u][v] > 0:
                d = dfs(v, min(pushed, cap[u][v]))
                if d > 0:
                    cap[u][v] -= d
                    cap[v][u] += d
                    return d
            ptr[u] += 1
        return 0

    while bfs():
        ptr = [0] * n               # 当前弧优化：每点记录下一条待试边
        while True:
            f = dfs(s, float('inf'))
            if f == 0:
                break
            flow += f
    return flow
```

逐行讲透重点：

- **`bfs()`**：建分层图，`level[v] = level[u] + 1` 只允许向"下一层"走。一次 BFS 成果供后续多次 DFS 复用。
- **`level[v] == level[u] + 1`**：DFS 只在分层图同向走，避免兜圈子。
- **`ptr`（当前弧优化）**：记录每个点"已经试过且失败"的边，下次直接跳过。没有它会退化，加上才到 $O(V^2 E)$ 的紧上界；单位容量图更快，达 $O(E\sqrt V)$。
- **阻塞流**：DFS 一直找直到分层图里再无 s→t 路径，称"该分层图的阻塞流"。然后重新 BFS。

**复杂度**：Dinic 最坏 $O(V^2 E)$，二分图匹配特例 $O(E\sqrt V)$。BFS 次数 $O(V)$（每次分层至少让最短增广路长度 +1），每层阻塞流用 DFS 找 $O(VE)$。Dinic 是竞赛和工程求最大流的主力。

### 4. 最大流 = 最小割：对偶定理

为什么"最大流 = 最小割"？给一个不靠符号堆砌的证明骨架。

**方向一：流 ≤ 任意割。** 任何从 s 到 t 的流量都必须穿过任意一个 s-t 割（因为割把 s 和 t 分开了）。所以"流值 ≤ 该割的容量"，对所有割成立，自然也 ≤ 最小割。

**方向二：当 Ford-Fulkerson 终止时，流值 = 某个割的容量。** 算法终止意味着再也找不到 s→t 增广路。此时把"从 s 可达的点"归为 S，其余归为 T。这是一个合法割（s∈S, t∈T，因为 t 不可达）。这个割里所有 S→T 的边都满了（否则还能增广），所有 T→S 的边流量都是 0（否则能反增广）。所以这个割的容量 = 当前流值。

方向一给下界（流 ≤ 最小割），方向二给上界（存在一个割 = 流值）。两相结合：**最大流值 = 最小割容量**。这就是对偶定理。

**用一个微缩图验证**。s→a(3), s→b(2), a→t(2), b→t(3), a→b(1)。手算最大流：可走 s→a→t 压 2、s→b→t 压 2、s→a→b→t 压 1，合计 5（此时 s→a 剩 0、s→b 剩 0，无法再增广）。再看割：取 S={s}, T={a,b,t}，割边 s→a+s→b = 3+2 = 5；取 S={s,a,b}, T={t}，割边 a→t+b→t = 2+3 = 5；其他割都更大。最小割 = 5，恰好等于最大流 5。定理在一个 5 条边的小图上就立得住。

这为什么算"数学之美"？因为它揭示了一个非平凡的等式：一个最大化问题（压最多流量）和一个最小化问题（最少代价切断），最优值恒等。这种"相反目标同解"的结构，正是对偶性（Duality）的标志。后续线性规划的对偶定理、博弈论的 minimax 定理，都是同一思想的回响。理解最大流-最小割，等于拿到了"对偶思维"的入场券。

### 5. 五指标评价

| 指标 | 匈牙利 | Edmonds-Karp | Dinic |
|---|---|---|---|
| 最坏 | $O(V \cdot E)$ | $O(V \cdot E^2)$ | $O(V^2 E)$ |
| 二分图特例 | $O(V \cdot E)$ | $O(V \cdot E^2)$ | $O(E\sqrt V)$ |
| 空间 | $O(V+E)$ | $O(V^2)$（邻接矩阵） | $O(V^2)$ |
| 依赖流值 | 否 | 否 | 否 |
| 工程首选 | 二分图匹配 | 教学/小图 | 竞赛/大图 |

注：Edmonds-Karp 用邻接矩阵实现是 $O(V^2)$ 空间，改邻接表可降到 $O(V+E)$。匈牙利在二分图匹配上最简洁，不需要建超级源汇。

**算法选择指南**：

- **纯二分图匹配**：直接匈牙利，代码短、常数小，无需建超级源汇。
- **教学或小图最大流**：Edmonds-Karp，BFS 逻辑直白、易调试，复杂度够用。
- **竞赛或中等以上规模最大流**：Dinic，分层 + 当前弧优化实测最快，是首选。
- **超大图或需要更优理论界**：Push-Relabel（预流推进），CLRS 第 26 章也讲它，最坏 $O(V^3)$，实测与 Dinic 互有胜负，工程实现更复杂。

本章重点放在 Dinic 之前——因为它和匈牙利、Edmonds-Karp 同属"增广路"家族，思路一脉相承，便于看清演进脉络；Push-Relabel 是另一条技术路线（"预流推进"而非"找增广路"），超出选读章篇幅，知道有这把刀即可。

### 6. 为什么这是"选读"章

匹配与流在面试中属低频考点——多数岗位不会手撕 Dinic。但这一章的价值不在"应试"，在三件事：①它是"图论 + 优化"的集大成，把匹配、调度、覆盖等问题统一进一个框架；②对偶定理是认识"对偶思维"最直观的入口，后续线性规划、博弈论都绕不开；③增广路 + 反向边的"可回退尝试"思想，是理解一大类优化算法的钥匙。学这章是为长线视野，不是为某道面试题。这也呼应了本章核心论点——网络流的美，在对偶。

## 三、实践与面试：对数器与选读题

### 1. 对数器：用不变量 + 交叉验证

最大流没有标准库可直接对照（Python 标准库不提供），所以对数器换两招：①用独立的朴素 Ford-Fulkerson（DFS 版）与 Edmonds-Karp 交叉比对结果；②用"容量约束"和"流量守恒"两个不变量自检。第二招比第一招更可靠——它直接对应"流"的数学定义。

```python
import random
from collections import deque

def ford_fulkerson_dfs(cap, s, t, n):
    """朴素 Ford-Fulkerson（DFS 找增广路），作为对照基准。"""
    cap = [row[:] for row in cap]      # 深拷贝，不污染原容量
    flow = 0
    while True:
        visited = [False] * n
        def dfs(u, bottleneck):
            if u == t:
                return bottleneck
            visited[u] = True
            for v in range(n):
                if not visited[v] and cap[u][v] > 0:
                    d = dfs(v, min(bottleneck, cap[u][v]))
                    if d > 0:
                        cap[u][v] -= d
                        cap[v][u] += d
                        return d
            return 0
        pushed = dfs(s, float('inf'))
        if pushed == 0:
            break
        flow += pushed
    return flow

def check_invariants(cap_orig, cap_final, s, t, n):
    """自检：每条边流量 ≥0 且 ≤ 容量；中间点流量守恒；源出=汇入。"""
    inflow = [0] * n
    outflow = [0] * n
    for u in range(n):
        for v in range(n):
            f = cap_orig[u][v] - cap_final[u][v]   # 实际流量 = 原容量 - 剩余
            if f < 0:                              # 正向流量不应为负
                return False
            outflow[u] += f
            inflow[v] += f
    for u in range(n):
        if u in (s, t):
            continue
        if inflow[u] != outflow[u]:                # 中间点流入=流出
            return False
    return inflow[t] == outflow[s]                 # 源出 = 汇入

def checker(times=500, max_n=8, max_cap=10):
    for _ in range(times):
        n = random.randint(4, max_n)
        cap = [[random.randint(0, max_cap) for _ in range(n)] for _ in range(n)]
        cap_a = [row[:] for row in cap]
        cap_b = [row[:] for row in cap]
        f1 = ford_fulkerson_dfs(cap_a, 0, n - 1, n)
        f2 = edmonds_karp(cap_b, 0, n - 1, n)
        if f1 != f2:                                # 两实现结果应一致
            return False
        if not check_invariants(cap, cap_b, 0, n - 1, n):  # 不变量自检
            return False
    return True
```

两个不变量比"数值相等"更可靠：

- **容量约束**：每条边实际流量 ≤ 原容量（`cap_orig - cap_final ≥ 0`）。
- **流量守恒**：除源汇外，每点流入 = 流出。

这两个不变量直接对应"流"的数学定义，比"和另一个实现结果一样"更根本。这是对数器进阶——**用问题的不变量做自检，而不只靠交叉对照**。

### 2. 二分图匹配 = 最大流的特例

把二分图匹配装进最大流：左点连超级源（容量 1），右点连超级汇（容量 1），原边容量 1。最大流值 = 最大匹配数。代码示意：

```python
def bipartite_via_maxflow(adj, n_left, n_right):
    """用最大流求二分图匹配。点编号：0=s, 1..n_left=左, 之后=右, 末=t。"""
    n = n_left + n_right + 2
    s, t = 0, n - 1
    cap = [[0] * n for _ in range(n)]
    for u in range(n_left):
        cap[s][u + 1] = 1                     # s -> 左点，容量 1
        for v in adj[u]:
            cap[u + 1][n_left + 1 + v] = 1    # 左 -> 右，容量 1
    for v in range(n_right):
        cap[n_left + 1 + v][t] = 1            # 右 -> t，容量 1
    return edmonds_karp(cap, s, t, n)
```

但二分图匹配通常还是用匈牙利更省事——匈牙利不需要建超级源汇，代码更短。最大流框架的价值在于"一类问题通用解法"：当问题从"01 匹配"升级到"带权匹配"或"多对多"时，匈牙利要改造成 Kuhn-Munkres（KM 算法）或干脆失效，而最大流框架只要改容量就能继续用。这是"统一框架"的工程价值。

### 3. 面试选读题：外卖配送匹配

**题面**：有 N 个骑手、M 个订单，每个骑手只接某些订单，每个骑手至多接 1 单、每单至多 1 人接。求最多能配送多少单。

这就是二分图最大匹配的直接套用，左骑手、右订单。面试时套匈牙利模板即可。注意这道是"选读"——匹配与流在面试里属低频，但一旦考到往往是压轴。

**讲解要点**（边写边讲）：

1. "这是二分图最大匹配。骑手是左点，订单是右点，能接的边连上，求最大匹配。"
2. "我用匈牙利算法，对每个骑手找增广路，找得到匹配数 +1。复杂度 $O(V \cdot E)$。"
3. 追问"骑手有等级、订单有难度，要让等级 ≥ 难度的匹配最大怎么办"——升级为带权匹配，套 KM 算法，或建带权二分图跑最大流。

### 4. 教科书做法 vs 生产做法

| 场景 | 教科书 | 生产 | 原因 |
|---|---|---|---|
| 二分图匹配 | 手写匈牙利 | 调库（如 scipy） | 工程少手写 |
| 中等规模最大流 | Edmonds-Karp | Dinic 或 Push-Relabel | Dinic 实测更快 |
| 超大规模流 | Dinic | 专用求解器或线性规划 | 通用算法可能不够 |

生产里直接手写网络流的场景不多，多见于：竞赛、图论库内核、运筹优化系统。但理解最大流-最小割对后续学线性规划、博弈论至关重要。

### 5. 两个真实工程坑

**坑一：忘了建反向边，结果不是最大流。**

```python
# 错误：只减正向容量，不加反向
v = t
while v != s:
    u = parent[v]
    cap[u][v] -= bottleneck
    # cap[v][u] += bottleneck   ← 漏了这行
    v = u
```

症状：结果偏小，且随数据变化不稳定。根因：没反向边就无法"反悔"，算法卡在局部最优。修复：必须加 `cap[v][u] += bottleneck`。

**坑二：用浮点容量跑 Ford-Fulkerson（DFS），不收敛。**

症状：算法跑不完或结果错误。根因：Ford-Fulkerson（DFS 版）在无理/浮点容量下不收敛，是教科书反例（ capacities 取无理数时增广路可无限细分）。修复：①用 Edmonds-Karp（BFS）保证 $O(VE^2)$ 终止 ②工程上容量一律用整数。

即便容量是整数，DFS 版也有"流值敏感"的坑。经典反例：s→a、s→b 容量都是 M，a→b、a→t、b→t 容量都是 M。DFS 若交替选 s→a→b→t 和 s→b→a→t 两条增广路，每次只压 1 单位，要跑 $2M$ 次才到最大流 $2M$。M 取 $10^9$ 时直接超时。根因：DFS 不保证最短增广路，瓶颈边被反复选中。修复：换 Edmonds-Karp（BFS 找最短增广路），复杂度脱离流值依赖，$O(VE^2)$ 收尾。

## 四、速查与自测

### 速查表：匹配与流算法对比

| 算法 | 解决问题 | 最坏复杂度 | 关键机制 |
|---|---|---|---|
| 匈牙利（Kuhn） | 二分图最大匹配 | $O(V \cdot E)$ | 交替路找增广 |
| Ford-Fulkerson | 最大流（框架） | $O(E \cdot \|f^*\|)$ | DFS 找增广路 |
| Edmonds-Karp | 最大流 | $O(V \cdot E^2)$ | BFS 找最短增广路 |
| Dinic | 最大流 | $O(V^2 E)$ | 分层图 + 阻塞流 + 当前弧 |

**核心定理**：最大流 = 最小割。最大匹配（二分图）= 最小点覆盖（König 定理）。

**反向边口诀**：正向减容量、反向加容量——允许"反悔"是网络流能到全局最优的关键机制。

### 自测三问

**问题一：** 为什么最大流算法需要反向边？去掉会怎样？

**参考答案：** 反向边记录"已压流量可被退回"的能力。没有反向边，算法一旦在某条增广路上压了流量，就无法纠错——后续若发现更优路径需要"撤回"之前的流量，没有机制实现。结果会卡在局部最优，得到的流小于最大流。反向边让每条增广路都能"反悔"前次决策，最终保证全局最优。这是网络流的本质机制。

**问题二：** Edmonds-Karp 为什么用 BFS 而不是 DFS 找增广路？复杂度差在哪？

**参考答案：** BFS 保证找到的是"边数最少"的增广路。这带来一个关键性质：每条边成为"瓶颈边"的次数有 $O(V)$ 上界，从而增广路总条数 $O(VE)$。配合每次 BFS 的 $O(E)$，总复杂度 $O(VE^2)$，与最大流值无关。DFS 版 Ford-Fulkerson 找增广路没这个保证，复杂度依赖最大流值 $|f^*|$，大容量或浮点容量时不收敛或极慢。

**问题三：** 请用"最大流 = 最小割"解释为什么二分图最大匹配 = 最小点覆盖（König 定理）。

**参考答案：** 二分图匹配装进最大流框架后，最大匹配数 = 最大流值。由对偶定理，最大流值 = 最小割容量。在二分图的 01 容量流网络里，最小割的边都连着源或汇，每个割边对应一个顶点（左点连源割边对应左点，右点连汇割边对应右点），这些顶点构成一个点覆盖，且割容量 = 点覆盖大小。所以最小割 = 最小点覆盖。串起来：最大匹配 = 最大流 = 最小割 = 最小点覆盖。这是对偶定理在二分图上的精彩回响。

### 算法思想 × 生活迁移

匹配与流的核心思想是"**用可回退的尝试逼近全局最优**"——增广路允许翻转已做决定，反向边允许退回已压流量。

**迁移一：项目排期允许返工。** 排一个迭代任务到人，第一版分配可能不是最优。允许"某个任务把已分配的人顶掉，被顶的人重排到别的任务"，正是增广路翻转的思路。强制"一次定终身"的排期，相当于没有反向边的网络流，容易卡在次优解。

**迁移二：交通调度的"反向边"。** 城市路网做流量分配时，某条路堵了，导航会建议绕路甚至"原路返回走另一条"。反向边在物理上对应"允许车辆掉头改走别的路"。没有这个能力，调度系统会强行把车塞进已堵的路。

**迁移三：对偶思维的迁移。** "最大流=最小割"的对偶性在生活中常见：要最大化某个目标，往往等价于最小化某个"瓶颈"。想提升团队产出（最大流），别只盯着"加人"，而要找"最窄的那道关口"（最小割）——可能是审批流程、可能是某个单点依赖。最大化问题的答案，常藏在最小化瓶颈的对偶视角里。

**为什么这些迁移成立：** 网络流成立的前提是"问题有可流通的资源和可识别的瓶颈"。排期有任务流、交通有车流、产出有工作流——都是"流"。只要能建模成流，就能用"找增广路 + 反向纠错 + 最小割瓶颈"这套思路。若问题没有"流"的结构（如纯创造性的灵感问题），这套思路就不适用。
