# 算法课·25｜并查集：用代表元管理集合的极简设计

> 前置知识：学完第 13 章（树的基础）和第 14 章（图的基础），尤其要懂"连通分量"和"无向图判环"
> 学完你能：①给纯小白讲清为什么动态连通性能近 $O(1)$ 维护 ②默写并查集三件套并讲透每一行 ③手撕并查集结构应对面试 ④理解 Ackermann 反函数 $\alpha(n)$ 为什么是"几乎常数" ⑤用对数器验证自己写的并查集对不对

## 一、问题与思路：动态连通性的痛点与并查集的破局思路

### 1. 一个生活场景：朋友圈里"我们是不是一伙的"

假设你在管一个 100 万人的社交平台，每秒钟都有人发起两类操作：①"我和他现在是朋友了"（连通操作）②"我和他是不是同一个朋友圈的？"（查询操作）。朋友关系是会传递的——A 认识 B，B 认识 C，那 A 和 C 就算同一个朋友圈。

如果用最朴素的做法——每来一次查询，就跑一次 BFS/DFS 看两人连不连通——每次查询 $O(n+m)$，100 万人、几千万关系，每秒几千次查询直接把数据库打挂。**反直觉的地方在于：明明每秒都在动态加边，查询却能"几乎 $O(1)$"完成**。这就是并查集（Union-Find / Disjoint Set Union, DSU）要解决的问题。

**核心论点**：并查集 = 找代表元 `find` + 按秩合并 `union` + 路径压缩，三件套缺一不可，合起来能把动态连通性的均摊复杂度压到 $O(\alpha(n))$，其中 $\alpha$ 是 Ackermann 反函数，对任何工程量级 $n \le 10^{80}$ 都不超过 4，**实际就等于常数**。

### 2. 朴素做法的痛点

**痛点一：每次查询都重新遍历整张图。** BFS/DFS 判连通性是 $O(n+m)$，对动态加边场景，每次查询都重跑一遍，总成本 $O(q(n+m))$（q 是操作总数），扛不住。

**痛点二：维护"集合"没有抓手。** 我们关心的是"两人是否在同一集合"，不关心具体路径。如果每次都把整张图遍历一遍，是"为了回答一个集合问题去解一个图问题"——大炮打蚊子。

**核心思想**：并查集的破局在于——**只维护"每个元素属于哪个集合"，不维护集合内部的结构**。具体做法是给每个集合选一个"代表元"（root），两个元素在同一集合当且仅当它们的代表元相同。这样"判连通"就退化成"比较两个代表元是否相等"，$O(1)$ 搞定。

### 3. 历史地位：从 Kruskal 算法到主流数据结构

并查集最早可追溯到 Galler & Fisher 1964 年的论文，但真正让它声名大噪的是 Kruskal 1956 年求最小生成树（MST）的算法——Kruskal 需要按边权排序后逐条加边，加边前判环，并查集是判环的最优工具。CLRS（算法导论）第 21 章用整章篇幅讲它，Tarjan 1975 年给出了带路径压缩的均摊复杂度严格证明。今天它是 LeetCode 高频题（朋友圈、冗余连接、岛屿数量 II）和工程系统（Kruskal、连通分量、最小生成树）的标配。

## 二、原理与实现：从代码到均摊复杂度

### 1. 最朴素的并查集（只有 find 和 union，无优化）

先看最简单的版本，理解机制：

```python
class UnionFind:
    def __init__(self, n):
        self.parent = list(range(n))     # 初始每个元素自成一派

    def find(self, x):
        """找 x 的代表元（根）。"""
        while self.parent[x] != x:       # 不是根就往上走
            x = self.parent[x]
        return x

    def union(self, x, y):
        """合并 x 和 y 所在的两个集合。"""
        rx, ry = self.find(x), self.find(y)
        if rx != ry:
            self.parent[rx] = ry         # 把 rx 挂到 ry 下
```

**逐行讲透**（每一行都不能含糊）：

- **`self.parent = list(range(n))`**：初始化 parent 数组，`parent[i] = i` 表示每个元素一开始自己是自己的代表元（自成一派）。这是并查集用"森林"表示集合的关键——每个集合是一棵树，根是代表元。
- **`while self.parent[x] != x`**：找代表元就是顺着 parent 指针一路往上，直到某个节点的 parent 是自己——那就是根。注意并查集的 parent 是"指向父亲"，不是"指向孩子"，所以是往上走。
- **`self.parent[rx] = ry`**：合并两个集合，就是把一棵树的根挂到另一棵树的根下面。挂完后两棵树变成一棵，根是 ry，所有原本以 rx 为根的元素的代表元现在变成 ry。

**用一个例子走一遍**。n=5，元素 0-4：

```
初始: parent = [0, 1, 2, 3, 4]   每个元素自成一派

union(0, 1): rx=0, ry=1, parent[0]=1
  parent = [1, 1, 2, 3, 4]   0 和 1 同集合，根是 1

union(2, 3): rx=2, ry=3, parent[2]=3
  parent = [1, 1, 3, 3, 4]   2 和 3 同集合，根是 3

union(0, 2): rx=find(0)=1, ry=find(2)=3, parent[1]=3
  parent = [1, 3, 3, 3, 4]   {0,1,2,3} 同集合，根是 3

find(0): 0 → 1 → 3, 返回 3
find(4): 4, 返回 4   （4 是独立集合）
```

**关键观察**：`find` 的复杂度取决于树的高度。如果一直把"大树挂小树"，树会越挂越深，最坏退化成链表，`find` $O(n)$。这就是为什么朴素版本不够用——必须优化。

### 2. 优化一：按秩合并（union by rank）

**做法**：合并两棵树时，**把矮的挂到高的下面**，这样树的高度不会增长（除非两棵一样高，那时高度才 +1）。

```python
class UnionFind:
    def __init__(self, n):
        self.parent = list(range(n))
        self.rank = [0] * n              # rank[i] 是以 i 为根的树的高度上界

    def find(self, x):
        while self.parent[x] != x:
            x = self.parent[x]
        return x

    def union(self, x, y):
        rx, ry = self.find(x), self.find(y)
        if rx == ry:
            return
        if self.rank[rx] < self.rank[ry]:   # 矮的挂到高的下面
            self.parent[rx] = ry
        elif self.rank[rx] > self.rank[ry]:
            self.parent[ry] = rx
        else:                                # 等高，挂任意一边，被挂的根高度 +1
            self.parent[rx] = ry
            self.rank[ry] += 1
```

**逐行讲透新增部分**：

- **`self.rank = [0] * n`**：rank 数组初始全 0。注意 rank 不是"精确高度"，是"高度的上界"——配合路径压缩后真实高度可能比 rank 小，但 rank 永远不会低于真实高度，所以比较 rank 仍然有效。
- **`if self.rank[rx] < self.rank[ry]: self.parent[rx] = ry`**：rx 这棵树矮，挂到 ry 下面。挂完后 ry 这棵树高度不变（因为 ry 更高）。
- **`else: self.parent[rx] = ry; self.rank[ry] += 1`**：两棵等高，挂任意一边，被挂的根 rank +1。为什么 +1？因为等高时，合并后 ry 这棵树的高度变成原 ry 高度 +1。

**复杂度**：只按秩合并，不带路径压缩，树高 $\le \log_2 n$，每次 `find` 和 `union` 都是 $O(\log n)$。证明：rank 为 k 的树至少包含 $2^k$ 个节点（归纳可证），所以 rank $\le \log_2 n$。

### 3. 优化二：路径压缩（path compression）

按秩合并只把树高压到 $O(\log n)$，还能更狠——**路径压缩**：`find` 时把路径上所有节点直接挂到根，让树变"扁平"。

```python
# 递归版
def find(self, x):
    if self.parent[x] != x:
        self.parent[x] = self.find(self.parent[x])   # 路径上所有节点指向根
    return self.parent[x]

# 非递归版（避免递归栈溢出，大数据量必备）
def find(self, x):
    root = x
    while self.parent[root] != root:        # 第一遍：找根，不修改 parent
        root = self.parent[root]
    while self.parent[x] != root:           # 第二遍：路径上所有节点指向根
        nxt = self.parent[x]
        self.parent[x] = root
        x = nxt
    return root
```

**逐行讲透**：

- **递归版 `self.parent[x] = self.find(self.parent[x])`**：这是"边找边压"的精髓。`find(self.parent[x])` 递归找到根，然后把 x 直接指向根。递归回溯过程中，路径上每个节点都被指向根。一行代码完成了"找根 + 压扁路径"两件事。
- **非递归版两遍循环**：第一遍找根（不修改 parent），第二遍把路径上每个节点指向根。注意 `nxt = self.parent[x]` 必须先存下来——直接 `self.parent[x] = root` 会丢失下一个节点的位置。**大数据量场景必用非递归版**，因为递归深度等于树高，链表状数据会栈溢出。

**为什么路径压缩和按秩合用最优**：单独用路径压缩（不按秩合并）均摊也是 $O(\log n)$；单独按秩合并 $O(\log n)$；**两者合用，均摊 $O(\alpha(n))$**，其中 $\alpha$ 是 Ackermann 反函数——这就是下一节要讲的高手向话题。

### 4. 高手向：Ackermann 反函数与均摊复杂度

Ackermann 函数 $A(m, n)$ 是个增长极快的双变量函数：

$$
A(0, n) = n + 1 \\
A(m, 0) = A(m-1, 1) \\
A(m, n) = A(m-1, A(m, n-1))
$$

它的反函数 $\alpha(n) = \min\{k : A(k, k) \ge n\}$ 增长极慢——慢到什么程度？**对任何可观测宇宙中的 $n \le 10^{80}$，$\alpha(n) \le 4$**。这意味着并查集单次操作的均摊成本，对任何工程量级数据，都不超过 4 次指针操作——**实际就是常数**。

Tarjan 1975 年证明：按秩合并 + 路径压缩，n 个元素、m 次操作的总成本是 $O(m \cdot \alpha(n))$，并且这个上界是紧的——存在让总成本达到 $\Omega(m \cdot \alpha(n))$ 的操作序列。换句话说，并查集的均摊复杂度"已经到顶了"，没法更好。

**对高手的启发**：$\alpha(n)$ 这种"理论非 $O(1)$、实际 $O(1)$"的复杂度，是算法分析里一个特殊现象。它提醒我们——理论复杂度和工程性能之间，有时隔着"宇宙级"的距离。理解这一点，能让你在面试中被问"并查集是不是 $O(1)$"时，给出有深度的回答："理论上是 $O(\alpha(n))$，但 $\alpha(n) \le 4$ 对任何工程量级都成立，实际就是 $O(1)$。"

### 5. 按秩合并 vs 按大小合并

按秩合并维护"高度上界"，按大小合并维护"节点数"——两者都把树控制在 $O(\log n)$ 高度，但实现和适用场景有差别：

| 维度 | 按秩合并（union by rank） | 按大小合并（union by size） |
|---|---|---|
| 维护量 | rank（高度上界） | size（节点数） |
| 合并策略 | 矮的挂高的 | 小的挂大的 |
| 单次合并 | $O(1)$ | $O(1)$ |
| 是否需要路径压缩配合 | 需要（rank 不准） | 需要 |
| 额外收益 | 无 | 能直接查"集合大小" |

**实践建议**：面试常考"按秩合并 + 路径压缩"标准版；但工程里如果需要查"每个集合有多少元素"（如朋友圈人数、连通分量大小），用按大小合并更直接——`size[root]` 就是集合大小，省去额外维护。

### 6. 五指标评价

| 指标 | 并查集（三件套） | 说明 |
|---|---|---|
| 单次 find | $O(\alpha(n))$ 均摊 | $\alpha(n) \le 4$，实际常数 |
| 单次 union | $O(\alpha(n))$ 均摊 | 含两次 find |
| 空间 | $O(n)$ | parent + rank/size |
| 建图 | $O(n)$ | 初始化 parent 数组 |
| 优势 | 近 $O(1)$ 维护动态连通性 | 比 BFS/DFS 每次 $O(n+m)$ 快得多 |
| 局限 | 只支持"合并"和"查询"，不支持"拆分" | 删除边需用可撤销并查集或离线 |

## 三、实践与面试：手撕结构、对数器、面试题

### 1. 面试手撕模板（标准三件套版）

面试中被要求手撕并查集，用"按秩合并 + 路径压缩"标准版——既能展示你知道均摊 $O(\alpha(n))$ 的深度，又是最通用的版本：

```python
class UnionFind:
    """并查集标准三件套：find + 按秩合并 + 路径压缩。"""
    def __init__(self, n):
        self.parent = list(range(n))
        self.rank = [0] * n
        self.count = n                     # 当前连通分量数

    def find(self, x):
        """路径压缩版 find。"""
        if self.parent[x] != x:
            self.parent[x] = self.find(self.parent[x])
        return self.parent[x]

    def union(self, x, y):
        """按秩合并 + 路径压缩。返回是否真的合并了。"""
        rx, ry = self.find(x), self.find(y)
        if rx == ry:
            return False                    # 本来就在同一集合
        if self.rank[rx] < self.rank[ry]:
            rx, ry = ry, rx                  # 保证 rx 的 rank >= ry
        self.parent[ry] = rx                 # ry 挂到 rx 下
        if self.rank[rx] == self.rank[ry]:
            self.rank[rx] += 1
        self.count -= 1                      # 连通分量数 -1
        return True

    def is_connected(self, x, y):
        return self.find(x) == self.find(y)
```

**面试时怎么讲**（按"技术的水有多深"八层面，挑重点讲）：

1. **问题层**：动态连通性问题——边上动态加，查询"两点是否连通"频繁，BFS/DFS 每次 $O(n+m)$ 扛不住。并查集用"代表元"把判连通降成"比较两个根是否相等"。
2. **原理层**：每个集合是一棵树，根是代表元。`find` 顺 parent 指针找根，`union` 把一棵树的根挂到另一棵下面。按秩合并控制树高 $O(\log n)$，路径压缩把路径压扁，两者合用均摊 $O(\alpha(n))$。
3. **优劣层**：近 $O(1)$ 维护动态连通性、代码极简（10 行核心）；但只支持合并和查询，不支持拆分（拆分要用可撤销并查集或离线）。
4. **演进层**：朴素版 $O(n)$ 最坏；按秩合并 $O(\log n)$；按秩合并 + 路径压缩 $O(\alpha(n))$，Tarjan 1975 证明这是紧的上界。

**面试加分点**：主动说"$\alpha(n) \le 4$ 对任何 $n \le 10^{80}$ 成立，实际就是常数"——这能立刻区分你和"只会背 $O(\alpha(n))$"的候选人。

### 2. 对数器：随机连通操作对比朴素实现

左神反复强调：**写完数据结构不要靠肉眼检查，要用对数器验证**。并查集的对数器思路是——生成随机操作序列，用你的并查集和"绝对正确的朴素实现"（每次 BFS 判连通）对比，跑足够多次，只要有一次不一致就说明你的并查集有 bug。

```python
import random
from collections import defaultdict, deque

class UnionFind:
    def __init__(self, n):
        self.parent = list(range(n))
        self.rank = [0] * n
    def find(self, x):
        if self.parent[x] != x:
            self.parent[x] = self.find(self.parent[x])
        return self.parent[x]
    def union(self, x, y):
        rx, ry = self.find(x), self.find(y)
        if rx == ry:
            return
        if self.rank[rx] < self.rank[ry]:
            rx, ry = ry, rx
        self.parent[ry] = rx
        if self.rank[rx] == self.rank[ry]:
            self.rank[rx] += 1

def naive_connected(adj, n, u, v):
    """朴素 BFS 判连通，绝对正确。"""
    if u == v:
        return True
    visited = [False] * n
    q = deque([u])
    visited[u] = True
    while q:
        x = q.popleft()
        for nb in adj[x]:
            if nb == v:
                return True
            if not visited[nb]:
                visited[nb] = True
                q.append(nb)
    return False

def checker(test_times=500, max_n=20, max_ops=50):
    """对数器：跑 test_times 次，每次随机操作序列对比。"""
    for _ in range(test_times):
        n = random.randint(1, max_n)
        uf = UnionFind(n)
        adj = defaultdict(list)             # 朴素实现用邻接表
        for _ in range(max_ops):
            op = random.choice(['union', 'query'])
            u, v = random.randint(0, n-1), random.randint(0, n-1)
            if op == 'union':
                uf.union(u, v)
                adj[u].append(v); adj[v].append(u)
            else:
                uf_result = (uf.find(u) == uf.find(v))
                naive_result = naive_connected(adj, n, u, v)
                if uf_result != naive_result:
                    print(f"出错！n={n}, u={u}, v={v}, UF={uf_result}, naive={naive_result}")
                    return False
    print("对数器验证通过！")
    return True

checker()
```

**对数器的价值**：

- **覆盖边界**：随机操作序列覆盖"先 union 再 query""反复 union 同一对""自环 union(u,u)"等容易遗漏的边界
- **不依赖人工**：跑 500 次只要几秒，比手动构造测试用例全面得多
- **左神原话**："对数器是算法工程师的基本功。写完一个数据结构，第一件事就是写对数器，而不是跑几个例子看看对不对。"

**对高手的启发**：并查集的 bug 往往出在"边界"——union 自己（u==v）、union 已连通的两点、find 时路径压缩写错把根改了。对数器能逼出这些边界 bug，比肉眼检查可靠得多。

### 3. 应用一：Kruskal 求最小生成树（MST）时判环

Kruskal 算法求无向连通图的最小生成树，思路是按边权排序后逐条加边，加边前判环——如果这条边的两端已经在同一连通分量，加入会成环，跳过。判环正是并查集的拿手好戏。

```python
def kruskal(n, edges):
    """edges: [(w, u, v), ...]，返回 MST 总权值。"""
    edges.sort()                            # 按边权升序
    uf = UnionFind(n)
    mst_weight = 0
    edge_count = 0
    for w, u, v in edges:
        if uf.find(u) != uf.find(v):        # 不在同一集合，加入不成环
            uf.union(u, v)
            mst_weight += w
            edge_count += 1
            if edge_count == n - 1:         # MST 有 n-1 条边，提前结束
                break
    return mst_weight
```

**关键观察**：`find(u) != find(v)` 就是判环——u 和 v 已连通意味着图中已有一条 u-v 路径，再加边 (u,v) 就成环。并查集把这个判环操作压到 $O(\alpha(n))$，让 Kruskal 总复杂度由"排序 $O(m \log m)$"主导，而不是"判环 $O(mn)$"主导。

### 4. 应用二：面试高频三题

**题一：朋友圈 / 省份数量（LeetCode 547）**

n×n 矩阵 `isConnected`，`isConnected[i][j]=1` 表示城市 i 和 j 直接相连，求省份总数（连通分量数）。

```python
def findCircleNum(isConnected):
    n = len(isConnected)
    uf = UnionFind(n)
    for i in range(n):
        for j in range(i+1, n):             # 只走上三角，避免重复
            if isConnected[i][j]:
                uf.union(i, j)
    return uf.count                         # 连通分量数 = 省份数
```

**讲解要点**：并查集的 `count` 字段直接给出连通分量数，不用扫一遍 parent 数组（扫还要去重，因为非根节点的 parent 不是自己）。维护 `count` 是面试加分点——每次成功 union 时 `count -= 1`，最后 `count` 就是连通分量数。

**题二：冗余连接（LeetCode 684）**

无向图有 n 个节点和 n 条边（比树多一条），找出一条可删除的边，使图变成树。返回最后出现的那条。

```python
def findRedundantConnection(edges):
    n = len(edges)
    uf = UnionFind(n + 1)                   # 节点编号 1..n
    for u, v in edges:
        if uf.find(u) == uf.find(v):        # 已连通，加这条边成环
            return [u, v]
        uf.union(u, v)
    return []
```

**讲解要点**：并查集判环的典型应用——`find(u) == find(v)` 说明 u-v 已连通，加边 (u,v) 成环。题目要求"最后出现的那条"，所以遍历到第一条成环的边就返回（因为按顺序遍历，第一条成环的就是最后出现的）。

**题三：岛屿数量 II（LeetCode 305）**

动态加陆地：每次操作把一个水域位置变陆地，要求每次操作后返回当前岛屿数。BFS/DFS 每次重算 $O(kmn)$ 扛不住，并查集增量维护 $O(\alpha)$。

```python
def numIslands2(m, n, positions):
    uf = UnionFind(m * n)
    grid = [0] * (m * n)
    res = []
    count = 0
    dirs = [(-1,0),(1,0),(0,-1),(0,1)]
    for r, c in positions:
        idx = r * n + c
        if grid[idx]:                       # 已经是陆地，跳过
            res.append(count)
            continue
        grid[idx] = 1
        count += 1                          # 先 +1，再尝试和邻居合并 -1
        for dr, dc in dirs:
            nr, nc = r + dr, c + dc
            if 0 <= nr < m and 0 <= nc < n:
                nidx = nr * n + nc
                if grid[nidx] and uf.union(idx, nidx):
                    count -= 1              # 成功合并，岛屿数 -1
        res.append(count)
    return res
```

**讲解要点**：每次加陆地先 `count += 1`，然后尝试和 4 个邻居合并，每合并成功一次 `count -= 1`。这是"增量维护连通分量数"的标准套路——比每次重算快几个数量级。

### 5. 教科书做法 vs 生产做法

| 场景 | 教科书做法 | 生产做法 | 原因 |
|---|---|---|---|
| 判连通 / 连通分量 | 手写并查集 | 手写并查集 | 并查集就是工业最优解，没有替代 |
| 最小生成树 | Kruskal + 并查集 | 大数据用 Prim + 斐波那契堆 | Kruskal 排序主导 $O(m \log m)$，Prim 在稠密图更优 |
| 离线 LCA | Tarjan 算法 + 并查集 | 倍增 / 树链剖分 | 在线查询倍增更灵活 |
| 动态连通性（带删除） | 可撤销并查集（离线） | Link-Cut Tree（在线） | LCT 支持在线删边，但实现复杂 |

**生产里并查集远比希尔排序常见**——Kruskal、连通分量、网络连通性、图判环、最小生成树，都是并查集的主场。它是少数"教科书做法 = 生产做法"的数据结构。

### 6. 三个真实工程坑

**坑一：find 不带路径压缩，链表状数据 $O(n)$ 退化。**

```python
def find_bad(self, x):
    while self.parent[x] != x:              # 不压缩路径
        x = self.parent[x]
    return x
```

症状：按顺序 `union(0,1), union(1,2), ..., union(n-2, n-1)` 后，树退化成链表，`find(0)` 要走 n 步。根因：每次 union 把前一个挂到后一个下面，形成长链。修复：加路径压缩（递归或非递归），把路径压扁。

**坑二：按秩合并写反，把高的挂到矮的下面。**

```python
def union_bad(self, x, y):
    rx, ry = self.find(x), self.find(y)
    if rx != ry:
        self.parent[rx] = ry                # 不比较 rank 直接挂
```

症状：树高不受控，最坏退化成链表。根因：没有按秩合并，可能反复把大树挂小树。修复：按秩合并或按大小合并，保证树高 $O(\log n)$。

**坑三：递归 find 在大数据量上栈溢出。**

```python
def find_recursive(self, x):
    if self.parent[x] != x:
        self.parent[x] = self.find_recursive(self.parent[x])   # 递归
    return self.parent[x]
```

症状：1000 万元素 + 链表状数据，递归深度 1000 万，Python 默认递归上限 1000 直接栈溢出。根因：路径压缩前树可能很深，递归 find 深度等于树高。修复：大数据量用非递归版 find（两遍循环，第一遍找根，第二遍压扁路径）。

## 四、速查与自测

### 速查表：并查集三件套与复杂度

| 操作 | 朴素 | 按秩合并 | 三件套（按秩+路径压缩） |
|---|---|---|---|
| find | $O(n)$ 最坏 | $O(\log n)$ | $O(\alpha(n))$ 均摊 |
| union | $O(n)$ 最坏 | $O(\log n)$ | $O(\alpha(n))$ 均摊 |
| 空间 | $O(n)$ | $O(n)$ | $O(n)$ |
| 实现难度 | 极简 | 简单 | 简单 |

**$\alpha(n)$ 关键事实**：$\alpha(n) \le 4$ 对所有 $n \le 10^{80}$ 成立，实际就是常数。

**按秩合并 vs 按大小合并**：

| 维度 | 按秩合并 | 按大小合并 |
|---|---|---|
| 维护量 | 高度上界 | 节点数 |
| 适用 | 通用 | 需查集合大小 |
| 单独复杂度 | $O(\log n)$ | $O(\log n)$ |
| 配合路径压缩 | $O(\alpha(n))$ | $O(\alpha(n))$ |

### 自测三问

**问题一：** 为什么按秩合并 + 路径压缩能把单次操作压到 $O(\alpha(n))$？单独用其中一个为什么不够？

**参考答案：** 单独按秩合并控制树高 $O(\log n)$，但每次 find 仍要爬 $\log n$ 步，单次 $O(\log n)$。单独路径压缩能让树变扁，但不按秩合并时下次 union 可能把大树挂小树，重新变深，均摊仍是 $O(\log n)$。两者合用才能达到 $O(\alpha(n))$——按秩合并保证树"不长高"，路径压缩保证树"变扁平"，Tarjan 1975 证明两者协同的均摊成本是 $O(\alpha(n))$ 且是紧上界。**核心思想是按秩合并控高、路径压缩压扁，缺一个都退回 $O(\log n)$**。

**问题二：** 路径压缩的递归版和非递归版有什么区别？什么场景必须用非递归版？

**参考答案：** 递归版 `self.parent[x] = self.find(self.parent[x])` 一行完成"找根 + 压扁路径"，简洁但递归深度等于树高。非递归版两遍循环：第一遍找根不修改 parent，第二遍把路径上每个节点指向根，需要临时变量 `nxt` 存下一个节点。**大数据量场景必须用非递归版**——1000 万元素 + 链表状数据，递归深度 1000 万，Python 默认递归上限 1000 会栈溢出。非递归版没有递归深度限制，适合任何规模。这是工程和面试的差异——面试写递归版够用，工程必须非递归。

**问题三：** Kruskal 算法为什么要用并查集？用 BFS/DFS 判环行不行？

**参考答案：** Kruskal 按边权排序后逐条加边，加边前判环——如果边 (u,v) 的两端已连通，加入成环，跳过。用 BFS/DFS 判环每次 $O(n+m)$，m 条边总共 $O(m(n+m))$，对稠密图 $m=O(n^2)$ 退化到 $O(n^4)$，扛不住。用并查集判环，`find(u) == find(v)` 即已连通，单次 $O(\alpha(n))$，m 条边总共 $O(m \alpha(n))$，加上排序 $O(m \log m)$，总复杂度由排序主导 $O(m \log m)$。并查集把"判环"从 $O(n+m)$ 压到 $O(\alpha(n))$，是 Kruskal 高效的关键。

### 算法思想 × 生活迁移

并查集的精髓是"**用代表元抽象集合，用扁平结构保证效率**"——不维护集合内部结构，只维护"谁属于哪个代表元"。

**迁移一：组织架构的"汇报线"管理。** 大公司几万人，HR 想知道"员工 A 和员工 B 是不是同一个事业部"。不必维护完整的部门树（每次重组都要改），只需维护"每个人的最终汇报对象是谁"（代表元）——这就是并查集的 parent 数组。部门合并就是 union，查询同部门就是 find 比较。路径压缩对应"扁平化管理"——把多层汇报线压成直接向事业部负责人汇报。

**迁移二：版本控制系统的分支合并。** Git 的分支合并图本质是"哪些 commit 属于同一连通分量"。每次 merge 就是 union，查询"两个 commit 是否同源"就是 find。Git 用类似并查集的结构快速判断分支关系，而不是每次遍历整个 commit 图。

**迁移三：网络路由的连通性判断。** 分布式系统里判断"节点 A 能不能通过现有链路到达节点 B"，链路动态增减。并查集能 $O(\alpha)$ 判连通，比每次跑 Dijkstra $O(m \log n)$ 快得多。注意并查集只能判"是否连通"，不能判"最短路径"——这是它的能力边界。

**为什么这些迁移成立：** 并查集成立的前提是"只关心归属关系，不关心内部结构"。组织架构、版本分支、网络连通性，都满足这个前提——我们只问"是不是一伙的"，不问"具体怎么连的"。如果问题需要"最短路径"或"具体路径"，并查集就不适用，得用 BFS/Dijkstra。
