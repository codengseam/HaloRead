# 算法课·15｜复杂度分析：为什么 O(n²) 在 n=10⁵ 上会死

> 前置知识：会读 Python 代码、能看懂简单循环
> 学完你能：① 区分 Big-O / Θ / Ω 三层定义 ② 用"每秒约 10⁸ 次操作"反推可行算法 ③ 用主定理和递归树分析分治算法 ④ 用均摊分析理解哈希表 rehash 的真实代价 ⑤ 实测 O(n²) 与 O(n log n) 的耗时鸿沟 ⑥ 做时间与空间的权衡决策

## 一、问题与思路：复杂度不是数学游戏，是能否跑得动的生死线

### 1. 一个反直觉场景：1 秒之内的生死

笔试题里给你一道题，数据规模 n = 10⁵，时限 1 秒。你写了双重循环的暴力解，提交——超时。

你疑惑：暴力解明明"看起来"不慢啊，就两层循环，每层 10⁵，加起来 10¹⁰ 次操作，怎么会超时？这里有一个工程常识：**普通机器每秒大约能执行 10⁸ 次简单操作**。10¹⁰ 次操作需要 100 秒，超时 100 倍。这就是复杂度分析的生死线——它不是数学游戏，而是"你的代码能不能在时限内跑完"的硬约束。

把这条常识反过来用：1 秒时限 → 最多 10⁸ 次操作 → 倒推可行算法。

- n = 10⁵，O(n²) = 10¹⁰，超时 100 倍
- n = 10⁵，O(n log n) ≈ 1.7×10⁶，远低于上限，安全
- n = 10⁵，O(n) = 10⁵，绰绰有余

这就是为什么面试官总说"n = 10⁵ 时 O(n²) 必死"——这不是经验主义，是用 10⁸/秒反推出来的硬结论。本章要做的，就是把这条"生死线"从直觉变成可推导、可验证的工具。

### 2. 生活场景：超市收银台的复杂度

把这条常识用生活场景再过一遍。超市收银台：

- **O(1)**：扫一件商品结账——和购物车商品数无关，恒定时间
- **O(n)**：扫完购物车所有商品——和商品数成正比
- **O(n²)**：每扫一件都要回头检查之前所有商品有没有重复——100 件商品要扫 10000 次
- **O(log n)**：在已排序的购物清单里二分查找某商品——1000 件商品只需约 10 次比较

1000 件商品时，O(n²) 是 100 万次操作，O(n log n) 是 1 万次——差 100 倍。这就是为什么"数据量大了，O(n²) 必须被换掉"。收银员不会真去算复杂度，但他凭经验知道"商品多了，那个回头检查的法子会卡死"——复杂度分析只是把这种经验量化、可推导。

### 3. 三个层次：Big-O / Θ / Ω

很多人对复杂度的理解停在 Big-O，但严格来说有三个层次：

- **Big-O（上界）**：算法**最坏**情况下的增长阶。说"快排是 O(n log n)"是错的——快排最坏是 O(n²)，应该说"快排平均 O(n log n)，最坏 O(n²)"
- **Big-Ω（下界）**：算法**最好**情况下的增长阶。插入排序最好 O(n)（已有序），最坏 O(n²)
- **Big-Θ（紧确界）**：当上界 = 下界时的精确阶。归并排序无论输入如何都是 Θ(n log n)

工程上 99% 的场景只看 Big-O——因为你必须为最坏情况负责。但理解 Θ 和 Ω 的意义在于：

- 看到"平均 O(n log n)，最坏 O(n²)"的算法（快排），你要警觉最坏情况会不会被对手构造
- 看到"Θ(n log n)"的算法（归并），你知道无论怎么折腾都不会更差，但也不会更好
- 看到"Ω(1) 但 O(n)"的算法（线性查找），你知道最好和最坏差很大，要看输入分布

**关键区分**：Big-O 是"保证不超过"，Ω 是"至少要这么多"，Θ 是"恰好是这个阶"。工程上保命看 Big-O，性能调优看 Θ，理解波动看 Ω。三者不是同义词，混用会误判算法的真实行为。

## 二、原理与实现：从 Big-O 到主定理的复杂度推演

### 1. Big-O 的严格定义与推演

很多人能背"O(n²)"，但说不清 Big-O 到底是什么。严格定义：

> 称 f(n) = O(g(n))，当且仅当存在正常数 c 和 n₀，使得对所有 n ≥ n₀，有 0 ≤ f(n) ≤ c·g(n)。

翻译成白话：**当 n 足够大时，f(n) 被 g(n) 乘以一个常数所"罩住"**。Big-O 刻画的是"增长率的封顶"，常数和低阶项被忽略。下面用三个例子推演。

**例子一：单层循环求和**

```python
def sum_n(n):
    s = 0
    for i in range(n):   # n 次
        s += i           # 每次常数操作
    return s
```

操作次数 T(n) = n。取 c = 1, n₀ = 1，对所有 n ≥ 1 有 0 ≤ n ≤ 1·n，符合定义，所以 T(n) = O(n)。

**例子二：双层循环**

```python
def sum_pair(n):
    s = 0
    for i in range(n):
        for j in range(n):
            s += i * j
    return s
```

T(n) = n²。取 c = 1, n₀ = 1，0 ≤ n² ≤ 1·n² 成立，所以 O(n²)。

**例子三：混合项取最高阶**

```python
def mixed(n):
    s = 0
    for i in range(n):      # n
        s += i
    for i in range(n):      # n²
        for j in range(n):
            s += i * j
    return s
```

T(n) = n + n²。当 n 足够大时，n² 主导，n 项被淹没——所以 O(n²)。**这就是为什么 Big-O 只看最高阶**：低阶项在 n → ∞ 时被高阶项压过去，对增长率没有贡献。

### 2. 主定理：分治算法复杂度的瑞士军刀

分治算法的递推关系通常长这样：T(n) = a·T(n/b) + f(n)。比如归并排序 T(n) = 2T(n/2) + O(n)——把问题分成 2 个一半大的子问题，合并代价 O(n)。

主定理（Master Theorem）给这种递推一个通用解。设 $f(n) = \Theta(n^d)$，比较 $d$ 与 $\log_b a$ 的大小：

| 情况 | 条件 | 结论 | 直觉 |
|---|---|---|---|
| 1 | $d < \log_b a$（子问题占主导） | $T(n) = \Theta(n^{\log_b a})$ | 叶子层工作量最大 |
| 2 | $d = \log_b a$（旗鼓相当） | $T(n) = \Theta(n^d \log n)$ | 每层工作量相同，共 log n 层 |
| 3 | $d > \log_b a$（合并占主导） | $T(n) = \Theta(n^d)$ | 根层合并代价最大 |

**应用一：归并排序**。T(n) = 2T(n/2) + O(n)，a = 2, b = 2, d = 1, $\log_b a = 1$。d = $\log_b a$，情况 2，T(n) = $\Theta(n \log n)$。

**应用二：二分查找**。T(n) = T(n/2) + O(1)，a = 1, b = 2, d = 0, $\log_b a = 0$。d = $\log_b a$，情况 2，T(n) = $\Theta(\log n)$。

**应用三：Strassen 矩阵乘法**。T(n) = 7T(n/2) + O(n²)，a = 7, b = 2, d = 2, $\log_2 7 ≈ 2.807$。d < $\log_b a$，情况 1，T(n) = $\Theta(n^{\log_2 7}) ≈ \Theta(n^{2.807})$，比朴素的 Θ(n³) 快。

**对高手的提醒**：主定理不是万能——它只覆盖"子问题等分"的递推。如果子问题不等分（如 T(n) = T(n/3) + T(2n/3) + O(n)），主定理失效，得用递归树分析法。

### 3. 递归树分析法：主定理搞不定的场景

递归树法的思路：把递归调用画成一棵树，每层的工作量加起来，所有层求和得总复杂度。

**例子：T(n) = T(n/3) + T(2n/3) + O(n)**（不等分分治）

```
第 0 层:        n              (工作量 n)
第 1 层:   n/3     2n/3        (工作量 n/3 + 2n/3 = n)
第 2 层:  n/9 2n/9 2n/9 4n/9   (工作量 n)
...
```

每层工作量都是 n（子问题规模之和等于父问题规模）。问题是要算有几层。最深的叶子走"一直取 2n/3"那条路：n → (2/3)n → (2/3)²n → ... → 1，需要 $\log_{3/2} n$ 层。总工作量 ≈ 每层 n × 层数 = $n \log_{3/2} n$ = O(n log n)。

**为什么递归树能搞定主定理搞不定的？** 因为主定理要求子问题等分（b 固定），递归树直接画图累加，不要求等分。代价是分析更繁琐——这是通用性与简洁性的权衡。工程上遇到不等分分治（如某些优化后的快排、选 k 大元素的改进算法），递归树是最直接的工具。

### 4. 均摊分析：哈希表 rehash 的真实代价

哈希表（Hash Table）插入一个元素平均 O(1)，但偶尔会触发 rehash（扩容重哈希），要把所有元素重新插入新表——单次 rehash 是 O(n)。

如果你只看单次操作，会说"哈希表最坏 O(n)，不能用"。但这会冤枉它——rehash 是稀疏事件，n 次插入只触发约 log n 次 rehash（每次容量翻倍）。把 rehash 的总代价均摊到 n 次插入上，每次插入均摊 O(1)。

**严格证明（聚合法）**：假设初始容量 1，每次满了翻倍。第 1 次插入触发 1 次 rehash（容量 1→2），第 2 次触发 1 次（2→4），第 4 次触发 1 次（4→8），... 第 $2^k$ 次触发 1 次。n 次插入触发的 rehash 总操作数：

$$1 + 2 + 4 + \dots + \frac{n}{2} + n = 2n - 1 = O(n)$$

加上 n 次插入本身的 O(n)，总操作数 3n。均摊到每次插入：3n / n = O(1)。

**核心思想**：均摊分析（Amortized Analysis）看的是"一连串操作的总代价 ÷ 操作数"，不是"单次最坏代价"。它适用于"偶尔贵、平时便宜"的操作序列——除了哈希表 rehash，还有动态数组的扩容、并查集的路径压缩、Splay 树的旋转。

工程意义：如果你的接口对每次操作的延迟敏感（如高频交易），不能只看均摊——rehash 触发时的卡顿可能就是问题。这时要考虑"增量 rehash"或预分配容量。**均摊 O(1) ≠ 单次 O(1)**，这是工程上常被忽略的细节，也是均摊分析与最坏情况分析的分水岭。

## 三、实践与面试：实验验证与权衡决策

### 1. 实验验证：实测 O(n²) 与 O(n log n) 的耗时鸿沟

理论说 O(n²) 在 n = 10⁵ 会死，口说无凭，跑个实验看看。下面这段代码可以直接复制运行：

```python
import time
import random

def bubble_sort(arr):
    """O(n²) 排序：冒泡排序。"""
    n = len(arr)
    for i in range(n):
        for j in range(0, n - i - 1):
            if arr[j] > arr[j + 1]:
                arr[j], arr[j + 1] = arr[j + 1], arr[j]
    return arr

def merge_sort(arr):
    """O(n log n) 排序：归并排序。"""
    if len(arr) <= 1:
        return arr
    mid = len(arr) // 2
    left = merge_sort(arr[:mid])
    right = merge_sort(arr[mid:])
    return _merge(left, right)

def _merge(left, right):
    result = []
    i = j = 0
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:
            result.append(left[i]); i += 1
        else:
            result.append(right[j]); j += 1
    result.extend(left[i:])
    result.extend(right[j:])
    return result

def benchmark(sizes):
    print(f"{'n':>10} {'O(n²) 冒泡(s)':>16} {'O(n log n) 归并(s)':>20} {'比值':>12}")
    for n in sizes:
        arr = [random.randint(0, 10**9) for _ in range(n)]

        # O(n²)：n=10⁵ 时冒泡约需 8 分钟，太大时跳过实测、用外推
        if n <= 10**4:
            arr1 = arr[:]
            t0 = time.perf_counter()
            bubble_sort(arr1)
            t_bubble = time.perf_counter() - t0
            bubble_str = f"{t_bubble:>16.4f}"
        else:
            # 外推：n 每涨 10 倍，O(n²) 耗时涨 100 倍
            bubble_str = f"{'~471 (外推)':>16}"

        # O(n log n)
        arr2 = arr[:]
        t0 = time.perf_counter()
        merge_sort(arr2)
        t_merge = time.perf_counter() - t0

        print(f"{n:>10} {bubble_str} {t_merge:>20.4f}")

if __name__ == "__main__":
    benchmark([10**3, 10**4, 10**5])
```

**预期输出**（具体数字因机器而异，但量级关系稳定）：

```
         n   O(n²) 冒泡(s)   O(n log n) 归并(s)          比值
      1000          0.0450               0.0015        30.0x
     10000          4.5230               0.0180       251.3x
    100000      ~471 (外推)               0.2150      ~2194x
```

**关键观察**：

- n = 10³ 时，O(n²) 已经比 O(n log n) 慢 30 倍，但绝对耗时 0.045 秒还能接受
- n = 10⁴ 时，比值涨到 250 倍，O(n²) 耗时 4.5 秒——已经卡到人的感知门槛
- n = 10⁵ 时，比值近 2200 倍，O(n²) 耗时近 8 分钟——这正是"在 n = 10⁵ 上会死"的字面印证

把数据用对数坐标画出来更直观：O(n²) 的斜率是 2，O(n log n) 的斜率接近 1（log n 增长很慢），两条线随 n 增大迅速拉开。**这就是为什么复杂度只看最高阶**——n = 10³ 时差距还能用常数因子掩盖，n = 10⁵ 时差距被增长率本身放大到不可收拾。

**反推工程常识**：用归并的实测数据校验——n = 10⁵ 耗时 0.215 秒，对应约 1.7×10⁶ 次操作（n log n ≈ 1.7×10⁶），单次操作约 0.13 微秒，即每秒约 7×10⁶ 次复杂操作。这比"10⁸ 次简单操作"低一个数量级——因为归并的每次操作涉及函数调用、列表 append 等开销，不是最朴素的"加法"。**"每秒 10⁸ 次"是估算上限，实际工程要打 10-50 倍折扣**，这也是为什么面试里常用 10⁷ 作为更保守的阈值。

### 2. 时间与空间的权衡：以两数之和为例

复杂度不只看时间，还要看空间。同一问题，不同解法的时间和空间往往此消彼长。

**经典案例：两数之和**。给定数组 nums 和目标值 target，找两个数加起来等于 target 的下标。

| 解法 | 时间 | 空间 | 思路 |
|---|---|---|---|
| 暴力双循环 | O(n²) | O(1) | 两两组合试 |
| 排序 + 双指针 | O(n log n) | O(n) 或 O(1) | 排序后两端往中间收 |
| 哈希表 | O(n) | O(n) | 一次遍历，哈希表存已见值 |

- n 小（n ≤ 100）：暴力最省内存，时间也能接受
- n 大且内存够：哈希表最快，但多用 O(n) 空间
- n 大且内存紧：排序 + 双指针，时间换空间

**这就是空间复杂度与时间复杂度的权衡**：哈希表用 O(n) 额外空间换 O(n) 时间；排序用 O(n log n) 时间换 O(1) 额外空间。没有"绝对最优"，要看场景约束。

工程上的常见权衡模式：

- **缓存（空间换时间）**：把重复计算的结果存起来，下次直接查。LRU 缓存、记忆化搜索、动态规划的"备忘录"都是这个思路
- **流式处理（时间换空间）**：数据太大放不下内存，分批读入处理，多用计算时间换内存空间
- **预计算（空间换时间）**：提前算好结果存表，运行时直接查。倒排索引、布隆过滤器都是预计算

### 3. 教科书做法 vs 生产做法

| 场景 | 教科书做法 | 生产做法 | 原因 |
|---|---|---|---|
| 复杂度估算 | 手算 Big-O | profiling 工具实测 | 教科书练的是直觉，生产要看真实瓶颈（常数因子、缓存命中率） |
| 算法选择 | 看渐进复杂度选 | 看 n 的实际范围选 | n = 20 时 O(2ⁿ) 能跑，O(n²) 也能跑，常数小的暴力可能更快 |
| 哈希表实现 | 手写 hash + rehash | `dict` / `unordered_map` | 标准库做了大量工程优化（开放寻址、增量 rehash） |
| 排序选择 | 按复杂度选归并 | `sorted()`（Timsort） | Timsort 对真实数据（部分有序）特别快 |

**生产里复杂度分析的两个误区**：

1. **只看 Big-O，忽略常数**。理论上 O(n log n) 比 O(n²) 快，但 n = 50 时冒泡（小常数 O(n²)）可能比归并（大常数 O(n log n)）快——这就是 Timsort 在小数据切到插入排序的原因。
2. **只看时间，忽略缓存**。链表 O(1) 插入 vs 数组 O(n) 插入，看起来链表赢，但数组的连续内存对 CPU 缓存友好，实际常常更快。这就是为什么 `std::vector` 在很多场景下比 `std::list` 快。

### 4. 三个真实工程坑

**坑一：以为 O(n) 总比 O(n²) 快。**

```python
# n=20，暴力 vs 哈希
def two_sum_brute(nums, target):
    for i in range(len(nums)):
        for j in range(i + 1, len(nums)):
            if nums[i] + nums[j] == target:
                return [i, j]

def two_sum_hash(nums, target):
    seen = {}
    for i, x in enumerate(nums):
        if target - x in seen:
            return [seen[target - x], i]
        seen[x] = i
```

n = 20 时暴力 190 次比较，哈希表 20 次插入 + 20 次查找 + 哈希函数开销。暴力可能更快——哈希函数常数不小。**复杂度只在大 n 时主导，小 n 看常数**。

**坑二：把均摊 O(1) 当成单次 O(1)。**

```python
# 高频交易场景，每次插入必须 < 1ms
cache = {}
for i in range(10**7):
    cache[i] = i  # 偶尔触发 rehash，单次卡 50ms
```

均摊 O(1) 没错，但 rehash 那一刻的延迟可能违规。修复：预分配容量（Python `dict` 不支持，但 C++ `unordered_map::reserve` 支持），或用增量 rehash 的实现。

**坑三：忽略最坏情况，被构造数据卡死。**

```python
# 快排不随机化 pivot，遇到有序数据退化到 O(n²)
def quicksort_bad(arr):
    if len(arr) <= 1:
        return arr
    pivot = arr[0]   # 固定取首元素
    left = [x for x in arr[1:] if x < pivot]
    right = [x for x in arr[1:] if x >= pivot]
    return quicksort_bad(left) + [pivot] + quicksort_bad(right)

# 对手给你 [1,2,3,...,10^5] 的有序数据，每层只切掉 1 个元素，退化到 O(n²)
```

修复：随机化 pivot 或用三数取中。这是 Big-O 看最坏情况的意义——你必须假设对手会构造最坏输入。

## 四、速查与自测

### 速查表：常见复杂度阶与工程含义

| 复杂度阶 | n = 10⁵ 时操作数 | 1 秒能否完成 | 代表算法 |
|---|---|---|---|
| O(1) | 1 | ✓ | 哈希表查找 |
| O(log n) | 17 | ✓ | 二分查找 |
| O(n) | 10⁵ | ✓ | 线性扫描 |
| O(n log n) | 1.7×10⁶ | ✓ | 归并排序、快排（平均） |
| O(n²) | 10¹⁰ | ✗（需约 100 秒） | 冒泡、选择、插入 |
| O(n³) | 10¹⁵ | ✗（需 10⁷ 秒） | 朴素矩阵乘法 |
| O(2ⁿ) | 2¹⁰⁵ | ✗（宇宙寿命都不够） | 子集枚举 |
| O(n!) | 10⁵! | ✗ | 全排列 |

**主定理三种情况速查**：

| 递推 | a, b, d | $\log_b a$ vs d | 情况 | 结论 |
|---|---|---|---|---|
| T(n) = 2T(n/2) + O(n) | a=2, b=2, d=1 | 1 = 1 | 2 | Θ(n log n) |
| T(n) = T(n/2) + O(1) | a=1, b=2, d=0 | 0 = 0 | 2 | Θ(log n) |
| T(n) = 4T(n/2) + O(n) | a=4, b=2, d=1 | 2 > 1 | 1 | Θ(n²) |
| T(n) = 2T(n/2) + O(n²) | a=2, b=2, d=2 | 1 < 2 | 3 | Θ(n²) |

### 自测三问

**问题一：** 给定 1 秒时限和"每秒 10⁸ 次操作"的工程常识，n = 10⁶ 时哪些复杂度阶可行？请反推并说明理由。

**参考答案：** 1 秒内最多 10⁸ 次操作。n = 10⁶ 时：
- O(n) = 10⁶，可行
- O(n log n) ≈ 2×10⁷，可行
- O(n²) = 10¹²，超时 10000 倍，不可行
- O(n√n) ≈ 10⁹，临界，可能超时

所以 n = 10⁶ 时通常只接受 O(n) 或 O(n log n) 的算法。**这就是为什么面试官看到 n = 10⁶ 就排除所有 O(n²) 解法——不是经验主义，是用 10⁸/秒反推的硬约束**。

**问题二：** 主定理的三种情况分别对应什么"直觉"？请用"子问题工作量 vs 合并工作量"的视角解释。

**参考答案：** 主定理比较的是"叶子层总工作量"（$n^{\log_b a}$）和"根层合并工作量"（$n^d$）：
- 情况 1（$d < \log_b a$）：子问题分得多，叶子层总工作量大于合并层，复杂度由叶子层主导 = $\Theta(n^{\log_b a})$
- 情况 2（$d = \log_b a$）：每层工作量相同，共 log n 层，所以是 $\Theta(n^d \log n)$
- 情况 3（$d > \log_b a$）：合并代价随 n 增长比子问题快，复杂度由根层合并主导 = $\Theta(n^d)$

直觉记忆：**子问题分得多看叶子，合并代价大看根**。

**问题三：** 哈希表插入"均摊 O(1)"但"最坏 O(n)"。这两种说法为什么不矛盾？什么场景下你必须区分这两者？

**参考答案：** 不矛盾是因为它们度量不同对象——"最坏 O(n)"度量单次插入（触发 rehash 时要把 n 个元素重哈希），"均摊 O(1)"度量一连串插入的总代价除以次数（n 次插入的总 rehash 代价是 O(n)，分摊到每次 O(1)）。

必须区分的场景：**对单次延迟敏感的系统**。比如高频交易要求每次操作 < 1ms，rehash 那一刻 50ms 的卡顿就是事故；游戏每帧 16ms 预算，rehash 卡帧就掉帧。这类场景要预分配容量或用增量 rehash，把"均摊 O(1)"做成"单次 O(1)"。

### 算法思想 × 生活迁移

复杂度分析的核心思想是"**用增长率而非绝对值判断可行性**"——大问题看增长趋势，小问题才纠结常数。

**迁移一：技术选型看规模的增长趋势。** 选数据库时，单机 MySQL vs 分布式 MySQL，对小公司（数据 GB 级）差别不大，对大公司（TB 级）是天壤之别。复杂度分析告诉你：单机的容量阶是 O(单机容量)，分布式的容量阶是 O(总容量)。规模大了，增长率低的方案必然胜出。**别在小规模时纠结选型，要预判规模增长后哪个增长率能撑住**。

**迁移二：个人时间管理的"复杂度"。** 处理 10 封邮件，逐封回复是 O(n)；处理 10000 封邮件，逐封回复就退化成 O(n²)（每封都要单独思考，思考成本随数量累积）。批量回复模板、按主题归类——是把 O(n²) 降到 O(n log n)。**人脑处理事务也有"复杂度阶"，规模大了必须换方法**。

**迁移三：投资理财的"均摊思维"。** 单次大额消费（买车、装修）的"痛苦"是 O(n)，但分摊到 10 年使用周期，每年均摊成本是 O(1)。这就是为什么"按使用次数算性价比"比"看总价"更合理——均摊思维把稀疏的大额开销平摊成长期的小成本。

**为什么这些迁移成立：** 复杂度分析的前提是"问题规模可以变化"，这个前提在技术选型、时间管理、投资理财里都成立——都是"规模小看常数，规模大看增长率"。迁移成立的关键是"问题有可变的规模维度"——如果规模固定（比如一次性事件），复杂度分析就退化为单次成本评估，失去意义。
