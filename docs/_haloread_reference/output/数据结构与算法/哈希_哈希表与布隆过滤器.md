# 算法课·10｜哈希：从哈希表到布隆过滤器

> 学完你能：①给纯小白讲清"为什么字典能秒查单词" ②默写拉链法哈希表并讲透每一行 ③手撕 LRU Cache 应对面试 ④推导 Bloom Filter 假阳性率公式，理解"用误差换空间"的工程哲学 ⑤用对数器对比 Python `dict` 验证自己写的哈希表

## 一、问题与思路：从"翻字典"到 $O(1)$ 查找

### 1. 一个生活场景：药房按药名找药

假设你在一间有 1 万种药的大药房当药剂师。有人拿着处方来取"阿莫西林"。

**方式一（线性查找式）**：从第一个药架开始，一格一格看药名——"感冒灵""板蓝根"……一直翻到第 8000 格才看到"阿莫西林"。1 万种药平均要翻 5000 次，慢到病人等不起。

**方式二（按字母分桶式）**：药架按药名拼音首字母分区——A 区、B 区……Z 区。"阿莫西林"拼音首字母是 A，直接走到 A 区，只在 A 区几十种药里找。从 1 万次降到几十次。

**方式三（哈希式）**：把"阿莫西林"四个字丢进一个函数 $h$，函数算出一个数字比如 4287，你直接走到第 4287 号抽屉，拉开就是这盒药。**1 次定位，不比较、不翻找**。

这就是哈希的本质——**用一个函数把"键"映射到"地址"，让查找从"逐个比较 $O(n)$"变成"一次定位 $O(1)$"**。

为什么这件事重要？因为现实世界里"按名字找东西"无处不在：查字典、查通讯录、查缓存、查数据库索引、查爬虫 URL 是否抓过……如果每次都线性扫，互联网早就卡死了。哈希是支撑现代软件"秒级响应"的底层基础设施。

### 2. 核心思想：哈希函数的四条铁律

哈希能做 $O(1)$ 查找，全靠那个"键 → 地址"的函数 $h$。但不是随便一个函数都能当哈希函数，它必须满足四条性质：

- **确定性**：同一个键每次调用 $h$ 必须返回同一个值。否则存进去是 4287，取出来变成 1024，整个表就废了。
- **均匀性**：不同键应尽量散布到不同地址，避免"挤在一个抽屉里"。这是性能的命根——后面会讲，挤在一起就退化成链表 $O(n)$。
- **雪崩效应**：输入差一个字符，输出应彻底改变。`h("abc")` 和 `h("abd")` 的结果应该完全无关。否则攻击者能构造大量"挤在一起"的键，把你的哈希表打回 $O(n)$（这就是著名的 HashDoS 攻击）。
- **单向性（部分场景）**：密码学哈希（MD5/SHA256）要求从哈希值反推原键几乎不可能。普通哈希表不要求，但理解这一点能区分"哈希表用的哈希"和"密码学哈希"。

**冲突不可避免——这是鸽笼原理的铁律**。把 1 万种药映射到 1000 个抽屉，至少有一个抽屉要装 10 种以上药。再好的哈希函数也救不了。所以哈希表设计的核心不是"消灭冲突"，而是"冲突来了怎么办"——这就是下一节要讲的拉链法和开放寻址。

### 3. 历史地位：从 IBM 实验室到互联网基础设施

哈希表的思想最早由 IBM 的 Hans Peter Luhn 在 1953 年的内部备忘录中提出（用"hashing"这个词是因为把键"切碎"成地址）。此后 60 年，哈希从一种"查找技巧"演变成现代软件的基础设施：

- **1953** Luhn 提出哈希表（拉链法雏形）
- **1970** Burton Bloom 发表 *Space/Time Trade-offs in Hash Coding with Allowable Errors*，提出 Bloom Filter——**主动允许少量错误换取巨大空间节省**，这是"概率数据结构"的开山之作 [12]
- **1974** Knuth 在 TAOCP 第三卷系统化哈希理论，给出拉链法与开放寻址的复杂度分析
- **1997** Karger 等人提出一致性哈希，解决分布式缓存的"加减节点雪崩"问题
- **2001** Pagh & Rodler 提出 Cuckoo Hashing，用"踢出重放"保证最坏 $O(1)$ 查找
- **2008** Python 把 `dict` 从伪随机探测改成"结合开放寻址"，Java 8 把 HashMap 长链表退化成红黑树

**这条历史线的关键洞察**：哈希的发展是"从确定性到概率性"的演进——经典哈希表追求"零误差 $O(1)$"，Bloom Filter 主动接受"假阳性"换空间，Cuckoo Hashing 用"最坏 $O(1)$"换"插入稍慢"。理解这条线，就理解了工程权衡的核心：**没有银弹，只有按场景选误差/速度/空间的组合**。

## 二、原理与实现：从拉链法到 Bloom Filter

### 1. 拉链法哈希表：put/get 逐行讲透

拉链法是最经典的哈希表实现——每个"抽屉"挂一条链表，冲突的元素都挂到同一条链上。看一个最小实现：

```python
class HashTableChaining:
    def __init__(self, capacity=8):
        self.capacity = capacity
        self.size = 0
        self.buckets = [[] for _ in range(capacity)]   # 每个桶是一条链表（Python 用 list 模拟）

    def _hash(self, key):
        return hash(key) % self.capacity               # 内置 hash 取模落到桶内

    def put(self, key, value):
        idx = self._hash(key)                          # ① 算桶下标
        bucket = self.buckets[idx]
        for i, (k, _) in enumerate(bucket):            # ② 在桶内线性扫描
            if k == key:
                bucket[i] = (key, value)               # ③ 已存在则覆盖
                return
        bucket.append((key, value))                    # ④ 不存在则追加
        self.size += 1
        if self.size > self.capacity * 0.75:           # ⑤ 负载因子超 0.75 触发扩容
            self._rehash(self.capacity * 2)

    def get(self, key):
        idx = self._hash(key)                          # ① 算桶下标
        for k, v in self.buckets[idx]:                 # ② 桶内线性扫描
            if k == key:
                return v                               # ③ 命中
        return None                                    # ④ 未命中

    def _rehash(self, new_capacity):
        old = self.buckets
        self.capacity = new_capacity
        self.buckets = [[] for _ in range(new_capacity)]
        self.size = 0
        for bucket in old:                             # 所有元素重新哈希到新桶
            for k, v in bucket:
                self.put(k, v)
```

**逐行讲透**（左神风格，每一行都不能含糊）：

- **`self.buckets = [[] for _ in range(capacity)]`**：初始化 capacity 个空桶，每个桶是一条链表。为什么用 list 模拟链表？因为 Python 没有内置链表，且桶内元素通常很少（负载因子 0.75 时平均 0.75 个/桶），list 的连续内存比真链表更缓存友好。
- **`hash(key) % self.capacity`**：先用 Python 内置 `hash` 算出大整数（雪崩性已经在 `hash` 里做了），再对桶数取模落到桶内。**关键：模的必须是桶数 `capacity`，不是某个固定数**——否则扩容后元素位置全乱。
- **`for i, (k, _) in enumerate(bucket)`**：桶内线性扫描。这就是冲突的代价——同一桶里多个键要逐个比。极端情况下所有键都落一个桶，退化成 $O(n)$ 链表。
- **`bucket[i] = (key, value)`**：键已存在就覆盖旧值。这是哈希表"键唯一"语义的核心——同一个键只能存一份值。
- **`bucket.append((key, value))`**：新键追加到桶尾。注意是"追加"不是"插入头部"——拉链法规范上插头尾都行，Python list 的 append 是均摊 $O(1)$，足够。
- **`if self.size > self.capacity * 0.75`**：负载因子 $\alpha = size/capacity$ 超过 0.75 就扩容。为什么是 0.75？这是 Java HashMap 的经验值——太小浪费空间，太大桶链变长查询慢。后面会讲这个数的权衡。
- **`self._rehash(self.capacity * 2)`**：扩容两倍，**所有元素必须重新哈希**。为什么不能直接搬？因为桶数变了，`hash(key) % capacity` 结果也变了，旧位置完全无效。rehash 是哈希表扩容的最大开销，单次 $O(n)$，但均摊到每次插入是 $O(1)$。

**走一个例子**：`capacity=8`，插入 `("apple", 1)`、`("banana", 2)`、`("apple", 99)`：

```
put("apple", 1):
  hash("apple")=1234, 1234 % 8 = 2
  bucket[2] = [("apple", 1)]
put("banana", 2):
  hash("banana")=5678, 5678 % 8 = 6
  bucket[6] = [("banana", 2)]
put("apple", 99):
  hash("apple")=1234, 1234 % 8 = 2
  bucket[2] 扫描到 "apple" 已存在 → 覆盖
  bucket[2] = [("apple", 99)]
get("apple") → 99，1 次定位命中
get("cherry") → None，bucket[?] 扫描未命中
```

**关键观察**：查找"apple"只需 1 次 `hash` + 1 次桶内比较，与表大小无关——这就是 $O(1)$ 的来源。但前提是"桶内链表很短"，而这要求负载因子控制 + 哈希函数均匀分布。

### 2. 开放寻址法：不挂链表，冲突了就找下一个空位

拉链法的缺点是每个桶要维护一个链表（额外指针开销）。开放寻址换个思路——**所有元素都住在数组里，冲突了就按规则找下一个空位**。三种探测规则：

- **线性探测**：冲突了就往后 +1、+2、+3…… 找空位。简单但容易"聚集"——一长串连续占用的位置，每次插入都要跨过整片。
- **二次探测**：冲突了往后 $+1^2, +2^2, +3^2, \dots$。缓解一次聚集，但仍有"二次聚集"。
- **双重哈希**：用第二个哈希函数算步长 `step = h2(key)`，冲突了每次加 `step`。聚集最轻，但实现复杂。

```python
class HashTableOpenAddressing:
    _DELETED = object()   # 哨兵：标记被删除的位置（区分"空"和"删过"）

    def __init__(self, capacity=8):
        self.capacity = capacity
        self.size = 0
        self.keys = [None] * capacity
        self.values = [None] * capacity

    def _probe(self, key, i):
        # 线性探测：(h(key) + i) % capacity
        return (hash(key) % self.capacity + i) % self.capacity

    def put(self, key, value):
        for i in range(self.capacity):
            idx = self._probe(key, i)
            if self.keys[idx] is None or self.keys[idx] is self._DELETED:
                self.keys[idx] = key
                self.values[idx] = value
                self.size += 1
                return
            if self.keys[idx] == key:
                self.values[idx] = value   # 覆盖
                return
        raise Exception("Hash table full")   # 探测完 capacity 次还没空位

    def get(self, key):
        for i in range(self.capacity):
            idx = self._probe(key, i)
            if self.keys[idx] is None:
                return None                  # 遇到空，肯定没存
            if self.keys[idx] == key:
                return self.values[idx]
        return None
```

**逐行讲透**：

- **`_DELETED = object()`**：删除元素的哨兵。为什么需要？因为线性探测的查找"遇到空就停"——如果直接把删除位置置 `None`，后续探测链会断在中间，原本该能找到的元素找不到了。所以删的位置要标记成 `_DELETED`（不是 `None`），查找时跳过、插入时可复用。
- **`for i in range(self.capacity)`**：最多探测 capacity 次。开放寻址的硬约束——**负载因子必须严格小于 1**，否则装满了就死循环。这也是为什么开放寻址表通常保持 $\alpha \le 0.7$。
- **`if self.keys[idx] is None or ... is self._DELETED`**：找到空位或"删过"位置就插入。
- **`raise Exception("Hash table full")`**：探测 capacity 次都没空位，说明表满了。生产代码应在到达这之前就 rehash。

### 3. 拉链法 vs 开放寻址：怎么选

| 维度 | 拉链法 | 开放寻址 |
|---|---|---|
| 负载因子上限 | 可 $>1$（链表能挂长） | 必须 $<1$ |
| 删除操作 | 直接删链表节点 | 需要哨兵标记 |
| 缓存友好性 | 一般（链表节点不连续） | 好（数据全在数组里） |
| 最坏情况 | 链表退化 $O(n)$ | 聚集导致探测长 $O(n)$ |
| 内存开销 | 每元素多一个指针 | 无额外指针，但有空槽浪费 |
| 典型实现 | Java HashMap（Java 7 前） | Python `dict`、Ruby Hash |

**生产趋势**：现代语言运行时越来越倾向开放寻址——因为缓存友好性在现代 CPU 上越来越重要（连续访问比指针跳转快 5-10 倍）。Python、Ruby、Lua 都用开放寻址。Java HashMap 仍用拉链法（Java 8 后长链表退化成红黑树，下面讲）。

### 4. 负载因子与 rehash：那个 0.75 是怎么来的

负载因子 $\alpha = \frac{\text{元素数}}{\text{桶数}}$ 是哈希表最重要的参数。它直接决定平均查找代价：

- 拉链法：成功查找平均 $\alpha + 1$ 次比较（CLRS [1] 第 11 章证明）
- 开放寻址（线性探测）：成功查找平均 $\frac{1}{2}\left(1 + \frac{1}{1-\alpha}\right)$ 次探测

**为什么 Java HashMap 选 0.75**？这是空间和时间的经验折中——0.5 太浪费空间（一半桶空着），1.0 探测代价飙升（开放寻址下接近无穷大）。0.75 时拉链法平均 1.75 次比较、开放寻址约 2.5 次探测，都是可接受范围，且空间利用率 75% 不算浪费。

**rehash 的代价**：扩容时所有元素要重新哈希到新桶，单次 $O(n)$。但均摊到每次插入仍是 $O(1)$——因为扩容是几何级数（每次翻倍），总开销 $\sum n/2^k = O(n)$。这是均摊分析的典范例子。

### 5. Bloom Filter：用误差换空间

经典哈希表追求"零误差 $O(1)$"，但有些场景"可以接受少量误差，但要省空间"。比如爬虫判断"这个 URL 抓过没"——10 亿 URL 存哈希表要几十 GB，但如果允许"误判 1% 没抓过说成抓过"（最多多爬 1%），就能用几百 MB 搞定。这就是 Bloom Filter [12]。

```python
import math
import mmh3   # MurmurHash3，非密码学哈希，速度快雪崩好

class BloomFilter:
    def __init__(self, n, p):
        # n: 预期元素数；p: 期望假阳性率
        self.m = int(-n * math.log(p) / (math.log(2) ** 2))   # 最优位数
        self.k = int((self.m / n) * math.log(2))               # 最优哈希函数数
        self.bits = [0] * self.m

    def _hashes(self, key):
        # 双哈希技巧：h1 + i*h2 生成 k 个独立哈希，避免真用 k 个函数
        h1 = mmh3.hash(str(key), 0) % self.m
        h2 = mmh3.hash(str(key), 1) % self.m
        for i in range(self.k):
            yield (h1 + i * h2) % self.m

    def add(self, key):
        for h in self._hashes(key):
            self.bits[h] = 1

    def contains(self, key):
        return all(self.bits[h] == 1 for h in self._hashes(key))   # 全 1 才"可能存在"
```

**逐行讲透**：

- **`self.m = int(-n * math.log(p) / (math.log(2) ** 2))`**：位数 $m$ 的最优公式。推导见下一节，给定 $n$ 和 $p$，这个 $m$ 让假阳性率最低、空间最省。
- **`self.k = int((self.m / n) * math.log(2))`**：哈希函数个数 $k$ 的最优公式。$k$ 太少覆盖不够，太多互相重叠，最优值是 $\frac{m}{n}\ln 2$。
- **`_hashes` 用双哈希技巧**：真用 $k$ 个独立哈希函数太慢。Kirsch 和 Mitzenmacher 2006 年证明 $h_1(x) + i \cdot h_2(x)$ 这一对哈希组合在实践上和 $k$ 个独立哈希效果相当，只需算 2 次。
- **`add` 把 $k$ 个哈希位置都置 1**：每个元素在位数组里"涂" $k$ 个点。
- **`contains` 检查 $k$ 个位置是否全 1**：全 1 → "可能存在"（可能是别的元素涂的，假阳性）；有 0 → "一定不存在"（**零假阴性**，因为如果元素加过，它的 $k$ 个位必然都是 1）。

**假阳性率公式推导**（高手向）：

设位数组大小 $m$、哈希函数数 $k$、已插入元素数 $n$。某一位在一次插入中被某个哈希函数置 1 的概率是 $1/m$，未被置 1 的概率是 $1 - 1/m$。插入 $n$ 个元素、每个用 $k$ 个哈希，某一位仍为 0 的概率是：

$$
\left(1 - \frac{1}{m}\right)^{kn} \approx e^{-kn/m}
$$

（最后一步用了 $(1-1/m)^m \to e^{-1}$ 的极限近似。）该位为 1 的概率是 $1 - e^{-kn/m}$。查询一个**未插入**的元素时，$k$ 个哈希位置全为 1 才误判，所以假阳性率：

$$
P \approx \left(1 - e^{-kn/m}\right)^k
$$

对 $k$ 求导令导数为 0，得最优 $k = \frac{m}{n}\ln 2$。代回得 $P_{\min} \approx (0.6185)^{m/n}$。这就是为什么 $m$ 越大、$n$ 越小，误差越低——空间换准确率。

### 6. 五指标评价

| 指标 | 拉链法哈希表 | 开放寻址哈希表 | Bloom Filter |
|---|---|---|---|
| 查找 | 平均 $O(1)$，最坏 $O(n)$ | 平均 $O(1)$，最坏 $O(n)$ | $O(k)$，固定 |
| 插入 | 平均 $O(1)$，最坏 $O(1)$ 摊还 | 平均 $O(1)$，最坏 $O(1)$ 摊还 | $O(k)$ |
| 空间 | $O(n)$ + 指针开销 | $O(n)$ + 空槽 | $O(m)$ bits，远小于 $n$ |
| 误差 | 无 | 无 | 假阳性 $\approx (1-e^{-kn/m})^k$，无假阴性 |
| 删除 | 支持 | 需哨兵 | **不支持**（删 1 会误伤其他元素） |
| 缓存 | 一般 | 好 | 极好（位数组连续） |

## 三、实践与面试：手撕模板、对数器、面试题

### 1. 面试手撕模板：LRU Cache（按"技术的水有多深"八层讲）

LRU Cache 是哈希表面试的"常驻题"——它把"哈希 $O(1)$ 查找"和"双向链表 $O(1)$ 增删"组合起来，是数据结构组合应用的典范。

```python
class Node:
    def __init__(self, key=0, value=0):
        self.key = key
        self.value = value
        self.prev = None
        self.next = None

class LRUCache:
    def __init__(self, capacity: int):
        self.cap = capacity
        self.cache = {}                    # key -> Node，哈希表 O(1) 定位
        # 哨兵节点：避免头尾边界判断
        self.head = Node()                 # 最近用
        self.tail = Node()                 # 最久没用
        self.head.next = self.tail
        self.tail.prev = self.head

    def _remove(self, node):
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_to_head(self, node):
        node.next = self.head.next
        node.prev = self.head
        self.head.next.prev = node
        self.head.next = node

    def get(self, key):
        if key not in self.cache:
            return -1
        node = self.cache[key]
        self._remove(node)                 # 从原位置摘下
        self._add_to_head(node)            # 移到表头（最近用过）
        return node.value

    def put(self, key, value):
        if key in self.cache:              # 已存在：更新值 + 提到表头
            node = self.cache[key]
            node.value = value
            self._remove(node)
            self._add_to_head(node)
            return
        node = Node(key, value)
        self.cache[key] = node
        self._add_to_head(node)
        if len(self.cache) > self.cap:     # 超容量：删尾 + 删哈希
            lru = self.tail.prev
            self._remove(lru)
            del self.cache[lru.key]
```

**面试时怎么讲**（按"技术的水有多深"八层面）：

1. **问题层**（核心思想）：缓存要"按访问顺序淘汰最久没用的"。朴素做法：链表按访问时间排，但更新位置要 $O(n)$ 查找。哈希表能把"找节点"降到 $O(1)$，链表能把"挪节点"降到 $O(1)$——两者结合就是 LRU。
2. **原理层**：哈希表存 `key → Node`，双向链表按"最近访问顺序"排。`get` 时哈希定位节点、链表把它提到表头；`put` 时新节点加表头，超容则删尾。
3. **优劣层**：所有操作 $O(1)$，但空间 $O(n)$ 双份（哈希 + 链表节点）；只支持 LRU 不支持 LFU。
4. **演进层**：LRU → LFU（按访问频次）→ ARC（自适应 LRU/LFU 混合）→ Redis 的近似 LRU（采样淘汰，不全表排序）。
5. **坑位层**：①为什么用双向链表不是单向？删尾要 $O(1)$ 必须能反向拿到 prev。②为什么用哨兵头尾节点？避免"链表为空""删的是头/尾"等边界判断。
6. **场景层**：Redis 缓存淘汰、MySQL Buffer Pool、CPU Cache、浏览器缓存。
7. **替代层**：Python 直接用 `OrderedDict`（底层就是哈希+双向链表）；生产环境别自己手写。
8. **追问层**：并发安全吗？不加锁不安全——生产用分段锁或 CAS。

**面试加分点**：主动提"Redis 用近似 LRU 而非精确 LRU，因为全表排序在大数据量上太贵"——能立刻展示你懂工程权衡，不只是会背模板。

### 2. 对数器：对比 Python `dict` 跑 5000 次

左神反复强调：写完数据结构不要靠肉眼检查，用对数器对比"绝对正确的标准库"。

```python
import random

class HashTableChaining:
    def __init__(self, capacity=8):
        self.capacity = capacity
        self.size = 0
        self.buckets = [[] for _ in range(capacity)]

    def _hash(self, key):
        return hash(key) % self.capacity

    def put(self, key, value):
        idx = self._hash(key)
        for i, (k, _) in enumerate(self.buckets[idx]):
            if k == key:
                self.buckets[idx][i] = (key, value)
                return
        self.buckets[idx].append((key, value))
        self.size += 1
        if self.size > self.capacity * 0.75:
            self._rehash(self.capacity * 2)

    def get(self, key):
        idx = self._hash(key)
        for k, v in self.buckets[idx]:
            if k == key:
                return v
        return None

    def remove(self, key):
        idx = self._hash(key)
        for i, (k, _) in enumerate(self.buckets[idx]):
            if k == key:
                self.buckets[idx].pop(i)
                self.size -= 1
                return True
        return False

    def _rehash(self, new_capacity):
        old = self.buckets
        self.capacity = new_capacity
        self.buckets = [[] for _ in range(new_capacity)]
        self.size = 0
        for bucket in old:
            for k, v in bucket:
                self.put(k, v)

def checker(test_times=5000, max_n=100, max_val=1000):
    """对数器：跑 5000 次，每次随机数据对比 Python dict。"""
    for t in range(test_times):
        ht = HashTableChaining()
        std = {}
        ops = random.randint(1, max_n)
        for _ in range(ops):
            op = random.choice(['put', 'get', 'remove'])
            key = random.randint(0, max_val)
            if op == 'put':
                value = random.randint(0, max_val)
                ht.put(key, value)
                std[key] = value
            elif op == 'get':
                if ht.get(key) != std.get(key):
                    print(f"出错！op=get key={key} mine={ht.get(key)} std={std.get(key)}")
                    return False
            else:
                if ht.remove(key) != (key in std):
                    print(f"出错！op=remove key={key}")
                    return False
                std.pop(key, None)
        # 最后全量比对
        for key in std:
            if ht.get(key) != std[key]:
                print(f"出错！最终不一致 key={key}")
                return False
    print("对数器验证通过！")
    return True

checker()
```

**对数器的价值**：覆盖三种操作（put/get/remove）的随机交叉、含覆盖写、含删不存在的键、含删后再插。跑 5000 轮只要几秒，比手写测试用例全面得多。**左神原话**："写完一个数据结构，第一件事就是写对数器对比标准库，跑够量再说自己写对了。"

### 3. 教科书做法 vs 生产做法

| 场景 | 教科书做法 | 生产做法 | 原因 |
|---|---|---|---|
| 通用键值存储 | 手写拉链法哈希表 | Python `dict` / Java `HashMap` | 标准库经过极端优化与多年实战 |
| 需要按访问顺序 | 手写 LRU（哈希+双向链表） | `collections.OrderedDict` / `functools.lru_cache` | 库已经实现，且并发安全 |
| 去重（允许少量误差） | 手写 Bloom Filter | `pybloom-live` / Redis `BF.ADD` | Redis 7 内置 Bloom Filter 模块 |
| 分布式缓存路由 | 手写一致性哈希 | Ketama / 一致性哈希库 | 工程实现要考虑虚拟节点、副本 |
| 海量字符串去重 | 哈希表存全部 | Bloom Filter / HyperLogLog | 哈希表存不下，必须用概率结构 |

**生产里几乎不手写哈希表**，但理解它的价值在于：

1. **理解 `dict` / `HashMap` 的性能边界**：知道为什么 `dict` 是 $O(1)$ 不是 $O(n)$，知道什么时候会退化（hash 冲突攻击、负载因子过高）。
2. **理解缓存淘汰策略**：Redis、MySQL Buffer Pool 都基于 LRU/LFU，不懂哈希+链表组合就读不懂它们的源码。
3. **理解概率数据结构的工程哲学**：Bloom Filter、HyperLogLog、Count-Min Sketch 都是"用误差换空间/时间"，理解 Bloom 是入门这条线的钥匙。

### 4. 三个真实工程坑

**坑一：用 mutable 对象当哈希键，行为突变。**

```python
d = {}
key = [1, 2, 3]
# d[key] = "value"   # TypeError: unhashable type: 'list'
```

症状：列表不能当 dict 的键。根因：哈希键必须"生命周期内哈希值不变"，list 是 mutable 改了内容哈希就变，会找不到。修复：用 tuple（immutable）`key = (1, 2, 3)`，或用 `frozenset`。

**坑二：哈希冲突攻击（HashDoS）。**

```python
# 攻击者构造大量哈希冲突的键
attack_keys = [str(i) for i in range(100000) if hash(str(i)) % 8 == 0]
# 这些键全落同一个桶，dict 退化成 O(n) 链表
d = {k: 1 for k in attack_keys}
# 后续每次 get 都 O(n)，CPU 100%
```

症状：服务被几个请求打挂。根因：旧版 Python/Java 用可预测的哈希函数，攻击者能离线算出冲突键。修复：Python 3.3+ 引入哈希随机化（每次启动进程 hash seed 不同），Java 8+ 长链表退化红黑树 $O(\log n)$。

**坑三：Bloom Filter 删元素，导致误判扩散。**

```python
bf = BloomFilter(n=1000, p=0.01)
bf.add("a")
bf.add("b")
# 两个元素的哈希位有重叠
bf.contains("a")  # True（正确）
# 如果"删"a：把 a 的所有哈希位置 0
# 那 b 的重叠位也被置 0，bf.contains("b") → False（误判"不存在"）
```

症状：删了 a 之后 b 也找不到了。根因：Bloom Filter 的位是共享的，删一个会影响所有重叠元素。修复：①不要删——大多数场景"只增不删"。②用 Counting Bloom Filter，每个位是计数器而非布尔，add +1、delete -1，但空间翻 4-8 倍。③用 Cuckoo Filter，原生支持删除。

### 5. 面试高频题

**面试题一：两数之和（LeetCode 1）**

> 给定数组和目标值 target，返回两数之和等于 target 的下标。

**朴素法 $O(n^2)$**：双重循环枚举所有对。

**哈希一次扫描 $O(n)$**：

```python
def twoSum(nums, target):
    seen = {}                          # 值 -> 下标
    for i, x in enumerate(nums):
        if target - x in seen:         # 配对元素已在表里
            return [seen[target - x], i]
        seen[x] = i
    return []
```

**讲解要点**：边扫边把"已见过的值"存哈希表，每来一个新元素查"它的配对在不在表里"。一次扫描 $O(n)$，空间 $O(n)$。**关键洞察**：哈希把"找配对"从 $O(n)$ 降到 $O(1)$——这是哈希在算法里最高频的用法："用空间换时间，把查找从线性降到常数"。

**面试题二：字符串变位词分组（LeetCode 49）**

> 把字母重排能得到彼此的字符串归为一组。

```python
from collections import defaultdict

def groupAnagrams(strs):
    groups = defaultdict(list)
    for s in strs:
        key = "".join(sorted(s))       # 排序后的字符串作为"规范化键"
        groups[key].append(s)
    return list(groups.values())
```

**讲解要点**：变位词的"身份"是排序后的字符串——`"eat"`/`"tea"`/`"ate"` 排序后都是 `"aet"`。用这个规范化键做哈希分组。复杂度 $O(n \cdot k\log k)$（$n$ 个串、每串长 $k$）。**高手向**：可以用 26 字母计数做键（`tuple(counts)`），降到 $O(n \cdot k)$——这是"键的设计决定复杂度"的典型例子。

**面试题三：LRU Cache（LeetCode 146）**

见上面"面试手撕模板"。**讲解要点**：哈希 + 双向链表组合，所有操作 $O(1)$。**追问"为什么不用单链表"**：单链表删尾要 $O(n)$ 找前驱；双向链表能 $O(1)$ 拿 prev，直接摘除。

### 6. 高手向分析：四个进阶话题

**话题一：一致性哈希（分布式缓存）**

经典哈希取模 `hash(key) % N` 在分布式缓存里有致命问题——加一台机器 $N \to N+1$，几乎所有键都要重新分配，缓存全失效引发雪崩。一致性哈希的解法：把哈希空间看成环 $[0, 2^{32})$，机器和键都映射到环上，键顺时针找到的第一台机器就是它的归属。加减机器只影响相邻一段，其他机器不动。

但朴素一致性哈希有"数据倾斜"问题——机器在环上分布不均，有的多有的少。解法是**虚拟节点**：每台物理机在环上放 100-200 个虚拟节点，统计上分布就均匀了。Memcached、Redis Cluster、Cassandra 都用这个方案。

**话题二：Java 8 HashMap 的红黑树退化**

Java 7 的 HashMap 拉链法在最坏情况下（hash 冲突攻击）链表退化成 $O(n)$ 链表，CPU 100%。Java 8 的改进：某桶链表长度超过 8 且表容量 ≥ 64 时，把链表重组成红黑树——查找从 $O(n)$ 降到 $O(\log n)$。当节点数降到 6 以下又退回链表（避免小数据上红黑树开销大）。

**为什么是 8**？这是泊松分布的统计结果——负载因子 0.75 时，桶内元素数服从泊松分布 $\lambda=0.5$，达到 8 的概率约 $6 \times 10^{-8}$，正常使用几乎不会触发，只在攻击或极端情况下生效。

**话题三：Python `dict` 的开放寻址实现**

Python 3.6+ 的 `dict` 用一种特殊的开放寻址——结合伪随机探测 + "紧凑表"布局：

- 哈希表只存"索引"（1 字节/2 字节/4 字节按表大小选），实际键值对存在另一个连续数组里。
- 查找时探测哈希表里的索引，命中后去键值对数组取数据。
- 好处：①键值对连续存放，缓存友好 ②哈希表小（只存索引），缓存也友好 ③插入顺序被保留（Python 3.7+ 保证）。

这是 Python `dict` 在不损失 $O(1)$ 查找的前提下，把内存占用降了 20-25% 的精妙设计。

**话题四：Cuckoo Hashing（布谷鸟哈希）**

Cuckoo Hashing 用两个哈希函数 $h_1, h_2$，每个键有两个候选位置。查找时只需看这两个位置——**最坏 $O(1)$ 查找**，没有"链表退化"问题。插入时若两个位置都占了，就"踢出"现有元素让它去自己的另一个候选位置，可能引发连锁踢出，最坏 $O(n)$ 但期望 $O(1)$。

名字来自布谷鸟——它把蛋下在别的鸟巢里，幼鸟把原来的蛋挤出去。Cuckoo Filter（Bloom Filter 的替代）就基于这个思想，支持删除且空间更省。

### 自测三问

**问题一：** 哈希表号称 $O(1)$ 查找，但链表退化时是 $O(n)$。请说明 $O(1)$ 成立的条件，以及工程上如何保证这些条件。

**参考答案：** $O(1)$ 成立需要两个前提：①哈希函数均匀分布，键不集中落在少数桶 ②负载因子 $\alpha$ 控制在有界范围（拉链法 $\alpha \le 0.75$、开放寻址 $\alpha \le 0.7$），保证桶内平均元素数是常数。工程上保证的方法：①负载因子超阈值就 rehash 扩容 ②用雪崩性好的哈希函数（MurmurHash、SipHash）③防 HashDoS 攻击——Python 用哈希随机化（每次进程启动 hash seed 不同）、Java 8 长链表退化红黑树 $O(\log n)$ ④分布式场景用一致性哈希 + 虚拟节点保证均匀。**核心洞察**：$O(1)$ 是"平均"复杂度，依赖"哈希函数均匀 + 负载因子有界"两个前提，工程上要主动维护这两个前提。

**问题二：** Bloom Filter 为什么"可能有假阳性、但绝无假阴性"？请用位运算逻辑解释，并给出假阳性率公式。

**参考答案：** **无假阴性**：add 时把元素 $x$ 的 $k$ 个哈希位置全置 1，所以 contains 时如果 $x$ 真在表里，它的 $k$ 个位必然都是 1，必然返回 True——绝不会"在表里却说不在"。**有假阳性**：contains 时即使 $x$ 没加过，它的 $k$ 个哈希位置可能被其他元素涂成全 1，就会误判"可能存在"。假阳性率公式 $P \approx (1 - e^{-kn/m})^k$，其中 $m$ 是位数、$k$ 是哈希函数数、$n$ 是已插入元素数。最优 $k = \frac{m}{n}\ln 2$，此时 $P_{\min} \approx (0.6185)^{m/n}$。**工程意义**：Bloom Filter 用"允许少量假阳性"换"巨大空间节省"——10 亿 URL 去重，哈希表要几十 GB，Bloom Filter 1% 误差只要几百 MB。

**问题三：** LRU Cache 为什么必须用"哈希表 + 双向链表"组合？单用哈希表或单用链表为什么不行？

**参考答案：** LRU 要在 $O(1)$ 内完成三件事：①按 key 查 value ②把刚访问的节点提到"最近用"位置 ③淘汰"最久没用"的节点。单用哈希表：能 $O(1)$ 查找，但无法维护"访问顺序"——哈希表无序，没法知道哪个最久没用。单用链表：能 $O(1)$ 维护顺序（增删节点），但 `get(key)` 要从头扫 $O(n)$。**组合的妙处**：哈希表存 `key → Node`，把"找节点"从 $O(n)$ 降到 $O(1)$；双向链表存"访问顺序"，把"挪节点到表头"和"删尾节点"都做到 $O(1)$（双向链表能 $O(1)$ 拿 prev，单向不行）。**这是"数据结构组合"的典范**——单一结构搞不定的需求，组合两个互补结构就能搞定。Redis 缓存淘汰、MySQL Buffer Pool、CPU Cache 都基于这个组合思想。

### 算法思想 × 生活迁移

哈希的核心思想是"**用一个函数把'键'直接映射到'地址'，让查找从'逐个比较'变成'一次定位'**"。这个思想在生活中无处不在。

**迁移一：图书馆按编号找书 vs 按书名找书。** 老式图书馆按书名找书——从 A 区扫到 Z 区，逐架比较，$O(n)$。现代图书馆给每本书一个索书号（如 `TP311.5/123`），索书号直接定位到第几排第几架第几层——这就是哈希，1 次定位 $O(1)$。**关键洞察**：哈希的本质不是"算得快"，是"提前给每个东西分配一个可计算的地址"——地址可计算，查找就 $O(1)$；地址不可计算（只能比较），查找就 $O(n)$。

**迁移二：分诊台 = 哈希函数。** 医院急诊的分诊台把病人按症状分到不同科室——胸痛去心内、腹痛去消化、外伤去外科。这就是哈希：症状是 key、科室是 bucket。好的分诊（好哈希函数）让每个科室负载均衡；差的分诊（差哈希函数）让所有病人挤一个科室，其他科室空闲——这就是"哈希冲突 + 数据倾斜"。**迁移启示**：生活中很多"分配问题"本质是哈希——任务分人、订单分机器、流量分服务器——好的分配函数（哈希）让负载均匀，差的让某节点过载。

**迁移三：Bloom Filter 的"用误差换空间"哲学。** 邮件垃圾过滤：把已知垃圾邮件地址存 Bloom Filter，新邮件来查——查到"可能垃圾"（有假阳性，正常邮件误删概率 1%），查到"一定不是垃圾"（无假阴性，绝不会漏放垃圾）。这是"宁可错杀 1% 也不漏放"的工程取舍。生活中也常做这种取舍：①机场安检"宁可多查 100 个无辜旅客也不放过 1 个危险品"——安检机就是 Bloom Filter，假阳性高但无假阴性。②简历筛选"宁可漏掉一些好简历也要快速过滤明显不合适的"——这是反过来的取舍，接受假阴性换速度。**核心方法论**：识别场景对"假阳性"还是"假阴性"更敏感——安全场景怕漏（无假阴性）、效率场景怕错杀（控制假阳性），数据结构的选择要匹配这种敏感性。

**为什么这些迁移成立：** 哈希成立的前提是"键可以被函数化映射到地址"——这个前提在生活里同样成立：书的索书号、病人的症状分类、邮件的发件人，都是"可计算的键"。理解了"键 → 地址"这个抽象，就能在任何"按属性快速定位"的场景里看出哈希的影子——这就是本质的迁移力。
