# 算法课·09｜非比较排序三兄弟：计数、桶、基数如何突破 O(n log n)

> 前置知识：学完第 03-08 章（排序总论到堆排），尤其要理解第 03 章讲的"比较排序下界 $O(n\log n)$"
> 学完你能：①讲清计数/桶/基数三种非比较排序的原理与适用条件 ②理解它们为什么能突破 $O(n\log n)$ 下界 ③知道每种算法的"数据前提"——什么时候能用、什么时候不能用 ④能判断生产场景该不该用非比较排序

## 生活场景：数钱与分快递

发工资那天，柜员清点一沓零钞——1 元、5 元、10 元、50 元、100 元混在一起。他不会两两比较"这张比那张大吗"，而是直接按面额分堆：100 元一摞、50 元一摞……最后按面额从大到小摞起来，整整齐齐。这就是计数排序——用"面额"做索引，$O(1)$ 定位，比两两比较快得多。

快递分拣中心也类似：传送带上的包裹按邮编分区——100 区一车、200 区一车、300 区一车……每个区里再按地址细排。这是桶排序——先按区间分桶，桶内再精排。

如果包裹要按"省-市-区"逐级排序，分拣员会先按"区"分（最低位），再按"市"调整，最后按"省"定位——每一级都保留前一级的相对顺序，最终得到正确排序。这是基数排序——按位分解，从低到高，稳定累加。

这三种做法有个共同点：**不靠两两比较，靠数据本身的结构定位**。这就是非比较排序。

## 一、问题与思路：绕开比较，用数据本身做索引

### 1. 为什么需要非比较排序

第 03 章讲了一个数学定理：**任何基于"比较"的排序算法，最坏情况至少 $\Omega(n\log n)$ 次比较**。这个下界是严格的——决策树模型证明了，n 个元素有 $n!$ 种排列，每次比较只能区分两种情况，至少要 $\log_2(n!) = \Theta(n\log n)$ 次比较才能区分所有排列。

这意味着：快排、归并、堆排这些 $O(n\log n)$ 的算法，已经逼近了比较排序的理论极限。想更快？不可能——只要你还靠"比较"。

那有没有办法突破这个下界？有。**放弃"比较"，用数据的额外信息做排序**。

这就是非比较排序（Non-comparison Sort）的思路：不比较两个元素的大小，而是直接用元素的值（或值的某一部分）作为数组下标，把"比较"换成"寻址"。寻址是 $O(1)$ 的，所以能突破 $O(n\log n)$，达到 $O(n)$。

**核心思想**：非比较排序的本质是"**用值做索引，把比较换成寻址**"。这个思想听起来简单，但有个苛刻的前提——**数据必须满足特定条件**（值域有限、可分配到桶、可按位分解）。条件不满足时，非比较排序要么不能用，要么退化到 $O(n\log n)$ 甚至更差。

这就是非比较排序的"代价"：**用数据条件换时间**。比较排序通用（任何可比较的数据都能排），非比较排序专用（只对特定数据形态有效）。这是"通用 vs 专用"的根本权衡。

### 2. 三兄弟的共同基因与各自特化

计数排序、桶排序、基数排序，三者共享一个基因：**都不做两两比较，而是用数据本身定位**。但具体定位方式不同：

| 算法 | 定位方式 | 数据前提 |
|---|---|---|
| 计数排序 | 值直接做下标，统计每个值出现次数 | 值是整数且值域小 |
| 桶排序 | 值映射到桶，桶内再用其他排序 | 数据分布均匀，可分桶 |
| 基数排序 | 按值的"位"从低到高排序 | 数据可按位分解（整数/字符串） |

理解了"用值定位"这个共同基因，三者只是定位粒度的不同：计数是"一个值一个桶"，桶排是"一段值域一个桶"，基数是"一位一位定位"。

### 3. 一个常被误解的核心思想：稳定性对基数排序是必需的

很多人以为稳定性只是"锦上添花"，但对基数排序来说，**稳定性是算法正确性的必需条件**。

基数排序按"位"排序——先按个位排，再按十位排，再按百位排……每一步都必须用稳定排序，否则前一位排好的顺序会被后一位破坏。举例：`[21, 11, 22]` 按个位排（稳定）得 `[21, 11, 22]` 或 `[11, 21, 22]`（个位 1,1,2），再按十位排（稳定）得 `[11, 21, 22]`。如果第二步不稳定，`[21, 11]` 可能变成 `[11, 21]` 也可能保持 `[21, 11]`，结果不确定。

所以基数排序内部通常用计数排序（稳定）做按位排序。**稳定性对基数排序不是特性，是前提**——这是非比较排序里一个微妙但关键的点。

## 二、原理与实现：三兄弟的机制与适用条件

### 1. 计数排序（Counting Sort）

**机制**：值是整数且值域有限时，开一个大小为"值域 + 1"的数组 count，count[v] 记录值 v 出现的次数。最后按 count 还原排序结果。

```python
def counting_sort(arr, max_val):
    """计数排序：值域 [0, max_val] 的整数排序。"""
    count = [0] * (max_val + 1)
    for x in arr:
        count[x] += 1   # 统计每个值出现次数
    
    # 计算"前缀和"——count[v] 变成"值 ≤ v 的元素个数"
    for v in range(1, max_val + 1):
        count[v] += count[v - 1]
    
    # 反向填充结果数组（保证稳定）
    result = [0] * len(arr)
    for x in reversed(arr):   # 反向遍历保证稳定
        count[x] -= 1
        result[count[x]] = x
    return result
```

**逐行讲透**（左神风格）：

- **`count = [0] * (max_val + 1)`** —— 开桶。值域多大就开多大，这一行是计数排序的命门：值域 $10^9$ 就要开 $10^9$ 长度，立刻爆内存。
- **`for x in arr: count[x] += 1`** —— 计数。每个值出现几次，count[x] 就是几。**这就是"用值做索引"的体现**：不比较，直接定位。
- **`for v in range(1, max_val + 1): count[v] += count[v - 1]`** —— 前缀和。这一步把 count[v] 从"值 v 出现几次"变成"值 ≤ v 的元素有几个"。**为什么这么做**？因为这样 `count[v] - 1` 就是值 v 的最后一个元素应放的最终位置。
- **`for x in reversed(arr):`** —— 反向遍历。**为什么反向**？保证稳定性——相同值的元素，原数组靠后的在结果里也靠后。
- **`count[x] -= 1; result[count[x]] = x`** —— 先把位置减一再放。**为什么先减再放**？因为 count[x] 是"≤ x 的元素个数"，下标从 0 算，所以最后一个 x 的位置是 `count[x] - 1`，先减一正好对应这个位置。

**关键细节**：

- **前缀和**：`count[v]` 变成"值 ≤ v 的元素个数"，这样 `count[v]-1` 就是值 v 的最后一个元素应放的位置
- **反向填充**：从原数组反向遍历，保证相同值的元素相对顺序不变（稳定性）
- **值域限制**：如果 max_val 是 $10^9$，count 数组要 $10^9$ 大小，内存爆炸

**复杂度**：$O(n + k)$，k 是值域大小。当 $k = O(n)$ 时是 $O(n)$，突破 $O(n\log n)$。当 $k \gg n$ 时退化（值域大不能用）。

**适用条件**：

- 值是整数（或可映射到整数）
- 值域小（$k = O(n)$ 量级）
- 典型场景：考试成绩（0-100）、年龄（0-150）、字符 ASCII（0-127）

**五指标评价**：

| 指标 | 计数排序 | 说明 |
|---|---|---|
| 时间 | $O(n + k)$ | k 是值域 |
| 空间 | $O(n + k)$ | count 数组 + 结果数组 |
| 稳定 | 是 | 前缀和 + 反向填充 |
| 数据前提 | **值域小整数** | 否则空间爆炸 |

### 2. 桶排序（Bucket Sort）

**机制**：数据分布均匀时，把值域分成 k 个桶，每个桶装一段值域的元素。桶内用其他排序（通常插入排序），最后按桶顺序合并。

```python
def bucket_sort(arr, bucket_size=10):
    """桶排序：数据分布均匀时高效。"""
    if not arr:
        return arr
    min_val, max_val = min(arr), max(arr)
    # 桶数量
    bucket_count = (max_val - min_val) // bucket_size + 1
    buckets = [[] for _ in range(bucket_count)]
    
    # 分配到桶
    for x in arr:
        idx = (x - min_val) // bucket_size
        buckets[idx].append(x)
    
    # 桶内排序（用插入排序，桶小所以快）
    result = []
    for bucket in buckets:
        insertion_sort(bucket)   # 桶内数据少，插入快
        result.extend(bucket)
    return result

def insertion_sort(nums):
    for i in range(1, len(nums)):
        key = nums[i]
        j = i - 1
        while j >= 0 and nums[j] > key:
            nums[j + 1] = nums[j]
            j -= 1
        nums[j + 1] = key
```

**逐行讲透**（左神风格）：

- **`min_val, max_val = min(arr), max(arr)`** —— 算值域。桶排序要知道数据范围才能分桶。
- **`bucket_count = (max_val - min_val) // bucket_size + 1`** —— 算桶数。`bucket_size` 是每个桶装的值域宽度，桶数 = 值域 / 桶宽 + 1（防整除丢尾）。
- **`buckets = [[] for _ in range(bucket_count)]`** —— 开桶。注意是 `[[] for _ in ...]` 不是 `[[]] * n`——后者所有桶是同一个 list 引用，会污染数据。
- **`idx = (x - min_val) // bucket_size`** —— 算桶号。`(x - min_val)` 把值平移到从 0 开始，再除以桶宽得到桶号。**这就是"用值域定位"的体现**。
- **`insertion_sort(bucket)`** —— 桶内用插入排序。**为什么不用快排**？桶内元素少（平均 n/k 个），插入排序常数小、对小数据更友好、且稳定。
- **`result.extend(bucket)`** —— 按桶顺序合并。**为什么直接 extend 就对**？因为桶号本身就是有序的——桶 0 装小值，桶 1 装稍大值，依次类推。

**关键细节**：

- **桶数量选择**：桶太多（每个桶 1 个元素）退化成计数；桶太少（1 个桶）退化成桶内排序
- **桶内排序**：桶小（平均 n/k 个元素），用插入排序常数小
- **数据分布假设**：均匀分布时每个桶 n/k 个元素，桶内 $O((n/k)^2)$，共 k 个桶 $O(k \cdot (n/k)^2) = O(n^2/k)$。当 $k \approx n$ 时是 $O(n)$

**复杂度**：

- 平均：$O(n + k)$（均匀分布时）
- 最坏：$O(n^2)$（所有数据挤一个桶，退化成桶内排序）

**适用条件**：

- 数据分布均匀（否则退化）
- 值域可分桶（浮点数也行，按值域等分）
- 典型场景：均匀分布的浮点数、按区间统计的数据

### 3. 基数排序（Radix Sort）

**机制**：数据可按"位"分解时，从最低位到最高位，每一位用稳定排序（通常计数排序）排一次。

```python
def radix_sort(arr):
    """基数排序：整数按位排序，从低位到高位。"""
    if not arr:
        return arr
    max_val = max(arr)
    exp = 1   # 当前位的权重：1, 10, 100, ...
    
    while max_val // exp > 0:
        # 按当前位做计数排序（必须稳定）
        arr = counting_sort_by_digit(arr, exp)
        exp *= 10
    return arr

def counting_sort_by_digit(arr, exp):
    """按第 exp 位做计数排序（稳定）。"""
    count = [0] * 10   # 数字 0-9
    for x in arr:
        digit = (x // exp) % 10
        count[digit] += 1
    
    for i in range(1, 10):
        count[i] += count[i - 1]
    
    result = [0] * len(arr)
    for x in reversed(arr):   # 反向保证稳定
        digit = (x // exp) % 10
        count[digit] -= 1
        result[count[digit]] = x
    return result
```

**逐行讲透**（左神风格）：

- **`max_val = max(arr)`** —— 找最大值。**为什么**？最大值的位数决定要排几轮。
- **`exp = 1`** —— 当前位的权重。个位 exp=1，十位 exp=10，百位 exp=100……
- **`while max_val // exp > 0:`** —— 循环到最高位。**为什么用 `max_val // exp > 0` 而不是位数循环**？这样写不用单独算位数，更紧凑——max_val 决定了几轮，自然终止。
- **`digit = (x // exp) % 10`** —— 取当前位。先除以 exp 把目标位移到个位，再 `% 10` 取出。**这是"按位分解"的体现**。
- **`count = [0] * 10`** —— 开 10 个桶。**为什么是 10**？十进制一位有 0-9 共 10 种值。如果用 256 进制（按字节排 IP），就是 `[0] * 256`。
- **`for x in reversed(arr):`** —— 反向填充。**和计数排序一样**，保证稳定性。
- **`exp *= 10`** —— 进入下一位。**为什么是 `*= 10`**？十进制每高一位权重乘 10。

**关键细节**：

- **从低位到高位**（LSD，Least Significant Digit）：先按个位排，再十位，再百位……高位排序时低位已排好的顺序靠稳定性保持
- **每位用稳定排序**：必须稳定，否则前一位的排序结果被破坏。通常用计数排序（值域 0-9，值域小）
- **轮数 = 最大位数 d**：d 轮计数排序，每轮 $O(n)$，总 $O(dn)$

**复杂度**：$O(d \cdot (n + k))$，d 是最大位数，k 是基数（十进制 k=10）。当 d 是常数（如 32 位整数 d=10）时是 $O(n)$。

**适用条件**：

- 数据可按位分解（整数、定长字符串）
- 位数 d 不能太大（否则 $O(dn)$ 退化）
- 典型场景：身份证号排序、手机号排序、定长字符串排序

**为什么基数排序从低位到高位**？如果从高位到低位（MSD），需要递归处理每个高位分组——复杂且递归深。从低位到高位（LSD）利用稳定性，每轮独立排序，无需递归，实现简单。这是 LSD 基数排序的工程优势。

### 4. 三兄弟横向对比

| 维度 | 计数 | 桶 | 基数 |
|---|---|---|---|
| 定位粒度 | 一个值一个桶 | 一段值域一个桶 | 一位一位定位 |
| 数据前提 | 值域小整数 | 分布均匀 | 可按位分解 |
| 时间 | $O(n+k)$ | $O(n+k)$ 平均 | $O(d(n+k))$ |
| 最坏 | $O(n+k)$ | $O(n^2)$ | $O(d(n+k))$ |
| 空间 | $O(n+k)$ | $O(n+k)$ | $O(n+k)$ |
| 稳定 | 是 | 是（桶内稳定） | **必须稳定** |
| 通用性 | 最差（值域限制） | 中（分布限制） | 中（位分解限制） |

**选型规律**：

- 值域小整数（考试成绩、年龄）→ 计数
- 分布均匀的浮点数 → 桶
- 定长字符串、固定位数整数 → 基数
- 数据特征不明确 → 老实实用 Timsort

## 三、实践与面试：对数器、生产场景、面试手撕

### 1. 面试手撕模板（计数排序 + LSD 基数排序）

```python
def counting_sort(arr, max_val):
    """面试白板版计数排序：值域 [0, max_val]。"""
    count = [0] * (max_val + 1)
    for x in arr:
        count[x] += 1
    for v in range(1, max_val + 1):
        count[v] += count[v - 1]
    result = [0] * len(arr)
    for x in reversed(arr):        # 反向，保稳定
        count[x] -= 1
        result[count[x]] = x
    return result


def radix_sort(arr):
    """面试白板版 LSD 基数排序。"""
    if not arr:
        return arr
    max_val = max(arr)
    exp = 1
    while max_val // exp > 0:
        count = [0] * 10
        for x in arr:
            count[(x // exp) % 10] += 1
        for i in range(1, 10):
            count[i] += count[i - 1]
        result = [0] * len(arr)
        for x in reversed(arr):
            d = (x // exp) % 10
            count[d] -= 1
            result[count[d]] = x
        arr = result
        exp *= 10
    return arr
```

**面试时怎么讲**（按"技术的水有多深"八层面，挑重点讲）：

1. **问题层**（核心思想）：比较排序下界 $\Omega(n\log n)$ 是数学定理，由决策树证明——$n!$ 种排列每次比较只能区分两种，至少 $\log(n!)$ 次。想突破，必须放弃"比较"。非比较排序用"值本身"做下标，把"比较"换成"寻址"，寻址 $O(1)$，所以能到 $O(n)$。
2. **原理层**：计数——值直接做下标，前缀和求位置，反向填充保稳定；桶——值映射到桶号，桶内插排，桶号有序合并；基数——按位分解，LSD 从低到高，每轮稳定计数排序。
3. **优劣层**：平均 $O(n)$、稳定（除桶最坏 $O(n^2)$）；但**有数据前提**——计数要值域小、桶要分布均匀、基数要可按位分解。前提不满足就退化或爆炸，不像 Timsort 通用。
4. **演进层**：计数（Hollerith 1890 人口普查打卡机）→ 桶（数据库直方图分桶）→ LSD 基数（早期卡片分选机）→ MSD 基数（字符串排序，Sedgewick 系列分析）→ 工程上的"按字节基数"（IP/字符串，4 或 8 轮即完）。

**面试加分点**：主动说出三点能立刻拉开和模板党的差距——①"非比较排序是**用数据条件换时间**，比较排序通用、非比较专用"②"基数排序内部必须稳定，所以用计数排序而不是快排"③"生产 99% 用 Timsort，非比较只在数据形态明确时才上（IP、成绩、定长字符串）"。

### 2. 对数器：左神风格的验证方法

左神反复强调：**写完排序不要靠肉眼检查，要用对数器验证**。对数器的思路是——生成随机数据，用你的排序和标准库排序（绝对正确）对比，跑足够多次，只要有一次不一致就说明你的排序有 bug。

```python
import random

def counting_sort(arr, max_val):
    """你的计数排序实现。"""
    count = [0] * (max_val + 1)
    for x in arr:
        count[x] += 1
    for v in range(1, max_val + 1):
        count[v] += count[v - 1]
    result = [0] * len(arr)
    for x in reversed(arr):
        count[x] -= 1
        result[count[x]] = x
    return result

def checker(test_times=5000, max_n=200, max_val=200):
    """对数器：跑 test_times 次，每次随机数据对比标准库。"""
    for _ in range(test_times):
        n = random.randint(0, max_n)            # 随机长度（含 0 和 1）
        arr1 = [random.randint(0, max_val) for _ in range(n)]
        arr2 = arr1[:]                          # 深拷贝
        sorted_mine = counting_sort(arr1, max_val)   # 你的排序
        arr2.sort()                             # 标准库（绝对正确）
        if sorted_mine != arr2:
            print(f"出错！输入: {arr2}, 你的结果: {sorted_mine}")
            return False
    print("对数器验证通过！5000 次随机对照一致。")
    return True

checker()
```

**对数器的价值**：

- **覆盖边界**：随机长度包含 0 和 1（空数组和单元素数组），随机值包含重复值，能覆盖手写时容易遗漏的边界
- **不依赖人工**：跑 5000 次只要几秒，比手动构造测试用例全面得多
- **左神原话**："对数器是算法工程师的基本功。写完一个算法，第一件事就是写对数器，而不是跑几个例子看看对不对。"

**对高手的启发**：对数器思想不止用于排序——任何"有标准答案可对照"的算法都能用对数器。写完红黑树？对数器对比 `std::map`。写完哈希表？对数器对比 `dict`。这是"工程化验证"的思维，比"理论上推导正确"更可靠。

### 3. 教科书做法 vs 生产做法

| 场景 | 教科书做法 | 生产做法 | 原因 |
|---|---|---|---|
| 考试成绩排序 | 手写计数 | 视场景，可计数或 Timsort | 成绩值域 0-100，计数 $O(n+101)$ 比 Timsort $O(n\log n)$ 快 |
| 大数据排序 | 手写基数 | 视场景 | 数据形态决定 |
| 字符串排序 | 手写基数 | `sorted()`（Timsort） | 通用，除非定长且量大 |
| 通用排序 | — | `sorted()` | 99% 场景 Timsort 够用 |
| 海量 IP 排序 | 手写基数 | 基数（IP 是 32 位） | IP 是定长 32 位整数，基数 $O(32n)$ 快 |

**非比较排序的生产价值**在于"特定数据形态下比 Timsort 快几倍"。但绝大多数场景数据形态不明确，Timsort 通用且足够快。**非比较排序是"专用武器"，不是"通用工具"**。

### 4. 三个真实工程坑

**坑一：值域大的整数用计数排序，内存爆炸。**

```python
# 错误：32 位整数用计数排序
arr = [10**9, 1, 10**9 - 1, 2]   # 值域 [1, 10^9]
counting_sort(arr, 10**9)   # count 数组 10^9 大小，4GB 内存
```

症状：内存爆炸或 OOM。根因：值域 $10^9$ 远大于数据量 4，count 数组大小 $10^9$ 浪费。修复：值域大时用基数排序（按位分解，count 数组恒为 10），或用 Timsort。

**坑二：分布极不均匀的数据用桶排序，退化。**

```python
# 错误：[1,2,3,1000000] 用桶排序，所有小数据挤一个桶
arr = [1, 2, 3, 1000000]
bucket_sort(arr, bucket_size=1000)
# 桶 0: [1,2,3]，桶 999: [1000000]
# 桶 0 内插入排序，但如果有大量小数据，桶 0 退化 O(n^2)
```

症状：数据分布不均时退化 $O(n^2)$。根因：桶排序假设均匀分布，不均匀时某些桶过大。修复：①动态调整桶边界（按分位数分桶）②不均匀数据用 Timsort。

**坑三：基数排序内部用了不稳定排序，结果错误。**

```python
# 错误：基数排序内部用快排（不稳定）
def radix_sort_bad(arr):
    exp = 1
    while max(arr) // exp > 0:
        # 错！快排不稳定，前一位的顺序被破坏
        arr = quick_sort_by_digit(arr, exp)
        exp *= 10
    return arr
```

症状：排序结果错误（部分元素顺序乱）。根因：基数排序要求每轮按位排序必须稳定，否则前一位的结果丢失。修复：内部必须用稳定排序（计数或桶）。

### 5. 一个生产级例子：IP 地址排序

```python
def ip_to_int(ip):
    """IP 地址转 32 位整数。"""
    parts = ip.split('.')
    return (int(parts[0]) << 24) + (int(parts[1]) << 16) + \
           (int(parts[2]) << 8) + int(parts[3])

def radix_sort_ip(ips):
    """基数排序排 IP 地址（32 位整数）。"""
    # IP 转 int
    nums = [ip_to_int(ip) for ip in ips]
    # 32 位整数，按字节（8 位）排 4 轮
    for byte_idx in range(4):   # 4 个字节
        shift = byte_idx * 8
        # 按当前字节做计数排序（值域 0-255）
        count = [0] * 256
        for x in nums:
            byte = (x >> shift) & 0xFF
            count[byte] += 1
        for i in range(1, 256):
            count[i] += count[i - 1]
        result = [0] * len(nums)
        for x in reversed(nums):
            byte = (x >> shift) & 0xFF
            count[byte] -= 1
            result[count[byte]] = x
        nums = result
    return nums
```

IP 排序是基数的典型场景：IP 是定长 32 位整数，按字节分 4 轮排序，每轮 $O(n)$，总 $O(4n) = O(n)$。比 Timsort $O(n\log n)$ 快——10 万 IP，Timsort 约 $1.7 \times 10^6$ 次操作，基数 $4 \times 10^5$ 次，快 4 倍。这是"数据形态明确时非比较排序的优势"。

### 6. 高手向：为什么能突破下界，以及 LSD vs MSD、字符串基数排序

**为什么非比较排序能突破 $O(n\log n)$ 下界？**

要回答这个问题，先要厘清"$O(n\log n)$ 下界"的适用范围——它只对**比较排序**成立。决策树模型证明下界的逻辑链是这样的：

1. n 个元素有 $n!$ 种排列，每种排列对应决策树的一个叶子。
2. 决策树是二叉树（每次比较只有"是/否"两种结果）。
3. 高度为 h 的二叉树最多 $2^h$ 个叶子。
4. 要覆盖 $n!$ 种排列，必须 $2^h \geq n!$，即 $h \geq \log_2(n!) = \Theta(n\log n)$。

**关键在第 2 步——"决策树是二叉树"**。这是比较排序的隐含前提：每次比较只区分两种情况。非比较排序打破了这一前提：

- **计数排序**：一次"寻址"能区分 k 种情况（k 是值域），相当于 k 叉决策树，高度降到 $O(n/k)$。
- **基数排序**：按位分解后，每位是 10 叉决策（十进制），d 轮排完，总操作 $O(dn)$。

所以下界没被"违反"，只是**不适用**——非比较排序根本不在比较排序的决策树模型里。这是初学者最容易混淆的点："$O(n\log n)$ 是排序的下界"这句话本身就不严谨，严谨说法是"**比较排序**的下界是 $\Omega(n\log n)$"。

但这个突破是有代价的：**用数据条件换时间**。比较排序对任意可比较数据都成立，非比较排序要求"数据有额外结构可利用"——值域有限、分布均匀、可按位分解。条件不满足就退化甚至不能用。这是"通用 vs 专用"的根本权衡。

**LSD vs MSD：两种基数排序的取舍**。

基数排序有两种实现方向，对应不同的工程取舍：

| 维度 | LSD（从低位到高位） | MSD（从高位到低位） |
|---|---|---|
| 实现 | 每轮独立稳定排序，无需递归 | 高位分组后递归处理每组 |
| 复杂度 | $O(dn)$，d 是位数 | 最坏 $O(dn)$，但短前缀提前结束 |
| 适合数据 | 等长整数/定长字符串 | 变长字符串（前缀不同可提前剪枝） |
| 稳定性要求 | 每轮必须稳定 | 子问题内部分组可不稳定 |
| 缓存友好性 | 顺序扫描，缓存友好 | 递归跳转，缓存不友好 |
| 典型应用 | 整数排序、IP 排序 | 字符串字典序排序 |

**LSD 的工程优势**：实现简单、无需递归、缓存友好，是定长数据的首选。**MSD 的工程优势**：变长数据能提前剪枝（前缀不同的字符串，高一位就分开了），适合字典序。但 MSD 递归深、缓存不友好，工程上常退化成"对短前缀用 LSD"的混合实现。

**字符串基数排序**：基数排序不止排整数，也能排字符串——把每个字符当作一位（基数为字符集大小，如 ASCII 128、Unicode 更大）。关键细节：

```python
def radix_sort_strings(strings, max_len):
    """LSD 基数排序排定长字符串（max_len 是定长）。"""
    # 从最后一位字符排到第一位
    for pos in range(max_len - 1, -1, -1):
        count = [0] * 128   # ASCII
        for s in strings:
            count[ord(s[pos])] += 1
        for i in range(1, 128):
            count[i] += count[i - 1]
        result = [None] * len(strings)
        for s in reversed(strings):
            count[ord(s[pos])] -= 1
            result[count[ord(s[pos])]] = s
        strings = result
    return strings
```

**变长字符串的两个坑**：①定长假设——变长字符串要先补齐到等长（短的补 0 或空格），否则不同长度对不齐。②字符集——ASCII 128 还好，Unicode 字符集大，count 数组大、缓存差，实际工程常退化到 Timsort。Sedgewick 的字符串排序系列对 MSD/LSD 的工程对比有详细分析。

**对高手的启发**：理解了"下界是**对比较排序**成立"和"LSD vs MSD 的取舍"，你对非比较排序的认知就超过了 95% 的候选人。生产中真正用到非比较排序的场景有限（IP、成绩、定长 ID），但知道"什么时候用、用哪种、为什么"，正是区分"会写算法"和"会选算法"的分水岭。

### 自测三问

**问题一：** 非比较排序为什么能突破 $O(n\log n)$ 下界？这个突破是"无条件的"还是"有条件的"？

**参考答案：** $O(n\log n)$ 下界的前提是"基于比较"——决策树模型下每次比较区分两种情况，要区分 $n!$ 种排列至少 $\log(n!) = \Theta(n\log n)$ 次比较。非比较排序不做比较，直接用值做下标（计数）、用值域分桶（桶）、按位分解（基数），把"比较"换成"寻址"。寻址 $O(1)$，所以能突破到 $O(n)$。但这个突破是**有条件的**——计数要求值域小、桶要求分布均匀、基数要求可按位分解。条件不满足时不能用或退化。**非比较排序是"用数据条件换时间"**，不像比较排序那样通用。

**问题二：** 基数排序为什么必须用稳定排序？请用一个具体例子说明，如果内部排序不稳定会怎样。

**参考答案：** 基数排序从低位到高位排序，每轮按一位排序。稳定性保证"前一轮排好的顺序在后一轮中相同键的元素间保持"。例：`[21, 11, 22]` 按个位排（稳定）得 `[21, 11, 22]`（个位 1,1,2，21 和 11 相对顺序保持）。再按十位排（稳定）得 `[11, 21, 22]`。如果第二轮不稳定，21 和 11 的十位都是 1，可能变成 `[11, 21]` 也可能 `[21, 11]`——结果不确定，可能错。**稳定性对基数排序不是特性，是正确性的前提**。这就是为什么基数内部通常用计数排序（稳定）——不能用快排（不稳定）。

**问题三：** 生产里什么场景该用非比较排序？什么场景该老老实实用 Timsort？请给一个判断框架。

**参考答案：** 判断框架：①数据是整数且值域小（如成绩 0-100）→ 计数，比 Timsort 快几倍。②数据是定长字符串或固定位数整数（如 IP、身份证号）→ 基数。③数据分布均匀的浮点数 → 桶。④数据特征不明确或值域大、分布不均 → Timsort，不要勉强用非比较。**核心原则**：非比较排序是"专用武器"，只在数据形态明确时用；Timsort 是"通用工具"，99% 场景够用。强行用非比较排序处理不满足条件的数据，会退化甚至爆炸（值域大的计数内存爆、分布不均的桶退化 $O(n^2)$）。**判断依据是"数据前提是否满足"——满足则用非比较赚速度，不满足则用 Timsort 求通用**。

### 算法思想 × 生活迁移

非比较排序的核心思想是"**用数据的额外信息换时间**"——不靠两两比较，靠数据本身的特性定位。

**迁移一：用"索引"代替"比较"做分类。** 图书馆按"中图法分类号"摆书，不是两两比较哪本该在前——每本书有分类号，直接按号定位到书架。这是计数排序的思路：用值（分类号）做索引，$O(1)$ 定位，比"两两比较"高效。前提是"每本书都有明确的分类号"——如果书没有分类号，只能两两比较内容（比较排序），慢得多。

**迁移二：按"维度"分层处理。** 基数排序按位分解，从低位到高位逐位排序。生活里"多维度决策"也用类似思路：先按最不重要的维度筛一遍，再按重要维度排。比如选房子：先按"是否在预算内"筛（最低位），再按"地段"排（中间位），最后按"房型"精排（最高位）。每一层都保留前一层的顺序（稳定性），最终得到综合排序。这种"分层决策"比"一次性多维度比较"高效。

**迁移三：识别"专用工具"的适用边界。** 非比较排序是专用武器，只在数据形态满足时用。生活里很多"专用工具"也是这样：电钻比螺丝刀快，但只适合拧螺丝，不适合拧螺母。判断"该用专用工具还是通用工具"，看"任务是否满足专用工具的前提"——满足则专用赚效率，不满足则通用求稳妥。这是"按场景选工具"的判断力，是算法思维向工具选型的迁移。

**为什么这些迁移成立：** 非比较排序成立的前提是"数据有额外信息可用（值域/分布/位结构）"。这个前提在图书分类、多维度决策、工具选型里都成立——只要问题有"额外结构"可利用，就能用"专用方法"赚效率。迁移成立的关键是"识别问题有没有额外结构"——有则用专用方法，无则用通用方法。**判断"问题是否有额外结构可利用"，是非比较排序思想能迁移成功的关键。**
