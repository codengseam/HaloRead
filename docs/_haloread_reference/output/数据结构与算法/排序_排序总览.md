# 算法课·03｜排序总览：评价一个排序算法好坏的五个指标

> 前置知识：学完第 01 章（为什么要学算法）和第 02 章（二分查找），理解 $O(n\log n)$ 的含义
> 学完你能：①讲清"为什么要排序"和"排序的本质是什么" ②用五个指标评价任何一个排序算法 ③理解"比较排序下界 $O(n\log n)$"这个数学定理 ④对后续每章排序算法建立统一的评价框架

## 一、问题与思路：为什么人类需要排序，以及"排序"到底是什么

### 1. 排序的真实目的：不是整齐，是查找

人类对"顺序"的需求比计算机早几千年。

书店按类别摆放，方便找书；药铺按药性分类，方便抓药；账本按日期排列，方便对账。这些场景的共同点是：**排序不是为了好看，是为了"快速定位"。**

第 02 章讲过，二分查找能在一个有序数组里 $O(\log n)$ 定位目标——前提是数组有序。10 亿个元素，$\log_2(10^9) \approx 30$，30 次比较就能找到。但如果数组无序，只能线性扫描 $O(n)$，10 亿次比较。差了 3000 万倍。

所以排序的本质目的，用一句话讲：**把数据从"只能线性查找"的状态，变成"可以二分查找"的状态。** 排序是几乎所有高效查找、统计、聚合算法的前置步骤。数据库索引、搜索引擎倒排表、文件系统目录树——底层都是"先排序，再快速定位"。

这个本质很重要，因为它解释了为什么计算机科学在 20 世纪 60 年代集中爆发了一批排序算法：1959 年 Shell 排序、1962 年 Hoare 的快速排序、1964 年 Williams 的堆排序。那个年代计算机刚普及，数据量开始变大，"怎么快速把数据排好序"是真实工程刚需。排序算法的演进史，就是"人类怎么把一件事越做越快"的缩影。

### 2. 排序的严格定义

数学上，排序是这样定义的：

> 给定 n 个元素 $a_1, a_2, \dots, a_n$，和一个**全序关系** $\le$（满足自反、反对称、传递），输出一个排列 $\pi$，使得 $a_{\pi(1)} \le a_{\pi(2)} \le \dots \le a_{\pi(n)}$。

有几个关键词：

- **全序关系**：任意两个元素都能比较出大小。这意味着排序的前提是"可比较"。如果元素之间只满足偏序（部分可比较），严格意义上不能排序，只能做拓扑排序（这个后面图论章讲）。
- **排列 $\pi$**：排序结果是原数组元素的一个重新排列，不是新元素。
- **非降序**：注意定义里是 $\le$ 不是 $<$，允许相等元素。相等元素在排序前后的相对位置怎么处理，引出后面要讲的"稳定性"。

### 3. 一个被忽视的核心：比较排序的下界

这一节是整个排序系列的认知地基。

定理：**任何基于"比较"的排序算法，最坏情况下的时间复杂度下界是 $\Omega(n\log n)$。** 不可能更快。

证明用决策树模型。n 个元素有 $n!$ 种排列。比较排序的每次比较，相当于在决策树上走一条边，把可能性分成两堆。要区分 $n!$ 种排列，决策树至少要有 $n!$ 个叶子。一棵有 L 个叶子的二叉树，高度至少是 $\lceil \log_2 L \rceil$。所以最坏情况至少要比较 $\lceil \log_2(n!) \rceil$ 次。

由 Stirling 公式：

$$\log_2(n!) = n\log_2 n - n\log_2 e + O(\log n) = \Theta(n\log n)$$

所以任何比较排序最坏至少 $\Omega(n\log n)$ 次比较。

**这个定理的意义**：

1. 它解释了为什么快排、归并、堆排都"恰好"是 $O(n\log n)$——不是巧合，是它们都逼近了下界，是"最优"的比较排序。
2. 它告诉你不要再试图发明一个 $O(n)$ 的比较排序算法，**不可能**。这条路走不通。
3. 它指出了突破方向：**想突破 $O(n\log n)$，只能放弃"比较"**，利用数据的额外信息（值域有限、可分配到桶等）。这就是计数排序、基数排序、桶排序的思路，它们能做到 $O(n)$，但前提是数据满足特定条件。

记住这个下界。后面每讲一个 $O(n\log n)$ 的算法，你都要意识到：它已经接近理论极限了。每讲一个 $O(n^2)$ 的算法，你都要意识到：它是直觉做法，还没突破。每讲一个 $O(n)$ 的非比较排序，你都要意识到：它绕开了比较，所以不受这个下界约束。

## 二、原理与实现：评价一个排序算法的五个指标

很多人学排序，学完十个算法还是选不出来该用哪个，因为脑子里没有"评价框架"。这一节建立这个框架——**五个指标**。后续每讲一个算法，都用这五个指标打分。

### 指标 1：时间复杂度（平均 + 最坏 + 最好）

这是最直观的指标，但也是最容易被简化的。

很多人只记"快排是 $O(n\log n)$"，这不够。完整的评价要记三个：

- **最好情况**（Best Case）：数据最理想时多快
- **平均情况**（Average Case）：随机数据多快
- **最坏情况**（Worst Case）：数据最恶劣时多快

为什么三个都要记？因为工程里数据千奇百怪，你不知道拿到的是哪种。快排平均 $O(n\log n)$ 但最坏 $O(n^2)$，工程上"最坏"是真实风险（用户数据恰好有序就是最坏）。归并和堆排最坏也是 $O(n\log n)$，更稳。

**复杂度的工程直觉**：$1$ 秒内能跑多少数据量？按现代 CPU 每秒约 $10^8$ 次操作算：

| 复杂度 | 1 秒能处理的数据量 |
|---|---|
| $O(n)$ | $10^8$（1 亿） |
| $O(n\log n)$ | $4 \times 10^6$（400 万） |
| $O(n^2)$ | $10^4$（1 万） |
| $O(n^3)$ | $460$（500） |

看到这个表你就明白：$O(n^2)$ 算法在 n=10 万的数据上跑不动（要 100 秒），$O(n\log n)$ 能轻松跑到百万级。这就是为什么排序算法的演进主线是"从 $O(n^2)$ 突破到 $O(n\log n)$"。

### 指标 2：空间复杂度

排序需要多少额外内存？

- **原地排序**（In-place）：$O(1)$ 额外空间，只用了几个临时变量
- **非原地**：需要 $O(n)$ 或更多额外空间

原地排序对内存敏感的场景重要（嵌入式设备、大数据集）。快排是原地的（只用了 $O(\log n)$ 递归栈），归并不是（合并需要 $O(n)$ 辅助数组）。

注意一个细节：递归调用栈也算空间。快排平均递归深度 $\log n$，所以严格说是 $O(\log n)$ 空间，不是 $O(1)$。但比起归并的 $O(n)$，快排已经算"原地"了。

### 指标 3：稳定性（Stability）

**稳定性**：如果两个元素相等，排序后它们的相对顺序不变。

举例：`[(Alice, 90), (Bob, 90), (Carol, 85)]` 按分数排序。稳定排序保证 Alice 还在 Bob 前面（因为原数组里 Alice 在前）。不稳定排序可能让 Bob 跑到 Alice 前面。

为什么稳定性重要？因为工程里常做"多关键字排序"。比如先按分数排，再按姓名排。如果排序不稳定，第二趟会打乱第一趟的结果。稳定排序才能保证"先按主键排、再按次键排"得到"先按次键、相同次键按主键"的正确结果。

主流排序的稳定性：

| 算法 | 稳定 | 原因 |
|---|---|---|
| 冒泡 | 是 | 相等元素不交换 |
| 选择 | 否 | 跨距离交换破坏相对顺序 |
| 插入 | 是 | 相等元素不后移 |
| 希尔 | 否 | 跨距离交换 |
| 快排 | 否 | 分区时跨距离交换 |
| 归并 | 是 | 合并时左半优先 |
| 堆排 | 否 | 建堆和取顶时打乱 |
| Timsort | 是 | 归并的变体 |
| 计数/桶/基数 | 是 | 按位分配天然稳定 |

### 指标 4：是否原地 + 是否自适应

这两个子指标常被忽略，但对工程选型很关键。

- **原地**（In-place）：是否需要额外 $O(n)$ 空间
- **自适应**（Adaptive）：对"基本有序"的数据是否能更快

自适应是个很有价值的能力。真实数据往往不是完全随机的——日志按时间近有序、商品按价格近有序。**自适应排序**能识别这种"部分有序"并加速，最坏 $O(n\log n)$，最好 $O(n)$。Timsort 就是自适应的典范，这也是它成为 Python/Java 默认排序的原因。

插入排序是自适应的（基本有序数据 $O(n)$），快排不是（无论数据多有序，仍然 $O(n\log n)$，甚至有序数据触发最坏 $O(n^2)$）。

### 指标 5：常数因子与缓存友好性

这是工程里"看不见但决定性能"的指标。

两个算法都是 $O(n\log n)$，但常数差几倍，实际性能差几倍。影响常数的因素：

- **比较代价**：比较两个简单整数很快；比较两个复杂对象（多字段、需调函数）慢。归并的比较次数固定，快排的比较次数约是归并的 1.4 倍——但快排的分区是顺序扫描，缓存命中率高，实际往往比归并快。
- **交换代价**：交换两个整数快，交换两个大对象慢。
- **缓存友好性**：顺序访问数组（快排分区）缓存命中率高；跳着访问（堆排调整堆，父子节点跨度大）缓存不友好。
- **递归调用开销**：每次函数调用有压栈开销，递归深度大时代价显著。

这就是为什么堆排虽然最坏 $O(n\log n)$ 比快排好，但实际跑起来快排更快——快排常数小、缓存友好。$O$ 符号隐藏了常数，工程选型时不能只看 $O$。

### 五指标汇总表（后续每章会用它打分）

| 算法 | 平均 | 最坏 | 空间 | 稳定 | 自适应 | 缓存友好 |
|---|---|---|---|---|---|---|
| 冒泡 | $O(n^2)$ | $O(n^2)$ | $O(1)$ | 是 | 是（加判断） | 好 |
| 选择 | $O(n^2)$ | $O(n^2)$ | $O(1)$ | 否 | 否 | 好 |
| 插入 | $O(n^2)$ | $O(n^2)$ | $O(1)$ | 是 | 是 | 极好 |
| 希尔 | $O(n^{1.3})$ | $O(n^2)$ | $O(1)$ | 否 | 否 | 好 |
| 快排 | $O(n\log n)$ | $O(n^2)$ | $O(\log n)$ | 否 | 否 | 极好 |
| 归并 | $O(n\log n)$ | $O(n\log n)$ | $O(n)$ | 是 | 否 | 中 |
| 堆排 | $O(n\log n)$ | $O(n\log n)$ | $O(1)$ | 否 | 否 | 差 |

## 三、实践与面试：怎么选排序算法

### 1. 教科书做法 vs 生产做法

| 场景 | 教科书做法 | 生产做法 | 原因 |
|---|---|---|---|
| 通用排序 | 手写快排 | `sorted()` / `list.sort()`（Timsort） | Timsort 综合最优，且稳定 |
| 小数据排序（n<50） | 手写插入 | `sorted()`（Timsort 内部对小子数组用插入） | 插入排序在小数据上常数小 |
| Top-K 问题 | 手写堆 | `heapq.nlargest` / `heapq.nsmallest` | 堆是 Top-K 的最优解 |
| 大数据外部排序 | 手写归并 | 框架（Spark/MapReduce 的 sort 阶段） | 分布式归并是工程问题 |
| 整数排序（值域小） | 手写计数 | 按需，Python 无内置 | 值域小时计数比 Timsort 快 |
| 对象多关键字排序 | 重写比较 | `key=lambda x: (主键, 次键)` | 声明式更不易错，且 Timsort 稳定 |

**生产原则：99% 的场景用 `sorted()`。** 原因：

1. Timsort 是混合算法，自适应、稳定、最坏 $O(n\log n)$，综合最优
2. 标准库是 C 实现的，比你手写快几倍
3. 不会写错边界（手写快排最容易写错分区）

**混合算法的工程智慧（Timsort 与 introsort）：** Timsort（Python/Java 默认）以归并为主，利用数据中已有的"run"段做到自适应；C++ 标准库的 `std::sort` 用的是 introsort（内省排序）——以快排为主，递归深度超过 $2\log n$ 时切到堆排以防最坏 $O(n^2)$，小子数组切到插入排序。两者思路一致：用混合算法把"平均快"和"最坏稳"都拿到手，单一算法很难两全。这也是为什么现代语言标准库不再用单一排序——工程选型走到极致，必然走向混合。

**什么时候必须手写**：

- 面试白板（无奈）
- 非标准排序（如"按自定义规则排序"，库不支持）
- 极端性能优化（如嵌入式、值域小的整数，计数排序快）
- 教学需要理解原理

### 2. 面试视角：常考的排序对比题

面试里排序题大致三类，对应不同的考察点：

- **对比题**：快排为什么平均快但最坏慢？堆排为什么最坏稳却用得少？归并为什么稳定却不原地？——考的是五指标框架，能用"常数/缓存/稳定性"讲清就过关。
- **手写题**：白板写快排的 partition、归并的 merge、堆的 sift-down。重点考边界（pivot 选点、递归终止、循环不变量）。
- **变体题**：Top-K（堆或快排分区）、合并 K 个有序链表（堆）、区间合并（按起点排序）、荷兰国旗（三路分区）。这些题排序只是工具，考的是把排序嵌入更大问题的能力。

记住一条：面试官问"为什么用 A 不用 B"，永远从五指标里挑维度回答，不要只说"快"。

### 3. 三个真实工程坑（贯穿后续章节）

**坑一：在有序数据上用快排触发最坏情况。** 手写快排取端点作 pivot，对有序数据每次只分出一个元素，退化 $O(n^2)$。修复：用 `sorted()`，或手写时随机化 pivot。

**坑二：对象排序丢失稳定性。** 用不稳定排序（快排/堆排）排对象，同键元素相对顺序被打乱。修复：用稳定排序（Timsort），或把次要键加进排序键。

**坑三：`key` vs `cmp` 的性能陷阱。** `key=` 对每个元素调用一次 $O(n)$，`cmp_to_key` 对每对调用 $O(n\log n)$。数据量大时差几倍。永远用 `key=`。

这三个坑在后续每章会结合具体算法展开。

### 4. 一个生产级例子：选择合适的排序

```python
import heapq

# 场景 1：通用排序——直接用 sorted
data = [3, 1, 4, 1, 5, 9, 2, 6]
sorted_data = sorted(data)   # Timsort

# 场景 2：Top-K——用堆，不要全排序
top5 = heapq.nlargest(5, range(10**6))   # O(n log k) 而不是 O(n log n)

# 场景 3：多关键字排序——用 key 元组
students = [("Alice", 90), ("Bob", 90), ("Carol", 85)]
# 先按分数降序，分数相同按姓名升序
sorted_students = sorted(students, key=lambda x: (-x[1], x[0]))
# [('Alice', 90), ('Bob', 90), ('Carol', 85)]

# 场景 4：值域小的整数——计数排序更快
def counting_sort(arr, max_val):
    count = [0] * (max_val + 1)
    for x in arr:
        count[x] += 1
    result = []
    for v in range(max_val + 1):
        result.extend([v] * count[v])
    return result

scores = [85, 90, 78, 90, 65, 100]   # 分数 0-100，值域小
sorted_scores = counting_sort(scores, 100)   # O(n+k)，k=101
```

每个场景选不同的算法，是因为数据特征不同。这就是"按数据状况选算法"——左程云反复强调的工程视角。

## 四、速查与自测

### 速查表：五指标评价框架

| 指标 | 关注什么 | 工程意义 |
|---|---|---|
| 时间复杂度（最好/平均/最坏） | 三种情况都要看 | 最坏决定下限保障 |
| 空间复杂度 | 原地 vs 非原地 | 内存敏感场景 |
| 稳定性 | 相等元素是否保序 | 多关键字排序必需 |
| 自适应性 | 部分有序数据是否更快 | 真实数据常部分有序 |
| 常数与缓存 | 实际速度 | $O$ 之外的性能决定因素 |

### 自测三问

**问题一：** 为什么"比较排序下界是 $\Omega(n\log n)$"这个定理重要？它对算法设计有什么指导意义？

**参考答案：** 它告诉你三件事：①任何 $O(n\log n)$ 的比较排序（快排/归并/堆排）都已经逼近理论最优，不要再试图优化到 $O(n)$；②想突破 $O(n\log n)$ 只能放弃"比较"，改用数据额外信息（计数/基数/桶）；③$O(n^2)$ 的算法（冒泡/选择/插入）是直觉做法，没突破下界，只在小数据或特殊数据下有用。这个定理是整个排序系列的认知地基，没有它你不知道哪些算法"已经够好了"、哪些"还有优化空间"。

**问题二：** 稳定性是什么？为什么工程里常需要稳定排序？请用一个具体场景说明。

**参考答案：** 稳定性指"相等元素排序后相对顺序不变"。场景：电商商品列表，要先按销量排序，销量相同按评分排序。做法是先按评分排（次键），再按销量排（主键）。如果第二次排序不稳定，销量相同的商品里评分高的可能被排到评分低的后面——第一趟排序的结果被破坏。稳定排序才能保证"先排次键再排主键"得到"先按主键、相同主键按次键"的正确结果。Python 的 `sorted()` 用 Timsort 是稳定的，所以这种"两趟排序"在生产里安全。

**问题三：** 快排和堆排都是 $O(n\log n)$，为什么生产默认用 Timsort 而不是堆排？堆排最坏更优啊。

**参考答案：** 三个原因。①堆排不稳定——对象排序会打乱同键元素顺序，是 bug 源。②堆排常数大——堆调整是"父子节点跳着访问"，跨度大，CPU 缓存不友好；Timsort 顺序扫描，缓存命中率高。③真实数据常"部分有序"，Timsort 自适应能接近 $O(n)$，堆排无论如何都是 $O(n\log n)$。堆排的优势在"最坏情况确定且原地"，定位是"Top-K 和优先队列"而不是"通用排序"。

### 算法思想 × 生活迁移

排序总览这一章的核心思想是"**建立评价框架**"——不急着评价具体算法，先把"用什么标准评价"想清楚。这个思想在生活里非常有用。

**迁移一：买东西先列指标，再看产品。** 买手机前，先列五个指标（性能、续航、拍照、价格、生态），再看具体型号在这五个指标上的得分。不列指标直接看型号，会被营销带跑——销售只强调强项，回避弱项。先有框架再选，才能避免"看完每个都觉得好"。

**迁移二：评价一个人，先定维度。** 招聘时先定评价维度（技术深度、沟通、责任心、学习能力、文化匹配），每个维度打分，再综合。不定维度直接凭感觉，会被"面试表现好"或"学历好"等单一维度主导。框架先行，是避免"以偏概全"的根本方法。

**迁移三：做技术选型先列约束，再看方案。** 选数据库时，先列约束指标（一致性要求、读写吞吐、延迟、成本、团队熟悉度），再对比 MySQL/Redis/MongoDB 在每个指标上的表现。不列约束直接选，会被"哪个方案流行"或"我用过"主导。先有约束框架再选型，才能避免"哪个火用哪个"的盲目。

**为什么这些迁移成立：** 排序算法的"五指标评价"成立的前提是"评价对象是多维度的、单一指标无法刻画全貌"。这个前提在产品选择、人才评价、方案对比里都成立。迁移成立的关键是"识别出评价对象是多维的"——一旦识别出来，自然需要先建框架再评价。这是"算法思维→决策思维"的迁移：**算法训练的"多维评价能力"，本质是"拒绝单一指标简化"的判断力。**
