# 算法课·06｜快速排序：分治思想的典范

> 前置知识：学完第 03-05 章（排序总览、简单排序、希尔排序），理解"逆序对"和"分治"
> 学完你能：①给纯小白讲清快排"找基准分队"的本质 ②默写 Lomuto/Hoare/三路三种分区并讲透每一行 ③手撕随机化快排应对面试 ④讲清快排平均 $O(n\log n)$ 的递推证明、最坏 $O(n^2)$ 的陷阱 ⑤知道 introsort 为什么诞生、为什么 C++ `std::sort` 内部不只纯快排 ⑥用对数器验证自己写的快排对不对

## 一、问题与思路：分治——把大问题拆成同构小问题

### 1. 一个生活场景：操场按身高排队

体育课，老师让 30 个学生按身高从矮到高排队。

**笨办法（冒泡式）**：两两比较相邻的，高的往后换。一轮下来最高的人到末尾，重复 29 轮。要比较 400 多次，学生累老师也累。

**聪明办法（快排式）**：老师随便拉一个同学站出来当"基准"，喊一声"比我矮的站左边，比我高的站右边"。这一喊，30 个人快速分成两堆，被拉出来的同学位置就定了——他左边都比他矮，右边都比他高，他不用再动了。然后左右两堆各自再拉一个同学出来继续分。几轮下来，队伍就排好了。

**为什么比冒泡快那么多？** 冒泡是"相邻两人比较交换"，每次只解决一个逆序对。快排是"一次分队，左半和右半之间的所有逆序对全部解决"——批量消除。30 个人，冒泡要 400+ 次比较，快排只要约 100 次。

**这就是快排的本质——找一个基准，把问题拆成两个独立小问题。**

### 2. 从希尔到快排：分治思想的成熟

上一章讲希尔排序时说，它的突破在于"跨步长消除逆序对"。但希尔仍是"在原数组上做分组插入"，没有真正"拆"问题。

1962 年，C. A. R. Hoare 在莫斯科大学交流时发明了快速排序（Quick Sort），把"分治"思想用到了极致。分治的精髓是：**不要一次处理整个数组，把它真正拆成两个独立子问题，分别解决，最后合并。**

快排的思路极其简洁：

1. 选一个元素作 pivot（基准）
2. 把小于 pivot 的放左边，大于 pivot 的放右边（这一步叫"分区" partition）
3. 对左右两部分分别递归快排

**关键洞察**：分区这一步，pivot 一次性"找到了自己的最终位置"——左边都比它小，右边都比它大，pivot 不用再动了。这就像操场上老师拉出来的那个同学，他站的位置就是他最终的排队位置，不用再动。

### 3. 核心思想：快排为什么快

快排为什么能 $O(n\log n)$？核心在于分区这一步的"批量消除逆序对"。

考虑分区：选 pivot，把小于它的放左、大于它的放右。这一步做完，**左半所有元素和右半所有元素之间的逆序对一次性全消除了**——左半任意元素都小于右半任意元素，它们之间不再有逆序对。

对比插入排序：每次操作消除 1 个逆序对。对比希尔：每次操作消除 1 个跨步长逆序对。快排分区：一次操作消除"半个数组 × 半个数组"个逆序对——批量消除，量级是 $O(n^2)$ 个逆序对用 $O(n)$ 次操作消除。

这就是分治能突破 $O(n^2)$ 的本质——**不是消除更多逆序对，是用"分区"批量消除跨区间逆序对**。后面讲归并时你会看到，归并的"合并"也是批量消除——但合并是把两个有序数组的逆序对一次消除，思路与快排对称。

### 4. 一个被忽视的本质：快排是"原地分治"

很多人没注意到快排和归并的一个本质差别：

- **归并**：先拆（拆是 $O(1)$，直接切两半），后合（合是 $O(n)$，要合并两个有序数组）→ "难在合"
- **快排**：先分（分是 $O(n)$，要把小于 pivot 的挪到左边），后不需要合（分完后左右各自排好就完事）→ "难在分"

快排是"**先苦后甜**"——分区时辛苦（$O(n)$ 工作量），但分完之后两边独立排好就结束，不需要合并步骤。归并是"**先甜后苦**"——拆分时轻松（直接切两半），但合并时辛苦（$O(n)$ 合并两个有序数组）。

这个差别决定了两者的空间特性：快排是原地的（分区在原数组上做，不需要额外空间），归并不是（合并需要 $O(n)$ 辅助数组）。也决定了稳定性：快排分区时跨距离交换，不稳定；归并合并时左半优先，稳定。

## 二、原理与实现：分区是快排的灵魂

### 1. Lomuto 分区：最容易理解的版本

```python
def quick_sort(nums, lo, hi):
    if lo >= hi:
        return
    pivot_idx = partition(nums, lo, hi)
    quick_sort(nums, lo, pivot_idx - 1)
    quick_sort(nums, pivot_idx + 1, hi)

def partition(nums, lo, hi):
    pivot = nums[hi]   # 取最后一个作 pivot
    i = lo             # i 指向"小于 pivot 区"的右边界
    for j in range(lo, hi):
        if nums[j] < pivot:
            nums[i], nums[j] = nums[j], nums[i]
            i += 1
    nums[i], nums[hi] = nums[hi], nums[i]   # pivot 放到正确位置
    return i
```

**逐行讲透**（左神风格，每一行都不能含糊）：

- **`if lo >= hi: return`**：递归终止。`lo == hi` 是单元素天然有序，`lo > hi` 是空区间。这行不写会无限递归，栈溢出。
- **`pivot = nums[hi]`**：取最后一个作 pivot。为什么取最后一个？Lomuto 的实现简化——从前往后扫，把小的挪到前面，最后把 pivot 放到中间。取端点是为了代码简洁，但有最坏情况风险（有序数据）。
- **`pivot_idx = partition(...)`**：分区，返回 pivot 的最终位置。这个位置已经就位，不用再动。
- **`quick_sort(nums, lo, pivot_idx - 1)`**：递归排左半（小于 pivot 的部分）。
- **`quick_sort(nums, pivot_idx + 1, hi)`**：递归排右半（大于等于 pivot 的部分）。注意 `pivot_idx` 本身不用排，它已经就位。
- **`i = lo`**：i 维护"小于 pivot 区"的右边界。初始指向 lo，意味着"小于 pivot 区"暂时为空。
- **`for j in range(lo, hi)`**：j 从 lo 扫到 hi-1（不含 hi，因为 hi 是 pivot 自己）。
- **`if nums[j] < pivot:`**：用严格小于。为什么不用 `<=`？因为相等元素不挪到"小于区"，留在原地——但这并不能保证稳定性（因为交换是跨距离的）。
- **`nums[i], nums[j] = nums[j], nums[i]`**：把小于 pivot 的元素挪到 i 位置。注意这是跨距离交换，破坏稳定性。
- **`i += 1`**："小于 pivot 区"扩张一格。
- **`nums[i], nums[hi] = nums[hi], nums[i]`**：扫描完，i 指向"第一个不小于 pivot 的位置"，把 pivot 放到这里，pivot 就到了最终位置。
- **`return i`**：返回 pivot 的最终位置，递归时左右两半排除这个位置。

**i 的语义是关键**：始终指向"下一个小于 pivot 的元素该放的位置"。扫描完，`[lo, i-1]` 都 < pivot，`[i+1, hi]` 都 >= pivot，i 是 pivot 的家。

**用一个例子走一遍**。数组 `[3, 6, 8, 10, 1, 2, 1]`，pivot = nums[6] = 1：

```
初始：[3, 6, 8, 10, 1, 2, 1]  i=0, j=0
j=0: nums[0]=3 >= 1，跳过
j=1: nums[1]=6 >= 1，跳过
j=2: nums[2]=8 >= 1，跳过
j=3: nums[3]=10 >= 1，跳过
j=4: nums[4]=1 >= 1（用<不用<=），跳过
j=5: nums[5]=2 >= 1，跳过
扫描完，i=0，交换 nums[0] 和 nums[6]：[1, 6, 8, 10, 1, 2, 3]
返回 i=0
```

pivot=1 到了位置 0，左边空（没有比 1 小的），右边 `[6, 8, 10, 1, 2, 3]` 都 >= 1。正确。

### 2. Hoare 分区：原始版本，更快

Hoare 1962 年原始论文用的分区是"双指针从两端往中间走"，比 Lomuto 快（交换次数少）：

```python
def partition_hoare(nums, lo, hi):
    pivot = nums[lo]   # 取第一个作 pivot
    i, j = lo - 1, hi + 1
    while True:
        i += 1
        while nums[i] < pivot:
            i += 1
        j -= 1
        while nums[j] > pivot:
            j -= 1
        if i >= j:
            return j   # 注意返回 j 不是 i
        nums[i], nums[j] = nums[j], nums[i]
```

**逐行讲透**：

- **`pivot = nums[lo]`**：取第一个作 pivot。Hoare 取首端，Lomuto 取末端，这是两者最直观的差别。
- **`i, j = lo - 1, hi + 1`**：i 和 j 初始化在边界外。为什么不在边界内？因为循环里先 `i += 1` / `j -= 1` 再判断，这样 i 和 j 第一次进入循环就指向 lo 和 hi。这是 do-while 风格的 Python 写法。
- **`while True:`**：无限循环，靠内部 `return` 退出。
- **`i += 1`**：i 先右移一格，再开始判断。即使本次循环没找到该停的，下一轮 i 也会先右移。
- **`while nums[i] < pivot: i += 1`**：i 从左往右找"第一个 >= pivot 的"。找到就停。
- **`j -= 1`**：j 先左移一格，再开始判断。
- **`while nums[j] > pivot: j -= 1`**：j 从右往左找"第一个 <= pivot 的"。找到就停。
- **`if i >= j: return j`**：i 和 j 相遇或交叉，分区完成。**注意返回 j 不是 i**——这是 Hoare 分区最容易写错的地方。返回 j 是因为 `[lo, j]` 都 <= pivot，`[j+1, hi]` 都 >= pivot，递归时用 `quick_sort(lo, j)` 和 `quick_sort(j+1, hi)`，注意 j 那个元素还要继续参与递归（不像 Lomuto 的 pivot 已经就位）。
- **`nums[i], nums[j] = nums[j], nums[i]`**：i 指向大的、j 指向小的，交换它们。一次交换解决两个"错位"。

**Hoare vs Lomuto 的关键差别**：

| 维度 | Lomuto | Hoare |
|---|---|---|
| pivot 位置 | 末端 | 首端 |
| 交换次数 | 多（每个小于 pivot 的都交换） | 少（只在"确实错位"时交换） |
| 返回值 | pivot 最终位置（已就位） | 分区边界（pivot 未必然就位） |
| 递归范围 | 排除 pivot：`(lo, p-1)` 和 `(p+1, hi)` | 包含边界：`(lo, j)` 和 `(j+1, hi)` |
| 代码难度 | 易 | 难（边界条件多） |

工程实现常用 Hoare（更快），教学常用 Lomuto（更易懂）。两者复杂度一样。

### 3. 复杂度分析：为什么平均 O(n log n)、最坏 O(n²)

**平均情况**：假设每次 pivot 把数组分成"近似两半"（比如 1:9 也算近似）。每层分区是 $O(n)$，递归深度 $O(\log n)$，总 $O(n\log n)$。

**最坏情况**：每次 pivot 都选到最值（最大或最小），分区后一边是 n-1 个元素、一边是 0 个。递归深度退化到 $O(n)$，每层 $O(n)$，总 $O(n^2)$。

什么时候触发最坏？**数组已经有序（升序或降序），且 pivot 取端点**。这是工程上的真实风险——很多数据天然有序（日志按时间、商品按价格），手写快排取端点 pivot 会直接退化 $O(n^2)$。

**复杂度的数学推导**（平均情况，高手向）：

设 $T(n)$ 为排序 n 个元素的平均时间。pivot 随机选，落在位置 k 的概率均等（$k = 0, 1, \dots, n-1$）。分区是 $O(n)$（扫描一遍），递归左右两部分：

$$T(n) = n + 1 + \frac{1}{n} \sum_{k=0}^{n-1} (T(k) + T(n-1-k))$$

化简（左右对称，$\sum_{k=0}^{n-1} T(k) = \sum_{k=0}^{n-1} T(n-1-k)$）：

$$T(n) = n + 1 + \frac{2}{n} \sum_{k=0}^{n-1} T(k)$$

用代入法证明 $T(n) = O(n\log n)$：假设 $T(k) \leq c \cdot k \log k$ 对所有 $k < n$ 成立，代入递推式，经过化简可得上界 $T(n) \leq c' \cdot n \log n$。证明的关键直觉是"随机 pivot 让递归深度期望 $\log n$"——每次随机选 pivot，期望分到 1:1，递归深度期望 $\log n$。

**对高手的启发**：快排的平均复杂度证明比最坏复杂度证明难得多。最坏情况是确定的（每次选最值），平均情况要算期望（所有 pivot 选择的加权平均）。工程上我们关心平均情况，因为随机化后最坏概率极低——这是"工程思维和数学思维的差别"。

### 4. 随机化 pivot：工程上的标准做法

避免最坏情况的标准做法是**随机化 pivot**：

```python
import random

def quick_sort_randomized(nums, lo, hi):
    if lo >= hi:
        return
    # 随机选一个 pivot，换到 hi 位置
    rand_idx = random.randint(lo, hi)
    nums[rand_idx], nums[hi] = nums[hi], nums[rand_idx]
    pivot_idx = partition(nums, lo, hi)
    quick_sort_randomized(nums, lo, pivot_idx - 1)
    quick_sort_randomized(nums, pivot_idx + 1, hi)
```

**逐行讲透**：

- **`rand_idx = random.randint(lo, hi)`**：在 `[lo, hi]` 闭区间内随机选一个索引。注意是闭区间，包含 hi。
- **`nums[rand_idx], nums[hi] = nums[hi], nums[rand_idx]`**：把随机选中的元素换到 hi 位置。这样后续 `partition` 仍然取 `nums[hi]` 作 pivot，但这个 pivot 是随机的。
- **`partition(nums, lo, hi)`**：复用 Lomuto 分区，pivot 已经在 hi 位置。

随机化 pivot 后，**最坏情况出现的概率极低**——要连续多次都选到最值，概率是 $1/n!$ 量级。对 $n=100$，连续 100 次都选到最值的概率约为 $1/100! \approx 10^{-158}$，比宇宙原子数还少。期望仍是 $O(n\log n)$。

**这是工程思维和数学思维的差别**：数学上随机化快排最坏仍是 $O(n^2)$（概率不为 0），工程上认为"概率极低的事件等于不会发生"。所以生产里如果一定要用快排（不用 Timsort），必须随机化 pivot。

### 5. 三路快排：解决大量重复元素

当数组有大量重复元素时，标准快排会退化——所有等于 pivot 的元素都堆在一起，分区不均衡。**三路快排**（3-way partition）把数组分成"小于 / 等于 / 大于 pivot"三部分，等于的部分不再递归：

```python
def quick_sort_3way(nums, lo, hi):
    if lo >= hi:
        return
    pivot = nums[lo]
    # lt 指向"小于 pivot 区"右边界
    # gt 指向"大于 pivot 区"左边界
    # i 是当前扫描位置
    lt, i, gt = lo, lo, hi
    while i <= gt:
        if nums[i] < pivot:
            nums[lt], nums[i] = nums[i], nums[lt]
            lt += 1; i += 1
        elif nums[i] > pivot:
            nums[i], nums[gt] = nums[gt], nums[i]
            gt -= 1   # 注意 i 不动，因为换过来的元素还没看
        else:
            i += 1
    quick_sort_3way(nums, lo, lt - 1)
    quick_sort_3way(nums, gt + 1, hi)
```

**逐行讲透**：

- **`pivot = nums[lo]`**：三路快排通常取首端作 pivot。
- **`lt, i, gt = lo, lo, hi`**：三个指针。lt 是"小于区"右边界（初始为空），i 是扫描位置，gt 是"大于区"左边界（初始为空）。扫描过程中维护三个区：`[lo, lt-1]` < pivot，`[lt, i-1]` == pivot，`[i, gt]` 待扫描，`[gt+1, hi]` > pivot。
- **`while i <= gt:`**：i 扫到 gt 为止。gt 右边都是大于 pivot 的，不用再扫。
- **`if nums[i] < pivot:`**：当前元素小于 pivot，归入"小于区"。交换 `nums[lt]` 和 `nums[i]`（lt 指向"等于区"的第一个，交换后 nums[lt] 变成小于 pivot 的，nums[i] 变成等于 pivot 的），然后 lt 和 i 都右移。
- **`elif nums[i] > pivot:`**：当前元素大于 pivot，归入"大于区"。交换 `nums[i]` 和 `nums[gt]`（gt 指向待扫描区最后一个，交换后 nums[gt] 变成大于 pivot 的），gt 左移。**注意 i 不动**——因为换过来的元素还没看过，下一轮要判断它。
- **`else:`**：当前元素等于 pivot，归入"等于区"。i 右移即可（等于区自动扩张）。
- **`quick_sort_3way(nums, lo, lt - 1)`**：递归排"小于区"。
- **`quick_sort_3way(nums, gt + 1, hi)`**：递归排"大于区"。**"等于区" `[lt, gt]` 不用排**——它们都等于 pivot，已经就位。

**三路快排的威力**：当数据有大量重复（比如 100 万元素只有 10 种值），三路快排能把递归规模从 100 万降到 10 种值的子问题，接近 $O(n)$。这是"按数据特征选算法变体"的典型——重复多就用三路，重复少用标准快排。Java 的 `Arrays.sort` 对基本类型用双轴快排（Dual-Pivot Quicksort，三路的扩展），就是基于这个思想。

### 6. 五指标评价

| 指标 | 快排 | 说明 |
|---|---|---|
| 最好 | $O(n\log n)$ | 每次 pivot 都平分 |
| 平均 | $O(n\log n)$ | 随机 pivot 下期望 |
| 最坏 | $O(n^2)$ | 有序数据 + 端点 pivot |
| 空间 | $O(\log n)$ | 递归栈，最坏 $O(n)$ |
| 稳定 | 否 | 跨距离交换 |
| 自适应 | 否 | 有序数据反而触发最坏 |
| 缓存 | 极好 | 分区是顺序扫描 |

## 三、实践与面试：手撕模板、对数器、面试题

### 1. 面试手撕模板（随机化快排）

面试中如果被要求手写快排，用随机化 Lomuto 版——既展示你知道"最坏情况陷阱"，又是最容易写对的版本：

```python
import random

def quick_sort(nums, lo=0, hi=None):
    """快速排序 - 随机化 Lomuto 版，面试手撕模板。"""
    if hi is None:
        hi = len(nums) - 1
    if lo >= hi:
        return
    # 随机化 pivot，避免有序数据退化 O(n²)
    rand_idx = random.randint(lo, hi)
    nums[rand_idx], nums[hi] = nums[hi], nums[rand_idx]
    # Lomuto 分区
    pivot = nums[hi]
    i = lo
    for j in range(lo, hi):
        if nums[j] < pivot:
            nums[i], nums[j] = nums[j], nums[i]
            i += 1
    nums[i], nums[hi] = nums[hi], nums[i]
    # 递归左右两半（pivot 已在 i 位置就位）
    quick_sort(nums, lo, i - 1)
    quick_sort(nums, i + 1, hi)
    return nums
```

**面试时怎么讲**（按"技术的水有多深"八层面，挑重点讲）：

1. **问题层**（核心思想）：排序本质是消除逆序对。冒泡/插入每次消一个相邻逆序对，$O(n^2)$。快排用分治，一次分区批量消除左半和右半之间的所有逆序对，突破到 $O(n\log n)$。
2. **原理层**：选 pivot → 分区（小于 pivot 的放左，大于的放右）→ 递归排左右。pivot 一次性就位。Lomuto 分区用 i 维护"小于区"边界，扫描一遍 $O(n)$。
3. **优劣层**：原地 $O(\log n)$ 栈空间、缓存极友好（顺序扫描）、平均 $O(n\log n)$ 且常数小；但不稳定、最坏 $O(n^2)$（有序数据 + 端点 pivot）、不自适应。
4. **演进层**：Hoare 1962 原始双指针版（快但难写）→ Lomuto 简化版（易写但慢）→ 随机化 pivot（避免最坏）→ 三路快排（解决重复）→ 双轴快排（Java Arrays.sort）→ introsort（快排+堆排+插入的混合，C++ std::sort）。

**面试加分点**：主动提"随机化 pivot 避免最坏情况"，并说出"有序数据取端点 pivot 会退化 $O(n^2)$"——这能立刻区分你和"只会背模板"的候选人。再提一句"生产里 C++ std::sort 用的是 introsort，不是纯快排"，直接留下深度印象。

### 2. 对数器：左神风格的验证方法

左神反复强调：**写完排序不要靠肉眼检查，要用对数器验证**。对数器的思路是——生成随机数据，用你的排序和标准库排序（绝对正确）对比，跑足够多次，只要有一次不一致就说明你的排序有 bug。

```python
import random

def quick_sort(nums, lo=0, hi=None):
    """你的快排实现。"""
    if hi is None:
        hi = len(nums) - 1
    if lo >= hi:
        return
    rand_idx = random.randint(lo, hi)
    nums[rand_idx], nums[hi] = nums[hi], nums[rand_idx]
    pivot = nums[hi]
    i = lo
    for j in range(lo, hi):
        if nums[j] < pivot:
            nums[i], nums[j] = nums[j], nums[i]
            i += 1
    nums[i], nums[hi] = nums[hi], nums[i]
    quick_sort(nums, lo, i - 1)
    quick_sort(nums, i + 1, hi)
    return nums

def checker(test_times=5000, max_n=100, max_val=1000):
    """对数器：跑 test_times 次，每次随机数据对比。"""
    for _ in range(test_times):
        n = random.randint(0, max_n)         # 随机长度（含 0 和 1）
        arr1 = [random.randint(0, max_val) for _ in range(n)]
        arr2 = arr1[:]                        # 深拷贝
        quick_sort(arr1)                      # 你的排序
        arr2.sort()                           # 标准库（绝对正确）
        if arr1 != arr2:
            print(f"出错！输入: {arr2}, 你的结果: {arr1}")
            return False
    print("对数器验证通过！")
    return True

checker()
```

**对数器的价值**：

- **覆盖边界**：随机长度包含 0 和 1（空数组和单元素数组），随机值包含重复值，能覆盖手写时容易遗漏的边界
- **不依赖人工**：跑 5000 次只要几秒，比手动构造测试用例全面得多
- **左神原话**："对数器是算法工程师的基本功。写完一个算法，第一件事就是写对数器，而不是跑几个例子看看对不对。"

**对高手的启发**：对数器思想不止用于排序——任何"有标准答案可对照"的算法都能用对数器。写完红黑树？对数器对比 `std::map`。写完哈希表？对数器对比 `dict`。这是"工程化验证"的思维，比"理论上推导正确"更可靠。

### 3. 教科书做法 vs 生产做法

| 场景 | 教科书做法 | 生产做法 | 原因 |
|---|---|---|---|
| 通用排序 | 手写快排 | `sorted()`（Timsort） | Timsort 稳定且最坏 $O(n\log n)$ |
| 基本类型排序（Java） | 手写快排 | `Arrays.sort`（Dual-Pivot Quicksort） | 双轴快排对基本类型优化 |
| 基本类型排序（C++） | 手写快排 | `std::sort`（introsort） | introsort 保证最坏 $O(n\log n)$ |
| 面试白板 | 手写快排 | — | 无奈 |
| 需要稳定 | 不能用快排 | Timsort / 归并 | 快排不稳定 |

**为什么 Java 对基本类型用快排、对对象用归并（Timsort）？** 因为基本类型不需要稳定性（两个相等的 int 没区别），快排的缓存友好性和低常数占优；对象需要稳定性（两个相等的对象可能有其他字段差异），必须用稳定排序。这是"按数据类型选算法"的工程范例。

### 4. 三个真实工程坑

**坑一：有序数据上用端点 pivot，退化 $O(n^2)$。**

```python
# 错误：对有序数据取最后一个作 pivot
def quick_bad(nums, lo, hi):
    if lo >= hi: return
    pivot = nums[hi]   # 有序数据下，这是最大值，分区极不均衡
    ...
```

症状：对有序数据排序耗时飙升。根因：有序数据下端点是最值，每次分区一边空。修复：①随机化 pivot ②三数取中（lo/mid/hi 的中位数）③直接用 `sorted()`。

**坑二：递归深度过大导致栈溢出。**

```python
# 错误：对 10 万元素递归，最坏递归深度 10 万，栈溢出
def quick_recursive(nums, lo, hi):
    # 最坏情况递归深度 O(n)，10 万层递归直接栈溢出
    ...
```

症状：大数据量下程序崩溃。根因：最坏情况递归深度 $O(n)$。修复：①随机化 pivot 避免最坏 ②尾递归优化——对较小的子数组先递归，较大的子数组用循环处理，把递归深度压到 $O(\log n)$ ③用迭代式快排（显式栈）。

```python
# 尾递归优化：先递归小的，大的用循环
def quick_tail_optimized(nums, lo, hi):
    while lo < hi:
        pivot_idx = partition(nums, lo, hi)
        # 先递归小的那半
        if pivot_idx - lo < hi - pivot_idx:
            quick_tail_optimized(nums, lo, pivot_idx - 1)
            lo = pivot_idx + 1   # 大的那半用循环
        else:
            quick_tail_optimized(nums, pivot_idx + 1, hi)
            hi = pivot_idx - 1
```

**坑三：对象排序用快排，丢稳定性。**

```python
students = [("Alice", 90), ("Bob", 90), ("Carol", 85)]
# 用快排按分数排，Alice 和 Bob 的相对顺序可能被打乱
```

症状：同分学生顺序乱。根因：快排分区时跨距离交换。修复：用稳定排序（Timsort），或把次要键加进排序键。

### 5. 高手向：introsort 为什么诞生

纯快排有一个绕不开的痛点：**最坏 $O(n^2)$**。即使随机化 pivot，理论上仍可能（概率极低）退化。但对一些场景（实时系统、安全敏感场景），"概率极低"不可接受——必须保证最坏 $O(n\log n)$。

1989 年，David Musser 提出了 **introsort**（内省排序）：**快排 + 堆排 + 插入排序的混合**。

```
introsort(nums, lo, hi, depth_limit):
    if hi - lo < 16:                      # 小数据切插入排序
        insertion_sort(nums, lo, hi)
    elif depth_limit == 0:                # 递归深度超限，切堆排
        heap_sort(nums, lo, hi)
    else:                                 # 否则用快排
        p = partition(nums, lo, hi)
        introsort(nums, lo, p-1, depth_limit - 1)
        introsort(nums, p+1, hi, depth_limit - 1)
```

**introsort 的三个机制**：

1. **递归深度限制**：初始 `depth_limit = 2 * log2(n)`。快排正常情况下递归深度 $\log n$，如果超过 $2\log n$ 说明 pivot 选得不好（接近最坏情况），立刻切到堆排（保证 $O(n\log n)$）。
2. **小数据切插入**：当区间小于 16 个元素时，切到插入排序。小数据上插入排序的常数更小，且对"基本有序"数据接近 $O(n)$。
3. **大数据用快排**：正常情况下用快排，享受缓存友好性和低常数。

**为什么 introsort 是工程上的"完美排序"？**

- 最坏 $O(n\log n)$：堆排兜底，保证不会退化
- 平均 $O(n\log n)$：快排主体，常数小
- 小数据 $O(n)$：插入排序收尾，常数更小
- 原地：三种排序都原地，空间 $O(\log n)$

C++ `std::sort`、.NET `Array.Sort`、Go `sort.Slice` 都用 introsort 或其变体。**这是"工程思维"的典范——不执着于单一算法的纯粹，按数据规模和递归深度动态切换算法，取各家之长。**

**对高手的启发**：生产级排序不是"选一个最好的算法"，而是"按场景动态切换"。这种"混合策略"思想在工程中无处不在：数据库查询优化器按数据量选索引/全表扫描、HTTP 按内容大小选 chunked/Content-Length、编译器按代码热度选解释/JIT。理解 introsort，能让你在面对"选 A 还是选 B"的工程决策时，想到第三条路——"按场景动态切换"。

### 6. 面试高频题

**面试题一**："手写快排，说明时间复杂度，如何优化。"

**满分回答模板**：用上面的随机化 Lomuto 模板，边写边讲：

1. "我先随机化 pivot，避免有序数据退化 $O(n^2)$。"
2. "Lomuto 分区：i 维护小于 pivot 区边界，j 扫描，遇到小于 pivot 的就交换到 i 位置。"
3. "时间复杂度：平均 $O(n\log n)$，最坏 $O(n^2)$（随机化后概率极低）。空间 $O(\log n)$ 递归栈。"
4. "不稳定——跨距离交换破坏相等元素相对顺序。"
5. "优化方向：①三路快排解决重复元素 ②尾递归优化压栈深度 ③小数据切插入排序 ④生产用 introsort（快排+堆排+插入混合）。"

**面试官追问"快排和归并怎么选"**：

- 需要稳定 → 归并
- 内存受限 → 快排（原地，归并要 $O(n)$ 辅助空间）
- 需要最坏 $O(n\log n)$ → 归并或堆排（快排最坏 $O(n^2)$）
- 通用场景 → 快排（缓存友好、常数小），但生产用 Timsort 或 introsort

**面试官追问"为什么快排缓存友好"**：分区是顺序扫描数组（`for j in range(lo, hi)`），CPU 预取能命中缓存行。归并的合并虽然也顺序扫描，但需要两个数组来回跳，缓存命中率略低。堆排的父子索引跳跃（`i -> 2i+1, 2i+2`）是随机访问，缓存最不友好。**$O$ 符号隐藏了常数，缓存友好性让快排实际比归并快 1.5-2 倍。**

## 四、速查与自测

### 速查表：快排五指标与变体

| 指标 | 标准快排 | 随机化快排 | 三路快排 | 双轴快排 |
|---|---|---|---|---|
| 平均 | $O(n\log n)$ | $O(n\log n)$ | $O(n\log n)$ | $O(n\log n)$ |
| 最坏 | $O(n^2)$ | $O(n^2)$（概率极低） | $O(n^2)$ | $O(n^2)$ |
| 重复数据处理 | 退化 | 退化 | **$O(n)$** | 较好 |
| 空间 | $O(\log n)$ | $O(\log n)$ | $O(\log n)$ | $O(\log n)$ |
| 稳定 | 否 | 否 | 否 | 否 |

**分区算法对比**：

| 分区 | 交换次数 | 代码难度 | 推荐度 |
|---|---|---|---|
| Lomuto | 多 | 易 | 教学/面试 |
| Hoare | 少 | 难 | 工程 |
| 三路 | 中 | 中 | 重复数据 |
| 双轴 | 中 | 难 | 大数据（Java Arrays.sort） |

### 自测三问

**问题一：** 快排为什么能 $O(n\log n)$？请用"批量消除逆序对"来解释，并对比插入排序。

**参考答案：** 插入排序每次操作消除 1 个相邻逆序对，总逆序对 $O(n^2)$，所以 $O(n^2)$。快排分区时，pivot 一次性"找到最终位置"，且**左半所有元素和右半所有元素之间的逆序对全部消除**——一次 $O(n)$ 的分区操作消除了 $O(\text{左半} \times \text{右半})$ 个逆序对，是批量消除。递归深度 $\log n$，每层 $O(n)$，总 $O(n\log n)$。本质区别是"每次消除 1 个"vs"每次消除一批"，分治让"批"的规模成平方级。

**问题二：** 快排最坏 $O(n^2)$，归并和堆排最坏 $O(n\log n)$。那为什么很多语言的库排序还是用快排（或快排变体）？

**参考答案：** 三个原因。①常数小：快排的分区是顺序扫描，缓存命中率极高，实际比归并快 1.5-2 倍。②原地：快排 $O(\log n)$ 栈空间，归并要 $O(n)$ 辅助空间，大数据上空间是瓶颈。③随机化后最坏概率极低：随机化 pivot 让 $O(n^2)$ 出现的概率是 $1/n!$ 量级，工程上视为不会发生。Java 对基本类型用双轴快排、对对象用 Timsort，C++ `std::sort` 用 introsort（快排+堆排+插入混合，堆排兜底保证最坏 $O(n\log n)$）。**$O$ 符号隐藏了常数，工程选型不能只看 $O$。**

**问题三：** 三路快排解决什么问题？为什么对"大量重复元素"特别有效？

**参考答案：** 解决"大量重复元素导致分区不均衡"的问题。标准快排（Lomuto/Hoare）把等于 pivot 的元素堆到一边，分区不均衡，递归深度增大。三路快排把数组分成"小于 / 等于 / 大于 pivot"三区，等于的部分一次分区就全部就位，不再递归——相当于"把重复元素提前剔除出递归"。当数据有大量重复（比如 100 万元素只有 10 种值），三路快排能把递归规模从 100 万降到 10 种值的子问题，接近 $O(n)$。这是"按数据特征选算法变体"的典型——重复多就用三路，重复少用标准快排。

### 算法思想 × 生活迁移

快排的核心思想是"**分治 + 找基准**"——用一个基准把问题拆成两个独立子问题。

**迁移一：用"中位数参考"做决策。** 团队要给 100 个提案排优先级，让所有人排 100 个太累。快排式做法：先挑一个"中等优先级"的提案作参考，把剩下的分成"比它重要"和"不如它重要"两堆。然后两堆各自再挑参考继续分。这种"找基准 + 二分"比"全两两比较"高效得多——100 个提案，快排式约 700 次比较，全两两比较要 4950 次。

**迁移二：分治解决冲突。** 两个团队对方案有分歧，不要试图一次性解决所有分歧（那是 $O(n^2)$ 的争吵）。先找一个"双方都能接受的中间方案"作 pivot，把分歧分成"比这个更激进的"和"比这个更保守的"两组，分别讨论。每组再找中间方案，继续分。这种"分治谈判"比"全量谈判"高效，且每次分治都有明确的"基准"避免讨论发散。

**迁移三：introsort 式"按场景切换策略"。** 不要执着于"一个方法解决所有问题"。小问题用简单方法（插入排序），中等问题用高效方法（快排），出现异常用兜底方法（堆排）。项目管理也类似：小任务直接做（插入），常规任务走标准流程（快排），出现风险立刻升级到兜底方案（堆排）。**"动态切换策略"比"坚持单一最优"更鲁棒。**

**为什么这些迁移成立：** 快排成立的前提是"能找到一个基准把问题分成两个独立子问题"。这个前提在优先级排序、方案谈判里都成立——总能找到一个"中间参考"把意见分成两堆。迁移成立的关键是"基准选得合理"——快排选错 pivot 会退化 $O(n^2)$，生活里选错基准也会让分治失效（比如选了个极端的方案，一边空一边满）。**选基准的能力，是分治思想能否迁移成功的关键。**
