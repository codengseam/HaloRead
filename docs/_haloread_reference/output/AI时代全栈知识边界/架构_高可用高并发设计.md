# AI时代全栈知识边界·19|高可用高并发设计

> 前置知识:了解 HTTP 请求生命周期、知道 Redis 是内存数据库、会用一门后端语言写 Web 接口、理解进程与线程的区别

### 一、概念

高可用(High Availability, HA)指系统在绝大部分时间可对外提供服务,用"可用性百分比"衡量,如 99.99%(四个 9)意味着全年停机不超过 52.6 分钟。高并发(High Concurrency)指系统在同一时刻能处理大量请求,用 QPS(Queries Per Second)或 TPS(Transactions Per Second)衡量。两者常被混用,但本质不同:高可用回答"服务在不在",高并发回答"服务能扛多少"。

工程上常把架构目标拆成四个互相牵制的维度:

- 高可用(HA):可用性几个 9,衡量"服务可访问时长占比"
- 高并发:QPS/TPS 峰值,衡量"同时能处理多少请求"
- 高性能:响应时间 P99/P999,衡量"单个请求多快返回"
- 可扩展(Scalability):水平扩展(加机器)与垂直扩展(加配置),衡量"容量增长时能否线性扩容"

可用性等级与年停机时间对照:99%(两个 9)≈ 87.6 小时;99.9%(三个 9)≈ 8.76 小时;99.99%(四个 9)≈ 52.6 分钟;99.999%(五个 9)≈ 5.26 分钟。从三个 9 到四个 9 通常靠冗余和故障转移就能达成,从四个 9 到五个 9 则必须多活架构 + 自动化运维,成本指数级上升。

后文反复出现的几组术语先对齐:Circuit Breaker(熔断器)、Token Bucket(令牌桶)、Leaky Bucket(漏桶)、Failover(故障转移)、Graceful Degradation(优雅降级)、Rate Limiting(限流)。SLA(Service Level Agreement,服务等级协议)是给客户的承诺,SLO(Service Level Objective,服务等级目标)是内部的工程目标,SLO 必须严于 SLA 才有缓冲。

### 二、原理

#### 1. 冗余为什么能提升可用性

单点故障(Single Point of Failure, SPOF)是可用性的头号杀手。一台机器宕机,无论是因为硬件老化、内核 panic 还是机房断电,挂在它上面的服务就全部不可用。冗余的本质是"把一份服务变成多份,任意一份挂掉仍有副本可用"。

冗余按拓扑分三类:

- 主从(Master-Slave):一主多从,写主、读从。主挂需提升某个从为主,有切换窗口
- 主主(Master-Master):多主互写,冲突需靠 timestamp 或向量时钟解决,实现复杂
- 多活(Active-Active):多地多中心同时对外服务,异地之间靠 Raft/Paxos 类共识协议同步数据

冗余只解决了"有副本",还要解决"如何切换到副本"。这就需要 Failover(故障转移):健康检查持续探活主节点,连续失败 N 次后触发切换。切换的关键不是"能不能切",而是"切换期间的数据一致性"——主从延迟会导致刚写入主的数据在从上还没同步,提升从为主后这部分数据丢失。这是金融级场景必须用同步复制而非异步复制的根本机制。

#### 2. 缓存为什么能提升并发

数据库是系统的吞吐瓶颈。一条 SQL 走 B+ 树索引、加锁、写 redo log,单机 QPS 上限通常在几千到几万。而 Redis 这种内存数据库,单机 QPS 可达 10 万级。缓存的本质是"用更快的存储挡住更慢的存储",把大部分读请求拦在 DB 之外。

缓存提升并发的核心原理有三:第一,内存访问比磁盘快 5-6 个数量级;第二,缓存通常是 KV 结构,无解析、无索引查找开销;第三,缓存可水平扩展,加机器即可线性提升读吞吐。但缓存引入新问题——缓存与 DB 的一致性、缓存穿透(查不存在的 key)、缓存击穿(热 key 过期瞬间大量请求打穿到 DB)、缓存雪崩(大量 key 同时过期)。这四类问题对应不同的工程解法:一致性靠"先更新 DB 再删缓存"加延迟双删;穿透靠布隆过滤器或缓存空值;击穿靠互斥锁或热点 key 永不过期;雪崩靠过期时间加随机抖动。

#### 3. 限流、熔断、降级三者的差别

这三者经常被一起提到,但触发时机和职责边界完全不同:

- 限流(Rate Limiting):入口保护。在请求进入系统前按阈值截断,保护系统不被流量打垮。发生在"系统还没出问题时"
- 熔断(Circuit Breaker):自身保护。下游依赖(如 DB、第三方接口)持续报错或超时,熔断器跳闸,后续请求直接快速失败不再调用下游。发生在"下游已经出问题时"
- 降级(Graceful Degradation):用户体验保护。系统扛不住时主动放弃部分非核心功能,如商品详情页不展示推荐列表、结算页不显示优惠券计算。发生在"系统快要扛不住时"

三者关系:限流是第一道闸门,挡住超额流量;熔断是第二道闸门,挡住对故障下游的调用;降级是兜底策略,在限流和熔断都生效后仍要给用户可用响应。一个健壮的系统三者必须同时具备。

#### 4. 限流算法的演进逻辑

四种主流限流算法对应不同的精度与开销权衡:

- 计数器(Counter):固定时间窗口内计数,超阈值拒绝。简单但有"临界突刺"——窗口切换瞬间可能放过 2 倍流量
- 滑动窗口(Sliding Window):把固定窗口切成多个小格,滑动统计。解决了临界突刺,精度取决于小格粒度
- 漏桶(Leaky Bucket):请求像水滴进桶,桶以恒定速率漏水,满则拒绝。强制平滑输出速率,适合保护下游
- 令牌桶(Token Bucket):以恒定速率往桶里放令牌,请求取走令牌才能通过,桶满则丢弃令牌。允许突发流量(桶里攒的令牌可瞬间消耗),适合入口限流

漏桶与令牌桶的本质差异:漏桶限制"输出速率恒定",令牌桶限制"平均速率恒定但允许突发"。对外提供 API 用令牌桶(允许客户端偶发突发),保护下游 DB 用漏桶(强制 DB 收到平稳流量)。

#### 5. 熔断器三态机

熔断器(Circuit Breaker)借鉴电路保险丝思想,有三个状态:

- Closed(关闭):正常放行请求,统计失败率。失败率超阈值则转 Open
- Open(打开):直接快速失败,不调用下游。经过冷却时间后转 Half-Open
- Half-Open(半开):放行少量探测请求,探测成功则转 Closed(下游恢复),探测失败则转 Open(下游仍未恢复)

三态机的关键参数:失败率阈值(如 50%)、统计窗口(如 10 秒内至少 20 个请求才统计)、冷却时间(如 5 秒)、半开探测请求数(如 1-3 个)。这套机制最早由 Michael Nygard 在《Release It!》中提出,Martin Fowler 在其文章中推广,Hystrix 是 Java 生态最知名的实现。

#### 6. 容量规划

容量规划(Capacity Planning)回答两个问题:单机能扛多少、什么时候该扩容。

单机 QPS 上限估算公式:`单机 QPS = (CPU 核数 × 单核并发数) / 平均响应时间`。例如 8 核 CPU、单核并发 100(典型 Web 服务器配置)、平均响应时间 50ms,则单机 QPS ≈ 8 × 100 / 0.05 = 16000。实际生产中要打 70% 折扣预留缓冲,即约 11000 QPS。

扩容时机判断看三个信号:CPU 利用率持续 5 分钟超 70%、P99 响应时间持续超 SLO、队列积压(消息队列消费滞后)。三者出现任一即应触发扩容。注意扩容不是越多越好——无状态服务(如 Web API)可线性扩容,有状态服务(如 MySQL 主库)扩容受限于分片策略,加机器不能直接提升写吞吐。

### 三、实践

#### 1. Python 令牌桶限流器

下面是一个线程安全的令牌桶实现,核心是"按时间差补充令牌、请求扣减令牌":

```python
import time
import threading


class TokenBucket:
    def __init__(self, capacity: float, refill_rate: float):
        """
        :param capacity: 桶容量,决定允许的突发流量上限
        :param refill_rate: 令牌补充速率(个/秒),决定平均通过速率
        """
        self.capacity = capacity
        self.refill_rate = refill_rate
        self.tokens = capacity
        self.last_refill = time.monotonic()
        self.lock = threading.Lock()

    def _refill(self):
        now = time.monotonic()
        elapsed = now - self.last_refill
        # 按经过时间补充令牌,但不超过桶容量
        self.tokens = min(self.capacity, self.tokens + elapsed * self.refill_rate)
        self.last_refill = now

    def acquire(self, tokens: float = 1) -> bool:
        """尝试取走 tokens 个令牌,成功返回 True,失败返回 False(不阻塞)"""
        with self.lock:
            self._refill()
            if self.tokens >= tokens:
                self.tokens -= tokens
                return True
            return False

    def try_acquire_or_wait(self, tokens: float = 1) -> bool:
        """取不到就短暂自旋等待,适合允许少量排队的场景"""
        while not self.acquire(tokens):
            time.sleep(0.01)
        return True


# 使用示例:容量 10、每秒补充 5 个令牌
bucket = TokenBucket(capacity=10, refill_rate=5)
for i in range(15):
    if bucket.acquire():
        print(f"请求 {i} 通过")
    else:
        print(f"请求 {i} 被限流")
```

这段代码的关键点有三个。第一,用 `time.monotonic()` 而非 `time.time()`,避免系统时钟回拨导致令牌数异常。第二,锁保护 `_refill` 和扣减的原子性,否则并发下会出现"两个线程同时看到 tokens=1 都通过"的超发。第三,令牌补充是惰性计算——不维护后台线程定时加令牌,而是在每次 `acquire` 时按时间差一次性补齐,这种"懒计算"模式省去了线程开销,是生产级令牌桶的标准写法。

生产环境通常不会自己写,直接用 Redis + Lua 脚本实现分布式令牌桶(保证多实例间令牌计数一致),或用 Sentinel、Resilience4j 这类成熟框架。但理解这段代码后,看框架源码会顺畅很多。

#### 2. 熔断器状态机伪代码

下面用 Python 实现一个最小可用的熔断器,完整呈现三态转移:

```python
import time
from enum import Enum
from collections import deque


class State(Enum):
    CLOSED = "closed"
    OPEN = "open"
    HALF_OPEN = "half_open"


class CircuitBreaker:
    def __init__(self, failure_threshold=0.5, min_requests=20,
                 window_seconds=10, cooldown_seconds=5, half_open_probes=3):
        self.state = State.CLOSED
        self.failure_threshold = failure_threshold      # 失败率阈值
        self.min_requests = min_requests                # 统计所需最小请求数
        self.window_seconds = window_seconds            # 统计窗口
        self.cooldown_seconds = cooldown_seconds        # Open 冷却时间
        self.half_open_probes = half_open_probes        # 半开探测请求数
        self.opened_at = None
        self.records = deque()  # (timestamp, is_success)

    def _record(self, is_success):
        now = time.monotonic()
        self.records.append((now, is_success))
        # 清理窗口外记录
        cutoff = now - self.window_seconds
        while self.records and self.records[0][0] < cutoff:
            self.records.popleft()

    def _failure_rate(self):
        if len(self.records) < self.min_requests:
            return 0.0
        failures = sum(1 for _, ok in self.records if not ok)
        return failures / len(self.records)

    def before_call(self):
        """调用下游前调用,返回 True 表示放行,False 表示熔断"""
        if self.state == State.OPEN:
            if time.monotonic() - self.opened_at >= self.cooldown_seconds:
                self.state = State.HALF_OPEN
                return True
            return False
        return True

    def after_call(self, is_success):
        """调用下游后调用,记录结果并可能触发状态转移"""
        self._record(is_success)
        if self.state == State.HALF_OPEN:
            if not is_success:
                # 探测失败,回到 Open
                self.state = State.OPEN
                self.opened_at = time.monotonic()
            elif len(self.records) >= self.half_open_probes:
                # 探测全部成功,恢复 Closed
                self.state = State.CLOSED
                self.records.clear()
        elif self.state == State.CLOSED:
            if self._failure_rate() >= self.failure_threshold:
                self.state = State.OPEN
                self.opened_at = time.monotonic()


# 使用示例
cb = CircuitBreaker()
def call_downstream():
    if not cb.before_call():
        raise RuntimeError("熔断中,请稍后重试")
    try:
        ok = do_http_call()  # 假设的下游调用
        cb.after_call(ok)
        return ok
    except Exception:
        cb.after_call(False)
        raise
```

状态转移逻辑有三个要点。第一,`before_call` 在 Open 状态下检查冷却时间,到期才转 Half-Open 并放行探测请求,未到期直接拒绝——这是"快速失败"的核心,避免请求堆积在下游超时上。第二,`after_call` 在 Half-Open 下只要一次探测失败就立刻回到 Open,这是"宁可错杀不可放过"的保守策略。第三,Half-Open 转 Closed 需要连续多个探测请求都成功,且要清空统计窗口,避免旧失败记录污染新状态。

实际生产中 Hystrix 用滑动窗口 + 半开计数器实现得更精细,Sentinel 用 LeapArray 滑动窗口。但状态机的骨架就是上面这段——理解了它,所有熔断器实现都是变体。

#### 3. 多级缓存架构

多级缓存(Multi-Level Cache)是高并发读场景的标准架构,典型三级:

```
请求 → L1 本地缓存(Caffeine/Guava,进程内) → L2 Redis(集中式) → L3 DB
         命中率约 30%-50%             命中率约 40%-60%       兜底
```

读路径(Read Path):

1. 请求先查 L1 本地缓存,命中直接返回(纳秒级)
2. L1 未命中查 L2 Redis,命中则回填 L1 后返回(毫秒级)
3. L2 未命中查 L3 DB,命中则回填 L2 和 L1 后返回(十毫秒级)
4. DB 也未命中,返回空值或默认值,可按需缓存空值防穿透

写路径(Write Path)与失效策略:

- Cache-Aside(旁路缓存):应用先更新 DB,再删除缓存。最常用,实现简单,有一致性窗口
- Write-Through(写穿透):应用写缓存,缓存同步写 DB。强一致但写延迟高
- Write-Behind(写回):应用写缓存即返回,缓存异步刷 DB。写吞吐最高但有丢数据风险

多级缓存的失效比单级缓存复杂。L1 是进程内缓存,多实例间各自一份,某个实例更新后其他实例的 L1 不会自动失效。解法有二:一是 L1 设较短 TTL(如 10 秒),接受短暂数据不一致;二是用 Redis Pub/Sub 或 Kafka 广播失效消息,各实例收到后清本地 L1。前者简单但有一致性窗口,后者强一致但增加基础设施依赖。电商商品详情页这类对一致性不敏感的场景用前者,金融账户余额这类强一致场景不该用 L1,直接走 L2+DB。

### 四、速查/自测

#### 限流算法对照表

| 算法 | 实现复杂度 | 突发流量 | 平滑输出 | 适用场景 |
|---|---|---|---|---|
| 计数器(固定窗口) | 低 | 允许(临界突刺) | 否 | 简单粗粒度限流 |
| 滑动窗口 | 中 | 允许(可控) | 较平滑 | 精度要求较高的入口限流 |
| 漏桶(Leaky Bucket) | 中 | 不允许 | 是(恒定速率) | 保护下游 DB、调用第三方 API |
| 令牌桶(Token Bucket) | 中 | 允许(桶内令牌) | 平均速率恒定 | 网关入口、对外 API 限流 |

#### 自测题

**问题一(原理层):** 从三个 9(99.9%)提升到四个 9(99.99%),技术上需要补齐哪些能力?从四个 9 提升到五个 9 又需要什么?

参考答案:三个 9 到四个 9 主要靠冗余 + 自动故障转移——主从切换、负载均衡健康检查、监控告警能在分钟级发现并切换故障节点。四个 9 到五个 9 单靠冗余不够,必须多活架构(异地双活/多活)+ 数据强一致同步(同步复制或 Raft 共识)+ 全自动化运维(故障自愈,人为介入的几分钟就够把五个 9 破掉)。成本上,四个 9 是工程努力可达,五个 9 几乎要重新设计整个架构。

**问题二(实践层):** 令牌桶容量 100、补充速率 10 个/秒。系统闲置 5 秒后,突然来 150 个请求,有多少个能立即通过?为什么?

参考答案:100 个立即通过。闲置 5 秒期间,桶里令牌从初始 100 已经满了(补充到 100 后再补被丢弃),所以桶里始终是 100 个令牌。150 个请求来时,前 100 个各取 1 个令牌通过,后 50 个因令牌不足被限流。这正是令牌桶"允许突发"的特性——桶容量决定了瞬时突发上限,补充速率决定了长期平均速率。

**问题三(思路层):** 熔断器在 Half-Open 状态下,为什么一次探测失败就要立刻回到 Open,而不是像 Closed 那样统计失败率?

参考答案:Half-Open 的语义是"试探下游是否恢复"。放行的探测请求本身就是有风险的——如果下游还在故障,这些请求会拖慢系统。如果用统计失败率的方式,需要积累足够样本(如 20 个请求)才能判断,这 20 个请求都可能堆积超时,反而加剧雪崩。一次失败立刻回 Open 是"快速止损"策略,代价是可能误判(下游其实刚恢复但偶发一次超时),但相比雪崩风险,误判代价小得多。这是熔断器设计上的"保守优先"原则。

**问题四(实践层):** 多级缓存 L1(本地)+ L2(Redis)+ L3(DB),某商品价格被更新。如何保证三个层级最终一致?给出 Cache-Aside 模式的具体步骤。

参考答案:Cache-Aside 标准步骤:① 先更新 DB 中的价格;② 删除 L2 Redis 中对应的 key;③ 通过 Redis Pub/Sub 或 Kafka 广播一条"价格变更"消息;④ 各应用实例收到消息后删除自己的 L1 本地缓存。注意是"删除"缓存而不是"更新"缓存——删除是幂等的,并发更新时不会出现"旧值覆盖新值"的脏写问题。即便如此仍有短暂不一致窗口(DB 更新后、缓存删除前的几毫秒),对极强一致场景要用"延迟双删":更新后立即删一次,休眠几百毫秒再删一次,兜掉并发读导致的脏回填。

**问题五(思路层):** 单机 QPS 估算公式 `(CPU 核数 × 单核并发数) / 平均响应时间` 在什么场景下会严重失真?

参考答案:三种典型失真场景。第一,I/O 密集型服务里"单核并发数"被高估——如果用同步阻塞模型,单核并发受限于线程数,而非公式假设的 100。第二,响应时间分布偏斜时,平均值严重低于 P99——用平均值算出的 QPS 看似很高,但实际高峰期 P99 飙升会导致大量请求超时,真实有效 QPS 远低于估算。第三,有状态服务(如带本地缓存的实例)扩容时缓存命中率下降,平均响应时间反而变长,出现"加机器性能更差"的反常现象。容量规划必须以压测数据为准,公式只用于前期粗估。

### 可交给 AI 的部分

这一章里,**限流算法对照表、容量规划估算公式、多级缓存架构图、熔断器参数推荐值**这几块可以放心交给 AI 生成。给定业务场景(如"电商商品详情页,峰值 QPS 5000,响应时间 P99 < 200ms"),AI 能给出合理的限流算法选型、缓存层级设计、熔断参数初值,准确率较高。UML 状态图、架构拓扑图这类可视化产物,AI 用 Mermaid 或 PlantUML 输出可用率也不错,适合直接放进设计文档。

但有几类内容**必须自己掌握、不能盲信 AI**,边界与风险如下。

第一类是熔断参数的线上调优。AI 给的失败率阈值 50%、冷却 5 秒是通用默认值,但真实业务里这些参数必须根据下游特性调。下游是 DB,失败率阈值要调低(DB 报错通常意味着主库挂了,要快速熔断);下游是第三方 API,失败率阈值要调高(第三方偶发抖动不该立即熔断)。这些判断依赖对下游 SLA 的实测认知,AI 无从知晓。盲目用 AI 默认值,要么熔断太灵敏(下游正常抖动就被熔断),要么熔断太迟钝(雪崩已经发生才触发)。

第二类是缓存一致性方案的选型。Cache-Aside、Write-Through、Write-Behind 三种模式 AI 能列出区别,但"你们这个业务该用哪种"涉及对一致性要求、写吞吐需求、可接受数据丢失程度的综合判断。金融账户场景误用 Write-Behind 会导致资金丢失,这类决策 AI 给不了负责任的建议,必须架构师基于业务语义拍板。

第三类是容量规划的压测验证。公式估算只是起点,真实容量必须靠压测。AI 能帮你写压测脚本、设计压测场景,但压测结果的分析(为什么 P99 飙升、瓶颈在 CPU 还是 I/O、加机器能不能解决)需要人看火焰图和监控面板。把压测分析完全交给 AI,容易得出"加 3 台机器就能解决"的错误结论——实际瓶颈可能是数据库连接池打满,加应用服务器无济于事。

第四类是故障应急的现场决策。线上熔断器全开、限流阈值被打满时,是先扩容还是先降级、是切流量还是重启服务,这类决策必须在几分钟内做出且不能回退。AI 在这种高压场景下来不及响应,且缺乏对全局拓扑的认知。Runbook 可以让 AI 提前帮你写好,但执行判断必须是人——这是 SRE 与架构师不可替代的核心能力。

区分"能交"与"不能交"的本质是:**参数表、架构图、压测脚本是机械产出,AI 强;参数调优的实测判断、一致性方案的语义权衡、应急现场的全局决策是经验推理,AI 弱**。把机械产出交给 AI 加速文档化,把经验推理留给自己练成现场判断力,这是 AI 时代架构师的知识边界。
