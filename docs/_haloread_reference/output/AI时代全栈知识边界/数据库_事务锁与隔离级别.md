# AI时代全栈知识边界·14|事务锁与隔离级别

> 前置知识:会写基本 SQL 增删改查、了解 InnoDB 是 MySQL 默认存储引擎、知道 B+ 树索引与主键的概念
> 学完你能:① 说清 ACID 四特性各自靠什么机制保证 ② 区分脏读/不可重复读/幻读三种并发异常 ③ 看懂两个 session 并发执行时为何出现诡异读现象 ④ 解释 MySQL InnoDB 默认 RR、PostgreSQL 默认 RC 的设计取舍 ⑤ 演示一个交叉更新死锁并用 `SHOW ENGINE INNODB STATUS` 排查 ⑥ 判断何时该用 `SELECT ... FOR UPDATE`、何时该靠 MVCC 兜底

### 一、概念

事务(Transaction)是数据库操作的最小逻辑执行单位,要么全部成功提交,要么全部失败回滚,不存在中间状态。隔离级别(Isolation Level)是数据库在并发事务之间"互不干扰"程度的配置档位——档位越高,并发越安全但吞吐越低;档位越低,并发越高但可能出现诡异读现象。

这两个概念要放到并发语境下理解。单事务场景下不存在隔离问题,只有当多个事务同时读写同一批数据时,才会出现"事务 A 看到了事务 B 中途的脏数据""事务 A 两次读同一行结果不一致"这类问题。隔离级别就是数据库提供给开发者的一个权衡旋钮。

需要先对齐几组术语。ACID 是事务的四个基本特性:原子性(Atomicity)、一致性(Consistency)、隔离性(Isolation)、持久性(Durability)。MVCC(Multi-Version Concurrency Control,多版本并发控制)是让"读不阻塞写、写不阻塞读"的核心机制。锁是写写串行化的保障,按粒度分记录锁(Record Lock)、间隙锁(Gap Lock)、临键锁(Next-Key Lock);按模式分共享锁(S 锁,Shared Lock)与排他锁(X 锁,Exclusive Lock)。这些术语在后文会逐一展开。

### 二、原理

#### 1. ACID 每一项为什么必要

四个特性不是凭空设计的,每一项都对应一类现实问题。

原子性解决的是"中途崩溃怎么办"。银行转账从 A 扣 100 元、给 B 加 100 元,如果在两步之间宕机,没有原子性就会出现"钱凭空消失"。InnoDB 通过 undo log(回滚日志)记录修改前的旧值,事务未提交就崩溃时,重启后按 undo log 反向恢复到事务开始前的状态。

一致性解决的是"数据必须合法"。这里的合法指满足所有约束(主键、外键、唯一索引、CHECK 约束)和业务语义(转账前后总余额不变)。一致性是目标,原子性、隔离性、持久性都是手段。一个常被忽略的点是:数据库约束只能保证结构合法(如主键不重复),业务语义合法(如"转账金额不能超过余额")必须靠应用层代码保证。

隔离性解决的是"并发互相干扰怎么办"。多个事务同时改同一行,不加隔离会出现丢失更新、脏读等。InnoDB 通过锁和 MVCC 共同实现隔离——写写靠行锁串行化,读写靠 MVCC 让读不阻塞写。

持久性解决的是"提交后崩溃怎么办"。事务一旦 COMMIT,即使下一秒拔掉电源,数据也不能丢。InnoDB 通过 redo log(重做日志)采用 WAL(Write-Ahead Logging,预写日志)机制,先写日志再写数据页,保证 crash 后能从日志恢复;binlog(归档日志)则用于主从复制和数据恢复。

#### 2. 四种隔离级别会产生的并发问题

并发事务下有三种典型异常:

- 脏读(Dirty Read):事务 A 读到了事务 B 尚未提交的修改,B 回滚后 A 读到的是"脏"数据
- 不可重复读(Non-Repeatable Read):事务 A 两次读取同一行,结果不同,因为 B 在中间提交了对该行的更新
- 幻读(Phantom Read):事务 A 两次按同一条件查询,后一次多出(或少掉)了 B 新插入(或删除)的行

三者的粒度递进:脏读读到的是"未提交数据",不可重复读针对"已有行的修改",幻读针对"新插入/删除的行"。

SQL 标准定义了四种隔离级别,从低到高:

- 读未提交(Read Uncommitted,RU):允许脏读,生产几乎不用
- 读已提交(Read Committed,RC):解决脏读,允许不可重复读和幻读。PostgreSQL 默认
- 可重复读(Repeatable Read,RR):解决脏读和不可重复读,SQL 标准下仍允许幻读。MySQL InnoDB 默认
- 串行化(Serializable):全部解决,但并发度最低

#### 3. MySQL InnoDB 默认 RR、PostgreSQL 默认 RC,为什么

这是数据库设计的经典差异,根源在历史包袱与实现哲学。

MySQL InnoDB 选 RR 作为默认,根本机制在于 InnoDB 的 RR 在 SQL 标准 RR 基础上额外用间隙锁解决了大部分幻读。配合早期 binlog 的 STATEMENT 格式主从复制,RR 能保证主库执行顺序与从库回放顺序一致——这一点对早期 MySQL 主从复制至关重要,RC + STATEMENT 格式会导致从库数据不一致。这是历史包袱下的工程权衡。

PostgreSQL 选 RC 作为默认,因为 PostgreSQL 的 MVCC 实现更激进——每条 SQL 都能拿到一致的快照,不需要像 InnoDB 那样维护复杂的回滚段。PostgreSQL 的设计哲学是"隔离级别应严格匹配 SQL 标准",RR 在 SQL 标准里允许幻读,PostgreSQL 的 RR 也允许;要彻底消除幻读应显式用 SSI(Serializable Snapshot Isolation,可串行化快照隔离)。PostgreSQL 没有 binlog 这种主从复制历史包袱,RC 不会带来复制问题。

实际生产中,越来越多互联网公司将 MySQL 从默认 RR 切到 RC。原因有二:RC 下间隙锁少、并发度高;RC 下 Next-Key Lock 退化为 Record Lock,死锁概率低。代价是必须把 `binlog_format` 改为 ROW,且业务代码要能接受"同一事务内两次读结果可能不同"。

#### 4. MVCC 为什么能提升并发

MVCC 的核心原理是:每行数据维护多个版本,事务读到的不是最新值,而是"该事务视图下可见的版本"。这样读操作不需要等写操作释放锁,写操作也不需要等读操作结束——读的是历史版本,写的是新版本。

InnoDB 的 MVCC 实现依赖三个组件:

- 隐藏字段:每行有 `DB_TRX_ID`(最近修改的事务 ID)、`DB_ROLL_PTR`(指向 undo log 的回滚指针)、`DB_ROW_ID`(行 ID)
- undo log 版本链:每次更新生成一条 undo log,通过 `DB_ROLL_PTR` 串成链表,链表头是当前值,越往后版本越老
- ReadView(一致性读视图):事务快照读时生成,记录"当前活跃事务列表",据此判断哪个版本对当前事务可见

RC 与 RR 的关键差异在于 ReadView 生成时机:

- RC:每条 SELECT 都生成新 ReadView,所以能看到该语句执行前已提交的修改——导致不可重复读
- RR:事务内第一条快照读生成 ReadView,整个事务复用——所以可重复读

注意一个常见误解:RR 下 ReadView 不是在 `BEGIN` 时生成,而是在事务内第一次执行快照读时才生成。若用 `START TRANSACTION WITH CONSISTENT SNAPSHOT`,则立即创建 ReadView。

#### 5. 死锁为什么不可避免

死锁(Deadlock)的本质是"两个事务互相持有对方想要的锁"。比如事务 A 锁住行 1 等行 2,事务 B 锁住行 2 等行 1,两者都不释放,永远等下去。

死锁不可避免的原因:数据库无法预知事务后续要锁哪些行。如果要求事务开始时声明所有要锁的行(两阶段锁定协议的严格版本),事务编程会变得极度不灵活,且很多业务场景下锁的范围是动态的(如按条件查询后锁定结果集)。

所以数据库只能走"检测 + 回滚"路线。InnoDB 默认开启 `innodb_deadlock_detect`,每次事务等待锁时检查等待图(wait-for graph)中是否有环,有环就选一个"代价最小"的事务回滚(通常是 undo log 量少的那个)。代价是高并发场景下死锁检测本身有 CPU 开销——这也是 MySQL 8.0 引入 `innodb_deadlock_detect=OFF` 选项的原因,关闭后用 `innodb_lock_wait_timeout` 超时回滚兜底。

### 三、实践

#### 1. 用两个 MySQL session 演示脏读、不可重复读、幻读

先建表造数据:

```sql
CREATE TABLE account (
  id INT PRIMARY KEY,
  name VARCHAR(20),
  balance INT
) ENGINE=InnoDB;

INSERT INTO account VALUES (1, 'alice', 100), (2, 'bob', 200);
```

**演示脏读(RU 级别)**。session A:

```sql
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
BEGIN;
SELECT * FROM account WHERE id = 1;
-- 看到 balance = 100
```

session B:

```sql
BEGIN;
UPDATE account SET balance = balance - 50 WHERE id = 1;
-- 未提交
```

回到 session A 再读:

```sql
SELECT * FROM account WHERE id = 1;
-- 看到 balance = 50(脏读:读到了 B 未提交的修改)
```

session B 回滚:

```sql
ROLLBACK;
```

session A 读到的 50 元从未真实存在过,这就是脏读的危害。

**演示不可重复读(RC 级别)**。session A:

```sql
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
BEGIN;
SELECT * FROM account WHERE id = 1;
-- balance = 100
```

session B:

```sql
BEGIN;
UPDATE account SET balance = balance + 50 WHERE id = 1;
COMMIT;
```

回到 session A 再读:

```sql
SELECT * FROM account WHERE id = 1;
-- balance = 150(不可重复读:同一事务内两次读结果不同)
COMMIT;
```

**演示幻读(RC 级别)**。session A:

```sql
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
BEGIN;
SELECT COUNT(*) FROM account WHERE balance > 100;
-- 1(只有 bob)
```

session B:

```sql
BEGIN;
INSERT INTO account VALUES (3, 'carol', 150);
COMMIT;
```

回到 session A 再读:

```sql
SELECT COUNT(*) FROM account WHERE balance > 100;
-- 2(多了 carol,幻读)
COMMIT;
```

把 session A 改成 RR 级别重做上述两个实验,会发现不可重复读和幻读都被解决——这就是 InnoDB 默认 RR 的价值。注意 RR 下幻读只是"快照读"被解决,"当前读"(`SELECT ... FOR UPDATE`、`UPDATE`、`DELETE`)仍可能看到新行,需配合 Next-Key Lock 才能彻底消除。

#### 2. 演示死锁场景(交叉更新)

死锁最典型的场景是两个事务以相反顺序更新同一批行。准备数据:

```sql
CREATE TABLE stock (
  id INT PRIMARY KEY,
  qty INT
) ENGINE=InnoDB;
INSERT INTO stock VALUES (1, 10), (2, 20);
```

session A:

```sql
BEGIN;
UPDATE stock SET qty = qty - 1 WHERE id = 1;  -- 锁住 id=1
```

session B:

```sql
BEGIN;
UPDATE stock SET qty = qty - 1 WHERE id = 2;  -- 锁住 id=2
```

session A 继续:

```sql
UPDATE stock SET qty = qty + 1 WHERE id = 2;  -- 等 B 释放 id=2 的锁
```

session B 继续:

```sql
UPDATE stock SET qty = qty + 1 WHERE id = 1;  -- 等 A 释放 id=1 的锁
-- ERROR 1213 (40001): Deadlock found when trying to get lock;
-- try restarting transaction
```

InnoDB 检测到死锁,主动回滚 session B(undo log 量较少的那个)。session A 随后能拿到 id=2 的锁继续执行。排查死锁用:

```sql
SHOW ENGINE INNODB STATUS\G
```

输出中的 `LATEST DETECTED DEADLOCK` 段会记录死锁发生时两个事务各自持有什么锁、等什么锁、执行的最后一条 SQL。这是定位死锁根因的关键证据。

预防死锁的工程实践:

- 同一事务内按固定顺序加锁(如按主键升序更新多行)
- 大事务拆小事务,缩短锁持有时间
- 必要时用 `SELECT ... FOR UPDATE` 提前显式加锁,把"隐式锁升级"变成"显式锁声明"
- 业务层捕获 1213 错误后短退避重试

#### 3. 演示 SELECT ... FOR UPDATE 排他锁

`SELECT ... FOR UPDATE` 是当前读,对查询命中的行加 X 锁,其他事务的当前读和写都会阻塞,但快照读不受影响。典型场景是"扣减库存前先锁住这一行"。

session A:

```sql
BEGIN;
SELECT * FROM stock WHERE id = 1 FOR UPDATE;
-- 拿到 id=1 的 X 锁,准备扣减
```

session B:

```sql
BEGIN;
SELECT * FROM stock WHERE id = 1 FOR UPDATE;
-- 阻塞,等 A 释放
```

session A 完成扣减并提交:

```sql
UPDATE stock SET qty = qty - 1 WHERE id = 1;
COMMIT;
```

session B 此时才能拿到锁继续执行。这正是 MVCC"读不阻塞写、写不阻塞读"的边界——快照读走 MVCC 不阻塞,但"写写"仍要串行。

`FOR UPDATE` 在不同隔离级别与版本下行为有差异,这是 MySQL 5.7 与 8.0 之间一个值得注意的演进点:

- MySQL 5.7 的 RR 下,`SELECT ... FOR UPDATE` 命中的行加 Next-Key Lock(行锁 + 行前面的间隙锁),范围查询会锁住较多间隙
- MySQL 8.0 引入了 `SKIP LOCKED`(跳过被锁的行)和 `NOWAIT`(不等待直接报错),用于实现任务队列场景下的"抢占式取任务"

```sql
-- MySQL 8.0+:取一个未锁的待处理任务,跳过已被其他 worker 锁住的
SELECT * FROM task_queue WHERE status = 'pending' LIMIT 1 FOR UPDATE SKIP LOCKED;
```

这段 SQL 是用 MySQL 实现轻量任务队列的标准写法,避免了引入 Redis 等外部依赖。

#### 4. 慢查询分析:show processlist 与 performance_schema

锁等待和长事务最终都会表现为慢查询。两个工具是排查主线。

`SHOW PROCESSLIST` 看当前所有连接在干什么:

```sql
SHOW PROCESSLIST;
```

关键字段:`State` 为 `Waiting for table metadata lock` 说明被元数据锁阻塞(常见于长事务持有表的 MDL 锁,阻塞后续 DDL);`State` 为 `updating` 且 `Time` 很长,说明在等行锁。

MySQL 8.0 推荐用 `performance_schema` 拿更精确的锁等待数据:

```sql
SELECT
  EVENT_NAME,
  COUNT_STAR AS wait_count,
  SUM_TIMER_WAIT / 1000000000 AS total_wait_ms
FROM performance_schema.events_waits_summary_global_by_event_name
WHERE EVENT_NAME LIKE 'wait/lock/row/innodb%'
ORDER BY total_wait_ms DESC;
```

要查具体哪条 SQL 在等谁,用 `sys.innodb_lock_waits` 视图(MySQL 8.0 内置,5.7 需手动开启 `performance_schema`):

```sql
SELECT
  waiting_pid, waiting_query,
  blocking_pid, blocking_query
FROM sys.innodb_lock_waits;
```

输出会直接告诉你"PID 1234 在等 PID 5678 的锁",配合 `KILL 5678` 可以快速止血。这套工具链要提前写进运维 runbook,线上告警时按手册执行即可。

### 四、速查/自测

#### 隔离级别 vs 并发问题对照表

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | InnoDB 实现要点 |
|---|---|---|---|---|
| Read Uncommitted(RU) | 允许 | 允许 | 允许 | 不加锁、不生成 ReadView |
| Read Committed(RC) | 解决 | 允许 | 允许 | 每条 SELECT 生成新 ReadView |
| Repeatable Read(RR,MySQL 默认) | 解决 | 解决 | 大部分解决 | 事务首次快照读生成 ReadView + Next-Key Lock |
| Serializable | 解决 | 解决 | 解决 | 普通 SELECT 隐式转 `LOCK IN SHARE MODE` |

#### 自测题

**问题一(原理层):** ACID 中"一致性"由谁保证?为什么不能只靠数据库?

参考答案:一致性是目标,由原子性 + 隔离性 + 持久性 + 业务约束共同保证。数据库约束(主键、外键、唯一索引、CHECK)只能保证结构合法,业务语义合法(如"转账金额不能超过余额""订单状态机只能从待支付跳到已支付")必须靠应用层代码。把一致性完全甩给数据库是常见误解。

**问题二(实践层):** 一个事务在 RR 下先 `BEGIN`,隔 10 分钟才执行第一条 `SELECT`,ReadView 何时生成?这会导致什么现象?

参考答案:ReadView 在该 SELECT 执行时生成,不是 `BEGIN` 时。这意味着这 10 分钟内其他事务提交的修改,本事务的快照读能看到——因为 ReadView 还没生成,没有"快照"的概念。如果想从 `BEGIN` 起就锁住视图,用 `START TRANSACTION WITH CONSISTENT SNAPSHOT`,它会立即创建 ReadView。这是数据导出场景常用技巧。

**问题三(思路层):** 公司要把 MySQL 从 RR 切到 RC,需要做哪些配套改动?有哪些风险?

参考答案:三件事必须做。第一,`binlog_format` 必须从 STATEMENT 改为 ROW,否则 RC + STATEMENT 会导致主从数据不一致(这是历史教训)。第二,审计业务代码中是否有"同一事务内多次读同一行期望结果一致"的逻辑,RC 下这类逻辑会失效。第三,间隙锁减少后并发度提升但死锁形态变化——Next-Key Lock 退化为 Record Lock,范围查询的锁范围变小,可能出现新的"丢失更新"模式。风险点是跨团队协调成本高,建议灰度切换、先在从库验证。

**问题四(实践层):** 写一段 SQL,演示用 `SELECT ... FOR UPDATE` 锁住 `account` 表中 `id = 1` 的行,并说明如果另一个事务想读这行(普通 SELECT)会发生什么、想写这行(UPDATE)会发生什么。

参考答案:

```sql
BEGIN;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 业务逻辑
UPDATE account SET balance = balance - 100 WHERE id = 1;
COMMIT;
```

另一个事务的普通 `SELECT * FROM account WHERE id = 1` 不受影响——因为普通 SELECT 是快照读,走 MVCC,读历史版本。但 `UPDATE account SET balance = balance + 1 WHERE id = 1` 会阻塞,因为 UPDATE 是当前读,需要 X 锁,等本事务 COMMIT 后才能继续。这正是 MVCC"读不阻塞写、写不阻塞读"的体现——但"写写"仍要串行。

**问题五(思路层):** InnoDB 的 RR 号称"大部分解决幻读","大部分"指什么?什么情况下仍会出现幻读?

参考答案:快照读场景下幻读被解决——因为整个事务复用同一份 ReadView,新插入的行版本号大于 ReadView 的高水位,不可见。当前读场景下(`SELECT ... FOR UPDATE`、`UPDATE`、`DELETE`)会触发 Next-Key Lock 锁住间隙,这本身能阻止其他事务在间隙内插入,所以也基本不出现幻读。"大部分"剩下的那一小部分场景,是事务内先快照读、后当前读的混合场景——先 `SELECT` 看到一批行,再 `SELECT ... FOR UPDATE` 可能因锁范围变化看到新行。彻底消除需在第一次读就用 `FOR UPDATE`,或全程用 Serializable。

### 可交给 AI 的部分

这一章里,**隔离级别选型表、锁类型对照表、长事务排查 SQL、死锁排查 SQL** 这几块可以放心交给 AI 生成。给定具体数据库版本和业务场景,AI 能准确给出推荐隔离级别、配套 `binlog_format` 设置、对应的 `information_schema` / `performance_schema` 查询语句,准确率较高。`SELECT ... FOR UPDATE SKIP LOCKED` 实现任务队列这类标准模式,AI 产出的代码可用率也很高。

但有几类内容**必须自己掌握、不能盲信 AI**,边界与风险如下。

第一类是死锁根因分析。AI 能给你死锁的标准示例代码,但真实业务里死锁的根因往往是"业务流程图里两个事务以相反顺序加锁",这种跨事务、跨代码模块的因果链,AI 看不到全貌。`SHOW ENGINE INNODB STATUS` 输出的死锁日志要自己读,对照业务代码画锁等待图,才能定位是哪两条 SQL 交叉。AI 生成"预防死锁"的通用建议(按主键顺序加锁、缩短事务)是对的,但具体哪段代码违反了,AI 判断不了。

第二类是 RC vs RR 的切换决策。AI 会列出两者的区别,但"你们公司该不该切"涉及业务代码审计、主从复制架构、灰度策略,这些 AI 无从知晓。盲目按 AI 建议切到 RC 而忘了改 `binlog_format=ROW`,会导致主从数据不一致——这种坑要自己把关。

第三类是慢查询与锁等待的现场处置。线上 CPU 飙升时,`SHOW PROCESSLIST` 看到 200 个连接都在 `Waiting for table metadata lock`,根因可能是某个长事务持有了 MDL 锁。这种现场判断要快、要准,AI 来不及响应,必须 DBA 自己上手 `KILL` 掉根因事务。`performance_schema` 的查询语句可以让 AI 提前帮你写好放进 runbook,但执行判断必须是人。

第四类是 MVCC 与锁的边界推理。给定一段并发 SQL,预测它会出现脏读、不可重复读还是幻读,需要同时推理隔离级别、ReadView 时机、锁范围三件事。AI 在简单场景下能答对,复杂场景(如混合快照读和当前读、间隙锁范围跨索引)容易出错。这类推理是面试和线上排障的高频考点,必须自己练熟。

区分"能交"与"不能交"的本质是:**SQL 模板和排查脚本是机械劳动,AI 强;跨事务因果分析、版本切换的架构决策、现场处置的快速判断是语义推理,AI 弱**。把机械劳动交给 AI 写进 runbook,把语义推理留给自己练成肌肉记忆,这是 AI 时代 DBA 与后端工程师的知识边界。
