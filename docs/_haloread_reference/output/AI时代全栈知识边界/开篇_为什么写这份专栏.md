# AI时代全栈知识边界·01|为什么写这份专栏

> 前置知识:无,本章是专栏起点
> 学完你能:说清 AI 时代全栈开发「必须掌握」与「可交给 AI」的边界判定标准,知道哪些知识甩不出去、为什么甩不出去,并能用这个标准规划自己的学习精力与工程责任

### 一、概念

**AI 时代全栈知识边界**,指在 AI 辅助开发已成为常态的当下,开发者主动划定「必须自己掌握」与「可交给 AI 实现」两部分知识的分界线,并据此分配学习精力与工程责任的一种认知框架。

它是「边界」,**不是**「清单」。清单告诉你学什么,边界告诉你哪些是你甩不出去的责任。它也**不是**「反 AI 手册」——本专栏默认你每天都在用 AI 写代码,讨论的只是哪些知识你一旦外包就会出事。

边界划定的核心标准是**责任归属**:凡是出了 bug 必须由你来定位、由你来向团队解释、由你来承担线上后果的部分,就是「必须掌握」;凡是可以靠生成、靠模板、靠工具链自动完成且失败成本低的部分,就是「可交给 AI」。这条标准与传统「全栈工程师要会所有东西」的要求表面相似,本质不同。传统全栈强调覆盖广度,AI 时代全栈强调**判断深度**——你不需要记得每一个 API 的参数顺序,但必须知道这段 AI 生成的代码在什么条件下会崩。

了解了边界的定义,下一步看这条边界为什么在 AI 时代反而变得更清晰而不是更模糊。

### 二、原理

#### 为什么 AI 时代反而需要更清晰的边界

AI 代码生成有一个根本特征:**生成成本与验证成本的不对称**。让 AI 写一段 200 行的 Python 异步任务调度器,只需要几秒钟和一句 prompt(提示词);但验证这段代码在并发场景下没有 race condition(竞态条件)、在异常路径下不会泄漏资源、在生产负载下不会撑爆连接池,可能需要你花两小时读代码、写测试、跑压测。

这种不对称带来一个直接后果:AI 让「写出来」变得几乎免费,但让「确认它是对的」变得比以前更贵。以前你自己写的代码,你在写的过程中已经把大部分边界条件想过一遍;现在 AI 给你的代码,你完全没有这个心智预热,验证时的认知负担反而更高。

Andrej Karpathy(安德烈·卡帕西)在 2017 年提出 Software 2.0 概念,指神经网络权重成为一种新的「程序」[1]。2024 年起他在多次公开演讲中把这一思路推进到 Software 3.0:大语言模型让自然语言成为一种可执行的编程语言,英文或中文 prompt 正在变成新的「源代码」[2]。但他在同一批演讲里反复强调一点:**架构判断与最终责任仍在人这一侧**。模型可以生成代码,但它不会替你回答「这个系统该不该上微服务」「这个事务边界划在哪」「这个数据该不该缓存」。

理解 Software 3.0 的关键,不是去争论「自然语言算不算编程语言」,而是看清它改变了什么、没改变什么。改变的是「实现成本」——把一个想法变成可运行代码的门槛被压到接近零;没改变的是「判断成本」——决定这个想法对不对、该不该做、做了会出什么事,仍然是人的工作。门槛降低会让更多人写更多代码,也会让更多未经判断的代码流入生产环境,这正是「必须掌握」在 AI 时代反而更重要的根本原因。

#### 责任不可外包的根本机制

软件开发中有一类决策叫**不可逆决策**(one-way door decision),与之相对的是可逆决策(two-way door)。架构选型、数据模型设计、安全边界划定,大部分属于不可逆——一旦上线、一旦有用户数据,改动的代价是迁移成本加停机风险加团队重新学习成本。

AI 擅长处理可逆决策:函数实现、UI 样式、测试用例、文档草稿——这些错了改回来就行,成本可控。但 AI **不擅长**不可逆决策,原因不是它不够聪明,而是它**看不到你的业务约束**。它不知道你的用户量级、不知道你的团队结构、不知道你的合规要求、不知道你的预算上限。这些约束只存在于你脑子里和团队的隐性共识里,AI 拿不到,自然也就无法替你做这类决策。

这就是「必须掌握」的本质:不是这些知识 AI 学不会,而是**承载这些知识的上下文 AI 拿不到**。知识边界,实际上是上下文边界。

#### 一个量化视角:错误放大系数

设一段代码的「错误放大系数」(error amplification coefficient)为它出 bug 后的影响半径与修复成本的乘积。一段 CRUD 接口代码出 bug,影响半径是一个接口,修复成本是改几行;一段数据库 schema(模式)设计出 bug,影响半径是整个应用,修复成本是一次数据迁移。前者放大系数低,后者高。

「必须掌握」的标准可以近似写成:**放大系数 > 你团队的容错阈值**。这个阈值因团队、因业务、因阶段而异,但量级判断是通用的:涉及数据一致性、安全、可用性的部分,放大系数普遍高,必须掌握;涉及展示、交互、模板生成的部分,放大系数普遍低,可以交给 AI。

原理讲清了,下面用一个真实场景看这条边界在实际开发里怎么落地。

### 三、实践

#### 场景一:AI 生成的代码出了 bug,你能不能定位

这是本专栏所有讨论的起点。假设你让 AI 写一段 Redis 分布式锁的实现,它给了你下面这段(Python 3.10+ + redis-py 5.x):

```python
import asyncio
import redis.asyncio as redis

class DistributedLock:
    def __init__(self, client: redis.Redis, name: str, ttl: int = 10):
        self.client = client
        self.name = name
        self.ttl = ttl
        self.identifier = None

    async def acquire(self) -> bool:
        import uuid
        self.identifier = str(uuid.uuid4())
        result = await self.client.set(self.name, self.identifier, nx=True, ex=self.ttl)
        return bool(result)

    async def release(self):
        await self.client.delete(self.name)
```

这段代码能跑,单机测试也通过。但它有三个 bug,每一个都可能在生产环境炸:

第一个,`release` 没有校验 `identifier`,任何拿到锁的客户端都能被别的客户端误删——经典的「删别人的锁」问题。第二个,`ttl` 到期后锁自动释放,但持有者并不知道,可能出现两个客户端同时以为自己持有锁。第三个,`acquire` 失败时没有重试,调用方需要自己写重试逻辑,但 AI 的注释里没说,你以为它「就是设计成一次性获取」。

如果你不懂 Redis 的 `SET NX EX` 语义、不懂分布式锁的释放必须用 Lua 脚本保证原子性、不懂 TTL 续期的必要性,这三个 bug 你一个都看不出来。你会把这段代码合并进主干,然后在某个凌晨被报警叫醒。

这个场景就是「必须掌握」的全部含义:**AI 能给你一段看起来合理的代码,但只有你能判断它是不是真的对**。判断的前提是你脑子里有 Redis 分布式锁的正确性证明,而不是你记得 `set` 方法的参数顺序。

正确的实现应该是这样,关键修复点在 `release` 用 Lua 脚本做条件删除、`_renew` 做 TTL 续期:

```python
import uuid
import asyncio
import redis.asyncio as redis

_RELEASE_SCRIPT = """
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
"""

class DistributedLock:
    def __init__(self, client: redis.Redis, name: str, ttl: int = 10):
        self.client = client
        self.name = name
        self.ttl = ttl
        self.identifier: str | None = None
        self._renew_task: asyncio.Task | None = None

    async def acquire(self, retry: int = 3, interval: float = 0.1) -> bool:
        self.identifier = str(uuid.uuid4())
        for _ in range(retry):
            ok = await self.client.set(self.name, self.identifier, nx=True, ex=self.ttl)
            if ok:
                self._renew_task = asyncio.create_task(self._renew())
                return True
            await asyncio.sleep(interval)
        return False

    async def _renew(self):
        while True:
            await asyncio.sleep(self.ttl / 2)
            await self.client.expire(self.name, self.ttl)

    async def release(self):
        if self._renew_task:
            self._renew_task.cancel()
        await self.client.eval(_RELEASE_SCRIPT, 1, self.name, self.identifier)
```

这段代码你可以放心交给 AI 去生成吗?可以,但前提是**你能在 30 秒内指出三个修复点分别解决什么问题**。如果指不出来,这部分知识就是你的「必须掌握」,而不是「可交给 AI」。

#### 场景二:架构判断为什么 AI 替不了

同样是分布式锁这个例子,再往上一层的问题是:**你这个业务到底该不该用分布式锁**。

假设你在做一个「用户秒杀抢券」场景,商品库存 1000 份,预计 10 万并发。AI 会很乐意给你写一段 Redis 分布式锁代码,但它不会主动告诉你:

- 用分布式锁会让所有请求串行化,吞吐量被锁的粒度卡死
- 秒杀场景更适合用 Redis 原子扣减(`DECR`)或 Lua 脚本一次性完成「判断 + 扣减」
- 更激进的做法是预热库存到本地内存,用本地原子操作加异步回补

这些判断需要你知道**业务的并发量级**、**数据一致性要求**、**团队对复杂度的承受能力**。AI 给不了你这些输入,因为它根本不知道你的业务是什么。架构判断的不可替代性,就在这里——它不是代码层面的「会不会写」,而是上下文层面的「该不该这么选」。

这两层加起来,正好对应本专栏要拆的边界:代码实现层(场景一)你能交给 AI,但前提是你看得懂、改得了;架构选型层(场景二)你交不出去,因为承载决策的业务上下文只在你脑子里。本专栏 14 大模块的每一章,都会落在这样一条分界线上——告诉你哪一层必须自己懂、哪一层可以让 AI 动手,以及两边交接时你要检查什么。

了解了边界在实践里怎么落地,下面用一张速查表和几道自测题把判定标准固化下来。

### 四、速查/自测

#### 「必须掌握」与「可交给 AI」速查表

| 维度 | 必须掌握 | 可交给 AI |
|---|---|---|
| 数据模型 | schema 设计、索引策略、事务边界 | CRUD SQL、ORM 映射 |
| 并发控制 | 锁的正确性证明、race condition 识别 | 锁的代码实现、重试模板 |
| 安全 | 鉴权流程、输入校验边界、密钥管理 | XSS/CSRF 防御代码、过滤器 |
| 架构 | 服务拆分、数据一致性边界、容灾策略 | 接口定义、DTO 转换、文档草稿 |
| 网络 | 协议语义(TCP/HTTP)、超时与重试边界 | 抓包脚本、curl 命令 |
| 算法 | 复杂度判断、正确性证明、边界 case | 标准算法实现(排序、查找) |
| 运维 | 部署拓扑、监控指标定义、回滚策略 | Dockerfile、CI 配置、告警模板 |
| AI 工程 | prompt 设计原则、评估方法、幻觉识别 | 调用代码、解析逻辑、流式输出 |

#### 自测题

**原理层**

Q1:为什么「生成成本与验证成本不对称」会让开发者的认知负担不降反升?

<details><summary>答案</summary>

因为自己写代码时,边界条件在写作过程中已被思考一遍;AI 生成代码时,这部分思考被跳过,全部转移到验证阶段。而验证一段陌生代码比写一段代码的认知负担更高——你需要先理解别人的意图,再判断对错,双重负荷叠加。所以 AI 省下的「写」的时间,被「验」的时间吃掉了大半,净收益没有想象中大。

</details>

**思路层**

Q2:判断一段知识属于「必须掌握」还是「可交给 AI」,核心标准是什么?为什么用「会不会」而不是「难不难」作为标准?

<details><summary>答案</summary>

核心标准是责任归属与错误放大系数。用「会不会」而不是「难不难」,是因为难的知识如果失败成本低、可快速回滚,仍可以交给 AI(如复杂正则);简单的知识如果失败后果严重(如密码学随机数生成),仍必须自己掌握。难度与责任是两个维度,不能用难度替代责任。

</details>

**实践层**

Q3:下面这段 AI 生成的 JWT 校验代码(Python 3.10+ + PyJWT 2.x)有哪些必须自己看出来的问题?

```python
import jwt

def verify_token(token: str, secret: str) -> dict:
    return jwt.decode(token, secret, algorithms=["HS256", "none"])
```

<details><summary>答案</summary>

至少两个问题。第一,`algorithms` 列表里包含 `"none"`,这是经典的 JWT 算法混淆漏洞——攻击者可以把 token 的 alg 改成 none 绕过签名校验。第二,没有校验 token 的 `exp`/`iat`/`aud` 等声明,过期 token 仍可用。正确做法是只允许预期的算法(如 `["HS256"]`),并显式校验相关声明。这类安全相关的知识,就是典型的「必须掌握」,因为它的 bug 不会在功能测试里暴露,只会在被攻击时暴露。

</details>

### 可交给 AI 的部分

本章是开篇,真正「可交给 AI」的部分不多,但有几件事可以放心外包。

**可交**:本专栏后续章节的代码示例,凡是有明确输入输出、有官方文档对照、失败后可以肉眼看出错的,都可以让 AI 先生成、你做 review。比如 CRUD 接口、Dockerfile、单元测试模板、正则表达式。这类内容错误暴露快、修复成本低,把 AI 当成「会出错的初级工程师」用就行。能交的根本原因是「可逆」——错了改回来代价小,不触及不可逆决策。

**可交**:本章速查表的初稿。让 AI 根据你给的维度先填一遍,你再根据自己团队的经验修正。AI 在结构化产出上比人快,但在「我们团队真正在意什么」上比人差。

**不能交**:你团队当前阶段的「知识边界」本身。哪些系统是核心链路、哪些可以容忍短暂故障、哪些数据一旦泄露就是事故——这些判断必须你自己做,AI 给不了答案。原因前面讲过:承载这些判断的上下文不在 AI 的训练数据里,而在你和团队的日常工作里。所以「必须掌握 vs 可交给 AI」这张表,可以借 AI 的框架,但最终的阈值必须你来填。能交与不能交的分界,本质就是「上下文在不在你脑子里」。

**风险提示**:把代码生成交给 AI 时,最大的风险并非「AI 写错了」,而是「AI 写错了你也没发现」。AI 写错的代码风格上往往非常干净、注释齐全、看起来很专业,这种「看起来对」比明显错误的代码更危险,因为它会降低你的警觉。应对办法只有一个:对 AI 生成的代码,review 时的标准要**比人写的更高**,而不是更低。人写的代码你会下意识怀疑,AI 写的代码你会下意识信任,这种信任偏差是 AI 时代最大的工程风险。这也是为什么本专栏把「必须掌握」段讲到可直接开发程度——只有你真懂,才能在 AI 给你一段「看起来对」的代码时,看出它哪里不对。
