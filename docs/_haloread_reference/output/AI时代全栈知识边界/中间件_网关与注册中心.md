# AI时代全栈知识边界·21|网关与注册中心

> 前置知识:理解 HTTP 请求/响应模型、知道进程间通信需要寻址、用过 Nginx 或任意反向代理
> 学完你能:① 说清网关统一处理哪些横切关注点 ② 区分反向代理与正向代理、客户端发现与服务端发现 ③ 写出可运行的 Nginx 反向代理配置与 Spring Cloud Gateway 路由 YAML ④ 解释注册中心如何做服务注册、健康检查、配置推送 ⑤ 讲清 Zookeeper 的 ZAB 协议为何能保证一致性 ⑥ 在 Nacos/Eureka/Consul/etcd 间做选型

### 一、概念

API Gateway(应用编程接口网关)是微服务架构中所有外部请求的统一入口,在客户端与后端服务之间充当"前台接待"——把路由转发、鉴权、限流、熔断、日志、协议转换这些每个服务都要做一遍的横切关注点(cross-cutting concerns)集中到一处。Service Registry(服务注册中心)则是服务实例的"电话簿":服务启动时登记地址,消费方按服务名查询再调用,使"调用方写死对方 IP"这种脆弱耦合被消除。

两者常成对出现,共同回答"请求怎么找到对的服务":网关解决"外部请求往哪转",注册中心解决"转进去后哪个实例接"。它们让业务服务专心写业务,把寻址、转发、容错交给基础设施。

主流网关各有定位:Nginx 是通用反向代理,配置驱动、性能强但自身无服务发现;Spring Cloud Gateway 是 Java 生态声明式网关,与 Spring Cloud 注册中心深度集成;Kong 基于 OpenResty,插件生态丰富;Traefik 云原生自动配置,是 Kubernetes Ingress 的常用选择。

需要先对齐几组术语。反向代理(Reverse Proxy)代理服务端,客户端不知真实后端;正向代理(Forward Proxy)代理客户端,服务端不知真实客户端。Service Discovery(服务发现)分客户端发现(client-side discovery,调用方自查注册中心选实例调用)与服务端发现(server-side discovery,由中间代理查注册中心,调用方只访问固定地址)两种。Health Check(健康检查)判断实例存活,有心跳、主动探测、TTL 租约三种实现。

### 二、原理

#### 1. 网关为什么能统一处理横切关注点

如果没有网关,每个微服务都得自己实现鉴权、限流、日志、HTTPS 终止。假设 20 个服务,这套代码写 20 遍——更糟的是会分叉:限流策略改了只有 5 个服务跟上,另 15 个用旧策略,事故就此埋下。

网关的本质是把"所有请求必经的公共逻辑"上移到入口层,这是"约束胜于约定"的设计:既然每个服务都要做,不如强制只在一个地方做。网关处在请求链路咽喉位置——外部请求先到网关,做完鉴权、限流、改写后再转发,后端拿到的请求已是"被预处理过"的,可假定前述逻辑都已完成。

网关的核心能力可拆为六项:

- 路由(Routing):按路径、Host、Header 把请求转发到不同后端。`/api/order/*` 走订单服务,`/api/user/*` 走用户服务。
- 鉴权(Authentication):统一校验 JWT、OAuth 令牌,验签通过才放行,后端无需重复校验。
- 限流(Rate Limiting):按 IP、用户、接口维度做 QPS 限制,超过阈值直接返回 429。
- 熔断(Circuit Breaking):后端错误率超阈值时快速失败,防止故障扩散,本质是"用拒绝请求来保护后端"。
- 协议转换(Protocol Translation):外部 HTTP 调用内部 gRPC,或做 TLS 卸载,网关完成编解码桥接。
- 日志与监控(Logging):统一记录访问日志、调用链 Trace ID,所有请求的可观测性在一处收敛。

前四项是"过滤",后两项是"加工"。判断"该加在网关还是后端"的标准:凡是所有服务都要做且策略一致的,放网关;只有某个服务关心的,放后端。

#### 2. 反向代理与正向代理:谁是"被代理"的一方

这对概念容易混淆,根本机制在于"代理谁"。正向代理代理客户端:典型如公司内网通过 Squid 访问外网,客户端把请求发给代理,代理替客户端访问目标服务器,服务器看到的是代理 IP——作用是突破访问限制、缓存、隐藏客户端。反向代理代理服务端:典型如 Nginx 部署在服务集群前,客户端以为访问 `example.com`,实际被 Nginx 转发到某台后端——作用是负载均衡、SSL 卸载、隐藏后端拓扑。

一句话区分:正向代理对服务端透明,反向代理对客户端透明。网关本质是"功能增强版的反向代理",在转发能力之上叠加鉴权、限流、协议转换等业务语义。

#### 3. 服务发现两种模式:谁来查注册中心

服务发现要解决的问题是调用方怎么知道被调方地址,两种模式的根本差异在于"查注册中心"由谁做。

客户端发现模式下,调用方进程内嵌客户端库(如 Eureka + Ribbon),启动时拉取实例列表缓存本地,调用时按负载均衡策略(轮询、随机、权重)选实例直接发起请求。优点是少一跳、延迟低;缺点是每种语言要维护一份发现库,负载均衡逻辑散落在各进程难以统一治理。

服务端发现模式下,调用方不直接查注册中心,而是把请求发到固定的负载均衡器或服务代理,由中间层查注册中心并转发,调用方无感知注册中心。Kubernetes Service 是典型:Pod 注册到 etcd,Service 通过 kube-proxy 转发,业务容器只看到 Service 虚拟 IP。优点是多语言友好、治理集中;缺点是多一跳、中间层要高可用。

选型上,云原生(Kubernetes)默认服务端发现(平台已内置);自建微服务(尤其 Java 单语言栈)常用客户端发现,省一跳且 Ribbon 库成熟。

#### 4. 注册中心如何保证"名单不乱"

注册中心存的是"服务名 → 实例地址列表",核心难点不是存,而是一致性与可用性的权衡。实例宕机迟迟不摘,调用方往死实例发请求;反之实例活着却被误摘,调用方少一个可用节点。

健康检查是这份数据"保鲜"的机制,有三种实现:

- 心跳(Heartbeat)上报:实例定期(如 30 秒)主动向注册中心发"我还活着"。Nacos、Eureka 用这种,超过若干周期没收到心跳则判定下线。实现简单、对注册中心压力小,代价是有探测延迟。
- 主动探测(Active Probe):注册中心定期向实例的健康检查端点(如 `/actuator/health`)发请求,不通则摘除。Consul 倾向这种,发现更及时,但实例多时探测压力大。
- TTL 租约(Lease):etcd 用这种,实例拿到带 TTL(如 10 秒)的租约定期续租,租约过期则对应 key 自动删除——本质是心跳的另一种实现,把"存活"绑定到 key 生命周期,摘除由 etcd 自动完成,无需主动探测。

配置中心与注册发现共享同一套推送通道。Nacos 把"服务发现"和"配置管理"合二为一:配置变更后,服务端通过长轮询(long polling)或 gRPC 长连接把新配置推到所有订阅客户端,客户端本地缓存并热更新。这回答了"为什么配置中心常与注册中心合体"——两者都需要"服务端感知变更、客户端实时同步"的推送通道,合体复用基础设施更经济。

#### 5. Zookeeper 与 ZAB 协议:为什么能保证一致

Zookeeper 常被当作注册中心(早期 Dubbo 默认用它)和分布式协调工具,核心能力是配置同步、分布式锁、Leader 选举。这三项能力的根本机制是对"写入顺序"的强保证:所有写请求经 Leader 串行化,以两阶段提交广播到 Follower,半数以上确认才算写入成功。

这一保证由 ZAB 协议(ZooKeeper Atomic Broadcast,原子广播协议)实现。ZAB 把每次写入编号成全局递增的 zxid(事务 ID),Leader 用类两阶段提交把提案发给所有 Follower,过半数确认后发 commit,所有节点按 zxid 顺序应用。崩溃恢复时"选举拥有最大 zxid 的节点为新 Leader"——已提交事务必然已被过半数节点持久化,新 Leader 一定拥有它们,从而保证已提交不丢、未提交不残留。

需要厘清 ZAB 与 Raft、Paxos 的关系:三者都是共识算法(Consensus Algorithm),目标都是让一组节点对一组值达成一致。ZAB 是 Zookeeper 专用,强调主备顺序广播;Raft 更易理解,etcd 用它;Paxos 是理论基础,工程很少直接用。etcd(Raft)与 Zookeeper(ZAB)都满足 CP,差异更多在 API 易用性与生态。

Zookeeper 近年逐步被 Nacos、etcd 取代作注册中心,根本原因有二:其 CP 特性在网络分区时拒绝写入保一致,而注册中心场景"可用性"往往比"强一致"更重要(Eureka 选 AP 正基于此);其 API 是树形 znode,对服务发现不够直白,Nacos 直接提供"服务名→实例"语义更贴合业务。

#### 6. 搜索引擎 Elasticsearch:何时该用它而非数据库

Elasticsearch(常简称 ES)是分布式全文搜索引擎,基于 Lucene,用倒排索引(inverted index)加速文本检索。它常与注册中心、网关同属中间件层基础设施,职责是"让数据可被高效检索"。两个典型场景:一是全文检索——电商商品搜索、文档检索,要按相关性打分、支持分词与高亮,这是关系数据库 `LIKE '%关键词%'` 做不到的(走全表扫描且无法按相关性排序);二是日志分析——ELK 栈把海量日志写入 ES 做聚合统计,关系数据库在海量数据下撑不住。

ES 与关系数据库的选型区别,根本机制在索引结构:数据库用 B+ 树,擅长精确等值、范围查询与强事务;ES 用倒排索引,擅长分词文本匹配与聚合,但不支持跨文档事务、写入有近实时延迟(默认 1 秒刷新才可搜到)。常见误用是把 ES 当主库——它的更新是"标记删除 + 重新索引",频繁更新代价高,不适合做唯一真相源。正确姿势是关系数据库做主库、ES 做检索副本,通过 Canal 监听 binlog 或应用双写同步,ES 只服务读侧检索。

### 三、实践

#### 1. Nginx 反向代理配置:把外部请求转发到后端集群

下面是一段可运行的 Nginx 反向代理配置,把 `api.example.com` 的请求按路径分发到两组后端,带被动健康检查与限流。

```nginx
upstream order_service {
    # 后端订单服务集群,ip_hash 保证同一客户端落到同一后端(会话保持)
    ip_hash;
    server 10.0.0.11:8080 max_fails=3 fail_timeout=30s;
    server 10.0.0.12:8080 max_fails=3 fail_timeout=30s;
    server 10.0.0.13:8080 max_fails=3 fail_timeout=30s backup;
}

upstream user_service {
    server 10.0.0.21:8080 max_fails=3 fail_timeout=30s;
    server 10.0.0.22:8080 max_fails=3 fail_timeout=30s;
}

server {
    listen 443 ssl http2;
    server_name api.example.com;

    # TLS 证书卸载,后端走明文 HTTP,减轻后端加解密负担
    ssl_certificate     /etc/nginx/ssl/api.pem;
    ssl_certificate_key /etc/nginx/ssl/api.key;

    add_header X-Frame-Options "SAMEORIGIN" always;

    # 订单接口转发到 order_service 集群
    location /api/order/ {
        proxy_pass http://order_service;
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_connect_timeout 2s;
        proxy_read_timeout    10s;
    }

    # 用户接口转发到 user_service 集群
    location /api/user/ {
        proxy_pass http://user_service;
        proxy_set_header Host $host;
    }

    # 限流:按客户端 IP 限制 /api/login,突发排队(需在 http 块预声明 limit_req_zone)
    location /api/login {
        limit_req zone=login burst=20 nodelay;
        proxy_pass http://user_service;
    }
}

# 通常放在 nginx.conf 的 http 段:
# limit_req_zone $binary_remote_addr zone=login:10m rate=10r/s;
```

这段配置体现了网关几项核心能力:`location` 路由分发、`upstream` 负载均衡、`ssl` 做 TLS 卸载、`limit_req` 限流、`max_fails/fail_timeout` 被动健康检查。Nginx 是通用反向代理,本身不做服务发现——`upstream` IP 写死,扩缩容要改配置 reload。要具备服务发现能力,需配合 Consul Template 或 nginx-upsync-module 动态注入后端列表。

#### 2. Spring Cloud Gateway 路由配置:声明式路由 + 过滤器

Spring Cloud Gateway 是 Java 生态主流网关,基于 Reactor,用路由(Route)、断言(Predicate)、过滤器(Filter)三件套描述转发规则。下面 YAML 把 `/api/order/**` 转发到 order 服务,叠加限流与鉴权过滤器。

```yaml
spring:
  cloud:
    gateway:
      # 服务发现集成:开启后,route 的 uri 可直接写 lb://服务名,网关自动从注册中心解析实例
      discovery:
        locator:
          enabled: true
          lower-case-service-id: true
      routes:
        - id: order_service_route
          uri: lb://order-service          # lb:// 表示走客户端负载均衡,从 Nacos 拿实例列表
          predicates:
            - Path=/api/order/**           # 断言:路径匹配才走此路由
            - Method=GET,POST
            - Header=X-Api-Version, \d+    # 必须带版本号 Header
          filters:
            - StripPrefix=1                # 转发前剥掉第一段路径(/api/order/x -> /order/x)
            - name: RequestRateLimiter     # 限流过滤器,基于 Redis 令牌桶
              args:
                redis-rate-limiter.replenishRate: 100   # 令牌填充速率 100/秒
                redis-rate-limiter.burstCapacity: 200   # 桶容量 200
                key-resolver: "#{@userKeyResolver}"     # 按用户 ID 限流
            - name: JwtAuth                 # 自定义鉴权过滤器,校验 JWT 后把 userId 注入 Header
              args:
                headerName: X-User-Id
        - id: user_service_route
          uri: lb://user-service
          predicates:
            - Path=/api/user/**
          filters:
            - StripPrefix=1
```

注意 `uri: lb://order-service`——`lb://` 是 Spring Cloud Gateway 接入服务发现的标志,它从注册中心(Nacos/Eureka)拉取实例列表,再用 Spring Cloud LoadBalancer 选实例转发。这是"网关 + 注册中心"配合的典型形态:Nginx 版 IP 写死,Spring Cloud Gateway 版 IP 由注册中心动态供给,扩缩容对网关透明。`RequestRateLimiter` 用 Redis 做分布式令牌桶限流,跨多网关实例计数一致——这是分布式限流与单机限流的根本区别。

#### 3. 服务注册的概念流程:从启动到调用

服务注册发现是一套多方协作的流程,文字描述如下。

**注册阶段**:订单服务实例启动后,内嵌的 Nacos Client 向 Server 发 POST,Body 带 `{serviceName: "order-service", ip: "10.0.0.11", port: 8080}`。Server 把记录写进内存注册表(`Map<服务名, Set<实例>>`),并追加到最近变更队列。注册失败会重试,超过次数则启动失败——"宁可起不来,也不要起来却没注册"的工程取舍。

**发现阶段**:调用方启动时,Nacos Client 向 Server 发"订阅"长轮询:`GET /nacos/v1/ns/instance/list?serviceName=order-service`。Server 返回实例列表并把该客户端加入订阅者集合。之后订单服务扩容或宕机,Server 把变更事件通过长轮询推给所有订阅者,客户端本地缓存随之更新。这是"变更秒级推送"的根本机制。

**调用阶段**:网关收到 `/api/order/123`,匹配 `lb://order-service` 路由,从本地缓存实例列表按负载均衡策略选一个(如 `10.0.0.11:8080`)发起请求。整个过程网关不查注册中心——本地缓存就是事实来源,这是客户端发现"低延迟"的来源。若缓存实例恰好刚宕机(尚未被推送摘除),请求失败,网关靠重试或熔断兜底——这是客户端发现"可能打到死实例"的代价。

**下线阶段**:服务正常停止时,客户端发一次 deregister 请求主动摘除;异常宕机时,心跳中断,Server 等过保护期(默认约 15-30 秒)后自动摘除。两种路径都保证注册表最终与真实存活实例一致。

### 四、速查/自测

#### 注册中心选型对照表

| 注册中心 | 一致性模型 | 健康检查 | 典型生态 | 适用场景 |
|---|---|---|---|---|
| Nacos | AP/CP 可切换 | 心跳 + 主动探测 | Spring Cloud 阿里系 | 国内 Spring Cloud 项目,服务发现 + 配置一体 |
| Eureka | AP | 心跳 | Spring Cloud Netflix(已停更) | 老旧 Spring Cloud 项目,可用性优先 |
| Consul | CP(Raft) | 主动探测 | HashiCorp 体系 | 多数据中心、非 Java 主力栈 |
| etcd | CP(Raft) | TTL 租约 | Kubernetes、CoreOS | Kubernetes 内核、自建平台需强一致元数据存储 |
| Zookeeper | CP(ZAB) | 心跳(会话) | Dubbo、HBase | 早期 Dubbo 注册中心、分布式协调(锁、选举) |

选型核心判断:可用性优先选 AP(Nacos AP 模式/Eureka),强一致优先选 CP(etcd/Zookeeper/Consul)。服务发现场景 AP 往往更合适——"名单短暂不一致"比"注册中心不可用致全部调用失败"代价小得多。

#### 自测题

**问题一(原理层):** 反向代理和正向代理的根本区别是什么?Nginx 在两种角色下网络拓扑分别处在什么位置?

参考答案:根本区别在"代理谁"。正向代理代理客户端,对服务端隐藏真实客户端(典型:公司翻墙代理);反向代理代理服务端,对客户端隐藏真实后端(典型:Nginx 负载均衡)。Nginx 做正向代理配 `resolver` + `proxy_pass $scheme://$host$uri`,处内网出口;做反向代理配 `upstream` + `proxy_pass`,处服务集群入口。

**问题二(实践层):** 给定一段 Spring Cloud Gateway 路由配置,如何让它具备服务发现能力?`lb://` 前缀背后发生了什么?

参考答案:两步:引入注册中心依赖(如 `spring-cloud-starter-alibaba-nacos-discovery`)并开启 `discovery.locator.enabled=true`;路由 `uri` 用 `lb://服务名` 而非具体地址。`lb://` 触发 Spring Cloud LoadBalancer 从注册中心拉取实例列表缓存本地,按负载均衡策略选实例,再由 Netty 发起请求。实例扩缩容由注册中心推送,网关无感。

**问题三(原理层):** 为什么 Eureka 选 AP 而 Zookeeper 选 CP?这对注册中心场景意味着什么?

参考答案:Zookeeper 的 ZAB 在网络分区时,少数派分区无法完成"过半数确认",拒绝写入保强一致(CP),但少数派分区的服务无法续约可能被误摘。Eureka 用去中心化复制,分区时各分区仍可读写(AP),可能短暂返回过时名单但注册中心始终可用。注册中心场景下,"名单短暂不准"可由调用方重试/熔断兜底,而"注册中心不可用致全链路阻断"是灾难,所以 AP 取舍更贴合,这也是 Nacos 默认 AP 的原因。

**问题四(思路层):** 公司要从自建 Nginx 网关(写死 upstream IP)迁移到 Spring Cloud Gateway + Nacos,需要做哪些配套改动?有什么风险?

参考答案:三件事:后端服务接入 Nacos Client 完成注册(加依赖、配地址与服务名);把 Nginx 路由规则翻译成 Gateway 的 Route DSL,自定义过滤器用 Java 重写,`limit_req` 对应 `RequestRateLimiter` 但从单机改分布式(Redis 令牌桶),限流行为会变;TLS 卸载保留在前置 Nginx,Gateway 只处理明文降低 Java 侧开销。风险:Nacos 集群部署 ≥3 节点保高可用;客户端发现模式下死实例未及时摘除致偶发失败,须配套重试熔断;灰度期间两套网关并存要控流量切分。

**问题五(原理层):** Zookeeper 的 ZAB 协议如何保证已提交的事务在 Leader 崩溃后不丢?

参考答案:ZAB 每个事务带全局递增 zxid,Leader 提交前需过半数 Follower 确认。Leader 崩溃后,集群选"已提交 zxid 最大"的 Follower 当新 Leader——已提交事务必然已被过半数节点持久化,新 Leader 一定拥有它们。新 Leader 上任后丢弃自身 zxid 之后"未过半确认"的提案,保证已提交不丢、未提交不残留。etcd 的 Raft 用"日志索引 + 选最新日志节点"达成同样效果。

### 可交给 AI 的部分

这一章里,**Nginx 与 Spring Cloud Gateway 的集群完整配置、注册中心集群部署清单、服务扩容缩容脚本、选型对照表** 可放心交给 AI 生成。给定业务场景,AI 能产出可直接落地的配置。ZAB 与 Raft 对比、Eureka AP 与 Zookeeper CP 取舍分析,AI 也能讲清,适合做学习导览。

但有几类内容**必须自己掌握、不能盲信 AI**,边界与风险如下。

第一类是生产配置调参。AI 给的 `fail_timeout`、`burstCapacity`、Nacos 心跳间隔都是默认值,生产要按真实流量与集群规模调。限流阈值设错要么挡不住洪峰要么误杀用户;心跳间隔设短把注册中心打满、设长宕机发现慢。这些参数没有通用答案,必须结合压测数据自己定。

第二类是注册中心可用性兜底。AI 会说"Nacos 集群 ≥3 节点",但真实生产要考虑:跨机房多数派怎么分布、整体宕机时客户端本地缓存能撑多久、推送延迟多少算正常。这些现场决策 AI 看不到你的拓扑。注册中心挂了致全站调用失败每年都在发生,兜底方案(客户端缓存、降级静态列表)必须自己设计演练。

第三类是服务发现与 K8s 平台的边界。Kubernetes 里 Service + kube-proxy 已做服务端发现,再叠一层 Nacos 做客户端发现会产生"双重注册、双重寻址"混乱。AI 不会主动提醒"在 K8s 上是否还需独立注册中心",它倾向按 Spring Cloud 教程直接配 Nacos。这种平台能力重叠的决策必须架构师把关,选错会让复杂度翻倍。

第四类是故障现场处置。线上"偶发 502""调用间歇超时",根因可能是网关 upstream 混入死 IP、注册中心推送延迟、客户端缓存脏数据、TLS 握手失败。这类跨组件因果链排查 AI 帮不上忙——它看不到你的流量、日志、监控。网关 access log、注册中心事件日志、调用链 Trace 要自己会读会串。

区分"能交"与"不能交"的本质是:**配置模板与协议对比是机械知识,AI 强;参数调优、架构边界判断、跨组件故障定位是经验推理,AI 弱**。把模板交给 AI 起草,调参与排障留给自己练熟,这是 AI 时代中间件工程师的知识边界。
