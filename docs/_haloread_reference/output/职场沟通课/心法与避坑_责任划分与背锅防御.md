# 出了事，没留痕的那个背锅

老陈的项目出事了。

三方合作：老陈的团队负责开发，乙方负责设计，运营部负责推广。上线当天，用户投诉页面打不开，损失几百万。

复盘会上，三方开始甩锅。

乙方说："设计稿我们按时交付了，是开发没实现好。"——乙方掏出了设计稿邮件、确认记录、交付时间戳。

运营部说："我们按计划推的，是产品本身有问题，用户才投诉。"——运营部掏出了推广排期、上线确认邮件、用户投诉记录。

老陈说："我们按设计稿开发的，需求是运营部提的……"——老陈掏不出东西。开发过程中的需求变更，是开会口头定的，没纪要；设计确认，是群里一句"行"，没邮件；上线时间，是领导口头拍板的，没记录。

最后锅全扣在老陈头上：开发没按需求做，导致上线失败。老陈百口莫辩——他明明记得需求是运营部临时改的，但他拿不出证据。

职场里出事，没留痕的那个，就是背锅的那个。

## 责任是事前定的，不是事后争的

很多人对"责任划分"有误解，以为责任是出事之后才争的——出了问题，大家坐下来掰扯谁该负责。

错了。责任划分是事前讲清楚 + 事中留好痕 + 事后按证据。出事之后再争，争的不是"谁该负责"，是"谁有证据"。老陈不是没道理，是没证据。

为什么事前要讲清楚？因为事前大家还讲理。项目还没出问题，谁都不怕担责，分工好谈。等出了事，人人自保，再想划责任，谁都不认。

为什么事中要留痕？因为人的记忆不可靠。三个月前开会定的需求，你记得是A，他记得是B，没记录就是罗生门。留痕不是为了"日后整人"，是为了"日后说清楚"。

为什么事后要按证据？因为职场不讲"我以为"，讲"白纸黑字"。你以为是常识的事，没写下来就不算。证据是保险，不是阴谋。

责任划分的本质，是把模糊的"大家的事"变成清晰的"谁干什么、谁担什么"。模糊的地方，就是甩锅的空间。划得越清，甩锅越难。

## 曹刿问战：凭什么打这一仗

两千七百年前，齐国打鲁国，鲁国弱小，眼看要输。

鲁庄公要迎战，有个叫曹刿的平民求见。曹刿问鲁庄公："您凭什么跟齐国打？"

鲁庄公先说："衣食这类安逸的东西，我不敢独享，一定分给别人。"曹刿说："小恩小惠，老百姓不会跟您拼命。"

鲁庄公又说："祭祀用的牛羊玉帛，我按规矩来，不敢虚报。"曹刿说："小信用，神不会保佑您。"

鲁庄公说了第三条："大大小小的诉讼案件，我虽然不能一一明察，但一定按实情判决。"曹刿这回点头了："这是尽心办事，可以打一仗。"（大意据《左传·庄公十年》）

这就是有名的"曹刿论战"。

曹刿为什么认第三条？因为前两条是"恩惠"和"祭祀"，是软的；第三条是"司法公正"，是硬的。司法公正的本质，是责任划分清晰——谁犯了什么罪，按律怎么办，不偏不倚。老百姓知道规矩在哪，知道犯了会罚、没犯不会冤，才会信任这个政权，才愿意为它拼命。

鲁国后来在长勺打败了齐国，以弱胜强。靠的不是兵多，是民心。民心靠的是司法公正，司法公正靠的是责任清晰。

这个故事放在职场，道理一样：责任清晰，团队才有凝聚力。大家都清楚谁干什么、担什么，就不会互相推诿，出事了一起扛。责任模糊，人人都留一手，出事了互相踩。

## 责任划分三件套

把责任划清楚，靠三样东西：RACI 矩阵 + 会议纪要 + 邮件确认。

**一：RACI 矩阵。**

RACI 是个简单的表格，把每件事的四个角色写清楚：

- R（负责）：谁具体干。
- A（批准）：谁拍板担责。一件事只能有一个 A。
- C（咨询）：动手前要问谁的意见。
- I（知会）：做完要告诉谁。

新项目启动，第一件事就是把 RACI 列出来。比如"上线发布"这件事：开发是 R（干活的），技术总监是 A（拍板的），测试是 C（要测过才能发），运营是 I（要知道发没发）。

RACI 的关键是 A 只能有一个。一件事两个 A，等于没 A——出事互相推，没事互相抢。把 A 钉死，责任就钉死了。

**二：会议纪要。**

开会定的事，会后必须出纪要，发给所有参会人确认。

纪要不用长，三段就够：决定了什么、谁负责什么、什么时候交。关键是发给所有人确认——大家回一句"同意"或"有异议"，这事就钉死了。三个月后谁说"我没同意过"，把邮件翻出来，没话说。

很多人嫌纪要麻烦，觉得"开会说清楚就行了"。等出事你就知道，"会上说的"和"纪要写的"是两个东西。会上说的可以赖，纪要写的赖不掉。

**三：邮件确认。**

口头沟通定的事，事后用邮件再确认一遍。

微信里聊定的需求变更，发封邮件："按今天沟通，需求从 A 调整为 B，请确认。"对方回个"确认"，这事就留痕了。不回，你追一封"如无异议，按此执行"——默认确认。

邮件把口头承诺变成书面承诺。口头承诺赖得掉，书面承诺赖不掉。老陈就栽在这——他所有的关键决策都是口头定的，出事没人认。

## 背锅防御四招

责任划清楚了，还得防着被甩锅。四招：

**一：不口头承诺。**

别人让你答应什么，别口头说"行"。回一句"我邮件回你"或"我确认下排期再答复"。把承诺落到书面，既给自己留思考时间，也避免口头承诺被曲解。

口头承诺最大的坑，是对方记的跟你说的不一样。你说"我尽量"，他记成"他答应了"。出事你百口莫辩。

**二：不代人签字。**

任何文件，不是你职责范围内的，别替人签。领导让你"帮我签个字"，你说"这个还是您签，我不清楚情况"。同事让你"代签一下，我赶时间"，你说"这个我不方便代签，你回来再签"。

签字就是担责。代签等于替人背锅——出了事，签字的人负责，不管是不是你"代"的。

**三：及时书面异议。**

发现责任划分有问题，或者被安排了不该你担的事，当场提异议，且落到书面。

会上领导说"这事老陈你盯着"，你觉得不该你管，当场说："领导，这块按分工是运营部的，我盯着名不正言不顺，要不还是运营部牵头，我配合？"——当场提，会后跟一封邮件确认。

异议不及时提，默认你接受。三个月后说"我当时就不同意"，没人认。异议留痕，是你日后自保的底牌。

**四：保留原始记录。**

跟项目相关的邮件、聊天记录、会议纪要、需求文档，全部留好。别删、别清空。项目结束后再存档，至少存到项目验收后一年。

这些记录平时是垃圾，出事是黄金。老陈如果有当时的会议纪要和需求变更邮件，锅就甩不到他头上。

保留记录是给自己留一条退路。你不留，别人留了，主动权就在别人手里。

## 实操：画一张 RACI 矩阵

拿出你正在做的项目，画一张 RACI 矩阵。

1. 列出项目里的关键任务（需求确认、开发、测试、上线、推广……）。
2. 列出相关的人（你、领导、合作方、客户……）。
3. 每个任务，标出 R（谁干）、A（谁拍板）、C（问谁）、I（告诉谁）。
4. 检查：每个任务是不是只有一个 A？有没有任务没人 R？有没有人被排除了却该参与？

画完发给大家确认。这一步最关键——你画的 RACI，没经过大家确认，就只是你的"我以为"。发出去，大家回"同意"，这才是共识。

## 几个坑

**坑一：RACI 列了不执行。** 矩阵画得再漂亮，干活时不按它来，等于没画。中途有人推活，你得拿出矩阵说"这块按分工不是我的"。矩阵是工具，得用。

**坑二：纪要写得太长。** 纪要越长，越没人看。三段话讲清楚就行：决定了什么、谁干、什么时候交。细节附件里给，正文别堆。

**坑三：留痕变成"留一手整人"。** 留痕是为了自保和说清楚，不是为了日后翻旧账。你拿着半年的聊天记录去挑同事的毛病，同事以后防你防得死死的，活没法干。留痕有度，点到为止。

**坑四：把留痕当万能药。** 有些事留了痕也说不清——比如口头承诺你来不及留痕就被对方赖了。所以第一招"不口头承诺"才是根本，留痕是补救。能事前挡住的，别等事后补。

## 最后一句

责任划得清，锅甩不到你身上。留痕是职场人的保险，不是心机。
