# 第 27 章：读写分离与并行复制

> 前置知识：第 25 章主从复制三线程、第 26 章分库分表、binlog 组提交
> 学完你能：①讲清读写分离的写后读一致性问题与三种解法 ②说清并行复制从 5.6 到 8.0 的演进与关键参数 ③从 SDET 视角设计主从切换 RTO<30s 的压测方案

## 概念

读写分离是把写流量打到主库、读流量分摊到从库，用复制链路把主库数据同步到从库。林晓斌在《MySQL 实战 45 讲》第 25 讲“备库为什么会延迟，怎么解决”一讲里指出，读写分离的收益是读扩展，痛点也正是从库延迟——刚写完立刻读不到，这是写后读一致性问题。

大白话：主库写、从库读，主库压力小一半。但复制是异步的，从库总比主库旧一点点，用户刚下单立刻刷新订单列表，可能从从库读到旧数据，于是“下单成功却查不到订单”。补这个洞，要么强制走主库，要么缓存兜底，要么上并行复制把延迟压到接近零。

并行复制是为了治“从库单线程重放慢”这个老毛病：让从库的 SQL 线程能多线程并发重放，追上主库的并发写入速度。边界要认清：读写分离解决读压力，并行复制解决延迟，两者配合才让“读从库”真正可用。

## 原理

### 读写分离实现

两种落法：

- **中间件代理**：ShardingSphere-Proxy、ProxySQL、MaxScale 在应用与 MySQL 之间，按 SQL 类型路由——写走主、读走从。应用无感知。
- **SDK 层**：在 ORM 或数据源层用动态数据源切换，如 ShardingSphere-JDBC。少一跳网络，但侵入应用。

### 写后读一致性

读写分离的头号痛点：事务 A 在主库 INSERT 提交，紧接着的 SELECT 路由到从库，而该事务还没重放到从库，于是读不到。三种解法：

1. **强制走主库**：写后一段时间内同会话的读强制路由主库。简单可靠，但主库压力大。
2. **缓存兜底**：写入后把结果写缓存，读先查缓存命中则返回，绕过从库延迟。
3. **半同步复制**：主库等从库 ACK 再返回，把延迟窗口从“异步”压到“已同步到 RelayLog”，配合 GTID 等待可做到读从库也强一致。

### 并行复制演进

从库 SQL 线程单线程是延迟根因，并行复制演进三代：

- **MySQL 5.6 库级并行**：`slave_parallel_type=DATABASE`，按库粒度并行。不同库的事务可并发重放，但单库内仍串行。库少时几乎没增益。
- **MySQL 5.7 组提交并行**：`slave_parallel_type=LOGICAL_CLOCK`，主库同一组提交（group commit）里的事务无锁冲突，从库可并发重放。林晓斌在第 25 讲里强调这是 5.7 并行复制的关键突破——并行度由主库组提交粒度决定。
- **MySQL 8.0 写集合并行**：`binlog_transaction_dependency_tracking=WRITESET`，基于写集合（修改的行集合）判断事务是否冲突，无冲突即可并行，比 LOGICAL_CLOCK 粒度更细、并行度更高。

### 关键参数

```sql
-- 从库并行 worker 数
SET GLOBAL slave_parallel_workers = 16;
-- 并行策略（5.7+）
SET GLOBAL slave_parallel_type = LOGICAL_CLOCK;
-- 8.0 依赖追踪
SET GLOBAL binlog_transaction_dependency_tracking = WRITESET;
-- 写集合历史大小
SET GLOBAL binlog_transaction_dependency_history_size = 25000;
```

`slave_parallel_workers` 通常设为 CPU 核数的 1/2 到 1 倍，过大反而调度开销高。

### 一主多从与分级读

核心读走强一致从库（SemiSync+GTID 等待），列表/统计读走延迟从库（异步，容忍秒级延迟），分析走专用离线从库。林晓斌在第 27 讲“一主多从，主库断电怎么找回丢失的数据”里讨论了一主多从在断电场景的找回逻辑——多从库既是读扩展，也是数据冗余兜底。

## 实践

### 面试场景

**问：读写分离写完读不到怎么办？**
回答模板：先点问题本质——异步复制导致从库落后。再给三解法：同会话强制读主、缓存兜底、SemiSync+GTID 等待。最后补一句：根本治法是并行复制把延迟压到秒级以内。

**追问：并行复制 5.6 到 8.0 怎么演进的？**
答：5.6 库级并行（DATABASE），粒度粗、库少无用；5.7 组提交并行（LOGICAL_CLOCK），同组事务并发重放，是 5.7 突破；8.0 写集合并行（WRITESET），按行冲突判断，粒度最细、并行度最高。

**追问：`slave_parallel_workers` 设多大合适？**
答：CPU 核数的 1/2 到 1 倍，常见 8-32。设太大调度和锁竞争反而拖慢，需压测定。

### 项目场景（SDET 视角）

主从切换 RTO<30s 的压测设计：

1. **故障注入**：用 chaos 工具 kill 主库 mysqld 进程，模拟主库宕机。
2. **切换观测**：MHA 或 Orchestrator 检测+提升新主，记录从故障到新主可写的耗时，断言 <30s。
3. **数据零丢失校验**：切换后用 GTID 对比，新主已执行的 GTID 集合必须包含原主已提交未同步的事务（或确认在容忍的丢失窗口内）。
4. **读写恢复**：应用层重连+DNS 切换后，核心写链路恢复时间 <60s。

QA 要把“主从切换”做成常态化演练：每月一次真切换，每次大促前必演。仅靠理论可靠性不够，要靠真故障喂出来的肌肉记忆。

从库健康度监控：`Seconds_Behind_Master` 设阈值告警（>10s 黄、>60s 红），但该指标有盲区——SQL 线程卡死时可能报 0。林晓斌在第 26 讲“主备切换可靠性如何保证”里建议配合心跳（heartbeat）：主库定期写心跳表，从库读心跳记录时间差，比 `Seconds_Behind_Master` 更准。

复制中断同样要纳入监控：SQL 线程因数据冲突（主键重复、行不存在）停止时，`Slave_SQL_Running=No` 且 `Last_Error` 给出原因。生产忌用 `slave_skip_errors` 跳过错误——那会掩盖数据不一致；正确做法是按 GTID 定位冲突事务，手工修复后再恢复重放，必要时用 `START SLAVE UNTIL SQL_AFTER_GTIDS` 控制重放点。

### 避坑

- **坑 1**：开了 `slave_parallel_workers` 但 `slave_parallel_type` 还是 DATABASE。5.7 必须显式设 LOGICAL_CLOCK 才生效组提交并行。
- **坑 2**：`Seconds_Behind_Master=0` 就以为没延迟。SQL 线程卡死时它可能误报 0，务必配心跳。
- **坑 3**：SemiSync 只一个从库 ACK，但读分散在多个从库。被读的从库未必是 ACK 的那个，仍可能延迟。要 GTID 等待读或所有从库都 SemiSync。

## 速查/自测

### 速查表

| 并行策略 | 版本 | 粒度 |
|---------|------|------|
| DATABASE | 5.6 | 库级 |
| LOGICAL_CLOCK | 5.7 | 组提交 |
| WRITESET | 8.0 | 写集合（行级） |

### 自测题

1. 判断：`Seconds_Behind_Master=0` 一定代表从库无延迟。
2. 读写分离写后读不到，给出三种解法。
3. 简答：MySQL 5.7 的 LOGICAL_CLOCK 并行复制依据是什么？为什么比 5.6 库级并行好？
4. 简答：从 SDET 视角，主从切换压测要验证哪几个指标？

<details><summary>参考答案</summary>

1. 错。SQL 线程卡死或 IO 线程断开时，`Seconds_Behind_Master` 可能误报 0 或 NULL；准确判断要配心跳表（主库周期写时间戳，从库读差值）。
2. 三种：①写后同会话强制读主库；②写入后写缓存，读先查缓存兜底；③SemiSync 复制+GTID 等待，保证读到的事务已同步。
3. 依据是主库组提交（group commit）——同一组里提交的事务彼此无锁冲突，从库可并发重放。比 5.6 库级并行好，因为不依赖“不同库”这一粗粒度，单库内同组事务也能并行，并行度大幅提升。
4. ①RTO：从主库故障到新主可写 <30s；②数据零丢失：GTID 集合校验；③读写恢复：应用重连+DNS 切换后核心链路恢复 <60s；④切换过程无脏读/双写。
