# 第 10 章：回表与覆盖索引

> 前置知识：第 09 章聚簇索引与二级索引、InnoDB 索引组织表
> 学完你能：①讲清回表的发生条件和代价；②用覆盖索引优化高频查询，看懂 Explain 里的 `Using index`；③在生产中设计覆盖索引把查询延迟降一个数量级

## 概念

**回表（Table Lookup）**：在二级索引树上查到主键值后，再回到聚簇索引树查完整数据行的过程。一次回表等于多走一棵 B+ 树，千万级数据约多 3-4 次磁盘 IO。

**覆盖索引（Covering Index）**：查询所需的列全部包含在索引里，无需回表，直接从索引树返回结果。注意覆盖索引是"查询的执行效果"，不是一种索引类型——索引本身可能是个普通二级索引或联合索引，只是恰好覆盖了某条查询的所有列，就称这条查询"命中了覆盖索引"。

林晓斌在《MySQL 实战 45 讲》"深入浅出索引（下）"里用一句口诀概括：`select` 的列都在索引里，就不用回表。判断方法看 Explain 输出的 Extra 列，出现 `Using index` 就是命中覆盖索引。

## 原理

### 回表为什么贵

表 `T(id PK, k KEY, name, age, addr)` 执行 `select * from T where k=5`：

1. 在 `k` 的二级索引 B+ 树上找 `k=5`，叶子节点拿到 `id=500`。
2. 拿 `id=500` 回到主键聚簇索引 B+ 树，找到完整行数据。
3. 返回 `*` 涉及的所有列。

第 2 步就是回表。如果 `k=5` 命中 100 行，就要回表 100 次，每次 3-4 次 IO，光是回表就 300-400 次随机 IO。这是为什么"二级索引扫描大量行"反而不如全表扫描——回表的随机 IO 比全表扫描的顺序 IO 还慢。

优化器在估算时如果判断回表代价太大，会直接放弃二级索引走全表扫描，Explain 里 `key` 列会是 NULL 或显示 `PRIMARY` 但 `rows` 接近全表行数。

### 覆盖索引如何省掉回表

把上面 SQL 改成 `select id from T where k=5`：`id` 是主键，InnoDB 二级索引叶子节点本来就存主键值，所以 `k` 索引树叶子节点已经包含 `(k, id)`，查询需要的列全在索引里，不用回表。Explain 的 Extra 列会显示 `Using index`。

再看一个更典型的例子。表 `user(id PK, id_card, name, age)`，高频查询"根据身份证号查姓名"：

```sql
-- 方案 A：单字段索引，需要回表
alter table user add index idx_idcard(id_card);
select name from user where id_card='110101199001011234';
-- idx_idcard 叶子存 (id_card, id)，name 不在索引里 → 回表

-- 方案 B：联合索引，覆盖索引
alter table user add index idx_idcard_name(id_card, name);
select name from user where id_card='110101199001011234';
-- idx_idcard_name 叶子存 (id_card, name, id)，name 在索引里 → Using index，不回表
```

方案 B 多存了一份 `name` 的冗余数据，但换来了"无回表"的查询路径。对高频查询，这个权衡几乎总是值得的。

### 覆盖索引的边界条件

不是所有"加列到索引"都叫覆盖索引。几个边界：

1. **主键自动在二级索引里。** InnoDB 二级索引叶子节点固定带主键值，所以 `select id` 永远是覆盖索引（只要走了二级索引），不用把 `id` 显式加进联合索引。
2. **`select *` 几乎不可能命中覆盖索引。** 除非表所有列都在索引里，否则 `*` 总要回表。生产 SQL 应避免 `select *`，按需列查询。
3. **前缀索引无法命中覆盖索引。** `email(6)` 这种前缀索引，MySQL 无法判断前缀是否覆盖完整值，必须回表验证。这是前缀索引的致命缺陷，第 13 章展开。
4. **`Using index` 与 `Using where` 可以同时出现。** `Using where` 表示 server 层做了过滤，`Using index` 表示用覆盖索引返回数据，两者不冲突。`Using index condition` 是另一回事——索引下推 ICP，下一章讲。

### 覆盖索引不是免费的

覆盖索引的代价是索引体积变大、写入变慢。每加一个字段到联合索引，索引 B+ 树叶子节点就多一份冗余数据，写入时要维护更多页。林晓斌提醒"索引字段维护有代价"，高频查询才值得用覆盖索引，低频查询走普通索引 + 回表更经济。

经验值：单表 5 个以内索引、联合索引字段不超过 5 个，写入和查询通常能兼顾。超过这个量级要拆表或上 ES。

## 实践

**面试场景：被问到"如何避免回表"**

回答两层：①`select` 列尽量少，按需取列不要 `select *`；②对高频查询设计联合索引，把 `where` 列和 `select` 列都包进去，让查询命中覆盖索引，Explain 看到 `Using index` 即成功。再补一句"覆盖索引是优化手段不是索引类型"，区分概念。

**项目场景：高频查询优化**

某 SDET 平台查询接口 `select id, status, create_time from task where user_id=? order by create_time desc limit 20`，QPS 5000，原索引 `idx_user_id(user_id)`，每次回表 20 次，P99 80ms。

优化方案：建联合索引 `idx_user_id_create_time(user_id, create_time)`，把 `where` 列和 `order by` 列都包进去（`order by create_time` 能用索引天然有序避免 filesort），但 `status` 还要回表。

进一步：建 `idx_user_id_create_time_status(user_id, create_time, status)`，把 `select` 列也包进去，命中覆盖索引。实测 P99 从 80ms 降到 8ms，提升 10 倍。代价是索引体积 +30%，写入 QPS 下降约 5%，业务可接受。

**项目场景：判断 Explain 是否命中覆盖索引**

```sql
explain select name from user where id_card='110101199001011234';
```

关键看 Extra 列：

- `Using index`：命中覆盖索引，不回表，最优。
- `NULL`（空）：走了索引但要回表，普通情况。
- `Using where`：server 层过滤，可能配合回表。
- `Using index condition`：索引下推 ICP，下一章展开。
- `Using filesort`：额外排序，索引没覆盖 `order by`，需要优化。

**避坑：`select *` 是覆盖索引杀手**

线上有个慢查询 `select * from order where user_id=? limit 100`，原本有 `idx_user_id`，Explain 显示要回表 100 次，平均 50ms。改成 `select order_id, amount from order where user_id=? limit 100` + 联合索引 `idx_user_id_amount(user_id, amount)` 后，命中覆盖索引，降到 3ms。根因就是 `select *` 强制回表。

**避坑：覆盖索引不是越多越好**

曾经给一张 2000 万行的表加了 8 个联合索引试图覆盖所有查询，结果写入 QPS 从 1.2 万掉到 4000，binlog 同步延迟严重。复盘后砍到 4 个核心索引，低频查询走回表，写入恢复。覆盖索引只对高频查询划算。
