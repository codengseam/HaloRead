# 第 14 章：锁体系与意向锁

> 前置知识：InnoDB 存储引擎基础、事务隔离级别（RR/RC）、SQL 执行流程
> 学完你能：① 按粒度（全局/表/行）和类型（S/X）完整画出 MySQL 锁体系 ② 说清意向锁与 MDL 解决了什么问题，避免把表锁当行锁用

## 概念

MySQL 的锁不是单一概念，而是按粒度和类型两个维度交叉构成的体系。粒度从粗到细分为全局锁、表锁、行锁；类型分为共享锁（S，Shared Lock）和排他锁（X，Exclusive Lock）。

**按粒度划分**

全局锁（Global Lock）是对整个数据库实例加锁，典型命令是 `FLUSH TABLES WITH READ LOCK`（简称 FTWRL）。加锁后整个库只读，连 DDL 和修改类语句都不能执行，事务的 commit 也会被阻塞。它的核心使用场景是全库逻辑备份，比如 mysqldump 在不支持一致性快照的引擎上需要靠它拿到一致性视图。

表锁（Table Lock）锁的是整张表，server 层实现，与存储引擎无关。一种是 `LOCK TABLES t READ/WRITE` 显式表锁，另一种是 MDL（Metadata Lock，元数据锁），MySQL 5.5 引入，用来防止 DDL 与 DML 冲突。MDL 不需要显式声明，访问表时自动加：增删改查加 MDL 读锁，DDL 加 MDL 写锁。

行锁（Row Lock）锁的是索引记录，存储引擎层实现。MyISAM 不支持行锁，InnoDB 支持，这是 InnoDB 高并发更新的根基。InnoDB 的行锁加在索引上，没有索引就会退化为表锁——这一点会在实践部分反复提到。

**按类型划分**

共享锁（S Lock）又称读锁，多事务可同时持有同一资源的 S 锁。排他锁（X Lock）又称写锁，独占资源，与任何其他锁互斥。`SELECT ... LOCK IN SHARE MODE`（8.0 推荐用 `FOR SHARE`）加 S 锁，`SELECT ... FOR UPDATE` 和 DML 语句加 X 锁。

## 原理

**意向锁（IntentLock，IS/IX）解决的是表锁与行锁的冲突判断问题**

设想一个场景：事务 A 想给表 t 加表写锁，事务 B 已经在某行加了行 X 锁。事务 A 怎么知道表里有没有行锁？如果没有意向锁，只能逐行扫描整张表，开销不可接受。

意向锁的解法是：加行锁前必须先在表上加对应的意向锁。加行 S 锁前先加 IS（意向共享锁），加行 X 锁前先加 IX（意向排他锁）。意向锁是表级锁，加锁开销小。事务 A 加表锁时只要看表上有没有不兼容的意向锁即可，O(1) 判断。

林晓斌在《MySQL 实战 45 讲》第 06 讲给出的锁兼容矩阵要点：意向锁之间互相兼容（IS-IS、IS-IX、IX-IX 都不冲突），表锁与意向锁按 S/X 关系互斥。意向锁不阻塞行级操作，只用来告诉其他事务"我想动这个表的部分行"，避免表锁误判。

**MDL 防止 DDL/DML 冲突**

MDL 是另一类表级锁，作用是事务进行中表的元数据结构不被改动。开启事务后，第一条访问表的语句会持有该表的 MDL 读锁，直到事务提交才释放。如果此时另一个事务想做 `ALTER TABLE` 加字段，需要 MDL 写锁，会被读锁阻塞。

这里有个长事务陷阱：如果一个长事务持有 MDL 读锁，DDL 申请写锁会被阻塞；更糟的是，后续所有访问该表的新事务也要申请 MDL 读锁，会被 DDL 的写锁请求队列挡住，造成连锁阻塞，业务侧表现就是整张表"卡死"。林晓斌在第 06 讲强调过这个案例，这是生产事故的高频源头之一。

**全局锁 vs FTWRL 的细节**

`FLUSH TABLES WITH READ LOCK` 包含两步：先关闭所有打开的表（FLUSH），再加全局读锁。如果此时有长事务或长查询，FTWRL 会被阻塞，可能卡住几十秒甚至更久。

另一种"近似全库只读"的方式是 `SET GLOBAL readonly=true`，但林晓斌明确不推荐：一是 readonly 主要用于主备角色切换，语义上不该用来做备份锁；二是 readonly 在异常断开时会保留状态，而 FTWRL 在连接断开时自动释放，更安全；三是 readonly 还会影响备库复制行为。

一致性备份更优的方案是 `mysqldump --single-transaction`，它利用 InnoDB 的 MVCC 在 RR 隔离级别下开启一个一致性读事务，整个备份过程不加全局锁，其他事务正常读写。前提是引擎支持事务（InnoDB 可以，MyISAM 不行）。

## 实践

**场景一：全库备份怎么选**

生产环境推荐 `mysqldump --single-transaction --master-data=2`，对 InnoDB 库零锁影响。只有当库里混用了 MyISAM 表、或者需要拿一致性 DDL 快照时，才用 FTWRL，且要预估长事务的影响。

**场景二：表锁 vs 行锁的开销与并发**

MyISAM 用表锁，写操作互斥，并发更新性能差，这是它被 InnoDB 替代的根本原因。InnoDB 行锁开销大（要在索引上加锁记录、维护锁结构），但并发度高。

需要警惕 InnoDB "行锁退化为表锁"的坑：当 `UPDATE` / `DELETE` 的 WHERE 条件没有走索引时，InnoDB 必须扫描全表，给每一行加锁，效果等同于表锁。生产事故常见于"给状态字段加索引前，对状态字段做批量更新"。

**场景三：DDL 被长事务阻塞**

排查命令：

```sql
SHOW PROCESSLIST;
-- 找到 State 为 "Waiting for table metadata lock" 的会话
SELECT * FROM information_schema.INNODB_TRX;
-- 找到长时间未提交的事务
```

修复手段：找到并 kill 持有 MDL 读锁的长事务，DDL 才能继续。预防上避免业务高峰期做 DDL，必要时用 `pt-online-schema-change` 或 MySQL 8.0 的 INSTANT DDL。

**场景四：意向锁的可见性验证**

执行下面语句观察意向锁：

```sql
-- session 1
BEGIN;
SELECT * FROM t WHERE id = 1 FOR UPDATE;

-- session 2
SELECT * FROM performance_schema.data_locks;
-- TABLE 级别会看到 IX（意向排他锁），RECORD 级别看到 X 锁
```

8.0 用 `performance_schema.data_locks` 替代了 5.7 的 `SHOW ENGINE INNODB STATUS` 来查锁，输出更结构化。

**速查表**

| 维度 | 锁类型 | 作用范围 | 典型场景 |
|---|---|---|---|
| 全局 | FTWRL | 整库只读 | 全库逻辑备份 |
| 表 | LOCK TABLES | 整表 | 显式串行化 |
| 表 | MDL | 表元数据 | 防 DDL/DML 冲突 |
| 表 | IS/IX 意向锁 | 表级标志 | 行锁前置 |
| 行 | S/X | 索引记录 | InnoDB 行级并发 |
