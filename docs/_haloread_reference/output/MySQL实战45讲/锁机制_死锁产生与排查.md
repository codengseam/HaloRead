# 第 15 章：死锁产生与排查

> 前置知识：InnoDB 行锁、事务隔离级别、两阶段锁协议
> 学完你能：① 用死锁四条件判断业务是否会死锁 ② 用 `SHOW ENGINE INNODB STATUS` 和死锁日志定位根因

## 概念

**死锁（Deadlock）定义**：两个或多个事务互相持有对方需要的锁，形成循环等待，所有事务都无法继续。

操作系统课本里的死锁四个必要条件，在 MySQL 中同样成立：

1. **互斥（Mutual Exclusion）**：行 X 锁同一时刻只能被一个事务持有
2. **请求与保持（Hold and Wait）**：事务持有已有锁的同时申请新锁，不释放已有锁
3. **不剥夺（No Preemption）**：锁不能被其他事务强行夺走，只能等持有者主动释放
4. **环路等待（Circular Wait）**：事务之间形成 A→B→A 的等待环

四个条件同时满足才会死锁，打破任意一个就能避免。InnoDB 无法干预前三个（互斥和持有是行锁语义决定的），只能在第四个上做文章：要么检测到环路主动回滚，要么让事务按统一顺序加锁避免成环。

**经典案例**

```
事务 A：UPDATE t SET ... WHERE id = 1;  -- 持有 id=1 的 X 锁
事务 B：UPDATE t SET ... WHERE id = 2;  -- 持有 id=2 的 X 锁
事务 A：UPDATE t SET ... WHERE id = 2;  -- 等 id=2 的锁
事务 B：UPDATE t SET ... WHERE id = 1;  -- 等 id=1 的锁
-- A 等 B，B 等 A，死锁
```

## 原理

**两阶段锁协议（Two-Phase Locking，2PL）**

林晓斌在《MySQL 实战 45 讲》第 07 讲明确：InnoDB 行锁不是语句结束就释放，而是事务提交时才统一释放。这就是两阶段锁协议——事务分为加锁阶段和解锁阶段，加锁阶段只加不放，解锁阶段一次性释放所有锁。

这条规则带来两个推论：

- 事务里锁多个行时，**热点行放最后**。如果业务要更新 10 行，其中一行是热点（会被很多事务争抢），把它放在事务的最后一条语句，能缩短热点锁的持有时间。
- **事务越短越好**。长事务持有锁的时间长，死锁概率和锁等待概率都高。

**InnoDB 死锁检测：`innodb_deadlock_detect`**

参数默认值 `ON`（默认开启）。一旦事务申请锁被阻塞，InnoDB 立即检测是否形成等待环。发现死锁后，回滚代价较小的事务（修改行数少的），让另一个继续执行。

死锁检测的代价在热点行更新场景暴露得很明显。林晓斌给出数据：1000 个并发线程争抢同一行，每个被阻塞的线程都要遍历等待链判断是否成环，时间复杂度 O(n)，整体开销达到百万量级 CPU 操作。表现是 CPU 飙到接近 100%，但 TPS 只有几十。这是热点行更新的典型瓶颈。

应对方案：

1. 业务侧把同一行更新打散到多行（如账户余额按 user_id 拆成 10 行随机命中，再聚合）
2. 队列化写入，单线程串行更新热点行
3. 极端高并发下临时关闭死锁检测（`innodb_deadlock_detect=OFF`），靠 `innodb_lock_wait_timeout` 兜底——但要承担真死锁会等满超时的代价

**锁等待超时：`innodb_lock_wait_timeout`**

默认 `50`（秒）。一个事务等锁超过这个时间，会被回滚，报错 `Lock wait timeout exceeded; try restarting transaction`。这个参数处理的是"锁等待过长"，不等同于死锁检测。死锁检测是瞬时回滚（毫秒级），超时是被动等待。

调优建议：在线交易场景适当调小（如 5-10 秒），让长时间等锁的请求快速失败回滚，避免连接堆积；批处理场景可调大。调小有副作用：正常的慢锁等待会被误伤。

## 实践

**排查步骤**

第一步，看最近一次死锁：

```sql
SHOW ENGINE INNODB STATUS\G
```

输出里的 `LATEST DETECTED DEADLOCK` 段会打印死锁发生时两个事务的 SQL、持有的锁、等待的锁。这是定位根因的核心信息。但默认只保留最近一次。

第二步，开启全量死锁日志：

```sql
SET GLOBAL innodb_print_all_deadlocks = ON;
```

之后所有死锁会写到 error log（通过 `SHOW VARIABLES LIKE 'log_error'` 查路径）。

第三步，结合 `information_schema.INNODB_TRX` 看当前活跃事务：

```sql
SELECT trx_id, trx_state, trx_started, trx_query, trx_mysql_thread_id
FROM information_schema.INNODB_TRX
ORDER BY trx_started;
```

**典型死锁案例与根因**

案例一：业务上更新顺序不一致。两个接口都要改 id=1 和 id=2，一个先改 1 后改 2，另一个先改 2 后改 1。修复方法是统一加锁顺序（按 id 升序加锁）。

案例二：NextKeyLock 引发的插入死锁。事务 A 和 B 都对同一间隙加了 GapLock（间隙锁之间不冲突，都能加成功），然后各自向间隙里插入记录，互相被对方的 GapLock 阻塞，形成死锁。这类死锁在高并发插入场景常见，根因是间隙锁机制本身，第 16 章会展开。

案例三：唯一键冲突死锁。三个事务并发插入同一唯一键值，每个事务因冲突先加 S 锁，再尝试加 X 锁，互相等待。林晓斌在第 40 讲专门分析过这个场景。

**避免死锁的工程实践**

1. 统一资源访问顺序：批量更新按主键排序
2. 缩短事务：把非数据库操作（RPC 调用、文件 IO）移出事务
3. 优化索引：避免无索引条件导致行锁退化为表锁，缩小锁范围
4. 降低隔离级别：RC 没有 GapLock，能消除一类间隙锁死锁（需配合 `binlog_format=row`）
5. 大批量更新分批 + 加 limit：减小锁范围
6. 监控告警：死锁次数突增要能及时发现
