# 第 12 章：索引失效场景与排查

> 前置知识：第 10 章回表与覆盖索引、第 11 章最左前缀、Explain 基本用法
> 学完你能：①背熟 7 类索引失效场景并讲清根因；②用 Explain 三步定位法排查慢查询；③用 `force index` 和 `analyze table` 解决优化器选错索引的问题

## 概念

索引失效是指明明有索引但执行计划没走索引，退化为全表扫描。失效原因分两类：**写法问题**（SQL 让索引无法被使用）和**优化器问题**（索引可用但优化器估算后放弃）。

林晓斌在《MySQL 实战 45 讲》第 10 讲"MySQL 为什么有时候会选错索引"里把第二类问题讲得很透：优化器选索引靠"统计信息估算扫描行数"，统计信息不准或采样偏差就会选错。这一类需要 DBA 介入修正，而不是改 SQL。

排查工具核心是 Explain，关键看四列：`type`（访问类型）、`key`（实际用的索引）、`rows`（预估扫描行数）、`Extra`（额外信息）。

## 原理

### 场景一：对索引列做函数或运算

```sql
where abs(id) = 10          -- ❌ 失效
where date(create_time) = '2026-01-01'   -- ❌ 失效
where id + 1 = 11           -- ❌ 失效
where id = 10               -- ✅ 走索引
where create_time between '2026-01-01 00:00:00' and '2026-01-01 23:59:59'  -- ✅
```

根因：B+ 树按列原值排序，函数运算后值无序，索引无法定位区间。MySQL 5.7+ 对部分函数支持"函数索引（Functional Index）"和"生成列索引"能绕过这个限制，但默认情况下函数运算一定失效。

### 场景二：左模糊

```sql
where name like '%abc'      -- ❌ 左模糊，失效
where name like '%abc%'     -- ❌ 全模糊，失效
where name like 'abc%'      -- ✅ 右模糊，能走索引（前缀匹配）
```

根因：B+ 树按字符串字典序排列，前缀固定能定位区间，前缀不固定（左模糊）无法定位起点。`like 'abc%'` 等价于 `between 'abc' and 'abd'`，是范围查询能用索引。

### 场景三：隐式类型转换

```sql
-- id 是 int
where id = '123'            -- ❌ 隐式转换，MySQL 把 id 转字符串比较，失效
where id = 123              -- ✅ 数值匹配，走索引

-- phone 是 varchar
where phone = 13800138000   -- ❌ 隐式转换，MySQL 把 phone 转数值比较，失效
where phone = '13800138000' -- ✅ 字符串匹配，走索引
```

根因：MySQL 的隐式转换规则是"把字符串转数值"，等价于对字符串列加了 `cast(phone as signed)` 函数，触发了场景一的失效。判断技巧：看 `where` 列类型和值类型是否一致。

### 场景四：OR 连接一侧无索引

```sql
-- a 有索引，b 无索引
where a = 1 or b = 2        -- ❌ 全表扫描
where a = 1 or a = 2        -- ✅ 走 a 索引（同一索引列）
where a = 1 or c = 2        -- ✅ 若 a、c 都有索引，可能走 index_merge
```

根因：OR 要求两侧都满足，只要一侧走不了索引，整条查询就要全表扫描保证不漏数据。两侧都有索引时，MySQL 5.0+ 支持 index_merge 合并两个索引结果，但效果不如单索引。

### 场景五：不等于 / is not null

```sql
where id != 10              -- ⚠️ 通常失效（数据量小或区分度低时）
where id <> 10              -- ⚠️ 同上
where name is not null      -- ⚠️ 通常失效
```

根因：`!=` 命中的是"除了某个值之外的所有行"，等价于全表扫描，优化器估算后通常放弃索引。例外：如果表里 `id != 10` 命中的行很少（比如 `id` 大部分是 10，只有少数不是），优化器仍可能走索引。这是统计信息相关的判断，不绝对。

### 场景六：联合索引不满足最左前缀

`idx(a,b,c)` 上 `where b=2` 直接失效。

### 场景七：数据量太小或优化器选错

```sql
-- 表只有 100 行，where status=1 命中 80 行
where status = 1            -- 优化器可能选全表扫描
```

根因：优化器估算"全表扫描的顺序 IO"比"走索引 + 大量回表的随机 IO"更快。这是合理选择，不算 bug。

更隐蔽的情况是**统计信息过期导致选错索引**。InnoDB 默认通过采样 8 个数据页估算索引基数（cardinality），采样有随机性，可能严重偏离真实分布。林晓斌在第 10 讲举过一个例子：同一张表同样的 SQL，优化器有时选索引 A 有时选索引 B，根因就是采样统计信息波动。

### Explain 三步排查法

拿到慢查询后：

**第一步：看 `type` 列**。从好到差：`const` > `eq_ref` > `ref` > `range` > `index` > `ALL`。`ALL` 是全表扫描，索引失效了。`index` 是扫描整棵索引树（比 `ALL` 好一点但仍是全扫）。

**第二步：看 `key` 和 `rows`**。`key` 是实际选用的索引，NULL 表示没用索引。`rows` 是预估扫描行数，如果接近全表行数说明索引没起作用。

**第三步：看 `Extra`**。`Using index` 命中覆盖索引（最优）；`Using where` server 层过滤；`Using index condition` 索引下推；`Using filesort` 额外排序；`Using temporary` 用了临时表（最差，常见于 group by/distinct）。

### 修正优化器选错索引

```sql
-- 强制走某索引
select * from T force index(idx_a) where ...;

-- 更新统计信息
analyze table T;

-- 查看索引基数
show index from T;
```

`force index` 是临时手段，根因是统计信息不准或索引设计有问题。`analyze table` 重新采样统计信息，通常能解决选错索引问题，但采样仍有随机性，不一定每次都准。生产环境建议定期 `analyze table` 高频更新表，或在低峰期手动执行。

跨版本差异：MySQL 8.0 引入了"直方图统计（Histogram）"和"持久化统计信息（`innodb_stats_persistent=ON`）"，统计信息更稳定。8.0 还支持"不可见索引（Invisible Index）"——让索引对优化器不可见但保留维护，方便测试索引删除影响。

## 实践

**面试场景：被问到"索引什么时候失效"**

背熟 7 类：函数运算、左模糊、隐式类型转换、OR 一侧无索引、不等于、联合索引不满足最左前缀、数据量太小/统计信息过期。再补一句"前 6 类是写法问题改 SQL，第 7 类是优化器问题用 force index 或 analyze table"，体现系统化思维。

**项目场景：慢查询排查实战**

某 SDET 平台告警：`select * from task where date(create_time)='2026-06-28'` 执行 8 秒，`task` 表 5000 万行。Explain 显示 `type=ALL`、`key=NULL`、`rows=5000万`，根因是 `date()` 函数让 `create_time` 索引失效。

改写：`select * from task where create_time between '2026-06-28 00:00:00' and '2026-06-28 23:59:59'`，复测 `type=range`、`key=idx_create_time`、`rows=50万`，执行时间 80ms。提升 100 倍。

**项目场景：隐式类型转换定位**

接口 `select * from user where phone = ?` 偶发慢，`phone` 是 `varchar(20)` 有索引。日志发现 ORM 传入的是数值 `13800138000` 而非字符串 `'13800138000'`，触发隐式转换。改 ORM 层强制传字符串后，`type=ref`、P99 从 200ms 降到 5ms。

**项目场景：优化器选错索引**

订单表有 `idx_user_id` 和 `idx_status`，查询 `where user_id=? and status=?`，优化器有时选 `idx_status`（区分度低，命中 80% 行 + 大量回表），有时选 `idx_user_id`（命中 0.1% 行）。`show index` 看 `idx_status` 的 `cardinality` 波动大。

处理：`analyze table order` 更新统计信息；长期方案建联合索引 `idx_user_id_status(user_id, status)` 让优化器无歧义。短期用 `force index(idx_user_id)` 兜底。

**避坑：`force index` 是临时方案**

不要长期依赖 `force index`。数据分布变化后强制走的索引可能不再最优，反而拖慢查询。根因解决永远是"优化索引设计 + 保持统计信息新鲜"。

**避坑：`!=` 不一定失效**

某查询 `where status != 1`，`status` 只有 0/1/2 三个值，大部分是 1，查 `!= 1` 命中很少。优化器仍可能走 `idx_status`。判定要看 Explain 的 `rows`，不是看 SQL 写法想当然。
