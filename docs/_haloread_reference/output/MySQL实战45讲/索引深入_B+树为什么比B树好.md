# 第 08 章：B+ 树为什么比 B 树好

> 前置知识：基本数据结构（二叉树、哈希表）、数据库索引概念、磁盘 IO 基本原理
> 学完你能：①说清 MySQL 为什么选 B+ 树而不是 B 树、红黑树或哈希表；②根据数据量估算 B+ 树高度与 IO 次数；③在面试中讲透"索引为什么快"的本质

## 概念

索引本质上是数据结构在存储介质上的落地，目标是把全表扫描的 O(n) 查询降到 O(log n)。但具体选哪种数据结构，要看数据库的访问模式：磁盘随机 IO 代价远高于内存访问，树高就等于磁盘 IO 次数。

候选结构有四种常见答案：哈希表（Hash Index）、有序数组、红黑树、B 树/B+ 树。

哈希表等值查询 O(1)，但不支持范围查询、不支持排序，业务里 `where age between 20 and 30`、`order by create_time` 这种场景直接失效。有序数组二分查找 O(log n)，范围查询也快，但插入要挪动后续所有记录，只适合静态数据。

所以问题收敛到树形结构：红黑树是平衡二叉树，B 树是多路搜索树，B+ 树是 B 树的变体。林晓斌在《MySQL 实战 45 讲》"深入浅出索引（上）"一讲里直接点出，InnoDB 选 B+ 树，核心是为了让磁盘 IO 次数尽量少、范围查询尽量快。

## 原理

### 红黑树为什么不行

红黑树是平衡二叉树，每个节点最多两个子节点。千万级数据树高约 log₂(10⁷) ≈ 24 层，意味着一次查询最多 24 次磁盘 IO。即使根节点常驻内存，也远高于 InnoDB 期望的 3-4 次。

二叉树的瓶颈在于"扇出（fanout）太低"。同样 16KB 的页，红黑树一个节点存一个 key 加两个指针，扇出只有 2；而 B+ 树一个非叶子节点能塞上千个 key，扇出上千。扇出越大、树越矮、IO 越少。

### B 树为什么不够好

B 树（B-Tree）每个节点既存 key 又存 data。非叶子节点里塞了数据行，单页能放的 key 数就被压缩。假设一条数据行 1KB，一个 16KB 的页只能放十几条记录，扇出十几，千万级数据树高会上升到 6-7 层。

B 树还有两个短板：第一，范围查询要回到根节点重新遍历，或者中序遍历整棵树，无法高效扫一段连续区间；第二，查询路径不固定，命中的数据可能在非叶子节点（离根近）也可能在叶子节点（离根远），性能不稳定。

### B+ 树的三个关键设计

B+ 树在 B 树基础上做了三处改造，恰好对应上述短板：

**第一，非叶子节点只存索引不存数据。** InnoDB 默认页大小 16KB，一个索引项大约 14 字节（12 字节 key + 4 字节子页指针），单页能存约 1170 个索引项。叶子节点才存完整数据行。这样非叶子节点扇出极高，三层 B+ 树就能撑起 1170 × 1170 × 16 ≈ 2190 万条记录，正好覆盖千万级数据，树高 3-4 层。

**第二，叶子节点之间用双向链表连接。** 范围查询 `where id between 100 and 200` 命中起点后，沿链表往后扫即可，不用回根节点。这是 B+ 树对 B 树最大的实战优势，业务里 90% 的列表查询、分页、排序都依赖这条链表。

**第三，所有查询都走到叶子节点。** B+ 树的 data 只存在叶子，非叶子节点纯索引。这样无论查哪条记录，路径长度一样，性能稳定可预期。林晓斌在课程里强调"查询效率稳定"是工程上比"个别查询快"更重要的特性——监控、压测、容量规划都需要稳定的延迟基线。

### 哈希表为什么不是默认选项

哈希表等值查询 O(1) 比 B+ 树 O(log n) 快，但 MySQL 业务大量是范围查询和排序。`Memory` 引擎默认哈希索引，适合做临时表；InnoDB 不用哈希做主索引，但提供了"自适应哈希索引（Adaptive Hash Index, AHI）"——对热点页自动建哈希缓存，等值查询能直接命中。这是 B+ 树 + 哈希的混合方案，鱼与熊掌兼得。

需要说明：AHI 默认开启，由 InnoDB 内部维护，开发者无法手动指定。MySQL 8.0 仍保留该特性，但高并发场景下 AHI 的锁竞争可能成为瓶颈，可用 `innodb_adaptive_hash_index=OFF` 关闭。

### 复杂度与版本差异

B+ 树查询复杂度 O(log n)，这里的 n 是数据量。千万级数据 log n ≈ 23，但因为扇出上千，实际树高只有 3-4，IO 次数等于树高而非 log₂ n。这是多路搜索树相对二叉树的本质优势。

跨版本差异：InnoDB 页大小从 MySQL 5.7 起可通过 `innodb_page_size` 配置为 4K/8K/16K/32K/64K，默认 16KB。改大页能进一步降低树高，但写入放大和锁粒度会变大，生产环境一般保持 16KB。

## 实践

**面试场景：被问到"为什么是 B+ 树不是 B 树"**

回答模板分三层：①非叶子节点不存 data，单页 key 多、树矮、IO 少；②叶子链表支持范围查询；③查询路径等长、性能稳定。再补一句"千万级数据 3-4 层 B+ 树，3-4 次磁盘 IO 就能定位"，落地感立刻拉满。

**项目场景：估算索引树高**

订单表 5000 万行，主键 bigint 8 字节 + 6 字节指针 = 14 字节索引项，叶子节点存整行约 200 字节，单页约 80 行。三层 B+ 树能存 1170 × 1170 × 80 ≈ 1.1 亿行，5000 万绰绰有余，树高 3。这就是为什么 DBA 说"千万级表加索引查询应该秒回"，超过 100ms 基本就是没走索引或回表过多。

**项目场景：判断是否需要哈希索引**

某 SDET 团队维护一个配置表，只有等值查询 `where config_key = 'xxx'`，数据量 50 万行。B+ 树已经够快，不需要单独上哈希。但如果遇到一个极端热点 key 被高频等值访问，可以观察 AHI 命中率（`SHOW ENGINE INNODB STATUS` 里的 `Hash table size` 和 ` searches`），命中率高就保持 AHI 开启。

**避坑：不要用业务无序字段做主键**

UUID 做主键时，新插入的 UUID 随机分布，B+ 树叶子节点要频繁页分裂，写入性能断崖式下跌。这就是为什么 InnoDB 推荐自增整型主键——顺序插入、页分裂少、树更紧凑。这个话题在下一章"聚簇索引与二级索引"展开。
