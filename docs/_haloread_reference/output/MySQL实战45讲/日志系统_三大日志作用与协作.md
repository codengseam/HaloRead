# 第 03 章：三大日志作用与协作

> 前置知识：InnoDB 存储引擎基础、事务 ACID、SQL 更新语句执行流程
> 学完你能：① 在面试中讲清 redo log / undo log / binlog / slow query log / error log 各自定位与边界 ② 在生产中根据日志现象快速定位崩溃恢复、主备不一致、慢查询等典型问题

## 概念

MySQL 的日志不是一份，而是按"层 + 用途"切成了多份。林晓斌在《MySQL 实战 45 讲》第 02 讲中按层把日志拆成两类：InnoDB 引擎层自己的 redo log 与 undo log，Server 层的 binlog。除此之外还有面向运维的 slow query log（慢查询日志）与 error log（错误日志）。理解每个日志"由谁写、写什么、给谁读"，是后续两阶段提交、崩溃恢复、主备一致的基础。

五大日志一句话定位：

- redo log（重做日志，Redo Log）：InnoDB 引擎层物理日志，记录"数据页做了什么修改"，循环写，保证 crash-safe（崩溃后可恢复）。
- undo log（回滚日志，Undo Log）：InnoDB 引擎层逻辑日志，记录"修改前的旧值"，用于事务回滚与 MVCC 多版本并发控制。
- binlog（归档日志，Binary Log）：Server 层逻辑日志，记录"语句或行变更的原始逻辑"，追加写，用于主从复制与数据恢复。
- slow query log（慢查询日志）：记录执行时间超过 long_query_time 的语句，性能优化入口。
- error log（错误日志）：记录启动、运行、停止过程中的错误与警告，排障第一现场。

大白话：redo log 是"粉板"——掌柜先记下没来得及抄账本的流水，机器断电也不丢；undo log 是"后悔药"——改错了能按它退回原样；binlog 是"账本流水"——按时间顺序追加，谁要复制这套账就照着重放一遍；slow query log 是"耗时单"——超过阈值的慢动作都记下来；error log 是"急诊病历"——出了事先翻它。

边界要分清：

- redo log 与 binlog 都写"变更"，但前者是物理页级（page_no + offset + bytes），后者是逻辑级（SQL 或行变更）；前者循环写会覆盖，后者追加写永久保留。
- undo log 与 redo log 都属 InnoDB，但 undo 是逻辑回滚（按反向操作恢复旧值），redo 是物理重做（按页重新打补丁）。
- slow query log 与 error log 不参与事务一致性，纯粹是观测与运维用途。

易混淆点：redo log buffer（redo 日志内存缓冲区）与 binlog cache（binlog 线程级缓存）都是内存结构，但前者全局共享、后者线程独立——这点在第 04 章讲组提交时再展开。

## 原理

### WAL：先写日志，再写磁盘

redo log 是 WAL（Write-Ahead Logging，预写式日志）的核心载体。一次 UPDATE 不是直接写磁盘数据页，而是先把"对哪个页做了什么修改"写进 redo log（顺序写），再由后台线程异步把 Buffer Pool 里的脏页刷到 ibd 文件（随机写）。顺序写比随机写快一到两个数量级，这是 InnoDB 高吞吐的根因。

Buffer Pool 默认 128MB，数据页默认 16KB，脏页先在内存改、再批量落盘。如果直接写磁盘，每条 UPDATE 都要随机寻道；走 redo log 则把随机写转成顺序写，崩溃后用 redo log 重放即可恢复。

### redo log 的循环写

redo log 由一组固定大小的文件组成（如 ib_logfile0、ib_logfile1），内部是一个环。两个指针：

- write pos：当前写入位置
- checkpoint：当前可覆盖的位置（该位置之前的脏页已刷盘）

write pos 追上 checkpoint 时，redo log "写满"，InnoDB 会暂停业务、强制刷脏页推进 checkpoint。生产环境 redo log 总大小建议 1G~8G，按 TPS 调整，避免业务被卡。

### binlog 的追加写与三种格式

binlog 不覆盖、按文件追加（mysql-bin.000001、mysql-bin.000002……），单文件达到 max_binlog_size（默认 1GB）滚动。三种格式：

| 格式 | 记录内容 | 一致性 | 体积 |
|------|---------|--------|------|
| statement | SQL 原文 | 弱（NOW()/RAND()/带 LIMIT 的 DELETE 可能主从不一致） | 小 |
| row | 行变更前后镜像 | 强（按主键精确操作） | 大 |
| mixed | 自动选择 | 中 | 中 |

生产环境几乎清一色 row，原因有二：row 格式主备绝对一致；row 格式支持数据恢复（delete 转 insert、insert 转 delete、update 前后值对调）。MySQL 8.0 默认值就是 row。

### undo log：回滚 + MVCC

每行数据隐藏两个字段：DB_TRX_ID（最后修改该行的事务 ID）与 DB_ROLL_PTR（指向 undo log 版本链）。事务回滚时按 undo 链逆向恢复；快照读时沿 DB_ROLL_PTR 走版本链，结合 ReadView 判断可见性，这就是 MVCC 可重复读的实现基础。

### 一条 UPDATE 的日志协作

以 `UPDATE t SET c=c+1 WHERE id=2` 为例：

1. 执行器找 id=2 的行，加载到 Buffer Pool
2. 写 undo log（记录 c 旧值）
3. 修改 Buffer Pool 中数据页，标记脏页
4. 写 redo log（prepare 状态）
5. Server 层写 binlog 并 fsync
6. redo log 改为 commit 状态，返回客户端

第 4~6 步就是两阶段提交（2PC），下一章详解。

### redo log vs binlog 五维度对比

| 维度 | redo log | binlog |
|------|---------|--------|
| 所属层 | InnoDB 引擎层 | Server 层 |
| 日志类型 | 物理日志（页级） | 逻辑日志（SQL/行） |
| 写入方式 | 循环写、可覆盖 | 追加写、永久 |
| 核心作用 | crash-safe + 持久性 | 主备复制 + 数据恢复 |
| 事务提交 | 两阶段提交的 prepare/commit | 两阶段提交中写盘点 |

## 实践

### 生产配置基线（MySQL 8.0）

```sql
-- 持久化双 1
SET GLOBAL innodb_flush_log_at_trx_commit = 1;
SET GLOBAL sync_binlog = 1;

-- binlog 格式与镜像
SET GLOBAL binlog_format = 'ROW';
SET GLOBAL binlog_row_image = 'FULL';

-- 慢查询
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 0.1;
```

### 常见坑

**坑 1：redo log 设太小，业务高峰期被卡。**
现象：QPS 突增时出现大面积阻塞，`SHOW ENGINE INNODB STATUS` 中 Log sequence number 与 Last checkpoint at 距离接近 redo log 总大小。
修复：调大 redo log 容量。8.0.30+ 用 `innodb_redo_log_capacity` 统一管理（默认 100MB，建议 2G~8G）。

**坑 2：binlog_format 仍是 statement，主备数据漂移。**
现象：备库数据量与主库对不上，常见于带 NOW() 或 LIMIT 的语句。
修复：升级到 row，低峰期切换；切换前确认 binlog_row_image=FULL。

**坑 3：sync_binlog=0，主机宕机丢 binlog。**
现象：主机异常重启后，备库补不上某些事务。
修复：金融级必须 sync_binlog=1，宁可损失部分性能。

**坑 4：误删数据后才发现 binlog 没开或已过期。**
现象：想做 PITR（Point-in-Time Recovery，时间点恢复）时无 binlog 可用。
修复：log_bin=ON、expire_logs_days≥7；定期验证备份可恢复性。

**坑 5：MySQL 8.0 还在找 query cache。**
现象：试图调优 query_cache_size。
修复：MySQL 8.0 已移除查询缓存，别再问"为什么没这个参数"。

### 教科书做法 vs 生产做法

- 教科书：redo log 与 binlog 各司其职，理论清晰。
- 生产：实际要盯 write pos 与 checkpoint 距离、binlog 增长速率、slow log TopN、error log 中的 warning。监控不到位，理论再对也是事后救火。

### 跨版本差异

- 5.7 及之前用 innodb_log_file_size + innodb_log_files_in_group 控制 redo log。
- 8.0.30+ 用 innodb_redo_log_capacity 统一控制，旧参数废弃。
- 8.0 移除查询缓存（query cache）。

## 速查/自测

### 五大日志速查表

| 日志 | 层 | 类型 | 写法 | 作用 |
|------|----|----|------|------|
| redo log | InnoDB | 物理 | 循环 | crash-safe |
| undo log | InnoDB | 逻辑 | — | 回滚 + MVCC |
| binlog | Server | 逻辑 | 追加 | 复制 + 恢复 |
| slow query log | Server | — | 追加 | 性能优化 |
| error log | Server | — | 追加 | 排障 |
