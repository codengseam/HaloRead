# 第 25 章：主从复制原理

> 前置知识：第 23 章“双 1”参数、binlog/redo log 协作、第 04 章两阶段提交
> 学完你能：①画出主从复制三线程模型并解释每条线程职责 ②讲清 binlog 三种格式的取舍与默认值演进 ③在生产中排查主从延迟并定位大事务

## 概念

主从复制是把主库（master）的变更通过 binlog 异步同步到一个或多个从库（slave），从库重放后保持与主库一致的数据状态。林晓斌在《MySQL 实战 45 讲》第 24 讲“主库出问题了，从库怎么办”一讲里指出，主备复制是 MySQL 高可用的底座——主库挂了切到备库，前提是备库数据跟得上。

大白话：主库写一条 UPDATE，先把变更记进 binlog；从库拉走这段 binlog 存成 RelayLog，再一条条重放。三段路，三线程接力。复制是“异步”的，主库不等从库确认，所以性能高，代价是从库会落后。

边界要分清：复制保证的是“最终一致”，不是强一致；从库读到的数据可能比主库旧。强一致需求要走 SemiSync 或在业务层强制读主。复制也不等于分库分表——前者解决高可用和读扩展，后者解决单库容量瓶颈，两者可叠加但不是一回事。

## 原理

### 三线程模型

主从复制靠三个线程协作：

- 主库的 **dump 线程**：负责把 binlog 发给从库。从库连接主库后，主库为这个连接起一个 dump 线程，读 binlog 并通过网络推送。
- 从库的 **IO 线程**：接收 binlog 事件，写入本地的 RelayLog（中转日志）。
- 从库的 **SQL 线程**：读 RelayLog，按顺序重放，把变更应用到从库数据文件。

林晓斌在第 24 讲里强调，把“拉取”和“重放”拆成 IO 线程与 SQL 线程两个线程，是为了让拉取不被重放拖慢——即使从库重放慢了，RelayLog 还能源源不断积攒，主库 binlog 也能尽早被消费掉，避免主库 binlog 长期积压。

### binlog 三种格式

复制依赖 binlog，而 binlog 有三种格式，直接决定从库重放行为：

| 格式 | 记录内容 | 优点 | 风险 |
|------|---------|------|------|
| STATEMENT | 原始 SQL 语句 | 日志小、省空间 | 含 `UUID()`、`NOW()` 等不确定函数时主从不一致 |
| ROW | 每行变更前后镜像 | 准确、主从强一致 | 日志大，大批量更新膨胀明显 |
| MIXED | 自动判断 | 折中 | 仍需理解混合策略 |

MySQL 5.7.7 起默认 `binlog_format=ROW`，8.0 沿用。STATEMENT 在老系统还能见到，但新项目务必用 ROW——准确性的优先级高于日志体积。

### 同步策略三档

按主库等待从库确认的程度，复制分三档：

- **异步复制**（默认）：主库写完 binlog 立即返回，不等从库。性能最高，主库宕机可能丢已提交未同步的事务。
- **半同步复制**（SemiSync）：主库提交后至少等一个从库 ACK 收到 binlog 才返回。参数 `rpl_semi_sync_master_timeout` 控制等待，默认 10000ms（10s），超时自动降级回异步。
- **组复制 MGR**：基于 Paxos 变种，多数节点确认才提交，强一致。配置复杂，适合对一致性要求极高的核心库。

### GTID 与传统位点复制

从库定位“该从哪条 binlog 续传”有两种方式：

- **传统位点**：`CHANGE MASTER TO MASTER_LOG_FILE='...', MASTER_LOG_POS=...`，靠文件名+偏移量。主库切换、binlog 文件名变化时易错位，运维心智负担重。
- **GTID**：每个事务分配全局唯一 ID（`server_uuid:transaction_id`），从库记录已执行 GTID 集合，自动从主库拉取缺失事务。`CHANGE MASTER TO ... MASTER_AUTO_POSITION=1` 即可，主备切换时不用手算位点，是 5.6+ 推荐方案。

GTID 配合 SemiSync 能精确表达“主库已提交、从库已同步”的事务集合，是写后读强一致等待的基础。

### 双 1 与数据安全

林晓斌在第 23 讲“MySQL 是怎么保证数据不丢的”一讲里把可靠性收敛为“双 1”：`sync_binlog=1` + `innodb_flush_log_at_trx_commit=1`。前者保证 binlog 每事务 fsync，后者保证 redo log 每事务 fsync。主库开双 1 才能在宕机后不丢已提交事务，复制链路才可靠。

### 主从一致性问题

复制链路常见三类故障：

1. **复制延迟**：SQL 线程单线程重放跟不上主库并发写入，`Seconds_Behind_Master` 持续增大。
2. **大事务阻塞**：主库一条 DELETE 删百万行，binlog 是一个巨型事务，从库重放数小时，期间整条复制链卡住。
3. **DDL 卡从库**：`ALTER TABLE` 在从库也要重建全表，大表 DDL 可能让从库延迟数小时甚至天级。

## 实践

### 面试场景

**问：讲一下 MySQL 主从复制原理。**
回答模板：先点三线程（dump/IO/SQL），再说 binlog 三格式与默认值，最后补同步策略三档和延迟成因。一句话收口：“复制是异步的、最终一致的，强一致要走 SemiSync 或 MGR。”

**追问：为什么 IO 线程和 SQL 线程要分开？**
答：解耦拉取与重放。重放慢时不影响拉取，RelayLog 做缓冲，避免主库 binlog 因从库消费慢而长期积压。

**追问：STATEMENT 格式为什么会主从不一致？**
答：`UUID()`、`NOW()`、`RAND()` 这类不确定函数在主从执行时取值不同，行级结果就不同。ROW 格式直接记录行镜像，绕过函数再执行，所以准确。

### 项目场景（SDET 视角）

电商大促，订单从库延迟飙到 5 分钟，下游报表和对账全乱。定位思路：

1. `SHOW SLAVE STATUS\G` 看 `Seconds_Behind_Master`，确认延迟量级。
2. 看 `Slave_SQL_Running_State`，若停在 `System lock` 或 `Reading event from the relay log`，重点查大事务。
3. 查 `information_schema.innodb_trx` 找长事务，定位到一条归档脚本 `DELETE FROM order_log WHERE create_time < ...` 一次删 200 万行。

修复：归档脚本改成分批 `DELETE ... LIMIT 1000` 循环执行，单事务控制在毫秒级；同时升级到 MySQL 5.7+ 开 MTS 并行复制，SQL 线程变多线程。压测验证延迟从 5 分钟降到 30 秒内。

QA 角色要做的是把“主从延迟”纳入大促前置压测的必过指标：用真实流量回放，监控从库延迟峰值，超过 60s 报警阻断发布。

### 避坑

- **坑 1**：以为 SemiSync 就不丢数据。SemiSync 只保证 binlog 到达从库，不保证从库已 apply；且超时降级回异步后等同裸奔，务必监控 `Rpl_semi_sync_master_status`。
- **坑 2**：从库用 STATEMENT 格式跑含 `NOW()` 的批处理，主从不一致。统一 ROW。
- **坑 3**：大表 DDL 直接在主库执行，从库重放卡死。用 pt-online-schema-change 或 gh-ost 在线变更。

## 速查/自测

### 速查表

| 线程 | 所在 | 职责 |
|------|------|------|
| dump 线程 | 主库 | 推送 binlog |
| IO 线程 | 从库 | 收 binlog 写 RelayLog |
| SQL 线程 | 从库 | 重放 RelayLog |

### 自测题

1. 判断：SemiSync 复制保证主库宕机不丢任何已提交事务。
2. MySQL 5.7.7 之后 binlog 默认格式是哪种？为什么？
3. 简答：IO 线程与 SQL 线程为何拆分？拆分后哪个环节容易成为瓶颈？
4. 简答：电商大促从库延迟 5 分钟，你会按什么顺序排查？

1. 错。SemiSync 只保证 binlog 已到达从库 IO 线程（写入 RelayLog），不保证从库已 apply；且 `rpl_semi_sync_master_timeout` 默认 10s 超时会降级回异步，降级期间提交的事务可能丢。
2. ROW。STATEMENT 在不确定函数下主从不一致，ROW 记录行镜像准确，新项目首选。

4. 顺序：`SHOW SLAVE STATUS` 看 `Seconds_Behind_Master` 与 `Slave_SQL_Running_State` → 查 `innodb_trx` 找长事务/大事务 → 确认是否有大表 DDL → 针对性拆分大事务或开并行复制，压测验证。
