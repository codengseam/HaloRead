# 第 09 章：聚簇索引与二级索引

> 前置知识：第 08 章 B+ 树索引模型、InnoDB 索引组织表概念
> 学完你能：①分清聚簇索引和二级索引的本质区别；②说清主键索引/普通索引/唯一索引/联合索引各自定位；③在表设计时给出"为什么 InnoDB 推荐自增整型主键"的工程理由

## 概念

InnoDB 把表按主键顺序组织成 B+ 树，这棵树叫"索引组织表（Index Organized Table）"。一张表上的索引不止一棵 B+ 树，按"叶子节点存什么"分为两类：

- **聚簇索引（Clustered Index）**：叶子节点直接存整行数据。一张表只能有一个，InnoDB 默认选主键做聚簇索引；没有主键则选第一个非空唯一索引；都没有就生成隐藏 6 字节 `ROW_ID` 列做聚簇。
- **二级索引（Secondary Index，又叫辅助索引）**：叶子节点只存索引列值 + 主键值。查询时如果需要非索引列，要拿主键再回聚簇索引查一次，这个过程叫"回表"。InnoDB 的二级索引都是非聚簇的。

注意 MyISAM 与 InnoDB 的差异：MyISAM 的主键索引和二级索引都是非聚簇的，叶子节点都存"数据行的物理地址（行偏移）"，数据和索引分开两个文件（`.MYD` 和 `.MYI`）。这是两个引擎在存储组织上最根本的不同。

按业务用途，索引还有几种常见叫法，对应不同的约束语义：

- **主键索引**：唯一且非空，一张表一个，InnoDB 下即聚簇索引。
- **普通索引**：无约束，仅加速查询，`KEY idx_name(col)`。
- **唯一索引（Unique Index）**：列值唯一允许 NULL，`UNIQUE KEY uk_email(email)`。
- **联合索引（Composite Index）**：多列组合，`KEY idx_ab(a,b,c)`，按字段顺序排序，受"最左前缀"约束（下一章展开）。

## 原理

### 为什么一张表只能有一个聚簇索引

聚簇索引的叶子节点要按主键顺序物理存放整行数据，所以一行数据在物理上只能"被组织一次"。二级索引的叶子节点存的是"主键值"，相当于"指向聚簇索引的指针"——查询二级索引拿到主键，再回聚簇索引找数据。

这种"一份数据 + 多份主键指针"的设计，让二级索引可以建很多棵，但都共享同一份行数据。林晓斌在《MySQL 实战 45 讲》"深入浅出索引（上）"里画过一张图：表 `T(id PK, k)` 有两棵 B+ 树，主键树叶子存整行，`k` 索引树叶子存 `(k, id)`。

### 二级索引为什么要存主键而不存行地址

存主键值而非物理地址，是 InnoDB 相对 MyISAM 的关键设计。好处是行数据页分裂或迁移时，二级索引不用重建——因为指向的是逻辑主键，不是物理位置。代价是回表要走一次聚簇索引 B+ 树查找，比 MyISAM 直接按地址定位多一次树高 IO（千万级数据约 3-4 次）。

这也解释了"为什么主键要尽量短"：主键越短，每个二级索引叶子节点能塞的索引项越多，二级索引树越矮。`bigint` 8 字节 vs `UUID` 36 字节，同样 16KB 页，二级索引扇出能差 4 倍。

### 自增整型主键为什么是 InnoDB 的默认推荐

林晓斌在课程里反复强调自增主键的两个好处：

**第一，顺序插入避免页分裂。** 自增主键下，新行总是追加到 B+ 树最右侧叶子节点，页满了就新建下一页，是顺序写。如果主键是无序的（如 UUID、身份证号），新行会随机插入到中间某页，目标页满了就触发页分裂——申请新页、挪动一半数据、更新前后页指针，写放大严重，空间利用率也降到约 50%。

**第二，整型比 UUID 短。** 4 字节 int 或 8 字节 bigint，比 36 字节 UUID 短得多。所有二级索引叶子节点都要存一份主键，主键越短，二级索引体积越小、缓存命中率越高。

业务字段做主键的唯一合适场景是 KV 表——只有主键一个索引、且必须唯一，林晓斌举的例子是用身份证号做主键且只有这一个唯一索引的表。这种情况下没有二级索引，主键长短不影响二级索引大小，且能避免回表。但凡有多个索引，自增整型主键几乎总是更优解。

### 唯一索引 vs 普通索引的查询差异

两者 B+ 树结构一样，区别在约束。查询层面，唯一索引找到第一条匹配就停止（因为唯一），普通索引要继续往下扫确认没有更多匹配。但 InnoDB 内部对普通索引的"继续扫描"做了优化，实际性能差距很小，通常 1-2%。

写入层面差异更大：唯一索引每次插入都要做唯一性校验（查 B+ 树），普通索引不用。高并发写入场景下，唯一索引的校验开销和死锁概率都更高。这个话题林晓斌在《MySQL 实战 45 讲》第 09 讲"普通索引和唯一索引，应该怎么选择"里有详细对比，结论是：业务层能保证唯一就用普通索引，除非有强一致约束需求。

### 跨版本/跨引擎差异

- InnoDB：主键即聚簇索引，二级索引存主键值。MySQL 5.6 起所有版本一致。
- MyISAM：主键和二级索引都是非聚簇，叶子存行物理地址。MySQL 5.5 之前是默认引擎，5.5 起默认换 InnoDB，8.0 起 MyISAM 已不推荐使用。
- 聚簇索引选择优先级（InnoDB）：主键 > 第一个非空唯一索引 > 隐藏 ROW_ID。隐藏 ROW_ID 是 InnoDB 内部 6 字节计数器，全局共享，业务感知不到。

## 实践

**面试场景：被问到"聚簇索引和非聚簇索引区别"**

回答四点：①叶子节点存什么（整行 vs 主键值）；②一张表能有几个（一个 vs 多个）；③查询是否要回表（聚簇不用、二级要）；④InnoDB 主键即聚簇、MyISAM 都是非聚簇。再补一句"二级索引存主键而非行地址，是为了页分裂时二级索引不用重建"，能直接拉开档次。

**项目场景：主键选型**

订单表设计，候选方案 A：`order_id bigint auto_increment`，方案 B：业务生成的订单号 `varchar(32)`。选 A。理由：自增整型顺序插入无页分裂、8 字节短主键让所有二级索引更紧凑、InnoDB 聚簇索引按主键排序利于按时间范围扫最近订单。业务订单号可以加唯一索引作为二级索引，查询走 `uk_order_no` 回表即可。

**项目场景：何时用业务字段做主键**

某 SDET 团队维护一张"配置项 KV 表"，结构 `config_key varchar(64), config_value text`，只有按 key 等值查询，没有其他索引。这种情况可以用 `config_key` 做主键（聚簇索引），查询直接命中叶子节点返回 value，不用回表，也省一个二级索引。前提是 `config_key` 唯一且不会变。

**避坑：不要用 UUID 做主键**

某次性能压测，订单表用 UUID 做主键，写入 QPS 到 2000 就卡住，磁盘 IO 打满。换成自增 bigint 后 QPS 上到 8000。根因就是 UUID 随机插入导致页分裂频繁、二级索引膨胀。如果业务强需求 UUID，可以用 UUID 做唯一二级索引，主键仍用自增 bigint。

**避坑：没有主键的表**

InnoDB 表不建主键时，会选第一个非空唯一索引做聚簇；都没有就用隐藏 ROW_ID。隐藏 ROW_ID 是全局自增的，多张无主键表共享同一计数器，业务无法感知，且 binlog 复制和故障恢复都更复杂。生产表必须显式声明主键。
