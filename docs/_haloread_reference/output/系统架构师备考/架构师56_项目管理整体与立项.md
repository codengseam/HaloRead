# 第56章：立项五阶段把关，整体管理六过程一气贯通

中级《系统集成项目管理工程师》把项目管理十大知识领域当作主战场，整体管理与立项管理是入门第一关。整体管理串起其余九大领域，立项管理决定"项目该不该生"。这章综合知识必考 5-8 分，高级架构师案例分析也常借整体变更控制出题。

## 概念

### 这章考什么
- **科目**：中级综合知识（每年 5-8 题），高级案例分析偶现
- **分值**：中级约 5-8 分；高级案例 0-5 分
- **考频**：中频稳定
- **难度**：中等，记忆为主，CCB 流程是分水岭

核心类比：项目像盖一栋楼。**立项管理**是动土前的报批——先写项目建议书"我想盖楼"，再做可行性研究"盖不盖得起、值不值"，然后招投标选施工队、签合同；**项目章程**是开工令，正式授权项目经理动用资源；**整体管理六过程**是贯穿始终的总指挥，把范围、进度、成本、质量等串成一条线；**整体变更控制**是门禁，任何改动都得过 CCB 评审，不能私下乱改图纸。

## 原理

### 【中级重点】项目立项管理五阶段

立项管理解决"项目该不该立项、由谁来做"。五阶段一条龙，顺序不能乱：

| 阶段 | 做什么 | 产出 |
|---|---|---|
| **项目建议书** | 提出项目设想，说明为什么要做 | 项目建议书 |
| **可行性研究** | 多维度论证值不值得做 | 可行性研究报告 |
| **项目论证与评估** | 第三方对可研报告评审 | 评估报告 |
| **招投标** | 选定承建方 | 中标通知书、合同 |
| **合同谈判与签订** | 敲定权利义务 | 项目合同 |

**项目建议书内容**（必背）：项目必要性、项目市场预测、产品方案或服务、项目选址、投资估算、资金筹措、进度设想、风险分析。

**可行性研究三层精度**（高频考点）：

| 层次 | 别名 | 精度 | 目的 |
|---|---|---|---|
| **机会研究** | 投资机会鉴定 | ±30% | 粗筛有没有戏 |
| **初步可行性研究** | 预可研 | ±20% | 判断是否值得详细研究 |
| **详细可行性研究** | 可研 | ±10% | 拍板决策的依据 |
| **评估与决策** | 项目评估 | — | 第三方评审后做最终决策 |

精度数字 ±30%/±20%/±10% 是中级常考记忆点，口诀：**机会三成、初步两成、详细一成**。

**招投标流程**（五步法定顺序）：

1. **招标**：招标人发招标公告或投标邀请书。分公开招标（公告）和邀请招标（≥3 家特定投标人）。
2. **投标**：投标人按招标文件编制投标文件，截止时间前送达。
3. **评标**：评标委员会按招标文件确定的评标标准和方法评审。评标委员会由招标人代表和有关技术、经济等方面专家组成，成员人数为 **5 人以上单数**，其中技术、经济等方面专家不得少于成员总数的三分之二。
4. **中标**：招标人确定中标人（中标人应不超过参与评标的候选人数），发中标通知书。
5. **合同谈判与签订**：招标人与中标人按招标文件和中标人的投标文件订立书面合同。

**易错点**：
- 评标委员会"5 人以上单数、专家≥2/3"是数字题命门。
- 中标通知书发出后、合同签订前，招标人与中标人不得再行订立背离合同实质性内容的其他协议。
- 投标人少于 3 家的，招标人应依法重新招标。

### 【中级重点】项目章程内容与项目经理授权

**项目章程（Project Charter）**是项目启动的"出生证"，由**项目发起人或高层**签发，正式**授权项目经理动用组织资源**。在章程签发前，项目经理没有正式权力。

项目章程核心内容（必背要点）：
- 项目目的或批准理由
- 可衡量的项目目标和成功标准
- 高层级需求、产品描述
- 项目边界、可交付成果
- 总体里程碑进度计划
- 预算
- 委派的项目经理及其权责
- 发起人或其他关键干系人姓名职权
- 职能组织及其参与
- 假设条件、制约因素

口诀：**目的边界与可交付，进度预算经理权，干系人参与加假设**。

### 【中级重点】项目整体管理六过程

整体管理是十大知识领域的"总线"，把其余领域拧成一股绳。六过程按时间顺序：

| 过程 | 所属阶段 | 做什么 | 产出 |
|---|---|---|---|
| **制定项目章程** | 启动 | 正式授权项目 | 项目章程 |
| **制定项目管理计划** | 规划 | 整合各子计划 | 项目管理计划 |
| **指导与管理项目工作** | 执行 | 带领团队干活 | 可交付成果、工作绩效数据 |
| **监控项目工作** | 监控 | 比对计划与实际 | 工作绩效报告、变更请求 |
| **实施整体变更控制** | 监控 | 审批变更 | 批准/拒绝的变更 |
| **结束项目或阶段** | 收尾 | 正式收尾 | 最终产品移交、组织过程资产更新 |

口诀：**章程—计划—指导—监控—变更—结束**，六过程两两成对：启动规划在前，执行监控在中，变更收尾在后。

**易错点**：
- 项目管理计划是**渐进明细**的，不是一次性写死的。
- 工作绩效数据（执行产出，原始）→ 工作绩效信息（监控加工）→ 工作绩效报告（监控产出，给干系人）。
- 变更控制属于监控过程组，不属于执行。

### 【中级重点】整体变更控制流程

变更不是洪水猛兽，但要按规矩走。CCB（Change Control Board，变更控制委员会）是变更审批的常设机构。完整流程：

```
提出变更申请 → 评估变更影响 → CCB 评审 → 批准/拒绝
        ↓ 批准                          ↓ 拒绝
    实施变更 → 验证变更 → 关闭变更   通知申请方并记录
```

**六步口诀**：**提—评—审—批—施—验**（提出、评估影响、CCB 审批、实施、验证、关闭）。

**易错点**：
- 变更申请必须**书面**，口头不算。
- 评估影响要全面：范围、进度、成本、质量、风险、干系人都得看，不能只看一方面。
- CCB 批准后才能实施，不能"先斩后奏"。
- 实施后要**验证**（测试、评审），验证通过才**关闭**变更。
- 拒绝也要记录在变更日志，留痕。

### 【高级衔接】架构师视角下的项目治理

中级教材讲流程，高级架构师要看到流程背后的**治理逻辑**。CCB 在架构变更中扮演的不是橡皮图章，而是**架构完整性的守门员**。架构师在 CCB 上要回答三个问题：

1. 这个变更是不是会破坏已确定的架构风格和关键约束？
2. 这个变更是不是会引入新的质量属性风险（如可修改性、安全性下降）？
3. 这个变更要不要更新 ADL 描述和 4+1 视图？

**架构决策记录 ADR（Architecture Decision Record）** 与变更控制的关系是高级常考的衔接点。ADR 是轻量级文档，记录每个重要架构决策的**背景、决策、后果**。两者关系：

- **变更控制**管"流程能不能走"，CCB 把审批关。
- **ADR** 管"决策为什么这么定"，留背景和理由。
- 架构变更走 CCB 时，应同步更新或新建 ADR，保证架构决策可追溯。

实战中，架构师常在 CCB 评审会上同时维护 ADR 仓库：CCB 批准一条架构变更，对应的 ADR 就入库一条。这样三年后回看架构演进，能讲清"为什么从单体拆成微服务""为什么选 Kafka 不选 RabbitMQ"。

**项目治理 vs 项目管理**也是高级考点。治理是董事会/PMO 层面定规则、定结构、定边界；管理是项目经理在边界内执行。架构师在大型项目里常兼"治理顾问"角色，向 PMO 反馈架构红线是否被项目实际遵守。

## 实践

### 案例分析答题公式

整体管理案例题答题套路："**缺什么过程 + 该过程该做什么 + 不做会有什么后果**"三段式。

> 例：某项目未制定项目管理计划就开工，导致各子计划互相矛盾。请指出问题并给出改进建议。

**答题模板**：
1. **问题定位**：未执行"制定项目管理计划"过程，缺少整合各子计划的总计划。
2. **该做什么**：项目经理应召集核心团队，整合范围、进度、成本、质量、风险、人力资源、沟通、采购等子计划，形成统一的项目管理计划，并经干系人评审批准后作为执行基线。
3. **不做后果**：子计划互相矛盾，执行时无所适从，监控无基准，变更无依据，最终范围蔓延、进度失控、成本超支。

### 真题示例

**考法一：可行性研究层次精度**

> 例：（2023年·中级综合知识·立项管理相关，考生回忆）可行性研究中，详细可行性研究的投资估算精度一般要求在（　）以内。
> A. ±5%　B. ±10%　C. ±20%　D. ±30%

**答题思路**：详细可研精度 ±10%，初步 ±20%，机会 ±30%。答案 **B**。

**考法二：评标委员会构成**

> 例：（2024年·中级综合知识·招投标相关，考生回忆）依法必须进行招标的项目，其评标委员会由招标人代表和有关技术、经济等方面的专家组成，成员人数为（　）以上单数，其中技术、经济等方面的专家不得少于成员总数的（　）。
> A. 5 人；二分之一　B. 5 人；三分之二　C. 7 人；二分之一　D. 7 人；三分之二

**答题思路**：评标委员会 5 人以上单数，专家不少于三分之二。答案 **B**。

**考法三：整体变更控制流程**

> 例：（2023年·中级案例分析·整体变更相关，考生回忆）项目经理接到客户增加功能的要求后，直接安排开发人员修改代码并上线，导致与原计划冲突。请指出变更管理存在的问题并给出正确流程。

**答题思路**：问题——未走变更控制流程，绕过 CCB，未评估影响，未经验证直接上线。正确流程：提出书面变更申请 → 评估对范围/进度/成本/质量的影响 → CCB 评审 → 批准后实施 → 验证（测试、评审）→ 关闭变更并更新基线与文档。

### 高级衔接考点：项目章程与架构边界

高级案例分析中，章程缺失或边界不清是常见扣分点。架构师在章程阶段就要把**架构边界、关键质量属性需求、技术约束**写进去，否则后期变更纠纷无据可依。一个常见错误是章程只写业务目标不写架构约束，导致团队在选型上各说各话。

### 立项五阶段速查卡

```
项目建议书 → 可行性研究(机会/初步/详细) → 项目论证评估
        → 招投标(招标-投标-评标-中标-合同谈判) → 合同签订
```

### 自测题

**自测 1**：项目建议书应包含哪些内容？至少列出 5 项。

> 参考答案：项目必要性、市场预测、产品方案或服务、项目选址、投资估算、资金筹措、进度设想、风险分析。任选 5 项即可。

**自测 2**：整体变更控制的完整流程是什么？项目经理接到客户口头变更要求后第一步该做什么？

> 参考答案：完整流程：提出书面变更申请 → 评估变更对范围/进度/成本/质量/风险的影响 → CCB 评审 → 批准或拒绝 → 批准则实施变更 → 验证变更（测试、评审）→ 关闭变更并更新基线。接到口头要求第一步：要求客户**书面提出变更申请**，口头要求不进入变更流程。

**自测 3**：评标委员会成员构成有何法定要求？某项目评标委员会共 7 人，其中技术经济专家至少要几人？

> 参考答案：评标委员会由招标人代表和技术、经济等方面专家组成，成员 5 人以上单数，技术经济专家不少于成员总数的三分之二。7 人委员会中专家至少 5 人（7×2/3≈4.67，向上取整为 5）。

### 一句话小结

立项五阶段（**建议—可研—评估—招标—合同**）把项目出生证办齐，整体管理六过程（**章程—计划—指导—监控—变更—结束**）把项目一生串起来，变更六步（**提评审批施验**）守住规范——三套口诀记牢，中级 5-8 分稳收。
