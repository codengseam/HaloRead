# 第44章：设计模式与UML题，箭头方向是命门

设计模式与 UML 建模是案例分析的高频题型，分值通常 10~15 分。考法稳定：给一段业务场景描述，让你补全类图（缺类名、缺关系、缺箭头方向）、识别用了什么设计模式、说明参与者角色、补全序列图。失分点很集中——**箭头方向画反**、**模式名称不全**、**角色对应错**。这一章把 UML 类图五种关系的箭头方向钉死，再配三道高频模拟题（观察者、策略、序列图填空），让这道题变成"按规则画图填空"。

## 概念

### 这章考什么

- **综合知识（单选）**：考 **2~3 题**。常考设计模式分类（创建/结构/行为）、UML 关系类型、关系箭头方向、模式意图。
- **案例分析**：**高频**。典型考法："请补全类图中的类名/关系""请说明该系统采用了哪种设计模式""请说明 X 类在模式中的角色""请补全序列图"。分值 **10~15 分**。
- **论文**：偶考。论文写"我采用了 X 设计模式"时，要交代意图、角色、解决的问题。

考频结论：案例高频，必练。

### 题型特征

设计模式与 UML 题的题干结构稳定，常见三种考法：

**考法一：补全类图 + 识别模式**

```
某 [业务场景] 系统，[场景描述，含若干类和交互]。
下图为其类图（部分缺失）。
1. 请补全类图中的类名（X1、X2）。
2. 请补全类图中的关系（Y1、Y2，需指明关系类型和方向）。
3. 请说明该系统采用了哪种设计模式，并说明各参与者角色。
```

**考法二：序列图填空**

```
某系统 [场景描述]，下图为其序列图（部分缺失）。
请补全序列图中的消息（M1、M2、M3），并说明各消息对应的操作。
```

**考法三：模式选择 + 类图设计**

```
某系统 [场景描述]，需采用设计模式解决 [问题]。
1. 请选择合适的设计模式并说明意图。
2. 请画出该模式的类图（含类名、关系、方向）。
3. 请说明各参与者角色。
```

读题时眼睛盯三处：**场景特征**（决定哪种模式匹配）、**类图/序列图缺什么**（决定补什么）、**问题问什么**（决定答什么）。

### GoF 23 种设计模式速查

详细内容见第 16 章，这里只放案例答题常用的子集。GoF（Gang of Four）23 种模式分三大类：

| 类别 | 模式 | 典型场景 | 关键角色 |
|---|---|---|---|
| **创建型** | 单例（Singleton） | 全局唯一实例 | 单例类 |
| 创建型 | 工厂方法（Factory Method） | 创建对象延迟到子类 | 抽象工厂、具体工厂、抽象产品、具体产品 |
| 创建型 | 抽象工厂（Abstract Factory） | 创建产品族 | 抽象工厂、具体工厂、抽象产品、具体产品 |
| 创建型 | 建造者（Builder） | 分步构建复杂对象 | Director、Builder、ConcreteBuilder、Product |
| 创建型 | 原型（Prototype） | 克隆已有对象 | 原型接口、具体原型 |
| **结构型** | 适配器（Adapter） | 接口转换 | Target、Adapter、Adaptee |
| 结构型 | 桥接（Bridge） | 抽象与实现分离 | Abstraction、Implementor |
| 结构型 | 组合（Composite） | 树形结构 | Component、Leaf、Composite |
| 结构型 | 装饰器（Decorator） | 动态加职责 | Component、ConcreteComponent、Decorator |
| 结构型 | 外观（Facade） | 子系统统一入口 | Facade、子系统类 |
| 结构型 | 享元（Flyweight） | 共享细粒度对象 | Flyweight、FlyweightFactory |
| 结构型 | 代理（Proxy） | 控制访问 | Subject、RealSubject、Proxy |
| **行为型** | 责任链（Chain of Responsibility） | 请求链式传递 | Handler、ConcreteHandler |
| 行为型 | 命令（Command） | 请求封装为对象 | Command、ConcreteCommand、Invoker、Receiver |
| 行为型 | 解释器（Interpreter） | 语言解释 | AbstractExpression、TerminalExpression |
| 行为型 | 迭代器（Iterator） | 顺序访问聚合 | Iterator、Aggregate |
| 行为型 | 中介者（Mediator） | 集中交互 | Mediator、Colleague |
| 行为型 | 备忘录（Memento） | 状态保存恢复 | Originator、Memento、Caretaker |
| 行为型 | **观察者（Observer）** | 一对多通知 | Subject、Observer、ConcreteSubject、ConcreteObserver |
| 行为型 | **状态（State）** | 状态切换行为 | Context、State、ConcreteState |
| 行为型 | **策略（Strategy）** | 算法族切换 | Context、Strategy、ConcreteStrategy |
| 行为型 | 模板方法（Template Method） | 算法骨架 | AbstractClass、ConcreteClass |
| 行为型 | 访问者（Visitor） | 操作与结构分离 | Visitor、Element |

案例高频模式：**观察者、策略、状态、工厂方法、抽象工厂、适配器、代理、装饰器、单例、责任链**。这十个必须能默写类图和角色。

记忆口诀：**"创五结七行十一"**（创建型 5 个、结构型 7 个、行为型 11 个）。

## 原理

### 四步答题公式

设计模式与 UML 题固定套这个四步公式：

```
① 识别问题模式（创建/结构/行为）
→ ② 套 GoF 模式名称 + 意图
→ ③ 画/补 UML 类图（关键箭头方向）
→ ④ 说明参与者角色
```

**第 1 步：识别问题模式（创建/结构/行为）**

读题，把场景特征归类到三大类：

- **创建型**：场景强调"如何创建对象"——单例（唯一）、工厂（延迟创建）、抽象工厂（产品族）、建造者（分步构建）、原型（克隆）。
- **结构型**：场景强调"如何组合类/对象"——适配器（接口转换）、代理（控制访问）、装饰器（动态加职责）、外观（统一入口）、组合（树形）、桥接（分离）、享元（共享）。
- **行为型**：场景强调"对象如何交互/职责如何分配"——观察者（一对多通知）、策略（算法切换）、状态（状态切换）、责任链（链式处理）、命令（请求封装）、模板方法（算法骨架）、迭代器（顺序访问）、中介者（集中交互）、备忘录（状态保存）、访问者（操作分离）、解释器（语言解释）。

判别抓手：题干说"通知所有依赖""一对多"→ 观察者；"算法可替换""多种策略"→ 策略；"状态改变行为变化"→ 状态；"统一接口适配老系统"→ 适配器；"控制访问/延迟加载"→ 代理。

**第 2 步：套 GoF 模式名称 + 意图**

明确给出模式名（全称，中英文），再写意图（一句话）。意图必须用 GoF 标准表述，不能自已编。

例：观察者模式（Observer Pattern），意图"定义对象间的一对多依赖关系，当一个对象状态发生改变时，所有依赖于它的对象都得到通知并自动更新"。

**第 3 步：画/补 UML 类图（关键箭头方向）**

UML 类图五种关系必须画对箭头方向。这是最高频失分点，下面专门讲。

**第 4 步：说明参与者角色**

列出模式的标准角色，并对应到题干的具体类。比如观察者模式：

- Subject（抽象主题）：题干中的 X 类
- ConcreteSubject（具体主题）：题干中的 Y 类
- Observer（抽象观察者）：题干中的 Z 类
- ConcreteObserver（具体观察者）：题干中的 W 类

角色对应错（如把具体主题说成抽象观察者）扣分。

### UML 类图箭头方向速查（命门）

UML 类图五种关系的箭头方向是案例最高频失分点。必须背下来：

| 关系 | 英文 | 箭头/线型 | 方向 | 含义 | 强弱 |
|---|---|---|---|---|---|
| **继承** | Generalization | **空心三角箭头 + 实线** | 子类 → 父类 | 子类继承父类 | 最强 |
| **实现** | Realization | **空心三角箭头 + 虚线** | 实现类 → 接口 | 类实现接口 | 强 |
| **组合** | Composition | **实心菱形 + 实线** | 整体 ◆—— 部分 | 整体与部分同生共死 | 强 |
| **聚合** | Aggregation | **空心菱形 + 实线** | 整体 ◇—— 部分 | 整体与部分可分离 | 中 |
| **关联** | Association | **实线箭头** | 拥有方 → 被拥有方 | 长期拥有关系 | 中 |
| **依赖** | Dependency | **虚线箭头** | 依赖方 → 被依赖方 | 临时使用关系 | 弱 |

记忆口诀：**"空继实继，实组空聚，实关虚依"**（空心三角实线=继承、空心三角虚线=实现、实心菱形=组合、空心菱形=聚合、实线箭头=关联、虚线箭头=依赖）。

#### 关系强弱排序

**继承 > 实现 > 组合 > 聚合 > 关联 > 依赖**

强弱记忆：继承和实现是"垂直"关系（is-a），组合聚合关联依赖是"水平"关系（has-a / use-a）。组合比聚合强（组合同生共死，聚合可分离），关联比依赖强（关联长期拥有，依赖临时使用）。

#### 组合 vs 聚合（高频易混）

- **组合（实心菱形）**：整体与部分**同生命周期**，部分不能脱离整体存在。例：鸟和翅膀（鸟死翅膀也没了）。
- **聚合（空心菱形）**：整体与部分**可分离**，部分可脱离整体存在。例：球队和球员（球员可转会）。

判别抓手：看"部分能否独立存在"——能独立是聚合，不能独立是组合。

#### 箭头方向判别

箭头方向判别有个简单规则：

- **继承/实现**：箭头指向**父类/接口**（"子指向父"）。
- **组合/聚合**：菱形在**整体**一端，线连到部分（"菱形在整体"）。
- **关联**：箭头指向**被拥有**的类（"拥有方指向被拥有方"）。
- **依赖**：箭头指向**被依赖**的类（"使用方指向被使用方"）。

例：类 A 持有类 B 的引用（A 有 B），关系是关联，箭头从 A 指向 B。

### 序列图核心元素

序列图（Sequence Diagram）描述对象间按时间顺序的消息交互，核心元素：

- **对象**：上方方框，下划线表示（如 `:Object` 或 `obj:Object`）。
- **生命线**：对象下方的虚线。
- **激活条**：生命线上的矩形，表示对象正在执行。
- **消息**：对象间的箭头，分同步（实心三角箭头）、异步（开口箭头）、返回（虚线箭头）。
- **自调用**：对象发给自己的消息（自环箭头）。

序列图填空题的踩分点：**消息方向**（谁发给谁）、**消息类型**（同步/异步/返回）、**消息顺序**（从上到下时间递进）。

### 踩分点清单

阅卷老师按以下踩分点给分：

1. **模式名称准确**：用全称（中英文），不能只写"观察者"不写"模式"。
2. **意图表述**：用 GoF 标准意图，不能自已编。
3. **箭头方向**：五种关系箭头方向画对，画反扣分。
4. **角色对应**：标准角色对应到题干具体类，对应错扣分。
5. **关系类型准确**：组合 vs 聚合不能混，关联 vs 依赖不能混。
6. **类图完整性**：类名、属性、方法、关系都要画齐。

### 通用答题模板

```
[第1步 识别] 本场景属于 [创建/结构/行为] 型问题，
  特征为 XX（题干关键词）。

[第2步 模式] 采用 [模式名（Pattern Name）] 模式。
  意图：[GoF 标准意图]。

[第3步 类图]
  [类图说明，含类名、关系、方向]
  示例：
    Subject <|-- ConcreteSubject（继承，空心三角实线）
    Subject o-- Observer（聚合，空心菱形）
    Observer <|-- ConcreteObserver（继承，空心三角实线）
    ConcreteSubject --> ConcreteObserver（依赖，虚线箭头，通知）

[第4步 角色]
  - Subject（抽象主题）：题干中的 X 类，职责 XX。
  - ConcreteSubject（具体主题）：题干中的 Y 类，职责 XX。
  - Observer（抽象观察者）：题干中的 Z 类，职责 XX。
  - ConcreteObserver（具体观察者）：题干中的 W 类，职责 XX。
```

## 实践

### 模拟题一：补全观察者模式类图

> **题目**：
> 某股票交易系统，当某只股票价格变化时，需通知所有订阅该股票的投资者。系统已有以下类：
> - `Stock`（股票类）：维护股票价格和订阅者列表，价格变化时调用订阅者的更新方法。
> - `Investor`（投资者类）：实现更新方法，收到通知后刷新显示。
> - `StockA`（具体股票）：继承 Stock。
> - `InvestorA`、`InvestorB`（具体投资者）：继承 Investor。
>
> 类图部分缺失：
> - X1、X2 为缺失的类名
> - Y1、Y2 为缺失的关系（需指明类型和方向）
>
> 请回答：
> 1. 补全 X1、X2 类名，并说明该系统采用了哪种设计模式。
> 2. 补全 Y1、Y2 关系（类型+方向）。
> 3. 说明各参与者角色。

#### 分步解答

**第 1 步：识别模式**

```
本场景特征"价格变化时通知所有订阅者"，是一对多依赖关系，属于
行为型问题，采用观察者模式（Observer Pattern）。
意图：定义对象间的一对多依赖关系，当一个对象状态发生改变时，
      所有依赖于它的对象都得到通知并自动更新。

X1 = Stock（抽象主题，对应 Subject）
X2 = Investor（抽象观察者，对应 Observer）
```

**第 2 步：补全关系**

```
Y1：Stock 与 Investor 的关系。
  类型：聚合（Aggregation），空心菱形 + 实线。
  方向：菱形在 Stock 端，线连到 Investor（Stock 持有 Investor 列表）。
  说明：Stock 聚合多个 Investor，Investor 可独立存在（可订阅多只股票），
       属聚合而非组合。

Y2：StockA 与 Stock 的关系。
  类型：继承（Generalization），空心三角箭头 + 实线。
  方向：箭头从 StockA 指向 Stock（子类指向父类）。
  说明：StockA 继承 Stock 的属性和方法。
```

**第 3 步：参与者角色**

```
- Subject（抽象主题）：对应 Stock 类。
  职责：维护观察者列表，提供 attach/detach/notify 方法。
- ConcreteSubject（具体主题）：对应 StockA 类。
  职责：具体股票，状态（价格）变化时调用 notify 通知观察者。
- Observer（抽象观察者）：对应 Investor 类。
  职责：声明 update 接口，收到通知时调用。
- ConcreteObserver（具体观察者）：对应 InvestorA、InvestorB 类。
  职责：实现 update，刷新自己的显示。
```

**踩分点标注**：①模式名用全称（Observer Pattern）；②意图用 GoF 标准表述；③Y1 关系类型为聚合（不是组合，因为投资者可独立存在）；④Y1 菱形方向在 Stock 端；⑤Y2 继承箭头从子类指向父类；⑥四个角色全部对应到题干类。

### 模拟题二：识别策略模式

> **题目**：
> 某电商系统的促销活动有多种折扣算法：满减、打折、返现。系统需支持运行时切换折扣算法，且新增算法时不修改原有代码。已有类：
> - `Order`（订单类）：维护订单金额，调用折扣算法计算最终价格。
> - `Discount`（折扣接口）：声明计算折扣方法。
> - `FullReductionDiscount`、`PercentageDiscount`、`CashbackDiscount`（具体折扣类）：实现 Discount 接口。
>
> 请回答：
> 1. 该系统采用了哪种设计模式？说明意图。
> 2. 画出类图关系（含方向）。
> 3. 说明各参与者角色。

#### 分步解答

**第 1 步：识别模式**

```
本场景特征"多种算法可替换、运行时切换、新增不修改原有代码"，
属于行为型问题，采用策略模式（Strategy Pattern）。
意图：定义一系列算法，将每一个算法封装起来，并使它们可以互相替换。
      策略模式使得算法可以独立于使用它的客户端变化。
```

**第 2 步：类图关系**

```
1. Order 与 Discount 的关系：
  类型：关联（Association），实线箭头。
  方向：箭头从 Order 指向 Discount（Order 持有 Discount 引用）。
  说明：Order 持有一个 Discount 引用，调用其计算方法。

2. Discount 与 FullReductionDiscount 的关系：
  类型：实现（Realization），空心三角箭头 + 虚线。
  方向：箭头从 FullReductionDiscount 指向 Discount（实现类指向接口）。
  说明：具体折扣类实现 Discount 接口。

3. Discount 与 PercentageDiscount、CashbackDiscount：
  同上，实现关系。
```

**第 3 步：参与者角色**

```
- Strategy（抽象策略）：对应 Discount 接口。
  职责：声明算法接口（calculate 方法）。
- ConcreteStrategy（具体策略）：对应 FullReductionDiscount、
  PercentageDiscount、CashbackDiscount。
  职责：实现具体的折扣算法。
- Context（上下文）：对应 Order 类。
  职责：持有 Strategy 引用，调用其算法；可由客户端注入具体策略。
```

**踩分点标注**：①模式名全称（Strategy Pattern）；②意图用 GoF 标准表述；③Order-Discount 为关联（不是依赖，因为长期持有引用）；④具体折扣类与 Discount 为实现（虚线空心三角）；⑤三个角色对应到题干类。

### 模拟题三：序列图填空

> **题目**：
> 某文件下载系统，客户端调用 `DownloadManager` 下载文件，`DownloadManager` 调用 `Authenticator` 鉴权，鉴权通过后调用 `Storage` 读取文件，`Storage` 返回文件内容，`DownloadManager` 返回给客户端。序列图缺失消息 M1、M2、M3、M4。
>
> 请回答：
> 1. 补全 M1~M4 消息（含发送方、接收方、消息名、消息类型）。
> 2. 说明序列图中同步消息和异步消息的区别。

#### 分步解答

**第 1 步：补全消息**

```
M1：Client → DownloadManager
  消息名：download(filename)
  类型：同步（实心三角箭头）
  说明：客户端调用下载方法。

M2：DownloadManager → Authenticator
  消息名：authenticate(user)
  类型：同步（实心三角箭头）
  说明：调用鉴权方法。

M3：DownloadManager → Storage
  消息名：read(filename)
  类型：同步（实心三角箭头）
  说明：鉴权通过后调用存储读取文件。

M4：DownloadManager → Client
  消息名：return fileContent
  类型：返回（虚线箭头）
  说明：返回文件内容给客户端。
```

**踩分点标注**：①四条消息发送方、接收方、消息名齐全；②M1、M2、M3 为同步消息（实心三角）；③M4 为返回消息（虚线）；④顺序正确（鉴权→读取→返回）。

**第 2 步：同步 vs 异步**

```
同步消息（Synchronous）：发送方发送后等待接收方处理完成才继续，
  箭头为实心三角。调用方阻塞。
异步消息（Asynchronous）：发送方发送后不等待，立即继续执行，
  箭头为开口箭头。调用方不阻塞。
返回消息（Return）：接收方处理完成后返回结果，虚线箭头。
```

**踩分点标注**：①同步等待、异步不等待；②箭头类型（实心三角 vs 开口 vs 虚线）；③阻塞行为差异。

### 易错提醒

**易错 1：箭头方向画反**

最高频失分。常见错误：

- 继承箭头从父类指向子类（错！应从子类指向父类）。
- 聚合菱形在部分端（错！应在整体端）。
- 关联箭头从被拥有方指向拥有方（错！应从拥有方指向被拥有方）。

**记忆抓手**：继承"子指向父"、聚合"菱形在整体"、关联"拥有方指向被拥有方"。

**易错 2：组合 vs 聚合混淆**

把"球队和球员"画成组合（实心菱形），把"鸟和翅膀"画成聚合（空心菱形）。判别：**部分能否独立存在**——能独立是聚合，不能独立是组合。

**易错 3：关联 vs 依赖混淆**

- **关联**：长期拥有关系，类 A 持有类 B 的引用作为属性（实线箭头）。
- **依赖**：临时使用关系，类 A 在方法内使用类 B（局部变量、参数、返回值，虚线箭头）。

判别：看 B 是否作为 A 的属性——是属性是关联，非属性是依赖。

**易错 4：模式名称不全**

只写"观察者"不写"模式"，或只写"Observer"不写中文。**模式名要全称中英文**："观察者模式（Observer Pattern）"。

**易错 5：角色对应错**

把具体主题说成抽象观察者，把上下文说成策略。**角色对应必须准确**，建议答完后再核对一遍四角色对应表。

**易错 6：意图自己编**

不用 GoF 标准意图，自己编一段"这个模式就是让对象能通知别的对象"。**意图要用 GoF 标准表述**，建议背熟高频模式的意图原文。

**易错 7：序列图消息类型混淆**

把返回消息画成实线箭头（错！应为虚线），把同步消息画成开口箭头（错！应为实心三角）。**消息类型必须画对**。

### 答题骨架速查卡

```
设计模式与 UML 题四步公式：
  ① 识别问题模式（创建/结构/行为）
  ② 套 GoF 模式名称 + 意图（GoF 标准表述）
  ③ 画/补 UML 类图（关键箭头方向）
  ④ 说明参与者角色（对应题干类）

UML 五种关系箭头方向：
  继承：空心三角 + 实线，子 → 父
  实现：空心三角 + 虚线，实现类 → 接口
  组合：实心菱形 + 实线，整体 ◆—— 部分（同生共死）
  聚合：空心菱形 + 实线，整体 ◇—— 部分（可分离）
  关联：实线箭头，拥有方 → 被拥有方
  依赖：虚线箭头，依赖方 → 被依赖方

关系强弱：继承 > 实现 > 组合 > 聚合 > 关联 > 依赖
```

### 自测题

**自测 1**：UML 类图中，"组合"和"聚合"分别用什么符号表示？二者有何区别？

> 参考答案：组合用**实心菱形+实线**，聚合用**空心菱形+实线**，菱形都在整体一端。区别：组合的整体与部分**同生命周期**（部分不能脱离整体存在，如鸟和翅膀）；聚合的整体与部分**可分离**（部分可独立存在，如球队和球员）。口诀"组合同死聚可离"。

**自测 2**：观察者模式中，Subject 和 Observer 之间是什么关系？箭头方向如何？

> 参考答案：Subject 和 Observer 之间是**聚合**关系（Subject 持有 Observer 列表，Observer 可独立存在）。菱形在 Subject 端，线连到 Observer。Subject 通过聚合维护多个 Observer，状态变化时调用各 Observer 的 update 方法通知。

**自测 3**：某场景"系统需支持多种排序算法，运行时可切换"，应采用哪种设计模式？说明意图和角色。

> 参考答案：采用**策略模式（Strategy Pattern）**。意图：定义一系列算法，将每一个算法封装起来，并使它们可以互相替换，使得算法可以独立于使用它的客户端变化。角色：①Strategy（抽象策略）——声明排序算法接口；②ConcreteStrategy（具体策略）——具体排序算法（如快排、归并）；③Context（上下文）——持有 Strategy 引用，调用其算法，由客户端注入具体策略。

**自测 4**：序列图中同步消息和异步消息在箭头上如何区分？

> 参考答案：同步消息用**实心三角箭头**（发送方等待接收方处理完成）；异步消息用**开口箭头**（发送方不等待，立即继续执行）；返回消息用**虚线箭头**（接收方处理完成返回结果）。

**自测 5**：适配器模式中，Adapter、Target、Adaptee 三者的关系分别是什么？

> 参考答案：①Adapter **实现** Target 接口（实现关系，空心三角虚线，从 Adapter 指向 Target）；②Adapter **关联** Adaptee（关联关系，实线箭头，从 Adapter 指向 Adaptee，Adapter 持有 Adaptee 引用）。Adapter 通过实现 Target 接口对外提供服务，内部委托给 Adaptee 完成实际工作，实现接口转换。

### 一句话小结

设计模式与 UML 题的四步公式"识模套名画图角"是骨架，**UML 五种关系箭头方向是命门**（空继实继、实组空聚、实关虚依），高频模式角色对应表是肌肉。把箭头方向画对、模式名称写全、角色对应准确，这道题稳拿 12 分以上。
