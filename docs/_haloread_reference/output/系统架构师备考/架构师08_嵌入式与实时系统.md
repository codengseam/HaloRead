# 第8章：硬实时与中断底半部，嵌入式的世界规则

嵌入式世界和服务器端是两套逻辑，实时性、确定性、资源约束才是这里的硬通货，错过截止时间可能就是事故。

## 概念

综合知识里嵌入式 **1-2 题/年**，分值不高但必点到，是论文里讲"工业控制、车载、物联网"架构时的差异化加分项。

核心类比：通用 OS 像出租车，按需调度谁拦拉谁；RTOS（Real-Time Operating System，实时操作系统）像救护车，**有截止时间红线**，到点必须到。硬实时是航天刹车（错过就出事），软实时是视频通话（偶尔卡顿能忍）。

## 原理

### 嵌入式架构特点

- **专用性强**：面向特定功能，不是通用计算。
- **资源受限**：CPU、内存、功耗都卡得死。
- **实时性要求**：响应必须在截止时间内。
- **高可靠性**：常年运行，故障代价高。
- **交叉开发**：在 PC 上编译，烧到目标板运行。

### RTOS 与实时性

实时操作系统核心是**确定性**——任务响应时间的上界可预测，而不是"快"。

- **硬实时（Hard Real-Time）**：错过截止时间 = 系统失败（航天、医疗、ABS 刹车）。
- **软实时（Soft Real-Time）**：偶尔错过可接受（流媒体、游戏）。

RTOS 关键指标：中断延迟（从硬件中断到开始执行 ISR 的最长时间）、任务切换时间、最坏响应时间。VxWorks、μC/OS、FreeRTOS、RT-Linux 是常见 RTOS。

### 中断处理：顶半部与底半部

嵌入式把中断分成两段：

- **顶半部（Top Half / 硬中断）**：硬件触发后立即执行，只做最紧急的事（读数据、清中断标志、登记事件），**绝不能阻塞、不能耗时**。
- **底半部（Bottom Half / 软中断/任务）**：把耗时的处理推后到软中断、tasklet 或内核线程中做，可被新的硬中断打断。

记法：顶半部"快进快出"，底半部"慢工细活"。Linux 里对应 softirq、tasklet、workqueue。这样设计是为了**保证高优先级中断的实时响应**。

### 任务调度算法

| 算法 | 策略 | 是否最优 | 说明 |
|---|---|---|---|
| RM（速率单调） | 周期越短优先级越高（静态） | 周期任务最优 | 简单但不灵活 |
| EDF（最早截止时间优先） | 距截止越近优先级越高（动态） | 单处理器最优 | 调度开销大 |
| 抢占式优先级 | 高优先级抢占低优先级 | — | RTOS 常用 |

**RM 可调度条件**（n 个周期任务）：总利用率 U = Σ(Ci/Ti) ≤ n×(2^(1/n) - 1)。n 很大时上界趋近 **ln2 ≈ 0.693**。利用率超 0.693 不能保证 RM 可调度。

EDF 理论上利用率可达 **100%**，但实现复杂、调度开销大。

### 嵌入式 Linux vs 裸机

| 维度 | 嵌入式 Linux | 裸机/RTOS |
|---|---|---|
| 上下文 | 有 OS、进程线程 | 无 OS、主循环+中断 |
| 实时性 | 需打 PREEMPT_RT 补丁 | 天生硬实时 |
| 资源 | MB~GB 级 | KB 级 |
| 开发 | 标准接口、生态丰富 | 直接操作寄存器 |
| 场景 | 网关、HMI、路由 | 单片机、传感器节点 |

选型原则：**资源够、要网络和文件系统 → 嵌入式 Linux；要硬实时、毫秒级响应、资源紧 → RTOS 或裸机**。

## 实践

**真题示例（2024年·综合知识·嵌入式调度相关，考生回忆）**：RTOS 中"硬实时"和"软实时"的本质区别？

答题公式：硬实时是错过截止时间会导致系统失败/灾难（用"失效"踩分），软实时是偶尔错过不影响系统正确性（用"可容忍"踩分）。区别在"后果"而非"快慢"。

**RM 调度题**：给两个周期任务，问 RM 是否可调度。先按周期短定高优先级，再画时间轴验证是否都赶在截止前完成，最后可代公式 U ≤ n×(2^(1/n)-1) 检验。

**论文应用**：讲"车载/工业控制架构"时，引硬实时与中断底半部设计——ABS 刹车信号走顶半部立即响应，日志上报走底半部。讲物联网网关时区分"边缘 RTOS 采集 + 网关 Linux 上云"的两层架构。

**踩分词**：硬/软实时、顶半部/底半部、RM、EDF、中断延迟、确定性。

## 速查/自测

**30 秒速查卡**

| 考点 | 一句话 |
|---|---|
| 硬实时 | 错过截止 = 系统失败 |
| 软实时 | 偶尔错过可容忍 |
| 顶半部 | 硬中断，快进快出不阻塞 |
| 底半部 | 软中断/tasklet，做耗时活 |
| RM | 周期短优先级高，静态 |
| EDF | 截止近优先级高，动态最优 |
| RM 利用率上界 | n 大时趋近 ln2≈0.693 |

**口诀**：顶半急活底半慢活，RM 看周期 EDF 看截止；硬实时错过就翻车，软实时卡顿能忍受；RM 利用率不过零点六九，EDF 满载也能跑。

**自测题**：
1. 中断顶半部能否调用可能阻塞的函数？答：**不能**，必须快进快出，阻塞会卡死中断响应。
2. RM 调度利用率上界 n 趋近无穷时是多少？答：**ln2 ≈ 0.693**。
3. EDF 单处理器利用率理论上限？答：**100%**，但实现复杂、开销大。

一句话小结：嵌入式拿分靠"硬软实时辨析 + 顶底半部分工 + RM/EDF 调度"，记住"顶半急活底半慢活、RM 不超 0.693"就够用。
