# 第16章：二十三招拆解变化的武林秘籍

GoF（Gang of Four）23种设计模式是前人踩坑总结的套路。综合知识考3-4题，案例分析里设计模式题是**必考大题**，动辄15-25分。这一章不能背概念，要会读类图、会判场景、会写代码填空。把23招当成对付"变化"的工具箱——创建型对付"new 的变化"，结构型对付"组合的变化"，行为型对付"交互的变化"。

## 概念

### 这章考什么

- **科目**：综合知识3-4题 + 案例分析必考设计模式大题
- **考频**：综合年年 **3-4题**；案例**必考**（类图阅读+代码填空+模式辨析）
- **分值**：综合3-4分；案例15-25分
- **难度**：★★★★★（量大、模式易混、需读图）

### 三大类速览

GoF 23种模式按"应对什么变化"分三类：

| 类别 | 数量 | 应对变化 | 记忆 |
|---|---|---|---|
| 创建型 | 5 | 对象创建 | "new 的封装" |
| 结构型 | 7 | 类/对象组合 | "搭积木" |
| 行为型 | 11 | 对象交互职责分配 | "谁干什么" |

记忆口诀：**创5结7行十一，创建管生，结构管拼，行为管事**。

23种全名单（必背）：

- **创建型**：工厂方法、抽象工厂、建造者、原型、单例
- **结构型**：适配器、桥接、组合、装饰器、外观、享元、代理
- **行为型**：责任链、命令、解释器、迭代器、中介者、备忘录、观察者、状态、策略、模板方法、访问者

## 原理

### 一、创建型模式（5个）——讲"怎么造对象"

#### 1. 工厂方法（Factory Method）

- **问题**：客户端直接 new 具体类，与具体类耦合，新增产品要改客户端
- **方案**：定义创建对象的接口，让子类决定实例化哪个类
- **结构**：Product 抽象产品 ← ConcreteProduct 具体产品；Creator 抽象工厂（含 factoryMethod）← ConcreteCreator 具体工厂
- **场景**：日志记录器（File/DB/Console 多种）、数据库连接器
- **真题考法**：给类图认模式（一个抽象工厂对应一个抽象产品，多个具体一一对应）

类比：肯德基点餐，你只说"来份汉堡"（抽象），具体是香辣还是劲脆由后厨（子工厂）决定。

#### 2. 抽象工厂（Abstract Factory）

- **问题**：需要创建一族相关产品（如"中式套餐"=中式主菜+中式汤+中式甜点），不能混搭
- **方案**：提供一个接口，创建一系列相关或依赖对象，无需指定具体类
- **结构**：AbstractFactory 声明多个创建方法；ConcreteFactory1 造一族、ConcreteFactory2 造另一族
- **场景**：跨平台 UI（Windows按钮+Windows文本框 / Mac按钮+Mac文本框）、数据库多表族
- **真题考法**：辨析"产品族"vs"产品等级"——抽象工厂造族，工厂方法造等级

类比：装修风格套餐——"北欧风"工厂造北欧沙发+北欧桌+北欧灯；"中式风"工厂造中式全套。一个工厂管一整套搭配。

**工厂方法 vs 抽象工厂**：工厂方法造**一种**产品（一个方法），抽象工厂造**一族**产品（多个方法）；工厂方法靠子类扩展，抽象工厂靠新增工厂类扩展。

#### 3. 建造者（Builder）

- **问题**：复杂对象有多个可选部件，构造参数爆炸，构造顺序复杂
- **方案**：将复杂对象的构造与表示分离，同样的构造过程造不同表示
- **结构**：Director 指挥者调 Builder 的方法；Builder 抽象构建步骤；ConcreteBuilder 实现并组装；Product 最终产品
- **场景**：SQL 语句构造、文档生成、StringBuilder、链式构造
- **真题考法**：识别 Director + Builder 结构；常考链式调用 `builder.a().b().build()`

类比：点套餐——你依次说"加可乐、加薯条、不要冰"，收银员（Director）按流程指挥后厨（Builder）组装出成品。

#### 4. 原型（Prototype）

- **问题**：创建对象成本高（深拷贝复杂对象），希望复制现有对象
- **方案**：通过克隆已有实例创建新对象，不调构造方法
- **结构**：Prototype 声明 clone()；ConcretePrototype 实现 clone
- **场景**：批量创建相似对象、保存"模板"对象、原型注册表
- **真题考法**：浅拷贝 vs 深拷贝辨析；clone() 方法

注意：**浅拷贝**只复制基本类型，引用类型还指向同一对象；**深拷贝**递归复制所有引用。原型模式必须想清楚用哪种。

#### 5. 单例（Singleton）——高频

- **问题**：全局只需一个实例（配置、连接池、日志器），多处 new 浪费且状态不一致
- **方案**：保证一个类只有一个实例，并提供全局访问点
- **结构**：构造私有、静态私有唯一实例、静态公有 getInstance()
- **场景**：配置管理、线程池、数据库连接池、日志器
- **真题考法**：5种实现写法辨析（饿汉/懒汉/双重检查/静态内部类/枚举）

五种实现：

```java
// 1. 饿汉：类加载即创建，线程安全
class Singleton {
    private static final Singleton INSTANCE = new Singleton();
    private Singleton(){}
    public static Singleton getInstance(){ return INSTANCE; }
}
// 2. 懒汉：延迟创建，需同步
class Singleton {
    private static Singleton instance;
    private Singleton(){}
    public static synchronized Singleton getInstance(){  // 锁影响性能
        if(instance==null) instance = new Singleton();
        return instance;
    }
}
// 3. 双重检查：推荐
class Singleton {
    private static volatile Singleton instance;  // volatile 防指令重排
    private Singleton(){}
    public static Singleton getInstance(){
        if(instance==null){
            synchronized(Singleton.class){
                if(instance==null) instance = new Singleton();
            }
        }
        return instance;
    }
}
// 4. 静态内部类：推荐，延迟加载且线程安全
class Singleton {
    private Singleton(){}
    private static class Holder { static final Singleton INSTANCE = new Singleton(); }
    public static Singleton getInstance(){ return Holder.INSTANCE; }
}
// 5. 枚举：最简洁，天然防反射反序列化
enum Singleton { INSTANCE; }
```

### 二、结构型模式（7个）——讲"怎么拼对象"

#### 6. 适配器（Adapter）

- **问题**：已有类接口与目标接口不匹配，又不能改源码
- **方案**：将一个类的接口转换成客户端期望的另一个接口
- **结构**：Target 目标接口；Adaptee 被适配者；Adapter 实现 Target 并持有 Adaptee
- **场景**：旧系统接口复用、第三方库适配、Type-C 转 USB
- **真题考法**：类适配器（继承）vs 对象适配器（组合）辨析

类比：电源转换头——你的国标插头（Adaptee）插不进美标插座（Target），加个转换头（Adapter）就行。

#### 7. 桥接（Bridge）

- **问题**：多维度变化（如形状×颜色），继承会产生类爆炸（圆形红、圆形蓝、方形红...）
- **方案**：将抽象部分与实现部分分离，使二者独立变化
- **结构**：Abstraction 持有 Implementor 引用；RefinedAbstraction 扩展 Abstraction；ConcreteImplementor 实现细节
- **场景**：跨平台 UI（形状×渲染引擎）、消息发送（方式×类型）、JDBC 驱动
- **真题考法**：辨析桥接 vs 适配器——**桥接是设计期分离多维度，适配器是事后补救接口不匹配**

类比：遥控器（抽象）和电视（实现）分离——遥控器可换（普通/语音），电视可换（索尼/小米），二者独立扩展用桥接，不互相耦合。

#### 8. 组合（Composite）

- **问题**：树形结构中，叶子节点和容器节点需要统一对待
- **方案**：将对象组合成树形结构表示"部分-整体"层次，客户端一致地处理单个对象和组合对象
- **结构**：Component 抽象构件；Leaf 叶子；Composite 容器（含子节点集合）
- **场景**：文件系统（文件/文件夹）、组织架构、菜单树、UI 控件树
- **真题考法**：识别 add/remove/getChildren 方法在 Component 中（透明式）还是 Composite 中（安全式）

#### 9. 装饰器（Decorator）

- **问题**：比继承更灵活地扩展功能，避免子类爆炸
- **方案**：动态地给对象添加职责，装饰器与被装饰者实现同一接口
- **结构**：Component 抽象构件；ConcreteComponent 被装饰者；Decorator 持有 Component 引用并实现 Component；ConcreteDecorator 加功能
- **场景**：Java IO 流（BufferedInputStream 装饰 FileInputStream）、咖啡加料、日志增强
- **真题考法**：装饰器 vs 继承辨析；多层包装链识别

类比：给手机贴膜、加壳、挂绳——每一层装饰后还是"手机"，但功能递增，且可自由组合。

**代理 vs 装饰器**（高频易混）：装饰器**强调增强**功能，代理**强调控制**访问；装饰器通常由客户端主动包装，代理通常对客户端透明。

#### 10. 外观（Facade）

- **问题**：子系统复杂，客户端需要与多个对象交互
- **方案**：为子系统提供统一的高层接口，简化调用
- **结构**：Facade 持有各子系统引用，提供简化方法
- **场景**：电脑开机（一键调用 CPU/内存/硬盘启动）、Spring JdbcTemplate、API 网关
- **真题考法**：识别"提供一个统一入口"特征；外观降低耦合但子系统仍可直接访问

#### 11. 享元（Flyweight）

- **问题**：大量细粒度对象（如百万个棋子、字符）耗内存
- **方案**：共享细粒度对象，内部状态共享、外部状态传入
- **结构**：Flyweight 抽象享元；ConcreteFlyweight 共享内部状态；FlyweightFactory 工厂池
- **场景**：Integer 缓存、String 常量池、围棋棋子、文本编辑器字符
- **真题考法**：内部状态（共享、不变）vs 外部状态（传入、可变）辨析

#### 12. 代理（Proxy）——高频

- **问题**：需要在访问对象时加控制（权限、延迟、远程、日志），但不能改原对象
- **方案**：为其他对象提供代理以控制访问
- **结构**：Subject 抽象主题；RealSubject 真实主题；Proxy 实现 Subject 并持有 RealSubject 引用
- **场景**：远程代理（RPC stub）、虚拟代理（延迟加载）、保护代理（权限）、智能引用（计数/缓存）、AOP
- **真题考法**：四种代理类型识别；代理 vs 装饰器辨析

四种代理：

- **远程代理**：为远程对象提供本地代表（RMI、Dubbo）
- **虚拟代理**：延迟创建开销大的对象（图片懒加载）
- **保护代理**：控制访问权限
- **智能引用代理**：访问时附加操作（计数、缓存、日志）

### 三、行为型模式（11个）——讲"对象怎么协作"

#### 13. 责任链（Chain of Responsibility）

- **问题**：多个处理者都可能处理一个请求，但具体谁处理运行时定，避免请求发送者与处理者耦合
- **方案**：将处理者连成链，请求沿链传递，直到某处理者处理或链尾
- **结构**：Handler 抽象处理者（含 next 指针和 handleRequest）；ConcreteHandler 处理或转发
- **场景**：审批流（组长→经理→总监）、过滤器链、Spring 拦截器、异常处理链
- **真题考法**：识别链式 next 结构；常考"请求可能不被处理"特性

#### 14. 命令（Command）

- **问题**：请求发送者与接收者耦合，且需要支持撤销、排队、日志
- **方案**：将请求封装成对象，包含接收者和动作
- **结构**：Command 抽象命令（execute/undo）；ConcreteCommand 持有 Receiver；Invoker 调用者；Receiver 接收者
- **场景**：GUI 按钮、菜单操作、事务回滚、宏命令、任务队列
- **真题考法**：识别 Invoker-Command-Receiver 三角；常考"解耦发送者与接收者"

#### 15. 解释器（Interpreter）

- **问题**：需要解释执行某种语言（正则、SQL、规则引擎）
- **方案**：给定一个语言，定义其文法表示，定义解释器解释句子
- **结构**：AbstractExpression 解释接口；TerminalExpression 终结符；NonterminalExpression 非终结符；Context 上下文
- **场景**：正则引擎、规则引擎、SQL 解析、计算器表达式求值
- **真题考法**：**易错！解释器归行为型**（虽体现虚拟机/编译思想，但 GoF 划在行为型）；识别终结符/非终结符

#### 16. 迭代器（Iterator）

- **问题**：需要遍历聚合对象内部元素，又不暴露其内部结构
- **方案**：提供一种方法顺序访问聚合对象各元素，不暴露内部表示
- **结构**：Aggregate 聚合接口（createIterator）；Iterator 迭代接口（hasNext/next）
- **场景**：Java 的 Iterator、各种集合遍历
- **真题考法**：识别 hasNext/next；常考"分离聚合与遍历职责"

#### 17. 中介者（Mediator）

- **问题**：多个对象网状交互，关系复杂，改一处牵动多方
- **方案**：用一个中介对象封装一组对象交互，使对象间不需显式相互引用
- **结构**：Mediator 中介接口；ConcreteMediator 具体中介（持有各 Colleague 引用）；Colleague 同事类（只跟中介通信）
- **场景**：聊天室、机场塔台、MVC 中 Controller、事件总线
- **真题考法**：识别"网状变星型"特征；同事类之间不直接通信

#### 18. 备忘录（Memento）

- **问题**：需要保存对象状态以便恢复，又不破坏封装
- **方案**：在不破坏封装前提下，捕获对象内部状态并保存到外部
- **结构**：Originator 发起人（创建/恢复备忘录）；Memento 备忘录（存状态）；Caretaker 管理者（保管备忘录）
- **场景**：撤销/重做、游戏存档、事务回滚
- **真题考法**：识别三角结构；常考"封装性"——Caretaker 不能修改 Memento 内部

#### 19. 观察者（Observer）——高频

- **问题**：一个对象状态变化需通知多个依赖对象，且不希望硬编码这些依赖
- **方案**：定义对象间一对多依赖，当一个对象状态变化，所有依赖者得到通知
- **结构**：Subject 主题（attach/detach/notify）；Observer 观察者接口（update）；ConcreteSubject 维护状态；ConcreteObserver 实现 update
- **场景**：事件监听、消息队列发布订阅、Vue 响应式、股价通知
- **真题考法**：识别 Subject-Observer 一对多；推模式（主动传数据）vs 拉模式（观察者主动取）辨析

类比：微信公众号——你关注（attach）公众号（Subject），发文时所有粉丝（Observer）收到通知（notify），你取关（detach）就不再收到。

#### 20. 状态（State）

- **问题**：对象行为随内部状态变化，用大量 if-else 判断状态会导致类臃肿
- **方案**：允许对象在内部状态改变时改变其行为，对象看起来像修改了它的类
- **结构**：Context 上下文（持有 State 引用）；State 抽象状态；ConcreteState 具体状态（实现该状态下行为）
- **场景**：订单状态机、自动售货机、游戏角色状态、工作流
- **真题考法**：识别"状态切换改变行为"；状态 vs 策略辨析

**状态 vs 策略**（高频易混）：策略模式由**客户端**主动选择算法，状态模式由**对象自身**根据状态切换行为；策略通常并列可选，状态通常有转移规则。

#### 21. 策略（Strategy）——高频

- **问题**：多种算法可互换，用 if-else 选择导致维护困难
- **方案**：定义一系列算法封装起来，使它们可互相替换
- **结构**：Strategy 抽象策略（algorithm 方法）；ConcreteStrategy 具体策略；Context 持有 Strategy 引用
- **场景**：支付方式、排序算法、促销折扣、出行路线规划
- **真题考法**：识别 Context-Strategy 结构；常结合 OCP 出题

#### 22. 模板方法（Template Method）——高频

- **问题**：多个类有相同算法骨架，但某些步骤不同，希望复用骨架
- **方案**：定义算法骨架，将某些步骤延迟到子类实现
- **结构**：AbstractClass 定义模板方法（final，调用 primitive 操作）+ 原始操作（abstract）；ConcreteClass 实现原始操作
- **场景**：Spring JdbcTemplate、HttpServlet 的 doGet/doPost、框架生命周期钩子、做饭流程（烧水→放食材→调味，食材不同）
- **真题考法**：识别"父类定骨架子类填细节"；钩子方法（hook）辨析

```java
abstract class Cook {
    public final void cook() {  // 模板方法，final 防止改骨架
        boilWater();
        if (needIngredient()) putIngredient();
        season();
    }
    abstract void putIngredient();  // 子类实现
    void boilWater() { System.out.println("烧水"); }  // 通用
    void season() { System.out.println("调味"); }
    boolean needIngredient() { return true; }  // 钩子，子类可重写改变流程
}
```

#### 23. 访问者（Visitor）

- **问题**：需要对一组结构稳定的对象添加新操作，但不想修改这些对象的类
- **方案**：表示一个作用于对象结构的操作，使你可在不改变各元素类前提下定义新操作
- **结构**：Visitor 访问者接口（visit A/visit B）；ConcreteVisitor 具体访问者；Element 元素接口（accept Visitor）；ConcreteElement 实现 accept，调用 visitor.visit(this)
- **场景**：编译器 AST 遍历、文档导出（XML/HTML/文本）、报表生成
- **真题考法**：**双分派**机制（accept 调 visit，visit 调元素方法）；要求"对象结构稳定，操作频繁变化"

**双分派**是访问者的灵魂：第一次分派是 accept(Visitor) 根据元素类型调对应的 visit；第二次分派是 visit(this) 根据 visitor 类型执行具体操作。运行时同时根据"元素类型+访问者类型"决定行为。

### 易错点汇总

1. **解释器归行为型**：虽体现虚拟机/编译思想，GoF 明确划在行为型（11个里）
2. **访问者双分派**：accept→visit→元素方法，两层动态绑定
3. **桥接 vs 适配器**：桥接是设计期分离多维度变化，适配器是事后补救接口不匹配
4. **代理 vs 装饰器**：代理控制访问，装饰器增强功能；代理对客户端透明，装饰器客户端主动包装
5. **状态 vs 策略**：状态由对象自身状态驱动切换，策略由客户端主动选择
6. **工厂方法 vs 抽象工厂**：工厂方法造一种产品，抽象工厂造一族产品
7. **组合模式透明式 vs 安全局**：透明式把 add/remove 放 Component（叶子要空实现），安全式放 Composite（叶子不需要空实现）

## 实践

### 真题示例

**2024年·综合知识·设计模式相关(考生回忆)**：以下模式中，用于"将抽象部分与其实现部分分离，使它们独立变化"的是？
A. 适配器
B. 桥接
C. 组合
D. 装饰器

**答案**：B。桥接的定义就是"抽象与实现分离，独立变化"。适配器是事后接口转换，组合是树形部分-整体，装饰器是动态加职责。

**2024年·综合知识·设计模式相关(考生回忆)**：访问者模式的核心机制是？
A. 单分派
B. 双分派
C. 代理转发
D. 链式传递

**答案**：B。访问者通过双分派（accept→visit）实现操作与元素分离。

### 案例设计模式题答题公式

案例必给一段需求描述+类图（或代码骨架），要求识别模式并填空。固定四步：

1. **判模式**：抓需求关键词
   - "创建对象不指定具体类" → 工厂/抽象工厂/原型
   - "动态加职责/功能" → 装饰器
   - "控制访问" → 代理
   - "一对多通知" → 观察者
   - "算法可替换" → 策略
   - "算法骨架复用" → 模板方法
   - "状态切换变行为" → 状态
   - "请求沿链传递" → 责任链
   - "树形部分整体" → 组合
   - "多维度独立变化" → 桥接
2. **填角色**：把类图中的类对号入座到该模式的标准角色（如 Context/Strategy/ConcreteStrategy）
3. **写代码**：照模板套，注意继承关系、方法签名、多态调用
4. **说原则**：补一句"体现了 OCP/DIP，符合开闭原则"加分

### 案例填空踩分词

填空题阅卷按关键词给分，务必写全：

"设计模式应用/软件架构设计"论文中，挑2-3个模式深入写：

> 针对支付场景的扩展性需求，系统采用策略模式封装支付宝、微信、银行卡三种支付算法，由 PaymentContext 持有策略引用，运行时通过依赖注入切换。新增支付方式只需实现 PaymentStrategy 接口，无需修改上下文代码，体现了开闭原则与依赖倒置原则。同时通过观察者模式在支付完成后通知库存、积分、通知服务，实现解耦的异步协同。

### 高频模式速记口诀

- **创建型**：工抽建原单（工厂、抽象工厂、建造、原型、单例）
- **结构型**：适桥组装外享代（适配、桥接、组合、装饰、外观、享元、代理）
- **行为型**：责命解迭中备观状策模访（责任链、命令、解释器、迭代器、中介者、备忘录、观察者、状态、策略、模板方法、访问者）

### 易错点速查

| 易错点 | 正确答案 |
|---|---|
| 解释器归哪类 | 行为型（11个之一） |
| 访问者核心机制 | 双分派 |
| 桥接 vs 适配器 | 桥接设计期分维度，适配器事后转接口 |
| 代理 vs 装饰器 | 代理控制访问，装饰器增强功能 |
| 状态 vs 策略 | 状态自驱动切换，策略客户端选 |
| 工厂方法 vs 抽象工厂 | 一个造一种，一个造一族 |
| 组合透明式 vs 安全局 | 透明式 add 在 Component，安全式在 Composite |

### 自测题

1. 一个系统需要支持"支付方式可扩展、不修改原代码"，应选哪种模式？体现哪条原则？
2. 某类图：Abstraction 持有 Implementor 引用，二者各有子类。这是什么模式？与适配器区别？
3. 订单有"待支付/已支付/已发货/已完成"四种状态，每种状态下"取消"行为不同。应用什么模式？
4. 文件系统需要统一处理文件和文件夹的"打开"操作，用什么模式？
5. 访问者模式的"双分派"两次分派分别依据什么？

### 一句话小结

23模式分创建5、结构7、行为11——创建管"怎么造"，结构管"怎么拼"，行为管"怎么协作"；案例必考读类图填空，记住"判模式→填角色→写代码→说原则"四步走，重点啃高频8式（单例/工厂/观察者/策略/适配器/装饰器/代理/模板方法）。
