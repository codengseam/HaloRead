# 第23章：C/S到B/S再到N层，Web架构的进化论

Web 架构的演进就是一部"层越拆越多、客户端越做越瘦"的历史。这一章把两层 C/S、三层 C/S、B/S、N 层架构的演进讲清，再拿下 MVC/MVP/MVVM 三种表现层模式和 REST、前后端分离，案例题的 Web 架构部分就稳了。

## 概念

### 这章考什么

- **综合知识**：每年约 **2 题**。高频考点是 MVC/MVP/MVVM 三者的区别辨析，以及 C/S 与 B/S 的特点比较。
- **案例分析**：**Web 架构题是案例必考项**，常让你为某系统设计架构、说明分层理由、或选 REST 还是 RPC、是否前后端分离。
- **论文**：Web 架构、微服务、前后端分离都是论文高频主题，分层理由和模式选型是论证核心。

### 一句话理解演进主线

Web 架构演进的主线是**"客户端越来越瘦、层越拆越细、前后端越来越分离"**：

> 两层 C/S（胖客户端）→ 三层 C/S（拆出业务层）→ B/S（浏览器替代胖客户端）→ N 层（业务层再拆，加缓存/网关等）→ 前后端分离 + REST API。

打类比：早期饭店老板自己又炒菜又收银又端盘子（两层 C/S 胖客户端）；后来分出后厨、前厅、收银三个岗（三层）；再后来变成外卖平台，顾客自己用手机点单（B/S，浏览器/手机当瘦客户端）；现在连后厨都拆成切配、炒菜、打包多个工序（N 层 + 前后端分离）。客户端越来越只管"点单"，重活全在服务端。

## 原理

### 演进一：两层 C/S 架构

**结构**：客户端（Client）+ 数据库服务器（Server）。客户端负责表现层 + 业务逻辑（胖客户端），数据库负责数据存储。

```
[客户端：表现层 + 业务逻辑] <---> [数据库服务器]
        （胖客户端）
```

**特点**：

- 客户端"胖"——承担表现和业务逻辑，要安装部署。
- 数据库直接被客户端访问。
- 性能好（业务逻辑在本地跑）、交互响应快。

**缺点**：客户端臃肿、每台机都要安装升级、维护成本高、跨平台差、安全难控（客户端直连数据库）。

### 演进二：三层 C/S 架构

**结构**：表现层（客户端）+ 业务逻辑层（应用服务器）+ 数据层（数据库服务器）。

```
[表现层：客户端] <---> [业务逻辑层：应用服务器] <---> [数据层：数据库]
   （瘦一些）            （业务逻辑上移）
```

**特点**：

- 业务逻辑从客户端抽到独立的应用服务器，客户端变瘦。
- 各层可独立部署、独立扩展。
- 客户端不直连数据库，安全性提升。

**优点**：客户端轻、维护集中、可扩展、安全可控。

**缺点**：比两层复杂、应用服务器是关键节点。

### 演进三：B/S 架构

**结构**：浏览器（Browser）+ Web 服务器+ 数据库服务器。本质是三层 C/S 的特例——客户端换成了标准浏览器。

```
[浏览器] <--HTTP--> [Web 服务器] <---> [数据库]
 （通用客户端）     （业务+页面生成）
```

**特点**：

- 客户端是通用浏览器，**零安装、零部署、跨平台**。
- 升级只在服务端做，用户无感知。
- 基于 HTTP/HTTPS，互联网友好。

**优点**：客户端零维护、易分发、跨平台、易升级。

**缺点**：交互体验不如胖客户端（早期）、服务器压力大、网络安全挑战大（暴露在公网）。

### 演进四：N 层架构

**结构**：在三层基础上继续拆分，常见为表现层、Web/接口层、应用/业务层、领域层、数据访问层、数据层，还可加缓存层、网关层等。

```
[表现层] -> [网关/API层] -> [应用层] -> [领域层] -> [数据访问层] -> [数据层]
                                                  (+缓存层、消息层等)
```

**特点**：层越拆越细、每层职责单一、可独立扩展和替换，是微服务、云原生架构的基础。

### C/S vs B/S 对比（高频考点）

| 对比维度 | C/S（两层/三层） | B/S |
|---|---|---|
| 客户端 | 专用客户端，需安装 | 通用浏览器，零安装 |
| 部署维护 | 客户端每台都要装升级 | 只维护服务端 |
| 跨平台 | 差（依赖客户端环境） | 好（浏览器标准） |
| 交互体验 | 好（胖客户端能力强） | 早期弱、现代已接近原生 |
| 性能 | 业务在本地，响应快 | 依赖网络，服务器压力大 |
| 安全 | 内网为主，相对可控 | 暴露公网，挑战大 |
| 典型场景 | 企业内网 ERP、专业软件 | 互联网应用、SaaS |

### 表现层三模式：MVC / MVP / MVVM（高频考点）

这三个模式是解决"表现层怎么组织"的，**MVC/MVP/MVVM 三者区别**是综合知识必考点。三者都把表现层拆成"模型-视图-控制/主持/视图模型"，区别在**视图和逻辑怎么通信**。

**MVC（Model-View-Controller）**

- Model：数据模型；View：视图；Controller：控制器。
- **Controller 直接操作 View**——控制器拿到用户输入、调 Model、然后直接更新 View。
- View 可直接观察 Model 变化（Model 通知 View 刷新）。
- 典型：Spring MVC（早期）、ASP.NET MVC。

```
用户 --输入--> Controller --调用--> Model
                  |                  |
                  +--直接操作--> View <--观察-- Model
```

**MVP（Model-View-Presenter）**

- Model：数据模型；View：视图（被动）；Presenter：主持者。
- **View 和 Presenter 通过接口交互**——View 只暴露接口，Presenter 通过接口操作 View，View 不直接碰 Model。
- Presenter 是中间人，View 完全被动，便于单元测试。
- 典型：Android（早期）、WinForms。

```
用户 --输入--> View --接口调用--> Presenter --调用--> Model
                  ^                     |
                  +----接口更新----------+
```

**MVVM（Model-View-ViewModel）**

- Model：数据模型；View：视图；ViewModel：视图模型。
- **View 和 ViewModel 双向绑定**——ViewModel 数据变了 View 自动刷新，View 上用户改动也自动同步回 ViewModel。
- 开发者不用手动操作 View，只需维护 ViewModel 状态，框架（如 Vue、WPF）自动同步。
- 典型：Vue.js、WPF、Knockout。

```
用户 --输入--> View <--双向绑定--> ViewModel --调用--> Model
              (框架自动同步)
```

### MVC / MVP / MVVM 三者区别表（必背）

| 模式 | 控制器/主持者/视图模型 | 与 View 的关系 | 通信方式 | 典型场景 |
|---|---|---|---|---|
| **MVC** | Controller | **直接操作 View** | Controller 主动更新 View | Spring MVC |
| **MVP** | Presenter | **通过接口操作 View** | View 暴露接口，Presenter 调用 | Android、WinForms |
| **MVVM** | ViewModel | **双向绑定** | 框架自动同步 View 和 ViewModel | Vue、WPF |

记忆抓手：**MVC 直接操作、MVP 走接口、MVVM 双向绑定**。这是三者最核心的差异点。

### REST 架构

REST（Representational State Transfer，表述性状态转移）是 Roy Fielding 在 **2000 年**博士论文提出的 Web 服务架构风格，核心约束：

- **资源（Resource）为中心**：一切皆资源，用 URI 标识（如 `/orders/123`）。
- **统一接口用 HTTP 方法**：GET 查、POST 增、PUT 改、DELETE 删。
- **无状态（Stateless）**：每次请求自带所有信息，服务器不保存客户端会话状态。
- **分层**：客户端不感知中间是否有代理、网关。
- **可缓存**：响应可标注是否可缓存。

REST 因简单、轻量、HTTP 友好，成为前后端分离时代的主流 API 风格。

### 前后端分离架构

**结构**：前端（SPA 单页应用，Vue/React）独立部署，通过 REST API 调后端服务；后端只提供数据接口，不渲染页面。

```
[前端 SPA] --REST API--> [后端服务] <---> [数据库]
（独立部署）               （只返回JSON）
```

**优点**：前后端解耦、可独立开发部署、前端可多端复用（Web/小程序/App 调同一套 API）、用户体验好（SPA 局部刷新）。

**缺点**：SEO 不友好（需 SSR 补救）、接口设计要规范、安全要靠 Token/网关保障。

## 实践

### 真题示例

**考法一：三模式辨析**

> 例：以下关于 MVC、MVP、MVVM 的比较，正确的是（　）。
> A. 三者中 Controller、Presenter、ViewModel 都直接操作 View
> B. MVP 中 Presenter 通过接口操作 View，View 不直接访问 Model
> C. MVVM 中 ViewModel 需要手动调用代码更新 View
> D. MVC 中 View 完全被动，不接受 Model 通知

**答题思路**：B 对。A 错（只有 MVC 直接操作，MVP 走接口，MVVM 双向绑定）；C 错（MVVM 是双向绑定自动同步，不需手动）；D 错（MVC 中 View 可观察 Model 变化刷新）。答案 **B**。

**考法二：C/S vs B/S**

> 例：相比两层 C/S 架构，B/S 架构的主要优势是（　）。
> A. 客户端交互能力更强
> B. 客户端零安装、易升级、跨平台
> C. 服务器压力更小
> D. 网络安全性更高

**答题思路**：B/S 最大优势是浏览器通用、零安装易升级跨平台。答案 **B**。A 是 C/S 的优势；C 错（B/S 服务器压力更大）；D 错（B/S 暴露公网，安全挑战更大）。

**考法三：REST 特征**

> 例：（2024年·综合知识·Web架构相关，考生回忆）以下属于 REST 架构风格核心约束的是（　）。
> A. 服务端保存客户端会话状态
> B. 资源用 URI 标识，使用 HTTP 方法操作
> C. 必须使用 SOAP 协议
> D. 客户端必须感知后端服务实现细节

**答题思路**：REST 核心约束包括资源用 URI 标识、用 HTTP 方法、无状态、分层。B 对。A 错（REST 无状态）；C 错（REST 用 HTTP，不用 SOAP）；D 错（REST 客户端不感知实现）。答案 **B**。

### 答题踩分词

- 三模式题：**MVC 直接操作、MVP 走接口、MVVM 双向绑定**。
- 选 B/S 时踩分词：**零安装、易升级、跨平台、服务端集中维护**。
- 选 REST 时踩分词：**资源为中心、URI 标识、HTTP 方法、无状态、分层、可缓存**。
- 选前后端分离时踩分词：**前后端解耦、独立部署、API 复用、SPA 局部刷新**。

### 案例论文应用

Web 架构是论文高频主题，要写出"为什么这么分层"。典型写法：

> "系统采用 B/S 架构下的前后端分离方案：前端用 Vue 构建 SPA，通过 REST API 与后端交互；后端按表现层（API 网关）、应用层、领域层、数据访问层分层。选择该方案的理由是：① 用户分布广、需跨平台，B/S 零安装最合适；② 前后端分离让前端可独立迭代、后端 API 可同时支撑 Web 和后续小程序；③ REST 无状态便于水平扩展，配合网关做鉴权和限流。表现层采用 MVVM 模式，借助 Vue 的双向绑定减少手动 DOM 操作，提升开发效率。"

这种"用了什么、为什么、各层怎么落、模式怎么选"的写法，是论文 Web 架构论证的标准结构。

### 记忆口诀

- 架构演进：**"两层胖、三层抽、B/S 瘦、N 层细"**。
- 三模式区别：**"MVC 直接操作，MVP 走接口，MVVM 双向绑"**。
- REST 约束：**"资源 URI、HTTP 方法、无状态、可缓存、分层"**。
- C/S vs B/S：**"C/S 体验强要安装，B/S 零装跨平台"**。

### 自测题

**自测 1**：MVP 模式中 Presenter 为什么通过接口操作 View？这样设计的好处是什么？

> 参考答案：MVP 中 View 暴露一组接口，Presenter 通过接口操作 View，View 不直接访问 Model。好处是 Presenter 不依赖具体 View 实现，可对 Presenter 进行单元测试（用 Mock View 替代真实 View），解耦了视图与逻辑，提升可测试性。

**自测 2**：MVVM 的"双向绑定"和 MVC 的"Controller 直接操作 View"有何区别？双向绑定解决了什么问题？

> 参考答案：MVC 中 Controller 需手动编写代码更新 View；MVVM 中 View 与 ViewModel 通过框架双向绑定，ViewModel 数据变化时 View 自动刷新、View 上用户改动也自动同步回 ViewModel，无需手动操作 DOM。双向绑定解决了 MVC/MVP 中大量手动视图更新代码的问题，让开发者只关注状态（ViewModel），提升开发效率。

**自测 3**：某互联网应用要求"用户无需安装客户端、可跨平台访问、服务端可水平扩展"。应选择 C/S 还是 B/S？为什么 REST 风格的 API 有利于水平扩展？

> 参考答案：应选 B/S——浏览器通用、零安装跨平台，满足前两点。REST 的无状态约束意味着每次请求自带全部信息、服务器不保存会话状态，因此任意请求可路由到任意服务器实例，天然支持水平扩展（加机器即可分担负载），这是有状态会话（如 Session 粘滞）做不到的。

### 一句话小结

Web 架构演进就是"客户端越来越瘦、层越拆越细"——**两层胖、三层抽、B/S 瘦、N 层细，三模式记牢"直接操作/走接口/双向绑"，REST 无状态好扩展**。
