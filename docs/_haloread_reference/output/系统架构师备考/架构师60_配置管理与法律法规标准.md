# 第60章：配置管理三类库三类基线，法律法规标准送分不丢分

配置管理是中级《系统集成项目管理工程师》案例分析必考内容，法律法规与标准化是综合知识的"送分题"。每年合计 6-10 分。配置库三类、基线三类、配置审计两种、变更控制流程——这几块背熟，案例题不丢分。法律法规七法记住关键词，综合稳收。以下法规条款以现行有效法规为准。

## 概念

### 这章考什么
- **科目**：中级综合知识（每年 6-10 题）+ 案例分析（配置管理必考）
- **分值**：中级约 6-10 分；案例 5-15 分
- **考频**：高频，配置管理案例必考
- **难度**：中等，记忆为主

核心类比：**配置管理**是项目的"档案室+门禁"——配置项是档案，基线是锁死的档案柜（改要走流程），三类库是不同权限的房间（动态库随便改、受控库锁版本、静态库存历史），CCB 是门禁审批员，配置审计是定期盘点；**法律法规**是项目红线——合同法管交易、招投标法管选商、政府采购法管公款、著作权法管创作、网络安全法/数据安全法/个人信息保护法管数据流转；**标准化**是行业通用语言——GB 是国标、ISO 是国际、IEEE 是工程、ITU 是电信。

## 原理

### 【中级重点】配置管理：三类库三类基线两种审计

配置管理（Configuration Management）是标识、控制、状态记录、审计配置项的全过程。核心活动：**配置标识、配置控制、配置状态记录、配置审计**。

**配置项 CI（Configuration Item）分类**：

| 类别 | 例子 |
|---|---|
| **环境类** | 操作系统、编译器、开发工具 |
| **代码类** | 源代码、可执行程序、脚本 |
| **文档类** | 需求规格、设计文档、用户手册、测试报告 |
| **数据类** | 测试数据、配置文件、初始化数据 |

**三类基线**（高频命门）：

| 基线类型 | 别名 | 建立时机 | 内容 |
|---|---|---|---|
| **功能基线** | Functional Baseline | 需求评审通过后 | 经评审的需求规格 |
| **分配基线** | Allocated Baseline | 概要设计/系统设计后 | 设计规格、接口定义 |
| **产品基线** | Product Baseline | 系统测试通过后 | 最终产品、源码、文档、运行环境 |

口诀：**功能需求、分配设计、产品交付**——三基线对应需求、设计、交付三个里程碑。

**三类配置库**（绝对命门）：

| 库类型 | 别名 | 权限 | 内容 | 谁能改 |
|---|---|---|---|---|
| **动态库** | 开发库、工作库 | 开发者自由 | 正在开发的代码、文档 | 开发者个人 |
| **受控库** | 主库、基线库 | 严格受控 | 已基线化的配置项 | 走变更流程 |
| **静态库** | 产品库、备份库 | 只读归档 | 历史版本、发布产品 | 几乎不改 |

口诀：**动态随便改、受控走变更、静态只归档**。

**易错点**：
- 动态库是开发者的"草稿本"，不受变更控制。
- 受控库存的是基线，任何修改必须走 CCB 变更流程。
- 静态库存历史版本和发布产品，一般只读，用于回溯和审计。

**变更控制流程**（与整体变更控制一致）：

```
提变更申请 → CCB 评估 → 批准/拒绝 → 实施变更 → 配置审计验证 → 更新基线与状态报告
```

**版本控制**：用版本号标识配置项状态。常见规则如 X.Y.Z（主版本.次版本.修订号）。每次变更产生新版本，旧版本保留可追溯。

**配置审计——两种**（高频易错命门）：

| 审计类型 | 查什么 | 通俗说法 |
|---|---|---|
| **功能配置审计 FCA** | 配置项是否满足需求、功能是否完整、测试是否通过 | "功能满不满足" |
| **物理配置审计 PCA** | 配置项是否齐全、版本是否正确、是否与基线一致 | "东西在不在、版本对不对" |

口诀：**功能查满足、物理查齐全**。

**配置状态报告 CSR（Configuration Status Report）**：记录配置项的状态变化（新建、变更、基线化、发布）。内容包括：配置项标识、当前版本、状态、变更历史、责任人。

**CCB（Change Control Board，变更控制委员会）**：审批配置项变更的常设机构，由项目经理、技术负责人、质量负责人、客户代表等组成。

### 【中级重点】信息系统集成及服务管理与 ITSS

**信息系统集成**：将计算机软件、硬件、网络通信、数据等整合为满足用户需求的系统。集成内容包括：网络集成、数据集成、应用集成、业务流程集成。

**信息系统集成及服务管理**涵盖：规划设计、系统集成、运行维护、技术服务。

**ITSS（Information Technology Service Standards，信息技术服务标准）**：我国自主制定的信息技术服务标准库，覆盖 IT 运维、云服务、咨询服务等。

ITSS 核心要素（人员、流程、技术、资源，"四要素"）：
- **人员**：人员能力、岗位结构
- **流程**：服务管理流程
- **技术**：技术服务能力
- **资源**：数据、知识、服务台

ITSS 与 ITIL 的区别：ITIL 是英国的最佳实践框架，ITSS 是中国国标化的服务标准。中级常考"ITSS 是我国自主制定"这一关键词。

### 【中级重点】法律法规七法必背

| 法律 | 核心要点 |
|---|---|
| **合同法（民法典合同编）** | 要约（希望订立合同的意思表示）、承诺（同意要约）、违约责任（继续履行、补救、赔偿） |
| **招标投标法** | 必须招标项目（大型基础设施、公共事业、国有资金等）、评标委员会 5 人以上单数专家≥2/3 |
| **政府采购法** | 公开招标为原则（限额以上必须公开招标）、采购方式：公开招标/邀请招标/竞争性谈判/单一来源/询价 |
| **著作权法** | 软件著作权自开发完成自动产生、登记非强制、保护期作者终生+死后50年 |
| **网络安全法** | 关键信息基础设施保护、网络安全等级保护制度、网络运营者主体责任 |
| **数据安全法** | 数据分类分级保护、重要数据目录、跨境数据传输监管 |
| **个人信息保护法** | 知情同意原则、最小必要原则、目的限制、公开透明 |

**合同法要点**：
- **要约**：希望与他人订立合同的意思表示，内容具体确定，表明承诺后受约束。
- **承诺**：受要约人同意要约的意思表示，到达要约人时生效。
- **要约邀请**：引诱他人向自己发出要约（如招标公告、商业广告），不是要约。
- **违约责任**：继续履行、采取补救措施、赔偿损失。

**招投标法要点**（此处聚焦必须招标项目与评标规则）：
- 必须招标项目：大型基础设施、公用事业关系社会公共利益公众安全的项目；全部或部分使用国有资金投资或国家融资的项目；使用国际组织或外国政府贷款援助资金的项目。
- 评标委员会：5 人以上单数，技术经济专家≥2/3。
- 中标后必须按招标文件和投标文件订立书面合同。

**政府采购法要点**：
- 公开招标为原则，限额以上必须公开招标。
- 五种采购方式：公开招标、邀请招标、竞争性谈判、单一来源采购、询价。
- 政府采购对象：货物、工程、服务。

**著作权法要点**（与软件著作权衔接）：
- 著作权自动产生，无需登记。
- 软件著作权由**中国版权保护中心**登记，登记非强制但为强证据。
- 职务开发归单位，委托开发无约定归开发方，合作开发共同享有。

**网络安全法要点**：
- **关键信息基础设施 CII**：公共通信、能源、交通、金融等行业一旦受损严重危害国计民生的设施，重点保护。
- **网络安全等级保护制度**：网络运营者按等级履行安全保护义务。
- 网络运营者收集个人信息须明示并取得同意。

**数据安全法要点**：
- **数据分类分级**：按重要程度分级，重要数据实行目录管理。
- 重要数据出境须安全评估。
- 数据处理者承担数据安全保护义务。

**个人信息保护法要点**：
- **知情同意原则**：处理个人信息须告知并取得同意。
- **最小必要原则**：只收集与处理目的最小相关的信息。
- **目的限制**：不得用于约定目的之外。
- 敏感个人信息（生物识别、宗教信仰、医疗健康、金融账户、行踪轨迹）需单独同意。

### 【中级重点】标准化分级与国际标准组织

**标准分级**（自上而下）：

| 级别 | 代号 | 制定机构 | 性质 |
|---|---|---|---|
| **国家标准** | GB（强制）/GB/T（推荐） | 国家标准化管理委员会 | 全国统一 |
| **行业标准** | 如 HJ（环保）、YD（通信）、JR（金融） | 国务院相关行政主管部门 | 行业内 |
| **地方标准** | DB | 省级市场监管局 | 地方统一 |
| **企业标准** | Q | 企业自己 | 企业内 |
| **团体标准** | T | 学会、协会等社会团体 | 团体内 |

**标准性质**：强制性标准（必须执行，如 GB）和推荐性标准（鼓励采用，如 GB/T）。注意：国家标准中 GB 为强制、GB/T 为推荐。

**国际标准组织**：

| 组织 | 全称 | 领域 |
|---|---|---|
| **ISO** | International Organization for Standardization | 综合性国际标准 |
| **IEC** | International Electrotechnical Commission | 电工电子 |
| **IEEE** | Institute of Electrical and Electronics Engineers | 电子电气工程、计算机 |
| **ITU** | International Telecommunication Union | 电信 |

**常见软考相关标准编号**：
- **ISO 9000 系列**：质量管理体系。
- **ISO/IEC 12207**：软件生命周期过程。
- **ISO/IEC 20000**：IT 服务管理。
- **ISO/IEC 27001**：信息安全管理体系。
- **ISO 33000 系列**：软件过程评估（取代 SPICE/CMMI 对应标准）。
- **GB/T 8566**：信息技术 软件生存周期过程（对应 ISO/IEC 12207）。
- **GB/T 16260**：软件工程 产品质量（对应 ISO/IEC 9126）。

## 实践

### 【高级衔接】配置管理与 DevOps/CI/CD 的关系

中级教材讲的传统配置管理（手动基线、CCB 审批）与现代 DevOps/CI/CD 看似矛盾，实则一脉相承。架构师要看到两者关系：

| 传统配置管理 | DevOps/CI/CD | 关系 |
|---|---|---|
| 受控库手动基线 | Git 分支保护 + 自动化构建 | 受控库变成 Git 受保护分支 |
| CCB 评审 | Pull Request 评审 + 自动化门禁 | CCB 评审人变成 PR Reviewer |
| 配置审计 FCA/PCA | 自动化测试 + 制品扫描 | 审计内嵌到流水线 |
| 配置状态报告 | 制品仓库 + 可追溯性矩阵 | 报告自动生成 |
| 三类库（动态/受控/静态） | feature 分支/main 分支/Tag 发布 | 库的概念映射到分支策略 |

**关键洞察**：DevOps 不是取消配置管理，而是把配置管理**自动化、左移、内嵌**。CI/CD 流水线本身就是配置管理的现代实现：
- **CI（持续集成）**：每次提交触发构建测试，等于动态库→受控库的自动门禁。
- **CD（持续交付/部署）**：通过自动化测试的制品进制品仓库，等于受控库→静态库。
- **基础设施即代码 IaC**：环境配置也版本化，环境类配置项纳入 Git 管理。

架构师推动 DevOps 时，要确保：① 受保护分支策略（main 分支不可直接 push）；② 强制 PR 评审（CCB 的现代版）；③ 制品签名和可追溯（配置审计的自动化）；④ 流水线门禁（测试通过才能合并）。这样配置管理的"标识、控制、状态、审计"四大活动一个不少，只是从手动变自动。

### 【高级衔接】架构文档化的配置管理实践

架构文档（ADL 描述、4+1 视图、ADR 决策记录）本身是配置项，必须纳入配置管理。架构师实践要点：

- **ADL 与设计文档进受控库**：架构基线一旦确定，ADL 文档进受控库，改动走 CCB/PR 流程。
- **ADR 仓库版本化**：每个架构决策记录有编号、版本、状态（提议/接受/废弃），用 Git 管理。
- **架构与代码同源**：架构图用代码生成（如 PlantUML、Structurizr），避免文档与代码脱节。
- **架构基线与产品基线对齐**：每次产品发布，对应一份架构快照，便于回溯。

### 真题示例

**考法一：三类配置库**

> 例：（2024年·中级综合知识·配置管理相关，考生回忆）在配置管理中，存放正在开发的代码和文档、开发者可自由修改的配置库是（　）。
> A. 动态库　B. 受控库　C. 静态库　D. 产品库

**答题思路**：开发者自由改的是动态库（开发库）。答案 **A**。

**考法二：配置审计**

> 例：（2023年·中级案例分析·配置管理相关，考生回忆）项目发布前，配置管理员要检查配置项是否齐全、版本是否与基线一致。这属于哪种配置审计？另一种配置审计查什么？

**答题思路**：检查齐全和版本一致性属**物理配置审计 PCA**。另一种是**功能配置审计 FCA**，查配置项是否满足需求、功能是否完整、测试是否通过。

**考法三：法律法规**

> 例：（2024年·中级综合知识·法律法规相关，考生回忆）根据《个人信息保护法》，处理个人信息应遵循的原则不包括（　）。
> A. 知情同意　B. 最小必要　C. 目的限制　D. 任意使用

**答题思路**：知情同意、最小必要、目的限制都是法定原则，"任意使用"明显错误。答案 **D**。

**考法四：标准化**

> 例：（2023年·中级综合知识·标准化相关，考生回忆）下列标准代号中，表示推荐性国家标准的是（　）。
> A. GB　B. GB/T　C. DB　D. Q

**答题思路**：GB 强制国标、GB/T 推荐国标、DB 地方标准、Q 企业标准。答案 **B**。

### 案例分析答题公式

配置管理案例题套路："**缺什么活动 + 该活动该怎么做 + 不做后果 + 改进措施**"四段式。

> 例：某项目开发人员直接在受控库修改代码，未走变更流程，导致版本混乱。请指出问题并给出改进方案。

**答题模板**：
1. **问题定位**：受控库的配置项必须走变更控制流程，开发者不能直接修改。该团队绕过 CCB，违反配置控制规范。
2. **该怎么做**：① 任何对受控库配置项的修改须提书面变更申请；② CCB 评估变更影响（范围、进度、成本、质量）；③ 批准后由配置管理员在受控库实施修改并生成新版本；④ 配置审计（物理+功能）验证；⑤ 更新基线和配置状态报告。
3. **不做后果**：版本混乱、基线失效、无法追溯、测试依据丢失、发布风险高。
4. **改进措施**：① 制定配置管理计划明确三类库权限；② 用 Git 分支策略实现受控库保护（main 分支禁止直接 push）；③ 强制 PR 评审替代手动 CCB；④ 培训团队配置管理规范；⑤ 定期做配置审计。

### 常见法律法规清单

```
合同法（民法典合同编）：要约-承诺-违约责任
```

### 自测题

**自测 1**：配置管理中"功能基线""分配基线""产品基线"分别建立于什么时机？分别对应什么内容？

> 参考答案：
> - **功能基线**：需求评审通过后建立，内容是经评审的需求规格说明书。
> - **分配基线**：系统设计/概要设计后建立，内容是设计规格、接口定义、各子系统分配的需求。
> - **产品基线**：系统测试通过后建立，内容是最终产品、源代码、文档、运行环境。

**自测 2**：某开发团队为赶进度，绕过 CCB 直接在受控库修改了已基线化的源代码。请指出违反了什么配置管理原则，并给出正确的变更流程。

> 参考答案：违反了**配置控制**原则——受控库中的基线配置项任何修改都必须走变更控制流程，不能直接改。正确流程：① 提交书面变更申请；② CCB 评估变更对范围、进度、成本、质量、风险的影响；③ CCB 批准或拒绝；④ 批准后由配置管理员在受控库实施修改并生成新版本；⑤ 配置审计（物理+功能）验证修改正确；⑥ 更新基线和配置状态报告。

**自测 3**：《个人信息保护法》的"最小必要原则"是什么含义？某 APP 收集用户手机号用于注册登录，但又索取通讯录、定位权限，是否合规？为什么？

> 参考答案：**最小必要原则**指只收集与处理目的直接相关、对实现目的最小必要的个人信息，不得过度收集。APP 收集手机号用于注册登录属必要；但若同时索取通讯录、定位，而这些权限与注册登录目的无关（除非注册流程明确需要），则违反最小必要原则，不合规。敏感信息（通讯录含他人信息、定位属行踪轨迹）还需单独取得用户同意。合规做法是分场景授权，注册时只索取手机号，需要通讯录/定位的功能再单独申请。

### 一句话小结

配置管理三类库（**动态随便改、受控走变更、静态只归档**）、三类基线（**功能-分配-产品**）、两种审计（**功能查满足、物理查齐全**）是案例必考命门；法律七法（**合同/招标/采购/著作权/网安/数安/个保**）记关键词、标准分级（**GB 强制、GB/T 推荐**）辨代号——双线并行，中级 6-10 分稳收。
