# 核心观点

二叉树与红黑树是**"树形结构的核心基石"**，从完全二叉树的数组表示到红黑树的工程应用，体现了**"用适度平衡换取稳定性能"**的设计哲学。理解二叉树的遍历、BST的有序性、AVL的严格平衡、红黑树的近似平衡，以及递归树的复杂度分析，能帮你在 TreeMap/TreeSet、数据库索引、Linux调度等场景中做出正确的技术选型。**"为什么有了散列表还需要二叉树？"** 这个问题的答案揭示了数据结构选择的深层逻辑：有序性、范围查询、稳定性往往比理论上的O(1)更重要。

## 核心概念解析（20% 重点深度解析）

### 1. 完全二叉树的数组存储：堆的实现基础

- **定义**：完全二叉树是一种特殊的二叉树，叶子节点都在最底下两层，最后一层的叶子节点都靠左排列
- **为什么重要**：
  - ⭐ **堆的实现基础**：堆就是完全二叉树，用数组存储最节省空间
  - **不需要指针**：通过下标计算即可找到父子节点关系
  - **缓存友好**：数组连续内存，CPU缓存命中率高

- **关键机制——寻址公式**：

  **核心思想：下标映射代替指针**
  ```
  假设节点X存储在数组下标为i的位置（i从0开始）：

  父节点：parent = (i-1) / 2
  左子节点：left = 2 * i + 1
  右子节点：right = 2 * i + 2

  （如果i从1开始：
   parent = i / 2
   left = 2 * i
   right = 2 * i + 1）
  ```

  **经典案例：为什么完全二叉树适合用数组存储？**
  ```
  对比三种二叉树的数组存储效率：

  ① 满二叉树（Perfect Binary Tree）
     特征：每个节点都有0或2个子节点，所有叶子在同一层
     数组存储：✅ 零浪费，完美利用每一个位置
     示例（7个节点）：
     数组：[A, B, C, D, E, F, G]
     结构：
           A(0)
          /   \
        B(1)  C(2)
       / \    / \
     D(3)E(4)F(5)G(6)

  ② 完全二叉树（Complete Binary Tree）⭐最常用
     特征：除最后一层外全满，最后一层靠左排列
     数组存储：✅ 几乎零浪费（最多浪费不到一半位置）
     示例（6个节点）：
     数组：[A, B, C, D, E, F]
     结构：
           A(0)
          /   \
        B(1)  C(2)
       / \
     D(3)E(4)
     /
   F(5)  ← 最后一层靠左

  ③ 普通二叉树（非完全二叉树）
     特征：不满足上述条件
     数组存储：❌ 大量浪费（为了保持下标关系需要留空）
     示例（退化为链表的二叉树）：
     数组：[A, _, B, _, _, _, C]  ← 浪费3个位置！
     结构：
     A
      \
       B
        \
         C

  结论：只有完全二叉树（包括满二叉树）才适合用数组存储！
  ```

- **性能特征**：
  ```
  空间复杂度：
  - 完全二叉树用数组：O(n)，无额外指针开销
  - 链式存储：O(n) + O(n)个指针（通常多占用50%-100%空间）

  时间复杂度（访问父/子节点）：
  - 数组存储：O(1) 直接通过公式计算
  - 链式存储：O(1) 通过指针直接访问

  工程意义：
  - 堆排序、优先级队列都用数组实现的完全二叉树
  - Java PriorityQueue、Python heapq 底层都是数组
  ```

### 2. 为什么有了散列表还需要二叉树？

- **定义**：散列表和二叉树都能支持动态数据的插入、删除、查找，但各有优劣
- **为什么重要**：
  - ⭐ **技术选型的核心问题**：什么时候用HashMap，什么时候用TreeMap？
  - 理解"理论最优 vs 实际可用"的权衡
  - 面试高频问题："你能说出至少5个必须用树的场景吗？"

- **关键机制——五大差异对比**：

  **差异①：有序性 vs 无序性**
  ```
  散列表：
  - ❌ 无序存储：元素位置由hash决定，与大小无关
  - ✅ 输出有序需要先排序：O(n log n)

  二叉搜索树（BST）：
  - ✅ 有序存储：中序遍历即有序输出：O(n)
  - 应用场景：
    * 数据库索引的ORDER BY操作
    * 范围查询：找出所有 score 在 [60, 80] 的学生
    * 找前驱后继节点

  实际代码示例（范围查询）：
  // BST支持高效的范围查询
  void rangeQuery(Node root, int low, int high) {
      if (root == null) return;
      if (root.val > low) {
          rangeQuery(root.left, low, high);  // 左子树可能存在
      }
      if (root.val >= low && root.val <= high) {
          print(root.val);  // 在范围内
      }
      if (root.val < high) {
          rangeQuery(root.right, low, high);  // 右子树可能存在
      }
  }
  // 时间复杂度：O(log n + k)，k=范围内元素个数
  // 散列表无法做到这一点！
  ```

  **差异②：性能稳定性**
  ```
  散列表：
  - 平均：O(1) ⭐看起来很美
  - 最坏：O(n) 当所有key冲突时（Hash DoS攻击）
  - 扩容时：单次O(n)，虽然均摊O(1)但会有偶发卡顿

  平衡二叉树（AVL/红黑树）：
  - 平均：O(log n)
  - 最坏：O(log n) ⭐非常稳定
  - 无扩容问题：每次操作都是可预测的

  实际影响：
  - Redis dict 用散列表：单线程模型，rehash会阻塞
  - Java TreeMap 用红黑树：性能稳定，适合实时系统
  - Linux进程调度用红黑树：不能接受偶发的长延迟
  ```

  **差异③：常数因子 vs 渐进复杂度**
  ```
  理论上：O(1) < O(log n)
  实际上：不一定！

  散列表的实际开销：
  1. 计算哈希值：需要遍历key的所有字符
     hash("a_very_long_string_key") → 多次乘法和取模
  2. 处理冲突：链表遍历或开放寻址探测
     负载因子0.75时，平均探测1.5次
  3. 缓存不友好：链表节点分散在内存中

  红黑树的实际开销：
  1. 比较操作：通常只是整数比较（1-2个CPU周期）
  2. 路径长度：log₂(100万) ≈ 20次比较
  3. 缓存较友好：局部性原理（虽然不如数组）

  结论：当n不是特别大时（比如几万以内），红黑树可能更快！
  ```

  **差异④：实现复杂度**
  ```
  散列表需要考虑：
  ✅ 哈希函数设计（均匀性、抗碰撞、速度）
  ✅ 冲突解决策略（链地址法 vs 开放寻址法）
  ✅ 动态扩容机制（何时扩容、如何rehash、渐进式rehash）
  ✅ 负载因子选择（0.5? 0.75? 0.9?）
  ✅ 线程安全（ConcurrentHashMap的分段锁/CAS）
  ✅ Hash DoS防御（ cryptographic hash function）

  平衡二叉树只需要考虑：
  ✅ 平衡维护（旋转+变色，规则固定且成熟）

  所以：如果语言标准库已经提供了红黑树实现，
        直接用就行；但散列表往往需要自己调参。
  ```

  **差异⑤：空间利用率**
  ```
  散列表（开放寻址法）：
  - 负载因子必须 < 0.7-0.8，否则性能急剧下降
  - 实际空间利用率：70-80%
  - 例如存100万个元素，需要130万个槽位

  红黑树：
  - 每个节点需要：data + left + right + parent + color
  - 空间开销：约3-5倍的数据大小
  - 但不需要预分配，按需分配

  权衡：
  - 内存充足时：红黑树更灵活
  - 内存紧张时：精心调优的散列表更省空间
  ```

### 3. 二叉树的核心概念体系

- **定义**：二叉树是每个节点最多有两个子节点的树形结构，是树形数据结构的基础
- **为什么重要**：
  - ⭐ **面试必考**：前中后序遍历、BST操作、平衡树
  - **实际应用广泛**：文件系统、数据库索引、决策树、语法树
  - **理解递归的最佳载体**：树的问题天然适合递归思维

- **关键机制——四大遍历方式**：

  **① 前序遍历（Pre-order）：根→左→右**
  ```
  递推公式：preOrder(r) = print r → preOrder(r.left) → preOrder(r.right)

  应用场景：
  - 树的拷贝：先复制根节点，再复制左右子树
  - 序列化/反序列化：可以唯一确定一棵二叉树
  - 表达式树的前缀表达式（波兰表达式）：+ A B

  代码实现：
  void preOrder(Node root) {
      if (root == null) return;
      print(root.val);        // 先处理根
      preOrder(root.left);    // 再处理左子树
      preOrder(root.right);   // 最后处理右子树
  }

  非递归实现（用栈）：
  void preOrderIterative(Node root) {
      if (root == null) return;
      Stack<Node> stack = new Stack<>();
      stack.push(root);
      while (!stack.isEmpty()) {
          Node node = stack.pop();
          print(node.val);
          if (node.right != null) stack.push(node.right);  // 先压右
          if (node.left != null) stack.push(node.left);    // 后压左
      }
  }
  ```

  **② 中序遍历（In-order）：左→根→右** ⭐BST专用
  ```
  递推公式：inOrder(r) = inOrder(r.left) → print r → inOrder(r.right)

  应用场景：
  - BST的有序输出：O(n)时间得到有序序列
  - 表达式树的中缀表达式：A + B（需要加括号才能消除歧义）
  - 验证是否为BST：检查输出序列是否严格递增

  经典示例——验证二叉搜索树：
  boolean isValidBST(Node root) {
      return isValidBST(root, Long.MIN_VALUE, Long.MAX_VALUE);
  }

  boolean isValidBST(Node node, long min, long max) {
      if (node == null) return true;
      if (node.val <= min || node.val >= max) return false;
      return isValidBST(node.left, min, node.val) &&
             isValidBST(node.right, node.val, max);
  }
  // 时间复杂度：O(n)
  // 关键：不只是判断左<根<右，还要保证整个左子树<根<整个右子树
  ```

  **③ 后序遍历（Post-order）：左→右→根**
  ```
  递推公式：postOrder(r) = postOrder(r.left) → postOrder(r.right) → print r

  应用场景：
  - 计算目录大小：先计算子目录大小，再加起来
  - 表达式树的后缀表达式（逆波兰表达式）：A B +
  - 删除整棵树：先删除子树，再删除根节点（防止内存泄漏）

  实际示例——计算目录大小：
  int calculateSize(DirectoryNode dir) {
      if (dir.isFile()) return dir.fileSize;

      int totalSize = 0;
      for (DirectoryNode child : dir.children) {
          totalSize += calculateSize(child);  // 先计算子目录
      }
      dir.size = totalSize;  // 最后再设置当前目录的大小
      return totalSize;
  }
  ```

  **④ 层序遍历（Level-order/BFS）** ⭐用队列
  ```
  思想：广度优先搜索，一层一层地遍历

  应用场景：
  - 按层级打印二叉树
  - 计算树的高度/最大宽度
  - 二叉树的右视图（每层最右边的节点）

  代码实现：
  void levelOrder(Node root) {
      if (root == null) return;

      Queue<Node> queue = new LinkedList<>();
      queue.offer(root);

      while (!queue.isEmpty()) {
          int levelSize = queue.size();  // 当前层的节点数
          for (int i = 0; i < levelSize; i++) {
              Node node = queue.poll();
              print(node.val);
              if (node.left != null) queue.offer(node.left);
              if (node.right != null) queue.offer(node.right);
          }
          println();  // 每层结束后换行
      }
  }
  // 时间复杂度：O(n)
  // 空间复杂度：O(w)，w=树的最大宽度
  ```

- **BST（二叉搜索树）的核心操作**：
  ```
  BST定义：
  - 左子树所有节点值 < 根节点值 < 右子树所有节点值
  - 左右子树也都是BST

  查找操作：O(log n) 平均，O(n) 最坏（退化成链表）
  Node find(int data) {
      Node p = root;
      while (p != null) {
          if (data < p.data) p = p.left;
          else if (data > p.data) p = p.right;
          else return p;  // 找到了
      }
      return null;  // 未找到
  }

  插入操作：找到合适的位置，作为叶子节点插入
  删除操作（分三种情况）：
  ① 叶子节点：直接删除
  ② 只有一个子节点：用子节点替换
  ③ 有两个子节点：用右子树的最小节点（或左子树的最大节点）替换

  中序遍历：O(n) 输出有序序列
  最大/最小值：O(log n) 一直向左/向右走
  前驱/后继节点：O(log n)
  ```

### 4. 红黑树：工程中的"平衡之选"

- **定义**：一种近似平衡的二叉搜索树，通过颜色标记和旋转操作保持大致平衡
- **为什么重要**：
  - ⭐ **工业界标准的平衡树**：Java TreeMap/TreeSet、C++ STL map/set、Linux CFS调度器
  - **性能稳定的代表**：插入/删除/查找都是稳定的 O(log n)
  - **理解"近似平衡"的设计哲学**：不过度追求完美，而是追求实用

- **关键机制——红黑树的5大性质**：

  **性质清单（必须同时满足）**
  ```
  ① 每个节点要么是红色，要么是黑色
  ② 根节点是黑色
  ③ 叶子节点（NIL空节点）是黑色
  ④ 红色节点的两个子节点都是黑色（不能有连续的红色节点）
  ⑤ 从任意节点到其所有叶子节点的路径中，包含相同数量的黑色节点

  性质④的含义：红色节点被黑色节点隔开
     合法的：黑→红→黑→红→黑
     非法的：黑→红→红→黑  （相邻红色！）

  性质⑤的含义：黑高相等（Black Height）
     从任一节点出发，到所有叶子的路径上的黑色节点数相同
     这保证了树不会严重偏向一边
  ```

  **红黑树 vs AVL树：工程选择的权衡**
  ```
  对比维度          AVL树                红黑树
  ──────────────────────────────────────────────
  平衡程度          严格平衡（|高度差|≤1） 近似平衡（高度≤2log n）
  查找效率          更快（树更矮）         稍慢（但仍是O(log n)）
  插入/删除效率     较慢（频繁旋转）       ⭐更快（旋转次数少）
  旋转次数（平均）   约1-2次              约0-1次
  适用场景          查找密集型            ⭐增删查均衡型

  为什么工程中选择红黑树？
  1. 大多数场景是增删查混合操作，而非纯查找
  2. AVL树的严格平衡维护成本高（每次插入/删除都可能触发多次旋转）
  3. 红黑树的"近似平衡"已经足够好（最多比AVL高1倍）
  4. 红黑树的实现经过数十年验证，稳定可靠

  典型应用：
  - Java 8+ HashMap：当链表长度≥8时转为红黑树
  - Java TreeMap/TreeSet：底层就是红黑树
  - C++ std::map/std::set：通常用红黑树实现
  - Linux内核CFS（Completely Fair Scheduler）：用红黑树管理进程
  - nginx的timer管理：用红黑树管理定时任务
  ```

- **关键机制——左旋和右旋操作**：

  **旋转的本质：调整树的形态，保持BST性质**

  **左旋（Left Rotate）围绕节点x**
  ```
  旋转前：
      x
       \
        y
       / \
      α   z

  旋转后：
        y
       /
      x
       \
        z
      /
     α

  操作步骤（伪代码）：
  leftRotate(x):
      y = x.right        // 1. 设y为x的右子节点
      x.right = y.left   // 2. y的左子树变为x的右子树
      if y.left != NIL:
          y.left.parent = x
      y.parent = x.parent// 3. y的父节点更新为x的父节点
      if x.parent == NIL:
          root = y       // x原来是根节点
      else if x == x.parent.left:
          x.parent.left = y
      else:
          x.parent.right = y
      y.left = x          // 4. x成为y的左子节点
      x.parent = y

  注意：左旋只改变结构，不影响中序遍历结果（BST性质保持不变）
  ```

  **右旋（Right Rotate）围绕节点y —— 与左旋对称**
  ```
  旋转前：
        y
       /
      x
       \
        z
      /
     α

  旋转后：
      x
       \
        y
       /
      α

  操作步骤：与左旋完全对称（left↔right互换）
  ```

  **插入操作的平衡调整（3种情况）**
  ```
  前提：新插入的节点默认为红色（为什么？因为插入红色不会破坏性质⑤）

  CASE 1：叔叔节点是红色
  解决方案：变色（父、叔变黑，祖父变红），然后继续向上处理
  示例：
       B(黑)              R(红)  ← 祖父变红，继续向上检查
      / \                 / \
    R(红)R(红)  →      B(黑)B(黑)  ← 父、叔变黑
    /                     /
  a(新红)               a(红)

  CASE 2：叔叔节点是黑色，且当前节点是右子节点
  解决方案：先左旋父节点，转化为CASE 3
  示例：
       B(黑)              B(黑)          B(黑)
        \                    \              \
        R(红)     左旋       a(红)  →       R(红)  → CASE3
        / \       父节点                   /
      a(新红)NIL(黑)                          NIL(黑)

  CASE 3：叔叔节点是黑色，且当前节点是左子节点
  解决方案：右旋祖父节点 + 父兄变色
  示例：
       B(黑)               R(红)  ← 父变黑，祖父变红
      / \         右旋     / \
    R(红)NIL(黑) →      a(红)B(黑)  ← 完成！
    /
  a(新红)

  时间复杂度：O(log n)（最多3次旋转，O(1)次变色操作）
  ```

### 5. 递归树分析算法复杂度

- **定义**：将递归算法的分解过程画成一棵树，通过分析树的结构来推导时间复杂度
- **为什么重要**：
  - ⭐ **直观易懂**：比递推公式更形象，尤其适合快速排序的平均情况分析
  - **通用方法**：适用于大多数分治算法的复杂度分析
  - **面试加分项**：能展示你对算法本质的理解深度

- **关键机制——三大实战案例**：

  **案例①：归并排序（Merge Sort）—— 完美二叉树**
  ```
  递推公式：T(n) = 2T(n/2) + O(n)

  递归树结构：
  层级    问题规模    本层代价    节点个数
  ──────────────────────────────────────
  1       n          O(n)       1
  2       n/2        O(n)       2        (每个n/2，共2个)
  3       n/4        O(n)       4
  ...     ...        ...        ...
  log₂n   1          O(n)       n^(log₂n)=n

  总代价 = 每层代价 × 层数 = O(n) × O(log n) = O(n log n)

  关键观察：
  - 归并排序的递归树是满二叉树（每次一分为二）
  - 每层的总代价都是O(n)（合并操作的总工作量）
  - 树的高度是log₂n
  ```

  **案例②：快速排序（Quick Sort）平均情况 —— 不完美但可控**
  ```
  假设每次分区比例为 1:9（比较极端的不均匀情况）

  递推公式：T(n) = T(n/10) + T(9n/10) + O(n)

  递归树结构：
  层级    左路径规模    右路径规模    本层代价
  ─────────────────────────────────────────
  1       n            n            O(n)
  2       n/10         9n/10         O(n)
  3       n/100        81n/100       O(n)
  ...     ...          ...           ...

  最短路径（一直走左边）：log₁₀n 层
  最长路径（一直走右边）：log₁₀/₉n 层

  总代价：介于 O(n log₁₀n) 和 O(n log₁₀/₉n) 之间
  统一写成：O(n log n)  （底数不影响大O表示法）

  重要结论：
  即使分区极度不均匀（1:99、1:999...），只要比例是固定的常数，
  快速排序的时间复杂度仍然是 O(n log n)！
  这就是快排在实践中表现优异的原因。
  ```

  **案例③：斐波那契数列（递归版）—— 指数级灾难**
  ```
  递推公式：T(n) = T(n-1) + T(n-2) + O(1)

  递归树结构：
  层级    节点总数    本层代价
  ────────────────────────────
  1       1          1
  2       2          2
  3       4          4
  4       8          8
  ...     ...        ...
  n       2^n        2^n

  总代价 ≈ 2^n  （指数级！）

  对比优化版本（动态规划/迭代）：
  int fib(int n) {
      if (n <= 2) return 1;
      int a = 1, b = 1;
      for (int i = 3; i <= n; i++) {
          int temp = a + b;
          a = b;
          b = temp;
      }
      return b;
  }
  // 时间复杂度：O(n)  ← 从指数级降到线性！
  // 空间复杂度：O(1)
  ```

  **主定理（Master Theorem）—— 通用公式**
  ```
  对于形如 T(n) = aT(n/b) + f(n) 的递归式：

  设 c_crit = log_b(a) （临界指数）

  情况1：f(n) = O(n^c) 其中 c < c_crit
        → T(n) = Θ(n^c_crit)
        例：T(n) = 4T(n/2) + n → Θ(n²)

  情况2：f(n) = Θ(n^c_crit * log^k n)
        → T(n) = Θ(n^c_crit * log^(k+1) n)
        例：T(n) = 2T(n/2) + n → Θ(n log n)

  情况3：f(n) = Ω(n^c) 其中 c > c_crit，且满足正则条件
        → T(n) = Θ(f(n))
        例：T(n) = 2T(n/2) + n² → Θ(n²)

  记忆技巧：
  - 叶子节点工作占主导 → 情况1（分治的工作量大）
  - 根节点工作和叶子相当 → 情况2（平衡）
  - 根节点工作占主导 → 情况3（合并工作量大）
  ```

## 次要信息速览（80% 快速带过）

- 【满二叉树 vs 完全二叉树 vs 完满二叉树的区别】：
  - **满二叉树（Full Binary Tree）**：每个节点都有0或2个子节点（没有只有1个子节点的节点）
  - **完全二叉树（Complete Binary Tree）**：除最后一层外全满，最后一层靠左
  - **完满二叉树（Perfect Binary Tree）**：所有叶子节点在同一层，所有非叶子节点都有2个子节点
  - **关系**：完满 ⊂ 满 ⊂ 完全（完满是满的特例，满是完全的特例）

- 【AVL树简介】：
  - **定义**：严格的平衡二叉树，任何节点的左右子树高度差不超过1
  - **优势**：查找速度最快（树的高度最小）
  - **劣势**：插入/删除时旋转次数多，维护成本高
  - **适用场景**：查找密集、增删较少的场景（如静态字典）

- 【Redis ZSET的实现选择：跳表 vs 红黑树】：
  - **跳表的优势**：
    - 实现简单（比红黑树容易得多）
    - 支持范围查询（天然有序链表）
    - 并发友好（局部性修改）
  - **Redis作者antirez的解释**："跳表在实现简单性和性能之间取得了更好的平衡"
  - **结论**：没有绝对的最优，只有最适合

- 【二叉树的常见变种】：
  - **B树/B+树**：多路搜索树，用于数据库索引（减少磁盘I/O）
  - **Trie树**：字典树，用于前缀匹配（搜索引擎的关键词提示）
  - **线段树**：区间树，用于区间查询（求区间最大值/最小值）
  - **笛卡尔树**：用于RMQ（Range Minimum Query）问题

- 【树的高度相关概念辨析】：
  - **高度（Height）**：从该节点到叶子节点的最长路径边数（从下往上数，根节点起始于0）
  - **深度（Depth）**：从根节点到该节点的边数（从上往下数，根节点为0）
  - **层数（Level）**：深度+1（根节点在第1层）

# 思考问答

1. **为什么红黑树新插入的节点默认是红色？如果是黑色会怎样？**
   - 提示：考虑性质⑤（黑高相等）和性质④（不能有连续红色节点）

2. **给定一组数据{1, 3, 5, 6, 9, 10}，可以构建出多少种不同的二叉搜索树？（注意：不是普通二叉树，是BST！）**
   - 提示：考虑卡特兰数（Catalan Number），n个节点可以构建 C(2n,n)/(n+1) 种BST

3. **如果让你设计一个支持以下操作的数据结构：insert(key)、delete(key)、find(key)、findMin()、findMax()、rangeQuery(low, high)，你会选择什么？为什么？**
   - 提示：权衡各数据结构的优劣势，考虑时间复杂度和实现难度

4. **快速排序在最坏情况下（数组已有序或逆序）的时间复杂度是O(n²)。如何避免这种情况？**
   - 提示：随机化选择基准值（pivot）、三数取中法、切换到插入排序（小数组）

5. **为什么Java HashMap在JDK 8中引入了红黑树？阈值为什么是8？**
   - 提示：泊松分布、链表长度概率、树化/退化的性能平衡

# 实践要点

- ✅ **优先使用语言内置的平衡树实现**：TreeMap/TreeSet（Java）、std::map（C++）、bisect模块（Python）
- ✅ **理解红黑树的核心思想即可**：除非特殊需求，否则不要手写红黑树（容易出错且难以调试）
- ✅ **掌握二叉树的三种递归遍历和非递归写法**：这是面试高频题，必须熟练
- ✅ **使用递归树分析复杂度**：遇到递归算法时，先画图再推导，比纯数学推导更直观
- ✅ **根据场景选择合适的树结构**：
  - 需要有序性+范围查询 → 红黑树/跳表
  - 需要极致查找性能 → AVL树
  - 需要磁盘存储 → B+树
  - 需要前缀匹配 → Trie树
- ❌ **在BST中存储重复数据时不做处理**：应采用链表+节点 或 放入右子树两种策略之一
- ❌ **忽略树的退化问题**：普通BST在极端情况下会退化成链表，务必使用平衡树
- ❌ **混淆树的"高度"、"深度"、"层数"三个概念**：它们虽然相关但定义不同
