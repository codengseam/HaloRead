# 核心观点

数组是**最基础的数据结构**，其核心价值在于**连续内存带来的随机访问O(1)**，但代价是插入删除需要数据搬移O(n)。理解数组的内存模型和从0开始编号的原因，能帮你更好地使用ArrayList/Vector等容器，并在性能敏感场景做出正确选择。

## 核心概念解析（20% 重点深度解析）

### 1. 数组的内存模型与随机访问机制

- **定义**：数组是一种**线性表**数据结构，用**连续的内存空间**存储**相同类型**的一组数据
- **为什么重要**：
  - 连续内存 + 相同类型 → 支持通过下标**直接计算内存地址**（随机访问O(1)）
  - 这是数组区别于链表的根本特性，决定了两者的适用场景
  - 理解这个原理才能明白"为什么数组从0开始"
- **关键机制**——寻址公式：
  ```
  a[i]_address = base_address + i * data_type_size
  ```
  - `base_address`：数组首地址（内存中第一个元素的地址）
  - `i`：下标（偏移量）
  - `data_type_size`：每个元素占用的字节数（int=4字节, long=8字节）

  **举例**：int[] a = new int[10]，首地址1000
  - a[0]的地址 = 1000 + 0×4 = **1000**
  - a[3]的地址 = 1000 + 3×4 = **1012**
  - a[9]的地址 = 1000 + 9×4 = **1036**

- **时间/空间复杂度**：
  - 随机访问：**O(1)**（核心优势！）
  - 插入/删除：**O(n)**（需要搬移元素保持连续性）
  - 空间效率：高（无额外指针开销，但可能浪费预分配空间）
- **实际应用**：
  - **MySQL索引**：B+树的叶子节点用数组存储，利用随机访问快速定位
  - **Redis**：String底层是SDS（简单动态字符串），本质是字符数组
  - **Java ArrayList**：封装了动态扩容的数组，大部分场景替代原生数组
- **关联知识**：[[T00_算法导论与复杂度]], [[T02_链表]]

### 2. 为什么数组从0开始编号？

- **定义**：大多数编程语言（C/Java/Python/Go）的数组下标从0开始而非1
- **为什么重要**：
  - 这是一个经典的面试题，考察对数组内存模型的理解深度
  - 理解偏移量的概念有助于学习指针、内存管理等底层知识
- **关键机制**：

  从**偏移量（offset）**的角度理解：
  - 下标 = 距离首地址的**偏移个数**
  - a[0] 表示偏移0个元素 → 就是首地址本身
  - a[k] 表示偏移k个元素 → 地址 = base + k×size

  如果从1开始编号：
  ```
  a[k]_address = base_address + (k-1) * data_type_size  // 多一次减法！
  ```

  **性能影响**：
  - 每次随机访问多一次减法运算
  - 对于CPU来说就是多一条指令
  - 数组是最基础的操作，积少成多影响显著

  **历史原因**：
  - C语言设计者从0开始计数（受汇编语言影响，偏移量思维）
  - 后续语言为了降低C程序员的学习成本，沿用了这个惯例
  - 反例：Matlab从1开始，Python支持负数下标

- **实际应用**：
  - 写循环时习惯用 `for (i=0; i<n; i++)` 而非 `for (i=1; i<=n; i++)`
  - 处理二维数组时注意行优先 vs 列优先的内存布局
  - 与数学中的1-indexed notation做转换时容易出错（off-by-one错误）
- **关联知识**：[[T02_链表]]（对比链表无法随机访问）, [[T08_二叉树]]（堆用数组存储）

### 3. 动态数组的扩容机制

- **定义**：当数组容量不足时，自动申请更大的内存空间并复制原有数据的机制
- **为什么重要**：
  - Java ArrayList / C++ vector / Python list 都依赖此机制
  - 理解扩容原理才能避免性能陷阱（如频繁扩容导致GC压力）
  - 均摊复杂度分析的经典案例
- **关键机制**（以Java ArrayList为例）：

  **扩容流程**：
  1. 检测容量不足（size == capacity）
  2. 申请新数组（通常是原大小的 **1.5倍** 或 **2倍**）
  3. 将旧数组数据**逐个拷贝**到新数组（O(n)操作）
  4. 释放旧数组，引用指向新数组

  **均摊时间复杂度分析**：
  - 假设当前容量为K，已满
  - 第K+1次插入触发扩容：需要拷贝K个元素（成本K）
  - 接下来K-1次插入都是O(1)
  - 总共K次操作的成本 = K（拷贝） + K-1（简单插入） ≈ 2K
  - **均摊每次插入成本 = 2K/K = O(1)** ✅

  **不同语言的扩容策略**：

| 语言 | 容器 | 扩容倍数 | 特点 |
|------|------|----------|------|
| Java | ArrayList | 1.5倍 | `(oldCapacity >> 1)` 位运算优化 |
| C++ | vector | 2倍 | 更激进，峰值内存更高 |
| Python | list | 约1.125倍 | 渐进式增长，更保守 |
| Go | slice | 2倍（<1024），1.25倍（≥1024） | 分段策略 |

- **时间/空间复杂度**：
  - 单次最坏：**O(n)**（触发扩容时）
  - 均摊：**O(1)**（摊还分析）
  - 空间：可能有**50%~100%的浪费**（未使用的预分配空间）
- **实际应用**：
  ```java
  // ✅ 推荐：预先指定容量避免多次扩容
  ArrayList<User> users = new ArrayList<>(10000);
  for (int i = 0; i < 10000; i++) {
      users.add(fetchUser(i));
  }

  // ❌ 不推荐：默认容量10，需要多次扩容
  ArrayList<User> users = new ArrayList();  // 触发约13次扩容
  ```

  - **高性能场景**：网络框架、游戏引擎等底层开发直接用固定大小数组
  - **业务开发**：直接用ArrayList即可，性能损耗可忽略
- **关联知识**：[[T00_算法导论与复杂度]]（均摊分析）, [[T02_链表]]（天然动态扩容）

## 次要信息速览（80% 快速带过）

- 【数组vs链表完整对比】：

| 操作 | 数组 | 链表 | 适用场景 |
|------|------|------|----------|
| 随机访问 | O(1) ⭐ | O(n) | 需要频繁按索引查询 |
| 插入头部 | O(n) | O(1) | 频繁在头部插入删除 |
| 插入尾部 | O(1)/O(n)* | O(1) | 只在尾部追加 |
| 删除指定位置 | O(n) | O(1)** | 频繁中间插入删除 |
| 内存连续性 | 是（缓存友好） | 否（缓存不友好） | CPU密集型计算 |
| 动态扩容 | 需要手动/自动 | 天然支持 | 数据规模不确定 |

  *数组如果在尾部插入且未满则是O(1)，否则需扩容O(n)
  **链表如果已知前驱节点则是O(1)，否则需遍历O(n)

- 【低效的插入/删除及优化技巧】：
  - **无序数组插入优化**：将第k位元素移到末尾，新元素放入第k位 → **O(1)**
    - 应用：快速排序的分区操作（partition）
  - **批量删除优化**：先标记删除，最后统一搬移 → 减少搬移次数
    - 类比：JVM的**标记-清除（Mark-Sweep）垃圾回收算法**

- 【数组越界的危险性】：
  - C/C++：越界访问不会报错，可能覆盖其他变量或导致死循环
  - 案例：`for (i=0; i<=3; i++) arr[i]=0` 当arr[3]恰好覆盖变量i时会无限循环
  - Java：抛出ArrayIndexOutOfBoundsException（安全但影响性能）

- 【何时使用原生数组 vs 容器】：
  - ✅ 用数组：基本类型(int/long)、已知大小、多维数组、极致性能要求
  - ✅ 用容器(ArrayList)：对象集合、大小不确定、业务代码、快速开发
  - ❌ 避免：在循环内频繁拼接字符串（用StringBuilder代替String+）

# 思考问答

1. **二维数组 `a[m][n]` 的内存寻址公式是什么？**
   - 提示：考虑行优先存储（C/Java）和列优先存储（Fortran）的区别

2. **为什么ArrayList选择1.5倍扩容而不是2倍？各有什么优劣？**
   - 提示：考虑内存浪费率 vs 扩容次数的权衡

3. **以下代码的时间复杂度是多少？如何优化？**
   ```java
   int[] arr = new int[n];
   for (int i = 0; i < n; i++) {
       for (int j = 0; j < n; j++) {
           arr[j] = i * j;  // 注意这里是j不是i
       }
   }
   ```
   - 提示：虽然有两层循环，但内层操作是否真的执行了n²次？

4. **如果实现一个LRU缓存，用数组还是链表更好？为什么？**
   - 提示：考虑访问模式（按值查找 vs 按位置移动）

5. **在高并发场景下，ArrayList线程安全吗？如果不安全有哪些替代方案？**
   - 提示：CopyOnWriteArrayList / Collections.synchronizedList / 手动加锁

# 实践要点

- ✅ **预分配容量**：创建ArrayList时尽量指定初始容量（`new ArrayList<>(expectedSize)`）
- ✅ **理解原理**：知道容器底层的扩容机制，避免踩坑（如批量add前ensureCapacity）
- ✅ **选择合适结构**：频繁头部操作用LinkedList，频繁随机访问用ArrayList
- ❌ **在循环内拼接String**：使用StringBuilder，否则O(n²)复杂度
- ❌ **忽视数组越界**：C/C++中务必检查边界，Java中也要避免异常开销
- ❌ **滥用Arrays.asList()**：返回的是固定长度列表，不能add/remove
