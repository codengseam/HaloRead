# 核心观点

字符串匹配是**"在文本中快速定位模式串位置"**的核心技术，从BF的暴力O(mn)到KMP/BM的O(m+n)，体现了**"利用已匹配信息避免重复比较"**的优化思想。KMP的next数组（部分匹配表）通过预处理模式串的前后缀信息，在失配时智能跳转而非回退；BM算法则从坏字符和好后缀两个维度最大化滑动距离，实际表现往往优于KMP。Trie树将字符串匹配转化为树路径查找，实现O(L)的前缀搜索；AC自动机融合Trie和KMP失败指针，一次扫描完成多模式串匹配。理解这些算法的设计哲学，能帮你在搜索引擎、敏感词过滤、IDE自动补全等场景中设计出高效的文本处理系统。

## 核心概念解析（20% 重点深度解析）

### 1. 从 BF 到 RK：哈希优化的威力

- **定义**：BF（Brute Force）是最朴素的字符串匹配算法，RK（Rabin-Karp）通过哈希优化减少字符比较次数
- **为什么重要**：
  - ⭐ **理解算法优化思路的起点**：从暴力到巧妙的完整演进过程
  - **哈希在字符串处理中的经典应用**：为后续理解BK树、SimHash打下基础

- **关键机制——两种算法的对比**：

  **① BF 算法（暴力匹配）**
  ```
  核心思路：
  模式串 P 与主串 T 的每个可能位置逐一比对

  示例：
  主串 T = "ABABCABD"
  模式串 P = "ABD"

  第1轮：T[0..2] vs P
  A B A B C A B D
  A B D           → T[2]='A' ≠ P[2]='D'，失配！

  第2轮：T[1..3] vs P（右移1位）
  A B A B C A B D
    A B D         → T[3]='B' ≠ P[2]='D'，失配！

  第3轮：T[2..4] vs P（再右移1位）
  A B A B C A B D
      A B D       → T[4]='C' ≠ P[2]='D'，失配！

  ... 继续右移 ...

  第6轮：T[5..7] vs P
  A B A B C A B D
          A B D   → 全部匹配！✅ 返回位置5

  代码实现：
  int bruteForce(String text, String pattern) {
      int n = text.length(), m = pattern.length();
      for (int i = 0; i <= n - m; i++) {       // 尝试每个起始位置
          int j;
          for (j = 0; j < m; j++) {
              if (text.charAt(i + j) != pattern.charAt(j))
                  break;                          // 失配，尝试下一位置
          }
          if (j == m) return i;                   // 完全匹配
      }
      return -1;                                  // 未找到
  }

  时间复杂度分析：
  - 最好情况：O(m)（第一个位置就匹配或首字符就不匹配）
  - 最坏情况：O(mn)（如 T="AAAAA...", P="AAAB"）
    每次都匹配到最后一个字符才失配

  为什么最坏情况这么慢？
  - 每次失配只右移1位
  - 已匹配的信息完全没有被利用！
  ```

  **② RK 算法（哈希优化）**
  ```
  核心创新：先比较哈希值，再比较实际字符

  步骤分解：

  Step 1: 计算模式串P的哈希值 hash_p
  Step 2: 计算主串T中每个长度为m的子串的哈希值 hash_t[i]
  Step 3: 如果 hash_t[i] == hash_p，再进行精确字符比对（防止哈希碰撞）

  关键技巧：滚动哈希（Rolling Hash）
  假设字符集为小写字母 a-z（a=1, b=2, ..., z=26），基数 base=26

  子串 "abc" 的哈希值：
  hash = 1×26² + 2×26¹ + 3×26⁰ = 1×676 + 52 + 3 = 731

  滚动计算（从 "abc" 到 "bcd"）：
  新hash = (旧hash - 首字符贡献) × base + 新字符
        = (731 - 1×676) × 26 + 4
        = 55 × 26 + 4
        = 1434

  验证：bcd = 2×26² + 3×26¹ + 4×26⁰ = 1352 + 78 + 4 = 1434 ✅

  代码实现（简化版）：
  int rabinKarp(String text, String pattern) {
      int n = text.length(), m = pattern.length();
      long base = 256;                    // 字符集大小
      long mod = 101;                     // 大质数（减少碰撞）
      long highestPow = 1;
      for (int i = 0; i < m - 1; i++)
          highestPow = (highestPow * base) % mod;

      long hashP = 0, hashT = 0;
      for (int i = 0; i < m; i++) {
          hashP = (hashP * base + pattern.charAt(i)) % mod;
          hashT = (hashT * base + text.charAt(i)) % mod;
      }

      for (int i = 0; i <= n - m; i++) {
          if (hashP == hashT) {
              if (text.substring(i, i+m).equals(pattern))
                  return i;                 // 找到匹配
          }
          if (i < n - m) {
              // 滚动更新：去掉最左边的字符，加上新字符
              hashT = ((hashT - text.charAt(i) * highestPow) * base
                      + text.charAt(i + m)) % mod;
              if (hashT < 0) hashT += mod; // 处理负数
          }
      }
      return -1;
  }

  时间复杂度：
  - 平均/最好：O(m + n)（哈希计算O(n)+比对O(1)每次）
  - 最坏：O(mn)（所有子串哈希都碰撞，退化成BF）

  工程优化：
  ✅ 使用多个哈希函数降低碰撞概率（类似布隆过滤器）
  ✅ 选择大质数作为模数
  ⚠️ 注意整数溢出问题（使用long或BigInteger）
  ```

### 2. BM 算法：坏字符 + 好后缀的双重优化

- **定义**：Boore-Moore 算法从模式串末尾开始匹配，利用坏字符规则和好后缀规则决定最大滑动距离
- **为什么重要**：
  - ⭐ **实际应用中最快的单模式串匹配算法**：GNU grep、编辑器查找功能
  - **理解"反向思考"优化策略的经典案例**

- **关键机制——两大核心规则**：

  **① 坏字符规则（Bad Character Rule）**
  ```
  场景：从右向左匹配时遇到不匹配的字符

  示例：
  主串 T: ... C H I N A S E A M ...
  模式串 P:     E X A M P L E
                              ↑
                         失配位置：T[j]='M', P[i]='E'

  坏字符 = 'M'（主串中与模式串不匹配的字符）

  处理方式：
  在模式串 P 中查找坏字符 'M' 的位置：
  - 情况A：'M' 存在于 P[0..i-1] 中
    将 P 右移，使 P 中最靠右的 'M' 对齐到 T[j]

  - 情况B：'M' 不存在于 P 中
    将 P 直接滑过这个 'M'

  具体示例：
  T: G O O D M A N I S E X A M P L E
  P:     E X A M P L E              ← 从右向左比
                ↑
            T[8]='N' ≠ P[5]='L'

  坏字符='N'，检查 P 中是否有 'N'：
  P中没有'N' → 右移整个模式串长度=7位

  移动后：
  T: G O O D M A N I S E X A M P L E
  P:                   E X A M P L E  ← 继续匹配
  ```

  **② 好后缀规则（Good Suffix Rule）**
  ```
  场景：已经匹配了一段后缀，但在前一个位置失配

  示例：
  T: . . . A B C D A B E F . .
  P:       A B C D A B F
                        ↑
                   T[j]='F'≠P[i]='E'

  好后缀 = "ABF" 已经匹配的部分？不对！
  好后缀 = 已经成功匹配的后缀 = "AB"（从失配位置右边开始）

  处理方式（三步判断）：
  1. 在 P 中查找好后缀是否再次出现
     如果找到，右移使两处对齐

  2. 如果找不到完整的好后缀，找好后缀的真后缀
     是否等于 P 的某个真前缀

  3. 都找不到，右移整个模式串

  详细示例：
  T: . . . M Y B C X A B C D A B C . .
  P:         A B C D A B C
                        ↑
                   T中='D'≠P中='A'

  好后缀 = "BC"（已经匹配的部分）

  在P中找"BC"：
  P: A B C D A B C
          ^ ^       ← 这里有一处"BC"

  右移对齐：
  T: . . . M Y B C X A B C D A B C . .
  P:               A B C D A B C      ← 对齐
  ```

  **③ 综合使用：取两者最大值**
  ```
  最终滑动距离 = max(坏字符规则移动距离, 好后缀规则移动距离)

  为什么取最大？
  - 两个规则独立计算，各自保证不会漏掉匹配
  - 取最大值可以跳得更远，效率更高

  时间复杂度：
  - 最好情况：O(n/m)（每次都能滑过整个模式串）
  - 最坏情况：O(mn)（如 T="AAAAA...", P="BAAA"）
  - 平均情况：O(n)（实际表现优秀，通常是亚线性的）

  预处理空间：O(σ + m)
  - σ：字符集大小（坏字符散列表）
  - m：模式串长度（好后缀数组）

  为什么 BM 在实践中通常比 KMP 快？
  - BM 从右向左匹配，更容易触发大跨度滑动
  - 坏字符规则在随机文本上效果很好
  - 好后缀规则在重复模式上效果很好
  ```

### 3. KMP 算法：next 数组的精髓

- **定义**：Knuth-Morris-Pratt 算法通过 next 数组（部分匹配表）记录模式串自身的重复信息，失配时智能跳转
- **为什么重要**：
  - ⭐ **面试高频考点**：手写 next 数组构造是经典考题
  - **理解"利用已匹配信息"思想的典范**

- **关键机制——KMP 的核心难点：next 数组**

  **① 直观理解：为什么能跳转？**
  ```
  问题场景：
  T: A B A B C A B A B ...
  P:   A B A B D
              ↑
          T[6]='C' ≠ P[4]='D'，失配！

  BF 算法做法：P 回到起点，T 前进1位（浪费了前面已匹配的 "ABAB"）

  KMP 的洞察：
  已匹配部分："ABAB"
  观察发现：这个已匹配部分有相同的前缀和后缀！
  - 前缀 "AB" = 后缀 "AB"

  这意味着什么？
  - 既然 "ABAB" 匹配成功了
  - 而且它的前后缀都是 "AB"
  - 那么我们可以直接让 P 的前缀 "AB" 跳到已匹配后缀 "AB" 的位置
  - 不需要重新比较这 "AB" 了！

  图示：
  T: A B A B [C] A B A B ...
  P:   A B A B [D]
              ↑ 失配

  KMP 跳转后：
  T: A B A B [C] A B A B ...
  P:         A B [A B D]  ← P向右滑动，但不是从头开始
                  ↑ 接着比较 C 和 A

  节省了多少？
  - 不需要重新比较前面的 "AB"
  - 直接从失配点附近继续
  ```

  **② next 数组的精确定义**
  ```
  next[i] = P[0..i] 这个子串中，
           最长相等前后缀的长度

  通俗解释：
  next[i] 告诉我们：当第 i 个字符失配时，
  模式串应该跳回到哪个位置继续比较

  示例（P = "ABABD"）：

  i   P[i]   P[0..i]   最长相等前后缀   next[i]
  ──────────────────────────────────────────────
  0   A      A         ""(空串)           -1（哨兵）
  1   B      AB        ""                 0
  2   A      ABA       "A"               1
  3   B      ABAB      "AB"              2
  4   D      ABABD     ""                0

  验证 next[3]=2：
  P[0..3] = "ABAB"
  前缀："", "A", "AB", "ABA"
  后缀："", "B", "AB", "BAB"
  最长相等前后缀："AB"，长度=2 ✅

  特殊说明：
  - next[0] = -1 是约定（表示需要整体右移一位）
  - 有些实现用 next[0]=0 并做特殊处理
  ```

  **③ next 数组的构造算法（核心！）**
  ```
  void getNext(String p, int[] next) {
      int m = p.length();
      next[0] = -1;                           // 哨兵值
      int k = -1;                             // k = next[j-1]，即前一个位置的最长前缀长
      int j = 0;                              // 当前正在计算的位置

      while (j < m - 1) {
          if (k == -1 || p.charAt(j) == p.charAt(k)) {
              k++;
              j++;
              next[j] = k;                    // next[j] = k
          } else {
              k = next[k];                    // 回溯！关键步骤
          }
      }
  }

  执行过程演示（P="ABABC"）：

  初始：k=-1, j=0, next[0]=-1

  第1轮：k=-1 → 进入if分支
          k=0, j=1, next[1]=0

  第2轮：p[1]='B' ≠ p[0]='A' → else分支
          k = next[0] = -1

  第3轮：k=-1 → 进入if分支
          k=0, j=2, next[2]=0  （p[2]='A'≠p[0]='A'? 不对，相等！）

  让我重新仔细追踪...

  正确追踪（P="ABABC"）：
  j=0: next[0]=-1, k=-1
  j=0→1: k=-1, p[0]==p[-1]? k==-1成立 → k=0,j=1,next[1]=0
  j=1→2: k=0, p[1]='B'≠p[0]='A' → k=next[0]=-1
         k=-1 → k=0,j=2,next[2]=0 (p[2]='A'==p[0]='A')
  j=2→3: k=0, p[3]='B'==p[1]='B' → k=1,j=3,next[3]=1
  j=3→4: k=1, p[4]='C'≠p[1]='B' → k=next[1]=0
         k=0, p[4]='C'≠p[0]='A' → k=next[0]=-1
         k=-1 → k=0,j=4,next[4]=0

  最终 next = [-1, 0, 0, 1, 0]

  时间复杂度：O(m)
  - while 循环虽然看起来有嵌套，但 k 只增不减（总共增加m次）
  - k 通过 next[k] 回溯时虽然减小，但总的回溯次数不超过增加次数
  - 类似于均摊分析
  ```

  **④ KMP 完整匹配流程**
  ```
  int kmpSearch(String text, String pattern) {
      int n = text.length(), m = pattern.length();
      int[] next = new int[m];
      getNext(pattern, next);                 // 预处理

      int i = 0, j = 0;                       // i:主串位置, j:模式串位置
      while (i < n && j < m) {
          if (j == -1 || text.charAt(i) == pattern.charAt(j)) {
              i++;                            // 匹配，各进一步
              j++;
          } else {
              j = next[j];                    // 失配，跳转！
          }
      }

      if (j == m) return i - j;               // 匹配成功
      return -1;                              // 匹配失败
  }

  时间复杂度：O(m + n)
  - 预处理 next 数组：O(m)
  - 匹配过程：O(n)（i 单调递增，j 通过 next 跳转但不回退）

  空间复杂度：O(m)（存储 next 数组）
  ```

### 4. Trie 树：字符串的字典结构

- **定义**：Trie（前缀树/字典树）是一种多叉树结构，用于高效存储和检索字符串集合
- **为什么重要**：
  - ⭐ **前缀搜索的最优数据结构**：搜索引擎自动补全、拼写检查
  - **空间换时间的典型案例**

- **关键机制——Trie 的结构与操作**：

  **① 数据结构**
  ```
  图示示例（存储：apple, app, apt, banana, bat）

           root（空）
            |
        a --|-- b
        |     |
    p -- |--  a
    |   |     |
    p   t     n --|-- t
    |   |        |  |
    l   |        a  a
    |   |        |  |
    e   |        n  t
        |        |
       [end]    [end]

  节点结构（Java实现）：
  class TrieNode {
      TrieNode[] children;    // 子节点数组（假设只有26个小写字母）
      boolean isEnd;          // 标记是否为单词结尾
      int count;              // 该单词出现的次数（可选）

      TrieNode() {
          children = new TrieNode[26];  // 初始化为null
          isEnd = false;
          count = 0;
      }
  }

  class Trie {
      private TrieNode root;

      public Trie() {
          root = new TrieNode();
      }
  }
  ```

  **② 三大基础操作**
  ```
  操作1：插入（Insert）- O(L)，L为字符串长度

  void insert(String word) {
      TrieNode node = root;
      for (char c : word.toCharArray()) {
          int index = c - 'a';             // 字符映射到下标
          if (node.children[index] == null) {
              node.children[index] = new TrieNode();  // 创建新节点
          }
          node = node.children[index];     // 向下移动
      }
      node.isEnd = true;                   // 标记单词结束
      node.count++;
  }

  操作2：查找（Search）- O(L)

  boolean search(String word) {
      TrieNode node = root;
      for (char c : word.toCharArray()) {
          int index = c - 'a';
          if (node.children[index] == null) {
              return false;                 // 路径不存在
          }
          node = node.children[index];
      }
      return node.isEnd;                    // 必须是完整单词
  }

  操作3：前缀搜索（StartsWith）- O(L)

  boolean startsWith(String prefix) {
      TrieNode node = root;
      for (char c : prefix.toCharArray()) {
          int index = c - 'a';
          if (node.children[index] == null) {
              return false;
          }
          node = node.children[index];
      }
      return true;                          // 前缀存在即可
  }
  ```

  **③ 工程优化**
  ```
  空间优化方案：

  方案1：排序字典树（Sorted Trie / Ternary Search Tree）
  - 每个节点只有3个子节点：left(<), equal(=), right(>)
  - 空间从 O(Σ×N) 降低到 O(N)
  - 但查找时间变为 O(L log Σ)

  方案2：用 HashMap 替代数组
  - children: Map<Character, TrieNode>
  - 只存储存在的字符，节省空间
  - 查找时间略慢（哈希开销）

  方案3：压缩Trie（Radix Tree / Patricia Trie）
  - 合并只有一个孩子的节点
  - 边上存储字符串而非单个字符
  - 大幅减少节点数量

  应用案例：
  1. 搜索引擎自动补全
     - 用户输入 "ap" → 提示 ["apple", "app", "apt", "application"...]
     - 实现：BFS收集以该前缀开头的所有单词

  2. 拼写检查
     - 输入 "aple" → 提示 "apple"（Levenshtein距离+Trie）

  3. IP路由最长前缀匹配
     - CIDR路由表中查找最长匹配前缀

  4. 词频统计
     - 大规模文本中统计每个词的出现次数
  ```

### 5. AC 自动机：多模式串匹配的终极武器

- **定义**：Aho-Corasick 自动机结合了 Trie 树和 KMP 的失败指针，能在 O(n) 时间内同时匹配多个模式串
- **为什么重要**：
  - ⭐ **敏感词过滤系统的核心技术**：内容审核、垃圾邮件检测
  - **理解"自动机"思想的绝佳案例**

- **关键机制——AC 自动机的三步构建**：

  **① 构建 Trie 树**
  ```
  模式串集合：{ "he", "she", "his", "hers" }

  构建Trie（与普通Trie相同）：

          root
           |
           h -- s

        e     i
        |     |
   [he]  r    s

  同时标记每个模式串的结束节点
  ```

  **② 构建失败指针（Failure Link）⭐核心**
  ```
  失败指针类似于 KMP 的 next 数组：
  当当前字符匹配失败时，应该跳转到哪个状态？

  构建方法（BFS 层序遍历）：

  void buildFailureLinks() {
      Queue<TrieNode> queue = new LinkedList<>();

      // 第一层节点的失败指针指向 root
      for (TrieNode child : root.children.values()) {
          child.fail = root;
          queue.offer(child);
      }

      while (!queue.isEmpty()) {
          TrieNode curr = queue.poll();

          for (Map.Entry<Character, TrieNode> entry :
               curr.children.entrySet()) {
              char c = entry.getKey();
              TrieNode child = entry.getValue();
              TrieNode failNode = curr.fail;

              // 沿着失败链往上找，直到找到有c子节点的节点
              while (failNode != root &&
                     !failNode.children.containsKey(c)) {
                  failNode = failNode.fail;
              }

              if (failNode.children.containsKey(c)) {
                  child.fail = failNode.children.get(c);
              } else {
                  child.fail = root;
              }

              queue.offer(child);
          }
      }
  }

  直观理解：
  - 对于节点 "she" 的 'e'
  - 它的父节点 's' 的失败指针指向 'h'
  - 检查 'h' 是否有 'e' 子节点 → 有！（就是 "he"）
  - 所以 'e'(she) 的失败指针指向 'e'(he)
  ```

  **③ 匹配过程**
  ```
  String acSearch(String text) {
      TrieNode state = root;
      StringBuilder result = new StringBuilder();

      for (int i = 0; i < text.length(); i++) {
          char c = text.charAt(i);

          // 沿着失败链找可转移的状态
          while (state != root &&
                 !state.children.containsKey(c)) {
              state = state.fail;
          }

          if (state.children.containsKey(c)) {
              state = state.children.get(c);
          } else {
              state = root;
          }

          // 检查是否匹配到了模式串（包括通过失败指针）
          TrieNode temp = state;
          while (temp != root) {
              if (temp.isEnd) {
                  result.append("Found: ").append(temp.word)
                        .append(" at position ").append(i).append("\n");
              }
              temp = temp.fail;  // 继续沿失败链检查
          }
      }
      return result.toString();
  }

  时间复杂度：
  - 预处理：O(M×L)（M=模式串数量，L=平均长度）
  - 匹配：O(N + Z)（N=文本长度，Z=匹配次数）
  - 空间：O(M×L×Σ)（Σ=字符集大小）

  示例：
  文本： "ushers"
  模式串：["he", "she", "his", "hers"]

  匹配过程：
  u → s → h → e → r → s
  state: root → s → sh → she(匹配!) → he(失败链匹配!) → hers(匹配!) → ...

  结果：
  Found: she at position 2
  Found: he at position 3
  Found: hers at position 4
  ```

## 次要信息速览（80% 快速带过）

- 【字符串匹配算法对比总结】：
  | 算法 | 预处理时间 | 匹配时间 | 空间 | 特点 |
  |------|-----------|---------|------|------|
  | BF | O(1) | O(mn) | O(1) | 简单粗暴，适合短模式串 |
  | RK | O(m) | O(n)平均 | O(1) | 哈希优化，适合多模式串初步筛选 |
  | BM | O(m+σ) | O(n)平均，最坏O(mn) | O(m+σ) | 实际最快，从右向左匹配 |
  | KMP | O(m) | O(n+m) | O(m) | 理论保证O(n)，适合流式数据 |
  | AC | O(M×L) | O(N+Z) | O(M×L×Σ) | 多模式串匹配最优解 |

- 【KMP 的工程变体】：
  - **MP 算法**：KMP 的简化版本，next 数组定义略有不同
  - **Booth 算法**：用于寻找字符串的最小旋转表示
  - **Sunday 算法**：BM 的改进版，关注主串中下一个字符

- 【Trie 的常见变种】：
  - **双数组 Trie（Double-Array Trie）**：用两个数组模拟Trie，空间极省，用于日语输入法
  - **基数树（Radix Tree/Patricia Trie）**：压缩单分支节点，Linux内核用于内存管理
  - **前缀哈希树（Prefix Hash Tree）**：分布式系统中用于范围查询

- 【实际应用场景】：
  ```
  1. IDE 的 Ctrl+F 功能
     - 通常使用改进的 Boyer-Moore 或其变种
     - 支持正则表达式时会切换到 NFA/DFA 引擎

  2. 搜索引擎的关键词高亮
     - 使用 AC 自动机一次性匹配所有关键词
  ```
