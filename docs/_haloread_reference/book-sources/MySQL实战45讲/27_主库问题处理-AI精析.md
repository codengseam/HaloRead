# 核心知识清单(20%重点)

## 🔹 一主多从架构的切换挑战

**场景**:主库A故障后,A'成为新主库,从库B、C、D需要重新指向A'

**核心问题**:从库B如何找到新主库A'的精确同步位点?

- 原来记录的是A的位点,但相同日志在A和A'上的位点不同
- 位点不精确会导致重复执行事务,引发主键冲突(1062错误)或删除失败(1032错误)

## 🔹 GTID(全局事务ID) - 核心解决方案

### GTID定义
```
GTID = server_uuid:gno
```

- **server_uuid**:实例首次启动自动生成,全局唯一
- **gno**:整数,事务提交时分配,每次递增

### GTID核心机制

**每个MySQL实例维护一个GTID集合,对应"执行过的所有事务"**

#### gtid_next的两种模式:

1. **automatic模式**(默认)
   - MySQL自动分配`server_uuid:gno`
   - 记录binlog时先写入`SET @@SESSION.GTID_NEXT='server_uuid:gno'`
   - 将GTID加入实例的GTID集合

2. **指定GTID模式**
   - 通过`set gtid_next='current_gtid'`指定
   - 如果current_gtid已存在于GTID集合 → **事务被忽略**
   - 如果不存在 → 分配给当前事务,gno不递增

**关键洞察**:GTID已存在则跳过事务,这是解决主备切换位点问题的核心

### GTID模式启动
```sql
gtid_mode=on
enforce_gtid_consistency=on
```

## 🔹 基于GTID的主备切换

### 切换命令(无需指定位点)
```sql
CHANGE MASTER TO 
MASTER_HOST=$host_name 
MASTER_PORT=$port 
MASTER_USER=$user_name 
MASTER_PASSWORD=$password 
master_auto_position=1
```

### 自动找位点逻辑

设A'的GTID集合为set_a,B的GTID集合为set_b:

1. B连接A',发送set_b
2. A'计算set_a与set_b的差集
3. 判断A'是否包含差集所需的所有binlog
   - 不包含 → 返回错误(拒绝同步)
   - 包含 → 从第一个不在set_b的事务开始发送

**设计思想**:主备关系必须保证日志完整性,主库负责判断

### GTID集合示例
- 切换前:B的GTID集合 = `server_uuid_of_A:1-N`
- 切换后:B的GTID集合 = `server_uuid_of_A:1-N, server_uuid_of_A':1-M`

## 🔹 GTID实战:跳过冲突事务

**场景**:从库X同步主库Y的事务时出现主键冲突

**解决方案**:提交空事务,将冲突GTID加入X的GTID集合
```sql
set gtid_next='aaaaaaaa-cccc-dddd-eeee-ffffffffffff:10';
begin;
commit;
set gtid_next=automatic;
start slave;
```

**原理**:GTID已存在,后续同步时自动跳过该事务

## 🔹 GTID在在线DDL中的应用

**场景**:双M结构下,在备库Y加索引,避免影响主库X

**传统方法**:关闭binlog → 导致数据和日志不一致

**GTID方法**:
1. 在Y上执行DDL(不关闭binlog)
2. 查出DDL对应的GTID = `server_uuid_of_Y:gno`
3. 在X上执行空事务:
```sql
set GTID_NEXT="server_uuid_of_Y:gno";
begin;
commit;
set gtid_next=automatic;
start slave;
```

**效果**:Y的DDL有binlog记录,X不会执行该DDL

# 次要信息速览(80%快速带过)

## 【基于位点的主备切换问题】
- 位点获取方法:等待中转日志同步→show master status→mysqlbinlog解析
- 位点不精确原因:切换瞬间可能已有事务同步到从库
- 跳过错误方法:`set global sql_slave_skip_counter=1`或`slave_skip_errors="1032,1062"`
- 风险:跳过错误仅适用于切换场景,稳定后需清空参数

## 【一主多从架构背景】
- 应用场景:读多写少,读写分离
- 主库职责:所有写入+部分读
- 从库职责:分担读请求
- 切换复杂性:从库需重新指向新主库

## 【GTID技术细节】
- 格式对比:官方`source_id:transaction_id` vs 实际`server_uuid:gno`
- gno vs transaction_id:gno在提交时分配,transaction_id在执行时分配(回滚也递增)
- Previous_gtids:binlog文件开头记录,用于快速定位GTID和启动时解析

# 思考问答

1. **为什么GTID模式下,主库发现从库需要的binlog已删除会直接报错,而不是像基于位点模式那样继续同步?**

   提示:思考"日志完整性保证"的设计理念差异。基于位点的协议由备库决定,GTID协议由主库保证完整性。

2. **在GTID模式下,如果从库执行start slave后主库报错"需要的binlog已删除",有哪些解决方案?**

   提示:考虑4种场景 - 允许主从不一致、要求主从一致、有其他从库保留全量binlog、binlog有备份。

3. **为什么在线DDL场景下,使用GTID方法比关闭binlog更优?**

   提示:从"数据一致性"和"日志完整性"两个维度分析。关闭binlog会导致数据和日志不一致,GTID方法既保留日志又避免执行。

# 实践要点

## GTID模式切换检查清单
- [ ] 确认MySQL版本支持GTID(5.6+)
- [ ] 配置gtid_mode=on和enforce_gtid_consistency=on
- [ ] 使用master_auto_position=1建立主备关系
- [ ] 监控GTID集合增长情况
- [ ] 定期备份binlog,避免主备切换时日志缺失

## 常见错误处理
- **1062错误**:唯一键冲突 → GTID空事务跳过
- **1032错误**:删除数据找不到行 → GTID空事务跳过
- **日志缺失错误**:重新搭建从库或应用缺失的binlog备份
