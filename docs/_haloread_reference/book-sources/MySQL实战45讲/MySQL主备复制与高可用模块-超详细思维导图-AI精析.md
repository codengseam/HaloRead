# 🔄 MySQL 主备复制与高可用模块 — 超详细思维导图

## 一、📊 主备复制全景概览

### 1.1 为什么需要主备复制？🎯

- **高可用（High Availability, HA）** 🛡️
  - **核心目标**：主库故障时快速切换到备库
  - **业务价值**：
    - 减少服务中断时间（从小时级降到秒级）
    - 保证业务连续性（7×24 小时在线）
    - 提升用户体验（故障无感知切换）
  - **典型场景**：
    - 主库硬件故障（磁盘损坏、内存故障）
    - 主库软件异常（MySQL Crash、死锁）
    - 计划内维护（版本升级、配置变更）

- **读写分离** ⚖️
  - **架构模式**：主库负责写操作，备库负责读操作
  - **性能提升**：
    - 分散数据库负载（读多写少场景效果显著）
    - 提升系统吞吐量（TPS/QPS 提升 2~10 倍）
    - 降低主库压力（避免读写互相干扰）
  - **实现方式**：
    - 应用层路由（代码判断读写走不同库）
    - 中间件代理（MyCat、ShardingSphere、ProxySQL）
    - DNS 层切换（VIP + Keepalived）

- **数据备份** 💾
  - **实时备份能力**：
    - 备库作为热备节点，数据接近实时同步
    - 避免传统冷备的时间窗口问题
    - 支持快速恢复（直接切换到备库）
  - **备份策略补充**：
    - 全量备份 + 增量 binlog 恢复
    - 备库可承担备份任务（不影响主库性能）
    - 多级备份保障（本地+异地）

- **灾难恢复** 🌍
  - **异地容灾**：
    - 跨机房部署（同城双活/异地多活）
    - 应对自然灾害（地震、火灾、洪水）
    - 满足合规要求（金融等行业监管要求）
  - **RPO/RTO 指标**：
    - RPO（ Recovery Point Objective）：数据丢失量目标（< 1 秒）
    - RTO（Recovery Time Objective）：恢复时间目标（< 30 秒）

### 1.2 主备复制的拓扑结构 🏗️

#### 基础拓扑：一主一从（Master-Slave）

```
                    ┌──────────┐
                    │  应用程序   │
                    └────┬─────┘
                         │
              ┌──────────┴──────────┐
              ▼                     ▼
       ┌──────────────┐     ┌──────────────┐
       │   Master     │     │    Slave     │
       │   （主库）    │     │   （从库）    │
       │   接收写入    │     │   只读       │
       └──────┬───────┘     └──────┬───────┘
              │ binlog            │ relay log
              │                   │
              └──────────┬────────┘
                         │ 复制通道
                         ▼
```

**特点说明**：
- ✅ **最简单的部署方式**：易于理解和维护
- ✅ **适合小规模应用**：单机即可满足需求
- ❌ **单点风险**：从库故障后无备用
- ❌ **资源利用率低**：从库大部分时间空闲

#### 进阶拓扑：一主多从（Master-MultiSlave）

```
                         ┌──────────────┐
                         │   Master     │
                         │   （主库）    │
                         │   接收写入    │
                         └──────┬───────┘
                                │ binlog
            ┌───────────────────┼───────────────────┐
            ▼                   ▼                   ▼
     ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
     │   Slave1     │   │   Slave2     │   │   Slave3     │
     │  （读库1）    │   │  （读库2）    │   │  （备份库）   │
     │  业务查询    │   │  报表统计    │   │  数据备份    │
     └──────────────┘   └──────────────┘   └──────────────┘
```

**适用场景**：
- 📊 **读多写少业务**：电商商品详情页、新闻资讯
- 📈 **报表分析系统**：BI 报表、数据仓库 ETL
- 💾 **备份分离**：专门从库做全量备份不影响业务

#### 高级拓扑：双主架构（Master-Master / 双M结构）

```
         ┌──────────────┐               ┌──────────────┐
         │   Master A   │◄─────────────►│   Master B   │
         │  （活跃主库）  │   双向复制     │  （ standby ）│
         └──────┬───────┘               └──────┬───────┘
                │                              │
                ▼                              ▼
         ┌──────────────┐               ┌──────────────┐
         │  读请求处理   │               │  故障切换目标  │
         └──────────────┘               └──────────────┘
```

**核心优势**：
- ✅ **切换无需改配置**：A、B 互为主备关系已建立
- ✅ **减少切换时间**：省去修改复制关系的步骤
- ✅ **双向容灾**：任一节点故障都可切换

**⚠️ 关键挑战：循环复制问题**

```
正常流程（期望）：
A 执行事务 → binlog → B 执行 → 完成 ✅

循环复制（危险）：
A 执行事务 → binlog → B 执行 → B生成binlog → 传回A → A再执行... 🔁❌
```

**解决方案：server_id 机制**
- 每个 MySQL 实例有唯一的 `server_id`（配置在 my.cnf）
- binlog 中记录执行该事务的原始 `server_id`
- 收到 binlog 时检查：如果 `server_id == 自己的 server_id` → 丢弃（防止循环执行）

```ini
# Master A 配置
[mysqld]
server-id = 1

# Master B 配置
[mysqld]
server-id = 2
```

## 三、📋 binlog 格式对主备一致性的影响

### 3.1 三种格式的核心差异对比

| 对比维度 | 📝 Statement 格式 | 📊 Row 格式（推荐） | 🔄 Mixed 格式 |
|---------|------------------|-------------------|--------------|
| **记录内容** | SQL 语句原文 | 行数据的变更镜像 | 自动选择 |
| **空间占用** | ⭐⭐⭐ 最小（几十字节/条） | ⭐ 最大（可能 MB 级） | ⭐⭐ 中等 |
| **主备一致性** | ❌ 可能不一致 | ✅ **绝对一致** | ⚠️ 基本一致 |
| **不确定性函数** | ❌ NOW()/RAND() 有问题 | ✅ 记录实际值 | ⚠️ 加 SET TIMESTAMP |
| **LIMIT 语句** | ❌ 可能删错行 | ✅ 按主键精确操作 | ⚠️ 自动转 Row |
| **触发器/函数** | ❌ 可能重复执行 | ✅ 只记录结果 | ⚠️ 有风险 |
| **CPU 消耗（备库）** | 高（需重新解析执行） | 低（直接应用变更） | 中等 |
| **数据恢复能力** | ❌ 弱 | ✅✅ **最强** | ⚠️ 中等 |

### 3.2 Statement 格式的致命缺陷案例 ⚠️

#### 案例 1：带 LIMIT 的 DELETE 导致主备不一致

```sql
-- 主库执行的 SQL
DELETE FROM t WHERE a>=4 AND t_modified<='2018-11-10' LIMIT 1;

-- 表结构
CREATE TABLE t (
  id INT PRIMARY KEY,
  a INT,
  t_modified TIMESTAMP,
  INDEX idx_a (a),
  INDEX idx_modified (t_modified)
);
```

**问题场景**：

```
主库（使用索引 idx_a）：
  DELETE FROM t WHERE a>=4 ... LIMIT 1;
  → 扫描索引 idx_a，第一个满足条件的是 id=4 的行
  → 删除了 id=4 这一行 ✅

备库（使用了索引 idx_modified）：
  DELETE FROM t WHERE a>=4 ... LIMIT 1;
  → 优化器选择了不同的索引 idx_modified
  → 第一个满足条件的是 id=5 的行（t_modified='2018-11-09'）
  → 删除了 id=5 这一行 ❌

结果：主库删除了 id=4，备库删除了 id=5 → 数据不一致！
```

**根本原因**：
- Statement 格式只记录 SQL 原文
- 主备的优化器可能选择不同的执行计划
- LIMIT 语句导致只删除一行，但具体是哪一行不确定

#### 案例 2：不确定性函数导致不一致

```sql
-- 使用 NOW() 函数
INSERT INTO logs (action, created_at) VALUES ('login', NOW());

-- 主库执行时间：2024-01-01 10:00:00
-- 备库执行时间：2024-01-01 10:00:05（延迟5秒同步）
-- 结果：created_at 字段值不同！

-- 使用 RAND() 函数
UPDATE users SET token=RAND() WHERE id=1;
-- 主库和备库生成的随机数必然不同
```

### 3.3 Row 格式的优势展示 ✨

#### 数据恢复能力的革命性提升

```
场景：误执行了 DELETE FROM users WHERE id=4;

Statement 格式的恢复困境：
  binlog 中只有：DELETE FROM users WHERE id=4;
  ❌ 无法知道被删除的数据是什么
  ❌ 无法自动生成反向的 INSERT 语句

Row 格式的恢复便利：
  binlog 中记录了被删除行的完整数据：
  ### DELETE FROM `test`.`users`
  ### WHERE
  ###   @1=4 /* INT meta=0 nullable=0 is_null=0 */     -- id
  ###   @2='张三' /* VARSTRING(30) */                   -- name
  ###   @3=25 /* TINYINT */                             -- age
  ###   @4='zhangsan@email.com' /* VARCHAR(100) */      -- email

  ✅ 可以精确转换为：INSERT INTO users VALUES(4, '张三', 25, 'zhangsan@email.com');
  ✅ 一键恢复误删数据！
```

#### 三种误操作的恢复对照表

| 误操作类型 | Statement 格式 | Row 格式 |
|-----------|---------------|---------|
| **DELETE 误删** | ❌ 不知道删了什么 | ✅ 转为 INSERT 恢复 |
| **INSERT 误插** | ❌ 需手动查找 | ✅ 转为 DELETE 删除 |
| **UPDATE 误改** | ❌ 不知道旧值 | ✅ 前后值对调恢复 |

### 3.4 Mixed 格式的智能切换机制 🤖

**MySQL 的自动判断逻辑**：

```sql
-- 场景 1：安全的确定性 SQL → 使用 Statement 格式
INSERT INTO config (key, value) VALUES ('version', '1.0');
-- binlog: INSERT INTO config (key, value) VALUES ('version', '1.0');

-- 场景 2：包含不确定性函数 → 自动切换为 Row 或加 SET TIMESTAMP
INSERT INTO logs (action, timestamp) VALUES ('click', NOW());
-- binlog:
SET TIMESTAMP=1704067200;  -- 固定时间戳
INSERT INTO logs (action, timestamp) VALUES ('click', '2024-01-01 00:00:00');

-- 场景 3：可能导致主从不一致的操作 → 强制使用 Row 格式
DELETE FROM temp_table LIMIT 1;
-- binlog: 记录实际删除的行数据（row 格式）
```

**Mixed 格式的局限性**：
- ⚠️ 判断逻辑可能有 bug（历史上出现过误判导致的主从不一致）
- ⚠️ 行为不可预测，开发者难以掌控
- 📉 生产环境中逐渐被 Row 格式取代

**现代最佳实践**：
> **强烈建议生产环境统一使用 ROW 格式**
> - 硬盘成本已很低，空间不再是瓶颈
> - 数据一致性和恢复能力远比节省空间重要
> - 配合 `binlog_row_image=MINIMAL` 可适当减小体积

## 五、⏳ 主备延迟问题深度剖析

### 5.1 什么是主备延迟？

#### 定义

**主备延迟（Replication Lag）**：同一个事务在备库执行完成的时间（T3）与主库执行完成的时间（T1）之间的差值。

```
时间轴 →

主库（Master）                    备库（Slave）
─────────                       ─────────
                                 
T1: 事务执行完成 ✅               
    写入 binlog                  
                                 
    ──────────── 网络传输 ───────►
                                 
T2: 备库接收到 binlog            
    写入 relay log              
                                 
    ──────────── 执行中 ──────────
    
T3: 备库执行完成 ✅               


主备延迟 = T3 - T1
```

#### 监控指标：Seconds_Behind_Master

```sql
-- 查看主备延迟
SHOW SLAVE STATUS\G

-- 关注字段：
-- Seconds_Behind_Master: 0    ← 理想状态（无延迟）
-- Seconds_Behind_Master: 5    ← 正常范围（秒级）
-- Seconds_Behind_Master: 3600 ← 严重告警（延迟1小时！）
```

**计算原理**：

```
Seconds_Behind_Master 的计算方法：

1. 每个 binlog 事件都有一个时间字段（记录主库执行时间）
2. 备库取出当前正在执行的事件的时间字段值
3. 计算它与当前系统时间的差值
4. 结果就是 Seconds_Behind_Master

注意：
- 时间精度：秒级
- 自动校正：如果主备系统时间不一致，会通过 SELECT UNIX_TIMESTAMP() 获取主库时间并扣除差值
```

### 5.2 主备延迟的四大根源 🔍

#### 根源一：备库机器性能差 💻

**典型场景**：

```
错误部署示例：
┌─────────────────────────────────────────┐
│ 主库：16核 CPU / 64GB 内存 / SSD 磁盘   │  ← 高配
│ 备库：4核 CPU  / 16GB 内存 / 机械硬盘   │  ← 低配 ❌
└─────────────────────────────────────────┘

原因："反正备库没有写请求，用差点的机器就行了"
后果：备库执行 relay log 的速度远跟不上主库生成 binlog 的速度
```

**正确做法**：
- ✅ 主备使用**相同规格**的硬件
- ✅ 对称部署（因为主备随时可能切换角色）
- ✅ 确保 I/O 性能对称（特别是磁盘 IOPS）

#### 根源二：备库压力大 📈

**典型场景**：

```
常见误区：
"主库负责写，备库可以承担一些读请求"

实际情况：
- 主库：大家比较克制，不敢乱跑查询
- 备库：运营后台、统计分析、报表查询...全部扔过来
- 结果：备库 CPU/IO 被大量查询占满，SQL Thread 被挤占资源
- 后果：主备延迟持续增大
```

**解决方案**：

| 方案 | 说明 | 效果 |
|------|------|------|
| **一主多从** | 将读请求分散到多个从库 | ⭐⭐⭐⭐ |
| **读写分离中间件** | 智能路由查询到合适的从库 | ⭐⭐⭐⭐ |
| **外部系统分流** | 将统计查询迁移到 Hadoop/ClickHouse | ⭐⭐⭐⭐⭐ |
| **缓存层卸载** | 热点查询走 Redis/Memcached 缓存 | ⭐⭐⭐⭐⭐ |
| **备库专用于同步** | 备库不做任何读查询 | ⭐⭐⭐⭐⭐ |

#### 根源三：大事务 🐘

**什么是大事务？**

定义：执行时间长、影响行数多的事务，包括但不限于：
- 批量 DELETE（如归档历史数据）
- 批量 UPDATE（如批量修改状态）
- 大表 DDL（ALTER TABLE）
- 长时间运行的存储过程

**典型案例**：

```sql
-- 案例 1：一次性删除百万级历史数据（危险操作！）
DELETE FROM orders WHERE create_time < '2020-01-01';
-- 影响：100万行数据，执行时间 10 分钟
-- 后果：备库需要串行重放这个事务，也卡住 10 分钟！

-- 案例 2：大批量更新
UPDATE users SET status = 'inactive'
WHERE last_login < DATE_SUB(NOW(), INTERVAL 2 YEAR);
-- 影响：50万行数据

-- 案例 3：大表添加索引（DDL 也是事务）
ALTER TABLE orders ADD INDEX idx_user_id (user_id);
-- 影响：表锁或长时间 MDL 锁
```

**大事务对主备的影响对比**：

| 影响维度 | 主库（Master） | 备库（Slave） |
|---------|---------------|---------------|
| **锁持有时间** | 长（阻塞其他事务） | N/A（单线程串行） |
| **Undo Log 膨胀** | 大（占用大量空间） | 较小 |
| **Redo Log 产生量** | 大（快速产生） | 无（只是回放） |
| **执行方式** | 并发执行（可与其他事务并行） | **串行执行**（必须排队！） |
| **总体影响** | 影响并发性能 | **直接导致主备延迟暴增** |

**解决方案**：

```sql
-- ✅ 正确做法：分批删除（每次 1 万行）
DELETE FROM orders WHERE create_time < '2020-01-01' LIMIT 10000;
-- 循环执行直到删完

-- ✅ 使用工具：pt-online-schema-change 做 DDL
pt-online-schema-change --alter "ADD INDEX idx_user_id (user_id)" D=test,t=orders

-- ✅ 使用 gh-ost 做 DDL（GitHub 开源，更安全）
gh-ost --database=test --table=orders --alter="ADD INDEX idx_user_id (user_id)"
```

#### 根源四：备库并行复制能力不足 ⚙️

**核心矛盾**：

```
主库（高并发写入）：
  事务A、B、C、D、E... 同时执行（并行）
  快速生成 binlog
       │
       ▼
备库（单线程回放）：
  SQL Thread 串行执行：A → B → C → D → E...
  来不及处理 → 延迟累积 → 越来越大
```

**这是 MySQL 5.6 及之前版本的最大性能瓶颈！**

## 七、🛡️ 高可用架构方案

### 7.1 主备切换策略对比

#### 可靠性优先策略（Reliability-First）⭐ 推荐

```
切换流程（详细步骤）：

Step 1: 判断备库延迟
┌─────────────────────────────────────────┐
│ 检查备库 B 的 Seconds_Behind_Master      │
│                                         │
│ IF SBM < 5 秒 → 继续下一步 ✅            │
│ IF SBM >= 5 秒 → 等待，持续重试 ⏳       │
│ （确保数据几乎完全同步后再切换）          │
└─────────────────────────────────────────┘
           │
           ▼
Step 2: 主库设为只读
┌─────────────────────────────────────────┐
│ SET GLOBAL read_only = ON;              │
│                                         │
│ 主库 A 停止接受新的写请求                │
│ （正在执行的请求会继续完成）              │
└─────────────────────────────────────────┘
           │
           ▼
Step 3: 等待备库追平
┌─────────────────────────────────────────┐
│ 持续监控备库 B 的 SBM                    │
│                                         │
│ 直到 SBM = 0（完全同步）                 │
│ （此时主备数据完全一致）                  │
│                                         │
│ ⚠️ 这一步可能耗时几秒钟                   │
└─────────────────────────────────────────┘
           │
           ▼
Step 4: 备库设为可读写
┌─────────────────────────────────────────┐
│ SET GLOBAL read_only = OFF;             │
│                                         │
│ 备库 B 成为新的主库，开始接受写请求       │
└─────────────────────────────────────────┘
           │
           ▼
Step 5: 切换客户端连接
┌─────────────────────────────────────────┐
│ 将业务流量切到新主库 B                   │
│ 方式：VIP 切换 / DNS 切换 / 配置中心推送 │
│                                         │
│ 切换完成！✅                             │
└─────────────────────────────────────────┘
```

**不可用时间窗口**：
```
时间轴 →

主库 A（readonly）  备库 B（等待SBM=0）  备库 B（新主库）
    │                    │                   │
    ├────────────────────┤                   │
    │   不可用时间窗口     │                   │
    │   （通常几秒钟）     │                   │
    │                    │                   │
    ▼                    ▼                   ▼
```

**优点**：
- ✅ **数据零丢失**：确保主备完全一致后才切换
- ✅ **数据一致性有保证**：不会出现主备数据分歧
- ✅ **适合核心业务**：金融、支付、订单等

**缺点**：
- ⚠️ **有几秒不可用时间**：Step 3 等待 SBM=0 需要时间
- ⚠️ **如果初始延迟大，不可用时间长**：比如初始延迟 30 分钟，就要等 30 分钟

#### 可用性优先策略（Availability-First）⚠️ 谨慎使用

```
切换流程（简化版）：

Step 1: 直接将备库设为主库
┌─────────────────────────────────────────┐
│ SET GLOBAL read_only = OFF;  （备库 B）  │
│                                         │
│ 不等待主备同步完成！                      │
│ 立即接受写请求                           │
└─────────────────────────────────────────┘
           │
           ▼
Step 2: 立即切换客户端连接
┌─────────────────────────────────────────┐
│ 业务流量切到新主库 B                     │
│                                         │
│ 不可用时间 ≈ 0 ✅                        │
└─────────────────────────────────────────┘
```

**数据不一致的风险演示**：

```
场景：自增主键表插入数据

初始状态（主备一致）：
  表 t: id=1(c=1), id=2(c=2), id=3(c=3)

时间线：
T1: 主库 A 执行 INSERT INTO t(c) VALUES(4);  → 插入 id=4
T2: 主备延迟 5 秒（备库还没收到这个 binlog）
T3: 发起主备切换（可用性优先策略）
T4: 备库 B 成为主库，开始接受写请求
T5: 新主库 B 执行 INSERT INTO t(c) VALUES(5);  → 插入 id=5
T6: 备库 B 终于收到 T1 的 binlog，执行 INSERT → 插入 id=6（因为 id=4,5 已被占用）

最终结果：
  主库 A（旧）: id=1,2,3,4
  主库 B（新）: id=1,2,3,5,4    ← 顺序不一致！
  主库 B（新）: id=1,2,3,5,6    ← id=4 变成了 id=6！

数据不一致！❌❌❌
```

**Row 格式的保护作用**：

```
同样的场景，但如果使用 binlog_format=ROW：

T6: 备库 B 尝试执行 INSERT id=4,c=4
    → 发现主键冲突！（id=4 已存在）
    → 报错：Duplicate key error
    → SQL Thread 停止复制
    → DBA 介入处理

虽然也出问题了，但至少：
✅ 能立刻发现数据不一致
✅ 不会静默地产生错误数据
✅ 给了人工干预的机会
```

**结论**：
> **生产环境强烈建议使用可靠性优先策略**
>
> 只有在特殊场景（如日志库、且业务可短暂不可用时）才考虑可用性优先
>
> 即使要用可用性优先，也务必配合 **ROW 格式的 binlog** 以便及时发现问题

#### 故障切换流程详解

```
正常状态：
  Master (活跃) ←── 复制 ──→ Slave1 (standby)
                        ←── 复制 ──→ Slave2 (只读)


故障发生时刻（T_fail）：
  Master 网络中断 / 进程崩溃 / 服务器宕机


MHA 检测与切换过程：

T_detect: Manager 检测到 Master 不可达（连续 3 次 ping 失败）
    │
    ▼
T_elect: 从存活 Slave 中选举新 Master
    │      选主标准（按优先级）：
    │      1. binlog 位置最新（数据最全）
    │      2. 配置中的优先级（candidate_master=1）
    │      3. 网络延迟最低
    │
    ▼
T_relay: 收集其他 Slave 的 relay log
    │      目的：尝试补齐新 Master 缺失的数据
    │      方法：从各个 Slave 获取差异部分的 binlog
    │
    ▼
T_promote: 将选中的 Slave 提升为新 Master
    │      • 清理复制配置
    │      • 设为可读写（read_only=OFF）
    │
    ▼
T_redirect: 将其他 Slave 指向新 Master
    │      • 执行 CHANGE MASTER TO
    │      • 启动复制
    │
    ▼
T_vip: 切换 VIP（虚拟 IP）到新 Master
    │      • ifconfig/ip 命令绑定 VIP
    │      • 或通知路由器/DNS 更新
    │
    ▼
T_notify: 通知应用层（邮件/短信/Webhook）
    │
    ▼
切换完成！总耗时：通常 10~30 秒
```

### 7.4 GTID（Global Transaction Identifier）🆔

#### 什么是 GTID？

**GTID = 全局事务标识符**，用于在全局范围内唯一标识一个事务。

**格式**：
```
UUID:sequence_number

示例：
3E11FA47-71CA-11E9-9E35-00155D123456:23

分解：
├─ UUID（源服务器标识）
│   3E11FA47-71CA-11E9-9E35-00155D123456
│   表示这个事务最初是在哪台服务器上执行的
│
└─ sequence_number（事务序号）
    23
    表示这台服务器上的第 23 个事务
```

#### GTID vs 传统 binlog 位置复制

| 对比维度 | 传统方式（file + pos） | GTID 方式 |
|---------|---------------------|----------|
| **复制起点标识** | `mysql-bin.000001:5678` | `3E11FA47-...:23` |
| **切换 Master 后** | ❌ 需要手动计算新的 file+pos | ✅ **自动定位**正确的起点 |
| **复制断裂恢复** | ❌ 困难，容易出错 | ✅ 简单，自动跳过已执行的事务 |
| **运维复杂度** | ⭐⭐⭐⭐ 高（容易配错） | ⭐⭐ 低（配置简单） |
| **故障排查** | 需要人工比对 binlog 位置 | 通过 GTID 直接追踪 |
| **多源复制** | ❌ 复杂（需手动管理） | ✅ 天然支持 |

#### GTID 配置方法

```ini
# ========== Master 和 Slave 都需要配置 ==========

[mysqld]

# 1. 开启 GTID 模式
gtid_mode = ON

# 2. 强制 GTID 一致性（禁止某些不兼容的操作）
enforce_gtid_consistency = ON

# 3. （可选）启用 GTID 下的事务压缩（MySQL 8.0+）
# binlog_transaction_compression = ON

# 4. server_id 必须唯一
server-id = 1  # Master 为 1，Slave 为 2、3、4...
```

```sql
-- 验证 GTID 是否生效
SHOW VARIABLES LIKE '%gtid%';

-- 应该看到：
-- gtid_mode = ON
-- enforce_gtid_consistency = ON
-- gtid_executed: 3E11FA47-...:1-100  （已执行的事务范围）
-- gtid_purged: 3E11FA47-...:1-50    （已清理的事务范围）

-- 查看已执行的 GTID 集合
SHOW MASTER STATUS\G
-- Executed_Gtid_Set: 3E11FA47-...:1-100
```

#### GTID 下的复制配置

```sql
-- 传统方式（不推荐）
CHANGE MASTER TO
  MASTER_HOST='192.168.1.100',
  MASTER_USER='repl',
  MASTER_PASSWORD='xxx',
  MASTER_LOG_FILE='mysql-bin.000001',
  MASTER_LOG_POS=5678;

-- GTID 方式（推荐）✅
CHANGE MASTER TO
  MASTER_HOST='192.168.1.100',
  MASTER_USER='repl',
  MASTER_PASSWORD='xxx',
  MASTER_AUTO_POSITION=1;  -- 自动找位置！
```

**MASTER_AUTO_POSITION=1 的魔法**：
- Slave 连接 Master 后，发送自己已执行的 GTID 集合
- Master 自动计算出 Slave 缺哪些事务的 binlog
- 从正确的位置开始发送
- **无需人工干预，永不配错！**

## 九、⏳ 延迟复制备库（Delayed Replica）

### 9.1 什么是延迟复制？

#### 定义

**延迟复制**：让 Slave 故意**延迟一段时间**再应用（执行）relay log 中的事务。

**本质**：Slave 的数据状态 = 主库 **N 秒前**的状态（时间机器！）

#### 配置方法

```sql
-- 设置延迟复制（延迟 1 小时 = 3600 秒）
CHANGE MASTER TO
  MASTER_DELAY = 3600;

-- 查看当前延迟配置
SHOW SLAVE STATUS\G
-- SQL_Relay_Sleep: 3600  （延迟秒数）

-- 取消延迟（改为实时复制）
CHANGE MASTER TO
  MASTER_DELAY = 0;
```

#### 工作原理

```
正常复制：
  Master binlog → Slave IO Thread 接收 → Slave SQL Thread 立即执行
  延迟 ≈ 0 秒

延迟复制：
  Master binlog → Slave IO Thread 接收 → [等待 3600 秒] → Slave SQL Thread 执行
  延迟 = 3600 秒（故意设计）


时间轴示意：

T=0:00   Master 执行 DELETE FROM important_data;  （误操作！）
T=0:00   Slave IO Thread 接收到该 binlog 事件
T=0:00   写入 relay log（但不执行）
         ...
T=0:05   运维人员发现误操作！（5分钟内发现）
T=0:05   立即 STOP SLAVE;  （停止 SQL Thread）
T=0:05   延迟 Slave 还没执行 DELETE！数据还在！✅
T=0:05   从延迟 Slave 导出被删数据
T=0:10   恢复到生产环境
T=0:10   问题解决！🎉
```

### 9.2 延迟复制的三大核心用途

#### 用途一：误操作快速恢复 🚑（最核心价值！）

**痛点背景**：
```
传统恢复方式的痛苦：

方案 A：从全量备份恢复
  1. 取最近的全量备份文件（假设昨天凌晨 3 点做的）
  2. 花费 2 小时恢复基础数据
  3. 应用 24 小时的 binlog（假设 binlog 有 10GB）
  4. 再花费 4 小时重放 binlog
  总耗时：6+ 小时 😱

方案 B：使用 Flashback 工具
  1. 前提：binlog_format=ROW
  2. 解析 binlog 找到误操作
  3. 生成反向 SQL
  4. 执行恢复
  总耗时：30 分钟~2 小时（取决于数据量）

方案 C：延迟复制备库恢复（最快！）
  1. STOP SLAVE（1 秒）
  2. 导出数据（取决于数据量，通常几分钟）
  3. 恢复到生产环境
  总耗时：**5~30 分钟** ✅✅✅
```

**支持的误操作类型**：

| 误操作类型 | 延迟复制能否恢复 | 恢复难度 |
|-----------|---------------|---------|
| **DELETE 误删** | ✅ 完美恢复 | 简单 |
| **DROP TABLE** | ✅ 完美恢复 | 简单 |
| **UPDATE 误改** | ✅ 完美恢复 | 中等（需定位误改数据） |
| **TRUNCATE** | ✅ 完美恢复 | 简单 |
| **ALTER TABLE 失败** | ✅ 可恢复到修改前的表结构 | 中等 |
| **GRANT/REVOKE 错误** | ✅ 可恢复到修改前的权限 | 简单 |

#### 用途二：历史数据快照查询 📸

**业务场景**：

```
场景 1：报表需要"上月末"的数据快照
  - 正常库：数据一直在变化，无法回到过去
  - 延迟备库：保留了过去某一时刻的数据状态
  - 用途：月度报表、审计报告、合规检查

场景 2：数据对比分析
  - 对比"本周 vs 上周"的用户行为
  - 延迟备库可以提供一周前的数据基线
  - 不影响主库性能

场景 3：回滚到某个时间点的测试
  - 测试新功能对历史数据的影响
  - 在延迟备库上模拟"当时的数据状态"
  - 不需要在生产环境冒险
```

#### 用途三：安全测试环境 🧪

**典型用法**：

```
场景 1：压力测试
  - 在延迟备库上执行高并发压测
  - 数据与生产环境几乎一致（只差几小时）
  - 测试结果更有参考价值
  - 不影响生产环境的稳定性

场景 2：新功能验证
  - 部署新版本到延迟备库
  - 验证数据迁移脚本是否正确
  - 观察新功能对现有数据的影响
  - 确认无误后再推广到生产

场景 3：SQL 优化实验
  - 测试某个 SQL 优化方案的实际效果
  - 使用真实的生产数据量级
  - 不会对生产库造成任何影响
```

### 9.3 延迟时间的设置策略

#### 推荐配置

| 业务类型 | 建议延迟时间 | 原因 |
|---------|------------|------|
| **核心业务（支付/订单）** | **1~2 小时** | 误操作通常 1 小时内发现 |
| **重要业务（用户/商品）** | **30~60 分钟** | 平衡恢复窗口和切换延迟 |
| **一般业务（日志/配置）** | **1~6 小时** | 发现时间可能较长 |
| **非核心业务** | **可不设置** | 数据可从其他途径恢复 |

#### 决策因素

```
设置延迟时间需要权衡的因素：

1. 误操作发现时间窗口
   └─ 统计团队平均多久能发现误操作
   └─ 建议延迟时间 = 平均发现时间 × 1.5~2 倍（留余量）

2. 对高可用切换的影响
   └─ 延迟备库不适合作为故障切换的目标
   └─ 因为切换后数据落后 N 秒
   └─ 解决方案：另外部署正常的实时备库

3. 存储成本
   └─ 延迟备库需要额外的磁盘空间
   └─ relay log 会累积更多（但最终会被清理）

4. 恢复速度要求
   └─ 延迟越长，恢复窗口越大
   └─ 但同时也意味着数据越旧（可能与当前状态差异大）
```

### 9.4 延迟复制的副作用与应对

#### 副作用一览

| 副作用 | 影响程度 | 应对方案 |
|-------|---------|---------|
| **高可用切换延迟** | ⚠️⚠️⚠️ 高 | 延迟备库专用于恢复，不参与 HA 切换 |
| **读写分离不一致** | ⚠️⚠️ 中 | 延迟备库不对外提供读服务 |
| **Seconds_Behind_Master 持续高位** | ⚠️ 低 | 监控系统需特殊处理此指标 |
| **额外存储开销** | ⚠️ 低 | relay log 略多，可接受 |
| **维护复杂度增加** | ⚠️ 低 | 需文档化延迟备库的管理规范 |

#### 最佳实践架构

```
推荐的混合部署架构：

                    ┌──────────────┐
                    │   Master     │
                    │   （主库）    │
                    └──────┬───────┘
                           │
          ┌────────────────┼────────────────┐
          ▼                ▼                ▼
   ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
   │  Real-Time   │ │  Real-Time   │ │   Delayed    │
   │   Slave1     │ │   Slave2     │ │    Slave3     │
   │  （实时备库）  │  （实时备库）  │  （延迟备库）   │
   │              │              │              │
   │ 用途：        │ 用途：        │ 用途：        │
   │ • HA 切换目标 │ • 读写分离    │ • 误操作恢复   │
```

## 十一、💡 常见面试题精解

### Q1：请简述 MySQL 主备复制的工作原理？🎯

<details>
<summary><b>点击查看标准答案</b></summary>

**答案要点（按顺序说出以下关键点）**：

1. **核心组件**：
   - Master 端：binlog（二进制日志）+ Binlog Dump Thread
   - Slave 端：IO Thread + SQL Thread + relay log（中继日志）

2. **三个线程的协作**：
   - **Binlog Dump Thread**（Master）：将 binlog 发送给 Slave 的 IO Thread
   - **IO Thread**（Slave）：连接 Master，接收 binlog，写入 relay log
   - **SQL Thread**（Slave）：读取 relay log，解析并执行其中的事件

3. **完整流程**（用一句话概括）：
   > 用户在 Master 执行更新 → 写入 binlog → IO Thread 拉取到 Slave → 写入 relay log → SQL Thread 重放 → Slave 数据同步完成

4. **关键技术点**：
   - binlog 是主备同步的基石（记录所有数据变更）
   - 支持断点续传（记录已同步的位置）
   - 异步执行（不阻塞 Master 的业务）

</details>

### Q2：什么是主备延迟？有哪些原因？如何解决？⏳

<details>
<summary><b>点击查看标准答案</b></summary>

**答案结构**：

**1. 定义**：
> 主备延迟 = 同一事务在备库执行完成时间（T3）- 主库执行完成时间（T1）
> 监控指标：`Seconds_Behind_Master`（通过 `SHOW SLAVE STATUS` 查看）

**2. 四大原因**（必答）：

| 序号 | 原因 | 说明 | 典型场景 |
|------|------|------|---------|
| 1 | **备库机器性能差** | 主备硬件规格不对称 | 为了省钱给备库用低配 |
| 2 | **备库压力大** | 备库承担大量读请求 | 把所有报表查询扔给备库 |
| 3 | **大事务** | 单个事务执行时间长 | 一次性 DELETE 百万行数据 |
| 4 | **单线程回放** | SQL Thread 是单线程（5.6及之前） | 高并发下备库来不及回放 |

**3. 解决方案**（按有效性排序）：

| 方案 | 解决的问题 | 实施难度 | 效果 |
|------|----------|---------|------|
| **拆分大事务** | 原因 3 | ⭐ 低 | ⭐⭐⭐⭐⭐ 最有效 |
| **并行复制** | 原因 4 | ⭐⭐ 中 | ⭐⭐⭐⭐⭐ 根本性解决 |
| **一主多从** | 原因 2 | ⭐⭐ 中 | ⭐⭐⭐⭐ 有效 |
| **硬件对称部署** | 原因 1 | ⭐⭐⭐ 高（花钱） | ⭐⭐⭐⭐ 根治 |

**加分项**：提到并行复制的 WRITESET 策略（MySQL 5.7.22+），以及 `slave_parallel_workers` 参数配置。

</details>

### Q3：异步复制、半同步复制、全同步复制有什么区别？如何选择？🔄

<details>
<summary><b>点击查看标准答案</b></summary>

**答案结构**：

**1. 核心区别（一句话版）**：
- **异步**：Master 不等 Slave，提交即返回（最快但可能丢数据）
- **半同步**：Master 等至少 1 个 Slave ACK（折中方案）
- **全同步**：Master 等所有 Slave 才返回（最安全但最慢）

**2. 详细对比表**（建议背诵）：

| 维度 | 异步 | 半同步 | 全同步 |
|------|------|--------|--------|
| **数据安全性** | 低 | 中高 | 最高 |
| **性能** | 最优 | 略差 | 很差 |
| **主备延迟** | 可能大 | 较小 | 几乎无 |
| **适用场景** | 日志/监控 | **一般业务（推荐）** | 金融核心 |

**3. 选择建议**：
> 生产环境**首选半同步复制**，配合以下配置效果最佳：
> - `rpl_semi_sync_master_timeout = 5000`（5 秒超时）
> - `rpl_semi_sync_master_wait_for_slave_count = 1`（至少 1 个从库确认）
> - 超时后自动降级为异步（保证可用性）

**4. 配置命令**（加分项）：
```sql
-- Master 端
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';
SET GLOBAL rpl_semi_sync_master_enabled = 1;

-- Slave 端
INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';
SET GLOBAL rpl_semi_sync_slave_enabled = 1;
```

</details>

### Q4：binlog 的 statement/row/mixed 三种格式有什么区别？推荐用哪种？📋

<details>
<summary><b>点击查看标准答案</b></summary>

**答案要点**：

**1. 三种格式核心区别**：

| 格式 | 记录内容 | 优点 | 致命缺陷 |
|------|---------|------|---------|
| **Statement** | SQL 语句原文 | 空间小、性能好 | ❌ 主备可能不一致（LIMIT、NOW()、RAND()） |
| **Row** | 行数据变更镜像 | ✅ 绝对一致、数据恢复能力强 | 空间较大 |
| **Mixed** | 自动选择 | 兼顾两者 | ⚠️ 判断逻辑可能有 bug |

**2. Statement 格式的不一致案例**（必答）：
```sql
-- 带 LIMIT 的 DELETE
DELETE FROM t WHERE a>=4 AND t_modified<='2018-11-10' LIMIT 1;
-- 主库用索引 a 删除了 id=4
-- 备库用了索引 t_modified 删除了 id=5
-- 结果：主备数据不一致！
```

**3. 强烈推荐 ROW 格式的理由**：
- ✅ 绝对保证主备一致性
- ✅ 强大的数据恢复能力（delete→insert, insert→delete, update→前后对调）
- ✅ 不依赖执行计划（不会因索引选择不同而出错）
- ✅ 现代硬盘成本低，空间不再是瓶颈

**4. 生产配置**：
```sql
SET GLOBAL binlog_format = 'ROW';  -- 强烈推荐
SET GLOBAL binlog_row_image = 'MINIMAL';  -- 减小体积（可选）
```

</details>

### Q5：什么是 GTID？相比传统的 file+pos 复制有什么优势？🆔

<details>
<summary><b>点击查看标准答案</b></summary>

**答案要点**：

**1. GTID 定义**：
> GTID（Global Transaction Identifier）= 全局事务标识符
> 格式：`UUID:sequence_number`
> 示例：`3E11FA47-71CA-11E9-9E35-00155D123456:23`

**2. 核心优势**（对比传统方式）：

| 传统方式（file+pos） | GTID 方式 |
|--------------------|----------|
| `mysql-bin.000001:5678` | `UUID:23` |
| 切换 Master 后需**手动计算**新位置 | **自动定位**正确的复制起点 |
| 复制断裂后恢复困难 | 自动跳过已执行的事务 |
| 运维复杂度高（容易配错） | 配置简单，不易出错 |

**3. 配置方法**：
```ini
[mysqld]
gtid_mode = ON
enforce_gtid_consistency = ON
server-id = 1  # 每个实例唯一
```

**4. 使用 GTID 复制**：
```sql
-- 传统方式（不推荐）
CHANGE MASTER TO MASTER_LOG_FILE='xxx', MASTER_LOG_POS=123;

-- GTID 方式（推荐）✅
CHANGE MASTER TO MASTER_AUTO_POSITION=1;
-- 自动找位置！永不配错！
```

</details>

### Q6：什么是延迟复制备库？有什么作用？⏳

<details>
<summary><b>点击查看标准答案</b></summary>

**答案要点**：

**1. 定义**：
> 延迟复制是一种特殊的备库配置，故意延迟 N 秒再执行 relay log。
> 配置：`CHANGE MASTER TO MASTER_DELAY = 3600;`（延迟 1 小时）

**2. 三大核心作用**（按重要性排序）：

| 作用 | 说明 | 价值 |
|------|------|------|
| **误操作快速恢复** | 误删数据后，延迟备库还未执行，可直接导出恢复 | 将恢复时间从**天级缩短到分钟级** |
| **历史数据快照** | 保留过去某时刻的数据状态 | 用于报表、审计、合规检查 |
| **安全测试环境** | 使用真实数据进行测试 | 不影响生产环境 |

**3. 典型恢复流程**：
```
1. 发现误操作（如 DELETE 误删）
2. 立即 STOP SLAVE（停止延迟备库的 SQL Thread）
3. 从延迟备库导出被删数据
4. 恢复到生产环境
5. 总耗时：5~30 分钟 ✅
```

**4. 推荐配置**：
- 核心业务：延迟 **1~2 小时**
- 一般业务：延迟 **30~60 分钟**
- 需另外部署**实时备库**用于高可用切换（延迟备库专用于恢复）

</details>

### Q7：MHA 和 Orchestrator 有什么区别？如何选择？🛡️

<details>
<summary><b>点击查看标准答案</b></summary>

**答案要点**：

**1. 核心区别**：

| 对比维度 | MHA | Orchestrator |
|---------|-----|-------------|
| **管理节点高可用** | ❌ 单点（需自行解决） | ✅ Raft 协议，内置集群化 |
| **可视化界面** | ❌ 命令行 | ✅ Web UI |
| **GTID 支持** | ⚠️ 部分 | ✅ 原生支持 |
| **社区状态** | ❌ 已停止维护（2020年） | ✅ 活跃维护中 |
| **适用规模** | 中小集群 | **大规模集群**（百节点级） |
| **学习成本** | 中等 | 较高 |

**2. 选择建议**：
- **中小规模（< 10 节点）**：MHA 足够，社区资料多
- **大规模（≥ 10 节点）或新建项目**：推荐 Orchestrator
- **云原生环境**：考虑云厂商提供的托管方案（如 AWS Aurora、阿里云 POLARDB）

**3. MHA 切换流程**（常考）：
```
检测 Master 不可达 → 选举新 Master（数据最新的 Slave）
→ 收集 binlog 补偿 → 提升新 Master → 其他 Slave 指向新 Master
→ VIP 切换 → 通知应用层
总耗时：10~30 秒
```

</details>
