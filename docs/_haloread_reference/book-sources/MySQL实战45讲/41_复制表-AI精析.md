# 怎么最快地复制一张表？- AI精析

> **思考问题**: 如果你要在凌晨2点执行一次100GB大表的迁移任务,哪种方法能让业务停机时间最短?为什么?



### 🔹 方法二:CSV文件导出(最灵活)

#### 核心流程
```sql
-- 导出
SELECT * FROM db1.t WHERE a>900 
INTO OUTFILE '/server_tmp/t.csv';

-- 导入
LOAD DATA INFILE '/server_tmp/t.csv' 
INTO TABLE db2.t;
```

#### 关键配置:secure_file_priv参数

| 参数值 | 含义 | 安全性 |
|--------|------|--------|
| 空 | 不限制文件位置 | ❌ 不安全 |
| 路径字符串 | 只能在指定目录 | ✅ 推荐 |
| NULL | 禁止导出操作 | 🔒 最安全 |

**主备同步机制(核心原理)**:
```
主库执行 → CSV内容写入binlog → 传到备库 
→ 备库写入临时文件/tmp/SQL_LOAD_MB-1-0 
→ 执行LOAD DATA LOCAL INFILE
```

**为什么需要LOCAL?**
- 不加LOCAL:读取服务端文件,受secure_file_priv限制
- 加LOCAL:读取客户端文件,绕过secure_file_priv限制
- **备库同步线程相当于客户端,必须用LOCAL**

**思考**: 如果备库的secure_file_priv=NULL,不加LOCAL会怎样?



### 🔹 三种方法对比决策表

| 维度 | mysqldump | CSV导出 | 物理拷贝 |
|------|-----------|---------|----------|
| **速度** | 慢 | 中等 | **最快** |
| **灵活性** | 中等(支持where) | **最高**(支持所有SQL) | 低(仅全表) |
| **跨引擎** | ✅ 支持 | ✅ 支持 | ❌ 仅InnoDB |
| **需要服务器权限** | ❌ 客户端执行 | ✅ 需要服务端文件权限 | ✅ 需要OS文件权限 |
| **适用场景** | 部分数据迁移 | 复杂查询导出 | 大表全量恢复 |

**决策树**:
```
需要全表拷贝?
├─ 是 → 能登录服务器?
│        ├─ 是 → 物理拷贝(最快)
│        └─ 否 → mysqldump
└─ 否 → 需要复杂SQL?
         ├─ 是 → CSV导出
         └─ 否 → mysqldump --where
```

## 关键记忆点

1. **mysqldump三剑客**: `--single-transaction`(不加锁) + `--no-create-info`(不导结构) + `--where`(条件过滤)

2. **CSV导出两注意**: `secure_file_priv`(路径限制) + `LOCAL`(主备同步必需)

3. **物理拷贝六步走**: 建表 → 删表空间 → 导出 → 拷贝 → 解锁 → 导入

4. **方法选择一句话**: 大表全量用物理,部分迁移用dump,复杂查询用CSV
