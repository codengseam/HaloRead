# 核心知识清单(20%)

## 1. MySQL边读边发机制

**定义**: MySQL查询结果采用"边读边发"策略,服务端不保存完整结果集,而是分段发送给客户端。

**核心流程**:
```
获取一行 → 写入net_buffer(16KB) → 写满后发送 → 清空继续
```

**关键机制**:
- **net_buffer_length**: 默认16KB,控制单次发送缓冲区大小
- **socket send buffer**: 网络栈缓冲区,写满时暂停读取
- **内存占用上限**: 单查询最大内存 = net_buffer_length,不会随数据量增长

**应用场景**: 
全表扫描200G表时,服务端内存占用仅16KB,不会打爆内存。

**常见错误**: 
误以为MySQL会缓存所有查询结果 → 实际是边读边发,客户端接收慢会导致服务端阻塞(状态显示"Sending to client")

## 3. InnoDB改进的LRU算法(5:3分区)

**核心设计**: 将LRU链表按5:3比例分为young区域和old区域,防止全表扫描污染Buffer Pool。

**算法流程**:
```
访问young区域数据页 → 移到链表头部
访问新数据页 → 插入old区域头部(LRU_old位置)
old区域数据页被访问 → 存在时间>1秒才移到链表头部
```

**关键参数**:
- **innodb_old_blocks_time**: 默认1000ms,控制old区域数据页晋升时间阈值
- **young区域**: 链表头部5/8,存放热点数据
- **old区域**: 链表尾部3/8,存放冷数据

**应用场景**:
扫描200G历史数据表时,数据页仅停留在old区域,扫描结束后快速淘汰,不影响young区域热点数据。

**核心收益**:
- 全表扫描不会污染Buffer Pool的young区域
- 保证正常业务查询的内存命中率(应>99%)

**思考**: 为什么设置1秒的晋升时间阈值?这个值是否需要调整?



## 5. 客户端性能对服务端的影响

**核心问题**: 客户端接收慢会导致服务端阻塞,但不会打爆内存。

**影响链条**:
```
客户端处理慢 → socket receive buffer满 
→ socket send buffer满 → 服务端暂停读取 → 事务执行时间变长
```

**更严重的场景**:
- 客户端磁盘满 → 无法写入结果 → 服务端长时间阻塞
- 频繁更新的事务 → undo表空间膨胀 → 服务端磁盘空间不足

**优化方案**:
- 小结果集 → 使用mysql_store_result(一次性缓存到客户端)
- 大结果集 → 使用mysql_use_result(逐行处理)
- 调整net_buffer_length → 减少服务端阻塞时间

# 思考问答

## 1. 核心机制理解

**问题**: 为什么MySQL采用"边读边发"机制,而不是一次性缓存所有结果再发送?

**解析**: 
- **内存效率**: 边读边发仅需16KB内存,一次性缓存需要存储完整结果集(可能GB级别)
- **并发能力**: 多个查询同时执行时,内存占用可控,不会因大查询打爆内存
- **实时性**: 客户端可以边接收边处理,减少等待时间
- **容错性**: 客户端中断连接时,服务端可立即停止查询,避免资源浪费

**代价**: 客户端接收慢会阻塞服务端,需要合理设计客户端处理逻辑。



## 3. 实践场景分析

**问题**: 如果在生产环境中发现大量线程处于"Sending to client"状态,应该如何处理?

**处理步骤**:
1. **紧急处理**: 调大net_buffer_length参数,快速减少阻塞线程
2. **根因分析**: 
   - 检查客户端处理逻辑是否过慢
   - 检查网络带宽是否充足
   - 检查查询结果集是否过大
3. **长期优化**:
   - 优化查询,减少返回数据量
   - 客户端使用mysql_store_result处理小结果集
   - 增加网络带宽或优化网络配置
   - 考虑分批查询,避免一次性返回大量数据

**预防措施**: 监控"Sending to client"状态线程数量,设置告警阈值。
