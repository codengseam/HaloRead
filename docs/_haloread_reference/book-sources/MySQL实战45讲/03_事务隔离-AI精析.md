# 核心知识清单(20%重点)

## 🔹 事务隔离级别(核心中的核心)

### 四种隔离级别定义

**读未提交(Read Uncommitted)**
- **定义**:事务未提交时,变更就能被其他事务看到
- **应用场景**:几乎不用,性能虽高但数据不一致风险极大
- **常见错误**:误以为能提升性能而使用→实际会导致脏读问题
- **实操建议**:生产环境禁用此级别

**读提交(Read Committed)**
- **定义**:事务提交后,变更才被其他事务看到
- **应用场景**:Oracle默认级别,适合大多数OLTP业务
- **视图机制**:每个SQL语句执行时创建新视图
- **实操建议**:Oracle迁移到MySQL时需显式配置`transaction-isolation=READ-COMMITTED`

**可重复读(Repeatable Read)**
- **定义**:事务执行期间看到的数据始终与启动时一致
- **应用场景**:数据校对、对账场景(需静态数据视图)
- **视图机制**:事务启动时创建视图,整个事务期间复用
- **实操要点**:MySQL InnoDB默认级别,适合需要数据一致性的业务

**串行化(Serializable)**
- **定义**:读写加锁,后访问事务必须等待前事务完成
- **应用场景**:对数据一致性要求极高的金融核心系统
- **性能代价**:并发性能最低,谨慎使用

### 隔离级别对比速查表

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 性能 |
|---------|------|-----------|------|------|
| 读未提交 | ✓ | ✓ | ✓ | 最高 |
| 读提交 | ✗ | ✓ | ✓ | 高 |
| 可重复读 | ✗ | ✗ | ✓ | 中 |
| 串行化 | ✗ | ✗ | ✗ | 最低 |



## 🔹 长事务的危害(实践要点)

### 三大核心风险

**1. 回滚日志膨胀**
- **原理**:长事务持有老的事务视图,所有可能用到的回滚记录必须保留
- **后果**:大量占用存储空间(MySQL 5.5及以前版本,回滚日志与数据字典共用ibdata文件,即使长事务提交也不会释放空间)
- **极端案例**:数据20GB,回滚段200GB,最终需重建整个库

**2. 锁资源占用**
- **原理**:长事务持有锁的时间过长
- **后果**:阻塞其他事务,可能拖垮整个数据库
- **实操建议**:监控锁等待时间,设置告警阈值

**3. 连接池污染**
- **场景**:客户端框架默认`set autocommit=0`,长连接导致意外长事务
- **后果**:连接池中连接被长事务占用,影响并发性能



# 次要信息速览(80%快速带过)

## 【ACID特性】
- 原子性(Atomicity):事务要么全部成功,要么全部失败
- 一致性(Consistency):事务前后数据完整性约束不变
- 持久性(Durability):事务提交后永久保存
- 【隔离性是本文重点,其他三个特性了解即可】

## 【事务应用场景】
- 经典案例:转账操作(查询余额→加减法→更新余额)
- 【MyISAM引擎不支持事务,已被InnoDB取代】

## 【Oracle迁移配置】
- Oracle默认隔离级别=读提交
- MySQL需显式配置:`transaction-isolation=READ-COMMITTED`
- 查看配置:`show variables like 'transaction_isolation';`

## 【备份策略补充】
- 一天一备vs一周一备:权衡RTO(恢复目标时间)与存储成本
- 【备份策略需根据业务重要性评估,非本文核心】

## 【常见问题】
- 脏读:读到未提交数据
- 不可重复读:同一事务内两次读取结果不同
- 幻读:同一事务内两次查询记录数不同
- 【串行化可解决所有问题,但性能代价最高】
