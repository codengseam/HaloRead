# 备库为什么会延迟好几个小时?

## 核心问题诊断

**备库延迟的根本原因**:备库执行日志的速度持续低于主库生成日志的速度,导致延迟累积到小时级别。

**关键现象**:
- 主库并发写入能力强(支持行锁,并发友好)
- 备库单线程执行relay log成为瓶颈
- `seconds_behind_master`值持续增大

## 核心知识清单(20%重点)

### 🔹 并行复制的基本模型

**架构演进**:将单线程`sql_thread`拆分为协调器+多个worker线程

```
coordinator(原sql_thread)
├─ 职责:读取relay log + 分发事务
├─ 不再直接更新数据
└─ worker线程数:slave_parallel_workers(建议8-16)
```

**分发两大铁律**:
1. **更新同一行的事务必须分发到同一个worker**(避免主备不一致)
2. **同一事务不能拆分到不同worker**(保证事务原子性)

*思考:为什么轮询分发会导致主备不一致?*

### 🔹 MySQL并行复制策略演进

#### 1. 按表分发策略(MySQL 5.5非官方)

**核心逻辑**:不同表的事务可并行执行

**分发规则**:
- 与所有worker都不冲突 → 分配给最空闲worker
- 与多个worker冲突 → coordinator等待,直到只与1个worker冲突
- 与1个worker冲突 → 直接分配给该worker

**缺陷**:热点表导致退化为单线程【次要:跨表事务需特殊处理】

#### 2. 按行分发策略(MySQL 5.5非官方)

**核心逻辑**:更新不同行的事务可并行

**关键技术**:
- Hash键:`库名+表名+唯一键名+值`(需考虑主键和唯一索引)
- 约束条件:binlog格式必须为row、表必须有主键、不能有外键

**大事务处理**:超过阈值(如10万行)退化为单线程模式

**缺陷**:大事务耗费内存和CPU【次要:删除100万行需记录100万个hash项】

#### 3. 按库并行(MySQL 5.6官方)

**优势**:
- Hash计算快(仅需库名)
- 不依赖binlog格式

**局限**:单库场景无效,多库压力不均衡效果差【次要:理论上可人为分库,但实际应用少】

#### 4. MariaDB组提交策略

**核心思想**:模拟主库并行模式

**实现机制**:
- 同一组提交的事务具有相同`commit_id`
- 相同`commit_id`的事务在备库并行执行
- 一组全部完成后才执行下一组

**缺陷**:
- 未真正模拟主库并发度(主库下一组在上一组commit时已在执行)
- 大事务拖后腿(整组等待最慢事务)

#### 5. MySQL 5.7 LOGICAL_CLOCK策略

**优化思路**:同时处于prepare状态的事务可并行

**关键洞察**:
- 进入prepare状态 = 已通过锁冲突检验
- prepare状态与commit状态的事务也可并行

**参数优化**:
```sql
binlog_group_commit_sync_delay      # 延迟多少微秒后fsync
binlog_group_commit_sync_no_delay_count  # 累积多少次后fsync
```

**作用**:制造更多"同时处于prepare阶段的事务",提升备库并行度

#### 6. MySQL 5.7.22 WRITESET策略

**三种模式**:
- `COMMIT_ORDER`:基于prepare/commit状态判断
- `WRITESET`:基于行冲突判断(无交集即可并行)
- `WRITESET_SESSION`:WRITESET + 同线程事务保持顺序

**核心优势**:
- Writeset在主库生成并写入binlog,备库无需解析
- 不依赖binlog格式(statement格式也支持)
- 节省内存和CPU

**约束**:无主键或有外键时退化为单线程

## 次要信息速览(80%快速带过)

### 【主备流程图解析】
- 主库箭头粗(并发写入强) vs 备库箭头细(单线程瓶颈)
- InnoDB行锁支持高并发(除热点行场景)

### 【按表分发详细流程】
- 每个worker维护hash表记录涉及表
- Key:`库名.表名`,Value:队列中修改该表的事务数
- 示例:hash_table_1显示worker_1有4个事务修改db1.t1

### 【按行分发唯一键处理】
- 示例:表t1有主键id和唯一索引a
- 更新语句`update t1 set a=1 where id=2`生成3个hash项:
  - 主键id=2
  - 唯一索引a=2(修改前)
  - 唯一索引a=1(修改后)

### 【MariaDB策略缺陷图解】
- 主库:trx1-3提交时,trx4-6已在执行
- 备库:必须等trx1-3全部完成,才开始trx4-6

### 【大事务影响】
- 主库影响:锁持有时间长、undo log大
- 备库影响:并行复制退化为单线程、整组等待
- 建议:拆分大事务(如删除100万行 → 100个事务各删1万行)

## 实践要点

### 1. 参数配置建议

**worker线程数**:
```sql
slave_parallel_workers = 8~16  # 32核物理机
```

**提升并行度**(MySQL 5.7):
```sql
binlog_group_commit_sync_delay = 1000      # 1毫秒
binlog_group_commit_sync_no_delay_count = 10
```

### 2. 策略选择决策树

```
是否单库场景?
├─ 是 → 使用WRITESET策略
│        └─ binlog-transaction-dependency-tracking = WRITESET
│
└─ 否 → 是否多库压力均衡?
         ├─ 是 → 可用DATABASE策略(MySQL 5.6)
         └─ 否 → 使用LOGICAL_CLOCK或WRITESET(MySQL 5.7+)
```

### 3. 大事务规避方案

**识别大事务**:
```sql
-- 查看长事务
SELECT * FROM information_schema.innodb_trx 
WHERE TIME_TO_SEC(TIMEDIFF(NOW(), trx_started)) > 60;
```

**拆分策略**:
- 批量删除:分批执行,每批1万行
- 批量更新:使用limit分页更新
- DDL操作:使用pt-online-schema-change工具

### 4. 监控指标

**延迟监控**:
```sql
SHOW SLAVE STATUS\G
-- 关注:Seconds_Behind_Master
```

**并行度监控**:
```sql
-- 查看worker线程状态

-- 关注:system user线程数量
```

## 思考问答

1. **场景题**:MySQL 5.7.22版本主库单线程插入大量数据,3小时后搭建备库并开启并行复制,应选择哪种策略?

   **答案**:WRITESET策略
   
   **原因**:
   - COMMIT_ORDER:主库单线程,无同时prepare的事务,退化为单线程
   - WRITESET:通过行冲突判断,可并行执行
   - WRITESET_SESSION:主库单线程=同一线程执行,退化为单线程

2. **架构题**:为什么coordinator不能将同一事务的不同语句分发给不同worker?

   **核心原因**:破坏事务原子性
   
   **场景**:
   - 事务T更新表t1和t2
   - 若分发给worker_1和worker_2
   - worker_1先提交,worker_2后提交
   - 提交间隙的查询会看到"事务执行一半"的状态

3. **优化题**:如何利用binlog_group_commit参数提升备库并行度?

   **机制**:故意延迟fsync,积累更多同时prepare的事务
   
   **效果**:
   - 主库:提交变慢(组提交优化)
   - 备库:并行度提升(更多可并行事务)
   
   **权衡**:需平衡主库性能与备库同步速度

## 最佳实践总结

1. **优先使用MySQL 5.7.22+版本**,启用WRITESET策略
2. **合理设置worker线程数**(8-16),避免CPU资源耗尽
3. **规避大事务**,拆分为小事务执行
4. **监控Seconds_Behind_Master**,及时发现延迟问题
5. **主库设计时考虑备库同步**,避免热点表和单库压力集中
