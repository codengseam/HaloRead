# 核心知识清单(20%重点)

## 一、覆盖索引 - 减少回表的性能优化利器

### 核心定义
**回表**:在二级索引树查到主键后,再到主键索引树查找完整数据行的过程

**覆盖索引**:查询所需字段已包含在索引中,无需回表即可直接返回结果

### 关键应用场景
**场景示例**:
```sql
-- 需要回表(查询所有字段)
select * from T where k between 3 and 5;

-- 覆盖索引(仅查询ID)
select ID from T where k between 3 and 5;
```

**实践要点**:
- 高频查询"根据身份证号查姓名"时,建立(身份证号,姓名)联合索引可避免回表
- 索引字段维护有代价,需权衡冗余索引的空间成本与查询性能收益

### 常见错误
**错误**:为每个查询都创建独立索引,导致索引过多
**规避**:评估高频查询场景,通过联合索引支持多种查询模式

## 三、索引下推(ICP) - MySQL 5.6的查询优化

### 核心定义
**索引下推**(Index Condition Pushdown):在索引遍历过程中,对索引包含字段先做判断,过滤不满足条件的记录,减少回表次数

### 关键应用场景
**示例**:(name,age)联合索引
```sql
select * from tuser
where name like '张%' and age=10 and ismale=1;
```

**MySQL 5.6之前**:
- 用"张"定位到ID3
- 逐条回表判断age和ismale
- 回表4次

**MySQL 5.6之后(索引下推)**:
- 在索引内部判断age=10
- 仅对满足条件的记录回表
- 回表2次

### 性能提升原理
- 减少回表次数(从4次降至2次)
- 减少主键索引树的访问

### 常见误区
**误区**:认为索引下推对所有条件都生效
**纠正**:仅对索引中包含的字段生效(如age在索引中,ismale不在)

## 问题2:联合索引的实战应用
表结构:
```sql
CREATE TABLE geek (
  a int, b int, c int, d int,
  PRIMARY KEY(a,b),
  KEY c(c),
  KEY ca(c,a),
  KEY cb(c,b)
)
```

查询语句:
```sql
select * from geek where c=N order by a limit 1;
select * from geek where c=N order by b limit 1;
```

问题:ca和cb两个索引是否都是必须的?为什么?

**分析思路**:
1. 主键(a,b)已按a,b排序
2. ca索引中c固定时a有序
3. cb索引中c固定时b有序
4. 考虑order by和limit的优化需求

# 核心要点总结

1. **覆盖索引**:查询字段在索引中,避免回表,显著提升性能
2. **最左前缀**:联合索引支持最左N字段/最左M字符,优先考虑索引复用
3. **索引下推**:MySQL 5.6+在索引内部过滤条件,减少回表次数
4. **设计原则**:减少资源消耗,平衡查询性能与空间成本

**实践建议**:
- 高频查询优先使用覆盖索引
- 联合索引字段顺序考虑复用能力和空间占用
- 利用索引下推减少回表开销
- 定期评估索引使用情况,删除冗余索引
