# 核心知识清单(20%重点)

## 1. 临时表的五大核心特性

### 1.1 会话隔离性
- **定义**:临时表只能被创建它的session访问,对其他线程不可见
- **应用场景**:多线程并发执行join优化时,无需担心表名冲突
- **常见错误**:连接池复用时未主动删除临时表,导致后续连接看到前一个连接的临时表
- **思考问题**:为什么连接池场景下需要主动删除临时表,而不是依赖自动回收?

### 1.2 自动回收机制
- **定义**:session结束时自动删除临时表,无需手动清理
- **应用场景**:异常断开或数据库重启后,不需要额外清理中间数据表
- **实操要点**:正常使用时仍应养成`DROP TEMPORARY TABLE`的好习惯

### 1.3 同名优先原则
- **规则**:session内有同名临时表和普通表时,所有操作优先访问临时表
- **验证方法**:`SHOW TABLES`不显示临时表,但`SHOW CREATE TABLE`会显示临时表结构

### 1.4 引擎灵活性
- **支持引擎**:InnoDB、MyISAM、Memory等
- **存储位置**:InnoDB临时表数据在5.7+版本存放在临时文件表空间

### 1.5 建表语法
```sql
CREATE TEMPORARY TABLE temp_t(id INT PRIMARY KEY)ENGINE=INNODB;
```

## 2. 临时表重名的底层实现机制

### 2.1 文件命名规则
- **frm文件**:`#sql{进程id}_{线程id}_序列号.frm`
- **ibd文件**(5.6及之前):`#sql{进程id}_{线程id}_序列号.ibd`
- **临时表空间**(5.7+):统一存放在临时文件表空间,无需单独ibd文件

### 2.2 内存标识机制
- **普通表table_def_key**:`库名+表名`
- **临时表table_def_key**:`库名+表名+server_id+thread_id`
- **关键点**:通过thread_id区分不同session的同名临时表

### 2.3 线程级管理
- 每个线程维护自己的临时表链表
- 操作表时先遍历链表查找临时表
- session结束时遍历链表执行`DROP TEMPORARY TABLE`

## 3. 分库分表跨库查询的典型应用

### 3.1 应用场景
- **问题**:查询条件未使用分区键,需要跨所有分库查询并排序
- **示例**:
```sql
SELECT v FROM ht WHERE k >= M ORDER BY t_modified DESC LIMIT 100;
```

### 3.2 实现流程
1. 在汇总库创建临时表`temp_ht`,包含字段v、k、t_modified
2. 在各分库执行查询:
```sql
SELECT v,k,t_modified FROM ht_x WHERE k >= M ORDER BY t_modified DESC LIMIT 100;
```
3. 将分库结果插入临时表
4. 在临时表上执行最终查询:
```sql
SELECT v FROM temp_ht ORDER BY t_modified DESC LIMIT 100;
```

### 3.3 优势分析
- **避免表名冲突**:多个session并发执行跨库查询时,临时表可重名
- **自动清理**:无需担心异常断开后的数据清理问题
- **性能优化**:可将临时表放在某个分库上,减少网络传输

## 4. 临时表与主备复制

### 4.1 binlog记录规则
- **statement/mixed格式**:记录临时表操作到binlog
- **row格式**:不记录临时表操作

### 4.2 备库同步机制
- 主库binlog中包含线程id
- 备库根据主库线程id构造不同的table_def_key
- 示例:
  - session A临时表:`库名+t1+M的serverid+session A的thread_id`
  - session B临时表:`库名+t1+M的serverid+session B的thread_id`

### 4.3 DROP TABLE语句改写
- **原因**:备库可能不存在临时表
- **改写示例**:
```sql
DROP TABLE `t_normal` /* generated by server */
```
- **标识**:`/* generated by server */`表示服务端改写过的命令

# 次要信息速览(80%)

## 临时表与内存表的区别
- 【内存表】:使用Memory引擎,数据在内存中,重启后数据丢失但表结构保留
- 【临时表】:可使用多种引擎,InnoDB/MyISAM临时表数据写入磁盘
- 【关键差异】:内存表是引擎类型,临时表是表属性

## 实践注意事项
- 【连接池使用】:连接池复用连接时,需主动删除临时表,避免污染后续连接
- 【异常恢复】:数据库重启后会扫描临时目录清理临时表
- 【binlog选择】:row格式可避免临时表同步问题,可作为binlog_format选择因素

## 思考题解析
- 【临时表改名】:只能用`ALTER TABLE`语法,不能用`RENAME TABLE`语法
- 【原因】:RENAME TABLE不支持临时表,官方文档明确说明

## 其他细节
- 【临时文件目录】:通过`SELECT @@tmpdir`查看
- 【内部临时表】:MySQL内部创建的临时表,与用户临时表不同
- 【BKA算法优化】:临时表常用于join优化场景
