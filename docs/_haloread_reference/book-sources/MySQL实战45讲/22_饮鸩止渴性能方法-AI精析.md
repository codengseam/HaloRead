# MySQL应急性能优化方法精析

## 核心问题场景

业务高峰期，MySQL压力过大无法正常响应，需要短期内临时性提升性能。这些方案多为"饮鸩止渴"，风险较高，仅作应急使用。



## 二、慢查询性能问题

### 三种慢查询原因及解决方案

#### 1. 索引没有设计好

**应急方案**：紧急创建索引
- MySQL 5.6+ 支持Online DDL
- 高效流程（一主一备架构）：
  1. 备库B执行：`set sql_log_bin=off; alter table ... add index ...`
  2. 主备切换（B变为主库）
  3. 新备库A执行：`set sql_log_bin=off; alter table ... add index ...`

**注意**：
- 平时变更建议使用 `gh-ost` 等工具
- 紧急情况上述方案效率最高

#### 2. SQL语句没写好

**应急方案**：使用查询重写功能（MySQL 5.7+）

**示例**：
```sql
-- 原语句（未使用索引）
select * from t where id + 1 = 10000

-- 添加重写规则
insert into query_rewrite.rewrite_rules
(pattern, replacement, pattern_database)
values
("select * from t where id + 1 = ?",
 "select * from t where id = ? - 1",
 "db1");

call query_rewrite.flush_rewrite_rules();
```

**效果**：自动将低效SQL改写为高效形式

#### 3. MySQL选错了索引

**应急方案**：通过查询重写强制使用正确索引

**预防措施**：
- 上线前SQL审计
- 使用 `pt-query-digest` 等工具检查SQL性能
- 新项目或表结构变更需全量回归测试



## 四、核心要点总结

### 应急方案风险等级

| 方案 | 风险等级 | 适用场景 | 备注 |
|------|---------|---------|------|
| kill空闲连接 | 中 | 短连接风暴 | 需判断事务状态 |
| skip-grant-tables | 极高 | 短连接风暴 | 强烈不建议 |
| 紧急创建索引 | 低 | 索引缺失 | Online DDL支持 |
| 查询重写 | 中 | SQL性能问题 | MySQL 5.7+ |
| 删除用户 | 低 | QPS突增 | 需独立用户 |
| 重写为select 1 | 高 | QPS突增 | 最后手段 |

### 预防优于应急

1. **架构层面**：避免大量短连接，使用连接池
2. **开发层面**：SQL审计，性能测试，正确的事务设计
3. **运维层面**：白名单机制，业务账号分离，监控预警
4. **测试层面**：上线前全量回归测试，使用性能分析工具
