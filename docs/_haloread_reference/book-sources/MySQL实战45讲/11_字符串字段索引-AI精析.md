# 怎么给字符串字段加索引？

## 核心知识清单（20%关键项）

### 🔹 前缀索引的本质权衡

**定义**：只取字符串前N个字符构建索引，节省空间但可能增加扫描行数。

**关键权衡点**：
- **空间节省**：前缀越短，索引占用空间越小
- **性能代价**：前缀越短，区分度越低，回表次数越多
- **致命缺陷**：无法使用覆盖索引优化

**实践示例**：
```sql
-- 完整索引：占用大，但扫描行数少
alter table SUser add index index1(email);

-- 前缀索引：占用小，但可能增加扫描行数
alter table SUser add index index2(email(6));
```

*思考：为什么email(6)会导致扫描4行，而email(7)只扫描1行？*



### 🔹 前缀索引的致命缺陷：无法使用覆盖索引

**场景对比**：
```sql
-- 查询只需要id和email字段
select id, email from SUser where email='zhangssxyz@xxx.com';
```

| 索引类型 | 是否使用覆盖索引 | 性能影响 |
|---------|----------------|---------|
| 完整索引email | ✅ 可以 | 无需回表，直接返回 |
| 前缀索引email(6) | ❌ 不能 | 必须回表判断完整值 |
| 前缀索引email(18) | ❌ 不能 | 即使包含完整信息也要回表 |

**根本原因**：MySQL无法确定前缀索引是否截断了完整信息，必须回表验证。

*思考：如果你的查询只需要返回索引字段，前缀索引会带来多大的性能损失？*



#### 方案二：Hash字段索引

**实现方式**：
```sql
-- 添加hash字段
alter table t add id_card_crc int unsigned, add index(id_card_crc);

-- 插入时计算hash
insert into t values (..., crc32(id_card), ...);

-- 查询时同时匹配hash和原值
select field_list from t 
where id_card_crc = crc32('input_id_card_string') 
  and id_card = 'input_id_card_string';
```

**优势**：
- 查询性能稳定（平均扫描行数≈1）
- 索引仅占4字节

**劣势**：
- 需要额外字段空间
- 不支持范围查询
- 需要额外计算crc32()



## 次要信息速览（80%快速带过）

- 【索引创建语法】：`alter table 表名 add index 索引名(字段名(前缀长度))`
- 【区分度计算】：使用`count(distinct left(字段, n))`统计不同前缀的区分度
- 【回表机制】：前缀索引必须回表验证完整值，即使前缀已包含所有信息
- 【覆盖索引限制】：前缀索引无法利用覆盖索引优化，必须回主键索引查询
- 【Hash冲突】：crc32()可能产生冲突，查询时必须同时验证原字段值
- 【函数消耗】：reverse() < crc32()，但差异不大
- 【身份证号特点】：前6位地址码区分度低，后6位无规律区分度高



## 实践要点总结

1. **前缀索引优先考虑**：字符串字段优先考虑前缀索引，通过区分度测试确定最优长度
2. **覆盖索引陷阱**：如果查询需要利用覆盖索引，避免使用前缀索引
3. **区分度不足时**：倒序存储适合后缀区分度高的场景，Hash字段适合仅等值查询场景
4. **范围查询限制**：倒序存储和Hash字段都不支持范围查询，需根据业务需求选择
5. **空间与性能平衡**：在满足区分度要求的前提下，选择最短的前缀长度
