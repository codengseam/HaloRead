## 一、两阶段提交崩溃恢复机制

### 1. 崩溃恢复判断规则

**定义**:当MySQL在两阶段提交过程中异常重启时,通过redo log和binlog的状态组合判断事务是否提交。

**判断逻辑**:
```
如果redo log有commit标识 → 直接提交
如果redo log只有prepare → 检查binlog:
  ├─ binlog存在且完整 → 提交事务
  └─ binlog不存在/不完整 → 回滚事务
```

**应用场景**:时刻B(binlog写完但redo log未commit)发生崩溃时,事务会被提交,保证主备数据一致性。

**常见错误**:误认为"commit步骤"是SQL语句中的`COMMIT`命令,实际是事务提交流程的最后一步。

**思考**:为什么时刻B崩溃后要提交事务,而不是回滚?【提示:binlog已被从库使用】

### 2. binlog完整性验证机制

**定义**:MySQL通过格式标记和校验和验证binlog事务完整性。

**验证方式**:
- **格式标记**:statement格式末尾有`COMMIT`,row格式末尾有`XID event`
- **校验和**:MySQL 5.6.2+引入`binlog-checksum`参数,防止日志中间出错

**实操**:查看binlog格式
```sql
SHOW VARIABLES LIKE 'binlog_format';
```

### 3. redo log与binlog的关联机制

**定义**:通过XID字段关联两个日志系统。

**关联流程**:
```
崩溃恢复扫描redo log:
  ├─ 有prepare+commit → 直接提交
  └─ 只有prepare → 用XID去binlog找对应事务
```

**关键点**:XID是两阶段提交的纽带,确保崩溃恢复时能正确匹配事务状态。



## 三、并发场景业务设计案例

### 问题场景:互相关注的并发问题

**业务需求**:A、B互相关注时自动成为好友。

**原始方案缺陷**:
```sql
-- 事务1(A关注B)
SELECT * FROM like WHERE user_id=B AND liker_id=A; -- 空
INSERT INTO like VALUES(A, B); -- 只插入单向关注

-- 事务2(B关注A)同时执行
SELECT * FROM like WHERE user_id=A AND liker_id=B; -- 空
INSERT INTO like VALUES(B, A); -- 只插入单向关注

-- 结果:双方都没成为好友(业务BUG)
```

**根本原因**:记录不存在时行锁无法生效,两个事务互不感知。

### 解决方案:唯一索引+按位或操作

**表结构优化**:
```sql
CREATE TABLE `like` (
  `id` int(11) NOT NULL AUTO_INCREMENT,
  `user_id` int(11) NOT NULL,
  `liker_id` int(11) NOT NULL,
  `relation_ship` int(11) NOT NULL, -- 1:A关注B, 2:B关注A, 3:互相关注
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_user_id_liker_id` (`user_id`,`liker_id`)
) ENGINE=InnoDB;
```

**核心逻辑**(A<B时):
```sql
BEGIN;
-- 强制保证user_id < liker_id,确保行锁冲突
INSERT INTO `like`(user_id, liker_id, relation_ship) 
VALUES(A, B, 1) 
ON DUPLICATE KEY UPDATE relation_ship=relation_ship | 1;

SELECT relation_ship FROM `like` WHERE user_id=A AND liker_id=B;
-- 如果返回1:单向关注,提交事务
-- 如果返回3:互相关注,插入friend表

INSERT IGNORE INTO friend(friend_1_id, friend_2_id) VALUES(A,B);
COMMIT;
```

**关键技巧**:
- **强制排序**:保证user_id < liker_id,确保并发操作同一行
- **按位或操作**:`relation_ship | 1`实现幂等性(1|2=3, 3|1=3)
- **唯一索引**:确保行锁生效,解决并发问题

## 课后思考题

**问题**:执行`UPDATE t SET a=2 WHERE id=1`,当a原本就是2时,MySQL如何处理?

**三种可能**:
1. Server层读出数据发现值相同,直接返回
2. 调用引擎接口,引擎发现值相同,直接返回
3. 引擎认真执行更新,加锁更新

**验证方法**:
```sql
-- 会话1
BEGIN;
UPDATE t SET a=2 WHERE id=1;
-- 不提交,观察锁状态

-- 会话2
UPDATE t SET a=2 WHERE id=1;
-- 被阻塞,说明会话1加了行锁
```

**结论**:MySQL采用第3种方式,即使值相同也会加锁更新,保证当前读的正确性。
