# MySQL高可用核心解析

## 核心知识清单（20%重点）

### 1. 主备延迟的本质

**定义**：同一事务在备库执行完成时间与主库执行完成时间的差值（T3-T1）

**关键时间点**：
- T1：主库执行完成事务，写入binlog
- T2：备库接收完binlog
- T3：备库执行完成事务

**监控指标**：`seconds_behind_master`
- 计算方法：备库当前执行事务的时间字段与系统时间的差值
- 精度：秒级
- 查看命令：`show slave status`

**思考**：为什么网络正常时主备延迟主要来自T3-T2而非T2-T1？

### 2. 主备延迟的三大来源

#### 2.1 备库机器性能差
- **错误认知**：备库无请求，可用差机器
- **实际情况**：更新操作对IOPS压力主备无差别
- **解决方案**：主备对称部署，相同规格机器

#### 2.2 备库压力大
- **场景**：备库承担读请求或分析查询
- **解决方案**：
  - 一主多从：分散读压力
  - binlog输出到外部系统（如Hadoop）：处理统计类查询

#### 2.3 大事务
- **典型场景**：
  - 一次性删除大量数据（delete语句）
  - 大表DDL操作
- **影响**：主库执行多久，备库就延迟多久
- **解决方案**：
  - 分批删除数据，控制每个事务的数据量
  - 使用gh-ost方案处理DDL

**思考**：如果监控发现备库延迟呈45°斜向上的线性增长，可能是什么原因？

### 3. 可靠性优先切换策略

**切换流程**（双M结构）：
1. 判断备库`seconds_behind_master < 5秒`，否则重试
2. 主库A改为只读（`readonly=true`）
3. 等待备库B的`seconds_behind_master = 0`
4. 备库B改为可读写（`readonly=false`）
5. 业务请求切到备库B

**特点**：
- 存在不可用时间（步骤2-5期间系统只读）
- 步骤3最耗时（需等待数据完全同步）
- 保证数据一致性

**思考**：为什么步骤1要先判断延迟小于5秒？如果延迟30分钟会怎样？

### 4. 可用性优先切换策略

**切换流程**：
- 直接切换到备库并设为可读写
- 不等待主备数据同步

**风险**：
- 可能出现数据不一致
- `binlog_format=mixed`：数据悄悄不一致
- `binlog_format=row`：报duplicate key error，更容易发现

**适用场景**：
- 操作日志库（数据不一致可通过binlog修补）
- 业务强依赖写入且可接受短暂不一致

**思考**：为什么row格式比mixed格式更容易发现数据不一致问题？

### 5. 异常切换的处理

**场景**：主库掉电，主备延迟30分钟

**问题**：
- 必须等待备库`seconds_behind_master = 0`才能切换
- 系统完全不可用（非只读问题）
- 直接切换会导致"暂时数据丢失"现象

**核心结论**：MySQL高可用系统的可用性依赖于主备延迟，延迟越小，故障恢复越快。



## 实践要点总结

### 减少主备延迟的关键措施
1. **硬件对称**：主备库使用相同规格机器
2. **压力分散**：一主多从，避免备库压力过大
3. **避免大事务**：分批删除、使用gh-ost
4. **监控预警**：实时监控`seconds_behind_master`

### 切换策略选择原则
- **默认选择**：可靠性优先策略
- **特殊情况**：日志库、可降级业务可用可用性优先
- **核心底线**：数据准确性优于可用性

### 高可用设计思路
- 主备延迟是高可用的核心指标
- 减少延迟 = 提升可用性
- 数据可靠性是数据库服务的底线
