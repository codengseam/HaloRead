# 核心知识清单(20%重点)

## 一、表定义自增值id

### 关键知识点

**1. 上限行为机制**
- **定义**: 自增id达到上限(2^32-1=4294967295)后,再申请时值保持不变
- **应用场景**: 频繁插入删除的表可能达到上限
- **错误表现**: 报主键冲突错误 `Duplicate entry '4294967295' for key 'PRIMARY'`
- **思考**: 为什么表自增id达到上限后选择报错而非覆盖数据?

**2. 规避方法**
- **建表评估**: 考察表是否可能达到上限
- **解决方案**: 使用 `bigint unsigned`(8字节)替代 `int unsigned`(4字节)
- **实践要点**: 对于高频写入表,建表时直接使用bigint unsigned

### 实操练习(30秒落地)
检查现有表的自增id类型:
```sql
SHOW CREATE TABLE your_table;
-- 如果是int unsigned,评估是否需要改为bigint unsigned
```



## 三、Xid(事务标识)

### 关键知识点

**1. 生成机制**
- **维护层**: Server层
- **全局变量**: `global_query_id`(纯内存,重启清零)
- **分配规则**: 事务第一条语句时赋值

**2. 唯一性保证**
- **同实例**: 不同事务Xid可能相同(重启后)
- **同binlog**: Xid保证唯一(MySQL重启生成新binlog文件)
- **理论上限**: 2^64-1(概率极低,可忽略)

### 思考问题
为什么Xid设计为纯内存变量?重启后清零如何保证binlog内唯一性?



## 五、thread_id(线程id)

### 关键知识点

**1. 唯一性保证机制**
- **全局变量**: `thread_id_counter`(4字节)
- **上限**: 2^32-1后归0
- **去重逻辑**: 使用唯一数组避免重复

**2. 代码实现**
```c
do {
  new_id= thread_id_counter++;
} while (!thread_ids.insert_unique(new_id).second);
```
