# 核心知识清单(20%重点)

## 🔹 幻读的定义

**核心概念**：一个事务在前后两次查询同一个范围的时候,后一次查询看到了前一次查询没有看到的行。

**关键特征**：
- 幻读仅专指"新插入的行",不包括已有行的修改
- 幻读在"当前读"下才会出现(快照读不会看到别的事务插入的数据)
- 在可重复读隔离级别下才会发生

**应用场景示例**：
```sql
-- Session A
begin;
select * from t where d=5 for update;  -- Q1: 返回id=5

-- Session B插入新行
insert into t values(1,1,5);

-- Session A再次查询
select * from t where d=5 for update;  -- Q2: 返回id=0, id=1, id=5
-- id=1这一行就是"幻读"
```

**思考问题**：为什么幻读只在当前读下出现,而快照读不会出现幻读？



## 🔹 间隙锁(Gap Lock)机制

### 定义与作用

**间隙锁**：锁住两个值之间的空隙,防止新记录插入到间隙中。

**关键特性**：
- 间隙锁之间**不冲突**
- 间隙锁只与"往间隙中插入记录"这个操作冲突
- 与行锁不同,行锁分为读锁和写锁,间隙锁没有这种区分

### 示例说明

表t初始化数据：(0,0,0), (5,5,5), (10,10,10), (15,15,15), (20,20,20), (25,25,25)

产生的7个间隙：
```
(-∞, 0), (0, 5), (5, 10), (10, 15), (15, 20), (20, 25), (25, +∞)
```

**加锁示例**：
```sql
select * from t where d=5 for update;
-- 不仅给6个记录加行锁,还给7个间隙加间隙锁
-- 确保无法插入新记录
```

**间隙锁不冲突示例**：
```sql
-- Session A: select * from t where c=7 for update;  -- 加间隙锁(5,10)
-- Session B: select * from t where c=8 for update;  -- 也加间隙锁(5,10)
-- Session B不会被阻塞,因为间隙锁之间不冲突
```

**思考问题**：为什么间隙锁之间不冲突？这样设计的目的是什么？



## 🔹 间隙锁带来的副作用

### 死锁问题

**典型场景**：
```sql
-- Session A: select * from t where id=9 for update;  -- 加间隙锁(5,10)
-- Session B: select * from t where id=9 for update;  -- 也加间隙锁(5,10),不冲突
-- Session B: insert into t values(9,9,9);            -- 被Session A的间隙锁阻塞
-- Session A: insert into t values(9,9,9);            -- 被Session B的间隙锁阻塞
-- 形成死锁
```

**原因分析**：
- 间隙锁之间不冲突
- 但插入操作会被间隙锁阻塞
- 导致互相等待形成死锁

### 并发度影响

**问题**：间隙锁会导致同样的语句锁住更大的范围,影响并发性能。

**解决方案**：
- 使用读提交隔离级别(没有间隙锁)
- 配合binlog_format=row
- 适用于不需要可重复读保证的业务场景

**思考问题**：什么场景下必须使用可重复读隔离级别？读提交+row格式binlog是否能满足所有场景？
