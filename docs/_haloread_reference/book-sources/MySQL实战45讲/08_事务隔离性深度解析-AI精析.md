# 核心知识清单(20%)

## 一、MVCC核心机制:多版本并发控制

### 1.1 数据多版本机制
**定义**:每行数据在InnoDB中存在多个版本,每个版本都有唯一的事务ID(row trx_id)

**实现原理**:
- 每次事务更新数据时,生成新版本并赋予transaction id作为row trx_id
- 旧版本通过undo log保留,形成版本链
- 版本链结构:最新版本→undo log→历史版本

**应用场景示例**:
```sql
-- 行数据被多次更新后的版本链
V4(row trx_id=25, k=22)
  ↓ undo log U3
V3(row trx_id=24, k=11)
  ↓ undo log U2
V2(row trx_id=23, k=...)
  ↓ undo log U1
V1(row trx_id=22, k=...)
```

**常见错误**:误以为需要物理拷贝整库数据实现快照
**正确理解**:利用版本链+视图数组实现"逻辑快照",无需物理拷贝



## 二、一致性读vs当前读

### 2.1 一致性读(快照读)

**定义**:基于事务启动时创建的视图,按照可见性规则读取历史版本

**执行流程**:
1. 从数据最新版本开始读取
2. 判断row trx_id是否可见
3. 不可见则通过undo log回溯历史版本
4. 直到找到可见版本或到达版本链头部

**应用场景**:
- 普通SELECT语句
- 可重复读隔离级别下的查询



## 三、可重复读(RR)vs读提交(RC)

### 3.1 核心区别

| 维度 | 可重复读(RR) | 读提交(RC) |
|------|-------------|-----------|
| 视图创建时机 | 事务启动时创建一次 | 每条语句执行前重新创建 |
| 视图生命周期 | 整个事务期间共用 | 仅当前语句有效 |
| 可见性范围 | 事务启动前已提交的数据 | 语句执行前已提交的数据 |
| start transaction with consistent snapshot | 有效,立即创建快照 | 无效,等效于普通start transaction |

### 3.2 实例对比

**场景**:事务A查询,事务B更新,事务C更新并提交

**可重复读下**:
- 事务A视图数组:[99,100]
- 查询时判断:
  - (1,3)未提交→不可见
  - (1,2)已提交但在视图后→不可见
  - (1,1)已提交且在视图前→可见
- **结果:k=1**

**读提交下**:
- 事务A查询时重新创建视图
- 查询时判断:
  - (1,3)未提交→不可见
  - (1,2)已提交→可见
- **结果:k=2**



# 次要信息速览(80%)

## 【基础概念补充】
- **视图(view)**:用查询语句定义的虚拟表,与MVCC的read-view是不同概念
- **undo log**:回滚日志,存储数据修改前的状态,用于版本回溯和事务回滚
- **两阶段锁协议**:事务执行过程中需要锁,锁在事务提交时才释放

## 【案例分析要点】
- 文中事务A/B/C的执行流程示例,用于演示MVCC工作机制
- 事务ID分配规则:按申请顺序严格递增
- 版本链遍历:从最新版本开始,通过undo log向前回溯

## 【实践陷阱】
- 表结构不支持可重复读:因为表结构没有row trx_id,只能当前读
- MySQL 8.0已支持将表结构放入InnoDB字典,未来可能支持表结构可重复读

## 【性能优化】
- MVCC避免物理拷贝整库数据,实现秒级快照创建
- 一致性读无需加锁,提高并发性能
- 当前读需要加锁,可能产生锁等待
