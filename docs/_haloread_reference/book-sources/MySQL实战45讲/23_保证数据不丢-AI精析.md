# MySQL数据持久化机制精析

## 核心问题
MySQL如何通过redo log和binlog保证数据不丢失?

## 一、binlog写入机制(20%核心)

### 1.1 写入流程
**核心路径**:事务执行→binlog cache→事务提交→binlog文件

**关键特性**:
- **原子性保证**:一个事务的binlog必须完整写入,不能拆分
- **线程隔离**:每个线程独立维护binlog cache(参数:`binlog_cache_size`控制大小)

### 1.2 write vs fsync的本质区别
| 操作 | 作用 | 性能影响 |
|------|------|----------|
| write | 写入文件系统page cache(内存) | 快,不占磁盘IOPS |
| fsync | 持久化到磁盘 | 慢,占用磁盘IOPS |

**思考**:为什么需要区分write和fsync?
> 通过延迟fsync时机,可以批量持久化多个事务,减少磁盘IOPS消耗

### 1.3 sync_binlog参数配置策略
```yaml
sync_binlog=0: 每次提交仅write,不fsync
  风险:主机异常重启丢失所有未fsync的binlog
  性能:最高

sync_binlog=1: 每次提交都fsync(双1配置)
  安全:最安全,不丢数据
  性能:最差,每次事务都刷盘

sync_binlog=N(N>1): 累积N个事务后fsync
  风险:丢失最近N个事务的binlog
  平衡:性能与安全的折中(推荐100-1000)
```

**实践建议**:生产环境不建议设为0,推荐100-1000



## 三、组提交机制(性能优化核心)

### 3.1 LSN(Log Sequence Number)概念
- **定义**:单调递增的日志序列号,对应redo log写入点
- **作用**:标识日志位置,用于组提交优化

### 3.2 组提交原理
**场景**:三个并发事务(trx1, trx2, trx3)LSN分别为50、120、160

```
执行流程:
1. trx1最先到达→成为leader
2. 等待期间trx2、trx3加入组
3. trx1执行fsync时带LSN=160
4. 所有LSN≤160的redo log都已持久化
5. trx2、trx3直接返回(无需再次fsync)
```

**性能收益**:组员越多,节约IOPS效果越好

### 3.3 两阶段提交优化
**原始流程**:
```
redo log prepare → 写binlog → redo log commit
```

**优化后流程**:
```
redo log prepare
→ binlog write到磁盘
→ redo log fsync(拖到此处)
→ binlog fsync(组提交)
→ redo log commit
```

**优化效果**:
- redo log和binlog都能组提交
- 大幅降低磁盘IOPS消耗

### 3.4 binlog组提交参数
```yaml
binlog_group_commit_sync_delay: 延迟多少微秒后fsync
binlog_group_commit_sync_no_delay_count: 累积多少次后fsync
触发条件:两个参数满足其一即可
```

**实践**:设置delay可增加组员数量,提升性能(会增加响应时间)



## 五、关键设计对比

### 5.1 binlog cache vs redo log buffer
| 特性 | binlog cache | redo log buffer |
|------|--------------|-----------------|
| 归属 | 每个线程独立 | 全局共享 |
| 原因 | binlog必须连续写,不能打断 | 可随时写入,支持搭便车 |
| 格式 | statement/row格式 | page页格式 |

### 5.2 crash-safe保证
**三种情况的处理**:
1. **收到成功消息**→事务一定持久化
2. **收到失败消息**→事务一定失败
3. **收到异常消息**→需重连查询状态,数据库保证内部一致



## 七、常见问题解析

### Q1: 为什么update后ibd文件看不到变化?
**原因**:WAL机制下,数据可能还在内存,未刷盘到ibd文件

### Q2: 事务执行期间crash会导致主备不一致吗?
**答案**:不会
- redo log和binlog都在内存中
- crash后两者都丢失
- 业务角度事务未提交,数据一致

### Q3: binlog写完但未返回就crash,是bug吗?
**答案**:不是
- crash-safe保证内部一致性
- 客户端需重连确认事务状态

## 核心记忆点
1. **write ≠ fsync**:write到page cache,fsync才持久化
2. **双1最安全**:每次事务两次刷盘
3. **组提交提性能**:批量fsync降低IOPS
4. **=0不推荐**:redo log=0风险太大
5. **顺序写+组提交**:WAL两大优势
