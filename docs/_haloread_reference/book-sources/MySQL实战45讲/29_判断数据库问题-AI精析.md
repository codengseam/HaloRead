# 核心知识清单（20%关键项）

## 🔹 数据库健康检测的四层演进逻辑

### 1. **select 1判断的致命缺陷**（核心中的核心）

**定义**：仅检测MySQL进程是否存活，无法发现实际运行问题

**应用场景示例**：
- 当`innodb_thread_concurrency=3`时，3个线程执行`select sleep(100)`占用全部并发线程
- 此时`select 1`仍能成功返回，但实际业务查询被阻塞
- **结论**：进程存活 ≠ 数据库可用

**常见错误及规避**：
- ❌ 误用：MHA默认使用此方法，但无法检测并发线程过载
- ✅ 正确理解：仅适用于检测进程崩溃，不适用于性能问题

**思考问题**：为什么并发连接数上千没问题，但并发查询数达到128就可能系统瘫痪？



### 3. **更新判断法的双刃剑特性**（实践核心）

**定义**：通过更新系统表检测数据库可用性

**实现方案**：
```sql
CREATE TABLE health_check (
  id int(11) NOT NULL,  -- 使用server_id作为主键
  t_modified timestamp,
  PRIMARY KEY (id)
);

-- 检测命令（避免主备冲突）
INSERT INTO mysql.health_check(id, t_modified)
VALUES (@@server_id, now())
ON DUPLICATE KEY UPDATE t_modified=now();
```

**关键设计要点**：
- 使用`server_id`作为主键，避免双M架构下主备检测冲突
- 更新操作可检测磁盘空间满（binlog写满时更新被阻塞）

**致命缺陷**：
- **IO资源竞争导致误判**：当IO利用率100%时，轻量级检测语句仍能获得IO资源成功执行，而业务SQL已严重超时
- **外部检测的随机性**：定时轮询可能错过问题窗口期

**思考问题**：为什么轻量级检测语句在IO满载时仍能成功执行，导致误判？



# 次要信息速览（80%快速带过）

## 【检测方法对比表】
- **select 1**：仅检测进程存活，MHA默认使用【次要：无法发现性能问题】
- **查表判断**：可检测并发线程过载，但无法发现磁盘空间满【次要：select * from mysql.health_check】
- **更新判断**：可检测磁盘空间问题，但存在IO竞争误判【次要：需用server_id避免主备冲突】
- **内部统计**：最精准但性能损耗约10%【次要：需手动开启监控项】

## 【参数配置建议】
- `innodb_thread_concurrency`：建议64-128【次要：根据CPU核数调整，虚拟机需谨慎】
- 监控超时时间N：根据业务容忍度设置【次要：一般30-60秒】

## 【主备切换场景】
- 主动切换：计划内维护【次要：需提前通知业务方】
- 被动切换：HA系统检测到主库问题【次要：需确保检测逻辑可靠】

## 【常见问题】
- 空间满导致连不上：事务无法提交累积连接数【次要：非空间满直接导致】
- 大表DDL影响：GTID等位点方案会超时10分钟【次要：建议业务低峰期操作】



# 推荐实践方案

**优先级排序**：
1. **基础方案**：update系统表（health_check）检测
2. **增强方案**：配合performance_schema监控IO异常
3. **兜底方案**：保留select 1检测进程崩溃

**关键原则**：
- 检测逻辑需覆盖：进程存活、并发线程、磁盘空间、IO性能
- 避免单一检测点的随机性问题
- 平衡检测准确性与性能损耗
