# 核心知识清单（20%关键项）

## 🔹 一、行锁基础概念

**定义**：针对数据表中行记录的锁，由存储引擎层实现。

**关键差异**：
- **InnoDB**：支持行锁，允许高并发更新
- **MyISAM**：不支持行锁，只能使用表锁，同一时刻只能有一个更新操作

**思考**：为什么MyISAM被InnoDB替代？行锁对业务并发度的影响有多大？



## 🔹 三、死锁与死锁检测

### 3.1 死锁定义

**现象**：不同线程出现循环资源依赖，涉及的线程都在等待别的线程释放资源，导致无限等待。

**典型场景**：
```
事务A：更新id=1 → 等待id=2的锁
事务B：更新id=2 → 等待id=1的锁
→ 互相等待，形成死锁
```

### 3.2 死锁处理策略

| 策略 | 参数 | 默认值 | 优缺点 |
|------|------|--------|--------|
| **超时等待** | `innodb_lock_wait_timeout` | 50s | 简单但等待时间长，设置过短会误伤正常锁等待 |
| **死锁检测** | `innodb_deadlock_detect` | on | 快速发现并处理，但有额外CPU消耗 |

### 3.3 死锁检测的性能问题

**问题根源**：
- 每个被阻塞的线程都要判断是否因自己的加入导致死锁
- 时间复杂度：O(n)
- **热点行更新场景**：1000个并发线程 → 死锁检测成本达100万量级
- **现象**：CPU利用率接近100%，但每秒执行不到100个事务

**思考**：为什么死锁检测会消耗大量CPU资源？



# 次要信息速览（80%快速带过）

## 【行锁实现细节】
- 行锁在引擎层实现，不同引擎支持情况不同
- InnoDB通过锁索引记录实现行锁【次要：无索引时可能锁表】

## 【死锁检测机制】
- 一致性读不加锁，不需要死锁检测
- 检测时只扫描相关事务链，不扫描所有事务【次要：检测算法优化】

## 【MDL锁与表锁区别】
- MDL锁：元数据锁，读读不互斥
- 表锁：MyISAM使用，同一时刻只能一个更新
- 两者是不同结构，需全部不互斥才能并行【次要：锁兼容性矩阵】

## 【实践建议】
- 事务中锁多个行时，热点行锁往后放
- 避免长事务，减少锁持有时间
- 监控死锁日志，优化业务逻辑【次要：`show engine innodb status`查看死锁】
