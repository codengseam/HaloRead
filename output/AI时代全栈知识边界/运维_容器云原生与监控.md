# AI时代全栈知识边界·24|容器云原生与监控

> 前置知识:理解进程与容器隔离、用过 Docker 跑过容器、了解 HTTP 服务与反向代理
> 学完你能:① 说清云原生四要素与声明式 API 优于命令式的根本机制 ② 在白板上画出 Pod / Deployment / Service / ConfigMap / Ingress 关系 ③ 解释 K8s 调谐循环与调度决策依据 ④ 画出日志采集链路(应用 → stdout → Filebeat → Loki → Grafana) ⑤ 用 USE / RED 体系定位 CPU 高是 user 还是 sys、IO 高是 read 还是 write ⑥ 区分 SLI / SLO / SLA 三层语义 ⑦ 写出可上线的 Deployment / Service / Prometheus 抓取配置 YAML

### 一、概念

云原生(Cloud Native)与可观测性(Observability)在全栈中的角色是「让服务在分布式环境下可部署、可调度、可观测」。一句话定义:云原生是一组让应用天然适配云环境(弹性、自愈、可移植)的工程实践,核心要素是容器(Containers)、微服务、声明式 API、不可变基础设施(Immutable Infrastructure);监控则是让这套分布式系统「看得见」的眼睛,通过指标、日志、链路三类信号把系统状态实时映射给工程师。

划清边界:

- 「云原生」不是「上云」。把单体应用搬上云虚拟机不叫云原生,只有应用本身具备容器化、声明式部署、自动伸缩、可观测这些特征,才算云原生。
- 「不可变基础设施」指服务器/容器实例一旦部署就不再修改,要更新就替换新实例,不在线上 ssh 改文件。这是与「宠物 vs 牲畜」同源的理念——实例不再被精心维护,坏了直接换。
- 「可观测性」不等同于「监控」。监控是「预先定义该看什么」,可观测性是「出问题时能临时下钻查什么」,后者要求系统暴露足够丰富的现场信号。

几个核心术语先对齐:

- Pod:K8s 最小调度单元,一个或多个紧密耦合容器共享网络与存储命名空间。
- Deployment:管理 Pod 副本数与滚动更新的控制器。
- Service:为一组 Pod 提供稳定虚拟 IP 与 DNS 名的抽象。
- ConfigMap:将配置与镜像解耦的键值对资源。
- Ingress:七层路由入口,把 HTTP / HTTPS 路径映射到 Service。
- SLI(Service Level Indicator,服务等级指标):可量化的服务表现度量,如「HTTP 5xx 率」「P99 延迟」。
- SLO(Service Level Objective,服务等级目标):对 SLI 设定的目标值,如「P99 < 200ms」。
- SLA(Service Level Agreement,服务等级协议):与客户签订的违约赔偿级别,通常比 SLO 更宽松。

### 二、原理

#### 1. 声明式 API 为什么优于命令式

命令式(Imperative)是「告诉系统做什么」:`kubectl run nginx --image=nginx` 启动一个容器;`kubectl scale deployment nginx --replicas=5` 扩容到 5 个。每条命令是一次动作,执行完不留痕迹,系统状态只能靠人记或脚本回放。

声明式(Declarative)是「告诉系统想要什么」:写一份 YAML 描述「我想要 5 个 nginx 副本」,交给 K8s,K8s 持续调谐(Reconcile)使实际状态趋向期望状态。其核心原理是控制循环(Control Loop):Controller 不断轮询「实际状态」与「期望状态」的差异,有偏差就执行动作消除偏差。

声明式优于命令式的根本机制有三:

- **可重入**:同一份 YAML 应用 N 次结果一致,因为是「设定目标」而非「执行动作」,系统自动跳过已满足的部分。命令式脚本则可能因重复执行出错(重复创建资源)。
- **可 diff**:两份 YAML 可以用 git diff 比较,变更评审、回滚、审计都有据可查。命令式操作只留下历史命令记录,无法结构化对比。
- **自愈**:节点宕机导致 Pod 数量从 5 降到 3,Controller 发现实际偏离期望,自动在新节点拉起 2 个新 Pod,无需人工干预。这是 K8s「自愈」能力的根。

#### 2. Prometheus pull 模型为什么比 push 模型更适合云原生

监控系统数据采集有两种模式:push(应用主动把指标推给监控服务器)与 pull(监控服务器主动到应用拉取指标)。Prometheus 选 pull,根本机制有四:

- **短连接、可控**:pull 模型下,Prometheus 知道有哪些目标、什么时候拉、拉取频率,流量与超时可控。push 模型下,任意应用可任意时刻推送,监控系统无法预知负载,易被雪崩打垮。
- **实例即目标**:每个 Pod 都是独立的抓取目标,Prometheus 通过服务发现拿到 Pod 列表,逐个抓取。Pod 频繁生灭时,push 模型会出现「应用已死但还在推」「新应用没注册推流」的混乱,pull 模型由服务发现统一管理目标,自然同步。
- **故障隔离**:push 模型下,监控服务器宕机时应用推送失败,可能阻塞业务流程(取决于实现);pull 模型下,监控服务器宕机只影响监控本身,业务无感知。
- **本地缓存兜底**:Prometheus 在被采集端可达时拉取,不可达时短期丢失数据,但应用本身不依赖监控服务器存活。配合 Pushgateway 处理短生命周期任务(如 CronJob)的指标推送,pull 模型对长生命周期服务最契合云原生「实例即目标」的模型。

push 模型适合「客户端不能被主动访问」的场景(如 IoT 设备、浏览器内埋点);云原生服务都有稳定网络端点,pull 是更自然的选择。

#### 3. USE / RED 指标体系为什么覆盖大多数故障场景

监控指标太多会迷失,太少会盲区。两套互补的方法论划清了「该看什么」:

- **USE(Utilization / Saturation / Errors)**:面向资源。对每类资源(CPU、内存、磁盘、网络)看「使用率(占多少)」「饱和度(排多长队)」「错误数(出多少错)」。CPU 利用率高不高、磁盘队列长不长、网卡丢包多不多,都属 USE 范畴。
- **RED(Rate / Errors / Duration)**:面向服务。对每个服务看「请求速率(QPS)」「错误率(5xx 占比)」「延迟(P99)」。一个 HTTP 服务出问题,基本逃不出 RED 三维。

USE 看硬件/资源层(是不是机器不行了),RED 看服务层(是不是服务出问题了)。两者叠加覆盖了「资源瓶颈 → 服务降级 → 用户体验劣化」这条因果链。故障定位时先看 RED 定位是哪个服务、哪个维度异常,再用 USE 下钻到该服务所在节点的资源水位,大多数故障可以在这两步内收敛。

进一步细分延迟与错误:

- Latency(延迟)看分位数(P50 / P90 / P99)而非平均,平均会被长尾掩盖——10 个请求 9 个 10ms、1 个 1s,平均 109ms 看着正常,P99 是 1s 才暴露问题。
- Errors(错误)要区分「业务错误」(4xx)与「系统错误」(5xx),前者是用户输入问题,后者才是系统故障。

告警体系对应三层:阈值告警(超阈值即报,简单粗暴但易抖动)、异常检测告警(基于历史基线偏离,适合周期性指标)、告警分级(P0 电话、P1 短信、P2 邮件、P3 静默)。生产环境必须三层叠加,只靠阈值告警会陷入「要么漏报要么噪声爆炸」的两难。

### 三、实践

下面给出可上线的 Deployment / Service / Prometheus 抓取配置三个 YAML,共同组成一个最小可观测的云原生服务。

#### 1. Deployment:声明期望状态

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
  namespace: prod
  labels:
    app: api-server
spec:
  replicas: 3                          # 期望副本数,Controller 调谐目标
  selector:
    matchLabels:
      app: api-server
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1                      # 滚动更新时最多多出 1 个
      maxUnavailable: 0                # 不允许减少可用副本,保证零停机
  template:                            # Pod 模板
    metadata:
      labels:
        app: api-server
      annotations:
        prometheus.io/scrape: "true"   # Prometheus 抓取开关
        prometheus.io/port: "8080"
        prometheus.io/path: /metrics
    spec:
      containers:
        - name: api
          image: registry.example.com/api:v1.2.0
          ports:
            - containerPort: 8080
          resources:
            requests:                  # 调度依据:节点必须剩这么多资源
              cpu: "250m"              # 250 millicpu = 0.25 核
              memory: "256Mi"
            limits:                    # 硬上限,超限触发限流或 OOM
              cpu: "500m"
              memory: "512Mi"
          livenessProbe:               # 存活探针:失败则重启容器
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 10
          readinessProbe:              # 就绪探针:失败则从 Service 摘除
            httpGet:
              path: /ready
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 5
          env:
            - name: LOG_LEVEL
              valueFrom:
                configMapKeyRef:
                  name: api-config
                  key: log_level
```

几处关键设计的根本机制:

- `replicas: 3` + `maxUnavailable: 0`:Controller 始终维持 3 个就绪副本,滚动更新时新副本就绪才替旧副本,实现零停机。
- `requests` 与 `limits` 分离:`requests` 给调度器用(决定 Pod 调度到哪台节点),`limits` 给 cgroups 用(决定运行时硬上限)。只设 limits 不设 requests 会让调度器无依据,容易把 Pod 调度到已满节点。
- `livenessProbe` 与 `readinessProbe` 分离:前者失败重启容器(治「卡死」),后者失败摘除流量(治「假活」)。混用会导致「服务还没起来就被摘除」或「服务已死还接流量」两类故障。
- 注解 `prometheus.io/scrape` 让 Prometheus 通过服务发现自动识别并抓取该 Pod。

#### 2. Service:稳定入口与负载均衡

```yaml
apiVersion: v1
kind: Service
metadata:
  name: api-server
  namespace: prod
spec:
  type: ClusterIP                       # 集群内可达虚拟 IP
  selector:
    app: api-server                     # 选中带此 label 的 Pod
  ports:
    - name: http
      port: 80                          # Service 对外端口
      targetPort: 8080                  # 转发到容器端口
      protocol: TCP
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-ingress
  namespace: prod
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: api-server
                port:
                  number: 80
```

Service 的核心原理是「虚拟 IP + kube-proxy 转发」:`api-server` 这个 DNS 名解析到一个 ClusterIP,该 IP 是虚拟的(没有网卡对应),kube-proxy 在每个节点上用 iptables / IPVS 规则把它重写到后端 Pod IP。Pod 频繁生灭,Pod IP 不断变化,但 Service 的虚拟 IP 与 DNS 名稳定,调用方无需感知后端变化。Ingress 进一步在七层做路由,把 `api.example.com/` 映射到 Service,完成从域名到 Pod 的整条链路。

#### 3. Prometheus 抓取配置

```yaml
global:
  scrape_interval: 15s                  # 默认每 15s 拉一次
  evaluation_interval: 15s              # 每 15s 评估一次告警规则
scrape_configs:
  - job_name: 'kubernetes-pods'         # 抓取带注解的 Pod
    kubernetes_sd_configs:
      - role: pod                       # 用 K8s 服务发现,role=pod 自动枚举 Pod
    relabel_configs:
      # 仅抓取注解 prometheus.io/scrape=true 的 Pod
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
        action: keep
        regex: true
      # 把注解里的端口写入抓取地址
      - source_labels: [__address__, __meta_kubernetes_pod_annotation_prometheus_io_port]
        action: replace
        regex: ([^:]+)(?::\d+)?;(\d+)
        replacement: $1:$2
        target_label: __address__
      # 把 Pod 名、命名空间写入指标 label,便于查询定位
      - source_labels: [__meta_kubernetes_namespace]
        target_label: namespace
      - source_labels: [__meta_kubernetes_pod_name]
        target_label: pod
```

这份配置展示 pull 模型的核心:Prometheus 主动连 K8s API 拿 Pod 列表,过滤出带抓取注解的,逐个拉取 `/metrics`。Pod 重启换 IP、扩容新增 Pod、缩容删除 Pod,服务发现自动同步,无需手动维护抓取目标列表。日志体系与监控同源:应用日志写 stdout / stderr → 节点上的 Filebeat / Fluentd 采集 → 中心化存储(ES / Loki)→ Kibana / Grafana 查询。这套链路与 Prometheus 指标链路并行,共同构成可观测性的两根支柱。

### 四、速查/自测

#### Kubernetes 对象速查表

| 对象 | 作用 | 典型字段 | 关联对象 |
|---|---|---|---|
| Pod | 最小调度单元,一个或多个容器 | containers, volumes | ConfigMap, Secret |
| Deployment | 管理 Pod 副本数与滚动更新 | replicas, strategy, template | Pod |
| Service | 给一组 Pod 提供稳定入口 | selector, ports, type | Pod |
| ConfigMap | 配置键值对,与镜像解耦 | data | Pod(以 env 或 volume 挂载) |
| Secret | 敏感配置(密码、证书) | data(base64) | Pod |
| Ingress | 七层路由入口 | rules, tls | Service |
| StatefulSet | 有状态应用(数据库) | serviceName, volumeClaimTemplates | Pod, PVC |
| DaemonSet | 每节点跑一个副本(日志采集) | template | Pod |
| HorizontalPodAutoscaler | 按 CPU / QPS 自动伸缩 | minReplicas, maxReplicas, metrics | Deployment |

#### 故障定位速查表

| 监控异常 | 第一步看 | 第二步看 | 可能根因 |
|---|---|---|---|
| CPU 高 | user / sys 占比 | 该 Pod 调用链 | user 高=业务计算重;sys 高=系统调用/上下文切换多 |
| 内存涨 | RSS 曲线 | GC 频率 | 内存泄漏 / GC 不及时 / OOM 即将触发 |
| IO 高 | read / write 占比 | iostat await | read 高=大量读盘;write 高=日志/落盘压力 |
| 延迟 P99 高 | RED 错误率 | 上游调用慢查询 | 下游慢 / 排队 / GC 停顿 |
| 5xx 突增 | 错误日志 | 该服务依赖 | 依赖故障 / 配置错误 / 版本回归 |
| Pod 重启 | liveness 失败次数 | OOMKiller 事件 | 探针太严 / 内存超限 / 进程崩溃 |

#### 自测题

1. **原理层**:声明式 API 与命令式 API 的本质区别是什么?为什么说 Controller 的控制循环让 K8s 具备「自愈」能力?

   <details><summary>参考答案</summary>
   命令式是「告诉系统做什么」,每条命令是一次动作,执行完不留痕迹;声明式是「告诉系统想要什么」,系统持续调谐使实际状态趋向期望状态。Controller 不断轮询实际与期望状态的差异,节点宕机导致 Pod 数从 5 降到 3,Controller 发现偏差后自动在新节点拉起 2 个新 Pod,无需人工干预——这就是自愈。
   </details>

2. **思路层**:某服务 P99 延迟突然从 50ms 涨到 800ms,但 5xx 率正常。用 RED + USE 体系该怎么定位?

   <details><summary>参考答案</summary>
   先用 RED 确认:Rate 是否同时上涨(可能流量突增)?Duration P99 高但 Errors 正常,说明请求都成功只是变慢,排除下游故障。再下钻到该服务所在节点的 USE:CPU 利用率与饱和度(load / 运行队列)是否高?如果 user 高可能是业务计算重,sys 高可能是上下文切换过多;如果 CPU 不高但内存饱和度高,可能是 GC 频繁导致 STW 停顿;如果 IO 高(看 read 还是 write),可能是日志写入或落盘阻塞。最后看该服务的调用链,定位是哪个下游变慢。
   </details>

3. **实践层**:Deployment 里只设了 `limits` 没设 `requests`,会出什么问题?

   <details><summary>参考答案</summary>
   调度器无依据判断节点是否够资源,可能把 Pod 调度到已经满了的节点,导致 Pod 启动后因资源不足被驱逐或 OOM。K8s 在只设 limits 不设 requests 时默认 requests 等于 limits,但显式分离二者是规范:`requests` 给调度用,`limits` 给 cgroups 用,二者分离才能既保证调度合理又限制运行时硬上限。
   </details>

4. **原理层**:Prometheus 为什么选 pull 而不是 push?短生命周期任务(如 CronJob)的指标怎么采集?

   <details><summary>参考答案</summary>
   pull 模型让监控服务器主动控制抓取目标与频率,流量可控、故障隔离(监控宕机不影响业务),配合服务发现能自动同步 Pod 生灭。短生命周期任务来不及被 pull,需用 Pushgateway:任务把指标 push 到 Pushgateway,Prometheus 再从 Pushgateway pull。Pushgateway 只适合这种短任务,长生命周期服务仍应直接被 Prometheus pull。
   </details>

5. **思路层**:某 Pod 频繁重启,`kubectl describe` 显示 `OOMKilled`。但该业务实际内存占用并不高,可能是什么原因?

   <details><summary>参考答案</summary>
   几种可能:① `limits.memory` 设得过低,业务正常峰值就被杀;② 容器内进程看不到 cgroups 限制,JVM / Python 按宿主机内存设堆,实际超限被杀;③ 内存泄漏,长时间运行后逐渐逼近上限;④ fork 炸弹或瞬时大对象分配,提交内存尖峰超限。排查:看 `limits.memory` 是否合理、看监控 RSS 曲线是平稳上涨(泄漏)还是尖峰(突发)、看 GC 频率、必要时调大 limit 或修复代码。
   </details>

### 可交给 AI 的部分

可以放心交给 AI 的:

- **完整 K8s YAML**:给定应用名、镜像、端口、资源需求,AI 能生成 Deployment / Service / ConfigMap / Ingress 一整套 YAML,含探针、资源限制、滚动更新策略。人工只审关键字段。
- **Jenkins / GitLab CI 流水线脚本**:从代码提交 → 构建镜像 → 推 registry → kubectl apply 的完整 pipeline,AI 出模板后由人补业务参数。
- **运维监控大盘配置脚本**:Grafana Dashboard 的 JSON、Prometheus 告警规则文件,AI 能按 RED / USE 体系生成初版。
- **Helm Chart 骨架**:Chart.yaml、values.yaml、templates/ 目录结构,AI 能搭好框架。
- **kubectl 排查命令组合**:把症状描述给 AI,能给出 `kubectl describe` / `kubectl logs` / `kubectl get events` 的组合与解读思路。

风险提示:

- **AI 写的 YAML 经常漏 resources 字段**:不设 requests / limits 会导致调度无依据、节点资源争抢,生产规范必须由人补。
- **AI 容易混淆 liveness 与 readiness 探针**:经常把同一个 endpoint 配给两个探针,导致「服务没起来就被杀」或「死了还接流量」。探针的 path / 延迟 / 阈值必须由人审。
- **AI 给的告警阈值是「拍脑袋」**:CPU 80% 报警、P99 500ms 报警这种阈值没有考虑业务基线,要么误报雪崩要么漏报。阈值必须基于历史数据调,不能直接用 AI 给的初值。
- **AI 不懂 SLO 的误差预算(error budget)**:让它配 SLO,经常给出「99.9% 但不写时间窗口」,忽略滚动窗口与预算消耗策略。SLO 的窗口与预算消耗策略必须人定。
- **AI 写的监控指标名经常不合规**:Prometheus 指标名应 snake_case + 单位后缀(如 `http_request_duration_seconds`),AI 经常给 `http_request_time` 这种不合规命名,导致后续聚合与单位换算出错。
- **AI 倾向于「什么都上 K8s」**:小规模服务用 Docker Compose 或 Serverless 更省心,AI 不会主动提醒「这个场景不需要 K8s」,选型必须人把关。

为什么这部分能交、那部分不能交:可交的都是「结构化、可验证、可回滚」的产出——YAML apply 失败能看、流水线跑挂能重跑、Dashboard 配错能改。不能交的是「需要工程判断」的决策:探针策略、资源配额、告警阈值、SLO 窗口、是否该上 K8s。这些一旦错,出的是线上事故而非构建报错。AI 是高产出但无线上意识的执行者,你是把关者,本章讲的就是把关所需的最小可观测与编排知识——分布式事务一章会反复用到这套 SLO 与告警分级视角。
