# AI时代全栈知识边界·32|需求建模与接口设计

> 前置知识:关系型数据库基础、HTTP 请求方法、看过一份 API 文档
> 学完你能:把一句模糊需求拆成用户故事与验收标准,画出 ER 图与状态机,按 RESTful 规范设计资源命名与状态码,用 OpenAPI 把前后端契约钉死,判断该选 REST 还是 GraphQL

## 一、概念

需求建模与接口设计解决的是"需求怎么不丢、接口怎么不乱"——把模糊的业务诉求翻译成结构化的模型,再把模型翻译成前后端都能照着写代码的接口契约。前者面向人和业务,后者面向代码和工程。

核心术语中英对照:

- User Story(用户故事):从用户视角描述一句话需求
- Use Case(用例):描述系统与外部交互者协作完成目标的完整流程
- Acceptance Criteria(验收标准):需求完成的可验证条件
- ER(Entity-Relationship,实体-关系):用实体、属性、关系刻画数据结构的方法
- Conceptual / Logical / Physical Model(概念模型/逻辑模型/物理模型):数据建模的三层抽象
- REST(Representational State Transfer,表述性状态转移):基于资源和 HTTP 动词的接口风格
- RESTful:符合 REST 约束的接口实现风格
- GraphQL(图查询语言):按需取字段、单端点查询的接口风格
- OpenAPI(开放 API 规范):描述 REST 接口的机器可读规范,前身叫 Swagger
- Mock(打桩):接口未实现时返回假数据供前端联调
- Contract(契约):前后端共同认可、变更需协商的接口约定

需求建模和接口设计不是两件事,是一条链:用户故事 → 用例 → 业务流程 → ER 图 → 数据模型 → 接口契约。前一段的产物是后一段的输入,任何一段出错都会向下游放大。

## 二、原理

### 为什么需要用户故事:从用户视角而非功能视角

需求最常踩的坑不是"没写清楚",而是"写错视角"。功能视角的需求长这样:"系统提供导出 Excel 功能"。这句话没毛病,但没法验证——导出给谁?什么场景下导?导出失败怎么办?开发按自己理解做出来,产品说不是想要的,返工。

用户故事强制把视角切到使用者:"作为一个运营,我希望能按筛选条件导出订单 Excel,以便月底做对账报表。"三段式 `As a <角色>, I want <功能>, so that <价值>` 把"为谁做、做什么、为什么做"钉死。价值的写法最关键,它逼着产品想清楚这个需求到底解决什么问题——如果写不出价值,大概率这个需求本身就不该做。

用户故事的精髓不在格式,在"可对话"。故事卡只是讨论的起点,真正的需求藏在产品和开发围绕这张卡的对话里。所以用户故事不追求写得多详细,追求"足够具体到能估算、足够开放到能对话"。

### 用例与验收标准:补全故事之外的流程与边界

用户故事偏轻,有些复杂交互单靠三段式讲不清,这时用 Use Case 补流程。用例描述"前置条件 + 主流程 + 备选流程 + 后置条件",把异常路径也写进来。比如"用户下单"的主流程是选商品→填地址→支付,备选流程包括"库存不足""支付超时""优惠券失效",这些分支用户故事里通常不写,但不写就会被开发漏掉。

验收标准(Acceptance Criteria)是需求"完成"的判定线,常用 Given-When-Then(Gherkin)格式:

```text
给定 用户已登录且购物车有 2 件商品
当 用户点击"去结算"
那么 系统跳转到订单确认页,显示 2 件商品和应付金额
```

验收标准的本质是"可测试"——它直接对应测试用例。没有验收标准的需求,开发完没法验,产品验收变成"凭感觉",扯皮不可避免。

### RESTful 资源设计的本质:资源 + HTTP 动词

REST 的核心是把后端能力建模成"资源(Resource)",用 HTTP 方法表达对资源的动作,用 URL 表达资源的位置。资源是名词,动作是动词,动词不写进 URL——这是 REST 与 RPC(远程过程调用)的根本分界。

RPC 风格的接口长这样:`POST /createOrder`、`POST /deleteUser?id=1`、`GET /getOrderList`。每个接口都是一个"动作",URL 是动词,参数塞 query 或 body。这种写法直观但失控:动作无限增长,`/approveOrder`、`/rejectOrder`、`/cancelOrder`、`/reopenOrder`……接口表越来越长,彼此关系看不出。

RESTful 风格把"订单"抽象成资源,URL 只剩名词 `/orders`,动作交给 HTTP 方法:`POST /orders` 创建、`GET /orders` 列表、`GET /orders/{id}` 详情、`PUT /orders/{id}` 更新、`DELETE /orders/{id}` 删除。动作类操作(批准、取消)用子资源表达:`POST /orders/{id}/cancellations`。这样接口表收敛在资源上,新增动作不会无限膨胀 URL,资源之间的关系(订单属于用户)也能用嵌套路径表达:`GET /users/{id}/orders`。

RESTful 不是宗教,几条硬约束守住即可:URL 用名词复数、HTTP 方法语义不乱用、状态码不撒谎(不要 POST 创建成功却返回 200 + body 里 code=0 这种自创协议)。守住这三条,接口的可读性和可维护性就已经超过大多数项目。

### 数据建模三层:从业务到存储的渐进细化

ER 图解决"系统里有哪些实体、它们什么关系"。设计思路分三步:

1. 实体识别:从用户故事和用例里圈名词。"用户下单购买商品"里能圈出用户、订单、商品三个实体。
2. 关系识别:动词连实体。用户"下"订单、订单"包含"商品。
3. 基数标注:1:1(一对一)、1:N(一对多)、N:M(多对多)。用户与订单是 1:N(一个用户多个订单,一个订单属一个用户),订单与商品是 N:M(一个订单多件商品,一件商品出现在多个订单),物理落地需中间表。

数据建模分三层,从抽象到具体:

- 概念模型:只画实体和关系,不关心字段和类型,产物就是 ER 图。给产品看。
- 逻辑模型:加属性、主键外键、范式化(第三范式),但不绑定具体数据库。给架构评审。
- 物理模型:落地到具体 RDBMS,加索引、字段类型、分区、表空间。给 DBA 和开发。

三层不是流程仪式,是控制变更影响范围的工具:概念模型变,逻辑物理都得动;物理模型调索引不影响逻辑概念。需求变更时先问"动到哪层",能省一半返工。

### OpenAPI 为什么成为标准:机器可读 + 多语言生成 SDK

接口文档用 Word 写的痛点:文档和代码两份,改了代码忘改文档,前端拿过时文档联调一路踩坑。OpenAPI 用一份 YAML(或 JSON)描述接口的全部要素——路径、方法、参数、请求体、响应、状态码、数据模型,这份文件既是文档也是契约。

它成为标准的根本机制是"机器可读":因为格式固定,工具链能围绕它生成一堆东西——前端用 `openapi-typescript` 生成 TypeScript 类型,后端用 `fastapi-codegen` / `openapi-generator` 生成路由骨架和客户端 SDK,测试用 `schemathesis` 跑契约测试,Swagger UI 直接渲染成交互式文档。一份契约文件驱动前后端测试四面八方,文档与代码天然同步。

### GraphQL vs REST:选型的真实权衡

GraphQL 不是 REST 的升级版,是另一种取舍。GraphQL 单端点、客户端按需指定字段、一次查询拼多个资源,适合"前端页面字段需求多变、聚合多个后端服务、移动端要省流量"的场景。代价是服务端要实现查询解析、权限按字段粒度控制、缓存复杂(REST 靠 HTTP 缓存,GraphQL 要应用层缓存)、N+1 查询要 DataLoader 兜底。

REST 适合"资源边界清晰、操作以 CRUD 为主、要利用 HTTP 缓存、团队对 REST 熟悉"的场景。多数业务系统用 REST 足够,GraphQL 引入的复杂度只有在"字段需求高度多变"时才回本。盲目上 GraphQL,把原本一个 GET 解决的事变成查询语言解析 + 鉴权 + 缓存三座大山,得不偿失。

## 三、实践

### 用户故事与验收标准的写法

一个完整的用户故事卡,包含故事本体、验收标准、估算和优先级:

```markdown
### US-018 会员用户导出订单 Excel

**故事**:
作为一个已注册的会员用户,
我希望能按日期范围和订单状态筛选并导出订单 Excel,
以便月底自己核对消费记录。

**验收标准**:
1. 给定 用户已登录且是会员,
   当 用户选择日期范围(最多 90 天)和订单状态后点击"导出",
   那么 系统在 10 秒内生成 Excel 并下载,文件名含导出时间戳。

2. 给定 用户未登录或非会员,
   当 用户点击"导出",
   那么 系统提示"请先登录并升级会员",不发起导出。

3. 给定 选中范围内订单数超过 10000,
   当 用户点击"导出",
   那么 系统改为异步任务,提示"导出已提交,完成后站内信通知",不阻塞页面。

**优先级**:P1
**估算**:5 故事点
```

三段式故事 + Given-When-Then 验收标准,这张卡丢给开发和测试,开发知道做什么、测试知道验什么、产品知道怎么验收。异步导出那条是最容易漏的——单靠"提供导出功能"这种需求写法,生产环境跑大订单一定超时。

### 业务流程梳理:流程图与状态机

需求里有两类流程要分别刻画:步骤流转用流程图(Flowchart),状态流转用状态机(State Machine)。流程图答"怎么一步步做",状态机答"当前能转到哪",两者刻画不同侧面——流程图关注动作的先后与分支,状态机关注对象在生命周期里合法的转移路径。

订单状态机把"哪些状态能转到哪些状态"显式化:

```text
CREATED → PAID → SHIPPED → DELIVERED → COMPLETED
   ↓        ↓        ↓
CANCELLED CANCELLED CANCELLED
```

没有状态机,代码里会出现 `if status == 'PAID' and ... and ...` 散落各处,加一个新状态要改十几个分支;有状态机,转移规则集中在一处,非法转移直接拒绝。比如"已发货订单不能取消"这条规则,在状态机里就是 `SHIPPED` 没有 `→ CANCELLED` 的边,代码层面一行拒绝逻辑搞定,而不是在每个取消入口各写一次判断。

### ER 图:实体关系落地

一个订单系统的 ER 模型:

- 实体:User(用户)、Order(订单)、OrderItem(订单项)、Product(商品)
- 关系:User 1:N Order;Order 1:N OrderItem;OrderItem N:1 Product
- N:M 落地:订单与商品本是 N:M,拆成 OrderItem 作为中间表,带 quantity、unit_price 属性

N:M 在关系库里不能直接存,必须拆中间表——这是逻辑模型到物理模型转换的典型步骤。中间表不只是两个外键,往往还要带业务属性(下单时的单价、数量),这些属性挂在 N:M 关系本身上,拆表时跟着搬。

### RESTful 资源命名对照表

把常见错误写法和推荐写法放一起,差距一目了然:

| 场景 | 不推荐(RPC 风格) | 推荐(RESTful) | 说明 |
|---|---|---|---|
| 获取用户列表 | GET /getUsers | GET /users | URL 用名词复数,不带动词 |
| 获取单个用户 | GET /getUserById?id=1 | GET /users/1 | 资源 ID 放路径,不放 query |
| 创建用户 | POST /createUser | POST /users | 创建用 POST,动作交给方法 |
| 全量更新 | POST /updateUser | PUT /users/1 | PUT 表达全量替换且幂等 |
| 部分更新 | POST /updateUserName?id=1 | PATCH /users/1 | 部分字段用 PATCH |
| 删除用户 | GET /deleteUser?id=1 | DELETE /users/1 | 删除用 DELETE,不用 GET |
| 用户的订单 | GET /getUserOrders?uid=1 | GET /users/1/orders | 嵌套表达从属关系 |
| 动作类(取消订单) | POST /cancelOrder?id=1 | POST /orders/1/cancellations | 非 CRUD 用子资源名词 |
| 搜索用户 | POST /searchUsers | GET /users?q=xxx | 查询参数走 query,GET 语义 |
| 批量删除 | POST /batchDeleteUsers | DELETE /users(body 带 ids) | 批量操作复用资源方法 |

第 8 行最容易吵——"取消"是动词,RESTful 不是不让动词进 URL 吗?这里的折中是:把"取消"抽象成名词"取消记录(cancellations)",用 POST 创建一条取消记录来表达动作。实在抽象不出名词的操作(如"重置密码"),也允许 `/users/1/password:reset` 这类写法,但要在团队内统一约定。

### OpenAPI 契约文档

下面是一份最小可用的 OpenAPI 3.1 文档,描述文章(article)资源的列表与创建:

```yaml
openapi: 3.1.0
info:
  title: 文章服务 API
  version: 1.0.0
  description: 内容管理模块的文章资源接口契约
servers:
  - url: https://api.example.com/v1
paths:
  /articles:
    get:
      summary: 获取文章列表
      parameters:
        - name: page
          in: query
          schema: { type: integer, minimum: 1, default: 1 }
        - name: status
          in: query
          schema: { type: string, enum: [draft, published] }
      responses:
        '200':
          description: 成功
          content:
            application/json:
              schema:
                type: object
                properties:
                  total: { type: integer }
                  items:
                    type: array
                    items: { $ref: '#/components/schemas/Article' }
    post:
      summary: 创建文章
      requestBody:
        required: true
        content:
          application/json:
            schema: { $ref: '#/components/schemas/ArticleCreate' }
      responses:
        '201':
          description: 创建成功
          content:
            application/json:
              schema: { $ref: '#/components/schemas/Article' }
        '400': { description: 参数错误 }
        '401': { description: 未登录 }
        '409': { description: 标题重复 }
components:
  schemas:
    Article:
      type: object
      properties:
        id: { type: integer, format: int64 }
        title: { type: string }
        status: { type: string, enum: [draft, published] }
        created_at: { type: string, format: date-time }
    ArticleCreate:
      type: object
      required: [title]
      properties:
        title: { type: string, minLength: 1, maxLength: 120 }
        content: { type: string }
        status: { type: string, enum: [draft, published], default: draft }
```

这份文件定稿后,前端立即用 Mock 跑通页面,后端按 schema 实现路由,联调时只对齐契约差异。状态码用 HTTP 标准语义:201 创建成功、400 参数错、401 未登录、409 冲突——不要一律返回 200 再在 body 里塞 `code: 0/1`,那是把 HTTP 状态码当摆设,前端要写两层错误处理。

### 前后端交互流程:契约 → Mock → 联调

契约先行的完整流程:

1. 评审阶段:产品和前后端一起评审需求,产出 OpenAPI 契约文件,签入仓库。
2. Mock 阶段:前端用 `prism mock openapi.yaml` 或 Mock Service Worker 起本地 Mock 服务,不等后端实现就开发页面。
3. 并行开发:后端按契约实现路由,前端按契约调用 Mock,两边进度互不阻塞。
4. 联调阶段:后端 ready 后,前端切到真实接口,理论上只对齐契约差异(字段名、状态码、边界返回)。
5. 契约测试:CI 里跑 `schemathesis run openapi.yaml --base-url http://staging`,自动校验真实接口是否符合契约。

契约一旦定稿,变更必须走 PR——改 OpenAPI 文件就是改契约,PR 评审时前后端都能看到。这把"口头改接口、改完不通知"的混乱彻底消灭。

### 接口版本管理:URL / Header / 参数

接口不可避免要演进,版本管理决定"老客户端怎么不被改死"。三种主流方案:

| 方案 | 写法 | 优点 | 缺点 |
|---|---|---|---|
| URL 版本 | `/v1/users`、`/v2/users` | 直观,浏览器和网关都能直接看到版本,缓存友好 | URL 里混版本,有人觉得"不 RESTful" |
| Header 版本 | `GET /users` + `Accept: application/vnd.example.v2+json` | URL 干净,版本属内容协商 | 不直观,Postman 调试麻烦,网关路由难 |
| 参数版本 | `/users?version=2` | 改动最小 | 容易和业务参数混淆,缓存失效 |

工程上 URL 版本最常用,直观且工具链支持好。版本不是越多越好——同时维护 v1/v2/v3 是负担,通常 v2 上线后给 v1 一个弃用期(6-12 个月),到期下线。版本号跳变要克制,字段增删、状态码调整不一定升版本(向后兼容的新字段不算破坏性变更),只有"响应结构无法兼容旧客户端解析"时才升大版本。

## 四、速查/自测

### HTTP 方法 vs 资源操作对照表

| HTTP 方法 | 语义 | 幂等 | 安全 | 典型场景 | 示例 |
|---|---|---|---|---|---|
| GET | 读取 | 是 | 是 | 获取资源/列表 | GET /users/1 |
| POST | 创建/触发动作 | 否 | 否 | 新建资源、非幂等动作 | POST /users |
| PUT | 全量替换 | 是 | 否 | 整体更新资源(带全部字段) | PUT /users/1 |
| PATCH | 部分更新 | 否 | 否 | 修改部分字段 | PATCH /users/1 |
| DELETE | 删除 | 是 | 否 | 删除资源 | DELETE /users/1 |

- 幂等(Idempotent):同一请求重复执行结果相同。PUT 重复执行结果一样(都是"设成这个值"),POST 重复执行会创建多条记录,所以 POST 不幂等。
- 安全(Safe):不改变服务器状态。GET 必须安全(理论上),DELETE 改状态所以不安全。安全方法可被缓存、被爬虫预取,不安全方法不行。

### 自测题

1. **概念层**:用户故事和用例的核心差别是什么?能否用一个例子说明何时该用哪个?
   <details><summary>参考答案</summary>用户故事是"用户想要什么价值"的轻量描述,三段式,偏需求意图,适合中小功能。用例是"系统与外部如何协作完成目标"的完整流程,含前置/主流程/备选/后置,偏交互细节,适合复杂业务流。例:做"密码登录"用用户故事够了;做"OAuth 第三方登录含回调、绑定、解绑"这种多分支流程,要写用例把异常路径钉死。</details>

2. **原理层**:为什么 RESTful 强调 PUT 的幂等性?这对客户端重试有什么实际意义?
   <details><summary>参考答案</summary>PUT 语义是"把这个资源设成我给的状态",重复执行结果相同,因此幂等。客户端在网络抖动时不知道请求有没有到服务端,可以安全重试 PUT——最坏情况是服务端多执行一次,结果不变。POST 不幂等,重试可能创建多条订单,所以 POST 重试必须靠幂等键(Idempotency-Key)等业务层机制兜底。</details>

3. **实践层**:"用户取消订阅"这个需求,以下两种设计哪个更合适?为什么?
   - A: `DELETE /subscriptions/{id}`
   - B: `POST /subscriptions/{id}/cancellations`
   <details><summary>参考答案</summary>B 更合适。DELETE 的语义是"删除资源",取消订阅并不是把订阅记录从数据库删掉,而是把状态置为 cancelled,要保留记录用于续费追溯、退款、审计。用 B 表达"创建一条取消记录"更贴合业务,也支持记录取消原因、取消时间。A 会让"已取消的订阅"在列表里消失,丢失业务上下文。</details>

4. **实践层**:ER 图里 User 和 Role 是 N:M(一个用户多个角色,一个角色多个用户),物理模型里怎么落地?
   <details><summary>参考答案</summary>N:M 不能直接在关系库实现,要拆中间表:建 `user_role(user_id, role_id)` 表,主键为 (user_id, role_id) 联合主键,各自外键指向 user 和 role。中间表可带额外属性(如 assigned_at 授权时间)。这是逻辑模型到物理模型转换的典型步骤——逻辑层画 N:M 直线,物理层落地成第三张表。</details>

5. **思路层**:GraphQL 在什么场景比 REST 更合适?什么场景反而是负担?
   <details><summary>参考答案</summary>合适:前端页面字段需求高度多变(同一资源不同页面要不同字段)、需聚合多个微服务、移动端要省流量(只取需要的字段)。负担:资源边界清晰、操作以 CRUD 为主、团队不熟 GraphQL——这时引入 GraphQL 要额外实现查询解析、字段级鉴权、应用层缓存、DataLoader 防 N+1,复杂度大幅上升而收益不明显,REST 更划算。</details>

### 可交给 AI 的部分

哪些内容可以放心交给 AI:

- 业务流程图与状态机绘制:把用例文本或状态描述喂给 AI,生成 Mermaid / PlantUML 流程图和状态图,人工只校验节点和转移是否完整。这类工作 AI 生成快、人工画慢,性价比高。
- ER 图初稿:把用户故事和用例喂给 AI,圈名词、标关系、标基数,产出 ER 草图,人工再补业务专属约束(如"订单号全局唯一")。
- 需求文档与用户故事初稿:把口头需求喂给 AI,生成用户故事卡 + Given-When-Then 验收标准初稿,人工补业务边界和优先级。
- OpenAPI 文档草稿:把资源列表喂给 AI,生成路径、参数、schema、状态码初稿,人工复核字段类型和约束。
- Mock 数据生成:按 OpenAPI schema 自动生成符合约束的 Mock 数据,几乎零风险。

风险提示:

- AI 画 ER 图容易漏关系或标错基数:它会把"用户登录日志"也当实体,把 1:N 标成 N:M。基数错了物理表结构全错,必须人工逐条核对。
- AI 写的用户故事容易写成功能视角:"系统提供导出功能"而不是"作为运营我希望能导出以便对账"。故事视角错了,验收标准会跟着错,必须人工改回用户视角。
- AI 生成的 OpenAPI 经常把 `integer` 写成 `number`、漏 `required`、把枚举写成自由字符串、状态码全用 200。契约文件下游全靠它生成代码,错一处全链路返工,必须人工逐字段对照需求复核。
- AI 画状态机容易漏异常转移:它画得出"创建→支付→发货"主流程,但漏掉"支付超时""发货失败""退款回滚"这些异常分支。状态机漏一条转移,生产就多一类无法处理的订单,必须人工列出所有异常场景再让 AI 补全。
- AI 选 GraphQL 还是 REST 容易给"流行度答案"而非"场景答案":它会默认推荐 GraphQL 因为更新潮,但忽略团队熟悉度和缓存复杂度。技术选型必须人工结合团队现状判断,AI 只能列权衡维度。
