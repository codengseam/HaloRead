# AI时代全栈知识边界·16|核心结构选型

> 前置知识:会用 Python 的 list/dict,知道时间复杂度大 O 记号的基本含义
> 学完你能:① 在数组、链表、哈希表、树、跳表、堆间按场景做出正确选型 ② 说清哈希表冲突解决的两种思路及链地址为何更稳健 ③ 讲清 B+ 树为什么是数据库索引首选 ④ 手写 LRU 缓存、heapq TopK、链表反转 ⑤ 判断哪些结构实现能交给 AI、哪些选型决策必须自己握住

### 一、概念

数据结构(Data Structure)是把数据按特定方式组织起来、以便高效读写的容器与规则的总和。核心结构选型,就是在「读多写少还是写多读少、要有序还是只要查得快、内存里还是磁盘上」这几个维度上,挑出代价最小的那一种结构。一句话定义:数据结构 = 数据的组织形式 + 操作的代价约束;选型的本质是「用空间换时间、用有序换范围、用指针换灵活性」的工程权衡。

需要划清几条边界。第一,「会用数据结构」不等于「会选型」:能背出哈希表 O(1) 查找,不代表知道 Redis 的 Sorted Set 为什么用跳表而不是红黑树。第二,「数据结构」不等于「算法」:结构是容器,算法是在容器上跑的操作;本章只讲结构的代价与选型,排序、最短路径等算法留给独立的《数据结构与算法》专栏。第三,复杂度只是选型的一半,另一半是「缓存局部性、内存布局、实现复杂度」这些工程因素——理论上跳表和红黑树都是 O(log n),但 Redis 选了跳表,因为实现简单、并发友好、范围查询直接。

理解这条边界,下面进入数组与链表、哈希表、树与跳表、堆四条主线,这是真正决定工程选型的内核知识。

### 二、原理

#### 1. 数组与链表:连续与离散的根本取舍

数组(Array)在内存里是一段连续空间,通过「基地址 + 偏移」直接算出任意元素地址,所以随机访问是 O(1)。代价是增删中间元素要搬移后续所有元素,O(n);且大小往往需要预先分配,扩容要整体复制。链表(Linked List)反过来,每个节点存数据加指向下一节点的指针,内存离散分布,插入删除只要改两个指针,O(1);代价是查找要从头遍历,O(n),且指针本身吃内存、缓存不友好。

选型逻辑就藏在这组代价里:

- 数组适合「查多写少、大小确定、内存连续」:查表、矩阵运算、随机访问下标。CPU 缓存预取对连续内存友好,数组遍历比链表快一个数量级,这是「缓存局部性」的收益。
- 链表适合「频繁增删、大小不确定、不要求随机访问」:消息队列中间节点、LRU 缓存的访问链、实现撤销栈。

工程里更常见的是两者的混合体:Python 的 list 底层是动态数组,扩容按倍数预留空间,摊还 O(1) 尾插;Java 的 ArrayList 同理。真要频繁头插,选 collections.deque(分块的双向链表)而不是 list。

#### 2. 栈与队列:操作受限带来的语义清晰

栈(Stack)遵循 LIFO(Last In First Out,后进先出),只在一端进出。队列(Queue)遵循 FIFO(First In First Out,先进先出),一端进另一端出。它们没有数组链表那样的能力扩展,但「受限」本身就是优势——语义明确,不易用错。

栈的典型场景:函数调用栈(每层调用的局部变量、返回地址压栈,return 时弹栈,递归过深就栈溢出)、编辑器撤销(每次操作压栈,Ctrl+Z 弹栈)、表达式求值(用双栈处理运算符优先级)、括号匹配。队列的典型场景:任务调度(先到先得)、消息缓冲(生产者写入队列尾部,消费者从头部取)、BFS(Breadth-First Search,广度优先搜索)逐层入队。

实现上,用数组还是链表?栈用数组(list 的 append/pop)即可,尾操作 O(1);队列若用 list 的 pop(0) 是 O(n),必须用 collections.deque 的 popleft(),才是 O(1)。这是新手最常踩的坑——拿 list 当队列用,数据量一大就慢。

#### 3. 哈希表:O(1) 查找的代价与冲突解决

哈希表(Hash Table)通过哈希函数把 key 映射到数组下标,平均查找、插入、删除都是 O(1)。代价是:无序(不能范围查询)、最坏情况退化为 O(n)、扩容时一次全量 rehash。Python 的 dict、Java 的 HashMap、Redis 的主存储都是哈希表。

哈希表的根本机制是「key → 哈希值 → 桶下标」。当两个不同 key 算出同一个下标,就发生冲突(Collision)。冲突不可避免(鸽巢原理),只能解决。主流解法两种:

- 链地址法(Chaining,拉链法):每个桶挂一条链表,冲突的 key 追加到链表尾部。Java 7 的 HashMap 用它。
- 开放寻址法(Open Addressing):冲突后按某种探测规则(线性探测、二次探测、双重哈希)找下一个空桶。Python dict、Java 8 之后的 HashMap 在部分场景用线性探测变体。

为什么链地址比开放寻址更稳健?关键在「删除」与「负载因子」两件事。第一,开放寻址删除元素不能真删,否则会打断探测链,后续查找会找不到本应顺着探测链找到的元素,只能打「墓碑」标记,墓碑一多查找效率崩塌;链地址删除直接摘节点,干净利落。第二,开放寻址在负载因子(load factor,元素数/桶数)超过 0.7 后冲突概率急升,性能急剧恶化,必须频繁扩容;链地址负载因子到 1 甚至更高仍能工作,链表虽长但可用。第三,开放寻址对哈希函数质量极敏感,函数稍差就会聚集(cluster),链地址有链表兜底容错更高。代价是链地址要额外存指针,内存开销略大。

Java 8 的折中:链地址法的链表长度超过 8 时转红黑树,把最坏 O(n) 降到 O(log n),防的就是「构造恶意 key 把哈希表打成链表」的 DoS(Denial of Service,拒绝服务)攻击。Python dict 则坚持开放寻址,靠高质量的哈希函数和高频 rehash 控制冲突,因为 dict 在 Python 里无处不在,内存紧凑比最坏情况防护更重要。

#### 4. 树:从二叉搜索树到 B+ 树

二叉搜索树(Binary Search Tree,BST):左子树都小于根,右子树都大于根,中序遍历得有序序列。查找、插入、删除平均 O(log n),但插入顺序决定形状——依次插入 1、2、3、4 会退化成链表,最坏 O(n)。平衡树就是为解决退化而生:AVL 树严格平衡(左右子树高度差不超过 1),查找快但增删旋转多;红黑树(Red-Black Tree)弱平衡(最长路径不超过最短两倍),增删旋转少,适合频繁修改,所以 C++ STL 的 map、Java 的 TreeMap、Linux 内核调度都用红黑树。

B+ 树(B+ Tree)是数据库索引的标准结构,它的两条核心设计决定了这一点:第一,所有数据只存在叶子节点,内部节点只存索引 key;第二,叶子节点用双向链表串联。这两条让 B+ 树天生适合磁盘。

为什么是 B+ 树而不是红黑树或跳表?算一笔账。磁盘 IO 是数据库性能的根本瓶颈,树高每多一层多一次随机 IO。红黑树是二叉树,存一亿行数据树高约 27 层(log₂(10⁸)≈26.6),意味着 27 次随机 IO。B+ 树每个内部节点扇出极大——InnoDB 默认页 16KB,主键 BIGINT 8 字节加 6 字节子指针,一个内部节点放约 1170 个 key,三层 B+ 树能存 1170×1170×16≈2190 万行,四层 250 亿行。从根到叶 3-4 次随机 IO,配合 Buffer Pool 缓存内部节点,实际查询常常只需一次磁盘 IO。红黑树做不到这个扇出。跳表虽然实现简单,但局部性差,磁盘随机 IO 多,Redis 的 Sorted Set 用跳表是因为它在内存里,关系库面向磁盘,B+ 树更优。叶子链表让范围查询变成顺序扫描,`where id between 100 and 200` 定位到 100 后沿链表扫到 200 即可,这是 B+ 树成为关系库索引标配的根本机制。

#### 5. 图:邻接矩阵与邻接表的取舍

图(Graph)由顶点和边组成,存储方式两种:邻接矩阵用 V×V 的二维数组存边,查「两点是否相邻」O(1),但空间 O(V²),稀疏图(边远少于 V²)浪费严重;邻接表每个顶点挂一条链表存相邻顶点,空间 O(V+E),适合稀疏图,查相邻要遍历链表。

选型看图的密度:稠密图(边接近 V²)用邻接矩阵,稀疏图(如社交网络、网页链接)用邻接表。工程里绝大多数真实图都是稀疏的——社交关系几十亿顶点,平均每人几百好友,边数远小于顶点数平方,只能用邻接表。图算法(最短路径 Dijkstra、拓扑排序、并查集)是独立的大主题,本章只点出存储选型,算法实现细节交给后续专栏。

#### 6. 跳表与堆:有序集合与优先队列

跳表(Skip List)通过在有序链表上加多层索引,实现 O(log n) 查找、插入、删除,且实现远比红黑树简单。Redis 的 Sorted Set(有序集合)用跳表,原因是:第一,跳表范围查询天然友好,定位到起点后沿最底层链表顺序扫即可,正是 ZRANGEBYSCORE 的需求;第二,跳表增删只改相邻节点的指针,并发控制比红黑树的旋转简单,适合 Redis 的单线程模型;第三,实现简单,出 bug 概率低。这是「工程选型不只看复杂度」的经典案例——红黑树和跳表都是 O(log n),但工程代价不同。

堆(Heap)是满足堆序性的完全二叉树:大顶堆每个节点都大于等于子节点。堆的核心价值是优先队列和 TopK 问题。优先队列:任务调度按优先级出队、Dijkstra 按距离取最近点,都用堆。TopK:从一亿个数里取最大的 10 个,排序要 O(n log n),用大小为 K 的小顶堆只要 O(n log K),K 远小于 n 时快一个量级。Python 的 heapq 模块就是小顶堆实现。

### 三、实践

#### 1. 手写 LRU 缓存

LRU(Least Recently Used,最近最少使用)缓存淘汰策略:容量满时淘汰最久没访问的元素。要求 get/put 都是 O(1)。实现思路是「哈希表 + 双向链表」:哈希表存 key 到节点的映射,O(1) 定位;双向链表按访问顺序排列,访问过的移到头部,淘汰尾部。Python 用 OrderedDict 三行就能写,但理解原理得手写一遍。

```python
# 哈希表 + 双向链表实现的 LRU,O(1) get/put
class Node:
    def __init__(self, key=0, val=0):
        self.key = key
        self.val = val
        self.prev = None
        self.next = None

class LRUCache:
    def __init__(self, capacity: int):
        self.cap = capacity
        self.cache = {}  # key -> Node
        # 哑节点,避免处理头尾边界
        self.head = Node()
        self.tail = Node()
        self.head.next = self.tail
        self.tail.prev = self.head

    def _remove(self, node: Node):
        """从链表摘除节点"""
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_to_head(self, node: Node):
        """节点插到头部(最近使用)"""
        node.next = self.head.next
        node.prev = self.head
        self.head.next.prev = node
        self.head.next = node

    def get(self, key: int) -> int:
        if key not in self.cache:
            return -1
        node = self.cache[key]
        self._remove(node)      # 访问即提升到头部
        self._add_to_head(node)
        return node.val

    def put(self, key: int, value: int) -> None:
        if key in self.cache:
            node = self.cache[key]
            node.val = value
            self._remove(node)
            self._add_to_head(node)
            return
        if len(self.cache) >= self.cap:
            # 淘汰尾部节点(最久未使用)
            lru = self.tail.prev
            self._remove(lru)
            del self.cache[lru.key]
        node = Node(key, value)
        self.cache[key] = node
        self._add_to_head(node)
```

几个实现要点:用头尾哑节点(Dummy Head/Tail)免去判空,所有节点都夹在两个哑节点之间,_remove 和 _add_to_head 不用处理边界;淘汰时从 tail.prev 取,因为 tail 本身是哑节点;双向链表让摘除节点 O(1),单向链表摘除要遍历找前驱退化成 O(n)。Python 标准库的等价写法更简洁:

```python
# OrderedDict 实现,等价但更简洁;move_to_end 维护访问顺序
from collections import OrderedDict

class LRUCacheOD:
    def __init__(self, capacity: int):
        self.cap = capacity
        self.od = OrderedDict()

    def get(self, key: int) -> int:
        if key not in self.od:
            return -1
        self.od.move_to_end(key)  # 移到末尾表示最近使用
        return self.od[key]

    def put(self, key: int, value: int) -> None:
        if key in self.od:
            self.od.move_to_end(key)
        self.od[key] = value
        if len(self.od) > self.cap:
            self.od.popitem(last=False)  # 弹出头部(最久未使用)
```

生产环境直接用 functools.lru_cache 装饰器或 Redis,这里手写是为了理解「哈希表 + 双向链表」的组合:哈希表负责 O(1) 定位,链表负责 O(1) 维护顺序。

#### 2. heapq 实现 TopK

从海量数据里取最大的 K 个,用小顶堆维护一个大小为 K 的堆:遍历每个元素,比堆顶大就替换堆顶再下沉。堆大小始终 K,单次操作 O(log K),总复杂度 O(n log K)。

```python
import heapq

def topk(nums: list[int], k: int) -> list[int]:
    """返回 nums 中最大的 k 个数(降序)"""
    if k <= 0:
        return []
    heap = nums[:k]
    heapq.heapify(heap)  # O(k) 建堆
    for x in nums[k:]:
        # 比堆顶(当前最小)大,就替换
        if x > heap[0]:
            heapq.heapreplace(heap, x)  # 弹出最小并压入 x,O(log k)
    return sorted(heap, reverse=True)  # 输出降序

# 流式场景:数据不断来,实时维护 TopK
class StreamingTopK:
    def __init__(self, k: int):
        self.k = k
        self.heap = []

    def add(self, x: int):
        if len(self.heap) < self.k:
            heapq.heappush(self.heap, x)
        elif x > self.heap[0]:
            heapq.heapreplace(self.heap, x)

    def topk(self) -> list[int]:
        return sorted(self.heap, reverse=True)
```

注意 heapq 是小顶堆,要取「最大的 K 个」就用小顶堆存当前候选;要取「最小的 K 个」就把元素取负塞进去,或用 nlargest/nsmallest 封装。直接 `heapq.nlargest(k, nums)` 等价于上面的手写实现,且内部对大数据集会优化成分块加堆归并,工程里优先用它。

#### 3. 链表反转

链表反转是面试高频题,也是理解指针操作的试金石。迭代法用三个指针逐个翻转,空间 O(1):

```python
class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

def reverse_list(head: ListNode) -> ListNode:
    """迭代反转,三指针逐步翻转,O(n) 时间 O(1) 空间"""
    prev = None
    curr = head
    while curr:
        nxt = curr.next  # 暂存下一节点
        curr.next = prev  # 翻转指向
        prev = curr       # prev 前进
        curr = nxt        # curr 前进
    return prev  # 新头

def reverse_list_recursive(head: ListNode) -> ListNode:
    """递归反转,O(n) 时间 O(n) 空间(递归栈)"""
    if not head or not head.next:
        return head
    new_head = reverse_list_recursive(head.next)
    head.next.next = head  # 后继指回自己
    head.next = None       # 断开原方向
    return new_head
```

迭代法的关键是「暂存 next 再翻转」,顺序不能错:先存 nxt,再改 curr.next,最后移动 prev 和 curr。递归法利用「反转后面的链表后,head.next 恰好是反转后链表的尾节点,让它指回 head」。生产代码优先用迭代,递归在万级节点以上会栈溢出。

### 四、速查/自测

#### 核心结构复杂度与场景对照表

| 结构 | 查找 | 插入 | 删除 | 有序 | 典型场景 |
|---|---|---|---|---|---|
| 数组(下标访问) | O(1) | O(n) | O(n) | 否 | 查表、矩阵、下标访问 |
| 链表(给节点) | O(n) | O(1) | O(1) | 否 | 频繁增删、LRU 链 |
| 栈 | O(n) | O(1) | O(1) | 否 | 调用栈、撤销、括号匹配 |
| 队列 | O(n) | O(1) | O(1) | 否 | 任务调度、消息缓冲、BFS |
| 哈希表 | O(1)均 | O(1)均 | O(1)均 | 否 | 缓存、字典、去重 |
| 二叉搜索树(平衡) | O(log n) | O(log n) | O(log n) | 是 | 有序字典、动态维护 |
| 红黑树 | O(log n) | O(log n) | O(log n) | 是 | STL map、内核调度 |
| B+ 树 | O(log n) | O(log n) | O(log n) | 是(范围友好) | 数据库索引 |
| 跳表 | O(log n) | O(log n) | O(log n) | 是(范围友好) | Redis ZSet |
| 堆 | O(1)堆顶 | O(log n) | O(log n) | 否 | 优先队列、TopK |
| 图(邻接表) | O(V+E) | O(1) | O(V) | 否 | 社交网络、路由 |

注:「均」指平均;链表插入删除 O(1) 的前提是已拿到目标节点,定位本身是 O(n)。

#### 自测题

1. **原理层**:为什么链地址法比开放寻址法更稳健?至少说出两点。
   参考答案:第一,开放寻址删除元素不能真删,只能打墓碑标记,否则打断探测链导致后续查找失败,墓碑堆积后查找效率崩塌;链地址直接摘节点,删除干净。第二,开放寻址负载因子超过 0.7 后冲突概率急升、性能恶化,必须频繁扩容;链地址负载因子到 1 甚至更高仍可用,链表虽长但功能正常。第三,开放寻址对哈希函数质量极敏感,函数稍差就聚集;链地址有链表兜底容错更高。

2. **思路层**:Redis 的 Sorted Set 为什么用跳表而不是红黑树?两者复杂度都是 O(log n)。
   参考答案:三方面。其一,跳表范围查询天然友好,定位到起点后沿最底层链表顺序扫描即可,正是 ZRANGEBYSCORE 的需求,红黑树范围查询要中序遍历实现更复杂。其二,跳表增删只改相邻节点指针,并发控制比红黑树的旋转简单,适合 Redis 单线程模型。其三,实现简单、出 bug 概率低。这是「工程选型不只看时间复杂度」的典型。

3. **实践层**:用 Python list 实现 LRU 缓存有什么问题?
   参考答案:定位 key 要 list.index() 是 O(n);把访问元素移到头部要 pop 加 insert,也是 O(n)。get/put 都退化成 O(n),违背 LRU 的 O(1) 要求。正确做法是哈希表加双向链表,或直接用 OrderedDict。

4. **原理层**:为什么 B+ 树 3-4 层就能存亿级数据,而红黑树做不到?
   参考答案:红黑树是二叉树,扇出固定为 2,存一亿行树高约 27 层,27 次随机 IO。B+ 树内部节点不存数据只存 key,扇出极大——InnoDB 16KB 页约 1170 key,三层存 1170×1170×16≈2190 万行,四层 250 亿行,3-4 次 IO 搞定。配合 Buffer Pool 缓存内部节点,实际查询常常只需一次磁盘 IO。

5. **思路层**:从一亿个数里取最大的 10 个,为什么用小顶堆而不是排序?
   参考答案:全排序 O(n log n),一亿个要排序上亿次比较。小顶堆维护大小 10 的堆,遍历每个元素比堆顶大就替换,O(n log 10)≈O(n),快一个量级。堆大小 K 远小于 n 时优势明显,且流式数据可实时维护,排序必须等全部到齐。

### 可交给 AI 的部分

能放心交给 AI 的,是「结构确定、可运行验证、错了能跑出来」的活儿:经典结构的完整实现(红黑树插入删除、AVL 旋转、图的最短路径、并查集带路径压缩)、LeetCode 风格的链表/树/堆题目代码、标准库调用的样板(用 heapq 写 TopK、用 OrderedDict 写 LRU)、复杂度分析的初稿。这类代码正确与否一组测试用例就能验,影响半径局限在单次调用,适合让 AI 出初稿、人工跑测试用例把关。

必须自己握住的,是「选型决策、复杂度预判、缓存局部性、内存与磁盘的差别」四件事。第一,AI 不会替你选型——你问它「用什么结构存这批数据」,它常默认 dict 或 list,不会主动问「读多写少吗、要有序吗、内存还是磁盘」,而选型恰恰由这些工程约束决定。第二,AI 写的代码常漏复杂度陷阱:用 list 当队列(pop(0) 是 O(n))、用 dict 存要排序的数据再排序、用红黑树存只查等值不要求有序的小数据集,这些局部看没错、数据量一大就慢的问题,AI 不会主动提醒。第三,缓存局部性是 AI 几乎不考虑的维度——同样 O(n) 遍历,数组比链表快一个量级,因为 CPU 缓存行预取连续内存,AI 不会替你算这笔账。第四,内存结构与磁盘结构的代价完全不同,内存里跳表红黑树差不多,磁盘上 B+ 树碾压一切,这是数据库选型的根本,AI 没有上下文知道你的数据在内存还是磁盘。

风险提示具体到三条:其一,AI 手写红黑树极易写错旋转与着色逻辑,这类代码必须用一套覆盖插入、删除、查找的测试用例验证,别只看能跑就交;其二,AI 实现的哈希表常忽略冲突处理与扩容,小数据能跑大数据就退化成 O(n),务必检查它是否处理了负载因子与 rehash;其三,AI 推荐结构时常基于「教科书最优」而非「工程最优」,例如让你用红黑树实现 LRU(正确应是哈希表加双向链表)、用数组实现频繁头插的队列(正确应用 deque)。判断标准很简单:涉及「选什么结构、复杂度会不会退化、数据在内存还是磁盘」这三类隐式决策时,自己过一遍原理;涉及「结构的标准实现、题目代码、标准库样板」时,大胆交给 AI 出初稿再跑测试把关。
