# AI时代全栈知识边界·13|关系库与索引原理

> 前置知识:能写出 select/insert/update/delete 四件套的基本 SQL,知道什么是表、什么是主键
> 学完你能:① 读懂 EXPLAIN 输出并判断索引是否命中 ② 说清 B+ 树为什么适合磁盘、聚簇索引为什么查询快 ③ 识别函数操作、隐式转换、最左前缀违反等索引失效场景 ④ 在范式与反范式之间做出工程取舍 ⑤ 判断哪些 SQL 活儿能交给 AI、哪些必须自己握住

### 一、概念

关系库(Relational Database,RDBMS)是把数据组织成「关系(二维表)」、用关系代数做查询的存储系统,核心承诺是表结构 + 约束 + 事务。索引(Index)是建在表上一列或多列上的辅助数据结构,目的是把全表扫描的 O(n) 查找降到 O(log n) 甚至 O(1)。一句话定义:关系库 = 表 + 关系约束 + ACID(Atomicity Consistency Isolation Durability,原子性一致性隔离性持久性)事务;索引 = 让查找不退化成全表扫描的有序结构。

需要划清几条边界。第一,「关系库」不等于「用 SQL 的数据库」:SQL(Structured Query Language,结构化查询语言)是查询语言,关系模型是数据模型;有 SQL 接口的不一定是关系库(例如 Hive 是 SQL on Hadoop,不保证 ACID)。第二,「索引」不是越多越好:每加一个索引,写操作就要多维护一棵 B+ 树,还要占磁盘与内存。第三,本章只讲关系库的存储结构与索引原理,事务、锁、隔离级别留给后续章节展开。

理解了这条边界,下面进入范式、聚簇索引、B+ 树三条主线,这是真正决定关系库工程决策的内核知识。

### 二、原理

#### 1. 范式与反范式的工程取舍

范式(Normal Form)是关系库设计的一组规则,目的是消除数据冗余与更新异常。最常用的四档:

- 1NF(第一范式):每个字段原子不可分。不要在「姓名」字段里塞「张三,李四」,也不要把「省市区」挤成一列。
- 2NF(第二范式):在 1NF 基础上,非主键字段必须完全依赖主键,不能部分依赖。这条主要针对联合主键——订单明细表若以 (order_id, product_id) 为联合主键,商品名称只依赖 product_id 而不依赖 order_id,就该拆到商品表,否则同一个商品名会被存多份,改一次要改千百行。
- 3NF(第三范式):在 2NF 基础上,非主键字段必须直接依赖主键,不能传递依赖。订单表里有 user_id 和 user_city,user_city 依赖 user_id,user_id 依赖订单主键,这就是传递依赖,该把 user_city 拆到用户表。
- BCNF(Boyce-Codd Normal Form):在 3NF 基础上,每个决定因素都必须是候选键。比 3NF 更严格,实际工程里很少强求到这一档。

范式的好处是冗余少、更新一致;代价是查询要 JOIN 多张表,读放大严重。反范式(Denormalization)是反其道而行:为了查询性能,故意冗余字段,用空间换时间、用写换读。典型场景是订单表冗余一份商品快照(名称、单价),这样查历史订单不用 JOIN 商品表,且商品改名不影响历史订单的展示。

取舍原则:写多读少、强一致要求高,优先范式;读多写少、查询性能吃紧,适度反范式。互联网业务绝大多数走「适度反范式 + 应用层缓存」,纯三范式的库在 OLTP(Online Transaction Processing,联机事务处理)场景里几乎不存在。

#### 2. 聚簇索引 vs 非聚簇索引:索引即数据

聚簇索引(Clustered Index)的本质是:索引的叶子节点就是数据行本身,数据行按主键顺序物理存储。一张表只能有一个聚簇索引,因为数据只能按一种顺序物理排列。InnoDB 的主键索引就是聚簇索引,这是「索引即数据」的由来——找主键等于直接定位到行,不需要第二次查找。

非聚簇索引(Secondary Index,二级索引)的叶子节点存的是主键值,而不是完整行。查到主键后,还要拿着主键回到聚簇索引里再查一次完整数据,这步叫回表(Table Lookup by PK)。理解回表,就能解释为什么「select *」常比「select 覆盖列」慢:前者大概率要回表,后者可能走覆盖索引(Covering Index)直接从二级索引返回,连回表都省了。

跨版本差异:MySQL 的 InnoDB 一直是聚簇索引,MyISAM 是非聚簇(索引和数据分离存储);PostgreSQL 没有聚簇索引概念,所有索引都是二级索引,数据存在堆表(heap)里,索引指向行的物理位置(ctid)。这意味着 PostgreSQL 的主键查询比 InnoDB 多一次堆表查找,但换来的是主键可灵活更换、表无物理排序约束。

#### 3. B+ 树为什么是数据库索引的标准结构

B+ 树(B+ Tree)是 B 树的变体,核心区别有两条:第一,所有数据只存在叶子节点,内部节点只存索引 key;第二,所有叶子节点用双向链表串联。这两条决定了 B+ 树天生适合磁盘存储。

逐个对比四种候选结构:

- B 树(B Tree):每个节点都存数据,导致一个节点能放的 key 少,树更高,磁盘 IO 更多。磁盘 IO 是数据库性能的根本瓶颈,树高每多一层,多一次随机 IO。
- Hash 索引:O(1) 等值查找极快,但不支持范围查询、不支持排序、不支持最左前缀。等值场景才用,如 Memory 引擎和 Redis。
- 跳表(Skip List):实现简单、内存友好,但局部性差,磁盘随机 IO 多。Redis 的 Sorted Set 用跳表是因为它在内存里;关系库面向磁盘,B+ 树更优。
- B+ 树:内部节点不存数据,扇出(fan-out)极大;叶子链表让范围查询变成顺序扫描。

为什么 B+ 树 3-4 层就能存亿级数据?算一笔账:InnoDB 默认页大小 16KB,假设主键是 BIGINT(8 字节),内部节点每个 key 配一个 6 字节子指针,一个内部节点能放约 1170 个 key(16384 / 14);叶子节点假设每行 1KB,一页放 16 行。三层 B+ 树能存 1170 × 1170 × 16 ≈ 2190 万行,四层能存 250 亿行。从根到叶三次随机 IO,配合 Buffer Pool 缓存内部节点,实际查询常常只需一次磁盘 IO。这是 B+ 树成为关系库标配的根本机制。

叶子节点的双向链表让范围查询友好:`where id between 100 and 200` 定位到 id=100 的叶子后,沿链表顺序扫到 id=200 即可,不用反复从根查找。这也是为什么主键自增的表插入快、范围查快——新行总追加到链表末尾,页分裂少,范围扫描连续。

#### 4. 索引为什么失效:函数操作破坏有序性

B+ 树是按字段原值排序的有序结构。一旦查询条件变成了 `where func(col) = x`,索引里存的是 col 原值,func(col) 不再保持原序,B+ 树的二分定位失效,优化器只能退化成全表扫描。这就是函数操作导致索引失效的核心原理。

同理可推:隐式类型转换的本质也是函数操作——字符串字段 `where col = 123` 会被 MySQL 转成 `where CAST(col AS SIGNED) = 123`,等价于在 col 上套了函数,索引失效。最左前缀违反的本质类似:联合索引 (a,b,c) 是先按 a 排序、a 相同按 b、b 相同按 c,跳过 a 直接查 b,等于在一组无序的 b 值里找,索引帮不上。`like '%xxx'` 左侧通配符同理,无法用前缀定位。

记住一句话:任何让索引字段「不再以原值、原序参与比较」的写法,都会让 B+ 树失效。

#### 5. SQL 执行顺序与慢 SQL 成因

写 SQL 的人常按语法书写顺序理解执行,但数据库的真实执行顺序是另一回事:

```
FROM/JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
```

这个顺序解释了几个常见困惑:为什么 `where` 里不能用 `select` 起的别名(因为 select 在 where 之后执行,别名还没生成);为什么 `having` 能用聚合函数而 `where` 不能(where 在分组前执行,还没有聚合结果);为什么 `order by` 里能用别名(select 已执行完,别名可用)。

慢 SQL 的成因主要三类,都对应 EXPLAIN 输出的 Extra 列:

- 全表扫描(type=ALL):没用上索引,逐行扫描整张表。千万行表上一次扫描就是秒级延迟。
- 临时表(Using temporary):GROUP BY、DISTINCT、UNION 等需要中间结果时,优化器建临时表(内存或磁盘),数据量大时落盘变慢。
- 文件排序(Using filesort):ORDER BY 的列不在索引里,无法用索引顺序,要在内存或磁盘单独排序。回表 + filesort 是慢 SQL 的最常见组合。

#### 6. 分库分表与读写分离的基本逻辑

单库扛不住的根因有三:数据量太大(单表数亿行,B+ 树层数上升、内存命中率下降)、写入太高(单机 IO 打满)、单点故障。解法分两层。

第一层是读写分离:主库(master)承担写,从库(replica)承担读,主从通过 binlog 复制同步。代价是复制延迟——刚写入的数据,从库可能还没同步到,读从库会读到旧值。业务上常见处理是「写后立即读主库」或「关键路径强一致读主」。

第二层是分库分表:垂直拆分按业务把不同表分到不同库(用户库、订单库、商品库),水平拆分按某个分片键(shard key)把一张表拆成多张,如按 user_id 取模拆成 64 张表。分片键选错是分库分表最大的坑:选 user_id,查某用户订单很快,但查某商户的全部订单要扫所有分片(跨片查询)。一旦分了,JOIN、事务、唯一约束都受影响,所以分库分表是「能不分就不分,分了就回不去」的决策,通常在单表过千万行、写入扛不住时才动用。

#### 7. SQL 注入与参数化查询

SQL 注入(Injection)的本质是:把用户输入当作 SQL 代码的一部分拼接执行。用户在登录框输入 `' OR '1'='1`,拼接后变成 `where password='' OR '1'='1'`,恒真就绕过了密码校验。根因是「数据与代码未分离」。

防御的标准做法是参数化查询(Parameterized Query):SQL 语句先用占位符 `?` 写好结构,再把用户输入作为参数传入。数据库驱动保证参数只被当作数据值,永远不会被解析成 SQL 语法。这是「数据与代码分离」的根本机制,所有 ORM 默认走参数化。

```python
# 错误:字符串拼接,有注入风险
cursor.execute(f"SELECT * FROM users WHERE name = '{name}'")

# 正确:参数化查询,MySQL 8.0 / PostgreSQL 14+ 通用
cursor.execute("SELECT * FROM users WHERE name = %s", (name,))
```

### 三、实践

#### 1. 建表 + 建索引 SQL

下面这张订单表演示了主键、唯一索引、联合索引的典型写法,字段选择贴近真实电商场景。

```sql
-- MySQL 8.0+ / InnoDB 引擎
CREATE TABLE orders (
  id          BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '订单ID',
  order_no    VARCHAR(32)     NOT NULL COMMENT '业务订单号',
  user_id     BIGINT UNSIGNED NOT NULL COMMENT '用户ID',
  merchant_id BIGINT UNSIGNED NOT NULL COMMENT '商户ID',
  amount      DECIMAL(10,2)   NOT NULL COMMENT '金额',
  status      TINYINT         NOT NULL DEFAULT 0 COMMENT '0待付 1已付 2已发 3已收',
  created_at  DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  UNIQUE KEY uk_order_no (order_no),
  KEY idx_user_status_created (user_id, status, created_at),
  KEY idx_merchant_created (merchant_id, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='订单表';
```

几个设计要点:主键用自增 BIGINT,保证 B+ 树顺序插入不触发页分裂;order_no 是业务订单号建唯一索引,避免外部暴露自增 ID;联合索引 `(user_id, status, created_at)` 服务于「查某用户某状态订单按时间倒序」这一高频查询,字段顺序遵循最左前缀 + 区分度从高到低;商户维度单独建 `(merchant_id, created_at)`,因为商户查订单与用户查订单是两条独立路径,共用索引会互相拖累。

#### 2. EXPLAIN 输出与解读

```sql
EXPLAIN SELECT * FROM orders
WHERE user_id = 10086 AND status = 1
ORDER BY created_at DESC LIMIT 20;
```

典型输出:

| id | select_type | table  | type | possible_keys          | key                    | key_len | ref         | rows | Extra                    |
|----|-------------|--------|------|------------------------|------------------------|---------|-------------|------|---------------------------|
| 1  | SIMPLE      | orders | ref  | idx_user_status_created| idx_user_status_created| 9       | const,const | 50   | Using index condition     |

逐列解读:

- `type=ref`:走非唯一索引的等值查找,属于较好档位。type 从好到差依次是 `system > const > eq_ref > ref > range > index > ALL`,出现 ALL 就要警惕全表扫描。
- `key=idx_user_status_created`:实际选用的索引。若为 NULL 说明没用索引。
- `key_len=9`:用到的索引字段长度(user_id 8 字节 + status 1 字节),据此可判断联合索引用到了几个字段。
- `rows=50`:优化器预估要扫描的行数,越小越好。
- `Extra=Using index condition`:用了索引下推(Index Condition Pushdown,ICP,MySQL 5.6+),在存储引擎层就过滤,减少回表次数。若出现 `Using filesort` 或 `Using temporary`,就要重点优化排序与分组。

#### 3. 索引失效 vs 命中对照查询

同一个联合索引 `idx_user_status_created (user_id, status, created_at)`,下面五组查询直观对照命中与失效:

```sql
-- 【命中】最左前缀 + 等值,且字段顺序对齐
SELECT * FROM orders WHERE user_id = 10086 AND status = 1;
-- type=ref,key=idx_user_status_created,走索引

-- 【失效一】函数操作:对索引列套函数
SELECT * FROM orders WHERE DATE(created_at) = '2026-06-30';
-- type=ALL,Using where,全表扫描
-- 改写为范围查询即可命中:
SELECT * FROM orders
WHERE created_at >= '2026-06-30' AND created_at < '2026-07-01';

-- 【失效二】隐式类型转换:order_no 是 VARCHAR,传数字
SELECT * FROM orders WHERE order_no = 20260630001;
-- type=ALL,全表扫描(MySQL 把字符串转数字比较,等价于套了 CAST 函数)
-- 改成字符串字面量即可命中唯一索引:
SELECT * FROM orders WHERE order_no = '20260630001';
-- type=const

-- 【失效三】最左前缀违反:跳过 user_id 直接查 status
SELECT * FROM orders WHERE status = 1;
-- type=ALL,全表扫描

-- 【失效四】LIKE 左模糊
SELECT * FROM orders WHERE order_no LIKE '%0630';
-- type=ALL,左侧通配符无法用 B+ 树前缀定位
-- 改成右模糊 '202606%' 才命中

-- 【失效五】OR 一边无索引
SELECT * FROM orders WHERE user_id = 10086 OR amount > 100;
-- type=ALL,因 amount 列无索引,优化器放弃索引,全表扫描
-- 两边都建索引时才可能走 index merge
```

规律很明显:失效写法能跑出结果,但 EXPLAIN 一跑就露馅——`type=ALL`、`key=NULL`、`rows` 接近全表行数。排查慢 SQL 第一步永远是 EXPLAIN。

### 四、速查/自测

#### 索引失效场景速查表

| 写法 | 是否命中索引 | 根本原因 |
|---|---|---|
| `where func(col) = x`(函数操作) | 失效 | func(col) 破坏 B+ 树原序,无法二分定位 |
| 字符串列 `where col = 123`(隐式转换) | 失效 | 等价于 `CAST(col AS ...)`,变成函数操作 |
| 联合索引跳过左列查询 | 失效 | 最左前缀违反,跳过左列后剩余列无序 |
| `where a = 1 or b > 10`(b 无索引) | 失效 | OR 两端必须都能走索引,否则全表 |
| `where col != 1` / `NOT IN` | 通常失效 | 范围过大,优化器判全表更快 |
| `where col like '%xxx'`(左模糊) | 失效 | 前缀无法定位 |
| `where col like 'xxx%'`(右模糊) | 命中 | 前缀匹配,可用 B+ 树范围查找 |
| `where a = 1 and b = 2 and c = 3` | 命中 | 联合索引完整最左前缀 |

#### 自测题

1. **原理层**:为什么 InnoDB 的主键索引叫聚簇索引?一张表为什么只能有一个聚簇索引?
   参考答案:聚簇索引的叶子节点就是数据行本身,数据按主键顺序物理存储,「索引即数据」。一张表的数据只能按一种顺序物理排列,所以聚簇索引唯一。InnoDB 没显式主键时会选第一个 NOT NULL 唯一索引,都没有就生成隐藏 6 字节 ROWID 作聚簇键。

2. **思路层**:联合索引 `(a,b,c)`,查询 `where a=1 and c=3` 走不走索引?用到几个字段?
   参考答案:走索引,但只用到了 a 字段(最左前缀)。c 在 b 缺失时无法参与索引定位,但 MySQL 5.6+ 的索引下推(ICP)会在存储引擎层用 c 过滤,减少回表。`key_len` 只反映 a 的长度。

3. **实践层**:慢 SQL 报告显示 `Using filesort`,你怎么排查?
   参考答案:看 ORDER BY 的列是否在索引里。若不在,加上包含排序列的联合索引,让排序走索引顺序;若已 in index 但仍 filesort,检查是否因 `select *` 触发回表打断了索引顺序扫描,改用覆盖索引。大表 filesort 落盘代价极高,优先消除。

4. **思路层**:订单表高频查询有两种——「某用户最近订单」和「某商户最近订单」,索引怎么建?
   参考答案:各建一条联合索引:`(user_id, created_at)` 与 `(merchant_id, created_at)`。两条索引各自服务一条查询路径,不要合并成单索引,因为合并后总有查询违反最左前缀。代价是写时要维护两棵 B+ 树,但读收益远大于写成本。

5. **原理层**:为什么 B+ 树 3-4 层就能存亿级数据?
   参考答案:内部节点不存数据,扇出极大(16KB 页约 1170 key);叶子节点存数据,每页约 16 行。三层 1170 × 1170 × 16 ≈ 2190 万,四层 250 亿。配合 Buffer Pool 缓存内部节点,根与中间层常驻内存,实际查询往往只需一次磁盘 IO 访问叶子层。

### 可交给 AI 的部分

能放心交给 AI 的,是「语法确定、可执行验证、错了能跑出来」的活儿:复杂多表联查 SQL、存储过程(Stored Procedure)、完整索引创建语句、报表 SQL、CTE(Common Table Expression,通用表表达式)与窗口函数写法、跨库迁移的方言转换。这类代码正确与否一条 SQL 就能验,影响半径局限在单条查询,适合让 AI 出初稿、人工 review 语法与字段。

必须自己握住的,是「索引设计、JOIN 正确性、慢查询定位、分片策略」四件事。第一,AI 写的查询常常漏索引——它只管 SQL 能跑出结果,不会主动检查这条查询走了哪个索引、扫了多少行,生产上跑成全表扫描它也不会提醒。第二,AI 写的 JOIN 可能产生笛卡尔积(Cartesian Product):漏写 ON 条件、关联键写错、JOIN 类型选错(该 LEFT 写成 INNER 丢数据、该 INNER 写成 LEFT 放大行数),行数悄悄爆炸,看着 SQL 没报错实则结果错。第三,慢查询定位必须人来做——AI 不会主动跑 EXPLAIN,要你来跑、来判断 type/rows/Extra,这是「能不能上线」的最后一道关。第四,分库分表的分片键选择是业务决策,涉及跨片查询代价与扩容难度,AI 没有业务上下文,不能替你决定。

风险提示具体到三条:其一,AI 生成的索引语句可能冗余或漏建——它常给每个 WHERE 字段单建索引,忽略联合索引的最左前缀复用;其二,AI 写的迁移脚本可能直接 `ALTER TABLE ADD INDEX`,大表上会长时间锁表,生产必须用 `pt-online-schema-change` 或 MySQL 8.0 的 INSTANT/online DDL;其三,AI 写的分页 SQL 常用 `LIMIT 1000000, 20`,深分页时 MySQL 要先扫 100 万行再丢弃,正确写法是用游标分页(`where id > last_id order by id limit 20`)。判断标准很简单:涉及「走不走索引、JOIN 行数对不对、能不能上线、分片键选什么」这四类隐式决策时,自己过一遍原理;涉及「SQL 语法、报表写法、迁移方言」时,大胆交给 AI 出初稿再人工把关。
