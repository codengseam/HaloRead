# AI时代全栈知识边界·02|知识边界的第一性原理

> 前置知识:读过本专栏开篇,了解"必须掌握 vs 可交给 AI"是 AI 时代全栈开发的核心命题
> 学完你能:①用一句话定义"知识边界"并划清"必须掌握"与"可交给 AI"的分野 ②用三条判据(识错/选型/定位)对任意知识点分类 ③用决策树快速判断一项知识该不该深学 ④理解边界随工具进化而移动的规律,动态调整学习重心 ⑤避开"全交给 AI"和"什么都自己学"两个极端

## 一、概念

### 1. 知识边界:你能在多大程度上对 AI 产出负责

一句话定义:**知识边界,是一个开发者在不依赖 AI 的情况下,仍能对系统正确性负责的最小知识集合**。边界之内是"必须掌握",边界之外是"可交给 AI"。

这个定义刻意没把"知识边界"等同于"你会多少"。会得多不代表该自己动手——会写 SQL 不代表每次都该手写;反过来,不会也不代表该全甩给 AI——不会看 EXPLAIN 又把所有 SQL 交给 AI 生成,等于把数据库性能交给运气。

划清三处容易混淆的边界:

- 知识边界不是"你的能力上限",而是"你能否兜底的下限"。上限决定你能做多难的事,下限决定 AI 出错时你能不能接住。
- 知识边界不是"AI 能做多少",而是"AI 做了之后你能不能验证"。AI 能写协程代码,但你能看出它有没有泄漏 event loop,这才是边界。
- 知识边界不是"静态的清单",而是"随工具进化动态移动的线"。今天必须死记的 API,明天 IDE 自动补全加静态检查后可能就退出必须掌握之列。

本专栏反复出现的一对术语先给中英对照:First Principles(第一性原理),指回到事物最根本的约束来推导结论,而不是类比已有做法;vibe coding(氛围编程)与 agentic engineering(Agent 工程)是 Karpathy 提出的两个对立概念,后面原理段会展开。

### 2. 划界的根本机制:责任不可委托

为什么用"能负责"而不是"能做"作为划界标准?因为编码这个动作可以委托,责任不能委托。AI 写的代码上线出事,担责的是提交它的人,不是 AI,也不是 AI 背后的模型厂商。

这条机制把"会不会做"和"该不该自己掌握"分开了。一个知识点,哪怕 AI 比你做得又快又好,只要"AI 做错了你识别不出来",它就落在你必须掌握的一侧——因为你没法对它的产出负责。反过来,一个知识点,哪怕你不太熟,只要"AI 做错了你能立刻发现并纠正",它就有可能落在可交给 AI 的一侧。

这条机制也解释了为什么知识边界是"个人化"的:同一个知识点,资深工程师能识别错误,就可以交给 AI;新手识别不出,就必须自己先学透。边界随人而动,没有统一答案。

### 3. 三条判据:把"能负责"拆成可操作的标准

"能不能兜底"太抽象,需要拆成三个具体场景,这就是贯穿全专栏的三条判据:

- **判据一·错误识别**:AI 产出含某项知识的代码,出错时你能否识别。例:AI 写的 SQL 漏建联合索引,你能否一眼看出会回表爆量。
- **判据二·选型判断**:涉及某项知识的技术选型,你能否基于业务场景判断。例:选 Redis 还是 Memcached,你能否基于"要不要持久化、要不要丰富数据结构"判断。
- **判据三·问题定位**:某项知识相关的线上问题,你能否定位根因。例:线上接口 P99 飙升,你能否定位是 GIL 串行化、网络抖动、磁盘 IO 慢,还是锁竞争。

三条判据是 OR 关系——只要有一条答"不能",这项知识就属于必须掌握。为什么是 OR 不是 AND,原理段会给出推导。

## 二、原理

### 1. 三条判据为什么成立

三条判据对应软件工程里三个不可委托给人的环节:审查、决策、排错。AI 在这三个环节的能力有结构性短板,短板的根源并非"模型还不够强",而在这些环节的本质要求。

判据一(错误识别)对应审查。AI 生成代码的错误有三个隐蔽特征:表面工整(命名规范、注释齐全),让人放松警惕;错误藏在边界条件、并发安全、权限校验这些"漂亮外表之下"的地方;偶尔引入你不熟悉的库或模式,审查时需要额外验证。审查能力的本质是"对正确性的先验判断"——你心里得有正确答案的样子,才能比对出偏差。没有这层先验,再强的 AI 输出你也审不动。

判据二(选型判断)对应决策。AI 能给出 Redis 与 Memcached 的对比表,但"选哪个"取决于业务上下文:数据要不要持久化、要不要发布订阅、集群规模、团队熟悉度。这些上下文是隐性的、跨系统的、往往连产品经理都说不全的。AI 拿不到完整上下文,就只能给"通用建议",而通用建议落到具体业务上经常是错的。选型能力的本质是"把隐性业务约束翻译成技术约束",这步必须在人脑里完成。

判据三(问题定位)对应排错。线上故障是"现象—根因"的多对多映射:接口慢可能是 GIL、网络、磁盘、锁、慢 SQL 中任何一个;CPU 飙高可能是死循环、内存泄漏触发频繁 GC、还是业务量上涨。AI 能分析日志,但定位需要"全链路心智模型"——你得在脑子里有一张从请求入口到磁盘 IO 的完整地图,才能在现象出现时迅速缩小范围。这张地图无法临时塞给 AI,它需要长期积累。

三条判据的共同点:它们都依赖"对上下文与约束的深刻理解",而上下文与约束恰恰是 AI 最难获取的隐性信息。这就是三条判据成立的根本机制。

### 2. 决策树:三条判据如何组合

把三条判据组织成一棵决策树,任何一项知识 K 都可以走一遍:

```
对知识点 K,逐条自问:
Q1  AI 产出含 K 的代码出错时,你能识别吗?
    ├─ 不能 → 必须掌握
    └─ 能 → 进 Q2
Q2  涉及 K 的技术选型,你能基于业务判断吗?
    ├─ 不能 → 必须掌握
    └─ 能 → 进 Q3
Q3  K 相关的线上问题,你能定位根因吗?
    ├─ 不能 → 必须掌握
    └─ 能 → 可交给 AI(原理仍要懂,出问题时回流)
```

三条判据是 OR 关系:任何一条"不能",结论就是必须掌握。为什么不是 AND?因为三条对应三个独立的失效场景,任一场景失效都会导致"无法对产出负责"。识错能力再强,选型判断不了,系统照样会在架构层崩;选型判断再准,问题定位不了,出事时只能干瞪眼。三个环节是串联的防线,任何一道被突破,责任就接不住。

决策树最后一行的括号很关键:"原理仍要懂,出问题时回流"。即便三条都答"能",可交给 AI 的也只是"日常实现",不是"原理本身"。原理始终在必须掌握一侧,因为出问题时(判据三)正是原理在起作用。这条回流机制是下一节"边界会移动"的核心。

### 3. 边界会移动:工具进化重塑分界线

知识边界不是刻在石头上的,它随工具进化而移动。移动有三个方向,理解这三个方向,才能动态调整学习重心。

方向一:必须掌握 → 可交给 AI。工具进化让某些能力从"必须"变成"可交"。典型例子是 SQL 索引建议:十年前你得手算回表代价,现在 EXPLAIN 加 AI 建议能覆盖大半。但注意——这里的"可交"是"日常实现",EXPLAIN 输出的解读能力(判据一)仍然必须掌握,否则你连 AI 给的建议对不对都判断不了。

方向二:可交给 AI → 必须掌握(出问题时回流)。平时交给 AI 的,出问题时会回到必须掌握一侧。典型是 asyncio 的 API 顺序:平时让 AI 写、查文档即可,但线上出现协程泄漏或事件循环阻塞时,你必须懂 event loop 的内部机制才能定位。API 是"可交"的,机制是"回流"的。

方向三:必须掌握的内容本身在迁移。从"会写代码"迁移到"会编排和审查代码"。Karpathy 的判语把这层迁移讲得最透,这就是下一节的注脚。

### 4. Karpathy 的注脚:vibe coding 与 agentic engineering

Karpathy 在 2025 年初于 X 平台提出 vibe coding(氛围编程),描述一种凭感觉让 AI 写代码、不太关心代码细节的工作方式。一年后,他在 2026 年 Sequoia Capital 主办的 AI Ascent 大会上做了一场炉边对话,把 vibe coding 与一个新概念 agentic engineering(Agent 工程)对立起来,并提出了 Software 1.0/2.0/3.0 的演化框架[1]。

vibe coding 的定位是"提升下限":让不懂代码的人也能用自然语言搭出能跑的东西,适合原型和个人工具。agentic engineering 的定位是"提升上限":在 vibe coding 之上长出严肃工程实践,用 Agent 编排、上下文工程、审查闭环来保证专业软件的质量、安全与可维护性[1]。

Karpathy 在那场对话里给出了一条对本文最关键的判断——**可以把思考外包,但不能把理解外包**。这条判语恰好是本专栏"知识边界"的注脚:思考(具体怎么写、写什么 API)可以交给 AI,理解(GIL 为什么影响并发、索引为什么失效)不能交给 AI。vibe coding 对应"边界向可交给 AI 移动",agentic engineering 对应"必须掌握的内容从写代码迁移到编排和审查"——两条方向合起来,正是上一节讲的"边界移动"。

Karpathy 还把这场迁移放进 Software 演化框架:Software 1.0 是人写显式代码,Software 2.0 是人准备数据训练神经网络,Software 3.0 是人通过提示词与上下文编程 LLM,上下文窗口成为新的操作杆[1]。在这个框架里,程序员的身份正从"代码书写者"重构为"Agent 编排者"。这不意味着编码能力归零,而是意味着编码能力从核心竞争力下沉为基础能力,编排与审查能力上升为新的核心竞争力。本专栏后续每一章的"必须掌握 vs 可交给 AI"划分,都建立在这层迁移之上。

## 三、实践

### 1. 用决策树分类:Python 异步的两个知识点

用同一技术栈里的两个知识点,演示决策树如何把它们分到不同分支。两个知识点都属 Python 异步,结论却截然相反。

**知识点 A:GIL(Global Interpreter Lock,全局解释器锁)**

- Q1 错误识别:AI 给你一段用 threading 加速 CPU 密集计算的代码,你能否一眼看出它不会提速?答不上来就是把雷埋进生产。→ 不能,必须掌握。
- Q2 选型判断:CPU 密集型服务,你能否判断该用 multiprocessing 还是 threading?IO 密集型能否判断该用 asyncio 还是线程池?判不清就会选错并发模型。→ 不能,必须掌握。
- Q3 问题定位:线上多线程服务 CPU 打不满一半,你能否定位是 GIL 串行化导致?→ 不能,必须掌握。
- 结论:三条全不能,**必须掌握**。

**知识点 B:asyncio 的具体 API 顺序(create_task / gather / wait 的参数顺序)**

- Q1 错误识别:AI 写错 API 顺序,运行时直接抛 TypeError 或 RuntimeError,你能否识别?能,因为会立即报错且易复现。→ 能。
- Q2 选型判断:API 顺序不涉及选型,跳过。→ 不适用。
- Q3 问题定位:平时写代码不涉及定位;但一旦线上出现协程泄漏、事件循环阻塞,定位要靠 event loop 原理——那是另一个知识点(回流到必须掌握)。
- 结论:API 顺序本身**可交给 AI**;但 event loop 原理**必须掌握**。

这两个知识点的对照揭示一条规律:**同一技术栈里,机制(GIL、event loop)必须掌握,写法(API 顺序、样板代码)可交给 AI**。机制决定识错、选型、定位三件事,写法决定不了任何一件。

下面这段代码可以直接运行,实证 GIL 对 CPU 密集任务的串行化效果(Python 3.8+,GIL 默认开启的常规构建):

```python
import threading
import time
import math

def cpu_work(n):
    """CPU 密集任务:对前 n 个整数开方求和。"""
    s = 0
    for i in range(n):
        s += math.isqrt(i)
    return s

def run_serial(n, tasks):
    t0 = time.perf_counter()
    for _ in range(tasks):
        cpu_work(n)
    return time.perf_counter() - t0

def run_threads(n, tasks):
    t0 = time.perf_counter()
    ts = [threading.Thread(target=cpu_work, args=(n,)) for _ in range(tasks)]
    for t in ts:
        t.start()
    for t in ts:
        t.join()
    return time.perf_counter() - t0

if __name__ == "__main__":
    n, tasks = 2_000_000, 4
    print(f"serial:  {run_serial(n, tasks):.3f}s")
    print(f"threads: {run_threads(n, tasks):.3f}s")
```

预期输出(具体数字因机器而异,但量级关系稳定):

```
serial:  0.9s
threads: 1.1s
```

四线程版本不会比串行快,甚至更慢——因为 GIL 让四个线程串行执行 Python 字节码,还要额外付线程切换开销。这就是 GIL 影响"选型"与"定位"的实证:CPU 密集任务用 threading 是错的,得用 multiprocessing(进程间不共享 GIL)或 C 扩展释放 GIL。

**跨版本差异**:Python 3.13(2024 年 10 月发布)经 PEP 703 引入实验性 free-threaded 构建,可在运行时禁用 GIL[3][4]。free-threaded 模式下,上面这段代码的四线程版本会真正并行,接近串行的四分之一耗时。但 free-threaded 目前仍是实验特性,单线程性能有折损,且要求 C 扩展显式标记支持[5]。这恰好是"边界会移动"的活样本:GIL 这个"必须掌握"的知识点本身在演变——free-threaded 普及后,"为什么多线程不提速"的答案会变,但"理解并发模型对选型的影响"这件事反而更复杂了(又多了 free-threaded 这个变量),并没有变成可交给 AI。

### 2. 三条判据的具象:三个真实场景

把三条判据各放进一个真实场景,看它们如何把"必须掌握"逼出来。

**判据一·错误识别——AI 写的 SQL 漏索引**

AI 生成查询:

```sql
-- AI 给的查询
SELECT * FROM orders
WHERE user_id = ? AND created_at > ?;
```

如果库里只有单列索引 `idx(user_id)`,这条查询会走索引找到该用户所有订单,再逐行回表判断 `created_at`。某用户有十万条订单时,回表十万次,接口直接拖垮。AI 不会主动告诉你"建议建联合索引",因为它不知道你的数据分布。你能否一眼看出这个问题?能,就必须掌握联合索引最左匹配与回表代价;不能,这条 SQL 就是定时炸弹。

**判据二·选型判断——Redis 还是 Memcached**

AI 能给两者的对比表,但"选哪个"要基于业务:

- 要持久化(RDB/AOF)→ Redis,Memcached 重启即丢。
- 要丰富数据结构(列表、集合、有序集合)→ Redis,Memcached 只有字符串。
- 要纯 KV 缓存、多线程、极致简单→ Memcached 的多线程模型在高并发纯缓存场景更省心。
- 要发布订阅、Stream、Lua 脚本→ Redis 独有。

你能否基于业务场景判断?判断不了,就会在"明明需要持久化却选了 Memcached"或"明明只是纯缓存却背上 Redis 单线程复杂度"这类错误上栽跟头。

**判据三·问题定位——线上接口慢**

接口 P99 从 50ms 飙到 2s。可能根因至少五个:GIL 串行化(Python 多线程没提速)、网络抖动(下游服务超时)、磁盘 IO 慢(日志或 DB 慢盘)、锁竞争(数据库行锁或分布式锁)、慢 SQL(索引失效或全表扫描)。你能否定位?这需要全链路心智模型——从网关到应用再到存储,知道每一层可能的瓶颈长什么样。定位不了,就只能盯着监控干着急,或盲目重启碰运气。

三个场景的共同点:AI 都能"参与",但都不能"收尾"。识别、判断、定位这三步,必须有人在脑子里完成。

### 3. 教科书 vs 生产:判据的应用差异

| 场景 | 教科书做法 | 生产做法 | 起决定作用的判据 |
|---|---|---|---|
| GIL | 背概念 | 用 multiprocessing 或 3.13+ free-threaded | 判据二(选型) |
| SQL 索引 | 背最左匹配 | EXPLAIN 看执行计划 + 慢日志 | 判据一(识错) |
| 接口慢 | 看一处 | 全链路 trace + 分层排查 | 判据三(定位) |

生产做法与教科书做法的差距,正是三条判据在起作用。教科书教你"是什么",生产要你"在压力下识别、判断、定位"。这也是为什么背了很多概念的人到了线上还是抓瞎——他们停在"是什么",没走到三条判据要求的"能不能负责"。

## 四、速查/自测

### 速查表:三条判据速查

| 判据 | 触发场景 | 必须掌握的标志 | 可交给 AI 的标志 |
|---|---|---|---|
| 一·错误识别 | AI 产出代码 | 能识别边界/安全/性能错误 | 错误会立即报错且易复现 |
| 二·选型判断 | 技术选型 | 能基于业务约束做决策 | 选型不依赖业务上下文 |
| 三·问题定位 | 线上故障 | 能定位根因到具体层 | 问题有明确报错且自愈 |

### 决策树速查

```
对知识点 K:
Q1 AI 产出含 K 的代码出错,你能识别吗?  不能→必须掌握
Q2 涉及 K 的选型,你能基于业务判断吗?  不能→必须掌握

```

### 自测题(三层次)

**原理层**:三条判据为什么是 OR 关系而非 AND?

<details><summary>参考答案</summary>

三条判据对应审查、选型、排错三个串联的防线,任一道被突破都会导致"无法对产出负责"。AND 关系要求三条全不能才算必须掌握,意味着只要有一条能就放手——这会在另两条失守时留下责任真空。OR 关系保证只要有一条不能就归入必须掌握,把责任兜底。防线串联的逻辑决定了必须用 OR。</details>

**思路层**:GIL 和 asyncio 的 API 顺序,为什么一个必须掌握一个可交给 AI?

<details><summary>参考答案</summary>

GIL 决定并发模型的选型(判据二:CPU 密集用 threading 是错的)、决定多线程不提速的定位(判据三)、决定能否识别"AI 用 threading 加速 CPU 任务"这类错误(判据一)——三条判据全部命中,必须掌握。asyncio API 顺序出错会立即抛异常、易复现(判据一能)、不涉及选型(判据二不适用)、平时不涉及定位(判据三只在出问题时回流到 event loop 原理)——三条都不卡,可交给 AI。差别在于:机制影响三件事,写法一件事都不影响。</details>

**实践层**:线上接口 P99 飙升,你按什么顺序排查?

<details><summary>参考答案</summary>

按"由外向内、由廉到贵"的顺序:先看监控分层指标(网关 P99 vs 应用 P99 vs DB 慢查询),定位瓶颈层;若卡在应用层,看 CPU/GC/线程池是否打满(判据三的 GIL/锁线索);若卡在 DB 层,看慢日志和 EXPLAIN(判据一的索引线索);若卡在网络层,看下游依赖超时与重试。每一步都在调用三条判据积累的心智模型。盲目重启或只看一处,都是判据三没练成的表现。</details>

## 可交给 AI 的部分

本章里能放心交给 AI 的,是"写法层"内容:三条例子里 asyncio 的 API 顺序、SQL 的具体写法(给表名给条件让 AI 生成)、Redis 与 Memcached 的对比表、决策树流程图的初版绘制。这些共同特征是:错误会立即暴露、不依赖业务上下文、出问题时有明确报错。

交给 AI 时的风险有三条,恰好对应三条判据的反面:

- **识错风险**:AI 写的 SQL 表面正确,但漏建联合索引导致回表爆量。AI 不会主动提醒,因为不知道你的数据分布。你必须懂 EXPLAIN 和最左匹配才能识破。
- **选型风险**:AI 给的对比表是通用的,落到你的业务上可能是错的。AI 不知道你要不要持久化、集群规模多大。你必须懂业务约束才能决策。
- **定位风险**:AI 能分析你贴给它的日志,但定位需要全链路心智模型,这张地图无法临时塞给它。你必须平时就懂机制,出事时才能用上。

为什么 API 顺序能交、GIL 不能交?因为 API 顺序错了运行时立刻报错,识错成本为零;GIL 错了不会报错,只是"多线程没提速"这个隐性失效,识别它要靠你对并发模型的理解。为什么 SQL 写法能交、索引设计不能交?因为写法错了语法检查会拦,索引设计错了不会拦,只在数据量上去后拖垮接口。分界线就一条:**错误是否隐性**。隐性错误的识别、判断、定位,就是必须掌握的边界;显性错误的部分,可以交给 AI。
