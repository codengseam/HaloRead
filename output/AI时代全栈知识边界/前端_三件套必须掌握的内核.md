# AI时代全栈知识边界·06|三件套必须掌握的内核

> 前置知识:会打开浏览器开发者工具、能读懂简单的标签和花括号语法
> 学完你能:① 看懂任意一份前端代码并定位常见 bug ② 说清闭包、this、事件循环的运行轨迹 ③ 解释 z-index 失效和重排重绘的性能差异 ④ 写出正确的跨域配置与防抖函数 ⑤ 判断哪些前端工作可交给 AI、哪些必须自己把关

### 一、概念

三件套的"内核",是浏览器把「结构、表现、行为」三股力量编织成一张可交互页面的那套最小不可再分的规则集。HTML(HyperText Markup Language,超文本标记语言)负责结构的语义契约,是 DOM(Document Object Model,文档对象模型)树的源头;CSS(Cascading Style Sheets,层叠样式表)负责表现的层叠规则,核心是盒模型与层叠上下文;JavaScript 负责行为的执行模型,核心是作用域链、原型链与事件循环。

这里要划清边界:所谓"内核",不是"怎么写一个漂亮页面",而是"为什么这样写能 work、那样写会出 bug"。前者可以交给 AI,后者必须自己掌握。一个全栈工程师即便主攻后端,只要产品有前端界面,就一定会遇到"线上前端报错、后端同学先顶上"的时刻——能不能在十分钟内判断这是闭包泄漏、跨域被拦、还是 z-index 失效,就是"看得懂前端"与"看不懂前端"的分水岭。

### 二、原理

#### 1. HTML 语义化:为什么不用满屏 div

HTML 标签的本质是"给内容打语义标签"。`<div>` 是无语义容器,`<article>` 表示独立完整的内容(一篇文章、一条评论),`<section>` 表示文档章节,`<nav>` 表示导航区域,`<header>`/`<footer>`/`<aside>`/`<main>` 各有归属。不用满屏 div 有三条理由:SEO(Search Engine Optimization,搜索引擎优化)上,爬虫依赖语义标签理解页面结构,`<article>` 内的正文权重高于裸 div;无障碍(Accessibility)上,屏幕阅读器靠语义标签给视障用户跳转导航;可维护性上,`<main>` 比 `<div class="main">` 自带"这是干嘛的"信息。语义化不增加任何渲染成本,只是把"意图"写进标签。

#### 2. 盒模型:content-box 与 border-box 都存在的来由

每个元素是一个矩形盒子,由内到外四层:content(内容)、padding(内边距)、border(边框)、margin(外边距)。问题出在 width 指的是哪一层。

- content-box(W3C 标准默认):width 只含 content。设 `width:100px` 再加 `padding:10px`、`border:1px`,实际占宽 122px。
- border-box(IE 盒模型):width 含 content + padding + border。设 `width:100px` 再加 padding 和 border,实际占宽仍是 100px,content 自动缩小。

历史原因是浏览器大战时代 IE 用 border-box,W3C 标准选了 content-box,开发者被"加了 padding 元素就撑爆"折磨了十几年,直到 CSS3 把 `box-sizing:border-box` 纳入标准。工程里几乎全员写 `*{box-sizing:border-box}`,因为算尺寸时只关心"最终占多宽"。理解这条,就能解释新人写的卡片为什么总是莫名超宽一行。

#### 3. 文档流与定位:元素为什么乱跑

默认元素按文档流(Normal Flow)排列:块级纵向堆叠,行内横向排列。`position` 五个值的本质是"这个元素参不参与文档流、以谁为参照":

| 值 | 是否脱离文档流 | 参照物 |
|---|---|---|
| static(默认) | 否 | 无,正常流 |
| relative | 否 | 自身原位置 |
| absolute | 是 | 最近的非 static 祖先 |
| fixed | 是 | 视口(可被 transform 改变) |
| sticky | 否 | 滚动到阈值前 relative、阈值后 fixed |

"脱离文档流"是布局错乱的根因:一个 absolute 元素不再占位,后面的元素顶上来,于是出现图片盖住文字。定位 bug 排查第一步永远是"这个元素脱离流了吗、参照谁"。

#### 4. flex 与 grid:一维与二维的根本差别

flex(Flexible Box,弹性盒)是一维布局:沿主轴分配空间,适合一行里几个元素怎么分。grid 是二维布局:同时管行和列,适合整个页面区域划分。选型原则很简单——只在一根轴上排就用 flex,要同时管行列就用 grid。新人常滥用 grid 做单行布局,或用 flex 套娃实现网格,都是没分清维度。

#### 5. 层叠上下文:z-index 失效的根因

z-index 不是全局深度排序,而是"在同一层叠上下文(Stacking Context)内的局部排序"。层叠上下文是一个独立的排序单元,内部元素的 z-index 只在本上下文里比,出了这个上下文就被当作整体参与外层比较。

形成层叠上下文的条件很多:根元素 html、position 非 static 且 z-index 非 auto、opacity 小于 1、transform 非 none、filter 非 none、will-change 等。一旦某元素形成了新层叠上下文,它内部子元素 z-index 再大也飞不出这个上下文——这就是 z-index 失效的最常见根因:你以为给子元素设 `z-index:9999` 能盖住外层,但父元素形成了一个层叠上下文且自身 z-index 比对方低,子元素再高也被父元素整体压在下面。

#### 6. 原型链与作用域链:属性的查找路径

每个对象有内部插槽 `[[Prototype]]`,访问属性时若自身没有就沿 `[[Prototype]]` 往上找,直到 null,这就是原型链。`Object.create(null)` 创建的对象没有原型,常用于纯净字典。原型链解释了"修改原型会影响所有实例"——它们共享同一条查找路径。

函数在创建时就确定了要访问的变量环境(词法作用域),这条链叫作用域链。闭包(Closure)本质是"函数 + 它创建时的词法环境引用",让函数即使在外部执行仍能访问定义处的变量。闭包导致内存泄漏的根因:只要闭包函数还被引用,它捕获的外部变量就不会被回收。常见场景是定时器、事件监听器、全局缓存持有闭包而未清理——一个 `setInterval` 回调闭包持有大数组,只要 interval 不 clear,数组就永远不释放。

#### 7. this 指向:箭头函数与普通函数的分水岭

普通函数的 this 在调用时确定,由调用方式决定:`obj.fn()` 里 this 是 obj,裸 `fn()` 严格模式下是 undefined,`new Fn()` 里是新对象。箭头函数没有自己的 this,在定义时继承外层作用域的 this。这条差异解释了两类经典 bug:回调里用普通函数 this 丢失(常见于事件处理器),以及用箭头函数当对象方法导致 this 指向窗外。

#### 8. 事件循环:microtask 为什么优先

浏览器事件循环(Event Loop)的核心是一个循环:取一个 task(宏任务)执行 → 清空所有 microtask(微任务)队列 → 渲染(若需要)→ 取下一个 task。microtask 优先于下一个 task 执行,根本机制在于规范规定"每轮 task 结束后、渲染前必须清空 microtask 队列"。

这样设计的好处:Promise 的 then 回调作为 microtask,能在本轮内拿到结果,保证 Promise 语义是"尽快 resolve";而 setTimeout 的回调是 task,至少要跨一个 task 才执行,所以 `setTimeout(fn, 0)` 不会真的 0ms 执行。async/await 的本质是 Promise + 生成器的语法糖,`await expr` 等价于暂停当前 async 函数、等 expr 的 Promise resolve 后把后续代码作为 microtask 排入队列。所以 await 之后的代码总是排在同步代码之后、下一个 setTimeout 之前。

#### 9. 浏览器渲染:重排与重绘的性能鸿沟

浏览器把页面画出来要经过:解析 HTML 生成 DOM 树 → 解析 CSS 生成 CSSOM(CSS Object Model,CSS 对象模型)→ 合成 Render Tree(渲染树)→ Layout(布局,算几何)→ Paint(绘制像素)→ Composite(合成图层)。

- 重排(Reflow):改变了几何(宽高、位置),触发 Layout 重新计算,代价最大。
- 重绘(Repaint):只改颜色、背景等不影响几何的样式,跳过 Layout 只触发 Paint,代价中等。
- Composite:只改 transform、opacity 这类能提升为独立图层的属性,只触发合成,GPU 加速,代价最小。

工程含义:动画尽量只动 transform 和 opacity,避免动 width、top、margin 这类触发重排的属性;批量改 DOM 时用 DocumentFragment 或先 `display:none` 再操作,减少重排次数。

#### 10. 跨域:同源策略与 CORS 预检

同源策略(Same-Origin Policy)规定:脚本只能访问与当前页面同源(scheme + host + port 完全一致)的资源。这是浏览器的安全基石,防止 evil.com 的脚本读取 bank.com 的数据。

CORS(Cross-Origin Resource Sharing,跨源资源共享)是同源策略的受控开口:服务器通过响应头 `Access-Control-Allow-Origin` 显式允许某些源跨域访问。对于"简单请求"(GET/HEAD/POST 且 Content-Type 为表单等少数值),浏览器直接发、看响应头决定是否给脚本读;对于"非简单请求"(PUT、DELETE、自定义头、application/json),浏览器先发一个 OPTIONS 预检(Preflight),问服务器"这个方法/头允许吗",允许后才发真正请求。

预检存在的根本原因:HTML 表单自古就能跨域发 POST,但表单的 Content-Type 和方法受限,且无法被脚本读取响应。当脚本想用更危险的能力(PUT、自定义头、JSON)跨域时,若直接发,服务器可能在不知情下执行了有副作用的操作。预检是浏览器替服务器先问一句"这种请求你接不接受",把是否允许跨域的决定权交给服务器,而不是让脚本直接突破同源策略。这就是为什么后端常收到"前端调接口报错、先看是不是 OPTIONS 没通"——预检不通,真请求根本发不出去。

### 三、实践

#### 1. 闭包经典坑:for + setTimeout + var/let

```javascript
// 坑:var 声明的 i 是函数级,循环结束后 i=5,五个定时器共享同一个 i
for (var i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 0);
}
// 输出:5 5 5 5 5

// 修复一:let 声明,每次迭代创建新的词法绑定,闭包捕获各自的 i
for (let i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 0);
}
// 输出:0 1 2 3 4

// 修复二:用 IIFE(立即调用函数表达式)手动创建闭包,捕获当时的 i
for (var i = 0; i < 5; i++) {
  (function (j) {
    setTimeout(() => console.log(j), 0);
  })(i);
}
// 输出:0 1 2 3 4
```

根因:`var` 没有"每轮新建绑定"的语义,五个闭包捕获同一个 i;`let` 在每次迭代开始时创建新绑定,五个闭包各捕获各的。理解这条,就能解释为什么循环里绑定事件总要 bind 或用 IIFE,而换成 `let` 就不出错。

#### 2. 事件循环顺序题

```javascript
console.log(1);                          // 同步 task

setTimeout(() => console.log(2), 0);     // task(宏任务)

Promise.resolve().then(() => {
  console.log(3);                        // microtask
}).then(() => {
  console.log(4);                        // microtask
});

console.log(5);                          // 同步 task
```

执行顺序:同步代码先跑完(1、5)→ 当前 task 结束,清空 microtask 队列(3、4)→ 取下一个 task(2)。

输出:
```
1
5
3
4
2
```

带 async/await 的版本(ES2017+):

```javascript
async function a() {
  console.log('a start');
  await Promise.resolve();
  console.log('a end');                  // 这行作为 microtask 执行
}
a();
console.log('main');
```

执行顺序:`a()` 同步部分先跑(打印 a start、遇 await 暂停返回)→ 同步打印 main → 当前 task 结束清 microtask(a end)。

输出:
```
a start
main
a end
```

速断:`await` 之后的代码 = `.then` 回调 = microtask,永远排在同步代码之后、下一个 setTimeout 之前。

#### 3. z-index 失效示例

```html
<div class="parent-a">
  <div class="child-a">我想盖住 B</div>
</div>
<div class="parent-b">
  <div class="child-b">我是 B</div>
</div>
```

```css
.parent-a { position: relative; z-index: 1; }   /* 形成层叠上下文 */
.parent-b { position: relative; z-index: 2; }   /* 形成层叠上下文 */
.child-a  { position: absolute; z-index: 9999; } /* 看似很高 */
.child-b  { position: absolute; z-index: 1; }
```

直觉上 child-a 的 9999 远大于 child-b 的 1,应该盖住 B。实际效果是 B 盖住 A 的所有子元素。原因:parent-a 和 parent-b 各自形成了层叠上下文,它们之间的比较只看各自的 z-index(1 vs 2),parent-b 赢,于是 parent-b 内部所有元素整体在 parent-a 之上;child-a 的 9999 只在 parent-a 内部排序有效,飞不出 parent-a。

修复思路二选一:把 child-a 提到与 parent-b 同级(脱离 parent-a 的上下文),或者去掉 parent-a 形成层叠上下文的条件(改 z-index 为 auto)。排查 z-index 失效,永远沿着 DOM 树往上找"谁形成了层叠上下文、谁的 z-index 比对手低"。

### 四、速查/自测

#### 事件循环顺序速查表

| 代码位置 | 类型 | 执行时机 |
|---|---|---|
| 顶层同步代码 | 同步 task | 立即 |
| `setTimeout(fn, 0)` | 宏任务(task) | 当前 task + microtask 全部清空后 |
| `Promise.then(fn)` | 微任务(microtask) | 当前 task 结束、渲染前 |
| `await` 之后的代码 | 微任务(microtask) | 等同于 `.then` 回调 |
| `requestAnimationFrame` | 渲染前回调 | microtask 清空后、Paint 前 |
| `MutationObserver` 回调 | 微任务(microtask) | DOM 变更后、下一 task 前 |

#### 自测题

1. **原理层**:为什么 microtask 必须在下一个 task 之前清空,而不是穿插执行?
   参考答案:microtask 的设计语义是"本轮内的收尾工作",保证 Promise 在同一轮事件循环内 resolve,使异步行为可预测。若允许 task 插队,microtask 可能被无限延迟,Promise 语义被破坏。

2. **思路层**:做"顶部固定导航 + 下方滚动内容"的布局,flex 还是 grid?
   参考答案:flex。导航和内容是纵向一维关系,`flex-direction:column` 配 `flex-shrink:0` 固定导航即可。grid 适合二维网格(如仪表盘卡片),这里用 grid 是杀鸡用牛刀。

3. **实践层**:写一个防抖(debounce)函数,要求最后一次触发后等待 wait 毫秒才执行,期间再次触发则重新计时。

```javascript
function debounce(fn, wait) {
  let timer = null;
  return function (...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), wait);
  };
}
```

要点:timer 在闭包里持有,每次调用 clearTimeout 重置;用 `apply(this, args)` 保证 this 和参数正确传递;这是闭包 + 事件循环(task)的直接应用。

4. **实践层**:接口报错"CORS 策略已阻止:不存在 Access-Control-Allow-Origin 头",后端说接口测过没问题,你怎么定位?
   参考答案:先确认是不是预检没通——看请求方法是 OPTIONS 还是不是目标方法。若是 OPTIONS 失败,检查后端是否对 OPTIONS 也返回了 `Access-Control-Allow-Origin` 和 `Access-Control-Allow-Methods`;若是简单请求失败,检查响应头是否真的带 `Access-Control-Allow-Origin` 且值匹配(或为 *)。常见误区是后端只在 POST 处理里加头,忘了 OPTIONS 单独走中间件。

5. **原理层**:为什么动画推荐用 transform 而不是 left?
   参考答案:left 改变几何,触发重排(Layout + Paint + Composite);transform 不影响布局,元素被提升为独立图层,只触发 Composite,GPU 加速,代价最小。60fps 动画下两者差距可达数十倍。

### 可交给 AI 的部分

能放心交给 AI 的,是"表现层与样板层"工作:精美页面样式、复杂 CSS 动画、多端适配的媒体查询、完整页面模板、表单校验正则、UI 组件初版代码、兼容性前缀、CSS 预处理代码。这些任务对就是对、错就是错,可肉眼验收,AI 写错了一眼能看出来;且不涉及运行时的隐式行为,改起来成本可控。

不能交给 AI、必须自己把关的,是"行为层与原理层":闭包导致的内存泄漏、this 指向错乱、事件循环顺序、z-index 失效、跨域预检配置、原型链污染。这些 bug 的特征是代码看起来对、运行时才爆,且爆点远离写错的地方——一个闭包泄漏可能在你卸载组件十秒后才体现,一个 z-index 失效可能只在特定滚动位置才出现。AI 写这类代码时,常常语法正确但语义错:它能写出闭包,但不会主动提醒你这个定时器要 clear;能写出 fetch,但可能漏掉 OPTIONS 预检的处理;能写出 `z-index:9999`,但不会检查父元素是否形成了层叠上下文。

风险提示具体到三条:第一,AI 写的 CSS 常漏浏览器兼容,尤其新特性(flex 的 gap、`:has()` 选择器),需要你核对 caniuse 兼容性表;第二,AI 写的 JavaScript 可能有原型链坑,比如 `Object.assign` 是浅拷贝、`==` 隐式转换,AI 不一定主动规避;第三,AI 写的跨域配置常常只配了简单请求、漏掉预检,导致本地能跑、上线报错。判断标准很简单:涉及"何时执行、谁持有引用、谁盖住谁、谁允许谁访问"这四类隐式行为时,自己过一遍原理;涉及"长什么样、什么动画、什么布局"时,大胆交给 AI。
