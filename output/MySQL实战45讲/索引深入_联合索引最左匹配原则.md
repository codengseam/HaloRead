# 第 11 章：联合索引最左匹配原则

> 前置知识：第 09 章联合索引概念、第 10 章回表与覆盖索引
> 学完你能：①判断任意 SQL 能否命中联合索引、能用到第几个字段；②说清"范围查询右侧失效"的根因；③讲透索引下推 ICP 的原理和触发条件

## 概念

联合索引（Composite Index）是多列组合的 B+ 树索引，如 `idx(a,b,c)`。B+ 树叶子节点按 `a → b → c` 的字典序排列：先按 `a` 排序，`a` 相同再按 `b` 排序，`b` 相同再按 `c` 排序。

**最左前缀原则（Leftmost Prefix）**：查询必须从联合索引最左列开始匹配，才能用到索引。中间字段一旦跳过或中断，右侧字段就用不上索引。林晓斌在《MySQL 实战 45 讲》"深入浅出索引（下）"里强调，最左前缀不是 SQL 书写顺序，而是优化器重排后的实际匹配顺序——`where b=2 and a=1` 也能命中 `idx(a,b)`，因为优化器会自动调整成 `a=1 and b=2`。

**索引下推（Index Condition Pushdown, ICP）**：MySQL 5.6 引入的优化。在存储引擎层（而非 server 层）对索引包含的列先做条件过滤，减少回表次数。Explain 的 Extra 列显示 `Using index condition` 表示触发 ICP。

## 原理

### 联合索引在 B+ 树上怎么排

`idx(a,b,c)` 的叶子节点排列示例：

```
(1,1,1) (1,1,2) (1,2,1) (1,2,3) (2,1,1) (2,1,3) (2,2,2) (3,1,1) ...
```

`a=1` 的记录连续排列，`a=1` 内部按 `b` 升序，`b` 相同再按 `c` 升序。这种"分层有序"结构决定了：能定位到一段连续区间的查询能用索引，定位不到就只能全扫描。

### 最左前缀的判定规则

以 `idx(a,b,c)` 为例，逐条看：

- `where a=1`：✅ 用到 a
- `where a=1 and b=2`：✅ 用到 a, b
- `where a=1 and b=2 and c=3`：✅ 用到 a, b, c
- `where b=2`：❌ 不满足最左前缀，全表扫描
- `where b=2 and c=3`：❌ 同上
- `where a=1 and c=3`：⚠️ 部分有效，只用到 a（c 用不上索引，因为中间跳过了 b，c 在 `a=1` 区间内无序）

判定口诀：**从最左列开始连续匹配，遇到跳过或范围就停止。**

### 范围查询右侧失效

`where a>1 and b=2`：`a` 是范围查询，匹配 `a>1` 的所有记录在 B+ 树上是一段连续区间，但这段区间内 `b` 不再有序（因为 `a` 不同，`b` 的顺序被打乱）。所以 `b` 用不上索引，只能在 `a>1` 命中的行里逐行判断 `b=2`。

常见范围操作符：`>`、`<`、`>=`、`<=`、`between`、`like 'abc%'`（前缀匹配也是范围）。等号 `=` 不算范围，右侧字段还能继续用索引。

这个规则有一个常被忽略的细节：`between` 在 MySQL 里被当作范围处理，右侧字段失效；但 `in` 在 MySQL 5.6 之后被优化器当作多个等值处理，右侧字段仍能用索引。所以 `where a in (1,2,3) and b=2` 能用到 a 和 b，而 `where a between 1 and 3 and b=2` 只用到 a。

### 索引下推 ICP 怎么减少回表

表 `tuser(id PK, name, age, ismale)`，联合索引 `idx(name, age)`，执行：

```sql
select * from tuser where name like '张%' and age=10 and ismale=1;
```

**MySQL 5.6 之前（无 ICP）**：

1. 在 `idx(name,age)` 上用 `name like '张%'` 定位到所有"张"开头的记录（假设 4 条）。
2. 拿这 4 条的主键逐条回表，到聚簇索引取整行。
3. 在 server 层判断 `age=10` 和 `ismale=1`，过滤掉不满足的。
4. 回表 4 次。

**MySQL 5.6 之后（有 ICP）**：

1. 在 `idx(name,age)` 上用 `name like '张%'` 定位到 4 条"张"开头的记录。
2. 在存储引擎层就用索引里的 `age` 字段判断 `age=10`，假设只有 2 条满足。
3. 只把这 2 条的主键回表，到聚簇索引取整行。
4. 在 server 层判断 `ismale=1`（`ismale` 不在索引里，无法下推）。
5. 回表 2 次。

ICP 把 `age` 的过滤从 server 层下推到了存储引擎层，回表次数从 4 降到 2。回表减少一半，性能直接翻倍。

### ICP 的触发条件

不是所有条件都能下推。能下推的条件必须满足：

1. 条件涉及的列在索引里（如上例 `age` 在 `idx(name,age)` 里，能下推；`ismale` 不在，不能下推）。
2. 条件能用索引列直接判断，不需要回表（`like '张%'` 是范围扫描，但 `age=10` 是等值，能下推）。
3. 存储引擎支持 ICP（InnoDB、MyISAM 都支持，NDB 不支持）。
4. 子查询、存储过程、触发器相关条件不下推。

跨版本说明：ICP 是 MySQL 5.6 引入的，5.6 之前的版本没有这个优化，`like '张%' and age=10` 会全量回表。生产环境如果还在 5.5 或更早版本，升级到 5.6+ 能直接拿到性能提升。

## 实践

**面试场景：被问到"联合索引 (a,b,c)，哪些查询能用索引"**

回答框架：①先看是否满足最左前缀（从 a 开始）；②看是否有范围查询让右侧失效；③看优化器重排（`b=2 and a=1` 等价 `a=1 and b=2`）；④提一句 ICP 对索引列条件的下推优化。再举一个"范围右侧失效"的例子，能立刻体现深度。

**项目场景：联合索引字段顺序设计**

订单表查询模式：`where user_id=? and status=? and create_time between ? and ?`。三个候选索引顺序：

- `idx(user_id, status, create_time)`：`user_id` 等值、`status` 等值、`create_time` 范围。三个字段都能用上索引。✅ 最优。
- `idx(user_id, create_time, status)`：`user_id` 等值、`create_time` 范围让右侧 `status` 失效。只用两个字段。
- `idx(create_time, user_id, status)`：`create_time` 范围在第一个，右侧 `user_id`、`status` 全失效。最差。

设计原则：**等值条件放前面、范围条件放后面**。区分度高的等值字段最优先，范围查询字段最后。

**项目场景：ICP 触发判断**

```sql
explain select * from tuser where name like '张%' and age=10 and ismale=1;
```

Extra 列出现 `Using index condition` 表示 ICP 生效。如果只看到 `Using where` 没有 `Using index condition`，说明 ICP 没触发，可能原因：①`age` 不在索引里；②MySQL 版本低于 5.6；③条件涉及子查询。

**避坑：`like '%张'` 直接失效**

`where name like '%张'` 是左模糊，B+ 树无法定位起点（所有节点都可能是匹配项），直接退化为全表扫描，连最左前缀都满足不了。右模糊 `like '张%'` 是范围查询，能用索引。模糊查询必须左固定。

**避坑：函数运算让索引失效**

`where date(create_time)='2026-01-01'` 对索引列加了函数，B+ 树上存的是 `create_time` 原值，函数运算后无序，索引失效。改成 `where create_time between '2026-01-01 00:00:00' and '2026-01-01 23:59:59'` 就能走索引。这个话题在第 12 章"索引失效场景与排查"展开。

**避坑：隐式类型转换**

`where id='123'`（`id` 是 int）会发生隐式转换，MySQL 把 `id` 转成字符串再比较，相当于对索引列加了函数，索引失效。改为 `where id=123`（数值匹配）即可。这个规则在第 12 章详细展开。
