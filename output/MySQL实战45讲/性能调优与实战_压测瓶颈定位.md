# 第 30 章：压测瓶颈定位

> 前置知识：sysbench、Explain、Buffer Pool、读写分离
> 学完你能：① 用 SDET 视角设计数据库压测方案 ② 按 QPS/TPS/连接数/命中率/锁等待判定瓶颈点 ③ 给出读写分离、缓存、分库分表的优化路径

## 概念

压测不是只看 QPS 和 TPS 两个数。很多团队压测报告写"QPS 8000，系统表现良好"，上线后却频繁告警——因为压测时数据库 CPU 才 30%，而真实流量下连接数、锁竞争、数据分布完全不同，瓶颈藏在压测没覆盖的维度里。SDET 做数据库压测，目标是要把"系统能撑多少、瓶颈在哪、优化后能提升多少"这三个问题量化回答清楚。

数据库压测的标准工具是 `sysbench`，它内置了 `oltp_read_write`、`oltp_read_only`、`oltp_point_select` 等场景，能模拟不同读写比例的负载。压测时要同时采集数据库侧的 CPU、IO、连接数、Buffer Pool 命中率、锁等待、慢 SQL，才能判断瓶颈到底在哪一层。

林晓斌在《MySQL 实战 45 讲》第 33 讲"我查这么多数据，会不会把数据库内存打爆"一讲里指出，查询扫描的数据量直接决定内存压力，Buffer Pool 命中率是判断"是不是被内存卡住"的核心指标。这条结论在压测中尤其重要。

## 原理

### 压测监控九维度

| 维度 | 指标 | 瓶颈信号 |
|---|---|---|
| 吞吐 | QPS、TPS | 随并发上升不再增长 |
| 连接 | `Threads_connected`、`Threads_running` | 接近 `max_connections` |
| 计算 | CPU 利用率 | 持续高于 80% |
| IO | `iostat` 的 `%util`、`await` | `%util` 接近 100% |
| 内存 | Buffer Pool 命中率 | 低于 95% |
| 锁 | `innodb_row_lock_waits`、锁等待时间 | 锁等待增长 |
| 慢 SQL | 慢查询计数 | 慢 SQL 数上升 |
| 网络 | 带宽、重传 | 带宽打满 |
| 复制 | `Seconds_Behind_Master` | 从库延迟 |

判定瓶颈的规则：CPU 持续高于 80% 是计算瓶颈；IO 的 `%util` 接近 100% 且 `await` 高是磁盘瓶颈；Buffer Pool 命中率低于 95% 是内存瓶颈；锁等待时间长是并发瓶颈。前两类要扩容或优化 SQL，第三类要扩 Buffer Pool，第四类要打散热点或降并发。

### sysbench 标准压测

```bash
sysbench oltp_read_write --mysql-host=127.0.0.1 \
  --mysql-db=test --tables=10 --table-size=1000000 \
  --threads=64 --time=300 run
```

`oltp_read_write` 是混合读写场景，最接近在线交易。`oltp_point_select` 测纯主键查询，能压出 Buffer Pool 命中率上限。压测要梯度加并发（16、32、64、128），观察 QPS 在哪个并发档位不再上升，那个点就是当前配置的容量拐点。

压测前要做数据预热。冷启动时 Buffer Pool 是空的，第一批请求会大量回磁盘，命中率极低，压出来的 QPS 偏低且不稳定。正确做法是先用 `sysbench` 的 `prepare` 阶段灌入数据，再跑一轮预热流量让热点数据进 Buffer Pool，然后清空计数器开始正式压测。压测数据分布也要接近真实，自增主键的顺序写入和真实业务的随机写入行为完全不同，必要时用 `--rand-type=uniform` 造随机分布，否则压出的 QPS 会虚高。另外压测时长至少 5 到 10 分钟，短时压测只能反映峰值爆发力，长时压测才能暴露 GC 停顿、连接泄漏、Buffer Pool 抖动这类稳态问题。

SDET 出压测报告时，要把每个并发档位的九维度指标列成表，标出瓶颈拐点，给研发明确的容量结论，而不是只甩一个 QPS 数字。

### 优化路径

按瓶颈类型选优化方向：

- 计算瓶颈（CPU 高）：先查慢 SQL 加索引，再考虑读写分离把读流量分到从库。
- 内存瓶颈（命中率低）：扩 `innodb_buffer_pool_size`，生产建议物理内存的 50% 到 70%。
- IO 瓶颈：用 SSD 替换机械盘，或把冷数据归档减少扫描量。
- 并发瓶颈（锁等待高）：热点行打散、降隔离级别、队列串行化。
- 容量瓶颈（单表过大）：单表超过 2000 万行考虑分库分表。

## 实践

**面试场景**

面试官问："怎么压测 MySQL，怎么定位瓶颈？"

回答模板："我用 `sysbench` 跑 `oltp_read_write`，梯度加并发找容量拐点。同时采集九维度指标：QPS 和 TPS 看吞吐，CPU 和 IO 看资源，Buffer Pool 命中率看内存，锁等待看并发。判定规则是 CPU 高于 80% 算计算瓶颈、命中率低于 95% 算内存瓶颈、`%util` 满算 IO 瓶颈。定位到瓶颈后按类型选优化：慢 SQL 加索引、读多走读写分离、命中率低扩 Buffer Pool、单表大走分库分表。"

追问"压测 QPS 上不去，怎么判断是数据库瓶颈还是应用瓶颈？"——看数据库侧 `Threads_running`：如果数据库连接数没打满、CPU 不高，瓶颈在应用；如果 `Threads_running` 接近上限且 CPU 或 IO 高，瓶颈在数据库。

**项目场景**

一次接口压测，`sysbench` 混合场景 QPS 卡在 2000 上不去，加并发到 128 反而下降。排查九维度：CPU 才 40%，IO 的 `%util` 为 60%，但 Buffer Pool 命中率只有 88%，`innodb_buffer_pool_size` 配的 4G，而数据量有 20G。根因是内存不足导致大量磁盘换页，命中率低。把 `innodb_buffer_pool_size` 扩到 16G（物理内存的 60%）后重压，命中率升到 99%，QPS 提到 8000。这个案例说明压测只看 CPU 和 IO 会漏判内存瓶颈，命中率是不可少的维度。

**避坑**

- 压测数据量要接近真实，10 万行压出的 QPS 上线后扛不住 1000 万行的真实表。
- 压测前关掉慢日志采样干扰，但压测后要分析慢 SQL。
- Buffer Pool 扩容要在线热扩，8.0 支持 `SET GLOBAL innodb_buffer_pool_size` 动态调整。

## 速查/自测

**速查表**

| 瓶颈类型 | 判定指标 | 优化方向 |
|---|---|---|
| 计算 | CPU 高于 80% | 慢 SQL 加索引、读写分离 |
| 内存 | 命中率低于 95% | 扩 Buffer Pool |
| IO | `%util` 接近 100% | SSD、冷数据归档 |
| 并发 | 锁等待增长 | 热点打散、降隔离级别 |
| 容量 | 单表大于 2000 万行 | 分库分表 |

**自测题**

1.（单选）压测时 CPU 才 40% 但 QPS 上不去，Buffer Pool 命中率 88%，瓶颈在？
A. CPU  B. 内存  C. 网络  D. 连接数
<details><summary>参考答案</summary>
B。命中率 88% 低于 95% 阈值，说明大量请求要回磁盘，是内存瓶颈。
</details>

2.（判断）压测报告写"QPS 8000 表现良好"就够了，不用看其他指标。
<details><summary>参考答案</summary>
错。QPS 高不代表没瓶颈，要同时看 CPU、IO、命中率、锁等待，否则上线后真实流量下瓶颈暴露会导致告警。
</details>

3.（单选）`innodb_buffer_pool_size` 生产环境建议设置为？
A. 物理内存 10%  B. 物理内存 50% 到 70%  C. 固定 128MB  D. 物理内存 90%
<details><summary>参考答案</summary>
B。生产建议物理内存的 50% 到 70%，默认值只有 128MB 远不够。
</details>

4.（判断）`sysbench` 的 `oltp_point_select` 场景主要压测 Buffer Pool 命中率上限。
<details><summary>参考答案</summary>
对。纯主键查询走聚簇索引，命中率最高，能压出 Buffer Pool 的上限表现。
</details>

5.（单选）单表数据量超过多少应考虑分库分表？
A. 10 万行  B. 100 万行  C. 2000 万行  D. 1 亿行
<details><summary>参考答案</summary>
C。业界经验单表 500 到 2000 万行后 B+ 树层级变深、维护成本上升，应考虑分表。
</details>

简答：压测发现 QPS 在并发 64 到 128 之间不再上升，如何定位是哪类瓶颈？
<details><summary>参考答案</summary>
采集九维度指标对照判定规则：CPU 持续高于 80% 是计算瓶颈；IO 的 `%util` 接近 100% 是磁盘瓶颈；Buffer Pool 命中率低于 95% 是内存瓶颈；`innodb_row_lock_waits` 增长是并发瓶颈。再按类型选优化路径：计算瓶颈加索引或读写分离，内存瓶颈扩 Buffer Pool，IO 瓶颈换 SSD，并发瓶颈打散热点。
</details>
