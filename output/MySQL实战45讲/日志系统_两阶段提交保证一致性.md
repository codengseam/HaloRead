# 第 04 章：两阶段提交保证一致性

> 前置知识：redo log / binlog 基本概念、WAL 机制、事务 ACID
> 学完你能：① 在面试中画出两阶段提交时序图并解释每一步崩溃后的恢复策略 ② 在生产中根据业务对数据丢失的容忍度选择 sync_binlog / innodb_flush_log_at_trx_commit 组合

## 概念

两阶段提交（Two-Phase Commit，2PC）是 MySQL 保证 redo log 与 binlog 一致性的提交协议。林晓斌在《MySQL 实战 45 讲》第 02 讲中强调：redo log 是 InnoDB 引擎层的物理日志，binlog 是 Server 层的逻辑日志，两者分属不同层、独立写入，必须有协议约束才能在崩溃后恢复出一致的状态。

大白话：单写 redo log 不行——备库拉不到 binlog 就同步不上；单写 binlog 也不行——主库自己崩溃恢复时数据丢了。两阶段提交就是先让 redo log 进入"准备态"（prepare），等 binlog 写完落盘了，再让 redo log 转正（commit），用 binlog 是否完整来决定崩溃时事务的去留。

边界：

- 2PC 解决的是"同一台机器上 redo log 与 binlog 的一致性"，不是分布式事务的 2PC（后者是跨节点协调）。
- 2PC 只覆盖事务提交路径，不覆盖纯查询。
- binlog 与 redo log 的一致性是"逻辑一致"——崩溃恢复后状态一致，不保证两者内容字节对齐。

易混淆：XA 事务也用 2PC 这个词，但那是跨 MySQL 实例的分布式事务协调，由协调者驱动多个分支；本章讲的是单实例内部的 redo/binlog 两阶段提交，两者别混。

## 原理

### 提交流程

以一条 UPDATE 为例，提交阶段的时序：

1. InnoDB 写 redo log，状态置为 PREPARE
2. Server 层写 binlog 到 binlog cache，再 write 到文件系统 page cache
3. 调用 fsync 将 binlog 持久化到磁盘
4. InnoDB 写 redo log commit 标记，状态置为 COMMIT
5. 返回客户端成功

伪代码（简化）：

```text
redo_log.write(PREPARE, txn)
binlog.write(txn)
binlog.fsync()
redo_log.write(COMMIT, txn)
return OK
```

关键在第 3 步：binlog 是否 fsync 决定了崩溃后事务的归属。

### 为什么必须是两阶段

反证两种单阶段方案：

**方案 A：先写 redo log，再写 binlog。** 崩溃在两步之间，主库用 redo log 恢复出事务（c=1），但 binlog 没写，备库同步不到这条变更（c=0），主备不一致。

**方案 B：先写 binlog，再写 redo log。** 崩溃在两步之间，主库 redo log 没写、事务无效（c=0），但备库 binlog 已写、应用后 c=1，主备不一致。

2PC 用 PREPARE 这个中间态把两份日志"绑"在一起，崩溃后用 binlog 完整性来仲裁事务去留，从而消除两份日志的不一致窗口。

### 崩溃恢复决策

InnoDB 重启时扫描 redo log，对每个事务按状态判定：

| redo log 状态 | binlog 状态 | 恢复动作 |
|--------------|------------|---------|
| COMMIT | — | 重做（事务已确认提交） |
| PREPARE | 完整（fsync 过、有 XID） | 提交（视为 binlog 已生效） |
| PREPARE | 不存在/不完整 | 回滚（事务作废） |

判定 binlog 是否完整的依据是事务对应的 XID（事务 ID）能否在 binlog 中找到完整记录。林晓斌在第 02 讲中把这个判定描述为"用 binlog 给 redo log 兜底"——binlog 在则提交，不在则回滚。

### 为什么用 binlog 仲裁而不是 redo log

因为主备一致性以 binlog 为准：备库只认 binlog。如果 redo log 提交了但 binlog 没写，主库有、备库没有，就是数据丢失；而 binlog 写了 redo log 没提交，主库回滚后用 binlog 重做即可对齐。所以"binlog 在则提交"是逻辑上唯一自洽的选择。

## 实践

### 双 1 与参数取值

涉及两个参数：

**innodb_flush_log_at_trx_commit**（控制 redo log 落盘）：

- `1`：每次事务提交都 fsync（最安全，双 1 之一）
- `0`：每秒由后台线程 fsync，提交时不刷（崩溃丢最近 1 秒）
- `2`：每次提交只 write 到 page cache，每秒 fsync（OS 崩溃丢最近 1 秒）

**sync_binlog**（控制 binlog 落盘）：

- `1`：每次提交都 fsync（最安全，双 1 之一）
- `0`：由 OS 决定 fsync 时机（崩溃丢未 fsync 部分）
- `N`（N>1）：累积 N 个事务后 fsync（崩溃丢最近 N 个）

金融级生产配置：

```sql
SET GLOBAL innodb_flush_log_at_trx_commit = 1;
SET GLOBAL sync_binlog = 1;
```

这就是"双 1"——每次提交 redo log 与 binlog 各 fsync 一次。代价是每事务两次刷盘，IOPS 开销大。

### 性能与可靠性的取舍

| 场景 | innodb_flush_log_at_trx_commit | sync_binlog | 风险 |
|------|-------------------------------|-------------|------|
| 金融核心 | 1 | 1 | 无（双 1） |
| 电商交易 | 1 | 100~1000 | 崩溃丢少量 binlog |
| 日志型业务 | 2 | 1000 | OS 崩溃丢 1 秒 redo + 少量 binlog |
| 测试环境 | 0 或 2 | 0 | 数据可能丢，仅供测试 |

非双 1 配置必须配合组提交（binary log group commit）来降低 fsync 次数：binlog_group_commit_sync_delay（延迟多少微秒后 fsync）与 binlog_group_commit_sync_no_delay_count（累积多少事务后 fsync），两者满足其一触发。组提交让多个事务共享一次 fsync，是双 1 性能救星。

### 教科书做法 vs 生产做法

- 教科书：双 1 最安全，所有业务都开。
- 生产：金融业务开双 1，非核心业务用 sync_binlog=100~1000 换性能；并配合组提交延迟（如 binlog_group_commit_sync_delay=1000 微秒）攒批，把 fsync 摊薄。监控指标是 IOPS 利用率与平均事务延迟，而非盲目上双 1。

### 常见坑

**坑 1：开了 sync_binlog=1 但 innodb_flush_log_at_trx_commit=0。**
解析：这不是双 1，redo log 每秒刷一次，崩溃丢 1 秒。两个参数必须同时为 1 才叫双 1。

**坑 2：双 1 后 QPS 砍半，直接回滚参数。**
解析：先开组提交（binlog_group_commit_sync_delay）攒批，再观察。多数场景能找回一半性能。

**坑 3：以为两阶段提交能防主从延迟。**
解析：2PC 只保证日志一致，不保证备库实时同步。备库延迟由 io_thread/sql_thread 消费速度决定，是另一个问题（见第 24 讲）。

**坑 4：崩溃后 redo log 处于 PREPARE、binlog 完整，但事务仍被回滚。**
解析：检查 binlog 是否真的 fsync 过、XID 是否匹配。如果 sync_binlog≠1，PREPARE 状态对应的 binlog 可能没落盘，恢复时按"binlog 不存在"处理，事务回滚——这正是双 1 的意义。

**坑 5：把内部 2PC 当成 XA 分布式事务。**
解析：本章 2PC 是单实例 redo/binlog 协调；XA 是跨实例分布式事务，由 XA START/END/PREPARE/COMMIT 驱动。两套东西，面试别答串。

### 跨版本差异

- 8.0 默认 binlog_format=ROW，5.7 默认 STATEMENT，升级时务必显式设置。
- 8.0.30+ redo log 容量统一用 innodb_redo_log_capacity 管理，旧的 innodb_log_file_size/innodb_log_files_in_group 废弃。
- 组提交参数在 5.7 与 8.0 行为一致，但 8.0 默认开启 binlog_transaction_dependency_tracking 提升并行复制。

## 速查/自测

### 两阶段提交速查表

| 阶段 | 动作 | 日志状态 | 崩溃后 |
|------|------|---------|--------|
| 1 | 写 redo log | PREPARE | 看 binlog 决定 |
| 2 | 写 binlog + fsync | — | redo 仍 PREPARE |
| 3 | 写 redo log commit | COMMIT | 重做事务 |

### 参数速查

| 参数 | 取值 | 含义 |
|------|------|------|
| innodb_flush_log_at_trx_commit | 1 | 每事务 fsync（双 1） |
| innodb_flush_log_at_trx_commit | 0 | 每秒 fsync |
| innodb_flush_log_at_trx_commit | 2 | 每事务 write，每秒 fsync |
| sync_binlog | 1 | 每事务 fsync（双 1） |
| sync_binlog | 0 | OS 控制 fsync |
| sync_binlog | N | 每 N 事务 fsync |

### 自测三问

1. 崩溃时 redo log 处于 PREPARE、binlog 已完整 fsync，重启后这个事务会被提交还是回滚？为什么？
2. 双 1 配置下，每条事务提交要做几次 fsync？分别刷的是什么？
3. sync_binlog=1 但 innodb_flush_log_at_trx_commit=0，这套配置算双 1 吗？崩溃会丢什么？
