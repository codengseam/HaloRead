# 第 31 章：订单表设计

> 前置知识：InnoDB 聚簇索引、B+ 树页分裂、联合索引最左匹配、分库分表
> 学完你能：① 设计高并发订单表的字段、主键、索引 ② 用雪花算法生成订单号 ③ 制定分库分表与冷热分离方案

## 概念

订单表是电商系统的核心表，也是面试高频考点。一张订单表要扛住日增百万级写入、亿级历史数据、多维度的查询（按用户、按商家、按状态、按时间），设计上要在"写入性能、查询效率、可扩展性"三者间找平衡。SDET 在评审表结构时，要能一眼看出字段类型、主键、索引、分表策略的隐患。

订单表设计的核心矛盾是：主键要保证写入顺序避免页分裂，索引要覆盖多维度查询但不能太多（影响写入），数据量增长后单表扛不住要分表但分表后跨片查询又难做。所以订单表设计不是一次性建表，而是一套"字段、主键、索引、分表、归档"的演进方案。

林晓斌在《MySQL 实战 45 讲》第 30 讲"用动态的观点看加锁"一讲里强调，表结构和索引设计直接决定锁的范围和并发能力，订单表这类高并发写入表尤其要考虑锁的粒度。

## 原理

### 字段设计

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `BIGINT UNSIGNED AUTO_INCREMENT` | 自增主键，保证顺序写入 |
| `order_no` | `BIGINT` | 雪花算法生成，业务唯一键 |
| `user_id` | `BIGINT` | 用户 ID，分片键 |
| `merchant_id` | `BIGINT` | 商家 ID |
| `amount` | `DECIMAL(15,2)` | 金额，禁用 FLOAT |
| `status` | `TINYINT` | 订单状态，0 待支付、1 已支付、2 已发货 |
| `create_time` | `DATETIME` | 创建时间 |
| `remark` | 拆到子表 | 长文本避免影响主表 IO |

金额必须用 `DECIMAL(15,2)`，浮点类型有精度损失，财务场景容不得错。状态用 `TINYINT` 而非 VARCHAR，存储小、索引快。订单号用雪花算法：64bit 拆成 1bit 符号、41bit 时间戳、10bit 机器、12bit 序列，保证全局唯一且趋势递增。

除了核心字段，订单表还要留 `is_deleted` 软删除标记（用 `TINYINT`），物理删除会让对账和退款找不到记录；`update_time` 记录最近修改时间，配合 binlog 做增量同步。这两个字段是订单表常被忽略却很重要的设计点，缺了对账和风控都会回头找你补数据。

### 主键与索引

主键选自增 `BIGINT`，不要用 UUID。UUID 无序，插入会引发 B+ 树页分裂，写入性能随数据量增长急剧下降。自增主键顺序写入，页填充率高。

索引设计：

```sql
KEY idx_user_create (user_id, create_time),        -- 用户订单列表
KEY idx_order_no (order_no),                        -- 订单号查询
KEY idx_status_create (status, create_time),        -- 按状态加时间筛选
KEY idx_merchant_create (merchant_id, create_time)  -- 商家维度
```

`(user_id, create_time)` 覆盖最高频的"我的订单"查询；`(status, create_time)` 支持后台按状态筛选；订单号单独建索引。索引不是越多越好，每多一个索引写入就要多维护一棵 B+ 树，订单表写入频繁，索引数量最好控制在 5 个以内。

### 分库分表与冷热分离

单表数据量超过 2000 万行后，B+ 树层级变深、维护成本上升，应分表。订单表按 `user_id` 取模分表，比如分 64 张表，`table_index = user_id % 64`，保证同一用户的订单落在同一张表，用户维度查询不跨片。

冷热分离：3 个月以上的已完成订单归档到历史表，主表只保留热数据。归档用 `INSERT INTO order_history SELECT ... WHERE create_time < ...` 加 `DELETE`，分批执行避免大事务。

读写分离：报表类查询（按商家统计、按时间汇总）打到从库，不压主库。可在应用层冗余 `user_name` 字段减少 `JOIN`，用空间换时间。

## 实践

**面试场景**

面试官问："设计一张日订单 100 万的订单表。"

回答模板："字段上 `id` 自增 BIGINT 主键、`order_no` 雪花算法 BIGINT、`amount` 用 `DECIMAL(15,2)`、`status` 用 TINYINT、长文本拆子表。索引建 `(user_id, create_time)`、`(order_no)`、`(status, create_time)`、`(merchant_id, create_time)` 四个。分表按 `user_id` 取模分 64 张表，保证用户维度不跨片。冷热分离把 3 个月以上订单归档历史表，报表查询走从库。日订单 100 万、单表到 8000 万时分 64 表，每表约 125 万行，性能稳定。"

追问"为什么不用 UUID 做主键？"——UUID 无序，插入引发 B+ 树页分裂，页填充率低、写入变慢、磁盘空间浪费；自增主键顺序写入避免这些问题。

追问"分表后按商家查询怎么办？"——商家维度查询是跨片查询，两种方案：一是建商家维度的汇总表（异构索引），二是用 ES 等搜索引擎做二级索引，订单写入时同步双写。

**项目场景**

某电商日订单 100 万，原设计单表，运行两年后单表 8000 万行，"我的订单"查询 P99 飙到 2s。重构方案：按 `user_id` 取模分 64 表，分表前用 `pt-archiver` 把 3 个月以上订单归档到 `order_history`，主表瘦身到 1500 万行。分表后用户查询走单表 `(user_id, create_time)` 索引，P99 降到 50ms；商家报表查询走从库加异构索引表。重构期间双写新老表灰度切换，数据校验用 SDET 编写的对账脚本比对一致性。

**避坑**

- 分表键一旦选定不可轻易更换，选 `user_id` 而非 `order_no`，因为用户维度查询最高频。
- 雪花算法的时钟回拨会生成重复 ID，要有时钟回拨检测和等待机制。
- 归档删除要分批，单次大事务会撑爆 undo 段。

## 速查/自测

**速查表**

| 设计点 | 方案 | 理由 |
|---|---|---|
| 主键 | 自增 BIGINT | 顺序写入，避免页分裂 |
| 订单号 | 雪花算法 BIGINT | 全局唯一、趋势递增 |
| 金额 | `DECIMAL(15,2)` | 避免浮点精度损失 |
| 分表键 | `user_id` 取模 | 用户维度不跨片 |
| 归档 | 3 个月以上进历史表 | 主表保热数据 |

**自测题**

1.（单选）订单表主键应选？
A. UUID  B. 自增 BIGINT  C. VARCHAR 订单号  D. MD5 哈希
<details><summary>参考答案</summary>
B。自增 BIGINT 顺序写入，避免 B+ 树页分裂；UUID 无序会引发页分裂降低写入性能。
</details>

2.（判断）订单金额可以用 `FLOAT` 存储，反正只是展示。
<details><summary>参考答案</summary>
错。FLOAT 有精度损失，财务场景必须用 `DECIMAL(15,2)` 保证精确。
</details>

3.（单选）雪花算法 64bit 的组成是？
A. 1 符号加 41 时间戳加 10 机器加 12 序列  B. 全部时间戳  C. 随机数  D. 1 符号加 63 随机
<details><summary>参考答案</summary>
A。1bit 符号、41bit 时间戳、10bit 机器、12bit 序列，保证全局唯一且趋势递增。
</details>

4.（判断）订单表分表应按 `order_no` 取模，因为订单号是唯一键。
<details><summary>参考答案</summary>
错。应按 `user_id` 取模，用户维度查询最高频，按 `user_id` 分片保证同一用户订单不跨片；按 `order_no` 分片会让用户查询跨多片。
</details>

5.（单选）单表数据量超过多少应考虑分表？
A. 10 万  B. 50 万  C. 2000 万  D. 10 亿
<details><summary>参考答案</summary>
C。单表 500 到 2000 万行后 B+ 树变深、维护成本上升，业界经验 2000 万行考虑分表。
</details>

简答：分表后商家维度查询如何解决？
<details><summary>参考答案</summary>
两种方案：一是建商家维度的异构索引汇总表，订单写入时双写；二是用 ES 等搜索引擎做二级索引，订单变更同步到 ES。查询时先查异构表或 ES 拿到订单号，再回主表取详情。
</details>
