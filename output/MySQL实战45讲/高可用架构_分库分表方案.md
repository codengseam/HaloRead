# 第 26 章：分库分表方案

> 前置知识：第 25 章主从复制、B+树索引、单表性能瓶颈
> 学完你能：①讲清四种拆分维度与各自适用场景 ②为订单表设计分片方案并选对 sharding key ③说清分库分表带来的五大副作用及对策

## 概念

分库分表是当单库单表撑不住业务量时，把数据拆到多个库或多张表的工程手段。林晓斌在《MySQL 实战 45 讲》里围绕主备与复制讨论高可用，但单库容量天花板是另一道坎——B+树高度、单表行数、单库 QPS 到了瓶颈，复制救不了，必须拆。

大白话：单表 8000 万行，查询慢、DDL 慢、写入也吃力，那就按某个键把数据打散到 64 张表甚至 64 个库，每张表降到百万级，查询自然快。难点不在“拆”，而在拆完之后跨库 JOIN、分布式事务、全局 ID、扩容迁移这一堆连锁问题。

边界：分库分表是“治标也治本”的容量方案，但代价是放弃单库事务和跨表查询的便利。能靠索引优化、读写分离、归档冷数据解决的，不要急着分库分表——一旦拆了，回不去。

## 原理

### 四种拆分维度

按拆分方向和粒度，分四种：

- **垂直分库**：按业务拆库，如订单库、用户库、商品库分开。本质是微服务化的数据库层落地。
- **垂直分表**：把一张宽表按字段拆成多张，如把 `user` 表的 `profile_text` 大字段拆到 `user_ext` 表。冷热字段分离，减少 IO。
- **水平分库**：按 sharding key 把同一张表的数据路由到不同数据库实例，如 `user_id % 4` 分到 4 个库。
- **水平分表**：同一个库内把一张表拆成多张，如 `order_0` 到 `order_63`。

垂直拆分解决“字段太多/业务耦合”，水平拆分解决“行数太多”。生产里常组合：先垂直分库理清业务，再水平分库分表扛量。

### 时机判断

不是数据一多就拆。常见触发线：

- 单表 500-2000 万行后 B+树层级、BufferPool 命中率下降，考虑分表。
- 单库 QPS > 5000，或磁盘 IO 饱和，考虑分库。
- 单表 DDL 耗时不可接受（如大表加索引几小时）。

低于这些线优先优化索引、SQL、参数；越过才上分库分表。

### sharding key 与路由

sharding key（分片键）决定一行数据落到哪个分片，是整个方案的命门。选错会导致大量跨分片查询。

常见路由策略：

| 策略 | 做法 | 特点 |
|------|------|------|
| 取模 | `shard = key % N` | 数据均匀，但扩容要 rehash 迁移全量 |
| 范围 | 按区间分片，如 0-100 万一片 | 扩容友好，但易热点 |
| 一致性哈希 | 节点在哈希环上分布 | 扩容只迁移相邻数据，实现复杂 |

sharding key 选择原则：查询高频且能带上、写入均匀分布。订单系统常用 `user_id`（按用户分片，买家维度查询多）；若要按商家查，则需广播或异构索引。

### 中间件

主流三类：

- **ShardingSphere**（Apache 顶级项目）：JDBC 模式无侵入、Proxy 模式对应用透明，国内生态最成熟。
- **MyCat**：早期 Proxy 方案，社区活跃度下降。
- **Vitess**：Google 出品，云原生，支撑 YouTube 规模，运维门槛高。

### 五大副作用

拆完不是终点，是新问题起点：

1. **跨库 JOIN**：分片后无法直接 JOIN。解法：绑定表（同分片键的表共置）、广播表（小表全量冗余）、应用层组装、ES 异构索引。
2. **分布式事务**：跨库写要 XA 或 TCC/Saga。XA 性能差，业务多用最终一致（消息+本地事务）。
3. **聚合查询**：`COUNT`、`SUM`、`GROUP BY` 要汇总各分片结果。`count(*)` 常用汇总表或离线 mapreduce。
4. **全局 ID**：自增主键失效，需分布式 ID 方案（见下）。
5. **扩容迁移**：取模分片从 4 库扩到 8 库要迁移全量数据，停机或双写灰度。

### 全局 ID 方案

| 方案 | 原理 | 优劣 |
|------|------|------|
| UUID | 本地生成随机 ID | 无中心、不递增，B+树插入碎片多 |
| Snowflake 雪花 | 时间戳+机器号+序列号 | 趋势递增、高性能，依赖时钟 |
| 号段模式 | DB 分配一段 ID 缓存到本地 | 递增、可控，需中心 DB |
| 数据库步长 | 每实例自增步长=N | 简单，扩容调整步长麻烦 |

Snowflake 最主流，但时钟回拨要处理；号段模式（如美团 Leaf）适合对连续性有要求的场景。

## 实践

### 面试场景

**问：订单表 8000 万行怎么分？**
回答模板：先说时机判断（单表超 2000 万、QPS 过万），再给方案：按 `user_id` 取模分 64 表（单表降到 125 万），sharding key 选 `user_id` 因为买家维度查询最多。最后说副作用对策：跨库 JOIN 用绑定表+ES 异构、全局 ID 用 Snowflake、`count(*)` 用汇总表。

**追问：为什么不用订单号分片？**
答：订单号若按时间或随机生成，按它分片会让买家查自己所有订单时跨遍所有分片。常见做法是订单号内嵌 `user_id` 后缀（如 `订单号 = 雪花算法 ID + user_id后4位`），既能按订单号定位分片，也能按 user_id 路由。

**追问：取模分片怎么扩容？**
答：取模扩容要 rehash 迁移全量，代价大。生产用“双倍扩容+一致性哈希”或预先多分片逻辑库映射到少量物理库，扩容时只搬逻辑库到新物理机，避免 rehash。

### 项目场景（SDET 视角）

订单表分 64 片上线前，QA 要验三件事：

1. **路由正确性**：构造同一 `user_id` 的多笔订单，断言全部落到同一分片；跨 `user_id` 断言分散。用 ShardingSphere 的 SQL 日志核对实际分片。
2. **扩容演练**：在预发环境模拟 64→128 分片迁移，验证数据零丢失、双写期间无重复。这是分库分表最容易翻车的环节，必须纳入回归。
3. **聚合查询对账**：分片后 `count(*)` 与迁移前单表总数比对，汇总表与各分片求和一致。

压测重点：分片键带不上的查询（如按商家统计）走 ES 异构，压测 ES 写入延迟和查询正确性，避免大促时 ES 落后导致报表错乱。

### 避坑

- **坑 1**：sharding key 选了 `create_time`。写入集中到当前时间所在分片，热点严重。时间适合做范围分片的辅助，不适合做取模主键。
- **坑 2**：分 4 表觉得够，半年后又爆。分表数量按 2-3 年容量预估，宁可一开始分 64 甚至 256，省去扩容痛苦。
- **坑 3**：分布式事务用 XA 求强一致，性能崩。业务允许就上最终一致（本地事务+消息队列），核心扣款才考虑 TCC。

## 速查/自测

### 速查表

| 拆分 | 解决 | 典型手段 |
|------|------|---------|
| 垂直分库 | 业务耦合 | 按域拆库 |
| 垂直分表 | 字段过多 | 冷热字段分离 |
| 水平分库 | 单库 QPS 瓶颈 | sharding key 路由多实例 |
| 水平分表 | 单表行数瓶颈 | 同库多表 |

### 自测题

1. 判断：单表 1000 万行就必须分库分表。
2. 取模分片从 4 库扩到 8 库，需要迁移多少数据？为什么？
3. 简答：Snowflake 方案有什么隐患？怎么解决？
4. 简答：分库分表后 `count(*)` 怎么实现？给出两种方案。

<details><summary>参考答案</summary>

1. 错。500-2000 万是参考线，1000 万若索引合理、QPS 不高未必需要拆；优先优化索引、SQL、归档冷数据，越过瓶颈再拆。
2. 几乎全量。取模分片 `key%4` 改 `key%8` 后，约一半数据要迁到新分片；这是取模路由的扩容痛点，所以常预分多逻辑库或用一致性哈希缓解。
3. 依赖机器时钟，时钟回拨会生成重复或倒退 ID。解决：回拨时拒绝服务或等待时钟追平；机器号分配用 ZooKeeper/Consul 防重复。
4. 两种：①汇总表——写入时同步更新一张全局计数表，查询直接读，注意分布式事务保证一致；②离线计算——各分片定时 dump 到数仓或 ES，mapreduce 聚合，准实时但有延迟。
