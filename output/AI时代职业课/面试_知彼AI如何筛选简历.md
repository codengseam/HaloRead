# 知彼 AI 如何筛选简历

## 一、你投出去的简历，先被谁读

2026 年，超过 **70%** 的 200 人以上规模企业已经引入 AI 简历筛选（来源：Moka，2026）。这意味着你投出去的简历，**第一关不是 HR，是模型**。

更扎心的一个数字：招聘专员每份简历的初筛时间只有 **6-8 秒**（来源：Moka，2026）。在 AI 接管之前，6-8 秒决定了你能不能进面试池；AI 接管之后，这 6-8 秒被压缩到毫秒级，但筛选逻辑从"扫一眼"变成了"算一遍"。

所以"知彼"的第一步，是搞清楚这一遍是怎么算的。你不知道算法，就只能靠运气；你知道算法，才能知道该往哪个方向使劲。

## 二、四个环节，串成一条流水线

AI 简历筛选不是一步到位的"打分"，而是一条四环节流水线（来源：Moka，2026-04-21，mokahr.com/blog/32906.html）。

### 2.1 环节一：解析（Parsing）

系统用 NLP + OCR 把你的简历拆成结构化字段。一份简历通常被抽出 **100+ 个字段**，覆盖基本信息、教育背景、工作经历、技能标签、项目经历、证书等。行业报告给出的字段抽取准确率在 **95%** 左右（来源：Moka，2026）。

这一步的关键不是"抽没抽出来"，而是**抽得对不对**。

- 你写"3 年 Java 经验"，模型要识别成"工作经验 = 3 年 + 技能 = Java"，而不是把"3 年 Java"当成一个完整短语。
- 你写"主导 Q3 营收增长 40%"，模型要识别成"业绩 = 营收增长 40% + 角色 = 主导"，而不是当成一句普通描述。

简历模板越花哨，解析出错率越高。**双栏布局、图片化技能图标、嵌入式图标**，都是 OCR 的天敌。一份在浏览器里好看的双栏简历，到了解析引擎里可能被切成无法理解的碎片。

### 2.2 环节二：语义理解（Semantic Understanding）

这一步是 AI 筛选区别于传统 ATS（Applicant Tracking System）的核心。

传统 ATS 靠关键词匹配：JD 里写"Python"，你简历里没有"Python"这个词，直接淘汰。问题是——你写的是"Py"或者"熟练使用 Python 3.10+"，传统 ATS 抓不到。

AI 筛选用 **Sentence-BERT** 之类的模型，把简历和 JD 都编码成向量，再算**余弦相似度**。这样即便你写"三年后端开发，主力语言 Python"，JD 写"需要 Python 工程师"，系统也能识别出语义接近。

行业数据显示，语义匹配相比关键词匹配，能**降低约 35% 的误筛率**（来源：Moka，mokahr.com/blog/37541.html）。这 35% 意味着——过去因为"用词不一致"被刷掉的好候选人，现在能被捞回来。

### 2.3 环节三：人岗匹配（Matching）

语义相似度只是"相关不相关"，匹配还要看"够不够格"。这一步把硬条件和软能力分开算。

- **硬条件**：学历、年限、必备技能、地域、语言。这一类是布尔判断，过/不过。
- **软能力**：沟通、抗压、协作、学习能力。这一类靠语义推断，从你简历里的行为描述反推（比如"主导跨部门项目"→ 协作能力 +1）。

### 2.4 环节四：智能排序（Ranking）

匹配完，系统给每个候选人打一个综合分，按分数从高到低排序。HR 看到的是一份**已经排好序的列表**，而不是原始简历堆。

排序的依据，就是下一节要讲的 ATS Score。

## 三、ATS Score：那个决定你命运的数字

学术界已经把 AI 简历筛选的打分逻辑公式化了。一份发表在 IJVRA（International Journal of Vocational and Recursive Assessment）的论文给出了这样的公式（来源：IJVRA，Resume Shortlisting with Transformers，ijpub.org/ijvra/papers/IJVRA2603597.pdf）：

> **ATS Score = round(100 × cos(r, j), 2)**

其中：

- `r` 是简历向量；
- `j` 是岗位 JD 向量；
- `cos(r, j)` 是两者的余弦相似度；
- `round(..., 2)` 是保留两位小数。

分数区间 0-100，按推荐等级划分（论文给出的推荐等级划分，来源：IJVRA）：

| 等级 | 分数区间 | 含义 |
|---|---|---|
| **S** | ≥ 90 | 极力推荐 |
| **A** | 80–89 | 推荐 |
| **B** | 70–79 | 可考虑 |
| **C** | 60–69 | 谨慎考虑 |
| **D** | < 60 | 不推荐 |

关键阈值是 **80 分**：**≥ 80 分进入 Highly Recommended 池**，HR 几乎一定会看；70-79 分进备选池，看 HR 心情和时间；60 分以下基本等于没投。

这意味着什么？**你的简历不是被"读"，是被"算"。** 你的目标不是让 HR 觉得你不错，而是让 `cos(r, j)` 算出来 ≥ 0.80。这是一个数学问题，不是一个文学问题。

## 四、语义匹配到底改变了什么

理解了 ATS Score，你就能理解语义匹配相比关键词匹配的三个本质变化。

### 4.1 关键词堆砌失效

过去 SEO 式的简历写法——在简历末尾堆一行"Java Python Go Kafka Redis MySQL Docker K8s"——对传统 ATS 有效，对 AI 筛选基本无效。因为向量是整篇简历的语义平均，堆砌的关键词会被稀释，反而把简历的语义重心拉向"技术名词罗列"，离具体岗位的 JD 越来越远。

### 4.2 同义表达被识别

你写"用户增长"，JD 写"获客"；你写"模型评测"，JD 写"Model Evaluation"——AI 都能识别为语义相近。这降低了"用词不一致"导致的误筛，但也提高了对**内容质量**的要求：你得真的写过相关的东西，才能在语义上接近。光靠同义词替换已经骗不过去了。

### 4.3 上下文权重被引入

"做了 3 年 Python"和"用 Python 重写了核心交易系统，TPS 提升 5 倍"——两句话都包含 Python，但向量完全不同。后者在语义上更接近"高级 Python 工程师"的 JD。**语境比关键词更重要。** 一句带数字、带成果、带技术栈的描述，其向量权重远高于一句干巴巴的经历陈述。

## 五、AI 筛选怎么识别"AI 工具使用能力"

2026 年的 JD 里，"有 AI 工具使用经验"几乎成了标配。AI 筛选系统也进化出了识别这一能力的能力（来源：Moka，2026）。

系统主要识别三类信号：

1. **工具使用**：Copilot、Cursor、Claude Code、Windsurf 等 AI 编程工具的使用记录。
2. **方法实践**：Prompt Engineering、RAG（检索增强生成）、Fine-tuning、Agent 编排等关键词。
3. **代表作验证**：GitHub Star 数、PR 贡献记录、Kaggle / 天池竞赛排名。

第三类最硬。AI 筛选系统会通过 GitHub API 拉取公开数据，验证你简历里写的"开源贡献 500 Star"是不是真的。**写假数据在这个环节会被直接识别**。

需要提醒的是，"AI 工具使用能力"的识别不是看你写了多少个工具名，而是看你的经历里有没有**用这些工具产生过可量化的成果**。光列"熟练使用 Cursor"没用，要写"用 Cursor 重构历史模块，代码量减少 30%，CR 通过率提升"——后者才有语义信号。

## 六、专业代表作怎么验证

"代表作验证"是 AI 筛选里最容易被低估的一环。系统不只看你写没写，还看你**写的是不是真的**。

### 6.1 GitHub 类

- **Star 数**：系统拉取仓库的真实 Star 数，跟你简历写的一致才计入。
- **PR 贡献**：系统识别你是不是真的提交过 PR、被合并了几个。**只 Fork 不 PR** 的仓库基本无效。
- **Commit 频率**：长期低频贡献比短期高频刷 commit 更被认可——后者是典型的刷量特征。

### 6.2 竞赛类

- Kaggle、天池、DataFountain 等平台的排名是公开可查的。
- 系统识别"Top 10%"和"参与过"是两回事——后者在 ATS 里几乎不加分。

### 6.3 技术博客类

- 掘金、知乎、个人博客的阅读量、点赞数被识别为"影响力信号"。
- 但系统会过滤掉刷量特征（短期高频点赞、互动率异常）。

## 七、本章边界

本篇讲的是 **AI 简历筛选的工作机制**：四环节流水线、ATS Score 公式、语义匹配的变化、AI 工具能力识别、代表作验证。

**本篇不讲**：候选人应该怎么写简历、怎么沉淀经历、怎么应对筛选。这些是"知己"篇的内容——见《知己如何丰富个人经历》和后续的简历呈现篇。

记住一句话：**知彼是看清对手，知己才是武装自己。** 看清对手之后，下一站是回到自己。知道算法怎么算，你才知道自己该补什么、该写什么、该藏什么——但具体怎么补怎么写，得等"知己"篇展开。
