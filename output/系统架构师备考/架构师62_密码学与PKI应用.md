# 第62章：密码学与PKI应用

密码学是中级信安工程师的"硬骨头"，也是分值最密集的一章——对称/非对称/哈希/签名/PKI/TLS 一套下来 10-15 分，且常出计算题。这章对架构师同样关键：HTTPS 怎么部署、国密算法怎么选、密钥怎么管理，都是系统设计里的实际决策。这章把密码学体系从底层算法到上层 PKI/TLS 应用一次讲透，配上国密清单和 TLS 握手流程，背完能直接答题，也能直接用于架构选型。

## 概念

### 这章考什么

- **科目**：中级信安工程师综合知识高频；高级架构师综合知识+案例偶现
- **考频**：中级信安约 10-15 分，密码学计算题必考
- **分值**：综合知识 10-15 分（算法对比+PKI+TLS+计算题）
- **难度**：★★★★（算法多、分组模式易混、TLS 握手流程长、计算题易错）

### 密码学全景

密码学解决两类问题：**加密**（保护机密性）和**认证**（保证完整性+抗抵赖）。三大类算法各有分工：对称加密管"快"（大数据量）、非对称加密管"分发+签名"（小数据量+身份）、哈希管"指纹"（完整性）。PKI 在非对称加密之上构建信任体系，TLS 把这些组合起来保护传输。理解这个分工，整章脉络就清楚了。

## 原理

### 【中级重点】对称加密与分组模式

对称加密（Symmetric Encryption）加密解密用同一把密钥，速度快，适合大数据量。

| 算法 | 密钥长度 | 分组长度 | 特点 |
|---|---|---|---|
| **DES** | 56 位 | 64 位 | 已被破解，不安全 |
| **3DES** | 168 位（3×56） | 64 位 | DES 三遍，过渡方案，慢 |
| **AES** | 128/192/256 位 | 128 位 | 当前主流，安全且快 |
| **SM4** | 128 位 | 128 位 | 国密对称算法 |

对称加密最大难题是**密钥分发**——双方怎么安全地把同一把钥匙传给对方？这正是非对称加密要解决的。

**分组密码工作模式**（一个分组算法怎么处理多块数据）：

| 模式 | 全称 | 特点 |
|---|---|---|
| **ECB** | 电子密码本 | 相同明文产生相同密文，不安全，禁止用于重复数据 |
| **CBC** | 密文分组链接 | 前块密文异或后块明文，需 IV，串行不可并行 |
| **CTR** | 计数器 | 用计数器加密后异或明文，可并行，流式 |
| **CFB** | 密文反馈 | 把分组密码当流密码用 |
| **OFB** | 输出反馈 | 类似 CFB，反馈的是加密器输出 |
| **GCM** | 伽罗瓦/计数器 | CTR+认证标签，提供加密+完整性，推荐 |

ECB 是经典考点：相同明文块加密后相同，会泄露数据模式（著名的"企鹅图"加密后还是企鹅轮廓）。GCM 是当前推荐模式，同时提供机密性和完整性。CBC 需要随机 IV，IV 相同会导致安全问题。

### 【中级重点】非对称加密与国密 SM2

非对称加密（Asymmetric Encryption）每个用户一对密钥：**公钥公开+私钥保密**。公钥加密只能私钥解密，私钥签名公钥验签。

| 算法 | 数学基础 | 密钥长度 | 特点 |
|---|---|---|---|
| **RSA** | 大整数分解难题 | 2048 位起步 | 最经典，慢 |
| **ECC** | 椭圆曲线离散对数 | 256 位（等价 RSA 3072） | 密钥短，适合移动端 |
| **SM2** | 椭圆曲线 | 256 位 | 国密非对称算法 |
| **SM9** | 标识加密（IBE） | — | 国密标识密码，用标识作公钥 |

非对称加密解决密钥分发问题，支持数字签名，但计算慢，不适合大数据量。实际系统几乎都用**混合加密**——用非对称加密协商一把对称会话密钥，再用对称加密传数据。HTTPS 就是这么干的：RSA/ECDHE 协商会话密钥，AES 加密报文。

### 【中级重点】哈希函数与数字签名

**哈希函数**（Hash Function）把任意长输入压缩成定长摘要，单向不可逆，用于完整性校验。两个核心性质：**抗原像性**（由哈希推不出原文）、**抗碰撞性**（找不到两个不同输入有相同哈希）。

| 算法 | 摘要长度 | 安全性 |
|---|---|---|
| **MD5** | 128 位 | 已不安全，可碰撞 |
| **SHA-1** | 160 位 | 已不安全，2017 被攻破 |
| **SHA-256** | 256 位 | 安全（SHA-2 家族） |
| **SHA-512** | 512 位 | 安全（SHA-2 家族） |
| **SM3** | 256 位 | 国密哈希算法 |

**数字签名**：发送方用私钥对哈希值加密生成签名，接收方用公钥验签。签名保证完整性+抗抵赖。

| 算法 | 类型 |
|---|---|
| **RSA 签名** | 基于大数分解 |
| **DSA** | 数字签名算法（仅签名） |
| **ECDSA** | 椭圆曲线数字签名 |
| **SM2 签名** | 国密签名算法 |

签名流程：发送方对消息 M 求 H=Hash(M)，用私钥加密 H 得签名 S，把 M+S 一起发；接收方用公钥解密 S 得 H，再对 M 求 Hash(M')，比对 H 与 H' 一致则验签通过。

**消息认证码 MAC**：用密钥参与的哈希，同时保证完整性和身份。HMAC 是基于哈希的 MAC（如 HMAC-SHA256），与签名区别：MAC 共享密钥，不提供抗抵赖（双方都能生成）；签名用私钥，提供抗抵赖。

### 【中级重点】密钥管理生命周期

密钥和算法同样重要——密钥泄露则全盘皆输。密钥管理全生命周期：

1. **生成**：用密码学安全的随机数生成器（CSPRNG），不能用普通 rand
2. **分发**：对称密钥靠非对称加密或密钥协商（DH/ECDH）分发
3. **存储**：密钥加密存储（KEK 加密 DEK）、硬件安全模块 HSM
4. **更新**：定期轮换，降低泄露风险
5. **销毁**：彻底擦除，防止残留
6. **归档/恢复**：必要时备份恢复（如 CA 根密钥）

口诀：**"生分发存更销复"**——生成、分发、存档（存储）、更新、销毁、恢复。KEK（密钥加密密钥）保护 DEK（数据加密密钥）是分层密钥管理的基本模式。

### 【中级重点】PKI 公钥基础设施

PKI（Public Key Infrastructure）用数字证书把公钥和身份绑定，解决"公钥怎么确认属于谁"的问题。

**核心组件**：

| 组件 | 全称 | 作用 |
|---|---|---|
| **CA** | 证书机构 | 签发证书，信任根 |
| **RA** | 注册机构 | 审核申请者身份 |
| **CRL** | 证书撤销列表 | 已吊销证书黑名单 |
| **OCSP** | 在线证书状态协议 | 实时查询证书是否有效 |
| **CP/CPS** | 证书策略/认证业务声明 | 规定证书使用规则 |

**X.509 数字证书结构**：版本、序列号、签名算法、签发者、有效期、主体、主体公钥、CA 签名。证书链验证：终端证书→中间 CA→根 CA，逐级验签到受信任的根。

证书吊销两种方式：CRL（定期发布的黑名单，有时延）和 OCSP（实时查询，更快）。证书包含有效期，过期需续期，吊销则提前失效。

### 【中级重点】SSL/TLS 握手与版本差异

TLS（Transport Layer Security，前身 SSL）用 PKI+混合加密保护传输安全。TLS 1.2 握手流程：

1. **ClientHello**：客户端发送支持的密码套件、随机数、TLS 版本
2. **ServerHello**：服务端选定密码套件、返回随机数、版本
3. **Certificate**：服务端发送证书（含公钥）
4. **ServerKeyExchange**（视算法而定）：服务端密钥交换参数
5. **ServerHelloDone**：服务端问候结束
6. **ClientKeyExchange**：客户端用服务端公钥加密预主密钥发送（或 ECDHE 协商）
7. 双方用三个随机数生成主密钥→会话密钥
8. **ChangeCipherSpec**：切换到加密通信
9. **Finished**：双方发送加密的握手摘要验证完整性

**TLS 1.2 vs 1.3 差异**：

| 对比项 | TLS 1.2 | TLS 1.3 |
|---|---|---|
| 握手往返 | 2-RTT | 1-RTT（支持 0-RTT 恢复） |
| 支持算法 | RSA/DSA/部分 ECDHE | 仅 ECDHE/DHE（前向安全） |
| 静态 RSA 密钥交换 | 支持 | 移除 |
| 已弃用算法 | CBC/RC4/MD5/SHA-1 | 全部移除 |
| 加密 | AES-CBC/AES-GCM | AES-GCM/ChaCha20-Poly1305 |

TLS 1.3 强制前向安全（Forward Secrecy）——即使长期私钥泄露，历史会话密钥也无法被解密，因为每次会话密钥由临时密钥协商生成。架构师部署 HTTPS 应优先 TLS 1.3。

## 实践

### 【高级衔接】架构师选型密码方案

架构师选密码方案看三个维度：**安全强度、性能、合规**。

1. **何时用对称何时用非对称**：大数据量加密用对称（AES-GCM/SM4），密钥分发和签名用非对称（RSA/ECC/SM2）。几乎所有实际系统都用混合加密。
2. **国密合规场景**：金融（银联/银行）、政务、关基系统强制或优先国密——SM2 替代 RSA、SM3 替代 SHA-256、SM4 替代 AES、SM9 用于标识加密。国密 HTTPS 双证书（签名证书+加密证书）是国密特色。
3. **HTTPS 部署架构**：LB 层卸载 TLS（7 层 LB 装证书，后端明文）适合内部可信网络；端到端加密（后端也装证书）安全性更高。证书选 CA（Let's Encrypt 免费/CFCA 国密）、有效期管理（90 天趋势）、OCSP Stapling 减少客户端查询延迟。
4. **密钥管理**：用 KMS（密钥管理服务）集中管理，应用拿 DEK 加密数据，KEK 留在 KMS/HSM；轮换周期根据数据敏感度（金融月轮换、一般季度）。
5. **避免的坑**：自己造加密算法（永远用标准算法）、ECB 模式加密重复数据、MD5/SHA-1 做安全校验、硬编码密钥进代码。

### 真题示例

**2024 年·综合知识·密码学(考生回忆)**：关于分组密码工作模式，下列说法正确的是？
A. ECB 模式下相同明文块产生相同密文块，适合加密大量重复数据
B. CBC 模式可并行加密，性能优于 ECB
C. GCM 模式同时提供加密和完整性认证
D. CTR 模式不能并行处理

**答案**：C。GCM = CTR 加密 + GHASH 认证标签，是 AEAD（带认证的加密）模式。A 错（ECB 相同明文相同密文正是不安全的原因）；B 错（CBC 串行，前块依赖后块）；D 错（CTR 可并行）。

## 速查/自测

### 30 秒速查卡

- **对称**：DES(56)/3DES(168)/AES(128/192/256)/SM4(128)，快但密钥分发难
- **非对称**：RSA(大数分解)/ECC(椭圆曲线)/SM2，慢但解决分发+签名
- **哈希**：MD5(128,弃)/SHA-1(160,弃)/SHA-256/SHA-512/SM3(256)
- **分组模式**：ECB(弃)/CBC(串行)/CTR(并行)/GCM(加密+认证,推荐)
- **签名**：RSA/DSA/ECDSA/SM2，私钥签公钥验
- **MAC**：HMAC-SHA256，共享密钥，无抗抵赖
- **PKI**：CA 签发/RA 审核/CRL 黑名单/OCSP 实时查询
- **国密四件套**：SM2(非对称)/SM3(哈希)/SM4(对称)/SM9(标识)
- **TLS 1.3**：1-RTT、强制 ECDHE 前向安全、移除静态 RSA/CBC/RC4

### 对称 vs 非对称对比表

| 维度 | 对称加密 | 非对称加密 |
|---|---|---|
| 密钥 | 一把（共享） | 一对（公钥+私钥） |
| 速度 | 快 | 慢（约慢 100-1000 倍） |
| 适合 | 大数据量 | 小数据量+密钥分发+签名 |
| 分发难题 | 有 | 无 |
| 抗抵赖 | 不能 | 能（签名） |
| 典型算法 | AES/SM4/DES | RSA/ECC/SM2 |
