# 第42章：架构风格选择题，五步公式+三大场景模板

架构风格选择是案例分析的高频题型，分值通常 10~15 分。它的考法看起来简单——"为某系统选一种架构风格并说明理由"，但失分率很高，因为考生只写"我选 X 风格，因为 X 适合"就交差，漏掉了"对比排除"和"折中说明"两个踩分点。这一章给出五步公式，再配三个高频场景模板（编译器、电商、语音识别），把这道题变成"按模板填空"。

## 概念

### 这章考什么

- **综合知识（单选）**：考 **1~2 题**。常考架构风格的分类、特征、典型应用。比如"管道-过滤器风格的特征是 X""以下属于独立构件风格的是 Y"。
- **案例分析**：**高频**。典型问法："请为该系统选择一种架构风格并说明理由""请对比 A、B 两种风格并给出选型建议""该系统采用了 X 风格，请说明其优缺点"。分值 **10~15 分**。
- **论文**：偶考。论文里写"我采用了 X 架构风格"时，要交代"为什么选这个、不选那个"，本质就是这道题的展开。

考频结论：案例高频，必练。

### 题型特征

架构风格选择题的题干结构稳定，永远是这个套路：

```
某 [业务场景] 系统，[业务特征描述]，[质量属性需求或约束]。
请为该系统选择一种架构风格，并说明理由。
（有时附带：候选风格有 A、B、C，请选择并对比说明）
```

读题时眼睛盯三处：**业务特征**（决定哪种风格匹配）、**质量属性需求**（决定选型优先级）、**候选列表**（如果给了候选，必须对比，不能只写自己选的那个）。

### 软考核心架构风格速查

案例分析涉及的架构风格集中在五大类，这里只放案例答题常用的子集：

| 大类 | 风格 | 典型场景 | 关键质量属性 |
|---|---|---|---|
| 数据流 | **管道-过滤器（Pipe-Filter）** | 编译器、ETL、数据处理流水线 | 可修改性、可重用性 |
| 数据流 | 批处理 | 日报、月结 | 吞吐量 |
| 调用返回 | 主程序-子程序 | 早期结构化程序 | 可理解性 |
| 调用返回 | **面向对象** | 业务系统 | 可维护性、可重用性 |
| 调用返回 | **分层（Layered）** | 操作系统、网络协议栈、企业应用 | 可移植性、可修改性 |
| 调用返回 | **客户端-服务器（C/S）** | 桌面应用、企业内网系统 | 可用性 |
| 独立构件 | **事件驱动（隐式调用）** | GUI、消息系统、IoT | 可扩展性、可修改性 |
| 独立构件 | **进程通信** | 分布式系统 | 可伸缩性 |
| 虚拟机 | 解释器 | 脚本语言、规则引擎 | 灵活性 |
| 虚拟机 | 基于规则的系统 | 专家系统 | 灵活性 |
| 仓库 | **数据库** | 事务系统 | 数据一致性 |
| 仓库 | **黑板（Blackboard）** | 语音识别、图像识别、AI 推理 | 灵活性、可扩展性 |
| 仓库 | 超文本系统 | Wiki、文档系统 | 可导航性 |
| 现代 | **微服务（Microservices）** | 互联网高并发应用 | 可伸缩性、可修改性 |
| 现代 | **SOA（面向服务架构）** | 企业集成、跨系统协作 | 互操作性 |
| 现代 | REST | Web API | 互操作性、可伸缩性 |

案例答题时，风格名称必须写全称（如"管道-过滤器风格"），不能只写"过滤器风格"或"管道风格"。这是踩分点，名字错了直接丢分。

## 原理

### 五步答题公式

架构风格选择题固定套这个五步公式，必须默写：

```
① 识别场景约束（质量属性需求）
→ ② 列出候选风格
→ ③ 逐风格分析对每个质量属性的影响
→ ④ 选型并说明理由
→ ⑤ 补充约束/折中
```

**第 1 步：识别场景约束（质量属性需求）**

读题，把题干里的"业务特征 + 质量属性需求"提取出来，列成清单。比如：

- 业务特征：处理流程线性、各阶段独立、需并行
- 质量属性需求：性能（吞吐）、可修改性（易扩展阶段）、可重用性（复用过滤器）

这一步是后续选型的依据，没列清单直接下笔，容易凭感觉选，选错或理由空洞。

**第 2 步：列出候选风格**

如果题干给了候选列表，直接用题干的列表；如果没给，自己列 2~3 个候选。候选必须有 **2 个以上**——只列 1 个就等于没得选，无法做"对比"，会丢"对比分析"踩分点。

候选风格的选择原则：选业务场景理论上匹配的 + 看似也能用但有缺陷的。比如编译器题，候选列"管道-过滤器风格 + 面向对象风格"，前者匹配，后者也能用但不够优——这样对比才有意义。

**第 3 步：逐风格分析对每个质量属性的影响**

这是答题的核心。建一个矩阵：

| 候选风格 | 性能 | 可修改性 | 可重用性 | 其他约束 |
|---|---|---|---|---|
| 管道-过滤器 | 高（并行） | 高（新增过滤器） | 高（过滤器复用） | 不适合交互 |
| 面向对象 | 中 | 中 | 中 | 适合业务建模 |

矩阵写在草稿纸上，正文里转成"分点 + 先结论后理由"的文字。每个风格对每个质量属性的影响都要写明（写正/负/中性），漏写一个就丢一个踩分点。

**第 4 步：选型并说明理由**

明确给出选型结论，再用"匹配业务特征 + 满足关键质量属性"两条理由支撑。结论必须一句话在前，理由分条在后。

**第 5 步：补充约束/折中**

任何选型都有代价，必须写"该风格带来的新问题 + 缓解措施"。比如选管道-过滤器要补"不适合交互式场景，需引入事件驱动模块处理交互"。这一步是高分的关键，没写折中只能拿 80% 分。

### 踩分点清单

阅卷老师按以下五个踩分点给分，缺一个扣 2~3 分：

1. **场景约束识别**：是否准确提取了业务特征和质量属性需求。
2. **风格名称准确**：是否用了全称（如"管道-过滤器风格"，不是"过滤器风格"）。
3. **对比分析**：是否对 2 个以上候选风格都做了质量属性影响分析。
4. **选型理由**：是否说明了选型匹配业务特征 + 满足关键质量属性。
5. **折中说明**：是否补充了该选型的局限和缓解措施。

### 通用答题模板

```
[第1步 约束识别] 该系统的业务特征为 XX，质量属性需求为：
  ①性能（XX）；②可修改性（XX）；③XX。

[第2步 候选列举] 候选架构风格有：
  A. 管道-过滤器风格（Pipe-Filter）
  B. 面向对象风格（Object-Oriented）

[第3步 对比分析]
  A 风格：①性能方面，XX（正/负）；②可修改性方面，XX；③XX。
  B 风格：①性能方面，XX（正/负）；②可修改性方面，XX；③XX。

[第4步 选型理由] 综上，选择 A（管道-过滤器风格）。
  理由：①匹配业务特征 XX；②满足关键质量属性 XX。

[第5步 折中说明] 该风格的局限是 XX，缓解措施为 XX。
```

模板背下来，所有架构风格选择题都能套。

## 实践

### 模拟题一：在线编译器选管道-过滤器风格

> **题目**：
> 某公司开发一个在线编译器，需要支持 C、Java、Python 等多种语言。编译过程包括词法分析、语法分析、语义分析、中间代码生成、优化、目标代码生成六个阶段。要求各阶段可以独立开发和替换，新增语言支持时只需替换部分阶段，整体编译效率高。请为该系统选择一种架构风格并说明理由，并对比另一种候选风格。

#### 分步解答

**第 1 步：识别场景约束**

- 业务特征：编译流程线性、六阶段顺序处理、各阶段独立、需支持多语言
- 质量属性需求：①可修改性（独立开发替换）；②可重用性（多语言复用阶段）；③性能（编译效率高）

**第 2 步：列候选**

- A. 管道-过滤器风格（Pipe-Filter）
- B. 面向对象风格（Object-Oriented）

**第 3 步：对比分析**

```
A. 管道-过滤器风格
  ①可修改性：高。每个阶段是独立过滤器，新增/替换阶段不影响其他阶段。
  ②可重用性：高。同一阶段（如优化器）可被多语言复用。
  ③性能：高。各过滤器可并行执行，整体编译效率高。
  ④局限：不适合交互式调试场景。

B. 面向对象风格
  ①可修改性：中。封装性好，但阶段间通过对象消息传递，新增阶段需改动调用方。
  ②可重用性：中。对象可复用，但编译流程的顺序耦合在对象方法调用中。
  ③性能：中。无并行优势，顺序执行。
  ④优势：适合业务建模，便于扩展调试交互。
```

**第 4 步：选型理由**

```
选择 A：管道-过滤器风格（Pipe-Filter）。
理由：
1. 匹配业务特征：编译流程为线性顺序处理，符合管道-过滤器的"数据流顺序流转"特征。
2. 满足关键质量属性：
   ①可修改性：新增语言只需替换词法/语法分析等过滤器，其他过滤器复用；
   ②可重用性：优化器、目标代码生成器可跨语言复用；
   ③性能：各过滤器可并行执行，编译效率高。
```

**第 5 步：折中说明**

```
该风格的局限：管道-过滤器为顺序数据流模型，不适合交互式调试、断点
设置等需要回滚或交互的场景。
缓解措施：在编译主流程外，引入事件驱动（隐式调用）模块处理调试交互，
调试事件触发编译器暂停或回放，与管道主流程解耦。
```

**踩分点标注**：①场景约束识别完整（六阶段、独立、多语言）；②风格名称用全称（Pipe-Filter）；③对比了两个候选（A 和 B 都分析）；④选型理由结合业务特征 + 三大质量属性；⑤折中说明了局限和缓解（事件驱动模块）。

### 模拟题二：电商平台选微服务架构

> **题目**：
> 某电商公司原系统为单体架构，随业务增长出现扩容困难、新功能上线慢、不同模块技术栈难以独立演进等问题。业务峰值 TPS 需求达 5 万，需支持"双 11"等大促活动快速扩容。团队考虑两种候选：①保持单体但做读写分离；②引入微服务架构。请选择并说明理由。

#### 分步解答

**第 1 步：识别场景约束**

- 业务特征：电商多模块（用户、商品、订单、支付、库存）、大促需快速扩容
- 质量属性需求：①可伸缩性（峰值 5 万 TPS，大促扩容）；②可修改性（新功能上线快）；③可演进性（独立技术栈）

**第 2 步：列候选**

- A. 单体 + 读写分离
- B. 微服务架构（Microservices）

**第 3 步：对比分析**

```
A. 单体 + 读写分离
  ①可伸缩性：低。仅缓解数据库读压力，应用层仍为单进程，无法针对
            热点服务（如订单）单独扩容。
  ②可修改性：低。代码耦合，新功能上线需整体回归测试。
  ③可演进性：低。技术栈统一绑定，无法独立演进。

B. 微服务架构
  ①可伸缩性：高。按服务独立扩容，订单服务可在大促期间单独扩容到
            几百实例，应对 5 万 TPS。
  ②可修改性：高。服务间松耦合，新增功能只需新增/修改对应服务。
  ③可演进性：高。每个服务可独立选技术栈（如支付用 Java、推荐用 Python）。
  ④局限：分布式复杂性、数据一致性挑战、运维成本上升。
```

**第 4 步：选型理由**

```
选择 B：微服务架构（Microservices）。
理由：
1. 匹配业务特征：电商多模块天然可按业务边界拆分服务（用户、商品、
   订单、支付、库存）。
2. 满足关键质量属性：
   ①可伸缩性：订单、支付等热点服务可独立扩容，应对大促 5 万 TPS；
   ②可修改性：服务松耦合，新功能上线周期从周级降到天级；
   ③可演进性：各服务独立技术栈，可按需演进。
```

**第 5 步：折中说明**

```
该风格的局限与缓解：
1. [分布式复杂性] 服务间网络调用引入延迟和故障。
   缓解：引入服务注册发现（如 Nacos）、熔断降级（如 Sentinel）、
        链路追踪（如 SkyWalking）。
2. [数据一致性] 跨服务事务难保证强一致性。
   缓解：采用最终一致性，通过消息队列（如 Kafka）做异步对账，
        关键交易用 Saga 模式补偿。
3. [运维复杂性] 服务多，部署监控成本高。
   缓解：引入容器化（Docker）+ 容器编排（Kubernetes）自动化部署。
```

**踩分点标注**：①场景约束识别（5 万 TPS、大促、独立技术栈）；②风格名称用全称（Microservices）；③对比两个候选；④选型理由结合三大质量属性；⑤折中说明完整（三个问题+缓解措施）。

### 模拟题三：语音识别系统选黑板架构

> **题目**：
> 某科研团队开发一个语音识别系统，输入为音频流，需综合声学模型、语言模型、语法分析、语义理解等多个知识源协同识别。各知识源随时可能更新（如新词汇加入），识别过程无固定流程（不同知识源按需参与），系统需支持不确定性和部分匹配。请为该系统选择一种架构风格并说明理由。

#### 分步解答

**第 1 步：识别场景约束**

- 业务特征：多知识源协同、无固定流程、需处理不确定性、知识源可更新
- 质量属性需求：①灵活性（无固定流程，按需调度）；②可扩展性（新增知识源）；③可修改性（知识源更新不影响其他）

**第 2 步：列候选**

- A. 黑板架构（Blackboard）
- B. 管道-过滤器风格（Pipe-Filter）

**第 3 步：对比分析**

```
A. 黑板架构
  ①灵活性：高。无固定流程，控制组件（控制器）根据黑板状态动态调度
          知识源（KS），处理不确定性。
  ②可扩展性：高。新增知识源只需注册到黑板，无需改动其他知识源。
  ③可修改性：高。知识源独立更新，互不影响。
  ④适用：语音识别、图像识别、AI 推理等非确定性问题。

B. 管道-过滤器风格
  ①灵活性：低。流程固定为线性顺序，无法按需调度。
  ②可扩展性：中。新增阶段需插入管道，破坏原有流程。
  ③可修改性：中。阶段独立，但流程顺序耦合。
  ④不适用：语音识别需要多知识源迭代协同，不是线性流水线。
```

**第 4 步：选型理由**

```
选择 A：黑板架构（Blackboard）。
理由：
1. 匹配业务特征：语音识别需多知识源（声学、语言、语法、语义）协同，
   无固定流程，符合黑板架构"知识源+黑板+控制器"三要素。
2. 满足关键质量属性：
   ①灵活性：控制器根据黑板当前状态动态调度知识源，处理不确定性
            和部分匹配；
   ②可扩展性：新增知识源（如方言模型）只需注册到黑板；
   ③可修改性：知识源独立更新（如新词汇加入语言模型），不影响其他。
```

**第 5 步：折中说明**

```
该风格的局限与缓解：
1. [性能] 黑板架构的动态调度和知识源轮询带来性能开销，不适合实时
        低延迟场景。
   缓解：引入知识源优先级机制，高置信度知识源优先调度；对实时性
        要求高的子任务（如端点检测）独立用管道-过滤器处理。
2. [一致性] 多知识源并发读写黑板可能冲突。
   缓解：黑板访问加锁，或采用事件驱动方式通知知识源。
3. [测试困难] 动态调度导致执行路径不确定。
   缓解：记录每次调度日志，便于回放测试。
```

**踩分点标注**：①场景约束识别（多知识源、无固定流程、不确定性）；②风格名称用全称（Blackboard）；③对比两个候选；④选型理由结合业务特征 + 三大质量属性；⑤折中说明三个问题+缓解措施。

### 易错提醒

**易错 1：风格名称写错或写半截**

考生常把"管道-过滤器风格"写成"过滤器风格""管道风格"，把"黑板架构"写成"黑板模式"，把"事件驱动"写成"事件机制"。**风格名称必须用全称**，错一个字扣踩分点。

**易错 2：只写选型不写对比**

题干如果给了两个候选，必须都分析；没给候选，自己也要列 2 个对比。只写"我选 X 因为 X 好"，丢"对比分析"踩分点。

**易错 3：术语只用缩写不用全称**

第一次出现"微服务架构（Microservices）""面向服务架构（Service-Oriented Architecture，SOA）""企业服务总线（Enterprise Service Bus，ESB）"必须写全称+缩写，之后才能用缩写。直接写"SOA、ESB、MVC"扣分。

**易错 4：理由空洞不结合题干**

"我选微服务，因为微服务可伸缩"——这种理由空洞。必须写"题干要求大促 5 万 TPS，微服务可对订单服务单独扩容，满足可伸缩性"，把题干关键词引进来。

**易错 5：折中只写局限不给措施**

"微服务会带来分布式复杂性"——只写问题没给缓解。必须"问题+措施"配对，每个局限都给一条具体的缓解措施。

**易错 6：把架构风格和设计模式混淆**

架构风格是"系统级"的（管道-过滤器、微服务、黑板），设计模式是"组件级"的（观察者、策略、单例）。架构风格选择题答设计模式直接零分。

## 速查/自测

### 答题骨架速查卡

```
架构风格选择题五步公式：
  ① 识别场景约束（业务特征 + 质量属性需求）
  ② 列候选（2 个以上）
  ③ 逐风格分析对每个质量属性的影响（建矩阵）
  ④ 选型并说明理由（匹配业务 + 满足质量属性）
  ⑤ 补充约束/折中（局限 + 缓解措施）

踩分点五项：
  场景约束识别 / 风格名称全称 / 对比分析 /
  选型理由结合题干 / 折中说明配对
```

### 三大场景模板速查

| 场景 | 选型 | 关键理由 |
|---|---|---|
| 编译器/ETL/数据处理流水线 | **管道-过滤器风格** | 线性流程、阶段独立、可并行、易扩展 |
| 电商/互联网高并发应用 | **微服务架构** | 业务可拆服务、独立扩容、独立技术栈 |
| 语音识别/图像识别/AI 推理 | **黑板架构** | 多知识源协同、无固定流程、处理不确定性 |
| GUI/消息系统/IoT | **事件驱动（隐式调用）** | 异步解耦、可扩展 |
| 操作系统/网络协议栈/企业应用 | **分层架构** | 抽象层级、可移植、可修改 |
| 跨企业集成/遗留系统协作 | **SOA** | 互操作性、服务复用 |

### 记忆口诀

- 五步公式：**"约候比分折"**（约束识别、候选列举、对比分析、选型理由、折中说明）。
- 三大场景：**"编管电微语黑"**（编译器→管道、电商→微服务、语音→黑板）。
- 风格名称要全：**"风格不写半，全称加缩写"**。

### 自测题

**自测 1**：请默写架构风格选择题的五步答题公式。

> 参考答案：①识别场景约束（业务特征 + 质量属性需求）→ ②列出候选风格（2 个以上）→ ③逐风格分析对每个质量属性的影响 → ④选型并说明理由（匹配业务 + 满足质量属性）→ ⑤补充约束/折中（局限 + 缓解措施）。

**自测 2**：在线编译器为什么选管道-过滤器风格而不是面向对象风格？

> 参考答案：编译流程为线性顺序处理（词法→语法→语义→代码生成），符合管道-过滤器的数据流顺序流转特征。各阶段独立可并行（性能高）、可独立替换（可修改性高）、可跨语言复用（可重用性高）。面向对象风格虽可建模但顺序耦合在方法调用中，无并行优势、阶段扩展需改调用方，不匹配编译器的线性流水线特征。

**自测 3**：某语音识别系统选黑板架构，请说明黑板架构的三要素及其作用。

> 参考答案：黑板架构三要素：①知识源（Knowledge Source，KS）——独立的领域知识模块（如声学模型、语言模型），监控黑板状态并在条件满足时修改黑板；②黑板（Blackboard）——共享数据结构，存放中间识别结果，知识源读写它；③控制器（Controller）——根据黑板状态动态调度知识源，决定下一个执行哪个 KS。三要素协同实现无固定流程的动态调度。

**自测 4**：架构风格选择题最容易丢的两个踩分点是什么？

> 参考答案：①**对比分析**——只写自己选的风格，不分析其他候选；②**折中说明**——只写选型理由不写该选型的局限和缓解措施。这两个踩分点丢了，即使选型正确也只能拿 60% 分。

**自测 5**：分层架构和微服务架构在可修改性方面有何异同？

> 参考答案：相同点：两者都通过"分层/分服务"实现关注点分离，提升可修改性。不同点：分层架构按"抽象层级"划分（表现层、业务层、数据层），层间依赖单向，修改某一层通常不影响其他层，但层内仍为单体；微服务按"业务边界"划分，服务间完全独立部署，可独立修改、独立部署、独立扩容，可修改性更强但带来分布式复杂性。

### 一句话小结

架构风格选择题五步公式"约候比分折"是骨架，三大场景模板（编译器→管道-过滤器、电商→微服务、语音→黑板）是肌肉，对比分析和折中说明是高分关键。把风格名称写全称、把题干关键词引进理由，这道题就能稳拿 12 分以上。
