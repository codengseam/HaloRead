# 从瀑布到 DevOps，模型选错全盘皆输

过程模型不是教科书怀旧——它直接决定你 11 月考场能不能拿下那 3-5 道送分题，更决定案例题"该选什么模型"的论证能不能站住脚。

## 概念

这章考什么：
- **科目**：综合知识（必考 3-5 题）、案例分析（选型论证常考）、论文（可作背景素材）
- **分值**：综合约 4-6 分，案例 5-15 分
- **考频**：高频，几乎每年必出

核心概念类比：开发过程模型就像建筑施工方式。瀑布是"图纸定死→打地基→砌墙→装修"一条龙走到黑；增量是"先搭个能住的小屋，再逐层加盖"；螺旋是"每盖一层先做个样板间验证风险"；敏捷是"客户每天来看工地，边盖边改图纸"。

## 原理

### 一、八大过程模型逐个拆

**1. 瀑布模型（Waterfall）**
1970 年 Winston Royce 提出。严格线性：需求→设计→编码→测试→维护，每阶段有里程碑文档，前阶段完成后才进入下一阶段。适用需求明确且技术成熟的项目，如政府合规系统、嵌入式固件。优点是文档完整、易于管理进度；致命缺陷是错误要等到测试阶段才暴露，返工成本高，且不适应需求变更。

**2. V 模型（V-Model）**
瀑布的测试驱动变种：开发阶段与测试阶段一一对应——需求分析对验收测试、概要设计对系统测试、详细设计对集成测试、编码对单元测试。适用强质量约束场景。比瀑布多了一点"测试先行"意识，但同样前期不验证需求。

**3. 增量模型（Incremental）**
把系统分成若干增量构件，每个增量都是一个可运行产品。第 1 个增量往往是核心功能（核心优先），后续增量逐步扩充。适用需求可分块、想快速上市的项目。优点是早期可用、风险分摊到各增量；缺点是要求开放架构、后期集成难。

**4. 螺旋模型（Spiral）**
Barry Boehm 1988 年提出。每轮迭代分四步：制定计划→风险分析→实施工程→客户评估，沿螺旋推进。最大特征是**风险驱动**。适用大型、高风险、需求不确定项目，如军工、航天、金融核心系统。优点是风险早暴露；缺点是依赖团队风险分析能力，成本高，小项目不值得。

**5. 喷泉模型（Fountain）**
面向对象（OO）开发专属。强调迭代无间隙、各阶段可重叠并行，像喷泉水循环。优点是开发高效、适合 OO；缺点是管理复杂、文档易遗漏。

**6. 原型模型（Prototype）**
需求模糊时先快速搭一个可演示原型，客户确认后再正式开发。分**抛弃型原型**（throwaway，验证完丢弃）和**演化型原型**（evolutionary，逐步演化为产品）。优点是需求澄清快；陷阱是客户以为原型就是产品，质量被牺牲。

**7. RAD 快速应用开发（Rapid Application Development）**
复用 + 可视化工具 + 短周期（60-90 天）。强调组件复用、自动代码生成、用户全程参与。适用中小型业务系统且组件可复用。不适合高性能计算、大并发、强实时项目。

**8. 敏捷 Scrum 与 XP**
Scrum：以 Sprint（2-4 周）为单位迭代，三大角色 Product Owner / Scrum Master / Development Team，三个工件 Product Backlog / Sprint Backlog / Increment，五个事件 Sprint 计划 / 每日站会 / Sprint 评审 / Sprint 回顾 / Sprint。XP（eXtreme Programming）：结对编程、TDD、持续集成、简单设计、重构。适用需求多变、中小团队。

### 二、模型对比表（必背）

| 模型 | 驱动 | 适用 | 致命短板 |
|---|---|---|---|
| 瀑布 | 文档 | 需求稳定 | 不适应变更 |
| V 模型 | 测试 | 强质量 | 前期不验证 |
| 增量 | 功能分块 | 快速上市 | 集成难 |
| 螺旋 | 风险 | 大型高风险 | 成本高 |
| 喷泉 | OO | OO 项目 | 管理复杂 |
| 原型 | 需求验证 | 需求模糊 | 易被当产品 |
| RAD | 复用 | 中小业务 | 性能妥协 |
| Scrum | 价值 | 多变小团队 | 文档弱 |
| XP | 工程 | 小团队 | 人耗大 |

### 三、方法论演进：结构化→OO→敏捷→DevOps

- **结构化方法**：自顶向下、分而治之，DFD + 数据字典 + SC 结构图，代表瀑布。
- **面向对象方法**：封装继承多态，UML 建模，代表喷泉、RUP。
- **敏捷方法**：价值观"个体互动高于流程工具、可工作软件高于详尽文档、客户合作高于合同谈判、响应变化高于遵循计划"，代表 Scrum/XP。
- **DevOps**：开发运维一体化，CI/CD + 自动化测试 + 监控反馈，把敏捷从开发延伸到运维。

易混点辨析：
- **增量 vs 螺旋**：增量按"功能分块"交付，每块都是产品；螺旋按"风险驱动"迭代，每轮都做风险分析。
- **喷泉 vs 敏捷**：喷泉强调"无间隙迭代"（OO 特性），敏捷强调"价值驱动短迭代"。
- **原型 vs RAD**：原型是验证需求，RAD 是快速交付产品。

## 实践

**真题示例（2024 年·综合知识·过程模型相关·考生回忆）**：题干描述某项目"需求明确、技术成熟、强文档要求"，问应选哪种模型。答案：瀑布。踩分词"需求明确 + 文档驱动"。

**案例分析答题公式**：
1. 识别项目特征关键词：需求是否明确 / 风险大小 / 团队规模 / 是否可复用 / 是否快速上市
2. 套模型特征：需求明确→瀑布；高风险大型→螺旋；快速上市→增量；需求多变小团队→敏捷；可复用组件→RAD；需求模糊→原型
3. 论证：写三句"项目具备 X 特征，模型 Y 强调 Z，因此适用"，再加一句风险提示

**论文应用**：在论文"项目背景"段交代"本项目采用 Scrum 敏捷开发，2 周一个 Sprint，共 12 个 Sprint"，体现你的工程实践。这是低成本加分项。

**自检陷阱**：题目说"需求模糊"未必是原型，可能是螺旋（如果还强调高风险）；"快速上市"未必是敏捷，可能是增量或 RAD。看完整关键词再下结论。

## 速查/自测

**30 秒速查卡**：

| 关键词 | 选模型 |
|---|---|
| 需求明确、文档严格 | 瀑布 |
| 测试与开发对齐 | V 模型 |
| 快速上市、分批交付 | 增量 |
| 大型、高风险、风险分析 | 螺旋 |
| OO、迭代无间隙 | 喷泉 |
| 需求模糊、先验证 | 原型 |
| 复用、短周期、可视化 | RAD |
| 多变、小团队、Sprint | Scrum |
| 结对、TDD、重构 | XP |

**口诀**：瀑布文档线性走，V 模测试对应齐；增量分块早上市，螺旋风险每轮析；喷泉 OO 无间隙，原型验证需求迷；RAD 复用短周期，敏捷迭代值优先。

**自测题**：
1. 某金融核心系统改造，需求部分明确但技术风险高，应优先考虑哪种模型？→ 螺旋
2. 瀑布模型最严重的缺陷是什么？→ 错误推迟到测试阶段才暴露，返工成本高
3. Scrum 的三个角色和三个工件分别是什么？→ 角色：PO/SM/Team；工件：Product Backlog/Sprint Backlog/Increment

**一句话小结**：模型选型是案例必考——背熟特征表 + 关键词映射，3 分钟拿满分。
