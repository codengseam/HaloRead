# 第33章：中间件六大门派，别再把JDBC当成RPC

中间件考的就是分类识记，每年综合知识稳出 **2~3 题**。难点不在理解，在别记混——JDBC 属数据库访问中间件不是 RPC，MQ 是面向消息不是分布式对象，这两个坑年年有人栽。这章把教材的权威分类、消息队列对比、RPC 框架、ESB 一次讲清。

## 概念

### 这章考什么

- **综合知识**：每年 **2~3 题**。考点集中在中间件分类辨识（给一个名词问属于哪类）、消息队列特性对比、ESB 与 SOA 关系。
- **案例分析**：偶尔出现，常结合微服务考 MQ 选型、RPC 框架选型。
- **论文**：直接考少，但写分布式系统、消息中间件选型时会用到。

### 核心概念类比

中间件（Middleware）= 操作系统和应用之间的"中间一层"，把分布式系统里共性的、难搞的问题（通信、数据访问、事务、消息）封装成现成组件，让应用专心写业务。

打类比：中间件像写字楼里的物业——你不直接管电梯、空调、保安、收发室，物业统一提供这些服务，你只管租办公室做自己的事。不同门派的中间件，就是不同类型的物业服务：数据库访问中间件是"档案室管理员"，RPC 是"内部电话总机"，MOM 是"收发室信箱"，分布式对象是"远程管家"，事务处理中间件是"账房先生"，Web 服务中间件是"对外前台"。

### 教材权威分类（必背六大门派）

系统架构设计师教材对中间件的分类是定论，**必须按教材口径答题**：

| 类别 | 全称 | 典型代表 | 一句话定位 |
|---|---|---|---|
| **数据库访问中间件** | Database Access | **ODBC、JDBC** | 屏蔽异构数据库差异，统一访问接口 |
| **远程过程调用中间件** | RPC | Dubbo、gRPC | 像调本地函数一样调远程服务 |
| **面向消息中间件** | MOM/MQ | **Kafka、RabbitMQ、RocketMQ** | 以消息异步通信，松耦合 |
| **分布式对象中间件** | Distributed Object | **CORBA、EJB、DCOM** | 远程对象调用，有对象总线 |
| **事务处理中间件** | TP Monitor | **Tuxedo** | 管理跨资源事务，保证 ACID |
| **Web 服务中间件** | Web Services | SOAP/WSDL/UDDI | SOA 的实现载体，跨平台互操作 |

注意：**CORBA** 在不同教材版本里有时归 RPC 有时归分布式对象，软考以"分布式对象中间件"为准（CORBA 是对象请求代理 ORB 的代表）。

## 原理

### 一、六大门派详解

**1. 数据库访问中间件**

代表 **ODBC（开放数据库互连）、JDBC（Java 数据库连接）**。作用是屏蔽底层不同数据库（Oracle、MySQL、SQL Server）的协议差异，提供统一编程接口。应用程序写一套 JDBC 代码，换数据库只换驱动不改业务。

**易错点（年年考）**：JDBC 是**数据库访问中间件**，**不是 RPC**。有人觉得"调数据库也是远程调用"就归 RPC，错。RPC 强调的是"调用远程的函数/方法"，JDBC 调用的是"数据库执行 SQL"，语义和接口都不同。

**2. 远程过程调用中间件（RPC）**

RPC（Remote Procedure Call）让调用远程函数像调本地函数一样简单，隐藏网络细节。代表 **Dubbo（阿里）、gRPC（Google）**。

- **Dubbo**：Java 生态，国内主流，配合 Nacos/Zookeeper 做注册发现。
- **gRPC**：跨语言，基于 HTTP/2 + Protobuf，性能高、强类型。

RPC 的核心是**存根（Stub）**：客户端调用存根像调本地函数，存根负责序列化、网络传输、调用对端、返回结果。同步阻塞为主。

**3. 面向消息中间件（MOM/MQ）**

MOM（Message-Oriented Middleware）以消息为单位异步通信。生产者发消息到队列，消费者按需取走，**生产者消费者不直接见面**，松耦合。代表 **Kafka、RabbitMQ、RocketMQ**。

与 RPC 的对比：RPC 是"打电话"——同步、实时、要对方在线；MQ 是"发邮件"——异步、解耦、不怕对方不在线。

**易错点**：MQ 是**面向消息中间件**，**不是分布式对象中间件**。分布式对象强调"远程对象的方法调用"（有对象引用、有方法签名），MQ 只管"投递消息"（消息内容自解释），二者语义不同。

**4. 分布式对象中间件**

代表 **CORBA、EJB、DCOM**。比 RPC 更进一步：不仅调远程方法，还能像操作本地对象一样操作远程对象，有对象引用、对象生命周期管理。CORBA 通过 ORB（Object Request Broker，对象请求代理）实现对象互操作。

这类技术属"上一代"分布式方案，互联网时代被 RPC+微服务取代，但教材仍列为分类，考试仍考。

**5. 事务处理中间件**

代表 **Tuxedo（BEA/Oracle）**。专门管理跨多个资源（多数据库、多服务）的分布式事务，保证 ACID。银行、电信老系统大量用 Tuxedo 管交易事务。

注意与现代分布式事务方案（Saga、TCC、Seata）的区别：Tuxedo 是重型事务监控器，现代方案偏轻量最终一致。

**6. Web 服务中间件**

代表 **SOAP、WSDL、UDDI** 这套 SOA 经典协议栈。SOAP 是通信协议（基于 XML），WSDL 是接口描述语言，UDDI 是服务注册发现目录。是 SOA 落地的标准载体，与 ESB 配套构成早期企业服务集成方案。

### 二、消息队列对比（高频考点）

| 维度 | Kafka | RabbitMQ | RocketMQ |
|---|---|---|---|
| **吞吐量** | **极高**（百万级 TPS） | 中等（万级 TPS） | 高（十万级 TPS） |
| **延迟** | 毫秒级 | **微秒级最低** | 毫秒级 |
| **可靠性** | 高（副本机制） | 高（确认机制） | **高（同步刷盘+同步双写）** |
| **设计取向** | 日志流、大数据场景 | 传统消息路由、复杂路由 | 金融级事务消息 |
| **典型场景** | 日志采集、流处理 | 业务消息、任务队列 | 电商交易、订单事务 |

选型口诀：**Kafka 重吞吐、RabbitMQ 重延迟与路由、RocketMQ 重可靠与事务**。案例题给场景选 MQ，按这三句口诀对应。

### 三、RPC 框架对比

| 维度 | Dubbo | gRPC |
|---|---|---|
| **语言生态** | Java 为主 | 跨语言 |
| **协议** | Dubbo 协议（默认 TCP） | HTTP/2 + Protobuf |
| **序列化** | Hessian/JSON 等可配 | Protobuf（强类型、紧凑） |
| **注册中心** | Nacos/Zookeeper | 自带或外接 |
| **典型场景** | 国内 Java 微服务 | 跨语言微服务、云原生 |

### 四、ESB 企业服务总线（SOA 核心）

**ESB（Enterprise Service Bus）** 是 SOA 架构的标志性组件，本质是一种**集成型中间件**，融合了消息路由、协议转换、数据变换、编排能力，把企业内异构系统粘合起来。

ESB 的核心职能：

- **路由**：把消息从源转发到目标服务。
- **协议转换**：HTTP 转 JMS、SOAP 转 REST。
- **数据变换**：XML 转 JSON、字段映射。
- **编排**：把多个服务调用串成一个业务流程。
- **中介**：鉴权、日志、监控的统一拦截。

ESB 是 SOA 的中枢，也是 SOA 的瓶颈——所有流量过 ESB，ESB 挂了全系统瘫。这正是后来微服务去 ESB、改用 API 网关+点对点轻量通信的原因。

### 五、三个高频易错点

**易错点一：JDBC 是 RPC 吗？**
不是。JDBC 是数据库访问中间件，调的是数据库执行 SQL；RPC 调的是远程函数/方法。两者接口语义不同。

**易错点二：Kafka/RabbitMQ 是分布式对象中间件吗？**
不是。它们是面向消息中间件（MOM），投递消息不调对象方法。分布式对象是 CORBA/EJB/DCOM。

**易错点三：CORBA 归哪一类？**
软考以**分布式对象中间件**为准（基于 ORB）。部分老教材归 RPC，答题以当前教材口径为准。

## 实践

### 真题示例

**考法一：分类辨识（最高频）**

> 例：以下技术中，属于面向消息中间件的是（　）。
> A. ODBC　B. JDBC　C. RabbitMQ　D. CORBA

**答题思路**：ODBC/JDBC 是数据库访问中间件，CORBA 是分布式对象中间件，RabbitMQ 是 MOM。答案 **C**。

**考法二：JDBC 归属陷阱**

> 例：JDBC 属于哪类中间件？（　）
> A. 远程过程调用中间件　B. 数据库访问中间件　C. 面向消息中间件　D. 事务处理中间件

**答题思路**：JDBC 屏蔽异构数据库差异，统一数据访问接口，是数据库访问中间件。答案 **B**。

**考法三：MQ 选型**（2024年·综合知识·中间件相关，考生回忆）

> 例：某日志采集系统要求每秒处理百万级消息，对单条延迟不敏感，应优先选用的消息中间件是（　）。
> A. RabbitMQ　B. Kafka　C. ActiveMQ　D. RocketMQ

**答题思路**：百万级 TPS + 延迟不敏感 = Kafka 的主战场（日志流、高吞吐）。RabbitMQ 强在低延迟路由，RocketMQ 强在事务消息。答案 **B**。

### 答题踩分词清单

- 分类辨识踩分词：**JDBC→数据库访问、MQ→面向消息、CORBA/EJB→分布式对象、Dubbo/gRPC→RPC、Tuxedo→事务处理、SOAP/WSDL→Web 服务**。
- MQ 选型踩分词：**Kafka 重吞吐、RabbitMQ 重延迟路由、RocketMQ 重可靠事务**。
- RPC 框架踩分词：**Dubbo Java 生态、gRPC 跨语言 HTTP/2 Protobuf**。
- ESB 踩分词：**SOA 中枢、路由/协议转换/编排/中介**。

### 案例论文应用

案例题常给场景让选中间件。答题公式：

> "该场景为电商交易，要求消息可靠不丢、支持事务消息（下单+扣库存原子性），对吞吐也有较高要求。对比三类 MQ：Kafka 吞吐极高但事务消息弱，RabbitMQ 延迟低但吞吐有限且事务支持弱，RocketMQ 同步刷盘+同步双写保证可靠性、原生支持事务消息，最契合电商交易场景。故选用 RocketMQ，配合 Seata 做分布式事务兜底。"

这种"场景需求 → 三类对比 → 选型理由"的三段式，是中间件选型题的标准骨架。

论文写消息中间件选型时，要讲清"为什么不是另外两个"——对比体现决策深度，比单夸一个拿分多。

### 记忆口诀

- 六大门派：**数据库访问 ODBC/JDBC、RPC Dubbo/gRPC、消息 Kafka/Rabbit、对象 CORBA/EJB、事务 Tuxedo、Web 服务 SOAP/WSDL**。
- JDBC 归属：**JDBC 是数据访问，不是 RPC**。
- MQ 选型：**Kafka 吞吐、Rabbit 延迟、Rocket 事务**。
- ESB 职能：**路由转换编排中介**。

### 自测题

**自测 1**：JDBC 为什么不属于远程过程调用中间件？两者的本质区别是什么？

> 参考答案：JDBC 屏蔽异构数据库的协议差异，提供统一的 SQL 执行接口，调用对象是"数据库执行 SQL 语句"，本质是数据访问抽象。RPC（远程过程调用）强调"像调用本地函数一样调用远程服务上的函数/方法"，调用对象是"远程服务的方法"。两者接口语义不同：JDBC 调的是 SQL 执行，RPC 调的是方法签名。故 JDBC 归数据库访问中间件，不归 RPC。

**自测 2**：Kafka、RabbitMQ、RocketMQ 三者在吞吐量、延迟、可靠性上各有何侧重？给一个金融交易场景应选哪个？

> 参考答案：Kafka 吞吐极高（百万级 TPS）适合日志流；RabbitMQ 延迟最低（微秒级）适合复杂路由的业务消息；RocketMQ 可靠性最强（同步刷盘+同步双写）且原生支持事务消息，适合金融交易。金融交易场景对消息不丢、事务原子性要求极高，应选 RocketMQ——它能保证下单与扣库存的事务一致性，Kafka 事务消息弱、RabbitMQ 吞吐和事务支持都不足。

**自测 3**：CORBA 在中间件分类中归入哪一类？它的核心组件 ORB 起什么作用？

> 参考答案：CORBA 归入分布式对象中间件。ORB（Object Request Broker，对象请求代理）是 CORBA 的核心，作用是充当对象通信的"总线"：客户端通过 ORB 找到远程对象的实现，ORB 负责把调用请求序列化、跨网络传到服务端、激活对象方法、把结果返回客户端。ORB 让开发者像操作本地对象一样操作远程对象，屏蔽了底层网络和异构平台差异，是分布式对象中间件的标志。

### 一句话小结

中间件六大门派背熟教材口径：**JDBC 是数据访问不是 RPC、MQ 是消息不是对象、CORBA 是分布式对象、Tuxedo 是事务、SOAP 是 Web 服务；MQ 选型按"Kafka 吞吐 Rabbit 延迟 Rocket 事务"对号入座**。
