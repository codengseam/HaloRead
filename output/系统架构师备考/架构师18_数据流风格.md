# 第18章：数据流风格，让数据自己跑完流水线

数据流风格的核心就一句：**系统的节奏由数据决定，不是由调用决定**。数据进来，被一串处理单元依次加工，出去。这一章把数据流大类的两个子风格——批处理和管道-过滤器讲透，编译器和 ETL 这类高频考点的底就铺好了。

## 概念

### 这章考什么

数据流风格是五大类里最"老实"的一种，考法直接：

- **综合知识**：每年 **1~2 题**，常见两种问法——一是给系统描述让你归类（编译器、ETL、数据清洗几乎必属此类），二是辨析批处理与管道-过滤器的区别。
- **案例分析**：偶有出现，常作为"备选风格之一"让你权衡，比如和黑板、层次架构放一起选。

分值不算大，但**编译器属于管道-过滤器**这个点几乎是常识级考点，丢不得。

### 数据流风格的本质

先记住一句话定位：**数据流风格里，数据是主角，处理单元是配角，连接件是管道**。

打个比方：流水线工厂。原料（数据）从一头进，经过一道道工序（过滤器），最后成品从另一头出。每道工序只管"拿到原料→加工→交出去"，不关心上一道是谁、下一道是谁，也不关心原料什么时候来——来一批处理一批。

这跟调用返回风格的区别很明显：调用返回是"我喊你干活，你干完还给我"（控制流主导）；数据流是"东西放这儿了，谁要谁拿，加工完放下一个传"（数据流主导）。

数据流大类下分两个子风格，区别在"数据是一坨一坨给，还是一股一股给"：

- **批处理（Batch Sequential）**：一坨一坨给，整批到位才处理。
- **管道-过滤器（Pipe-Filter）**：一股一股给，边来边处理。

## 原理

### 子风格一：批处理风格

**结构描述**：批处理由**一组顺序连接的处理组件**构成，前一个组件的完整输出作为后一个组件的输入，组件之间通过"完整数据文件"交接。

```
[处理1] --完整数据--> [处理2] --完整数据--> [处理3] --> 输出
```

**关键特征**：

- 数据是**完整的批量**，必须前一步全部处理完、整体落地，下一步才开始。
- 组件之间**不并行**，串行执行。
- 中间结果通常以**文件**形式持久化保存。

**典型例子**：早期银行的日终批处理——白天记账，晚上把全天交易一次性结算、对账、生成报表，每一步都是吃完整批数据才开始。

**优点**：简单清晰，步骤解耦，中间结果可追溯（出错能从某一步重跑）。

**缺点**：吞吐受最慢环节限制，无并行，实时性差，中间数据占存储。

### 子风格二：管道-过滤器风格

**结构描述**：管道-过滤器由**过滤器（Filter）**和**管道（Pipe）**构成。过滤器是处理单元，管道是数据通道。数据以**流**的形式在管道中流动，过滤器一边读、一边处理、一边写。

```
[过滤器1] --数据流(管道)--> [过滤器2] --数据流(管道)--> [过滤器3] --> 输出
```

**关键特征**：

- 数据是**流式**的，过滤器可以边收边处理边输出，不必等全部数据到齐。
- 过滤器之间**可并行**（只要上游有产出、下游就能消费）。
- 过滤器**独立**，不共享状态，只通过管道交互。

**典型例子**：

- **Unix 管道**：`cat file | grep "error" | sort | uniq -c`，每个命令是一个过滤器，`|` 是管道，数据逐行流过。
- **编译器**：词法分析→语法分析→语义分析→中间代码生成→优化→目标代码生成，每一步读上一步输出、产生下一步输入。

**优点**：

- **松耦合**：过滤器只认输入输出格式，不认彼此，换一个过滤器不影响其他。
- **可并行**：多个过滤器可同时跑，提升吞吐。
- **可复用**：过滤器像积木，能拼到不同流水线里。
- **支持流式**：不用等数据全部到齐，适合实时处理。

**缺点**：

- **不适合交互**：数据单向流动，难以回传控制信号。
- **错误处理难**：某个过滤器出错，错误顺着流往下传，难定位源头。
- **性能开销**：数据在过滤器间频繁搬运、可能格式转换，有额外开销。
- **难处理共享状态**：过滤器独立、无状态，需要共享数据的场景不适合。

### 批处理 vs 管道-过滤器：四维对比

这是软考最爱考的辨析，背下这张表：

| 对比维度 | 批处理风格 | 管道-过滤器风格 |
|---|---|---|
| **数据形态** | 完整批量（一坨） | 数据流（一股） |
| **处理时机** | 整批到齐才处理 | 边到边处理（流式） |
| **耦合度** | 低（通过完整文件交接） | 极低（只认输入输出格式） |
| **并行性** | 无（串行） | 有（过滤器可并行） |
| **中间结果** | 通常持久化为文件 | 通常不持久化，流式传递 |
| **实时性** | 差（批处理周期） | 好（流式实时） |
| **典型场景** | 银行日终结算、月度报表 | 编译器、Unix 管道、ETL、流处理 |

记忆抓手：**批处理是"等齐再动"，管道-过滤器是"边来边干"**。

### 一个易混点

考生常把"管道-过滤器"和"调用返回里的层次架构"搞混。区分关键看**数据怎么走**：

- 层次架构：上层**调用**下层，下层**返回**结果，控制流是来回的（请求向下、响应向上）。
- 管道-过滤器：数据**单向**流动，过滤器之间没有"返回"，只有"传给下一个"。

编译器里如果你看到"调用语法分析模块、等它返回"，那是调用返回；如果看到"词法分析的输出流进语法分析的输入"，那是管道-过滤器。软考里编译器一般按**管道-过滤器**考，因为它强调"前一阶段输出作为后一阶段输入"的数据流动。

## 实践

### 真题示例

**考法一：归类题**

> 例：某数据清洗系统将原始数据依次经过去重、格式转换、缺失值填充、异常值剔除等步骤处理，每个步骤独立运行，前一步输出直接作为后一步输入，可并行执行。该系统属于（　）架构风格。
> A. 批处理　B. 管道-过滤器　C. 层次　D. 主程序-子程序

**答题思路**：抓两个关键词——"前一步输出直接作为后一步输入"（数据流动）+ "可并行执行"（过滤器并行）。这是管道-过滤器的标志性特征。答案 **B**。注意区分批处理：题干强调"可并行"，批处理是串行的，所以不是 A。

**考法二：辨析题**

> 例：以下关于批处理风格与管道-过滤器风格的比较，正确的是（　）。
> A. 两者都支持过滤器并行执行
> B. 批处理以完整数据交接，管道-过滤器以数据流交接
> C. 管道-过滤器必须等全部数据到齐才处理
> D. 批处理实时性优于管道-过滤器

**答题思路**：B 对。A 错（批处理不并行）；C 错（管道-过滤器是流式，边到边处理）；D 错（管道-过滤器实时性更好）。答案 **B**。

### 答题踩分词

案例题里若选管道-过滤器风格，理由段要写出这些踩分词：**数据单向流动、过滤器独立、松耦合、可并行、流式处理、支持增量处理**。

若要否定批处理，踩分词：**串行无并行、需整批到齐、实时性差**。

### 案例论文应用

数据流风格在论文里通常作为"特定场景的选型论证"出现。比如写数据中台、ETL 系统的论文：

> "数据清洗链路采用管道-过滤器风格：去重、格式转换、字段映射、质量校验等过滤器独立部署，通过 Kafka 主题作为管道串联。该风格使各过滤器可独立开发部署、可并行扩容，且新增清洗规则只需插入新过滤器，不影响已有链路。"

这种写法既交代了风格，又讲了组件怎么落、为什么合适，比单纯说"用了管道-过滤器"扎实得多。

### 记忆口诀

- 数据流两子类：**"批处理等齐动，管道过滤边到边"**。
- 归类抓手：**"前出后入、单向流动、可并行"= 管道-过滤器**。
- 编译器归属：**"编译器是管道过滤器的活招牌"**。

### 自测题

**自测 1**：Unix 命令 `cat log.txt | grep "ERROR" | wc -l` 体现了哪种架构风格？说明理由。

> 参考答案：管道-过滤器风格。每个命令是一个过滤器，`|` 是管道，数据逐行流过，过滤器独立、可并行、单向流动，符合管道-过滤器特征。

**自测 2**：批处理风格和管道-过滤器风格在"数据形态"和"并行性"上有何区别？

> 参考答案：数据形态上，批处理以完整批量交接，管道-过滤器以数据流交接；并行性上，批处理串行无并行，管道-过滤器的过滤器之间可并行。

**自测 3**：某系统要求"前一步全部处理完并落盘后，下一步才开始处理"，这更接近哪种风格？为什么不能用管道-过滤器？

> 参考答案：更接近批处理风格。管道-过滤器是流式边到边处理、可并行，而该系统要求"全部处理完再开始"，是典型的整批交接、串行执行，符合批处理特征。

### 一句话小结

数据流风格靠"数据自己跑流水线"组织系统——**记住批处理等齐动、管道过滤边到边，编译器和 ETL 归这里就稳了**。
