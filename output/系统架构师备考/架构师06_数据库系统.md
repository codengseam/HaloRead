# 第6章：从SQL到分布式事务，数据库的规范化与一致性

数据库是架构师的看家本领，规范化到第几范式、事务隔离怎么选、分布式下 CAP 怎么取舍，每一条都直接落到线上系统设计。

## 概念

综合知识里数据库 **4-5 题/年**，是单章分值最高的之一；**案例分析必有一道数据库题**（ER 图、SQL、规范化），论文里讲数据架构也必涉及。这一章投入产出比最高。

核心类比：数据库表像"档案柜"。1NF 是抽屉里不塞散纸（每格只存一个值）；2NF 是每份档案只依赖档案号，不依赖档案号+抽屉号的组合；3NF 是档案里不写"部门名"（部门名依赖部门号，不直接依赖员工号）；BCNF 是连决定因素都必须是主键。范式越高，重复越少，但查询要 join 更多表。

## 原理

### 关系代数

- **选择 σ**：按条件选行（对应 WHERE）。
- **投影 π**：选列（对应 SELECT 列）。
- **连接 ⋈**：按条件拼两张表（对应 JOIN）。
- **除法 ÷**：选"包含全部"的行，常考"选了所有课程的学生"。

### SQL 三类

- **DDL（Data Definition Language，定义）**：CREATE/ALTER/DROP，改结构。
- **DML（Data Manipulation Language，操纵）**：INSERT/UPDATE/DELETE，改数据。
- **DCL（Data Control Language，控制）**：GRANT/REVOKE，管权限。

连接查询：内连接（取交集）、左外连接（左全保留）、右外连接、全外连接。子查询：IN/EXISTS/ANY/ALL，EXISTS 通常比 IN 快（相关子查询走索引）。

### 规范化（必考反例）

函数依赖：X → Y 表示 X 决定 Y。

| 范式 | 要求 | 反例 |
|---|---|---|
| 1NF | 属性不可分 | 一个电话字段塞"座机,手机" |
| 2NF | 非主属性完全依赖主键（消除部分依赖） | 主键(学号,课程号)却存学生姓名（只依赖学号） |
| 3NF | 非主属性不传递依赖主键（消除传递依赖） | 员工表存部门号又存部门名 |
| BCNF | 每个决定因素都必须是候选键 | 仓库(仓库,物品)→管理员，管理员→仓库 |

记法：1NF 原子性，2NF 无部分依赖，3NF 无传递依赖，BCNF 决定因素全主键。规范化越高冗余越少但 join 越多，**实际项目常做到 3NF 后按查询性能反规范化**。

### 并发控制

锁：共享锁 S（读锁）、排他锁 X（写锁）。S-S 兼容，X 与任何锁互斥。

事务隔离级别（由弱到强）：读未提交 < 读已提交 < 可重复读 < 串行化。

| 级别 | 脏读 | 不可重复读 | 幻读 |
|---|---|---|---|
| 读未提交 | 有 | 有 | 有 |
| 读已提交 | 无 | 有 | 有 |
| 可重复读 | 无 | 无 | 有 |
| 串行化 | 无 | 无 | 无 |

两阶段锁协议：先扩展（加锁）后收缩（解锁），可保证串行化但不能防死锁。

### 分布式数据库

**两阶段提交 2PC（Two-Phase Commit）**：协调者发 prepare → 各参与者表决 → 协调者发 commit/abort。问题：协调者宕机会阻塞，单点故障。

**CAP 定理**：一致性 C、可用性 A、分区容错 P 三选二。网络分区不可避免，实际在 CP 和 AP 间取舍。强一致选 CP（如 ZooKeeper），高可用选 AP（如 Cassandra）。

**BASE**：基本可用、软状态、最终一致。是 AP 牺牲强一致后的补偿方案，电商大促常用。

### 数据仓库与数据挖掘

OLTP vs OLAP：OLTP 是日常交易（增删改多，行式存储），OLAP 是分析查询（聚合多，列式存储）。

ETL（Extract-Transform-Load）：抽取-转换-加载，把业务库数据搬到仓库。

模型：星型模型（一张事实表 + 多张维表，简单快）、雪花模型（维表再规范化，省空间慢）。

数据挖掘：聚类（无监督，如 K-means）、分类（有监督，决策树）、关联规则（Apriori，"啤酒尿布"）。

## 实践

**真题示例（2024年·综合知识·规范化相关，考生回忆）**：给关系模式 R(A,B,C,D)，函数依赖 A→B, B→C，问属于几范式、最高可到几范式。

答题公式：先找候选键（这里 A→B→C，若 A→D，则 A 是键），再看依赖。A→B 是完全依赖，B→C 是传递依赖（A→B→C），所以 **3NF 不满足，属于 2NF**。要消除传递依赖需拆成 R1(A,B,D) 和 R2(B,C)，可达 BCNF。

**案例数据库题**：ER 图转关系模式 → 指出函数依赖 → 判范式 → 拆表到 3NF/BCNF。踩分词：候选键、部分依赖、传递依赖、无损连接、保持函数依赖。

**论文应用**：讲"数据架构演进"时，CAP 取舍是黄金素材——电商从强一致（订单库 CP）到最终一致（库存缓存 AP）。讲数据仓库时引星型 vs 雪花、ETL 流程。

**踩分词**：候选键、部分/传递依赖、BCNF、2PC、CAP、BASE、星型模型、OLAP。

## 速查/自测

**30 秒速查卡**

| 考点 | 一句话 |
|---|---|
| 关系代数 | σ选行 π选列 ⋈连接 ÷除法 |
| 1NF | 属性不可分 |
| 2NF | 无部分依赖 |
| 3NF | 无传递依赖 |
| BCNF | 决定因素全候选键 |
| CAP | C/A/P 三选二 |
| 隔离级别 | 串行化最高，无幻读 |

**口诀**：1 原子 2 部分 3 传递，BCNF 决定因素全是键；CAP 三选二，分区不可避；BASE 最终一致，大促不宕机。

**自测题**：
1. R(学号,课程号,姓名,成绩)，主键(学号,课程号)，姓名只依赖学号，违反几范式？答：违反 2NF（部分依赖），只到 1NF。
2. 两阶段提交的协调者宕机会怎样？答：参与者阻塞等待，**无法 commit 也无法 abort**，需人工介入。
3. 电商秒杀库存系统该选 CP 还是 AP？答：通常 AP + 最终一致，避免超卖用补偿，强一致会让系统扛不住。

一句话小结：数据库拿分靠"范式反例 + CAP 取舍 + 隔离级别表"，案例题务必画出函数依赖链再判范式。
