# 需求没搞清，架构就是空中楼阁

需求工程是软件工程的起点，也是案例题 DFD 画图题的命题高地——把 DFD 三层画法和需求管理流程吃透，案例至少多拿 10 分。

## 概念

这章考什么：
- **科目**：综合知识（2-3 题）、案例分析（DFD 高频，几乎每年必考）
- **分值**：综合约 3 分，案例 10-15 分
- **考频**：DFD 是案例"钉子户"

核心类比：需求工程像盖房前的勘探。需求获取是"问客户要什么房"（访谈/问卷/原型）；需求分析是"画施工图"（DFD/用例图）；SRS 是"合同"；需求评审是"客户签字"；需求管理是"施工中改图纸的流程"。

## 原理

### 一、需求工程五大活动

需求工程 = 需求开发（获取→分析→编写 SRS→验证）+ 需求管理（基线 + 变更控制）。

**1. 需求获取（Elicitation）**
- **访谈**：一对一深入，结构化（按提纲）/ 非结构化（自由），适合关键干系人
- **问卷**：广覆盖，适合大样本用户，深度不足
- **JAD 联合应用开发**：多干系人集中开会，1-2 周出共识，效率高
- **原型**：需求模糊时搭可演示原型，比文字直观
- **观察法**：跟随用户工作，挖掘隐性需求

**2. 需求分析（Analysis）**
两条技术路线：
- **结构化分析 SA**：用 DFD（数据流图）+ DD（数据字典）+ 加工说明（小说明/判定表/判定树）。面向过程。
- **面向对象分析 OOA**：用用例图 + 类图 + 状态图。用例驱动，识别"参与者-用例-关系"。

**3. 需求规格说明书 SRS**
需求工程的交付物。内容含功能需求、非功能需求（性能/安全/可用/可靠）、约束条件、接口需求。SRS 应满足 IEEE 830 标准：正确、完整、一致、可验证、可追踪、可修改。

**4. 需求验证（Verification）**
- **需求评审**：技术评审 + 用户签字
- **原型验证**：用原型走查
- **形式化验证**：数学方法验证关键需求

**5. 需求管理（Management）**
- **需求基线**：经评审通过的 SRS 冻结为基线，后续变更走流程
- **变更控制**：变更申请→影响分析→CCB（变更控制委员会）审批→实施→验证
- **需求跟踪**：双向跟踪矩阵，从需求→设计→代码→测试用例

### 二、DFD 数据流图（案例必考）

DFD 是 SA 的核心图，描述系统逻辑功能与数据流向。**图元规范（必背）**：

| 图元 | 符号 | 含义 |
|---|---|---|
| 外部实体 | 方框 | 系统外的人/外部系统，数据源/汇 |
| 加工 | 圆（圆角矩形也可） | 对数据的处理转换 |
| 数据存储 | 双横线（开口矩形） | 数据保存位置，文件/数据库 |
| 数据流 | 带箭头的线 | 数据流向，必须命名 |

**DFD 分层画法**：
- **0 层（顶层）**：把整个系统看作一个加工，画外部实体与系统间的数据流。叫"上下文图"。
- **1 层**：把 0 层那个加工分解为若干子加工（通常 3-7 个），画出存储。
- **2 层**：继续对 1 层复杂加工分解，直到每个加工足够简单（一个加工一个动作）。

**画图纪律**：
1. 数据流必须命名，不能空箭头
2. 数据守恒：进入加工的数据 = 加工输出 + 加工读存储 - 加工写存储
3. 父图子图平衡：父图中某加工的输入输出流 = 子图边界流
4. 外部实体不出现在子图（只出现在 0 层）
5. 数据存储一般从 1 层开始出现

### 三、需求分类

- **功能需求**：系统"做什么"——输入、处理、输出
- **非功能需求**：性能、安全、可靠性、可用性、可维护性
- **设计约束**：必须用某技术、必须运行在某平台
- **接口需求**：UI、硬件、软件、通信接口

易混点：
- **业务需求 vs 用户需求 vs 系统需求**：业务需求是组织目标（高层），用户需求是用户任务描述，系统需求是系统应具备的功能/性能（SRS 主体）。
- **功能 vs 非功能**：响应时间≤2 秒是非功能；登录验证是功能。

## 实践

**真题示例（2024 年·案例分析·DFD 相关·考生回忆）**：给出系统说明，要求补全 0 层/1 层 DFD 的数据流命名和加工。

**DFD 答题四步公式**：
1. **找外部实体**：从题干"用户/管理员/外部系统"提取，画方框
2. **找数据流**：题干动词+名词（"提交订单""查询库存"），命名箭头
3. **找加工**：系统内部处理动作，画圆，编号（1, 1.1, 1.2）
4. **找存储**：题干"文件/表/库"，画双横线，连接读写加工

**踩分词**：外部实体名称、数据流名称、加工编号、数据存储命名——每个空 1-2 分，命名要"名词+动词"或"动宾短语"。

**案例变更控制答题模板**：
"变更应走 CCB 流程：① 提交变更申请 ② 进行影响分析（成本/进度/质量） ③ CCB 审批 ④ 实施变更 ⑤ 验证并更新基线 ⑥ 通知相关干系人。"

**论文应用**：论文"需求分析"段写"本项目采用 JAD 会议集中获取需求，输出 SRS 经评审签字后建立基线，所有变更走 CCB 流程，保证需求可追溯"，体现规范工程实践。

## 速查/自测

**30 秒速查卡**：

| 活动 | 关键产物 |
|---|---|
| 需求获取 | 用户需求列表 |
| 需求分析（SA） | DFD + DD |
| 需求分析（OOA） | 用例图 + 类图 |
| 编写 SRS | 需求规格说明书 |
| 需求验证 | 评审签字 |
| 需求管理 | 基线 + 变更记录 |

**DFD 图元口诀**：方框是外部，圆圈是加工，双横是存储，箭头是数据流。

**自测题**：
1. DFD 0 层图把系统画成几个加工？→ 1 个（整个系统）
2. 数据存储从第几层开始出现？→ 1 层
3. CCB 是什么？→ Change Control Board 变更控制委员会，审批需求变更
4. 父图子图平衡是什么？→ 父图某加工的输入输出流 = 子图边界流

**一句话小结**：DFD 三层 + 变更控制流程是案例必拿分——图元规范背死，画图纪律记牢。
