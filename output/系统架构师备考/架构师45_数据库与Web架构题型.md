# 第45章：数据库与Web架构题，ER图+规范化+MVC三连击

数据库与 Web 架构是案例分析的高频题型，分值通常 10~15 分。考法分两条线：数据库线考 ER 图补全、关系模式规范化、SQL 优化；Web 线考 B/S 多层架构设计、MVC/MVP/MVVM 选型、REST 接口设计。失分点集中在**规范化级别判断错**和**MVC 三模式区别讲不清**。这一章把 ER 建模、三大范式、SQL 优化、MVC 三模式、DFD 数据流图全部钉死，再配三道完整模拟题（ER+规范化、Web 架构、SQL 优化），把这道题变成"按规则填空"。

## 概念

### 这章考什么

- **综合知识（单选）**：考 **2~3 题**。常考范式定义、ER 图元、MVC 三模式区别、REST 特征。
- **案例分析**：**高频**。典型考法："请补全 ER 图""请说明关系模式满足第几范式、如何规范化""请设计 Web 架构""请说明 MVC/MVP/MVVM 的区别"。分值 **10~15 分**。
- **论文**：偶考。论文写"Web 系统架构"时，MVC 选型和多层架构是核心论证点。

考频结论：案例高频，必练。

### 题型特征

数据库与 Web 架构题的题干结构稳定，分两条线：

**数据库线**

```
某 [业务场景] 系统，[场景描述，含若干实体和关系]。
1. 请补全 ER 图中的实体/联系/属性。
2. 请说明关系模式 R 满足第几范式，若不满足 BCNF 请规范化。
3. 请给出某查询 SQL 的优化建议。
```

**Web 线**

```
某 [Web 业务] 系统，[需求描述]。
1. 请设计该系统的 Web 架构（多层）。
2. 请说明采用 MVC/MVP/MVVM 哪种模式及理由。
3. 请设计某功能的 RESTful 接口。
```

有时两线混合（先 ER 后 Web），需分别套公式。

### ER 图核心元素

ER 图（Entity-Relationship Diagram）三要素：

| 元素 | 符号 | 含义 |
|---|---|---|
| **实体（Entity）** | 矩形 | 客观存在的事物（如学生、课程） |
| **属性（Attribute）** | 椭圆 | 实体的特征（如学号、姓名） |
| **联系（Relationship）** | 菱形 | 实体间的关系（如选课） |
| **连接线** | 实线 | 实体-属性、实体-联系的连接 |

联系的三种基数：

| 基数 | 表示 | 例子 |
|---|---|---|
| **1:1** | 1—1 | 一个丈夫对应一个妻子 |
| **1:N** | 1—N | 一个部门对应多个员工 |
| **M:N** | M—N | 一个学生选多门课，一门课被多学生选 |

记忆抓手：1:1 罕见，1:N 常见（外键在 N 端），M:N 需新建关系表（含两端主键）。

### 关系模式规范化（必背）

详细内容见第 6 章，这里只放案例答题用的速查表。规范化（Normalization）的目标是消除冗余和异常，范式从低到高：

| 范式 | 要求 | 一句话 |
|---|---|---|
| **1NF** | 每个属性不可再分（原子性） | 字段不嵌套 |
| **2NF** | 满足 1NF + 非主属性完全依赖候选键（无部分依赖） | 非主属性不依赖主键的一部分 |
| **3NF** | 满足 2NF + 非主属性不传递依赖候选键（无传递依赖） | 非主属性不依赖其他非主属性 |
| **BCNF** | 满足 3NF + 每个决定因素都包含候选键 | 决定因素必须是候选键 |

记忆抓手：**1NF 原子、2NF 无部分依赖、3NF 无传递依赖、BCNF 决定因素必含候选键**。

判别流程：

```
看属性是否原子 → 不原子则不满足 1NF
↓ 满足
看非主属性是否完全依赖候选键 → 有部分依赖则只满足 1NF
↓ 满足
看非主属性是否传递依赖候选键 → 有传递依赖则只满足 2NF
↓ 满足
看每个决定因素是否含候选键 → 不含则只满足 3NF
↓ 满足
BCNF
```

### MVC / MVP / MVVM 三模式区别（高频易混）

| 模式 | 全称 | 控制器角色 | 通信方向 | 典型场景 |
|---|---|---|---|---|
| **MVC** | Model-View-Controller | Controller 中转 | View ↔ Controller ↔ Model，Model 可通知 View | Web 后端（Spring MVC） |
| **MVP** | Model-View-Presenter | Presenter 中转且包揽 View 逻辑 | View ↔ Presenter ↔ Model，单向，Model 不通知 View | Android、桌面 |
| **MVVM** | Model-View-ViewModel | ViewModel 双向绑定 | View ↔ ViewModel（双向绑定） ↔ Model | Vue、WPF、前端框架 |

三模式核心区别：

- **MVC**：Model 可直接通知 View（观察者模式），Controller 只中转用户输入。
- **MVP**：View 和 Model 完全隔离，所有交互经 Presenter，Presenter 持有 View 接口。
- **MVVM**：ViewModel 和 View 双向数据绑定，View 改变自动同步 ViewModel，反之亦然，无需手动更新。

记忆抓手：**MVC 是爷爷（最老，Model 能通知 View）、MVP 是爸爸（隔离 View 和 Model，Presenter 包揽）、MVVM 是儿子（双向绑定，自动同步）**。

### DFD 数据流图层与图元

DFD（Data Flow Diagram，数据流图）分层：

| 层级 | 名称 | 内容 |
|---|---|---|
| **0 层** | 顶层 | 系统作为一个加工，外部实体和数据流 |
| **1 层** | 一层 | 把 0 层加工分解为若干子加工 |
| **2 层** | 二层 | 把 1 层子加工进一步分解 |

DFD 图元规范（必须画对）：

| 图元 | 符号 | 含义 |
|---|---|---|
| **外部实体** | 方框 | 系统外数据源/汇（如用户、外部系统） |
| **加工** | 圆或圆角矩形 | 数据处理过程 |
| **数据存储** | 双横线（开口矩形） | 数据保存（如文件、数据库表） |
| **数据流** | 箭头 | 数据流向，需标注数据名 |

DFD 画法要求：①每个加工至少有一个输入流和一个输出流；②数据流需命名；③分层时父子图平衡（父图某加工的输入输出 = 子图整体输入输出）。

## 原理

### 答题公式

数据库与 Web 架构题分两条线，分别套公式。

#### 数据库线三步公式

```
① ER 建模（实体/属性/关系 1:1 / 1:N / M:N）
→ ② 关系模式规范化（1NF → 2NF → 3NF → BCNF）
→ ③ SQL 优化（索引/连接/子查询）
```

**第 1 步：ER 建模**

读题，提取实体（名词）、属性（特征）、联系（动词+基数）。

- 实体：题干里的核心名词（如学生、课程）。
- 属性：实体的特征（如学号、姓名、课程号、课程名）。
- 联系：实体间的关系（如"选课"是 M:N）。

ER 图补全题的踩分点：实体名、联系名、属性名、联系基数（1:1/1:N/M:N）。

**第 2 步：关系模式规范化**

判别范式级别（按上面流程图），若不满足 BCNF 则分解。

规范化分解原则：

- **2NF 分解**：把部分依赖的属性单独成表，主键为依赖的那部分。
- **3NF 分解**：把传递依赖的属性单独成表，主键为中间属性。
- **BCNF 分解**：把非候选键的决定因素及其依赖属性单独成表。

例：关系模式 R（学号，姓名，系号，系名），函数依赖：学号→姓名、系号，系号→系名。
- 学号→系号→系名 是传递依赖，**只满足 2NF**。
- 分解：R1（学号，姓名，系号），R2（系号，系名），均满足 3NF 且 BCNF。

**第 3 步：SQL 优化**

常见优化手段：

| 手段 | 说明 |
|---|---|
| **建索引** | 在 WHERE/JOIN/ORDER BY 涉及的列上建索引 |
| **避免 SELECT \*** | 只查需要的列，减少 IO |
| **避免子查询** | 用 JOIN 替代子查询，减少临时表 |
| **避免前置 %** | LIKE '%abc' 不走索引，LIKE 'abc%' 走索引 |
| **LIMIT 分页** | 大结果集用 LIMIT 分页，避免一次取全表 |
| **避免函数包裹索引列** | WHERE YEAR(date)=2024 不走索引，改成范围查询 |

#### Web 线四步公式

```
① 识别需求（业务特征 + 质量属性）
→ ② 选 B/S 多层架构（表现/业务/数据）
→ ③ MVC / MVP / MVVM 选型
→ ④ REST 接口设计
```

**第 1 步：识别需求**

读题，提取业务特征（高并发、跨平台、富交互等）和质量属性（性能、可修改性、可测试性等）。

**第 2 步：选 B/S 多层架构**

Web 系统通常选 B/S（Browser/Server）三层架构：

- **表现层（Presentation）**：Web 前端，负责 UI 展示和用户交互。
- **业务逻辑层（Business Logic）**：后端服务，负责业务规则处理。
- **数据访问层（Data Access）**：数据库访问，负责持久化。

复杂系统可加"服务层""缓存层"等，但答题时三层最稳。

**第 3 步：MVC / MVP / MVVM 选型**

按场景选：

| 场景 | 选型 | 理由 |
|---|---|---|
| Web 后端（Spring MVC 等） | MVC | Controller 中转，Model 可通知 View，适合请求-响应模型 |
| Android / 桌面应用 | MVP | View 和 Model 隔离，Presenter 包揽 View 逻辑，便于单元测试 |
| 前端富交互（Vue、WPF） | MVVM | 双向绑定，View 改变自动同步 ViewModel，减少手动 DOM 操作 |

**第 4 步：REST 接口设计**

REST（Representational State Transfer，表述性状态转移）核心原则：

- **资源导向**：URL 表示资源（如 `/users/123`），不是动作。
- **HTTP 方法语义化**：GET 查、POST 增、PUT 改、DELETE 删。
- **无状态**：每个请求包含所有信息，服务器不保存会话状态。
- **统一接口**：资源通过统一接口访问。
- **分层**：客户端不感知中间层。

REST 接口设计示例：

| 操作 | HTTP 方法 | URL | 示例 |
|---|---|---|---|
| 查列表 | GET | /users | GET /users?page=1 |
| 查单个 | GET | /users/{id} | GET /users/123 |
| 新增 | POST | /users | POST /users（body 含用户数据） |
| 全量更新 | PUT | /users/{id} | PUT /users/123（body 含完整数据） |
| 部分更新 | PATCH | /users/{id} | PATCH /users/123（body 含部分字段） |
| 删除 | DELETE | /users/{id} | DELETE /users/123 |

### 踩分点清单

阅卷老师按以下踩分点给分：

**数据库线**：

1. **ER 图元齐全**：实体、属性、联系、基数都画对。
2. **范式级别准确**：判别 1NF/2NF/3NF/BCNF 不能错。
3. **规范化分解正确**：分解后每关系满足更高范式，且无损连接。
4. **SQL 优化建议具体**：不能只说"加索引"，要说加在哪列。

**Web 线**：

5. **多层架构清晰**：表现层/业务层/数据层分清。
6. **MVC 三模式区别准确**：不能把 MVC 说成 MVP。
7. **REST 接口规范**：URL 资源导向、HTTP 方法语义化。
8. **结合题干**：选型理由要结合业务特征。

### 通用答题模板

```
[数据库线]
第1步 ER 建模：
  实体：A（属性 a1, a2）、B（属性 b1, b2）
  联系：A-B 为 M:N（联系名 R，属性 r1）

第2步 规范化：
  关系模式 R(A, B, C, D)，函数依赖：A→B, A→C, C→D。
  A→C→D 是传递依赖，R 满足 2NF 不满足 3NF。
  分解：R1(A, B, C)，R2(C, D)，均满足 BCNF。

第3步 SQL 优化：
  原 SQL：SELECT * FROM t WHERE ...
  优化：①在 WHERE 列建索引；②避免 SELECT *，只查必要列；
        ③用 JOIN 替代子查询；④...

[Web 线]
第1步 需求：业务特征 XX，质量属性 XX。
第2步 架构：采用 B/S 三层架构（表现层/业务层/数据访问层）。
第3步 选型：采用 MVC/MVP/MVVM，理由 XX。
第4步 REST：接口示例 GET /users/{id}、POST /users 等。
```

## 实践

### 模拟题一：ER 图补全 + 规范化

> **题目**：
> 某图书馆管理系统，需管理图书、读者、借阅记录。已知：
> - 图书（Book）：书号、书名、作者、出版社、库存数
> - 读者（Reader）：读者号、姓名、电话、所属部门
> - 一名读者可借多本图书，一本图书可被多名读者借阅，每次借阅记录借阅日期和应还日期。
>
> 关系模式 R（读者号，姓名，电话，部门，书号，书名，作者，出版社，借阅日期，应还日期）。
> 函数依赖：读者号→姓名、电话、部门；书号→书名、作者、出版社；（读者号，书号）→借阅日期、应还日期。
>
> 请回答：
> 1. 补全 ER 图（实体、联系、基数）。
> 2. 关系模式 R 满足第几范式？说明理由并规范化到 BCNF。

#### 分步解答

**第 1 步：ER 图**

```
实体：
  - Book（图书）：属性书号、书名、作者、出版社、库存数
  - Reader（读者）：属性读者号、姓名、电话、所属部门

联系：
  - 借阅（Borrow）：Reader 与 Book 的 M:N 联系
    属性：借阅日期、应还日期
    基数：M:N（一名读者借多本图书，一本图书被多名读者借阅）

ER 图描述：
  [Reader] —< 借阅 >— [Book]
              |
           借阅日期、应还日期
```

**踩分点标注**：①两个实体名准确；②M:N 基数准确；③联系属性（借阅日期、应还日期）准确。

**第 2 步：规范化**

```
关系模式 R（读者号，姓名，电话，部门，书号，书名，作者，出版社，
借阅日期，应还日期），候选键为（读者号，书号）。

函数依赖：
  读者号 → 姓名、电话、部门  （部分依赖，依赖候选键的一部分）
  书号 → 书名、作者、出版社   （部分依赖，依赖候选键的一部分）
  （读者号，书号）→ 借阅日期、应还日期  （完全依赖）

判别：R 存在部分依赖（读者号、书号各自决定部分非主属性），
      不满足 2NF，只满足 1NF。

规范化（分解到 BCNF）：
  R1（读者号，姓名，电话，部门）—— 主键读者号，BCNF
  R2（书号，书名，作者，出版社）—— 主键书号，BCNF
  R3（读者号，书号，借阅日期，应还日期）—— 主键（读者号，书号），BCNF
    其中读者号是外键参照 R1，书号是外键参照 R2。

分解后三个关系均满足 BCNF，且无损连接、保持函数依赖。
```

**踩分点标注**：①判别 R 满足 1NF（存在部分依赖）；②分解为三个关系；③每个关系主键标注；④外键关系说明；⑤说明满足 BCNF 且无损连接。

### 模拟题二：Web 架构设计 + MVC 选型

> **题目**：
> 某在线教育平台，需支持 Web 端和移动端 H5 访问，功能包括课程浏览、视频播放、订单购买、用户中心。要求高并发、易扩展、前后端可独立演进。请回答：
>
> 1. 请设计该系统的 Web 架构（多层）。
> 2. 前端采用 MVVM 还是 MVC？后端采用 MVC 还是 MVP？说明理由。
> 3. 请设计"获取用户订单列表"的 RESTful 接口。

#### 分步解答

**第 1 步：Web 架构**

```
采用 B/S 三层架构（增加缓存层和 CDN）：

1. 表现层（Presentation）：
   - Web 前端（H5/SPA）：Vue.js 或 React，负责 UI 展示和交互
   - 移动端 H5：复用 Web 前端响应式设计
   - CDN：缓存静态资源（视频、图片、JS/CSS），加速访问

2. 业务逻辑层（Business Logic）：
   - 后端服务（Java/Spring Boot 或 Node.js）
   - 按业务拆分微服务：课程服务、订单服务、用户服务、视频服务
   - 网关：统一入口，做鉴权、限流、路由

3. 数据访问层（Data Access）：
   - 数据库：MySQL（订单、用户等结构化数据）
   - 缓存：Redis（热点课程、会话）
   - 对象存储：OSS/S3（视频、图片文件）

辅助层：
   - 消息队列：Kafka（异步处理订单、通知）
   - 监控：Prometheus + Grafana
```

**踩分点标注**：①三层架构清晰；②表现层含 CDN（高并发加速）；③业务层含微服务+网关；④数据层含数据库+缓存+对象存储；⑤辅助层（消息队列、监控）。

**第 2 步：MVC/MVVM 选型**

```
前端采用 MVVM，后端采用 MVC。

前端 MVVM 理由：
  在线教育前端为富交互 SPA（视频播放、订单购买、用户中心），
  需要频繁更新 UI。MVVM 的双向绑定让数据变化自动同步到 View，
  无需手动操作 DOM，开发效率高、Bug 少。典型框架 Vue.js 即 MVVM。
  不选 MVC：MVC 的 Model 可直接通知 View，前端场景下 View 逻辑
  复杂，MVC 会导致 Controller 臃肿。

后端 MVC 理由：
  后端为请求-响应模型（HTTP 请求→处理→返回 JSON），适合 MVC。
  Controller 接收请求并中转，Model 处理业务，View 返回 JSON
  （或模板）。Spring MVC 是典型实现。
  不选 MVP：MVP 主要用于桌面/移动端，Presenter 包揽 View 逻辑，
  后端无 GUI View，不适合。
  不选 MVVM：后端无 View 双向绑定需求。
```

**踩分点标注**：①前端选 MVVM，理由结合富交互；②后端选 MVC，理由结合请求-响应模型；③对比排除其他模式；④典型框架提及（Vue.js、Spring MVC）。

**第 3 步：REST 接口**

```
获取用户订单列表接口：
  HTTP 方法：GET
  URL：/users/{userId}/orders
  路径参数：userId（用户 ID）
  查询参数：page（页码）、size（每页条数）、status（订单状态筛选）
  示例：GET /users/123/orders?page=1&size=20&status=paid
  返回：订单列表 JSON

设计原则：
  ①资源导向：URL 表示资源（用户订单），不含动词
  ②HTTP 方法语义化：GET 表示查询
  ③无状态：每个请求含 userId 等所有信息
  ④分页参数：page、size 标准化
```

**踩分点标注**：①HTTP 方法 GET；②URL 资源导向（/users/{userId}/orders）；③分页参数；④设计原则四条说明。

### 模拟题三：SQL 优化

> **题目**：
> 某订单系统查询缓慢，原 SQL 如下：
> ```sql
> SELECT * FROM orders o, customers c
> WHERE o.customer_id = c.id
>   AND c.city = '北京'
>   AND YEAR(o.create_time) = 2024
> ORDER BY o.amount DESC;
> ```
> 订单表 orders 约 1000 万行，客户表 customers 约 50 万行。请给出至少 4 条优化建议。

#### 分步解答

```
优化建议：

1. [避免 SELECT *] 只查询必要列，减少 IO 和网络传输。
   改为：SELECT o.order_id, o.amount, c.name, c.city FROM ...

2. [用 JOIN 替代隐式连接] 隐式连接（FROM A, B WHERE）可读性差且
   优化器可能选错执行计划，改用显式 JOIN。
   改为：FROM orders o JOIN customers c ON o.customer_id = c.id

3. [建索引]
   ①customers.city 建索引（加速城市筛选）；
   ②orders.customer_id 建索引（加速连接）；
   ③orders.create_time 建索引（加速时间筛选）；
   ④orders.amount 建索引（加速排序）。
   可考虑复合索引：(customer_id, create_time, amount)。

4. [避免函数包裹索引列] YEAR(o.create_time) = 2024 会导致
   create_time 索引失效，改为范围查询。
   改为：o.create_time >= '2024-01-01' AND o.create_time < '2025-01-01'

5. [分页] 如前端只需展示部分结果，加 LIMIT 分页。
   加：LIMIT 20 OFFSET 0

优化后 SQL：
   SELECT o.order_id, o.amount, c.name, c.city
   FROM orders o JOIN customers c ON o.customer_id = c.id
   WHERE c.city = '北京'
     AND o.create_time >= '2024-01-01'
     AND o.create_time < '2025-01-01'
   ORDER BY o.amount DESC
   LIMIT 20;
```

**踩分点标注**：①至少 4 条建议；②每条说明原因；③建索引具体到列；④YEAR 函数问题指出并改范围查询；⑤优化后 SQL 完整。

### 易错提醒

**易错 1：范式级别判错**

最常见错误：把存在部分依赖的关系误判为 3NF。判别流程要按"原子→部分依赖→传递依赖→决定因素含候选键"四步走，**有部分依赖就只满足 1NF，不是 2NF**。

**易错 2：M:N 联系不新建关系表**

ER 图里 M:N 联系，转换为关系模式时必须新建一个关系表（含两端主键 + 联系自身属性），不能把外键直接放到任一实体表。例：学生-课程 M:N，需新建"选课"表（学号，课程号，成绩）。

**易错 3：MVC 三模式区别讲不清**

很多人答"MVC 就是 Model View Controller"——没说区别。必须答：**MVC 的 Model 可通知 View、MVP 的 View 和 Model 完全隔离经 Presenter、MVVM 双向绑定**。三者的核心差异在"View 和 Model 是否直接通信"和"控制器的包揽程度"。

**易错 4：REST 接口 URL 含动词**

`/getUserOrders`、`/deleteOrder` 这种 URL 不符合 REST（含动词）。REST 的 URL 应**资源导向**：`/users/{id}/orders`、用 DELETE 方法表示删除。

**易错 5：SQL 优化建议空洞**

只写"加索引""优化查询"——没具体到列和改法。必须说**加在哪列、为什么、原 SQL 怎么改**。

**易错 6：ER 图基数判错**

把 1:N 判成 M:N，或把 M:N 判成 1:N。判别抓手：**A 的一个对应 B 的几个，B 的一个对应 A 的几个**——两边都多个是 M:N，一边多个是 1:N。

**易错 7：DFD 图元画错**

把外部实体画成圆（错！应是方框）、把数据存储画成方框（错！应是双横线）。**图元规范必须背**：方框=外部实体、圆=加工、双横线=数据存储、箭头=数据流。

**易错 8：BCNF 分解后忘说"无损连接"**

规范化分解后必须说明"分解无损连接、保持函数依赖"，否则阅卷老师无法确认分解质量。

## 速查/自测

### 答题骨架速查卡

```
数据库线三步公式：
  ① ER 建模（实体/属性/联系 1:1 / 1:N / M:N）
  ② 规范化（1NF 原子 / 2NF 无部分 / 3NF 无传递 / BCNF 决定因素含候选键）
  ③ SQL 优化（索引/避免 SELECT */JOIN 替代子查询/避免函数包裹索引列/分页）

Web 线四步公式：
  ① 识别需求
  ② 选 B/S 三层架构（表现/业务/数据访问）
  ③ MVC / MVP / MVVM 选型
  ④ REST 接口设计（资源导向/HTTP 方法语义化/无状态/统一接口）

ER 三要素：实体=矩形、属性=椭圆、联系=菱形
联系基数：1:1 / 1:N / M:N（M:N 需新建关系表）

DFD 图元：外部实体=方框、加工=圆、数据存储=双横线、数据流=箭头
DFD 分层：0 层（系统+外部）/ 1 层（分解）/ 2 层（再分解）
```

### MVC / MVP / MVVM 区别速查

| 维度 | MVC | MVP | MVVM |
|---|---|---|---|
| Model 通知 View | 可（观察者） | 否（隔离） | 否（隔离） |
| 控制器 | Controller 中转 | Presenter 包揽 View 逻辑 | ViewModel 双向绑定 |
| View 和 Model | 可直接通信 | 完全隔离 | 完全隔离 |
| 典型场景 | Web 后端 | Android/桌面 | Vue/WPF/前端 |
| 典型框架 | Spring MVC | Android原生 | Vue.js |

口诀：**"MVC 通、MVP 隔、MVVM 绑"**。

### 范式判别流程速查

```
属性原子？否 → 不满足 1NF
  ↓ 是
非主属性完全依赖候选键？否 → 满足 1NF
  ↓ 是
非主属性不传递依赖候选键？否 → 满足 2NF
  ↓ 是
每个决定因素含候选键？否 → 满足 3NF
  ↓ 是
BCNF
```

口诀：**"原子、无部分、无传递、决定含候选"**。

### REST HTTP 方法速查

| 操作 | 方法 | URL | 说明 |
|---|---|---|---|
| 查列表 | GET | /resources | 返回资源列表 |
| 查单个 | GET | /resources/{id} | 返回单个资源 |
| 新增 | POST | /resources | body 含新资源 |
| 全量更新 | PUT | /resources/{id} | body 含完整资源 |
| 部分更新 | PATCH | /resources/{id} | body 含部分字段 |
| 删除 | DELETE | /resources/{id} | 删除资源 |

### 自测题

**自测 1**：关系模式 R（学号，姓名，系号，系名），函数依赖：学号→姓名、系号；系号→系名。R 满足第几范式？如何规范化到 BCNF？

> 参考答案：候选键为学号。学号→系号→系名 是传递依赖（非主属性系名传递依赖候选键学号），但**无部分依赖**（所有非主属性都完全依赖学号）。所以 R 满足 **2NF**，不满足 3NF。规范化分解：R1（学号，姓名，系号），R2（系号，系名），均满足 BCNF 且无损连接、保持函数依赖。

**自测 2**：MVC、MVP、MVVM 三者的核心区别是什么？Web 后端和前端富交互各应选哪种？

> 参考答案：核心区别在"View 和 Model 是否直接通信"和"控制器包揽程度"：①**MVC**：Model 可通知 View（观察者），Controller 只中转用户输入；②**MVP**：View 和 Model 完全隔离，所有交互经 Presenter，Presenter 持有 View 接口；③**MVVM**：ViewModel 和 View 双向绑定，自动同步。Web 后端选 **MVC**（请求-响应模型，Spring MVC）；前端富交互选 **MVVM**（双向绑定减少 DOM 操作，Vue.js）。

**自测 3**：DFD 数据流图有哪四种图元？分别用什么符号表示？

> 参考答案：①**外部实体**——方框；②**加工**——圆或圆角矩形；③**数据存储**——双横线（开口矩形）；④**数据流**——箭头（需标注数据名）。DFD 分层：0 层（系统作为一个加工+外部实体）、1 层（分解为子加工）、2 层（再分解），父子图需平衡（输入输出一致）。

**自测 4**：原 SQL `WHERE YEAR(create_time)=2024` 为什么慢？怎么优化？

> 参考答案：YEAR(create_time)=2024 对索引列使用了函数包裹，导致 **create_time 上的索引失效**，数据库需全表扫描。优化：改为范围查询 `create_time >= '2024-01-01' AND create_time < '2025-01-01'`，这样能走索引，加速查询。

**自测 5**：某系统有"用户-订单-商品"三个实体，一名用户下多个订单，一个订单含多个商品，一个商品可被多个订单包含。请说明这三个联系的基数。

> 参考答案：①用户-订单：**1:N**（一名用户下多个订单，一个订单只属一名用户）；②订单-商品：**M:N**（一个订单含多个商品，一个商品可被多个订单包含），需新建"订单明细"关系表（订单号，商品号，数量，单价）。判别抓手：A 的一个对应 B 的几个、B 的一个对应 A 的几个，两边都多个是 M:N。

### 一句话小结

数据库与 Web 架构题的两条线公式（数据库：ER+规范化+SQL；Web：需求+三层+MVC+REST）是骨架，**范式判别流程**和**MVC 三模式区别**是命门，DFD 图元规范和 ER 基数判别是高频失分点。把范式判对、模式区别讲清、SQL 优化具体到列，这道题稳拿 12 分以上。
