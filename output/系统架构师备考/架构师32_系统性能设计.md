# 第32章：让系统扛住百万流量的性能工程套路

性能问题从来不是单点优化，而是一条从指标度量到工程落地的完整链路。这章讲的就是这条链：怎么测、怎么扛、怎么缓、怎么削。综合知识稳出 **2 题**，案例分析几乎必有一道性能优化题，论文写高并发系统也绕不开这些套路。

## 概念

### 这章考什么

- **综合知识**：每年约 **2 题**，考的是负载均衡层次、缓存策略选型、性能指标含义。
- **案例分析**：**高频性能题**。常给一个慢系统让你诊断瓶颈、给优化方案，负载均衡+缓存+数据库优化是标配答题组合。
- **论文**：写"高并发系统设计""性能优化实践"几乎必用本章套路。

### 与第26章质量战术的边界（必须分清）

这是本章最容易混的点。两章都讲性能，但定位不同：

- **第26章 质量属性战术**：讲**原则层**。性能战术分三类——资源需求（提升计算效率、减少开销）、资源管理（调度、缓存）、资源仲裁（先来后到、先重要后次要）。回答"用什么思路管性能"。
- **第32章 系统性能设计**：讲**工程实现层**。负载均衡、CDN、Redis 缓存、读写分离、消息队列削峰——都是 26 章战术的具体落地。回答"具体用什么组件、怎么配"。

一句话区分：**26 讲"为什么这么管"，32 讲"具体怎么做"**。案例答题时，先讲战术原则（体现架构思维），再讲工程实现（体现落地能力），两层都踩到才拿满分。

### 核心概念类比

把系统性能优化想成一家餐厅应对客流：

- **响应时间**：客人从进门到上菜的总时长。
- **吞吐量**：餐厅每小时能上多少桌菜。
- **并发数**：同时坐在餐厅里吃饭的人数。
- **资源利用率**：厨房、服务员忙到几成。
- **负载均衡**：把客人分流到多个分店或多个收银台。
- **缓存**：把热销菜提前做好放着，来了就上。
- **异步削峰**：高峰期发号牌，客人拿号先逛商场，叫号再回来吃。

记住这个类比，后面所有概念都能对号入座。

## 原理

### 一、性能指标：四个必背量

| 指标 | 定义 | 关键细节 |
|---|---|---|
| **响应时间（Response Time）** | 从发起请求到收到响应的总时长 | 必看分位数：**P90、P99** 比平均值更真实 |
| **吞吐量（Throughput）** | 单位时间内系统处理的请求数 | 常用 QPS（每秒查询数）、TPS（每秒事务数） |
| **并发数（Concurrency）** | 系统同时处理的请求数 | 与响应时间、吞吐量满足 Little 定律：**L = λ × W** |
| **资源利用率（Utilization）** | CPU、内存、网络、IO 等资源占用率 | 不是越高越好，留余量应对突发 |

**响应时间的分位数是必考点**。平均值会被极端慢请求拉高，看不出真实体验；**P90** 表示 90% 的请求快于这个值，**P99** 表示 99% 的请求快于这个值。线上系统常说"P99 < 200ms"，意思是最慢的 1% 请求也得在 200ms 内——这才是用户体感的真实刻画。Little 定律 **L = λ × W**（并发数 = 到达率 × 平均响应时间）串起三个量，案例题常考。

### 二、负载均衡：四层 vs 七层是核心

负载均衡（Load Balancing）把流量分摊到多台后端服务器。按工作层级分两大类：

| 层级 | 代表 | 工作原理 | 特点 |
|---|---|---|---|
| **DNS 轮询** | DNS 服务器 | 域名解析返回多个 IP，客户端轮换访问 | 最外层、调度粒度粗、有缓存延迟、不知后端死活 |
| **四层负载（LVS）** | LVS、F5 | 基于 IP+端口转发，不解析应用层 | 性能极高、协议无关、不入应用 |
| **七层负载（Nginx）** | Nginx、HAProxy | 基于 HTTP URL/Header/Cookie 转发 | 智能路由、可做灰度、性能略低于四层 |
| **反向代理** | Nginx | 客户端只看到代理，代理代为转发后端 | 兼具负载均衡、缓存、安全防护 |

**四层 vs 七层是高频考点**。一句话区分：**四层看 IP 端口转发快，七层看 URL Header 路由灵**。生产典型架构是 DNS → LVS（四层）→ Nginx（七层）→ 应用集群，层层分流。

负载均衡策略（决定流量怎么分）：

| 策略 | 规则 | 适用场景 |
|---|---|---|
| **轮询（Round Robin）** | 依次分配 | 后端配置一致 |
| **加权轮询** | 按权重分配 | 后端配置不均 |
| **最少连接（Least Connections）** | 分给当前连接数最少的 | 请求时长差异大 |
| **IP 哈希（IP Hash）** | 同一 IP 固定到同一后端 | 需要会话保持 |
| **一致性哈希** | 节点增减时只影响相邻数据 | 缓存场景，减少缓存失效 |

记忆抓手：**轮询均匀、最少连接灵、IP 哈希粘、一致性哈希稳**。

### 三、缓存策略：五层缓存与三大风险

缓存（Cache）是性能优化的第一杀器。从浏览器到数据库，五层缓存各管一段：

| 层次 | 位置 | 作用 | 失效粒度 |
|---|---|---|---|
| **浏览器缓存** | 客户端 | 减少重复请求 | 单用户 |
| **CDN 缓存** | 边缘节点 | 就近响应静态资源 | 区域用户 |
| **反向代理缓存** | Nginx | 缓存后端响应 | 全局 |
| **本地缓存** | 应用进程内（Caffeine/Guava） | 极快访问、不跨网络 | 单实例 |
| **分布式缓存** | Redis/Memcached | 跨实例共享、容量大 | 全局 |

**热点数据从近到远查**：浏览器 → CDN → 反向代理 → 本地缓存 → 分布式缓存 → 数据库。每一层命中就不用往下走，miss 才查下一层。

**缓存三大风险**（必背，案例高频）：

| 风险 | 现象 | 对策 |
|---|---|---|
| **缓存穿透（Penetration）** | 查不存在的数据，每次都打到数据库 | 缓存空值；**布隆过滤器**拦截 |
| **缓存击穿（Breakdown）** | 热点 key 过期瞬间，海量请求打穿到数据库 | 互斥锁重建；**热点 key 永不过期** |
| **缓存雪崩（Avalanche）** | 大量 key 同时失效，请求全打到数据库 | 过期时间加随机抖动；多级缓存；限流降级 |

三者的区分口诀：**穿透查不到、击穿热 key 挂、雪崩批量挂**。穿透是"数据压根不存在"，击穿是"一个热点 key 失效"，雪崩是"一堆 key 同时失效"——量级递增。

### 四、数据库优化：四把刀

数据库往往是性能瓶颈的根。优化手段从浅到深：

1. **索引优化**：建合适的索引、避免索引失效（函数操作、左模糊、隐式类型转换都会让索引失效）。
2. **SQL 优化**：避免 SELECT *、减少 JOIN、用 EXPLAIN 看执行计划、避免大事务。
3. **读写分离**：主库写、从库读，缓解主库压力。注意主从延迟带来的脏读问题。
4. **分库分表**：单表数据量大时拆分。**垂直拆分**按字段拆（专库专用），**水平拆分**按行拆（按用户 ID 取模）。引入分布式事务、跨库 JOIN 难题。

分库分表是"重武器"，引入复杂度大，能不分就不分，分了就要做好一致性代价。

### 五、异步处理：消息队列削峰填谷

同步调用链长，一个慢全链路慢。**消息队列（MQ）** 把同步变异步，核心价值是**削峰填谷**：

- **削峰**：秒杀瞬时百万请求，先全写入 MQ，后端按自己节奏消费。
- **填谷**：消费端按固定速率处理，把尖峰流量摊平成稳定流量。
- **解耦**：生产者不关心消费者死活，系统松耦合。
- **事件驱动**：订单完成发事件，库存、积分、通知各自订阅，互不阻塞。

代价：引入消息丢失、重复消费、顺序性、最终一致性等分布式问题，不是免费午餐。

## 实践

### 真题示例

**考法一：负载均衡层次**

> 例：以下关于负载均衡技术的描述，错误的是（　）。
> A. DNS 轮询实现简单，但无法感知后端服务器健康状态
> B. LVS 工作在传输层，性能高于 Nginx
> C. Nginx 可基于 URL 实现七层负载均衡
> D. 四层负载均衡必须解析 HTTP 协议才能转发

**答题思路**：D 错。四层负载均衡基于 IP+端口转发，不解析应用层协议，正是"不解 HTTP"才快。答案 **D**。

**考法二：缓存风险识别**

> 例：某系统在缓存服务器重启后，大量请求直接打到数据库导致数据库宕机。该现象属于（　）。
> A. 缓存穿透　B. 缓存击穿　C. 缓存雪崩　D. 缓存预热

**答题思路**：缓存重启后大量 key 同时失效，符合雪崩特征。穿透是查不存在的数据，击穿是单个热点 key 失效。答案 **C**。

**考法三：性能指标含义**（2024年·综合知识·性能相关，考生回忆）

> 例：评估一个在线系统的用户体感性能，以下哪个指标最能反映"绝大多数用户的真实体验"？
> A. 平均响应时间　B. P99 响应时间　C. 最大响应时间　D. 吞吐量

**答题思路**：平均值被极端值拉偏，最大值过于苛刻，吞吐量与体感无直接关系。P99 反映 99% 用户的体验上限。答案 **B**。

### 答题踩分词清单

- 性能诊断题踩分词：**响应时间 P90/P99、吞吐量 QPS、并发数、资源利用率瓶颈点**。
- 负载均衡踩分词：**四层 LVS 转发快、七层 Nginx 路由灵、轮询/最少连接/IP 哈希**。
- 缓存题踩分词：**穿透布隆过滤器、击穿互斥锁、雪崩过期抖动**。
- 数据库优化踩分词：**索引/SQL/读写分离/分库分表，垂直拆字段水平拆行**。
- 异步削峰踩分词：**消息队列削峰填谷、解耦、最终一致**。

### 案例论文应用

案例题常见套路："某电商系统大促时响应缓慢，请给出优化方案"。答题公式：

> "经分析瓶颈在数据库与缓存层。优化分四步：① 接入层引入 LVS+Nginx 四七层负载均衡，扩容应用集群；② 缓存层加 Redis 分布式缓存，对热点商品 key 设置不过期+互斥锁防击穿，过期时间加随机抖动防雪崩，布隆过滤器防穿透；③ 数据库层读写分离，主写从读，对订单表按用户 ID 水平分表；④ 削峰层引入 Kafka，秒杀请求先入队后端按速率消费，把瞬时峰值从 10 万 QPS 削到 1 万 QPS。"

这套"负载均衡+缓存+数据库+异步"四件套，是案例性能题的标准答案骨架，几乎万能。

论文写性能优化要避免堆术语，要写出**量化对比**：优化前 P99 800ms、优化后 120ms；优化前单库 QPS 2000、读写分离后 8000。有数字才有说服力。

### 自测题

**自测 1**：某系统 P99 响应时间 500ms，平均响应时间 80ms，为什么不能只看平均值做性能评估？

> 参考答案：平均值会被大量快速请求拉低，掩盖少数极慢请求的真实影响。P99 反映 99% 用户体验的上限，500ms 意味着最慢的 1% 请求（往往就是出问题的请求）超过 500ms。线上事故常由尾部慢请求引发，所以必须用 P90/P99 等分位数刻画长尾，平均值只看整体趋势不够。

**自测 2**：缓存穿透、击穿、雪崩三者区别是什么？分别用什么对策？

> 参考答案：穿透是查询根本不存在的数据，每次都穿过缓存打数据库，对策是缓存空值或用布隆过滤器拦截；击穿是单个热点 key 失效瞬间，海量并发请求同时打向数据库，对策是互斥锁重建缓存或热点 key 永不过期；雪崩是大量 key 同时失效，请求全打到数据库，对策是过期时间加随机抖动、多级缓存兜底、限流降级。三者危害量级递增：单次查询、单 key、批量 key。

**自测 3**：第 26 章质量战术也讲性能，第 32 章也讲性能，案例答题时如何区分使用？

> 参考答案：第 26 章是原则层，性能战术分资源需求（提升效率）、资源管理（调度缓存）、资源仲裁（优先级）三类，回答"用什么思路管性能"；第 32 章是工程实现层，讲负载均衡、CDN、Redis、读写分离、消息队列等具体组件，回答"具体用什么做"。案例答题应先讲战术原则体现架构思维，再讲工程实现体现落地能力，两层都踩才满分。比如先说"采用资源管理战术中的缓存调度"，再说"具体落地为 Redis 分布式缓存+本地 Caffeine 二级缓存"。

### 一句话小结

性能优化是一条链：**指标用 P99、负载均衡四七层、缓存五层防三险、数据库四把刀、MQ 削峰填谷——和第 26 章的边界是"原则对实现"，案例答题两层一起踩就稳**。
