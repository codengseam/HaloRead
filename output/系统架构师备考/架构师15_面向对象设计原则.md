# 第15章：用SOLID五原则守住扩展性底线

设计原则是设计模式的"宪法"。23种模式不过是 SOLID 五原则在具体场景的落地。综合知识考1-2题，案例分析里设计模式题的答题逻辑全靠它撑。记住一句话：**原则是心法，模式是招式**。

## 概念

### 这章考什么

- **科目**：综合知识1-2题 + 案例分析
- **考频**：综合稳定 **1-2题**；案例答设计模式必引原则
- **分值**：综合1-2分；案例按论述给分
- **难度**：★★★☆☆（原则好背，辨析题易混）

五大原则缩写 SOLID，加一条迪米特法则（LoD），共六条：

| 缩写 | 原则 | 中文 |
|---|---|---|
| S | SRP | 单一职责 |
| O | OCP | 开闭原则 |
| L | LSP | 里氏替换 |
| I | ISP | 接口隔离 |
| D | DIP | 依赖倒置 |
| - | LoD | 迪米特法则 |

类比：SRP 是"一个萝卜一个坑"，OCP 是"老房不拆加新楼"，LSP 是"儿子能替老子顶班"，ISP 是"别给客户推销不需要的功能"，DIP 是"高层别直接依赖底层小弟，都去依赖合同"，LoD 是"少跟陌生人说话"。

## 原理

### SRP 单一职责原则（Single Responsibility Principle）

**定义**：一个类应该只有一个引起它变化的原因。

违反：

```java
class Employee {
    void calculatePay() { }   // 财务变化
    void save() { }           // 存储变化
    void generateReport() { } // 报表变化
}
```

三个职责三个变化源，改一处影响别处。

遵守：

```java
class Employee { }                       // 只装业务数据
class EmployeePayService { void calc(){} }   // 财务
class EmployeeRepository { void save(){} }   // 存储
class EmployeeReporter { void gen(){} }      // 报表
```

**对应模式**：外观、代理可辅助拆分职责。

### OCP 开闭原则（Open-Closed Principle）

**定义**：对扩展开放，对修改关闭。新增功能加新代码，不改老代码。

违反：

```java
class Discount {
    double calc(String type, double price) {
        if (type.equals("VIP")) return price * 0.8;
        if (type.equals("SVIP")) return price * 0.7;
        // 每加一种会员就改这里
        return price;
    }
}
```

遵守（用多态）：

```java
interface Discount { double calc(double price); }
class VipDiscount implements Discount { public double calc(double p){return p*0.8;} }
class SvipDiscount implements Discount { public double calc(double p){return p*0.7;} }
// 新增会员加新类，老代码不动
```

**对应模式**：策略、工厂、装饰器、观察者都直接服务于 OCP。

### LSP 里氏替换原则（Liskov Substitution Principle）

**定义**：所有引用父类的地方必须能透明地使用子类对象，程序行为不变。

经典反例——正方形不该继承长方形：

```java
class Rectangle { void setWidth(int w){} void setHeight(int h){} }
class Square extends Rectangle {
    void setWidth(int w){ super.setWidth(w); super.setHeight(w); }  // 改宽连高一起改
}
// 调用者按 Rectangle 用，setWidth 后 Height 变了，行为不一致
```

子类不能让父类已成立的不变量失效。LSP 是继承的契约，违反就别用继承。

**对应模式**：策略、状态、模板方法依赖 LSP 保证可替换。

### ISP 接口隔离原则（Interface Segregation Principle）

**定义**：客户端不应被迫依赖它不使用的方法。接口要小而专。

违反：

```java
interface Worker {
    void work();
    void eat();  // 机器人 Worker 被迫实现 eat，抛异常或空实现
}
class Robot implements Worker {
    public void work(){ }
    public void eat(){ throw new UnsupportedOperationException(); }  // 噪音
}
```

遵守：

```java
interface Workable { void work(); }
interface Eatable { void eat(); }
class Robot implements Workable { public void work(){} }  // 只实现需要的
class Human implements Workable, Eatable { ... }
```

**对应模式**：适配器、外观可帮接口瘦身。

### DIP 依赖倒置原则（Dependency Inversion Principle）

**定义**：高层模块不依赖低层模块，二者都依赖抽象；抽象不依赖细节，细节依赖抽象。

违反：

```java
class UserController {
    private MySQLDatabase db = new MySQLDatabase();  // 高层直接依赖具体低层
}
```

遵守：

```java
interface Database { void save(); }
class MySQLDatabase implements Database { public void save(){} }
class UserController {
    private Database db;  // 依赖抽象
    public UserController(Database db){ this.db = db; }  // 注入
}
```

**对应模式**：工厂、抽象工厂、依赖注入、桥接都体现 DIP。

### LoD 迪米特法则（Law of Demeter）

**定义**：一个对象应该对其他对象保持最少的了解。只和"直接朋友"说话。

"直接朋友"：当前对象本身、方法参数、成员变量、方法内创建的对象。**不是朋友就别调用它的方法**。

违反：

```java
class Customer {
    void pay(Order o) {
        o.getPayment().getWallet().deduct(100);  // 链式穿越，摸到钱包
    }
}
```

遵守：

```java
class Order {
    void deductFromWallet(int amount) { payment.deduct(amount); }  // Order 自己处理内部
}
class Customer {
    void pay(Order o) { o.deductFromWallet(100); }  // 只跟直接朋友 Order 说话
}
```

**对应模式**：外观、中介者都是 LoD 的体现——集中沟通，减少网状依赖。

### 六原则与设计模式对应速查

| 原则 | 直接服务的模式 |
|---|---|
| SRP | 外观、代理 |
| OCP | 策略、工厂、装饰器、观察者、状态 |
| LSP | 策略、状态、模板方法 |
| ISP | 适配器、外观 |
| DIP | 工厂、抽象工厂、桥接、依赖注入 |
| LoD | 外观、中介者 |

## 实践

### 真题示例

**2024年·综合知识·设计原则相关(考生回忆)**：以下违背里氏替换原则的是？
A. 子类重写父类方法时增强功能
B. 子类抛出父类未声明的受检异常
C. 子类返回值是父类返回值的子类型
D. 子类方法访问权限比父类更宽

**答案**：B。LSP 要求子类不能比父类"更严格"——异常不能新增、访问权限不能更窄。B 新增异常破坏了调用者的异常处理预期。

### 答题公式

案例设计模式题论述时，固定套路：

> 该模式体现了【XX 原则】。【抽象角色】依赖【抽象】，新增【具体角色】无需修改原代码，符合【OCP/DIP】；客户端通过【多态/注入】使用，符合【LSP】，保证了系统的可扩展性与可维护性。

### 论文应用

"软件架构设计/质量属性"论文中，原则是论证架构合理性的标尺：

> 针对可维护性需求，系统遵循单一职责原则拆分服务边界，遵循开闭原则通过策略模式扩展业务规则，遵循依赖倒置通过接口隔离层间依赖，使各模块可独立演进。

## 速查/自测

### 30秒速查卡

| 原则 | 一句话 | 关键词 |
|---|---|---|
| SRP | 一个类一个变化原因 | 单一职责 |
| OCP | 扩展开放修改关闭 | 不改老代码 |
| LSP | 子类能替父类 | 不破坏契约 |
| ISP | 不依赖不用方法 | 接口小而专 |
| DIP | 依赖抽象不依赖具体 | 面向接口 |
| LoD | 最少知道 | 少跟陌生人说话 |

### 口诀

**单职开闭里氏换，接口隔离依赖倒，迪米特法则最少知，六原则守住扩展性。**

### 自测题

1. "对扩展开放，对修改关闭"是哪条原则？实现它最常用的面向对象机制是什么？

3. 高层模块直接 new 一个低层具体类，违反了哪条原则？

### 一句话小结

SOLID 五原则加迪米特是设计模式的宪法——SRP 管职责、OCP 管扩展、LSP 管继承、ISP 管接口、DIP 管依赖、LoD 管通信，案例答题必引其一。
