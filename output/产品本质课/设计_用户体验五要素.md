# 第 29 章　用户体验五要素

> 边界声明：本篇讲用户体验五要素框架，不讲情绪曲线思维（见思维_用户体验与情绪曲线 篇）。五要素在本篇定义并挂出处，后续设计章不重复挂。

## 一、入戏

产品评审会开了两个小时，会议室白板上贴满了改版方案。设计师在解释首屏卡片为什么用浅蓝而不是深蓝，开发在争论按钮圆角是 8 像素还是 12 像素，老板突然插话："这个功能到底给谁用？"

全场安静了三秒。PM 翻了翻演示稿，说："给 C 端用户用的。"

"哪类 C 端用户？解决他们什么问题？"

PM 卡住了。

这种场景在产品团队里太常见。讨论进入视觉细节的速度永远比进入战略层的速度快——视觉细节谁都能说两句，而战略层需要沉下来想清楚。但视觉层做得再漂亮，战略层错了，整个产品还是白搭。

要讲清楚"为什么战略层决定表现层"，需要一个被反复引用却被反复误归的框架——用户体验五要素。

## 二、破题

用户体验五要素由 Jesse James Garrett 于 2000 年提出，他先把模型发布在个人网站 jjg.net，2002 年成书《The Elements of User Experience》，2011 年第二版扩展到 Web 以外的产品与服务（副题改为 "for the Web and Beyond"）。这里要特别交代一句——五要素的提出者是 Garrett，不是 Don Norman。Norman 在更早的 1988 年写了《设计心理学》、在 2003 年写了《情感化设计》，但五要素这个分层框架是 Garrett 的贡献，常被误归给 Norman，本篇按核验版本写。

五层名称自底向顶依次是：战略层 / 范围层 / 结构层 / 框架层 / 表现层（Strategy / Scope / Structure / Skeleton / Surface）。

Garrett 在原书第二章 "Building from Bottom to Top" 明确指出：每一层依赖其下一层，从最抽象的战略到最具体的视觉表现，是一条自底向顶的决策依赖链。

本篇要给出的洞察是：五要素的真正价值不在分类，而在这条因果链——战略层决定范围层，范围层决定结构层，结构层决定框架层，框架层决定表现层。绝大部分产品失败不是表现层丑，是战略层和表现层脱节；越往下越稳定，越往上越易变，做产品得先把下盘扎稳。

## 三、方法论

### 1. 战略层：为谁、解决什么

战略层回答两个问题——我们做这个产品是要解决谁的什么问题？我们的商业目标是什么？

这一层是整条依赖链的根。根错了，上面四层全错。战略层模糊的产品长这样：所有人都在做事，但没人能一句话说清这个产品到底给谁用、解决什么具体问题。开会讨论方向时，每个人脑里的"用户"都不一样。

判断战略层是否清晰有一个简单的方法——让团队的五个人各自写一句话："我们为 [谁] 解决 [什么问题]，让他们 [得到什么价值]。"如果五句话里的"谁"和"什么问题"高度一致，战略层算过关；如果五个人写出五种用户、五个问题，战略层就是漏的。

### 2. 范围层：做哪些、不做什么

战略层清晰后，范围层回答——为了实现战略，我们要做哪些功能？更重要的是不做什么？

范围层的核心动作是取舍。每个团队资源有限，做十个平庸的功能，不如做三个做到极致的功能。范围层的产物是功能清单，但清单的灵魂是"砍掉了什么"——一份没有"不做什么"的功能清单，等于没有范围层。

Garrett 在原书中把范围层进一步细分为功能规格（针对功能型产品）和内容需求（针对信息型产品），但实操中可以把这两类合并理解为"产品要做哪些事"。

### 3. 结构层：信息架构与用户流程

范围层定了"做什么"，结构层回答"这些东西怎么组织"。

结构层有两块——信息架构（内容怎么分类、怎么导航）和交互设计（用户从 A 走到 B 的路径怎么走）。这两块决定了用户进入产品后能不能找到他要的东西、能不能顺畅地完成任务。

结构层是看不见的，但它的影响最深远。一个信息架构混乱的产品，无论表现层多精致，用户都会迷路。改版时如果动了结构层，影响范围远大于改表现层——这是为什么大改版往往需要重构信息架构。

### 4. 框架层：界面布局与导航

结构层定了"组织方式"，框架层回答"具体每个界面长什么样"——界面布局、导航设计、信息设计。

框架层比结构层更具体，但比表现层更抽象。框架层关注"按钮放在哪里、导航怎么排、表格怎么设计"，但不涉及颜色、字体、图片这些视觉细节。线框图就是框架层的典型产物。

### 5. 表现层：视觉设计

最上面一层是表现层——用户最终看到的视觉呈现，包括色彩、字体、图标、图片、动效。

表现层是所有人都能评头论足的层，所以讨论最热闹，但它其实是最容易改的层。改个颜色半天，改个信息架构可能要重构整个产品。

### 自底向顶的决策依赖链

五层不是并列的清单，是一条自底向顶的依赖链——战略层不清楚，范围层就拍脑袋；范围层不清楚，结构层就只能堆功能；结构层不清楚，框架层就只能靠设计师临场发挥；框架层不清楚，表现层做得再好看也只是包装。

Garrett 在原书里把这层关系讲得很直白：每一层依赖其下一层。理解了这条链，就知道为什么"先想清楚再动手"比"边做边想"省力——下层改一次，上层全得跟着改；上层改一次，下层可以不动。

## 四、避坑

### 坑一：跳层

最常见的坑是跳过战略层和范围层，直接进入框架层甚至表现层。老板拿着一张竞品截图说"我们也做个这样的"，团队就开始画线框图、做视觉稿。等做完上线，发现这个功能根本不是我们的用户要的——战略层从一开始就没人想。

跳层的代价是返工。下层没想清楚，上层做完等于白做。一次次跳层，一次次返工，团队疲了，老板也失去耐心。

### 坑二：用表现层反推战略层

另一种常见情况是——产品已经上线，团队看着表现层和数据反推"我们这个产品到底解决什么问题"。这种倒推经常得出牵强的结论，因为表现层的设计本身可能就是拍脑袋做的，反推出来的"战略"当然也站不住。

正解是先补战略层，哪怕补得晚。补完战略层，再看现有的范围层和结构层是否对得上——对得上的保留，对不上的标记待改。

### 坑三：把五要素当清单，不当因果链

五要素最大的价值是那条自底向顶的依赖链，而不是"五个层都要做"。如果一个团队只是机械地填五张表，但没有用依赖链去检查每一层是否真的支撑上一层，五要素就退化成了一张表格，失去了它的诊断力。

## 五、践行

1. **本周给正在做的产品做一次五要素体检**。拿出一张白纸，画五层，每层写一句话。如果某层写不出一句话，那一层就是漏的。
2. **找最薄弱的一层，下周补一项**。不要五层一起补，一次补一层。战略层最薄弱就补战略层，结构层最薄弱就补结构层。
3. **把战略层一句话贴到团队可见的地方**。会议室墙上、协作工具置顶、产品文档首页——让所有人每天看见这句话，避免开会时各想各的。
4. **下次评审功能时，按五层顺序提问**。先问"这个功能服务哪个用户、解决什么问题"（战略层），再问"为什么是这个功能不是别的"（范围层），再问"放在哪个位置"（结构层），再问"界面怎么布局"（框架层），最后问"长什么样"（表现层）。顺序倒了，讨论就乱了。
5. **警惕表现层讨论热度**。如果一场评审会八成时间在讨论颜色和按钮，说明团队在下层没想清楚，靠表现层讨论来回避战略层问题。这种会要主动打断，把讨论拉回战略层。
