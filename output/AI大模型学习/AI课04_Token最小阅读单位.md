# AI 课 04 · Token：AI 的最小阅读单位

> 前置知识：学完第 03 章即可
> 学完你能：①理解 Token 是什么、为什么不是"字"或"词" ②估算一段话大概消耗多少 Token ③理解 Token 对成本和上下文的影响

## 一、讲清楚：理解 Token，是理解一切的基础

上一章讲 AI 的能力边界，提到一个反直觉的现象：让 AI 数"strawberry 里有几个 r"，它经常答错。这一章就来回答这个问题，顺便讲清楚一个贯穿后续所有章节的基础概念——Token。

很多人以为 AI 跟人一样，读的是"字"或"词"。其实不是。AI 读的是一种叫 Token 的东西。理解 Token，你才能理解：为什么 AI 会数错字母、为什么中文比英文"贵"、为什么上下文有长度限制、为什么按 Token 计费。

这一章不写公式，只讲直觉。看完之后，你拿到任何一段文字，应该能大致估出它消耗多少 Token，以及为什么。

## 二、看案例：两个反直觉的现象

**案例一：strawberry 里有几个 r？**

正确答案是 3 个。但很多大模型会答 2 个。你追问它"再数一遍"，它可能改口，也可能继续错。

这件事让很多人困惑：一个能写代码、能做翻译的模型，怎么会数错字母？

答案跟它"看"这个词的方式有关。在 AI 眼里，"strawberry"不是一个一个字母拼出来的，它被切成了几个 Token，可能是 "straw" + "berry"，也可能是 "str" + "awberry"，具体取决于分词器。AI 看到的是这几个 Token，不是 10 个字母。让它数"r"，相当于让你看一幅抽象画数里面有几根红线——你看到的不是线，是色块。

**案例二：同一句中文，不同模型算出来的"长度"不一样。**

把"我今天去公园散步"这句话发给三个不同的大模型，问它们各自消耗了多少 Token。你会发现答案不一样。同一个模型，把这句话翻译成英文 "I went to the park for a walk today" 再算，Token 数又变了。

为什么？因为 Token 不是"字"也不是"词"，它是每个模型自己定义的"阅读单位"。不同模型用不同的**分词器**（Tokenizer，把文本切成 Token 的工具），切法不一样，结果就不一样。

这两个案例指向同一个事实：要理解 AI 怎么"看"世界，必须先理解 Token。

## 三、拆原理：Token 到底是什么

### 1. Token 是子词切分的结果

Token 不是字，不是词，是**子词**（Subword，介于字符和整词之间的文本片段）。

为什么要搞出"子词"这种东西？因为如果按"字"切，词表太小，表达力不够——"人工智能"被切成"人""工""智""能"四个孤立字符，模型很难学到"人工智能"是一个完整概念。如果按"词"切，词表太大，遇到没见过的词就抓瞎——"ChatGPT""元宇宙""区块链"这些新词，训练时没出现过，词表里没有，模型直接不认识。

子词是折中：把常见的整词保留，把罕见词拆成更小的片段。

举个例子。英文里 "playing" 这个词，可能被切成 "play" + "ing" 两个 Token。"play" 是常见词，"ing" 是常见后缀。这样切的好处是：模型见到 "jumping""running""walking" 时，能识别出 "ing" 这个共同模式，知道它们都是"进行时"。哪怕遇到训练时没见过的 "glomping"，也能猜出它跟 "glomp" 这个动作有关，且是进行时。

中文也类似。"人工智能"可能被切成 "人工" + "智能"，也可能切成 "人" + "工" + "智" + "能"，取决于分词器在训练时学到了什么。

这种切分方法有个正式名字：**子词切分**（Subword Tokenization）。

### 2. BPE：最常用的切分算法

目前主流大模型用的子词切分算法叫 BPE，全称 **Byte Pair Encoding**（字节对编码）。GPT 系列、Llama 系列等大多数主流模型，分词器都以 BPE 或其变体为基础。

BPE 的核心思想用一句话讲：**统计高频组合，合并成新单元**。

打个比方。假设你有一大堆英文文本，最初把每个字符当成一个单元：p、l、a、y、i、n、g……然后你统计哪些"相邻两个单元"出现得最频繁。发现 "p" 后面经常跟 "l"，于是把它们合并成 "pl"。再统计，发现 "pl" 后面经常跟 "ay"，合并成 "play"。继续下去，"play" 后面常跟 "ing"，但 "ing" 本身也是个高频组合，已经被合并过了，于是 "playing" 可能被切成 "play" + "ing"。

BPE 就这么一遍遍扫描、合并，直到达到预设的词表大小。词表大小是模型设计时就定好的，GPT-4 用的词表大约十万级别（据 OpenAI 的 tiktoken 工具，cl100k_base 词表约 10 万个 Token）。

合并出来的每个单元——无论是一个字符、一个子词、还是一个整词——都是一个 Token。

BPE 的妙处在于：常见词被切成一个 Token（省），罕见词被切成多个 Token（虽然不省，但至少能表达）。比如 "the" 是一个 Token，"supercalifragilistic"（一个生造的长词）会被切成好几个 Token。这就像汉语里"的"是高频字单独成词，而"饕餮"这种罕见词可能要靠偏旁部首拆开猜。

### 3. 中文和英文的 Token 差异

这是中文用户最该关心的一段。

英文里，一个常见单词通常对应 1 个 Token，比如 "hello" 是 1 个 Token。长一点或罕见一点的词可能 2-3 个 Token，比如 "understanding" 可能被切成 "under" + "stand" + "ing"。

中文不一样。一个汉字通常对应 1-2 个 Token，常见词组可能更省，但很多情况下，一个汉字就是 1 个 Token，甚至有的字会被切成 2 个 Token。原因在于，主流分词器主要是按英文语料训练的，对中文不够"熟"——它没见过足够多的中文，学不到"人工智能"应该作为一个整体，于是只能按字符或小片段切。

这意味着什么？同样表达一个意思，中文往往比英文消耗更多 Token。

举个例子。"我爱北京天安门"这句话，7 个汉字，用 OpenAI 的分词器算，大约消耗 7-10 个 Token。翻译成英文 "I love Beijing Tiananmen"，4 个单词，可能只消耗 4-5 个 Token。

这件事直接影响成本。如果你用按 Token 计费的 API，写中文 prompt 比写英文 prompt 贵——不是汇率问题，是 Token 数本身就多。这也是为什么很多开发者写系统提示词时偏好英文：同样的逻辑，Token 更省，成本更低，模型理解也更准（因为训练数据里英文占比大）。

### 4. Token 与上下文窗口

每个大模型都有一个**上下文窗口**（Context Window，模型一次能处理的 Token 总数上限），指的是它一次能"看"的 Token 总数。

比如某模型上下文窗口是 128K，意思是你给它输入的所有内容——系统提示词、对话历史、当前问题、附带的文件——加起来不能超过 12.8 万个 Token。超了，要么报错，要么模型自动截断前面的内容（通常是忘掉最早的对话）。

上下文窗口不是"内存"。它更像是模型的"短期记忆容量"。你塞进去的东西越多，模型能"看到"的越多，但同时也越容易"走神"——研究发现，当上下文很长时，模型对中间部分的注意力会下降，对开头和结尾的内容反而记得更清楚。这个现象在业界被称为"中间遗忘"（Lost in the Middle，Liu et al., 2023 的论文系统研究过这一现象）。

所以不要觉得"窗口大就可以随便塞"。塞满上下文，模型反而可能答得更差。一个朴素的策略：把最重要的信息放在开头和结尾，中间放次要的。

### 5. Token 与成本

主流大模型 API 按 Token 计费，分两部分：

- **输入 Token**（Input Tokens / Prompt Tokens）：你发给模型的内容。
- **输出 Token**（Output Tokens / Completion Tokens）：模型生成的内容。

输出 Token 通常比输入 Token 贵几倍。原因：输入只是"读"，输出是"写"，写的时候要逐个 Token 预测，计算量大得多。

这就引出一条成本优化原则：**能少说就少说，能少生成就少生成**。让模型写一万字的报告，比让它写一千字摘要贵十倍不止——不只是字数多，输出 Token 单价也高。

另一个常被忽视的成本：**重复发送**。如果你在一个长对话里反复来回，每次请求都会把之前的对话历史全部重发一遍（除非用了 prompt caching 之类的优化）。聊到第 20 轮，每次请求的输入 Token 可能已经膨胀到几万。这是很多人账单爆炸的原因。

举个具体的账。假设某模型输入 1 美元每百万 Token，输出 5 美元每百万 Token。你跟它聊了 30 轮，每轮平均 2000 输入 Token + 1000 输出 Token。到第 30 轮时，单次请求的输入 Token 已经累积到约 6 万（前面 29 轮的历史 + 本轮新内容）。仅这一轮，输入费用就是 0.06 美元，输出费用 0.005 美元。30 轮加起来，总输入费用不是 30 × 0.002 美元，而是从 0.002 一路涨到 0.06 的累加——总账远比你按"每轮固定"估算的高。

### 6. 怎么估算 Token

实操层面，有几个经验法则：

- 英文：1 个 Token ≈ 0.75 个单词，或者说 100 个单词 ≈ 130-150 个 Token。
- 中文：1 个汉字 ≈ 1-2 个 Token，保守按 1.5 估算。
- 代码：跟英文接近，但符号、空格、缩进都算 Token，一段 100 行的代码大约 1500-2500 Token。
- 标点和空格也算 Token，别忽略。

最准的办法是用工具实测。OpenAI 官方提供 tiktoken（Python 库）和在线 tokenizer 工具，把文本贴进去就能看到精确 Token 数和切分结果。其他模型厂商也有各自的工具。做成本估算时，别凭感觉，用工具量一次。

## 四、避坑指南：三个不要

**第一，不要以为 Token 等于单词数。** 英文里大致接近（常见词 1 个 Token ≈ 1 个单词），但中文差很远。一个汉字可能 1-2 个 Token，一个英文单词可能 1-3 个 Token。估算成本时，别拿字数或词数直接套，要用工具实测。把"这段话有 500 字，所以是 500 Token"这种算法丢掉——中文场景下，500 字往往是 700-1000 Token。

**第二，不要忽视中文的 Token 成本。** 同样一段业务逻辑，中文 prompt 可能比英文 prompt 多消耗 50%-100% 的 Token。如果你的应用是 to C、调用量大，这笔账必须算。优化方向：系统提示词用英文写，用户输入用中文，模型输出按需中英文切换。这不是崇洋媚外，是 Token 经济学。

**第三，不要把上下文窗口当内存用。** 上下文窗口是"短期记忆"，不是"数据库"。把整个知识库塞进上下文让模型"现场查"，技术上可行，成本上灾难，效果上还会因为中间遗忘而打折。正确做法是用 **RAG**（Retrieval-Augmented Generation，检索增强生成）——先检索出最相关的几段，再塞进上下文。RAG 后面会专门讲，这里先记住：上下文窗口要省着用，留给真正需要的信息。

## 五、悟本质：Token 是 AI 的"视神经"

把这一章串起来，可以提炼出一个心智模型：

**Token 是 AI 的"视神经"，决定了 AI 怎么"看"世界。**

人看文字，看到的是字符、是词、是句子。AI 看文字，看到的是一串 Token。这两种"看"法不一样，导致很多反直觉的现象：

- AI 数不清 "strawberry" 里有几个 r，因为它看到的不是 10 个字母，是几个 Token。
- AI 对中文更"贵"，因为中文一个字常常就是一个 Token，比英文单词"重"。
- AI 会"忘"前面说过什么，因为上下文窗口有上限，超了就丢。
- AI 越聊越贵，因为每轮对话都在重发历史 Token。

理解了 Token，你就理解了 AI 的"感官限制"。它不是无所不能的精灵，它是一个有特定"视觉系统"的概率机器。它的强项和弱项，很多都根源于"它怎么看见这个世界"。

这个心智模型会贯穿后续所有章节。讲提示词工程，本质是"用 AI 看得清的方式说话"；讲上下文工程，本质是"在 Token 限额内排兵布阵"；讲 RAG，本质是"用检索代替塞满上下文"。所有这些，都建立在"理解 Token"之上。

## 结语

Token 是 AI 的视神经。看清它怎么看世界，才能用好它。

## 自测三问

**问题一：** 为什么大模型经常数不清一个单词里有几个特定字母？请用 Token 的概念解释。

**参考答案：** 因为大模型看到的不是逐个字母，而是 Token。一个单词被切成几个子词 Token，模型看到的是这些子词，不是字母序列。让它数字母，相当于让人看几个色块数里面有几根线——它根本没有"逐字母"的视角。这也是 "strawberry 里有几个 r" 经常答错的原因。

**问题二：** 同样一段业务说明，写成中文和写成英文，哪种消耗的 Token 更多？为什么？这对成本有什么影响？

**参考答案：** 中文通常消耗更多 Token。因为一个汉字常常对应 1-2 个 Token，而一个英文常见单词通常对应 1 个 Token。同样意思的文本，中文 Token 数往往比英文多 50%-100%。这直接影响 API 成本——按 Token 计费的情况下，中文 prompt 比英文 prompt 贵。调用量大的应用，系统提示词常优先用英文写。

**问题三：** 某模型上下文窗口是 128K Token。你在对话里聊了 50 轮，每轮平均 2000 Token。到第 50 轮时，会发生什么？为什么账单会越来越贵？

**参考答案：** 50 轮 × 2000 Token = 10 万 Token，还没超 128K 上限，模型不会报错。但每发一次新请求，之前的对话历史都会作为输入 Token 重发一遍。到第 50 轮，单次请求的输入 Token 已经接近 10 万。如果输出也有 2000 Token，那么这一轮就要为 10 万输入 + 2000 输出付费。越往后聊，每轮的输入 Token 越多，账单指数级增长。这就是长对话成本爆炸的原因，也是 prompt caching 等优化技术存在的原因。
