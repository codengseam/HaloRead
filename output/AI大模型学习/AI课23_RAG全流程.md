# 第 23 章：RAG 全流程：从文档到问答

> 前置知识：学完第 16 章（RCR 框架）即可
> 学完你能：①理解 RAG 的完整流程 ②知道每一步用什么工具 ③能搭一个简单的 RAG 应用

## 一、讲清楚：让 AI"查资料再回答"

第 16 章讲了 RCR 框架——检索、压缩、重排，那是企业级 RAG 的上下文管理心法。但 RCR 只回答了"怎么从一堆片段里选出最相关的"，没回答一个更基础的问题：**这些片段从哪来？怎么来的？**

这一章就把这条链路从头到尾走一遍。从一份原始文档（PDF、Word、网页），到最终用户问一句话、AI 给出一个有出处的回答，中间到底发生了什么。

RAG（Retrieval-Augmented Generation，检索增强生成）这个名字是 2020 年 Facebook AI 在一篇论文里提出的（「Lewis et al., Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks, NeurIPS, 2020」）。论文要解决的问题很直接：**大模型知道很多，但不可能什么都知道**。训练数据有截止日期、企业内部文档模型没见过、最新资讯模型来不及学。让模型"凭记忆"回答这些问题，它就只会编。

RAG 的解法也直接：**别让模型凭记忆答，让它先查资料再答**。

本章解决三个问题：RAG 的完整流程长什么样、每一步用什么工具、怎么用 Coze 零代码搭一个能跑的 RAG 应用。学完之后，你能自己搭一个"喂文档→问问题→得答案"的小系统。

## 二、看案例：员工问"报销流程是什么"

来看一个最常见的场景——企业内部知识库问答。

某公司有一份《财务管理制度》，80 页 PDF，里面写了差旅报销、采购报销、会议报销的各种规则。员工小王出差回来，想报销一笔差旅费，但他不知道：

- 发票抬头怎么写？
- 超过多少金额需要部门经理审批？
- 出差补贴按什么标准算？

小王有三个选择：

**选择一：问同事**。同事也不一定记得清，而且每次都问很烦人。

**选择二：翻 PDF**。80 页文档，找到差旅报销那一节，再找到审批金额、补贴标准这些细节，可能要翻十几分钟。

**选择三：问 AI**。如果直接问 ChatGPT，它会一本正经地编："发票抬头一般写公司全称，超过 5000 元需要部门经理审批，出差补贴按城市等级分为一类 200 元/天、二类 150 元/天……"——听起来很专业，但全是它根据"一般公司大概什么样"猜的，和这家公司的实际制度八竿子打不着。

**选择四：问一个接了公司文档的 RAG 应用**。小王问："差旅报销的审批流程是什么？"应用先去公司文档里检索"差旅报销""审批流程"相关的内容，找到《财务管理制度》第 23 页那段，然后把这段连同问题一起喂给大模型，模型基于这段真实内容回答："根据《财务管理制度》第 4.2 节，差旅报销金额超过 3000 元需部门经理审批，超过 10000 元需财务总监审批……"

差别一目了然：选择三是闭卷考试，AI 凭记忆瞎编；选择四是开卷考试，AI 翻资料作答。RAG 就是把 AI 从闭卷考试变成开卷考试。

## 三、上手步骤：RAG 的六步流程

RAG 的完整流程可以拆成六步：**数据准备 → 向量化 → 存储 → 检索 → 生成 → 应用层**。前三步是"建库"，一次性做完；后三步是"问答"，每次提问都跑一遍。

### 第 1 步：数据准备——文档切分（Chunking）

原始文档格式五花八门：PDF、Word、Excel、网页、Markdown、PPT。第一步要把这些统一处理成"纯文本"，然后切成小块。

为什么要切？因为检索的基本单位是"片段"，不是整篇文档。一篇 80 页的 PDF 如果整篇当一个片段，检索时要么召回整篇（信息过载），要么召回不到（粒度太粗）。切成小块，每块讲一个完整的小主题，检索时才能精准召回"刚好相关"的那段。

切多大合适？业界经验是**每块 500-1000 字**（中文按字数，英文按 token 数）。太小，每块信息不完整，模型看到一段没上下文的话；太大，一块里混了多个主题，检索精度下降。

切分时还要注意"重叠"（Overlap）——相邻两块之间留 50-100 字的重叠。为什么要重叠？因为切分点是任意的，一句话可能被切成两半，前一块的尾巴和后一块的开头都看不懂。留点重叠，能保证被切开的句子至少在一块里是完整的。

工具上，Python 生态用 LangChain 的 `RecursiveCharacterTextSplitter` 或 LlamaIndex 的 `SentenceSplitter` 最常见。Coze 这种低代码平台则内置了切分功能，上传文档时自动切，你只需要选"切分粒度"（粗/中/细）。

### 第 2 步：向量化（Embedding）——把文本变成向量

切好的文本块是字符串，计算机没法直接算"相似度"。要把每块文本转成一个向量（一串数字，通常是 768 维或 1536 维）。

向量化用的是 Embedding 模型（嵌入模型，把文本映射成向量的模型）。第 16 章讲过，语义相近的文本，向量距离也近——这是向量检索能工作的基础。

主流的 Embedding 模型：

- **OpenAI text-embedding-3-small / text-embedding-3-large**：闭源商业 API，效果好、支持多语言，按 token 计费。
- **智谱 embedding-3**：国产商业 API，中文效果好，价格便宜。
- **Jina embeddings v3**：开源模型，支持长文本（最长 8192 token），多语言。
- **BGE 系列（BAAI 出品）**：开源，中文效果拔尖，可本地部署，免费。

选哪个？看预算和场景。预算充足、要省心，选 OpenAI 或智谱的 API；要本地部署、不想花钱，选 BGE。中文场景下 BGE-large-zh 和智谱 embedding-3 是性价比最高的两个选择。

### 第 3 步：存储——存进向量数据库

向量化和原始文本块要一起存起来，方便检索时既能算相似度，又能把原文取出来。存的地方叫**向量数据库**（Vector Database）。

主流的向量数据库：

- **Chroma**：轻量级，本地一行命令就能跑，适合原型开发和小规模数据（10 万片段以内）。Python 一行 `pip install chromadb` 就能用。
- **Milvus**：开源、分布式、能扛亿级向量，适合生产环境大规模部署。国内团队 Zilliz 出品，中文文档全。
- **Pinecone**：商业 SaaS，全托管，不用自己运维，按用量付费。海外用得多。
- **Qdrant、Weaviate、pgvector**：其他常见选项，pgvector 是 PostgreSQL 的扩展，适合"已经在用 Postgres、不想再装一个数据库"的团队。

新手怎么选？本地试水用 Chroma，上生产用 Milvus 或 Pinecone。Coze 这种平台则完全不用你管存储——上传文档后，平台自己存。

### 第 4 步：检索——用户问题向量化，召回 Top-K

前三步是"建库"，做完一次就能反复用。从这一步开始是"问答"，每次用户提问都跑一遍。

用户问"差旅报销的审批流程是什么"，系统做三件事：

1. 把这个问题用同一个 Embedding 模型转成向量。
2. 拿这个向量去向量数据库里做相似度搜索（通常是余弦相似度或点积），找出距离最近的几段。
3. 返回 Top-K 段（K 通常取 3-5），这就是召回结果。

注意一个细节：**检索时用的 Embedding 模型必须和建库时用的一样**。不同模型的向量空间不同，混用等于乱来——用 OpenAI 的模型建库，用智谱的模型检索，召回结果完全没意义。

第 16 章讲的 RCR 框架，就是在这一步之后加压缩和重排，把召回的 Top-K 进一步筛成"最相关的几段"。简单 RAG 可以不做压缩重排，直接把召回结果喂给模型；企业级 RAG 一定要做，效果差距很大。

### 第 5 步：生成——把片段和问题拼成 prompt

拿到召回的片段后，把它们和用户问题拼成一个 prompt，喂给 LLM，让 LLM 基于这些片段回答。

一个典型的 prompt 长这样：

```
你是一个企业知识库助手。请根据以下参考资料回答用户问题。
如果参考资料里没有相关内容，请直接说"知识库中未找到相关内容"，不要编造。

参考资料：
【片段1】（来源：《财务管理制度》第 4.2 节）
差旅报销金额超过 3000 元需部门经理审批，超过 10000 元需财务总监审批……

【片段2】（来源：《财务管理制度》第 4.3 节）
差旅补贴按城市等级分类，一类城市 200 元/天，二类城市 150 元/天……

用户问题：差旅报销的审批流程是什么？
```

这个 prompt 有三个关键点：

- **明确角色和任务**：告诉模型"你是知识库助手，根据资料回答"。
- **明确约束**：告诉模型"资料里没有就说没有，不要编"。这是防幻觉的关键一行——第 09 章讲过，模型最大的毛病就是爱编。这一行约束能把幻觉率压下去一大截。
- **明确来源**：每段资料标注来源（哪份文档、哪一节）。这样模型回答时可以引用来源，用户能去原文核对。

LLM 拿到这个 prompt，基于片段内容生成回答："根据《财务管理制度》第 4.2 节，差旅报销审批流程为：金额 3000 元以下由直属主管审批；3000-10000 元需部门经理审批；10000 元以上需财务总监审批。"

### 第 6 步：实操——在 Coze 里搭一个 RAG 应用

讲了五步原理，现在动手搭一个。最简单的路径是用 Coze（字节出品的 Bot 搭建平台），零代码，半小时能跑通。

操作步骤：

1. **登录 Coze**（国内版 coze.cn，海外版 coze.com），创建一个新 Bot。
2. **进入 Bot 编辑页**，左侧找到"知识库"模块，点击"创建知识库"。
3. **上传文档**：支持 PDF、Word、TXT、Excel、网页 URL 等。把《财务管理制度》PDF 传上去。
4. **配置切分**：Coze 会自动切分，你可以选"自动"或"自定义"。自定义时建议块大小 500 字、重叠 50 字。
5. **配置分段规则**：可以选择"按段落切分"或"按字符数切分"。结构化文档（如带章节的 PDF）建议按段落，纯文本建议按字符数。
6. **等待向量化完成**：Coze 自动用内置 Embedding 模型向量化并入库，几十页文档通常一两分钟搞定。
7. **回到 Bot 编辑页**，在 prompt 里写明："你是公司财务制度助手，请根据知识库内容回答员工问题，不要编造。"
8. **测试**：在右侧预览框输入"差旅报销审批流程是什么"，看 Bot 是否能基于文档回答。
9. **发布**：测试通过后，可以发布到飞书、微信、网页等渠道。

整个过程不用写一行代码。Coze 帮你做了切分、向量化、存储、检索、prompt 拼接这一整套，你只需要提供文档和 prompt。

## 四、验收标准

怎么判断你的 RAG 应用搭对了？三条硬标准：

**标准一：能基于文档回答，不瞎编**

拿一个文档里明确写过的问题去问，看回答是否和文档一致。再拿一个文档里没写过的问题去问，看 Bot 是否说"知识库中未找到相关内容"。如果文档里没写它还硬答，说明 prompt 里的"不要编造"约束没起作用，或者模型太强势盖过了资料。

**标准二：能给出答案的来源**

回答里应该能追溯到"这个答案来自哪份文档、哪一节"。Coze 在回答时会显示引用的片段，点开能看到原文。如果答案没有来源支撑，等于又回到了闭卷考试。

**标准三：换一种问法还能答对**

"差旅报销审批流程是什么"和"出差回来报销要找谁签字"是同一个问题，只是问法不同。好的 RAG 应用应该都能答对——这考验检索的语义理解能力。如果换种问法就答不上来，多半是 Embedding 模型太弱，或者切分粒度不合理。

## 五、悟本质：RAG 是给 AI 一个开卷考试的机会

讲了这么多步骤，RAG 的本质可以用一句话概括：**把 AI 从闭卷考试变成开卷考试**。

闭卷考试时，AI 只能凭训练时记住的内容回答。训练数据没见过的东西，它要么不会，要么瞎编。开卷考试时，AI 先翻资料，再基于资料回答——能力上限从"记住了多少"变成"会查多少、会用多少"。

这个本质有几个推论：

**推论一：RAG 的天花板是文档质量，不是模型能力。**

模型再强，喂给它的是过时的、错误的、混乱的文档，回答也好不到哪去。所以搭 RAG 应用，功夫要花在文档上——清洗、去重、结构化、定期更新。很多人花大力气调模型、调 prompt，却懒得整理文档，方向错了。

**推论二：RAG 的瓶颈通常在检索，不在生成。**

模型生成能力已经很强了，只要给它对的片段，它就能答好。问题往往出在"对的片段没被召回"——切分粒度不对、Embedding 模型差、查询没改写、没做重排。所以调 RAG，先调检索，再调生成。第 16 章讲的 RCR 就是专门治检索瓶颈的。

**推论三：RAG 不是"让 AI 变聪明"，而是"让 AI 知道自己不知道"。**

没有 RAG 时，AI 遇到不会的问题也硬答，因为它没有"我不知道"这个概念。有了 RAG，AI 先查资料，查不到就老实说"知识库中未找到"。RAG 给 AI 装的不是一个"知识库"，而是一个"知道自己不知道"的机制。这个机制比知识本身更重要。

理解了"开卷考试"这个本质，你再看市面上的 RAG 产品——Coze 知识库、Dify 知识库、阿里云百炼、智谱 GLM 知识库——会发现它们做的都是同一件事：把企业文档变成 AI 的"开卷资料"。区别只在切分策略、检索精度、引用展示这些工程细节。

## 六、结语

RAG 不是什么高深技术，它就是让 AI 先查资料再回答——把闭卷考试改成开卷考试，能力上限就从"记住了多少"变成"会查会用多少"。
