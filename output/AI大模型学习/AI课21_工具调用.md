# 第 21 章：工具调用：给 AI 装上手和脚

> 前置知识：学完第 20 章（ReAct 范式）即可
> 学完你能：①理解工具调用的基本原理 ②能定义一个简单的工具 ③在 Coze/Claude 里配置工具

## 一、讲清楚：AI 光有脑子不够，得有"手"

上一章讲 ReAct 时，我们把 Action 当成一个黑盒——"调一个工具，拿一个结果"。这一章把这个黑盒打开。

工具调用（Tool Calling），也叫函数调用（Function Calling，模型按约定格式生成对一个外部函数的调用请求），是给大模型"装手"的技术。模型本身只会"说"——生成文本。它不会查数据库、不会发邮件、不会算复杂数学、不会调外部 API。这些"动手"的事，得靠工具。

打个比方：大模型像一个博学的顾问，脑子快、知识广，但被困在玻璃房里——他能告诉你"应该怎么查天气"，但自己伸不出手去查。工具调用就是在玻璃房上开个口子，让顾问能把纸条递出去，外面的人按纸条上的指令去查，再把结果递回来。

为什么这件事重要？因为只有装上"手"，AI 才能从"聊天工具"变成"干活工具"。没有工具调用的 AI，最多是个高级问答机器人；有了工具调用，它才能查实时数据、操作数据库、控制外部系统——这才是 Agent 能"干活"的根基。

本章解决三个问题：工具调用到底怎么工作、怎么定义一个工具、在 Coze 和 Claude 里怎么实际配置。学完之后，你能自己给 AI 装一个"手"。

## 二、看案例：同一个问题，没工具 vs 有工具

### 案例 1：查天气

问 ChatGPT（不联网）："上海明天会下雨吗？"

它可能回答："上海明天可能有阵雨，建议带伞。"——听起来合理，但完全是猜的。它不知道"明天"是哪天，也没有任何实时天气数据。这是第 09 章讲过的幻觉（Hallucination，模型生成看似合理但实际错误的内容）。

给它接一个天气查询工具，再问同样的问题。模型会先判断"这需要实时数据"，调用天气工具，传入参数 `{"city": "上海", "date": "tomorrow"}`，工具返回 `{"weather": "小雨", "temperature": "18-23℃"}`，模型基于真实数据回答："上海明天小雨，气温 18-23℃，建议带伞。"——这次是真的查过的。

### 案例 2：算数学

问："1234 乘以 5678 等于多少？"

模型可能直接给答案，但大数字乘法它经常算错——因为它是语言模型，不是计算器，精确数值运算是它的弱项。给它接一个计算器工具，模型会调用 `calculator(expression="1234*5678")`，工具返回 `7006652`，模型照着答。准确无误。

### 案例 3：查数据库

问："我们公司上个月销量最高的三个产品是什么？"

模型连你们公司的数据库都看不到，更别说销量数据。瞎编的话会编出一堆不存在的商品名。给它接一个 SQL 查询工具，模型生成 SQL `SELECT product_name, sales FROM orders WHERE month='last' ORDER BY sales DESC LIMIT 3`，工具执行 SQL 返回真实数据，模型基于真实数据回答。

三个案例的共同点：**模型本身做不到的事，靠工具做到了**。模型负责"判断要不要调工具、调哪个、传什么参数"，工具负责"实际执行"。这就是工具调用的分工。

## 三、上手步骤

### 1. 工具定义：用 JSON Schema 描述工具

工具调用要工作，第一步是让模型"知道有哪些工具可用、每个工具干什么、需要什么参数"。这件事靠工具定义（Tool Definition）完成。

工具定义通常用 JSON Schema 格式描述，包含三个核心字段：name（工具名）、description（用途说明）、parameters（参数结构）。

一个天气查询工具的定义长这样：

```json
{
  "name": "get_weather",
  "description": "查询指定城市指定日期的天气。支持中国主要城市。",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "城市名，如'北京'、'上海'"
      },
      "date": {
        "type": "string",
        "description": "日期，格式 YYYY-MM-DD，默认今天"
      }
    },
    "required": ["city"]
  }
}
```

几个要点：

- **name 要简洁明确**：模型靠名字判断"该不该调这个工具"。`get_weather` 一眼能看出是查天气的。
- **description 要写清"干什么"和"什么时候用"**：模型主要靠这段描述决定是否调用。写得越清楚，模型选错工具的概率越低。比如"支持中国主要城市"这条，能避免模型拿它去查国外城市。
- **parameters 用 JSON Schema**：每个参数说明类型和含义。`required` 标注哪些参数必填。
- **description 是给模型看的，不是给人看的**：要写得让模型容易理解。比如"日期，格式 YYYY-MM-DD"比"日期"好——模型知道该传什么格式。

工具定义传给模型时，会作为请求的一部分（OpenAI 叫 `tools` 字段，Claude 也叫 `tools` 字段，格式略有差异但思路一致）。模型看到这些定义后，就能在回答时决定"要不要调工具"。

### 2. 工具调用流程：五步循环

工具调用的完整流程分五步，和上一章讲的 ReAct 循环高度重合：

**第一步：模型决定调哪个工具。** 模型收到用户问题 + 工具定义列表，判断"要不要调工具、调哪个"。如果不需要（比如问"什么是光合作用"），直接回答；如果需要，进入下一步。

**第二步：模型生成调用参数。** 模型输出一个结构化的调用请求，包含工具名和参数值。比如 `{"name": "get_weather", "arguments": {"city": "上海", "date": "2026-06-24"}}`。注意：模型只生成调用请求，不执行——执行是外部的事。

**第三步：外部执行工具。** 你的代码（或平台）拿到模型生成的调用请求，实际去执行——调真实 API、查真实数据库、跑真实计算。这一步模型完全不参与。

**第四步：把结果返回给模型。** 工具执行完，把结果（通常是 JSON）作为一条新消息塞回对话上下文。比如 `{"role": "tool", "tool_call_id": "xxx", "content": "{\"weather\": \"小雨\", \"temperature\": \"18-23℃\"}"}`。

**第五步：模型基于结果继续。** 模型看到工具返回的结果，组织自然语言回答用户。如果一次不够，模型可以再调下一个工具——这就回到第一步，形成循环。

用伪代码表示：

```python
def chat_with_tools(user_message, tools):
    messages = [{"role": "user", "content": user_message}]
    while True:
        response = llm.generate(messages, tools=tools)
        if response.tool_calls:
            # 模型要调工具
            for call in response.tool_calls:
                result = execute_tool(call.name, call.arguments)
                messages.append({
                    "role": "tool",
                    "tool_call_id": call.id,
                    "content": result
                })
        else:
            # 模型直接回答了
            return response.content
```

这个循环就是工具调用的骨架。OpenAI、Claude、Coze 底层都是这个流程，只是接口封装不同。

### 3. 实操 A：在 Coze 里添加一个"天气查询"插件

Coze（字节跳动的 Agent 搭建平台）把工具叫"插件"（Plugin）。下面是在 Coze 国内版里给一个 Bot 配天气查询插件的步骤。

**环境准备**：注册并登录 Coze（coze.cn），创建一个 Bot。

**操作步骤**：

1. 进入 Bot 编辑页，左侧找到"插件"区域，点"添加插件"。
2. 在插件市场搜索"天气"，会看到若干现成插件（如"墨迹天气"、"和风天气"）。选一个，点"添加"。
3. 添加后，插件会出现在 Bot 的工具列表里。点开插件，能看到它包含的工具（如 `get_weather`、`get_forecast`）和参数定义。
4. 在 Bot 的"人设与回复逻辑"里写一句引导："当用户问天气时，调用天气插件查询。" 这一步不是必须，但能提高模型调用工具的命中率。
5. 在右侧预览区测试：发"上海明天天气怎么样"，看 Bot 是否调用了天气插件、返回是否正确。

**预期结果**：Bot 回答里会显示"已调用天气插件"，并给出基于真实数据的天气信息。

**常见报错**：

- **Bot 不调工具，直接瞎编**：通常是插件描述不够清楚，或者人设里没引导。改插件描述，加"涉及天气问题时必须调用"。
- **调了工具但参数错**：比如城市名传了"上海市"而不是"上海"。检查插件的参数 description，写清"不要带'市'字"。
- **插件返回报错**：可能是 API Key 过期、配额用完。在插件管理页查看错误日志。

### 4. 实操 B：在 Claude 里用 tool_use 功能

Claude（Anthropic 的模型）原生支持工具调用，接口字段叫 `tool_use`。下面是用 Claude API 调用一个简单工具的代码示例。

**环境准备**：安装 Python SDK `pip install anthropic`，设置环境变量 `ANTHROPIC_API_KEY`。

**代码示例**：

```python
import anthropic
import json

client = anthropic.Anthropic()

# 1. 定义工具
tools = [{
    "name": "get_weather",
    "description": "查询指定城市的当前天气",
    "input_schema": {
        "type": "object",
        "properties": {
            "city": {"type": "string", "description": "城市名"}
        },
        "required": ["city"]
    }
}]

# 2. 第一次请求：让模型决定调不调工具
response = client.messages.create(
    model="claude-3-5-sonnet-20241022",
    max_tokens=1024,
    tools=tools,
    messages=[{"role": "user", "content": "上海今天天气怎么样？"}]
)

# 3. 检查模型是否要调工具
if response.stop_reason == "tool_use":
    # 找到工具调用请求
    tool_use_block = next(b for b in response.content if b.type == "tool_use")
    print(f"模型要调: {tool_use_block.name}")
    print(f"参数: {tool_use_block.input}")

    # 4. 实际执行工具（这里用假数据模拟）
    tool_result = {"weather": "多云", "temperature": "22℃"}

    # 5. 把结果返回给模型，让它继续
    follow_up = client.messages.create(
        model="claude-3-5-sonnet-20241022",
        max_tokens=1024,
        tools=tools,
        messages=[
            {"role": "user", "content": "上海今天天气怎么样？"},
            {"role": "assistant", "content": response.content},
            {"role": "user", "content": [
                {
                    "type": "tool_result",
                    "tool_use_id": tool_use_block.id,
                    "content": json.dumps(tool_result)
                }
            ]}
        ]
    )
    print(f"最终回答: {follow_up.content[0].text}")
```

**预期结果**：模型先输出"要调 get_weather，参数 city=上海"，拿到模拟数据后回答"上海今天多云，气温 22℃"。

**常见报错**：

- **`stop_reason` 不是 `tool_use`**：模型判断不需要调工具，直接回答了。检查工具描述是否清晰、问题是否真的需要工具。
- **`tool_use_id` 不匹配**：返回结果时 `tool_use_id` 必须和模型生成的 id 一致，否则会报错。
- **参数类型错**：模型传的参数类型和 schema 不符（比如该传 string 传了 number）。检查 schema 定义。

### 5. 关于工具调用的几个关键认知

这部分参考「菜鸟教程 · AI Agent，核心结构」中关于 Tool use 的部分，结合实操经验整理几个关键认知：

- **模型不执行工具，只生成调用请求**：模型输出的是"我想调 get_weather，参数是 city=上海"这段结构化文本，实际执行靠你的代码。模型没有"手"，它只是告诉你"该伸手了"。
- **工具描述是模型选工具的唯一依据**：模型看不到工具的实现代码，只看 name 和 description。描述写得烂，模型就选错工具或不用工具。
- **一次可以定义多个工具**：你可以同时给模型 10 个工具，模型会根据问题选合适的。但工具太多会降低准确率——模型容易选错。一般 5-10 个以内比较稳。
- **工具调用可以串行多次**：复杂任务里模型可能调 A、再调 B、再调 C，每一步基于上一步结果。这就是 ReAct 循环的体现。
- **不同模型的工具调用能力差异大**：GPT-4、Claude 3.5 Sonnet、Gemini 这些强模型工具调用准确率高；小模型经常选错工具或参数错。生产环境别用太弱的模型做工具调用。

## 四、验收标准

学完这一章，用下面三条验收自己：

1. **能说清工具调用的五步流程**：模型决定调工具 → 生成参数 → 外部执行 → 结果返回 → 模型继续。能用大白话给同事讲明白，不卡壳。
2. **能写出一个工具的 JSON Schema 定义**：给一个简单需求（比如"查询股票价格"），能写出包含 name、description、parameters 的完整定义，参数类型和 required 标注正确。
3. **能在 Coze 里成功配置一个工具并让 Agent 调用它**：建一个 Bot，加一个天气插件，发"北京今天天气"能触发工具调用，返回基于真实数据的回答。如果用 Claude API，能跑通上面那段代码，看到模型先调工具再回答。

第三条是硬指标。跑不通就回去查插件描述、参数格式、API Key 这几个常见坑点。

## 五、悟本质：语言能力与执行能力的对接

工具调用的本质，可以用一句话概括：**把大模型的语言能力和外界的执行能力对接起来**。

大模型擅长理解意图、生成文本、做规划——这些是"语言能力"。但它不擅长精确计算、不擅长实时数据、不擅长操作外部系统——这些是"执行能力"，得靠工具。

工具调用就是这两者的桥梁：模型用语言能力判断"该做什么"，工具用执行能力"实际去做"，结果再用语言能力"组织成回答"。

这个本质有几个推论：

**推论 1：工具调用让 AI 的能力边界从"模型知道什么"扩展到"工具能做什么"。**

模型本身不知道你们公司的销量数据，但接上数据库工具后它就能查；模型本身不会发邮件，但接上邮件工具后它就能发。AI 的能力上限不再是模型本身，而是"模型 + 工具集合"。这也是为什么 Agent 框架都在疯狂集成工具——每多一个工具，Agent 能干的事就多一件。

**推论 2：工具的质量决定 Agent 的质量。**

模型再聪明，工具返回错数据，结果就是错的。一个返回脏数据的工具，比没有工具还糟糕——因为模型会基于错数据生成"看起来合理但实际错误"的回答，比直接说"我不知道"更有迷惑性。所以搭 Agent 时，工具的可靠性比模型的选择更关键。

**推论 3：工具描述是"模型和工具之间的接口文档"。**

写工具描述和写 API 文档是一回事——读者不同而已。API 文档写给程序员看，工具描述写给模型看。两者都要求：清楚说明"干什么、需要什么输入、返回什么"。工具描述写得烂，模型就用得烂；工具描述写得好，模型就能用得准。

**推论 4：工具调用是 Agent 的"手"，但不是 Agent 的全部。**

工具调用解决"怎么调一个工具"的问题，但 Agent 还需要规划（什么时候调、调几个、什么顺序）、记忆（前面调过什么、结果是什么）、错误处理（工具报错怎么办）。这些是后面章节的内容。工具调用是基础能力，但搭一个完整 Agent 还需要更多组件配合。

## 六、结语

工具调用给大模型装上了"手"——模型负责想，工具负责做，结果再回到模型组织成回答。这一对接，让 AI 从"只会说"变成"能干活"。
