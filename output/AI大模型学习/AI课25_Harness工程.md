# 第 25 章：Harness Engineering：套在模型外面的壳

> 前置知识：学完第 15 章（上下文五大组成）即可
> 学完你能：①理解什么是 Harness 以及它为什么重要 ②看懂主流 AI 产品的 Harness 设计 ③理解"模型能力"和"产品能力"的差距

## 一、讲清楚：你用的不是模型，是一整套驾驭系统

第 15 章讲过上下文五大组成——指令、历史、知识、工具、记忆。如果你只是把这几样东西手动拼一拼喂给模型，能跑通一个 demo，但离一个真正能用的产品还差得远。

差在哪？差在"工程系统"。

举个最常见的场景：你在 ChatGPT 网页上和 GPT-4 聊天，问它"帮我写一段 Python 代码读取 Excel"。你看到的是——输入框、对话气泡、流式打字、代码高亮、可以一键复制。背后真正发生的事远不止"模型生成文本"这么简单：

- 你输入的"帮我写一段 Python 代码读取 Excel"会被加上一段 system prompt（定义 ChatGPT 的助手身份、安全规则、回答风格）。
- 之前的对话历史会被拼进去（如果你之前问过 pandas，它会记得你倾向用 pandas）。
- 模型生成时是流式的——边生成边往前端推，前端按字符渲染，遇到代码块要切换到代码高亮组件。
- 如果模型决定调用代码解释器（Code Interpreter，ChatGPT 内置的代码执行沙箱）执行代码，Harness 要把代码送进沙箱、跑出结果、把结果再喂回模型。
- 如果模型生成了不安全的内容，Harness 的安全过滤器会拦截或改写。
- 如果生成失败（网络断了、超时、触发限流），Harness 要负责重试或给用户友好提示。

这一整套"套在模型外面的工程系统"，就是 Harness。

Harness Engineering（驾驭工程，设计和实现套在模型外面的工程系统的实践），就是研究怎么把这层"壳"做好。它不是提示词工程，不是 Fine-tuning，而是介于两者之间的"产品工程"——决定一个 AI 应用最终好不好用、能不能落地。

本章解决三个问题：Harness 到底是什么、它由哪些核心组件组成、为什么同样的模型套不同 Harness 表现天差地别。学完之后，你看 ChatGPT、Claude、Cursor、Coze 这些产品，就不再只是看"它用的什么模型"，而是能看到"它的 Harness 怎么设计的"。

## 二、看案例：同一个 GPT-4，三种壳

为了让你直观感受 Harness 的差距，我们做一个对比实验：拿同一个 GPT-4 模型（同一时间点的同一版本），分别套上三种 Harness，让它完成同一个任务——"在我的项目里加一个用户登录功能"。

### 1. 套上 ChatGPT 的壳

你在 ChatGPT 网页里输入这句话。GPT-4 会怎么回答？

它会给你一段"通用建议"：建议用什么认证库（比如 bcrypt、jsonwebtoken）、给一段示例代码、提醒你注意密码哈希、HTTPS、CSRF。代码是孤立的——它不知道你的项目用什么框架、什么数据库、目录结构长什么样。

你拿这段代码去用，多半要改不少地方才能塞进你的项目。

### 2. 套上 Cursor 的壳

你在 Cursor 里打开你的项目，按 Cmd+I 唤起 Composer，输入同样的话。GPT-4（或 Cursor 当时支持的某个模型）会怎么表现？

它会先扫一遍你的项目——读 package.json 知道你用 Express + TypeScript，读现有路由文件知道你的代码风格，读数据库 schema 知道用户表长什么样。然后它生成的代码会：

- 直接放到合适的目录（比如 `src/routes/auth.ts`）。
- 用你项目里已有的工具函数（比如 `db.query`）。
- 遵循你已有的代码风格（命名、缩进、错误处理方式）。
- 自动修改相关文件（比如在 `app.ts` 里挂载新路由）。
- 跑一下 TypeScript 编译，如果有报错自己再改。

同样是 GPT-4，为什么表现差这么多？因为 Cursor 的 Harness 帮模型做了大量"翻译"工作——把"你的项目"这个现实问题，翻译成了模型能懂的上下文（项目结构、相关文件、代码风格）；把模型的输出（一段代码）翻译成了用户能直接用的结果（写到对应文件、跑编译、修 bug）。

### 3. 套上 Coze 的壳

你在 Coze 里搭一个工作流，把 GPT-4 作为一个节点。输入同样的话，Coze 会怎么处理？

Coze 不会让 GPT-4 直接回答，而是按你设计的工作流走：可能先调一个"代码分析"节点解析用户意图，再调一个"知识库检索"节点找相关代码片段，再调 GPT-4 节点生成代码，最后调一个"代码格式化"节点输出。每一步的输入输出都被 Coze 的工作流引擎管理着。

同样是 GPT-4，在 Coze 里它的角色是"工作流中的一个节点"，而不是"直接面对用户的助手"。它的能力被工作流结构塑造——能做什么、不能做什么、什么时候被调用，都由 Harness 决定。

### 4. 三种壳的差距

把三种 Harness 放一起对比：

| 维度 | ChatGPT | Cursor | Coze |
|---|---|---|---|
| 上下文来源 | 用户输入 + 对话历史 | 项目文件 + 对话历史 + 代码索引 | 工作流上游节点输出 |
| 工具调度 | 代码解释器、联网搜索 | 文件读写、终端、代码索引 | 工作流节点（自定义） |
| 输出处理 | 流式渲染、代码高亮 | 直接写文件、跑编译、改 bug | 传给下游节点 |
| 错误恢复 | 重试、安全过滤 | 编译报错自动修 | 节点失败重试或分支 |

模型还是那个模型，但表现出来的"能力"完全不同。这就是 Harness 的力量——**它决定了模型能"看到什么"、"做到什么"、"做错了怎么办"**。

## 三、拆原理：Harness 的四大核心组件

看完案例，我们来拆 Harness 的内部结构。一个完整的 Harness 通常由四个核心组件组成：上下文组装器、工具路由器、结果解析器、错误恢复器。

### 1. 上下文组装器（Context Assembler）

上下文组装器负责"把现实问题翻译成模型能懂的输入"。它的活儿是：在模型每次推理前，把该塞进上下文的东西按对的顺序、对的大小塞进去。

第 15 章讲过上下文五大组成——指令、历史、知识、工具、记忆。上下文组装器就是这五部分的"调度员"：

- **指令**：从配置文件读 system prompt，加上当前任务规则。
- **历史**：从对话存储里读历史消息，按滑动窗口或摘要策略截取。
- **知识**：调 RAG 检索，把相关文档片段注入。
- **工具**：从工具注册表读工具定义，按需注入（常用工具常驻，非常用按需）。
- **记忆**：从记忆库读用户画像、长期偏好。

不同产品的上下文组装器差别巨大。ChatGPT 的组装器相对简单——主要是 system prompt + 历史 + 工具定义。Cursor 的组装器复杂得多——要扫项目文件、做代码索引、按相关性筛选文件、把相关代码片段注入上下文。这就是为什么 Cursor 写代码比 ChatGPT 强——它的上下文组装器把"项目"这个关键信息翻译成了模型能懂的形式。

类比：上下文组装器像"翻译官"。模型只懂"文本输入"，但现实问题是"我的项目要加登录功能"——这个现实问题里有项目结构、代码风格、数据库 schema、依赖库版本，全是模型看不见的东西。组装器把这些"现实"翻译成"文本"，模型才能干活。

### 2. 工具路由器（Tool Router）

工具路由器负责"决定模型该用哪个工具、什么时候用"。它的活儿是：管理工具注册表、把工具定义按需注入上下文、执行模型生成的工具调用、把结果返回给模型。

工具路由器不只是"模型说调啥就调啥"。它还要做：

- **工具筛选**：如果注册了几十个工具，全塞进上下文太占空间。路由器要根据当前问题筛选相关工具，只把相关的注入。
- **参数校验**：模型生成的工具调用参数可能不对（类型错、缺字段）。路由器要校验，错了要让模型重新生成。
- **执行调度**：工具可能要并发执行（比如同时查两个 API），可能要串行执行（先查 A 再用 A 的结果查 B）。路由器要管调度。
- **结果预处理**：工具返回的原始结果可能很脏（HTML、长文本、嵌套 JSON）。路由器要做预处理，转成模型好理解的结构化数据。

Cursor 的工具路由器就管着一堆工具：读文件、写文件、跑终端命令、跑代码索引、调 LSP（Language Server Protocol，语言服务器协议，提供代码补全、跳转、诊断等功能）。模型说"读 src/routes/auth.ts"，路由器就执行文件读取，把内容返回给模型。模型说"跑 tsc 检查类型"，路由器就跑终端命令，把报错信息返回给模型。

### 3. 结果解析器（Result Parser）

结果解析器负责"把模型输出翻译成用户能用的结果"。它的活儿是：解析模型生成的文本（或结构化输出）、按场景渲染、触发后续动作。

模型输出的本质就是一段文本。但用户看到的不是"一段文本"，而是：

- ChatGPT 里的对话气泡、代码高亮、Markdown 渲染。
- Cursor 里直接写进文件的代码、终端里跑的命令、diff 视图。
- Coze 工作流里传给下游节点的结构化数据。

这些"翻译"都是结果解析器干的。它要把模型的原始输出拆解——哪些是代码、哪些是说明、哪些是要执行的动作（写文件、跑命令、调 API），然后分别处理。

举个 Cursor 的例子。模型生成了一段输出：

```
我先在 src/routes/auth.ts 里创建登录路由：

​```typescript
import { Router } from 'express';
// ... 代码
​```

然后在 app.ts 里挂载：

​```typescript
app.use('/auth', authRouter);
​```
```

结果解析器要把这段输出拆成：

- 一个"创建文件"动作：路径 `src/routes/auth.ts`，内容是第一段代码。
- 一个"修改文件"动作：路径 `app.ts`，要在合适位置插入第二段代码。
- 一段"说明文字"：展示给用户看。

拆完之后，解析器调用工具路由器执行这些动作（写文件、改文件），并把执行结果反馈给模型（"文件已创建"、"app.ts 已修改"）。模型看到反馈，可能再生成下一步——比如"现在跑一下 tsc 看看有没有报错"。

### 4. 错误恢复器（Error Recoverer）

错误恢复器负责"出错了怎么办"。它的活儿是：检测错误、决定恢复策略、把错误信息翻译成模型能理解的形式。

AI 应用里出错是常态，不是异常：

- 模型生成的代码有语法错误，跑不起来。
- 模型调用的工具报错（API 限流、文件不存在、权限不足）。
- 模型生成的工具调用参数不对，工具拒绝执行。
- 模型输出的格式不符合预期（该是 JSON 却生成了纯文本）。
- 模型陷入循环，反复生成同样的输出。

错误恢复器要处理这些情况。常见策略：

- **重试**：网络错误、限流，等一会儿重试。
- **反馈给模型**：把错误信息（比如编译报错）拼进上下文，让模型自己改。这是 Cursor 自动修 bug 的核心机制——跑编译、拿报错、喂回模型、模型改、再跑。
- **降级**：如果强模型超时，切到弱模型；如果工具不可用，换备用工具。
- **兜底**：如果实在恢复不了，给用户友好提示，而不是直接崩。

错误恢复器是 Harness 里最容易被忽视、但最影响用户体验的组件。一个没有错误恢复器的 AI 应用，跑顺的时候挺好，一出错就崩——这种产品没人愿意用。

### 5. 四大组件怎么配合

把四大组件串起来看一次完整的推理流程：

1. 用户输入"帮我加登录功能"。
2. **上下文组装器**：扫项目、读历史、检索相关代码，组装成 messages 数组。
3. 模型推理，生成输出（含代码和动作）。
4. **结果解析器**：拆解输出，识别出"创建文件"、"修改文件"两个动作。
5. **工具路由器**：执行这两个动作，写文件、改文件。
6. 执行时发现 app.ts 里没有合适插入位置（报错）。
7. **错误恢复器**：把报错信息拼进上下文，让模型重新生成。
8. 模型重新生成，这次给出了正确的修改方式。
9. **工具路由器**：再次执行，成功。
10. **结果解析器**：把"成功"反馈给用户，展示 diff。

这就是一个完整的 Harness 工作流程。模型只参与了第 3 步和第 7 步的"推理"，其他全是 Harness 在干活。

## 四、避坑指南：Harness 的三个常见误区

### 1. 以为换个更强的模型就能解决所有问题

这是最常见的误区。产品体验不好，第一反应是"换个更强的模型"——GPT-4 不行换 Claude，Claude 不行换 Gemini，再不行等下一代。

但很多时候，问题不在模型，在 Harness。比如：

- 模型回答不准——可能是上下文组装器没把相关知识注入，模型在瞎猜。
- 模型调工具老出错——可能是工具描述写得不清楚，或工具返回的数据格式乱。
- 模型生成的代码塞不进项目——可能是结果解析器没把"项目结构"这个上下文喂给模型。
- 模型一遇到报错就崩——可能是错误恢复器没把报错反馈给模型让它重试。

正确做法：遇到问题先看是模型的问题还是 Harness 的问题。一个简单的判断方法——把当前上下文（messages 数组）打印出来，自己看一眼。如果连你都觉得"信息不够、规则不清"，那就是 Harness 的问题，换再强的模型也没用。

### 2. 忽视 Harness 的工程价值

很多团队把精力全放在"调模型"上——选哪个模型、调什么参数、用什么提示词——却忽视了 Harness 的工程实现。结果 demo 跑得挺好，一到生产环境就一堆问题：上下文超限、工具调用失败、错误恢复不了、用户体验卡顿。

Harness 是工程，不是技巧。它需要：

- **可观测性**：每次推理的上下文、工具调用、错误日志都要记录，方便排查。
- **可测试**：Harness 的每个组件都要能单独测试——上下文组装器给定的输入是否产出预期的 messages？工具路由器是否能正确处理各种工具调用？
- **可迭代**：Harness 要能快速迭代——加新工具、改 system prompt、调上下文策略，不能每次改都要重新部署整个系统。
- **可监控**：生产环境要监控关键指标——Token 消耗、延迟、错误率、用户满意度。

这些工程能力，是 AI 应用能不能从 demo 走到生产的关键。

### 3. 把 Harness 和 Fine-tuning 混淆

有人会问：Harness 这么麻烦，为什么不直接 Fine-tune 一个模型，让它"天生"就会做这件事？

这是个好问题，但答案不是二选一。Harness 和 Fine-tuning 解决的是不同层面的问题：

- **Fine-tuning**：改变模型本身的"能力倾向"。比如用大量医学问答训练，让模型在医学领域回答更准。改的是模型权重。
- **Harness**：在不改模型的前提下，通过工程手段让模型在特定场景下表现更好。改的是模型外的系统。

两者的关系：Fine-tuning 让模型"更会做某类事"，Harness 让模型"在某个具体场景里把这件事做好"。Fine-tune 一个医学模型，还要配医学知识库、症状查询工具、处方格式化解析器——这些 Harness 组件一个都不能少。

更关键的是：Fine-tuning 成本高、周期长、改起来慢；Harness 改起来快、可迭代、可灰度。绝大多数场景下，先把 Harness 做好，比急着 Fine-tune 更划算。很多看似需要 Fine-tuning 的问题，其实只要 Harness 设计得好，通用模型就能解决。

## 五、悟本质：模型是发动机，Harness 是整车

讲了这么多，Harness 的本质可以用一个类比概括：**模型是发动机，Harness 是整车**。

发动机再强，不能直接开上路。你要装上方向盘（控制）、变速箱（调速）、刹车（安全）、底盘（承载）、车身（外观）、座椅（舒适）——这些加起来才是"能开的车"。发动机决定了车的上限（马力），但整车设计决定了车的实际表现（操控、安全、舒适、油耗）。

AI 应用也一样。模型决定了能力上限，Harness 决定了实际表现。一个 GPT-4 加上简陋的 Harness，可能还不如一个 GPT-3.5 加上精心设计的 Harness——因为后者把模型的能力"榨"出来了，前者让模型的能力"漏"掉了。

这个本质有几个推论：

**推论 1：模型能力 ≠ 产品能力。**

OpenAI 的模型很强，但 ChatGPT 之所以好用，不只是因为模型，还因为它的 Harness——流式渲染、对话管理、代码解释器、安全过滤、错误恢复，每一项都是工程。同样，Cursor 之所以写代码强，不只是因为它用了强模型，更因为它的代码上下文管理、文件操作、编译反馈这套 Harness。

**推论 2：Harness 是产品的护城河。**

模型是公开的——大家都能用 GPT-4、Claude。但 Harness 是自己做的——你怎么组装上下文、怎么调度工具、怎么处理错误，这些是产品差异化的核心。同样是 AI 编程工具，Cursor、GitHub Copilot、Trae、Windsurf 用的模型可能差不多，但体验差异巨大——差异就在 Harness。

**推论 3：通用模型 + 专用 Harness > 专用模型。**

过去做 AI 应用，思路是"训练一个专用模型"——医学模型、法律模型、客服模型。现在思路变了：用通用大模型 + 专用 Harness。因为通用模型能力强、迭代快（跟着 OpenAI/Anthropic 升级），而专用 Harness 改起来灵活、成本低。绝大多数场景下，后者更划算。

**推论 4：Harness 工程师是 AI 时代的新工种。**

提示词工程师关注"怎么写好一句话"，Harness 工程师关注"怎么搭好一整套系统"。后者需要懂模型（知道模型能干什么、不能干什么）、懂工程（能写上下文管理、工具调度、错误恢复的代码）、懂产品（知道用户要什么、体验怎么优化）。这个工种正在快速崛起——Cursor、Anthropic、OpenAI 都在招这类人，叫法可能是 "AI Engineer"、"Product Engineer"、"Agent Engineer"，本质都是 Harness 工程师。

理解了这个本质，你再看任何 AI 产品，都会多一个视角：不只是看"它用的什么模型"，而是看"它的 Harness 怎么设计的"——上下文怎么组装、工具怎么调度、结果怎么解析、错误怎么恢复。这个视角，是 AI 时代产品工程师的核心能力。

## 六、结语

模型是发动机，Harness 是整车——你用的不是模型，是一整套驾驭系统。换更强的发动机未必让车更好开，把整车设计好才是关键。

## 七、自测三问

**问题 1：用大白话讲清 Harness 和提示词工程的区别。**

参考答案要点：提示词工程关注"怎么写好一次输入"——system prompt 怎么写、user 消息怎么问、few-shot 怎么给，焦点是"当前这一次"的文本质量。Harness 关注"怎么搭好一整套系统"——上下文怎么组装、工具怎么调度、结果怎么解析、错误怎么恢复，焦点是"模型外的工程系统"。提示词工程是 Harness 的一个组成部分（system prompt 属于上下文组装器），但 Harness 远不止提示词——它还包括工具路由、结果解析、错误恢复这些工程组件。

**问题 2：为什么同样的 GPT-4 模型，在 Cursor 里写代码比在 ChatGPT 里强？从 Harness 的角度分析。**

参考答案要点：①上下文组装器的差距——Cursor 的组装器会扫项目文件、做代码索引、把相关代码片段注入上下文，模型能看到"你的项目长什么样"；ChatGPT 的组装器只拼用户输入和对话历史，模型看不到项目。②工具路由器的差距——Cursor 有文件读写、终端、LSP 等工具，模型能直接操作项目；ChatGPT 只有代码解释器，不能操作用户的项目。③结果解析器的差距——Cursor 把模型输出拆解成"写文件、改文件、跑命令"等动作并执行；ChatGPT 只是把代码展示给用户。④错误恢复器的差距——Cursor 能跑编译、拿报错、喂回模型让它改；ChatGPT 没有这个闭环。模型还是那个模型，但 Harness 把模型的能力"榨"出来了。

**问题 3：你的 AI 应用体验不好，怎么判断是该换模型还是该改 Harness？**

参考答案要点：①先把当前上下文（messages 数组）打印出来，自己看一眼——如果连你都觉得"信息不够、规则不清、工具描述模糊"，那就是 Harness 的问题，换再强的模型也没用。②看错误模式——如果是模型"不会做某类事"（比如医学知识不准），可能需要换更强的模型或 Fine-tune；如果是模型"做错了但能改"（比如代码塞错位置、工具调错参数），那是 Harness 的问题，应该改上下文组装或工具描述。③看一致性——如果同一个问题有时好有时差，多半是 Harness 不稳定（上下文组装、工具调度有随机性）；如果一直差，可能是模型能力不够。④成本考量——换模型成本高（API 价格、迁移工作），改 Harness 成本低（改 prompt、改工具描述、改错误处理），优先尝试改 Harness。绝大多数体验问题，根因在 Harness 不在模型。
