# 第 13 章：防幻觉与约束：让 AI 乖乖说话

> 前置知识：学完第 09 章（幻觉从哪来）即可
> 学完你能：①掌握 5 种以上防幻觉技巧 ②知道怎么约束 AI 的输出格式 ③在关键场景降低幻觉风险

## 一、讲清楚：第 09 章讲了"为什么"，这一章讲"怎么办"

第 09 章我们拆过幻觉的根：大模型本质是概率续写，它优化的是"看起来对"，不是"事实对"。模型自己没有"事实"这个概念，也没有"我不知道"这种元认知。

知道了病因，下一步自然是开药方。

这一章就是药方。它不解决"模型为什么会幻觉"——那是机制问题，得改模型才能动；它解决的是"作为使用者，我怎么把幻觉压到能接受的范围"。换句话说，**模型不变，靠约束把它的自由度收窄，让它没机会胡说**。

这套思路适用于所有大模型——GPT、Claude、Gemini、文心、通义、Kimi 都一样。具体到工具里，Coze、Claude API、Cursor、Trae 都能落地。本章的代码片段以 Claude API 和 Coze 为主，因为这两个工具的约束设置最直观。

## 二、看案例：约束前后的天壤之别

### 案例 1：让 AI 写技术文档，它编了一个不存在的 API

程序员小张让 AI 帮他写一份"用 Python 调用某支付平台 SDK"的技术文档。他给的提示词很随意：

```
帮我写一份 Python 调用 XXPay SDK 的技术文档，包括初始化、下单、退款。
```

AI 给他写了一份漂亮文档，里面赫然写着：

```python
from xxpay import XXPayClient

client = XXPayClient(api_key="xxx", mode="sandbox")
order = client.create_order(
    amount=100,
    out_trade_no="ORDER_001",
    callback_url="https://example.com/callback"
)
```

小张复制粘贴，运行，报错：`ModuleNotFoundError: No module named 'xxpay'`。一查 PyPI，根本没有这个包名。AI 把"XXPay"这个虚构名字当成了真实包名，还顺手编了 `XXPayClient`、`create_order`、`mode="sandbox"` 这些看起来非常合理的 API。

加了约束之后，提示词变成：

```
你是一位技术文档工程师。请根据我提供的官方 SDK 文档片段，写一份 Python 集成说明。
规则：
1. 只能使用文档中明确出现的类名、方法名、参数名。
2. 文档里没有的 API，一律不得编造，遇到缺失部分请写"文档未提供"。
3. 如果文档片段不足以覆盖初始化/下单/退款三个环节，请明确指出哪一环缺失。

【官方 SDK 文档片段】
（这里粘贴真实文档）
```

AI 这次老老实实按文档写，遇到文档没写的退款接口，直接回："文档未提供退款接口，建议查阅官方文档退款章节。"——它没有再编。

### 案例 2：让 AI 做法律咨询，它瞎引法条

一位实习律师让 AI 帮他起草一份"房屋租赁合同纠纷"的法律意见书。提示词：

```
帮我写一份关于房东提前收房的租赁合同纠纷法律意见书，引用相关法条。
```

AI 写得头头是道，引用了"《合同法》第二百二十七条""《城市房地产管理法》第五十四条"。问题是——《合同法》早在 2021 年就被《民法典》废止了，AI 引的是一部已经不存在的法律。

加约束之后：

```
你是一位中国执业律师。请基于以下材料回答：
1. 仅可引用现行有效的法律条文，禁止引用已废止的法律（如《合同法》《经济合同法》）。
2. 引用法条时必须标注"《XX法》第X条"，并附条文原文。
3. 如果材料中没有相关依据，请直接说"现有材料不足以判断"，不要补充外部推测。

【材料】
（粘贴租赁合同、双方陈述）
```

AI 这次只引了《民法典》第七百零三条、第七百二十九条，并且每条都附了原文。没再出现废止法律。

这两个案例说明同一件事：**约束不是"让 AI 更聪明"，而是"让 AI 没机会胡说"**。下面进入具体操作。

## 三、上手步骤：六种防幻觉技巧

### 技巧 1：让 AI 承认不知道

最便宜的一招，零成本，效果立竿见影。

在提示词里加一句：

```
如果不确定或没有依据，请直接回答"我不知道"，不要编造。
```

这一句的原理是给模型一个"合法的退路"。默认情况下，模型倾向于"必须给出回答"——因为训练数据里"回答了"的样本远多于"拒绝了"的样本。你明确允许它说不知道，它就有了拒绝的概率路径。

进阶版：

```
对每个事实性陈述，请在句末标注置信度（高/中/低）。低置信度的内容请额外标注"建议核实"。
```

让 AI 自评置信度不能完全消除幻觉，但能让你一眼看出哪些段落需要重点核查。

**常见报错**：加了"不知道"指令后，AI 变得过度保守，什么都说"我不知道"。这时把指令改成"如果不确定，请说明哪部分不确定，并给出你已知的部分"，避免一刀切。

### 技巧 2：提供事实来源（RAG 的最小化形态）

第 09 章讲过，幻觉的根是模型没有外部事实。最直接的解法就是把事实喂给它。

提示词模板：

```
请只根据以下材料回答问题，不要使用材料之外的知识。如果材料中没有答案，请回答"材料中未提及"。

【材料】
（粘贴文档、网页、数据）

【问题】
（你的问题）
```

这就是 RAG（Retrieval-Augmented Generation，检索增强生成）的最朴素形态——人工检索 + 喂给模型。工业级 RAG 用向量数据库自动检索，但原理一样：先找资料，再让模型基于资料回答。

**关键细节**：

- "只根据以下材料"这句话必须明确，否则模型会混入训练数据里的相关知识。
- 材料太长时，模型可能"看不到"中间部分（注意力衰减）。建议单次喂的材料控制在 5000 Token 以内，超长就分段。
- 材料之间有冲突时，让 AI 标注冲突，而不是自己选一个。

**预期结果**：模型回答里的事实都能在材料里找到对应。你可以做反向核查——把 AI 的回答逐句对照材料，找不到出处的就是潜在幻觉。

### 技巧 3：结构化输出

自由文本最容易藏幻觉，因为模型可以"自由发挥"。把它关进结构的笼子里，幻觉空间大幅压缩。

要求 JSON 输出：

```
请按以下 JSON 格式输出，不要输出任何其他内容：
{
  "summary": "一句话总结",
  "key_facts": ["事实1", "事实2"],
  "sources": ["对应材料中的段落编号"],
  "uncertain": ["不确定的部分"]
}
```

要求表格输出：

```
请用 Markdown 表格输出，列固定为：
| 事实 | 来源 | 置信度 |
每条事实必须对应一个来源编号，无来源的事实不要写入表格。
```

结构化的好处有三个：

- **格式约束 = 内容约束**：要填 sources 字段，模型就被迫去找来源，找不到就不能写。
- **便于程序化核查**：JSON 可以直接被代码解析，自动校验 sources 字段是否在材料范围内。
- **降低自由发挥空间**：模型把"算力"花在填字段上，花在"编造流畅叙述"上的就少了。

**常见报错**：模型有时会在 JSON 外面多加一句"好的，这是结果："。解决方法是在提示词末尾加"直接输出 JSON，不要任何前后缀"。

### 技巧 4：分步验证（Chain-of-Verification）

让 AI 一次性写完长文，幻觉概率高。把它拆成"先列要点 → 再展开 → 最后自查"三步，每步都给一次纠错机会。

提示词模板：

```
请按以下三步完成回答，每步单独输出：

第一步：列出回答这个问题需要的关键事实点（不超过 5 条），每条标注来源。
第二步：基于第一步的事实点展开回答，不得引入第一步未列出的事实。
第三步：对照第一步的清单，逐条检查第二步的回答是否有事实编造或越界。如有，请指出并修正。
```

这个思路对应一篇专门研究幻觉缓解的论文提出的 Chain-of-Verification（链式验证）：让模型先生成、再自检、再修正。「Dhuliawala et al., Chain-of-Verification Reduces Hallucination in Large Language Models, arXiv, 2023」

**关键细节**：

- 三步必须分开输出，不能让模型"在心里自查"。分开输出才能强制它把自查过程外化。
- 第三步的修正结果才是最终答案，前两步是过程。
- 模型自查不能 100% 发现幻觉，但能抓出明显的那一类（比如第二步引入了第一步没列的事实）。

### 技巧 5：调低温度

前面四个技巧都是提示词层面的。温度（Temperature，控制模型生成随机性的参数，值越高越随机）是 API 层面的参数。

- temperature = 1.0：默认值，随机性适中，适合创意写作。
- temperature = 0.7：常见对话场景，平衡流畅与稳定。
- temperature = 0.2：低随机性，模型倾向选概率最高的 Token，适合事实性任务。
- temperature = 0：几乎确定性输出，同一输入每次得到几乎相同的结果。

在 Claude API 里设置：

```python
import anthropic

client = anthropic.Anthropic(api_key="your-api-key")
response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=1024,
    temperature=0.2,
    system="你是一位严谨的技术文档工程师，只基于提供的材料回答。",
    messages=[
        {"role": "user", "content": "（你的问题）"}
    ]
)
```

在 OpenAI API 里同理：

```python
from openai import OpenAI

client = OpenAI(api_key="your-api-key")
response = client.chat.completions.create(
    model="gpt-4o",
    temperature=0.2,
    messages=[
        {"role": "system", "content": "你是一位严谨的技术文档工程师。"},
        {"role": "user", "content": "（你的问题）"}
    ]
)
```

**注意**：温度调低不等于零幻觉。温度只影响"选哪个 Token"，不影响"模型脑子里有没有这个事实"。模型不知道的事，温度调到 0 它还是会编——只是编得更"稳定"，每次编得一样。

### 技巧 6：在 Claude / Coze 里的具体设置

**Claude（API）**：

- system 参数里写防幻觉规则（"只基于材料回答""不确定就说不知道"）。
- temperature 参数设 0.2-0.5。
- 如果用工具调用（Tool Use），可以让模型在回答前先调用搜索工具查证，再基于查到的内容回答。这是工业级 RAG 的形态。

**Claude（网页版）**：

- 网页版不暴露 temperature 参数，Anthropic 默认设置了一个相对保守的值。
- 可以在对话开头贴一段"防幻觉规则"作为 system 替代物（效果不如 API 的 system，但聊胜于无）。

**Coze**：

- Bot 配置 → "人设与回复逻辑"输入框里写防幻觉规则。
- Bot 配置 → "模型设置"里可以调温度（Coze 把这个参数叫"随机性"或"Temperature"，不同版本叫法略有差异）。
- Coze 的"插件"机制可以接搜索、知识库。开启"知识库"后，Bot 会先检索知识库再回答，相当于内置 RAG。在 Bot 配置 → "知识库"里挂上自己的文档即可。
- Coze 的"工作流"功能可以搭"检索 → 生成 → 自查"的流水线，把分步验证做成自动化。

**Cursor / Trae**：

- 这两个代码工具的"规则文件"（Cursor 的 `.cursorrules`、Trae 的项目规则）就是 system prompt 的持久化形态。把防幻觉规则写进规则文件，每次对话都自动生效。
- 对代码场景特别有用的规则："只能引用项目里实际存在的函数和变量""不确定的 API 必须先查文档再写""禁止编造不存在的库"。

## 四、验收标准

学完这一章，你应该能做到下面三件事。建议拿一个真实任务对照验收。

1. **能用至少 3 种技巧组合约束 AI 输出**。比如同时用"承认不知道 + 提供事实来源 + 结构化输出"三件套，跑一个法律咨询或技术文档任务，对比加约束前后的输出差异。
2. **能让 AI 在不知道时说"不知道"**。故意问一个材料里没有答案的问题，看 AI 是否按规则回答"材料中未提及"而不是编一段。
3. **能在 API 调用里设置 temperature**。用 Claude 或 OpenAI 的 API，写一段带 `temperature=0.2` 的调用代码，跑通并观察输出稳定性。

如果三项都过，本章的核心就吃透了。

## 五、悟本质：防幻觉的本质是"用约束换确定性"

把这一章的六种技巧摊开看，它们其实在做同一件事：**收窄模型的自由度**。

- 让 AI 承认不知道：收窄了"必须回答"的自由度。
- 提供事实来源：收窄了"从训练数据里随便取"的自由度。
- 结构化输出：收窄了"自由发挥叙述"的自由度。
- 分步验证：收窄了"一次性写完不回头"的自由度。
- 调低温度：收窄了"选概率次高 Token"的自由度。

每收窄一分，幻觉就少一分，但模型的"创造力"也少一分。这是一个 trade-off（权衡）。

理解了这个本质，几个推论就清楚了：

**推论 1：不同任务该用不同强度的约束。**

写诗、写文案、做创意发散——这些任务要的是"惊喜"，约束要松，温度可以高一点。
写技术文档、做法律咨询、做医疗问答——这些任务要的是"准确"，约束要严，温度要低。
用同一套约束套所有任务，要么扼杀创意，要么放任幻觉。

**推论 2：约束是有成本的。**

每加一条约束，提示词就长一点，Token 消耗就多一点，模型推理就慢一点。结构化输出还要写代码解析。分步验证要跑三次模型。RAG 要搭检索系统。这些都不是免费的。

所以防幻觉不是"越多越好"，而是"够用就好"。一个内部 demo，能容忍 5% 幻觉，用"承认不知道 + 提供材料"就够了；一个面向客户的法律产品，幻觉率必须压到 1% 以下，那就要上 RAG + 结构化 + 分步验证 + 人工复核全套。

**推论 3：约束不能消灭幻觉，只能压低概率。**

这一点最重要。哪怕你把六种技巧全用上，模型还是可能在某个角落编一句。因为根源在机制——概率模型没有"事实"这个概念。

所以工业级防幻觉永远是"约束 + 复核"两条腿走路：约束把概率压低，复核把漏网的抓出来。完全依赖约束、不做复核的产品，迟早会出事。

## 六、结语

防幻觉不是让 AI 变聪明，是让 AI 没机会胡说。你给的笼子越结实，它越乖——但你也别指望它能飞。
