# 第 31 章　端到端实战：从 0 搭一个 AI 小项目

> 前置知识：学完第 23、29 章（RAG、AI 编程工具）即可
> 学完你能：①把前 5 个模块的技能串起来 ②完成一个能跑的 AI 小项目 ③理解从需求到上线的完整流程

## 一、讲清楚：学了这么多，得串起来才算真会

前面三十章走下来，你已经听过 Transformer、跑过提示词、用过 RAG、玩过 Agent、也用 Cursor 写过代码。但只要这些技能还散落在脑子里，它们就只是"知识点"，不是"能力"。

这一章是整个专栏的"期末考试"。题目很简单：**从零搭一个能跑的 AI 小项目**。不要求多复杂，但要求完整——从需求拆解、方案选型、动手实现，到上线测试、收集反馈，一条龙走完。

为什么必须做这一步？因为真实世界里，没有人会按章节给你派活。老板扔过来一句"做个能查公司文档的 AI 助手"，你得自己判断用零代码、低代码还是写代码，自己决定切分粒度，自己处理"AI 答非所问"的反馈。这种"把单点技能编织成系统能力"的过程，是看再多文章都替代不了的。

本章给三条路径，难度递增，选一条走通即可：

- **方案 A（零代码）**：用 Coze 拖拽搭建，半天能上线
- **方案 B（低代码）**：用 Dify 配 RAG 流程，能调参能扩展
- **方案 C（写代码）**：用 Cursor 辅助写 Python + LangChain + Chroma，从 0 到 1

## 二、看案例：一个"个人知识库助手"

场景很朴素。你有一堆 Markdown 笔记、PDF 资料、网页收藏，散落在各处。每次想找点东西，要么靠记忆，要么靠搜索框硬翻。你想做一个助手：把笔记丢进去，然后用自然语言提问，它基于你的笔记回答，还能告诉你这段话出自哪篇笔记。

这就是典型的 RAG（Retrieval-Augmented Generation，检索增强生成）应用——先检索，再生成。第 23 章讲过原理，这一章把它跑起来。

需求拆解成五步流水线：

1. **上传文档**：支持 txt / md / pdf
2. **文档切分**：把长文档切成小块（chunk），便于检索
3. **向量化**：每个 chunk 用 Embedding 模型转成向量，存进向量库
4. **检索**：用户提问 → 问题向量化 → 在向量库里找最相似的几个 chunk
5. **生成回答**：把找到的 chunk 喂给大模型，让它基于这些内容回答

这五步是 RAG 的骨架，无论用 Coze、Dify 还是手写代码，本质都一样。区别只在于：谁帮你做了前三步，谁留给你调。

## 三、上手步骤

### 方案 A：用 Coze 搭（零代码）

Coze（扣子）是字节出的 Bot 搭建平台，国内版网址 coze.cn，国际版 coze.com。这里以国内版为例。

**第一步：建 Bot**

登录后点"创建 Bot"，起个名字比如"我的笔记助手"，选一个 Bot 模式（标准模式即可）。进入编排页，左侧是 Bot 配置区，中间是预览区。

**第二步：写人设（Persona）**

在"人设与回复逻辑"里写清楚 Bot 该干什么。人设不是装饰，它直接决定 Bot 的行为边界。一段够用的人设：

```
你是一个个人知识库助手。用户会基于自己上传的笔记提问。
回答规则：
1. 只能基于知识库中检索到的内容回答，不要编造。
2. 如果知识库里没有相关信息，直接说"笔记里没找到"，不要硬答。
3. 回答末尾标注来源文件名。
```

**第三步：加知识库**

在"知识库"模块点"+ 添加"，新建一个知识库，类型选"文本格式"。把你的笔记文件传上去（支持 txt / md / pdf / docx）。Coze 会自动做切分和向量化，你只需要等它处理完。

处理完成后可以点进知识库看分块结果。如果发现切得太碎或太长，可以在知识库设置里调"分块大小"和"分块重叠"——这两个参数第 23 章讲过，这里就是它们落地的地方。

**第四步：配模型与调用策略**

在 Bot 设置里选一个大模型（豆包、DeepSeek 等都行）。打开"知识库调用"开关，让 Bot 在回答前自动检索知识库。可以调"召回数量"（Top-K），一般 3-5 比较合适。

**第五步：测试与发布**

在右侧预览区直接提问测试。比如笔记里有篇关于"番茄工作法"的，你就问"番茄工作法具体怎么用"。看回答是否基于笔记、是否标注来源。

测试没问题后点"发布"，可以选择发布到飞书、微信公众号、API 等渠道。最简单的是发布到"Bot Store"或生成 API 接口，自己用 curl 就能调。

**常见报错**：

- Bot 不调用知识库：检查"知识库调用"开关是否打开，人设里是否明确要求"基于知识库回答"
- 回答还是编造：在人设里加更硬的约束，比如"如果检索结果为空，必须回复'未找到'"
- 上传 PDF 后检索效果差：PDF 解析容易丢格式，建议先转成 Markdown 再传

### 方案 B：用 Dify 搭（低代码）

Dify 是开源的 LLM 应用开发平台，可以云端用，也可以自己部署（dify.ai）。比 Coze 更"工程师友好"，能看见 RAG 的每一步参数。

**第一步：创建应用**

登录 Dify，点"创建空白应用"，选"聊天助手"，起个名字。进入应用编排页。

**第二步：配知识库**

左侧点"知识库"→"创建知识库"。上传文档，Dify 会让你选分段策略：

- **自动分段**：Dify 帮你切，省事
- **自定义分段**：自己设 chunk size 和 overlap，可控

建议先用自动分段跑通，再回头调参。处理时选一个 Embedding 模型（OpenAI 的 text-embedding-3-small，或 BGE 系列），Dify 会算好向量存进它内置的向量库。

**第三步：编排 RAG 流程**

回到应用编排页，在"上下文"里关联刚才的知识库。在"提示词"里写系统提示，跟 Coze 的人设类似。在"模型"里选一个大模型。

Dify 的优势是"检索策略"可调。点开"检索设置"，能看到：

- **向量检索**：纯语义相似度
- **全文检索**：传统关键词
- **混合检索**：两者结合，效果通常最好
- **Top-K**：召回几个 chunk
- **Score 阈值**：相似度低于多少不要

这些参数第 23 章都讲过原理，这里就是调它们的地方。一般从"混合检索 + Top-K=3 + 阈值 0.5"开始试。

**第四步：调试与发布**

右上角有调试框，直接提问测试。Dify 还提供"标注回复"功能——你可以人工标注几个高质量问答，Bot 遇到相似问题时优先用标注答案。

测试通过后点"发布"，可以生成 Web 链接、嵌入 iframe，或者通过 API 调用。Dify 的 API 文档很完整，调用方式是标准的 RESTful。

**常见报错**：

- 检索结果不准：先看 chunk 切得对不对，再看 Embedding 模型选对没，最后调 Top-K 和阈值
- 回答太长或太短：在提示词里约束长度，比如"回答不超过 200 字"
- 混合检索报错：Dify 社区版某些 Embedding 模型不支持混合检索，换模型或换检索方式

### 方案 C：用 Cursor 辅助写代码（从 0 到 1）

这是最硬核的一条路，也是收获最大的一条路。走完之后，你对 RAG 的每一行代码都心里有数。

**环境准备**

- Python 3.10+
- 一个 OpenAI API Key（或国内大模型的 Key，比如 DeepSeek、智谱）
- 安装依赖：`pip install langchain langchain-openai chromadb pypdf`

**第一步：用 Cursor 开一个新项目**

在 Cursor 里新建文件夹 `kb-assistant`，新建 `main.py`。打开 Cursor 的 Chat（Cmd/Ctrl+L），把需求贴给它：

```
我要做一个个人知识库助手，用 LangChain + Chroma。
功能：
1. 读取 ./docs 目录下的 md 和 pdf 文件
2. 切分文档，chunk_size=500，overlap=50
3. 用 OpenAI 的 embedding 模型向量化，存进 Chroma
4. 用户提问时检索 top 3 相关 chunk，喂给 gpt-4o-mini 生成回答
5. 回答末尾标注来源文件名

请帮我写 main.py，分函数写，加注释。
```

Cursor 会给你一版代码。**不要直接跑**，先读一遍。重点看这几处：

- 文档加载用的是 `DirectoryLoader` 还是 `PyPDFLoader` + `TextLoader`
- 切分用的是 `RecursiveCharacterTextSplitter`（推荐）还是 `CharacterTextSplitter`
- Embedding 用的是 `OpenAIEmbeddings` 还是其他
- Chroma 是持久化到本地还是内存

**第二步：补全与修正**

Cursor 给的代码通常有 80% 能用，剩下 20% 需要你修。常见问题：

- API Key 没设置环境变量：在终端 `export OPENAI_API_KEY=sk-xxx`
- 文件路径写死：改成相对路径或参数化
- Chroma 持久化目录没建：`mkdir -p ./chroma_db`
- 中文切分效果差：在 splitter 里加中文分隔符 `separators=["\n\n", "\n", "。", "！", "？", "，", " "]`

修的过程就是学习的过程。每改一处，问 Cursor"为什么这么改"，它会解释。

**第三步：核心代码骨架**

一个能跑的最小版本大概长这样（具体实现以你和 Cursor 协作为准）：

```python
from langchain_community.document_loaders import DirectoryLoader, PyPDFLoader, TextLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import Chroma
from langchain.chains import RetrievalQA

# 1. 加载文档
def load_docs(path):
    loader = DirectoryLoader(path, glob="**/*.{md,txt,pdf}")
    return loader.load()

# 2. 切分
def split_docs(docs):
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=500, chunk_overlap=50,
        separators=["\n\n", "\n", "。", "！", "？", "，", " "]
    )
    return splitter.split_documents(docs)

# 3. 向量化 + 存储
def build_vectorstore(docs, persist_dir="./chroma_db"):
    embeddings = OpenAIEmbeddings()
    return Chroma.from_documents(docs, embeddings, persist_directory=persist_dir)

# 4. 检索 + 生成
def make_qa_chain(vectorstore):
    llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
    return RetrievalQA.from_chain_type(
        llm=llm, chain_type="stuff",
        retriever=vectorstore.as_retriever(search_kwargs={"k": 3}),
        return_source_documents=True
    )

if __name__ == "__main__":
    docs = load_docs("./docs")
    chunks = split_docs(docs)
    vs = build_vectorstore(chunks)
    qa = make_qa_chain(vs)
    
    while True:
        q = input("问：")
        if q == "exit":
            break
        result = qa.invoke({"query": q})
        print("答：", result["result"])
        for doc in result["source_documents"]:
            print("来源：", doc.metadata.get("source"))
```

**第四步：跑起来**

```bash
export OPENAI_API_KEY=sk-xxx
mkdir -p ./docs ./chroma_db
# 把笔记文件丢进 ./docs
python main.py
```

**常见报错**：

- `AuthenticationError`：API Key 没设或写错
- `RateLimitError`：调用太频繁，加 sleep 或换模型
- Chroma 报"dimension mismatch"：换了 Embedding 模型但没清空旧的向量库，删掉 `./chroma_db` 重来
- 中文乱码：终端编码问题，Windows 下设 `chcp 65001`
- 检索结果跟问题无关：chunk_size 太大或太小，调一调

### 上线与测试

不管走哪条路，跑通本地测试后还要做两件事。

**第一件：效果验证**

准备 10-20 个测试问题，覆盖三种情况：

- 答案在笔记里（应该答对）
- 答案不在笔记里（应该说"没找到"，而不是编造）
- 问题模糊（应该追问或给保守回答）

跑一遍，统计准确率和"幻觉率"（编造的比例）。这是基线，后面优化都跟它比。

**第二件：收集反馈**

呼应第 28 章讲过的反馈闭环。最简单的做法：在回答下面加两个按钮"👍 有用 / 👎 没用"，用户点"没用"时让他补一句"哪里不对"。这些数据攒起来，定期回看，就能发现"哪类问题答不好"——然后针对性调切分、调检索、调提示词。

进阶一点可以记日志：每次问答的 query、检索到的 chunk、最终回答、用户反馈，都存下来。这是后续做 RAG 优化的金矿。

## 四、验收标准

走完任一方案，能用以下三条判断是否"真跑通了"：

1. **能上传**：至少支持 md 和 pdf 两种格式，上传后能在知识库/向量库里看到内容
2. **能检索**：提一个笔记里明确写过的问题，回答里能引用到对应内容，并标注来源
3. **能拒答**：提一个笔记里没写的问题，回答是"笔记里没找到"而不是编造一段

第三条最容易被忽略，也最能区分"真 RAG"和"套壳聊天"。一个不会说"我不知道"的助手，比一个不会回答的助手更危险。

## 五、悟本质：把单点技能编织成系统能力

回头看这三条路径，你会发现一个有意思的事：**Coze、Dify、手写代码，做的是同一件事**。区别只在于"封装程度"。

- Coze 把五步全包了，你只写人设
- Dify 把前四步包了，你能调检索参数
- 手写代码每一步都自己来，你能改任何东西

这就是"端到端实战"的本质：**理解每一层在做什么，才能选对封装层级**。

很多人卡在"该用哪个工具"上，其实答案很简单——看你的需求边界：

- 只是个人用、不想写代码 → Coze
- 要给团队用、要调参、要 API → Dify
- 要深度定制、要学原理、要写简历 → 手写代码

更深一层，端到端实战教你的不是"搭一个 RAG"，而是"**怎么把一个模糊需求变成一个能跑的系统**"。这个能力迁移到任何 AI 项目都通用：

1. 把需求拆成可执行步骤
2. 每一步选合适的工具或模型
3. 串起来跑通最小版本
4. 用反馈数据持续优化

前 30 章学的是"零件"，这一章学的是"装配"。零件再好，不会装配也造不出车。

## 六、结语
