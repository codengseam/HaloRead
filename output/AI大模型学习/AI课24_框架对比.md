# 第 24 章：框架对比：LangChain/LangGraph/Dify/Coze 怎么选

> 前置知识：学完第 19、23 章（Agent、RAG）即可
> 学完你能：①理解四大框架的定位差异 ②根据自己的需求选框架 ③知道什么场景该用代码、什么场景该用低代码

## 一、讲清楚：框架太多，容易选错

前面几章讲了 Agent（第 19 章）、ReAct（第 20 章）、RAG（第 23 章）。你可能已经摩拳擦掌想动手搭一个了。但真要动手时，第一个问题就是：**用什么框架搭？**

市面上能搭 AI 应用的框架一抓一大把：LangChain、LangGraph、LlamaIndex、Dify、Coze、FastGPT、Flowise、AutoGen、CrewAI……光看名字就头大。每个框架都自称"最好用""最强大""最适合生产"，文档动辄几百页，看完更懵。

这一章不打算把所有框架都过一遍——那是工具书该干的事。我们只聚焦四个最有代表性的：**LangChain、LangGraph、Dify、Coze**。这四个覆盖了从"纯代码、完全控制"到"零代码、完全托管"的整个光谱，理解了它们的定位差异，其他框架你也能对号入座。

本章解决三个问题：四大框架各自定位是什么、同一个需求用四种框架分别怎么做、拿到一个具体需求怎么选框架。学完之后，你面对"该用什么框架"这个问题时，能在一分钟内给出答案。

## 二、看案例：同一个客服机器人，四种做法

拿一个具体需求来对比——**搭一个客服机器人**。

需求描述：某电商公司要搭一个客服 Bot，能回答用户关于订单、物流、退换货的问题。要求：

- 能查订单状态（调内部订单 API）
- 能查物流信息（调物流 API）
- 能回答退换货政策（基于公司文档，用 RAG）
- 能在复杂问题上转人工

同一个需求，四种框架分别怎么做？

### LangChain 的做法：写代码拼组件

LangChain 是 Python 生态最老牌的 AI 应用框架，定位是"AI 应用的瑞士军刀"——什么都有一把，但每把都不一定最锋利。

用 LangChain 搭这个客服 Bot，你要写 Python 代码：

1. 用 `ChatOpenAI` 初始化一个 LLM。
2. 用自定义 tool 封装订单 API、物流 API。
3. 用 `RecursiveCharacterTextSplitter` 切分退换货政策文档，用 `OpenAIEmbeddings` 向量化，存进 Chroma。
4. 用 `RetrievalQA` 把 RAG 链路接上。
5. 用 `AgentExecutor` 把 LLM、工具、RAG 串成一个 Agent，用 ReAct 范式调度。
6. 自己写转人工的逻辑（比如检测到 Agent 连续两次没答上来，就触发转人工）。

代码量大概 200-500 行。优点是**完全可控**——每一步都能调；缺点是**抽象层多、版本变动快**——LangChain 的 API 隔几个月就大改一次，去年写的代码今年可能跑不通，得重写。

### LangGraph 的做法：画状态图

LangGraph 是 LangChain 团队后来出的新框架，专门解决 LangChain 的一个痛点：**复杂流程难管理**。

LangChain 的 Agent 是"一条链"，跑到底就完了。但客服机器人不是一条链——它有分支（简单问题直接答、复杂问题转人工）、有循环（用户追问时回到上一步）、有状态（要记住对话历史、用户身份）。LangChain 的链式结构表达不了这些，硬写会很乱。

LangGraph 的核心抽象是**状态图**（State Graph）——把应用拆成若干节点（每个节点是一个函数），节点之间用边连接，边可以有条件。整个应用是一个有限状态机。

用 LangGraph 搭客服 Bot：

1. 定义状态对象（包含对话历史、用户身份、当前阶段等字段）。
2. 定义节点：`classify_intent`（分类用户意图）、`query_order`（查订单）、`query_logistics`（查物流）、`rag_answer`（RAG 回答）、`escalate_human`（转人工）。
3. 定义边：`classify_intent` 根据意图分别走到 `query_order`、`query_logistics`、`rag_answer`；如果 RAG 答不上来，走到 `escalate_human`。
4. 编译成图，跑起来。

代码量比 LangChain 略多，但**流程清晰、可调试、可持久化**（LangGraph 支持把状态存数据库，断点续跑）。适合流程复杂、需要精细控制的 Agent 应用。

### Dify 的做法：拖拽工作流

Dify 是开源的低代码 AI 应用开发平台，定位是"让非程序员也能搭 AI 应用"。它把 LangChain 那些组件做成了可视化块，你拖拽连线就能搭应用。

用 Dify 搭客服 Bot：

1. 在 Dify 后台创建一个"工作流"应用。
2. 拖一个"开始"节点，配置输入参数（用户问题、用户 ID）。
3. 拖一个"LLM"节点，写 prompt 做意图分类，输出"订单/物流/退换货/其他"。
4. 拖一个"条件分支"节点，根据上一步的输出走不同分支。
5. 订单分支：拖一个"HTTP 请求"节点调订单 API。
6. 物流分支：拖一个"HTTP 请求"节点调物流 API。
7. 退换货分支：拖一个"知识库检索"节点 + 一个"LLM"节点做 RAG。
8. 所有分支汇到一个"LLM"节点，组织最终回答。
9. 转人工逻辑：在条件分支里加一条"如果 RAG 置信度低，走转人工分支"。

全程拖拽，几乎不写代码（只在 HTTP 节点里填 API 地址、在 LLM 节点里写 prompt）。Dify 是开源的，可以自己部署，数据不出公司——这是它比 Coze 强的地方。

### Coze 的做法：配 Bot

Coze 是字节出品的 Bot 搭建平台，定位比 Dify 还要"傻瓜"——Dify 是低代码，Coze 是零代码。你不用拖拽工作流，直接配一个 Bot 就行。

用 Coze 搭客服 Bot：

1. 创建一个 Bot，写 prompt："你是某电商的客服，能查订单、查物流、回答退换货政策。"
2. 在"插件"里加订单查询插件、物流查询插件（Coze 内置一些，也可以自己接 API）。
3. 在"知识库"里上传退换货政策文档，Coze 自动做 RAG。
4. 在"工作流"里可选地搭转人工逻辑（也可以让 Bot 自己判断什么时候转）。
5. 发布到飞书、微信、网页。

全程不写代码，半小时能搭完。缺点是**控制力弱**——Coze 帮你做了大部分决策（用什么模型、怎么切分、怎么检索），你想精细调也调不了多少。而且数据存在字节那边，对数据敏感的企业不合适。

## 三、上手步骤：四大框架定位 + 选型决策树

看完上面四个案例，你应该对四个框架的"手感"有了印象。这一节把它们各自的定位、优缺点、适用场景讲清楚，最后给一个选型决策树。

### 1. LangChain：最全的瑞士军刀

**定位**：AI 应用开发的"标准库"，提供 LLM、Prompt、工具、记忆、检索、Agent 等全套组件的 Python/JS 封装。

**优点**：

- 组件最全——市面上能见到的 AI 能力，LangChain 基本都有封装。
- 生态最大——教程、案例、第三方集成最多，遇到问题好搜答案。
- 适合快速原型——几十行代码能搭一个能跑的 demo。

**缺点**：

- 抽象层多——同一个事往往有三四种写法（Chain、LCEL、Runnable、AgentExecutor），新手容易懵。
- 版本变动快——LangChain 的 API 隔几个月大改一次，老代码经常跑不通。
- 生产环境坑多——抽象层多导致调试困难，性能也不如手写。

**适用场景**：程序员快速搭原型、做技术验证、学习 AI 应用开发。不适合直接拿去上生产（生产环境建议用 LangGraph 或自己封装）。

### 2. LangGraph：基于状态图的复杂 Agent

**定位**：LangChain 团队出品，专门做复杂 Agent 流程的框架。核心抽象是状态图，适合有分支、有循环、有状态的 Agent 应用。

**优点**：

- 流程清晰——节点和边把应用结构显式化，看图就懂。
- 支持复杂流程——分支、循环、人工审批、断点续跑都能做。
- 可调试、可观测——状态可以序列化存数据库，每一步都能回放。
- 生产友好——比 LangChain 更稳定，更适合上生产。

**缺点**：

- 学习曲线陡——要理解状态图、节点、边、reducer 这些概念。
- 代码量比 LangChain 多——简单应用用 LangGraph 是杀鸡用牛刀。
- 生态比 LangChain 小——第三方集成少一些。

**适用场景**：程序员搭复杂 Agent 应用（多步推理、多 Agent 协作、需要人工审批节点、需要持久化状态）。菜鸟教程的框架对比表里也把 LangGraph 归为"适合复杂 Agent 流程"的一类（参见「菜鸟教程 · AI Agent，学习资源」）。

### 3. Dify：开源低代码平台

**定位**：开源的 AI 应用开发平台，可视化拖拽搭应用，支持 RAG、Agent、工作流三种模式。可以自己部署，数据不出公司。

**优点**：

- 低代码——拖拽连线，非程序员也能上手。
- 可私有部署——开源，数据不出公司，对企业合规友好。
- 功能全——RAG、Agent、工作流、模型管理、prompt 管理都有。
- 支持多模型——OpenAI、Claude、智谱、通义、本地模型都能接。

**缺点**：

- 控制力不如代码——复杂逻辑用拖拽表达起来很别扭，有些细节调不了。
- 性能开销——可视化层有 overhead，高并发场景不如纯代码。
- 部署运维——自己部署要管 Docker、数据库、模型服务，有运维成本。

**适用场景**：非程序员但要定制 AI 应用、企业内部要私有部署、需要快速搭一个能用的应用并持续迭代。

### 4. Coze：零代码 Bot 平台

**定位**：字节出品的 Bot 搭建平台，零代码，配一配就能发布到飞书、微信、网页。国内版 coze.cn，海外版 coze.com。

**优点**：

- 零代码——完全不用写代码，配 prompt、加插件、传文档就行。
- 上手最快——半小时能搭一个能跑的 Bot。
- 发布渠道多——一键发布到飞书、微信、抖音、网页、API。
- 内置能力强——插件市场、知识库、工作流、图像生成、语音都有。

**缺点**：

- 控制力最弱——大部分决策平台帮你做了，想精细调调不了多少。
- 数据在平台方——数据存在字节那边，对数据敏感的企业不合适。
- 受平台限制——平台改规则、下线功能，你的应用就受影响。
- 海外版和国内版能力有差异——模型、插件、发布渠道都不一样。

**适用场景**：普通人快速做应用、个人玩家、轻量级企业应用、验证想法。

### 5. 选型决策树

把四个框架的定位画成一条光谱：

```
控制力强 ←─────────────────────────────→ 控制力弱
易用性弱 ←─────────────────────────────→ 易用性强

LangGraph ── LangChain ── Dify ── Coze
（代码/复杂）  （代码/原型）（低代码）  （零代码）
```

选型决策树：

- **你是普通人，不会写代码，想快速做个应用** → 选 Coze。半小时搭完，发布到微信飞书。
- **你是非程序员（产品、运营），但要定制逻辑、要私有部署** → 选 Dify。拖拽搭，自己部署，数据在自己手里。
- **你是程序员，要快速搭原型验证想法** → 选 LangChain。几十行代码出 demo。
- **你是程序员，要搭复杂 Agent 上生产** → 选 LangGraph。状态图管复杂流程，可调试可持久化。
- **你要搭一个简单的 RAG 问答** → Coze 或 Dify 都行，看要不要私有部署。
- **你要搭一个多 Agent 协作系统** → LangGraph（代码）或 Dify（低代码）。
- **你的应用要上高并发生产环境** → 别用框架的默认实现，用 LangGraph 控制流程 + 自己优化性能，或者直接基于 OpenAI SDK / 智谱 SDK 手写。

一句话总结：**普通人 Coze，非程序员 Dify，程序员原型 LangChain，程序员生产 LangGraph**。

## 四、验收标准

怎么判断你选对了框架？三条标准：

**标准一：能说清四个框架的定位差异**

用一句话讲清每个框架：LangChain 是组件最全的代码框架，适合快速原型；LangGraph 是基于状态图的代码框架，适合复杂 Agent；Dify 是开源低代码平台，适合非程序员私有部署；Coze 是零代码 Bot 平台，适合普通人快速做应用。如果讲不清，说明还没真正理解差异。

**标准二：能为一个具体需求选出合适的框架**

拿一个真实需求来测——比如"搭一个能查公司内部文档、能调内部 API、要私有部署的客服 Bot"。正确答案：Dify（要私有部署 + 非程序员可维护）或 LangGraph（程序员搭 + 复杂流程）。如果选 Coze，错在"要私有部署"这条没满足；如果选 LangChain，错在"生产环境用 LangChain 坑多"。

**标准三：知道什么场景该用代码、什么场景该用低代码**

判断依据是两个问题：**要不要精细控制？要不要私有部署？** 两个都要 → 代码（LangGraph）。要私有部署但不要精细控制 → 低代码（Dify）。都不要 → 零代码（Coze）。能讲清这个判断逻辑，说明选型思维过关了。

## 五、悟本质：框架选型是在控制力和易用性之间找平衡

讲了这么多框架，本质可以用一句话概括：**框架选型是在控制力和易用性之间找平衡**。

控制力和易用性是一对矛盾：

- **控制力强**的框架（LangGraph、LangChain），你能调每个细节，但要写代码、要懂原理、要花时间。
- **易用性强**的框架（Coze、Dify），你拖拽配置就能用，但平台帮你做了很多决策，你想改也改不了。

这就像开车和坐公交：

- LangGraph 是开手动挡赛车——你想怎么开怎么开，但得会开车、得懂换挡。
- LangChain 是开自动挡——好开一些，但偶尔还是会熄火。
- Dify 是坐网约车——你说去哪，司机带你走，路线你大致能看懂但改不了。
- Coze 是坐公交——固定路线、固定站点，上车就行，但只能去公交到的地方。

选哪个，取决于你的需求：

- 你要去的地方公交到不了（特殊需求）→ 自己开车（代码）。
- 你要去的地方公交能到，但你想要点私密性（私有部署）→ 打网约车（Dify）。
- 你要去的地方公交能到，不在乎私密性 → 坐公交（Coze）。

这个本质有几个推论：

**推论一：没有"最好的框架"，只有"最合适的框架"。**

很多人争论"LangChain 好还是 Dify 好"，这是伪问题。它们解决的不是同一个问题——LangChain 给程序员用，Dify 给非程序员用。比较它们就像比较"赛车和公交哪个好"——看你是什么人、要去哪。

**推论二：框架会变，但选型思维不变。**

LangChain 的 API 明年可能又大改，Coze 后年可能下线某个功能，Dify 可能被新平台取代。具体框架会变，但"控制力 vs 易用性"这条光谱不会变。掌握了选型思维，新框架出来你也能迅速定位它在光谱的哪个位置。

**推论三：框架是脚手架，不是终点。**

很多人学了 LangChain 就离不开 LangChain，所有应用都用它搭。其实框架是脚手架——帮你快速搭起来，但成熟的应用往往会脱离框架，自己封装底层逻辑。OpenAI、Anthropic 内部肯定不用 LangChain，他们直接调自己的 SDK。所以学框架要学它的思想（Agent 怎么调度、RAG 怎么拼），而不是死记它的 API。

理解了"控制力 vs 易用性"这个本质，你再看市面上新出的框架——AutoGen、CrewAI、Flowise、FastGPT——都能对号入座：AutoGen、CrewAI 是代码框架（偏 LangGraph 那一端），Flowise、FastGPT 是低代码平台（偏 Dify 那一端）。框架再多，光谱就那么一条。

## 六、结语

框架没有好坏，只有合不合适——普通人 Coze、非程序员 Dify、程序员原型 LangChain、程序员生产 LangGraph，选型本质是在控制力和易用性之间找平衡。
