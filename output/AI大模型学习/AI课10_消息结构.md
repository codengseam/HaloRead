# 第 10 章：消息结构：system/user/assistant 的权力游戏

> 前置知识：学完第 04 章（Token）即可
> 学完你能：①理解 system/user/assistant 三种消息的作用 ②知道怎么用 system 设定 AI 的"人设" ③看懂 API 调用时的消息结构

## 一、讲清楚：你以为在聊天，其实在建一个"权力结构"

大多数人第一次用 ChatGPT、Claude、文心一言，都把它当成一个聊天框：你打字，它回话，一来一回。

但只要你稍微深入一点——比如用 API 开发、用 Coze 搭 Bot、用 Cursor 配置规则——你会发现一个反直觉的事实：**和 AI 对话不是"你问我答"，而是"消息有角色之分，角色决定权重"**。

同样是"请用中文回答"这句话，放在不同位置，效果天差地别：

- 放在 system 消息里：AI 全程都会用中文，哪怕你后面用英文问。
- 放在 user 消息里：AI 这次会用中文，下一轮可能就忘了。
- 放在 assistant 消息里（作为历史回复）：基本没效果，AI 不会"听从自己说过的话"。

为什么会有这种差别？因为消息结构背后是一套"权力分层"。理解这套分层，你才能真正驾驭 AI——而不是被它的"随机性"搞得一头雾水。

本章讲清楚三件事：三种消息各自的作用、为什么 system 权重最高、多轮对话是怎么拼起来的。学完之后，你看 API 文档、看 Coze 配置、看别人写的 prompt 模板，都能一眼看懂"为什么这么写"。

## 二、看案例：同一个问题，不同 system 给出完全不同的回答

### 案例 1：同一个"解释量子力学"的问题，三种人设

我们拿"请用一段话解释量子力学"这个问题，配三个不同的 system prompt，看看 AI 的回答怎么变。

**配置 A：system = "你是一位大学物理教授，讲课严谨，喜欢用公式和实验佐证。"**

AI 的回答大概是这样的：从普朗克的能量量子化假说讲起，提到波粒二象性、薛定谔方程、不确定性原理，引用双缝干涉实验，最后落到"量子力学是描述微观世界的概率性理论"。语气严肃，术语密集。

**配置 B：system = "你是一位给小学生讲科普的博主，语言活泼，多用比喻，避免术语。"**

AI 的回答变成这样：把量子力学比作"微观世界里的捉迷藏"，说电子有时候像小球有时候像波，"你不去看它，它就在好几个地方同时存在；你一看它，它就只在一个地方了"。语气轻松，没有公式。

**配置 C：system = "你是一位喜欢抬杠的哲学家，对任何理论都先质疑再讨论。"**

AI 的回答又变了：先问"你说的量子力学是指哥本哈根诠释还是多世界诠释？"，然后质疑"不确定性原理是真的不确定，还是我们测量工具不够？"，最后说"量子力学的'解释'本身就是一个未解决的哲学问题"。

同一个问题，三个 system，三种完全不同的回答。这就是 system 消息的力量——它定义了 AI"以什么身份、用什么风格、在什么边界内"来回答你。

### 案例 2：在 Coze / Claude / ChatGPT 里设置系统消息

具体到产品里，system 消息的设置入口各不相同，但本质一样：

- **ChatGPT 网页版**：在"自定义 GPT"（Custom GPT）里有一栏 Instructions，这里写的就是 system prompt。普通对话里你看不到它，但每次新建对话，模型都默认带着一套 OpenAI 预设的 system。
- **Claude 网页版**：Anthropic 在系统层预设了 system，普通用户改不了；用 API 时可以自己传 system 参数。
- **Coze**：搭 Bot 时第一步就是填"人设与回复逻辑"，这个输入框就是 system prompt 的可视化入口。Coze 还支持在里面引用变量、调用插件，但底层还是 system 消息。
- **API 调用**：直接在 messages 数组里传 `{"role": "system", "content": "..."}`，这是最原始的形态。

不管入口长什么样，底层都是一回事：一段最高优先级的指令，告诉 AI "你是谁、你该怎么做"。

## 三、拆原理：三种消息各自的分工

### 1. system 消息：岗位说明书

system 消息是整个对话的"设定层"。它定义四件事：

- **身份**：你是谁（"你是一位资深 Python 工程师"）
- **能力边界**：你能做什么、不能做什么（"只回答编程相关问题，拒绝闲聊"）
- **风格**：你怎么说话（"用中文，简洁，先给结论再给解释"）
- **规则**：你必须遵守什么（"不要编造 API，不确定时说不知道"）

类比一下：system 就像一份"岗位说明书"。你招一个实习生，第一件事不是给他派活，而是告诉他"你的岗位是 XX，你的职责是 XX，你汇报给谁，哪些事可以做哪些不能做"。这份岗位说明书一旦定下来，他后续所有的工作都在这个框架里。

system 消息的几个特点：

- **优先级最高**：当 system 和 user 冲突时，模型倾向于服从 system。比如 system 说"用中文回答"，user 用英文问，模型还是会用中文回。
- **全局生效**：一次对话里 system 只设一次，但每一轮都生效。
- **位置固定**：通常放在 messages 数组的第一条。

### 2. user 消息：工作任务

user 消息就是你的输入。它是"这一次"要 AI 做的事。

类比：user 消息就像你给实习生派的"具体任务"。"帮我写一个 Python 函数，输入一个列表返回排序后的结果。"——这是一个任务。

user 消息的特点：

- **优先级低于 system**：如果 user 的要求和 system 冲突，user 通常输。
- **逐次生效**：每一轮 user 消息只对当前这一轮和后续生成有直接影响。
- **数量不限**：一次对话可以有任意多条 user 消息（多轮对话）。

### 3. assistant 消息：工作记录

assistant 消息是 AI 的历史回复。它不是"指令"，而是"先例"。

类比：assistant 消息就像实习生之前做过的工作记录。你下次给他派活时，他会参考自己之前是怎么做的，保持一致性。

assistant 消息的作用常被新手忽略，其实它很关键：

- **维持一致性**：如果 AI 上一轮用了某种风格、某种术语，下一轮会倾向于延续。
- **影响后续生成**：模型生成下一句话时，看到的不只是 system 和 user，还有自己之前说过的话。如果之前说错了，后面可能会"将错就错"。
- **可以人为构造**：在 API 调用里，你可以手动写 assistant 消息，相当于"伪造"一段 AI 的工作记录，引导它后续按这个模式走。这就是 few-shot prompting（少样本提示，给模型几个输入输出示例来引导其行为）的底层原理。

### 4. 为什么 system 权重最高

很多人会问：凭什么 system 优先级最高？同样是消息，凭什么 system 说了算？

答案在训练阶段。

第 07 章讲过指令微调（SFT，Supervised Fine-Tuning，用带标注的指令数据继续训练模型）。在 SFT 阶段，训练数据被构造成"system + user + assistant"的形式，其中 assistant 是要模型学习的"正确回答"。训练样本里，system 消息的内容会被设计成"必须服从的指令"——比如 system 说"用中文回答"，训练样本里的 assistant 就一定用中文；system 说"你是 Python 专家"，assistant 就一定按专家风格回答。

模型在大量这样的样本上训练后，学到了一个统计规律：**system 消息里的要求，服从的概率最高**。

这不是硬编码的规则，而是训练数据塑造的"行为倾向"。但因为这个倾向被强化得足够强，实际使用时，system 的权重就表现为"最高"。

有一个例外值得注意：如果 user 消息特别长、特别强调，而 system 消息很弱，模型有时会被 user 带跑。这就是为什么"重要指令要放 system"——不是因为 user 一定没用，而是 system 更稳。

### 5. 多轮对话的消息拼接

很多人以为多轮对话是"AI 记得之前说过什么"。其实不是。

AI 没有"记忆"。每一轮对话，模型都是"重新算一遍"。所谓"记得"，是因为 API 把整个对话历史拼成一个长 messages 数组，重新喂给模型。

举个例子，三轮对话的 messages 数组长这样：

```json
[
  {"role": "system", "content": "你是一位 Python 专家。"},
  {"role": "user", "content": "什么是装饰器？"},
  {"role": "assistant", "content": "装饰器是一种语法糖……"},
  {"role": "user", "content": "给个例子。"},
  {"role": "assistant", "content": "比如 @log 装饰器……"},
  {"role": "user", "content": "怎么带参数？"}
]
```

模型看到这一整串，预测下一个 assistant 消息应该是什么。它"记得"之前的内容，是因为之前的 user 和 assistant 消息都在这个数组里。

这带来几个重要推论：

- **Token 是累积的**：对话越长，messages 数组越大，消耗的 Token 越多，费用越高、速度越慢。
- **上下文窗口有上限**：第 04 章讲过 Token 窗口。对话长到一定程度，最早的消息会被截断或压缩，AI 就"忘了"开头说过什么。
- **历史消息会影响生成**：如果中间某一轮 AI 说错了，这个错误会进入 assistant 消息，影响后续回答。这就是为什么"纠错"时要明确指出之前哪里错了，而不只是重新问一遍。

## 四、避坑指南：消息结构的常见误用

### 1. 把重要指令放在 user 消息里

典型错误：

```
user: 你是一个翻译专家，请把下面这句话翻译成英文：你好。
```

这里"你是一个翻译专家"放在 user 消息里，效果远不如放在 system 里。因为 user 消息优先级低，而且只在当前轮生效——下一轮你换个问题，"翻译专家"这个人设可能就没了。

正确做法：

```
system: 你是一位翻译专家，专门把中文翻译成英文。
user: 你好。
```

### 2. 写超长 system，导致被截断

system 消息能写多长？理论上可以很长，但实际有上限：

- 受上下文窗口限制：system 占的 Token 多了，留给对话的空间就少了。
- 受注意力衰减影响：第 04 章讲过注意力机制，system 太长时，模型对 system 中间部分的"关注"会变弱，重要指令可能被忽略。

实际经验：system 控制在 500-1000 Token 以内比较稳。如果规则特别多，用结构化格式（Markdown 标题、编号列表）写，比一大段散文更有效。

### 3. 忽视历史消息的累积影响

很多人遇到这种情况：对话进行到第 10 轮，AI 突然"变了个人"，风格、立场都和开头不一样了。

原因往往是：中间某一轮 AI 说了一句偏离 system 的话，这句话进入了 assistant 消息，后续生成时模型"参考"了它，越走越偏。

避坑方法：

- 重要规则在 system 里写死，不要依赖"对话中提醒"。
- 发现 AI 跑偏，及时明确纠正（"你刚才说的 XX 不对，应该按 YY 来"），让纠正后的内容进入 assistant 消息。
- 长对话定期开新对话，避免历史消息累积干扰。

### 4. 在 user 消息里反复强调同一件事

"请用中文回答。请用中文回答。请用中文回答。"——这种重复在 user 消息里基本没用，甚至可能让模型困惑。

如果要强制某种行为，放 system 里写一次，比在 user 里写三次都管用。

### 5. 把 few-shot 示例放在 system 里

few-shot 示例（给 AI 看几个输入输出的例子）应该放在 user 和 assistant 的对话历史里，而不是塞进 system。

错误做法：

```
system: 你是一个翻译器。例子：输入"你好"，输出"Hello"。输入"谢谢"，输出"Thanks"。
user: 再见。
```

正确做法：

```
system: 你是一个翻译器，把中文翻译成英文。
user: 你好。
assistant: Hello.
user: 谢谢。
assistant: Thanks.
user: 再见。
```

后者利用了 assistant 消息的"先例"作用，模型学到的模式更稳。

## 五、悟本质：消息结构是"权力分层"

讲了这么多，其实消息结构的本质可以用一句话概括：**权力分层**。

把一次 AI 对话想象成一个组织：

- **system 是宪法**：最高规则，定义组织的根本性质和边界。一次制定，长期有效，所有行为都不能违背。
- **user 是提案**：每一次具体的诉求。"我们要做 XX"。提案要在宪法框架内提出，否则会被驳回。
- **assistant 是先例**：之前做过的决策。先例会影响后续决策，但先例本身不能违背宪法。

这个类比有几个推论：

**推论 1：宪法改一次，影响全局。**

改 system 消息，等于修宪。整个对话的"基调"都会变。所以 Coze、Custom GPT 这些产品里，改 system 是最重的操作——改完最好开新对话，避免历史 assistant 消息和新 system 冲突。

**推论 2：提案要符合宪法，否则无效。**

user 消息里提的要求，如果和 system 冲突，模型倾向于按 system 来。所以"重要规则写 system"不是习惯，是机制决定的。

**推论 3：先例有惯性，但不如宪法。**

assistant 消息会形成路径依赖，但它的优先级低于 system。如果先例跑偏了，可以用 system 强行拉回来——但更好的做法是开新对话，清掉跑偏的先例。

**推论 4：没有宪法，组织就乱。**

不开 system 直接对话，等于让 AI 用默认人设回答。默认人设是模型厂商预设的（通常是"一个乐于助人的助手"），不一定符合你的需求。所以哪怕你觉得"不需要特别设定"，也建议写一个简短的 system，把基本规则定下来。

理解了"权力分层"这个本质，你写 prompt 时就不会再把所有要求都堆在 user 消息里，而是会下意识地分层：哪些是根本规则（system）、哪些是当前任务（user）、哪些是参考先例（assistant）。这个分层意识，是从"会用 AI"到"驾驭 AI"的关键一步。

## 六、结语

消息不是平等的。system 定调，user 提问，assistant 留痕——三层权力，决定了 AI 每一次回答的形状。

## 七、自测三问

**问题 1：为什么"请用中文回答"放在 system 里比放在 user 里更有效？**

参考答案要点：system 消息在训练阶段被设计成"必须服从"的指令，优先级最高，且全局生效；user 消息优先级低，且只在当前轮直接生效。同样的要求放 system 一次，比放 user 多次都管用。

**问题 2：多轮对话里，AI 为什么会"记得"之前说过的话？**

参考答案要点：AI 本身没有记忆，每一轮都是重新计算。所谓"记得"，是因为 API 把整个对话历史（包括之前的 user 和 assistant 消息）拼成一个长 messages 数组，重新喂给模型。模型看到完整历史，所以能"接着说"。这也意味着对话越长，Token 消耗越大，且历史消息会影响后续生成。

**问题 3：如果你发现对话进行到第 8 轮时 AI 风格跑偏了，你会怎么处理？**

参考答案要点：①检查是不是中间某一轮 assistant 消息说了偏离 system 的话，形成了"跑偏的先例"；②在 user 消息里明确指出"你刚才的 XX 不对，应该按 YY 来"，让纠正后的内容进入 assistant 消息；③如果跑偏严重，直接开新对话，清掉历史 assistant 消息的干扰，让 system 重新主导；④根本预防措施是把重要规则写死在 system 里，不依赖对话中提醒。
