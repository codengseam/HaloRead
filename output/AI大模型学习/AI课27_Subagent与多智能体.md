# 第 27 章：Sub-agent 与多智能体协作

> 前置知识：学完第 19、26 章（Agent 心智模型、Loop Engineering）即可
> 学完你能：①理解 Sub-agent 的概念 ②看懂多智能体协作的几种模式 ③知道什么时候该用多 Agent、什么时候不该

## 一、讲清楚：一个 Agent 干不了的事

前面两章我们讲了 Agent 和 Loop。一个 Agent 配上 Loop，能干不少事——写代码、修 bug、查资料、订机票。但任务一复杂，单 Agent 就开始力不从心。

举个具体场景。你让一个 Agent "做一个完整的电商网站"——它要懂需求分析、懂架构设计、懂前端、懂后端、懂数据库、懂测试、懂部署。一个 Agent 把这些全干了？理论上可以，实际上会出问题：

- 上下文越来越长，模型注意力分散，质量下降（第 18 章讲的 Lost in the Middle）。
- 一个角色干所有事，没有"专业分工"，每个环节都做得不深。
- 单点失败——Agent 在某一步卡住，整个任务停摆。
- 没有交叉校验，错了没人发现。

人遇到这种复杂任务会怎么办？组队。找产品经理理需求、找架构师设计、找前端工程师写界面、找后端工程师写接口、找测试工程师验收。每个人专精一块，协作完成。

AI 也一样。一个 Agent 干不了的复杂任务，可以拆给多个 Agent 协作——这就是多智能体（Multi-Agent，多个智能体通过通信和协作共同完成任务的系统）。

本章解决三个问题：Sub-agent 到底是什么、多智能体有哪几种协作模式、什么时候该用多 Agent 什么时候不该。学完之后，你看 Trae 的专家团、MetaGPT 的软件公司、AutoGen 的多 Agent 对话，都能一眼看懂它们的协作骨架。

## 二、看案例：两个真实的多智能体产品

### 案例 1：Trae 的专家团评审

Trae（一个 AI IDE）里有一个"计划评审"功能，背后就是一个多智能体协作。你写好一个开发计划，触发评审，背后不是单个 Agent 在看，而是三个 Agent 同时在看：

- **架构师 Agent**：从架构合理性角度评审——技术选型对不对、模块划分清不清楚、有没有过度设计。
- **测试 Agent**：从可测试性角度评审——验收标准明不明确、测试用例覆盖全不全、有没有漏掉的边界。
- **规则 Agent**：从规范一致性角度评审——是否符合项目既定的目录结构、命名规范、技术栈约束。

三个 Agent 各自拿到计划，各自独立评审，最后把意见汇总成一份评审报告。这种"并行评审"的好处是：每个 Agent 专注一个视角，看得更深；多个视角交叉，能发现单个 Agent 发现不了的问题。

注意一个细节：这三个 Agent 不是各自为政，它们有一个共同的"汇总点"——评审报告。汇总时还要解决冲突（比如架构师说"用 Redis 缓存"，测试 Agent 说"加 Redis 会增加测试复杂度"），这种冲突需要仲裁机制。这是多智能体协作的核心难点之一，后面会展开讲。

### 案例 2：MetaGPT 模拟软件公司

MetaGPT（一个多智能体框架）做了一个更激进的尝试——用多个 Agent 模拟一整个软件公司。你给它一个需求："做一个待办事项 App"，它会自动派出几个角色：

- **产品经理 Agent**：把需求拆成用户故事、功能列表。
- **架构师 Agent**：根据功能列表设计技术架构、模块划分。
- **开发工程师 Agent**：根据架构写代码。
- **测试工程师 Agent**：根据需求写测试用例、跑测试。
- **QA Agent**：验收、出报告。

这些 Agent 按软件开发的真实流程串起来——产品经理的输出（PRD）是架构师的输入，架构师的输出（设计文档）是开发工程师的输入，开发工程师的输出（代码）是测试工程师的输入。一个 Agent 的产出喂给下一个 Agent，像流水线一样。

这种"串行流水线"的好处是：每个 Agent 专注自己的角色，输出质量高；流程清晰，每一步都有明确的交付物。坏处是：上游错了下游跟着错（产品经理需求理解错，后面全错），而且串行执行慢。

这两个案例展示了多智能体协作的两种典型模式——并行评审和串行流水线。下面我们拆开讲。

## 三、拆原理：Sub-agent 与四种协作模式

### 1. Sub-agent 是什么：项目经理分包给组员

Sub-agent（子代理，主 Agent 调用子 Agent 处理子任务的机制），是多智能体协作的最小单元。

类比一下：你是一个项目经理，接到一个大任务"做一个电商网站"。你不会自己一个人干，你会把任务拆成几块——需求分析交给产品经理、架构设计交给架构师、前端交给前端工程师、后端交给后端工程师。每个组员就是一个 sub-agent，你（项目经理）就是主 Agent。

Sub-agent 的工作流程大致是：

1. 主 Agent 接到任务，判断"这事我一个人干不了，得拆"。
2. 主 Agent 把任务拆成几个子任务，每个子任务派给一个 sub-agent。
3. 每个 sub-agent 在自己的上下文里独立工作——它只看自己那部分任务，不被其他子任务干扰。
4. sub-agent 完成后，把结果交回主 Agent。
5. 主 Agent 汇总所有 sub-agent 的结果，决定下一步——继续派活、调整计划、或交付。

这里有一个关键设计：**sub-agent 有独立的上下文**。主 Agent 派活时，只把"这个子任务需要的信息"传给 sub-agent，不把整个任务的上下文全塞过去。这样 sub-agent 的上下文短、专注度高、不容易跑偏。这是 sub-agent 相对"单 Agent 干所有事"的最大优势——上下文隔离。

第 26 章讲 Claude 的 sub-agent 循环时提过这个思路。主 Agent 跑规划-执行 Loop，每个 sub-agent 跑自己的执行-反馈 Loop，分层处理。Sub-agent 是 Loop Engineering 在"分工"维度的延伸。

### 2. 多智能体协作的四种模式

Sub-agent 是单元，多智能体是组织方式。多个 Agent 怎么协作，有四种典型模式。

#### 模式一：串行流水线（Pipeline）

**结构**：A 的输出是 B 的输入，B 的输出是 C 的输入，依次往下。

```
Agent A → Agent B → Agent C → ... → 最终结果
```

**典型场景**：流程明确的任务，每一步有明确的输入输出。

- 软件开发：需求分析 → 架构设计 → 编码 → 测试 → 部署（MetaGPT 模式）。
- 内容生产：选题 → 撰写 → 编辑 → 校对 → 发布。
- 数据处理：采集 → 清洗 → 分析 → 可视化 → 报告。

**关键设计**：

- **每步的输入输出要标准化**：上一步的输出要能直接喂给下一步，不能格式对不上。
- **错误要能传播**：上游错了，下游要么能纠正、要么能停下来报错，不能把错的一路传下去。
- **整体延迟是各步之和**：5 步流水线，每步 10 秒，整体 50 秒。对延迟敏感的场景慎用。

串行流水线的本质是"专业分工"——每个 Agent 专精一个环节，整体质量比单 Agent 全干要高。但代价是"错误传播"和"延迟累积"。

#### 模式二：并行评审（Parallel Review）

**结构**：多个 Agent 同时看同一个东西，各自给意见，最后汇总。

```
                ┌→ Agent A（架构视角）→ 评审意见 A
输入（计划/代码）─┼→ Agent B（测试视角）→ 评审意见 B
                └→ Agent C（规范视角）→ 评审意见 C
                                          ↓
                                       汇总仲裁 → 最终意见
```

**典型场景**：需要多视角交叉验证的任务。

- 计划评审：架构师、测试、规则三个 Agent 并行评审（Trae 的专家团）。
- 代码 review：安全、性能、可读性三个 Agent 并行审代码。
- 文案审核：合规、品牌、吸引力三个 Agent 并行审文案。

**关键设计**：

- **视角要互补**：三个 Agent 看同一个角度没意义，要看不同角度才能交叉发现盲点。
- **汇总要有仲裁机制**：多个 Agent 意见冲突时（一个说"加缓存"、一个说"别加，增加复杂度"），要有仲裁——可以是主 Agent 仲裁、可以是规则仲裁、可以是人工仲裁。
- **并行比串行快**：N 个 Agent 并行，整体延迟约等于最慢的那个 Agent，不是 N 倍。

并行评审的本质是"多视角交叉"——单个 Agent 视角有限，多个视角交叉能发现更多问题。代价是"汇总复杂度"和"成本翻倍"。

#### 模式三：辩论对抗（Debate）

**结构**：多个 Agent 互相挑战，逼近最优解。

```
Agent A 提主张 ─→ Agent B 反驳 ─→ Agent A 修正 ─→ Agent B 再反驳 ─→ ... → 收敛
```

**典型场景**：答案不唯一、需要深度推理的任务。

- 战略决策：Agent A 主张"扩张"，Agent B 主张"收缩"，互相挑战，最后收敛到"先扩张 X 业务、收缩 Y 业务"。
- 投资判断：Agent A 看多，Agent B 看空，互相挑战论据，最后得出"短期看多、长期看空"。
- 算法设计：Agent A 提方案 A，Agent B 提方案 B，互相挑刺，最后融合成方案 C。

**关键设计**：

- **辩论要有规则**：不能变成"为反对而反对"，要有"基于证据反驳"的约束。
- **要有裁判**：辩论不能无限进行，要有裁判判断"什么时候收敛、采纳哪方观点"。
- **Agent 要有差异**：两个一样的 Agent 辩论意义不大，要让它们代表不同立场、不同偏好。

辩论对抗的本质是"通过对抗逼近真理"——和人类辩论赛一个道理。代价是"Token 消耗大"（多轮辩论）和"收敛难控"（可能陷入死循环）。

#### 模式四：层级委派（Hierarchical Delegation）

**结构**：主 Agent 拆任务，子 Agent 各自执行，子 Agent 还能再拆给孙 Agent。

```
                ┌→ Sub-agent A1 ─→ ...
主 Agent ─→ 拆任务 ┼→ Sub-agent A2 ─→ ...
                └→ Sub-agent A3 ─→ ...
```

**典型场景**：任务太大、层次太多，单层拆不够。

- 大型项目：主 Agent 拆成"前端、后端、运维"三大块，每块再拆给子 Agent，子 Agent 再拆给孙 Agent。
- 复杂研究：主 Agent 拆成"市场、技术、财务"三个方向，每个方向再拆子课题。
- 企业级自动化：主 Agent 是"CEO"，下面有"部门经理"Agent，再下面有"员工"Agent。

**关键设计**：

- **层级不要太深**：每加一层，通信成本和错误率都上升。一般 2-3 层够用，再深就难管了。
- **每层的目标要清晰**：上层只管"拆任务、汇总结果"，下层只管"执行子任务、返回结果"，职责分明。
- **跨层通信要克制**：孙 Agent 不要直接和主 Agent 通信，要通过中间层转发，否则层级就乱了。

层级委派的本质是"分层管理"——和人一样，CEO 不管具体活，部门经理管方向，员工管执行。代价是"层级越深越难调试"和"信息失真"。

### 3. 四种模式的关系

四种模式不是互斥的，可以组合。一个复杂的多智能体系统往往同时用几种：

- **外层是层级委派**：主 Agent 把大任务拆给几个 sub-agent。
- **某个 sub-agent 内部是串行流水线**：比如"开发 Agent"内部又拆成"写代码 → 跑测试 → 修 bug"的流水线。
- **某个环节用并行评审**：比如代码写完后，并行调"安全 Agent、性能 Agent、可读性 Agent"评审。
- **关键决策用辩论对抗**：比如架构选型时，让两个 Agent 辩论"用 Redis 还是 Memcached"。

组合使用能处理更复杂的任务，但也带来更高的复杂度。设计多智能体系统时，先想清楚"这个任务需要哪种模式"，不要一上来就堆四种。

### 4. 协作的三个关键问题

不管哪种模式，多智能体协作都要解决三个关键问题：信息共享、冲突解决、终止条件。

**信息共享**：Agent 之间怎么传信息？

- **直接传消息**：A 把输出直接传给 B。简单，但 B 只能看到 A 给的，看不到 A 的思考过程。
- **共享黑板**：所有 Agent 把中间结果写到一个"黑板"上，谁需要谁去读。灵活，但要管理读写权限和并发。
- **通过主 Agent 中转**：A 把结果交给主 Agent，主 Agent 决定给谁、给多少。可控，但主 Agent 成为瓶颈。

不同模式适合不同的信息共享方式。串行流水线适合直接传消息，并行评审适合通过主 Agent 汇总，层级委派适合分层中转。

**冲突解决**：Agent 意见不一致怎么办？

- **主 Agent 仲裁**：主 Agent 看各方意见，自己拍板。简单，但主 Agent 的判断力要够。
- **规则仲裁**：预先定好规则（比如"安全 > 性能 > 可读性"），按规则裁决。可预测，但规则要全。
- **投票**：少数服从多数。民主，但少数派可能正确。
- **人工仲裁**：冲突升级到人，人来决定。可靠，但破坏自动化。

冲突解决是多智能体协作里最难的部分。没有银弹，要根据任务特点选。

**终止条件**：协作什么时候算完？

- **目标达成**：所有 sub-agent 都完成各自任务，主 Agent 汇总后判定"目标达成"。
- **最大轮次**：辩论对抗、并行评审这类多轮协作，要设最大轮次兜底。
- **收敛检测**：连续几轮没有新进展（辩论不再有新论点、评审不再有新意见），强制收敛。
- **外部中断**：用户喊停、超时、预算用完。

终止条件不设计好，多智能体系统要么提前结束（任务没完成），要么无限跑（烧钱）。

### 5. 主流框架对比

业界有几个主流的多智能体框架，各有侧重（参见「菜鸟教程 · AI Agent，多智能体框架」）：

- **AutoGen**（微软开源）：以"对话"为核心，多个 Agent 通过对话协作。适合辩论对抗、多轮讨论场景。特点是灵活——Agent 之间可以自由对话，不预设流程。
- **CrewAI**：以"角色"为核心，每个 Agent 有明确角色（研究员、作家、编辑），按角色分工。适合串行流水线、层级委派。特点是易用——角色定义清晰，上手快。
- **MetaGPT**：以"流程"为核心，模拟真实组织流程（软件公司的产品-架构-开发-测试流水线）。适合流程明确的复杂任务。特点是结构化——每一步有明确输入输出，整体像一个 SOP。

三个框架不是非此即彼，而是侧重不同。AutoGen 偏"自由对话"，CrewAI 偏"角色分工"，MetaGPT 偏"流程编排"。选哪个，看你的任务更像哪种——需要多轮讨论选 AutoGen，需要角色分工选 CrewAI，需要严格流程选 MetaGPT。

## 四、避坑指南：多 Agent 的三个坑

### 1. 为了多 Agent 而多 Agent

典型表现：一个简单任务，硬要拆成多个 Agent。比如"写一句问候语"，拆成"需求分析 Agent + 文案撰写 Agent + 校对 Agent"三个 Agent 协作。

这就好比一个人能拧的螺丝，非要三个人一起拧——一个扶螺丝、一个拿螺丝刀、一个喊口号。除了显得"高大上"，没有任何好处，反而增加沟通成本、降低效率。

避坑方法：

- **先判断任务复杂度**：任务能不能一个 Agent 在合理上下文长度内搞定？如果能，就不要拆。
- **拆分的依据是"专业分工"**：只有当任务确实需要不同"专业"（架构、测试、安全）时，才拆成多个 Agent。专业相同的拆没意义。
- **算清成本**：每多一个 Agent，多一份 Token 成本和延迟。多 Agent 的收益要明显大于成本才值得。

一个简单判断：**单 Agent 跑不动的原因是"上下文太长"或"需要不同专业视角"时，才考虑多 Agent**。其他原因（模型不够强、prompt 没写好、工具没配齐）应该先优化单 Agent，而不是急着上多 Agent。

### 2. 忽视 Agent 间的通信成本

典型表现：多个 Agent 之间频繁通信，每次通信都把大量上下文传来传去，Token 成本爆炸，延迟也爆炸。

多智能体协作不是"多个 Agent 各自干活"那么简单，Agent 之间要交换信息——而每次信息交换都消耗 Token。一个 5 个 Agent 的系统，如果每个 Agent 都和其他 4 个通信，通信路径是 5×4=20 条，每条都传上下文，成本会非常高。

避坑方法：

- **通信要克制**：能不通信就不通信，能少传就少传。每个 Agent 只传"对方需要的信息"，不传整个上下文。
- **优先用"汇总点"模式**：与其让 N 个 Agent 两两通信，不如让它们都把结果交给一个汇总点（主 Agent 或共享黑板），由汇总点分发。
- **设计好通信协议**：什么时机通信、传什么格式、传多少量，都要预先设计，不要让 Agent 自己决定（它们倾向于过度通信）。

### 3. 让 Agent 数量失控

典型表现：任务一复杂就加 Agent，最后系统里有 20 个 Agent，没人能搞清楚谁在干什么、谁依赖谁、出错是谁的责任。

Agent 数量失控的后果：

- **调试困难**：一个错误结果，不知道是哪个 Agent 出的问题。
- **责任不清**：每个 Agent 都觉得自己没问题，但整体结果不对。
- **成本失控**：Agent 越多，Token 成本越高，预算很快烧完。
- **延迟爆炸**：Agent 越多，协作链路越长，整体延迟越高。

避坑方法：

- **从少开始**：先用 2-3 个 Agent 跑通，再根据需要加。不要一上来就铺一堆。
- **每个 Agent 有明确职责**：能一句话说清"这个 Agent 干什么"。说不清的，要么合并、要么删掉。
- **设上限**：根据任务复杂度设 Agent 数量上限（比如不超过 5-7 个），超过就重新设计。
- **可视化协作流程**：把 Agent 之间的协作关系画出来（谁调谁、谁依赖谁），定期 review，发现冗余就精简。

## 五、悟本质：多智能体的本质是分工

讲了这么多，多智能体的本质可以用一句话概括：**分工——和人一样，复杂任务靠协作，简单任务别过度组织**。

人解决复杂问题的方式就是这样。一个人能干的事，不组队；干不了的事，组队。组队时还要分工——谁干什么、谁听谁的、什么时候算完。这些规则搞不清楚，组队反而比单干还乱。

AI 也一样。多智能体不是"越多越好"，而是"该用的时候用、不该用的时候别用"。判断标准很简单：**任务复杂到一个 Agent 干不好（上下文太长、需要不同专业、需要交叉验证），才用多 Agent**。

这个本质有几个推论：

**推论 1：多智能体的收益是"专业化"，成本是"协作开销"。**

多 Agent 的收益来自每个 Agent 专注一个领域，做得更深。成本来自 Agent 之间的通信、汇总、冲突解决。只有当"专业化收益"大于"协作开销"时，多 Agent 才划算。简单任务协作开销大于收益，不如单 Agent。

**推论 2：多智能体的可靠性，由"协作机制"决定，不由"Agent 数量"决定。**

很多人以为"Agent 越多越可靠"。其实反过来——Agent 越多，协作链路越长，出错点越多。多智能体的可靠性取决于协作机制设计得好不好（信息共享、冲突解决、终止条件），不取决于 Agent 数量。一个设计良好的 3 Agent 系统，比一个设计糟糕的 10 Agent 系统可靠得多。

**推论 3：多智能体的天花板，是"任务本身的可分解性"。**

不是所有任务都能拆。有些任务天然耦合——比如"写一首诗"，你拆成"选题 Agent + 选词 Agent + 押韵 Agent"，结果一定不如一个 Agent 一气呵成。多智能体适合"可分解"的任务（软件开发、研究分析、流程审批），不适合"高度耦合"的任务（创作、综合判断、直觉决策）。

**推论 4：多智能体的未来，不是"更多 Agent"，而是"更聪明的协作"。**

业界早期的多智能体尝试倾向于"堆 Agent"——MetaGPT 模拟整个公司、AutoGPT 堆一堆 sub-agent。但实践发现，Agent 堆多了反而难用。现在的趋势是"更聪明的协作"——更少的 Agent、更清晰的分工、更高效的通信、更可靠的冲突解决。少即是多。

理解了这个本质，你再看任何多智能体产品，都会多一个视角：它为什么用多 Agent？用的哪种模式？协作机制设计得怎么样？这个视角，是看懂当前多智能体应用的关键。

## 六、结语

多智能体的本质是分工——复杂任务靠协作，简单任务别过度组织。该用的时候用，不该用的时候别用。

## 七、自测三问

**问题 1：用大白话讲清 Sub-agent 和多智能体的关系，并说明 Sub-agent 相对单 Agent 的最大优势是什么。**

参考答案要点：Sub-agent 是多智能体协作的最小单元——主 Agent 调用子 Agent 处理子任务。多个 Sub-agent 协作构成多智能体系统。Sub-agent 相对单 Agent 的最大优势是"上下文隔离"——每个 sub-agent 只看自己那部分任务的上下文，不被其他子任务干扰，专注度高、不容易跑偏、不会因为上下文太长而质量下降。这相当于把"一个 Agent 背着大包袱赶路"变成"多个 Agent 各背小包袱分头赶路"。

**问题 2：串行流水线、并行评审、辩论对抗、层级委派四种模式各适合什么场景？各举一个例子。**

参考答案要点：①串行流水线适合流程明确的任务，每步有明确输入输出——如软件开发（需求→设计→编码→测试→部署）。②并行评审适合需要多视角交叉验证的任务——如计划评审（架构师、测试、规则三个 Agent 并行看同一份计划）。③辩论对抗适合答案不唯一、需要深度推理的任务——如战略决策（一个 Agent 主张扩张、一个主张收缩，互相挑战，逼近最优解）。④层级委派适合任务太大、层次太多的场景——如大型项目（主 Agent 拆成前端/后端/运维，每块再拆子 Agent）。四种模式可以组合使用。

**问题 3：什么情况下不该用多 Agent？为什么？**

参考答案要点：①简单任务不该用——能一个 Agent 在合理上下文内搞定的（写一句问候语、翻译一个短语），用多 Agent 是浪费。②高度耦合的任务不该用——任务本身不可分解（写一首诗、做综合判断），拆开反而质量下降。③对延迟敏感的任务慎用——多 Agent 协作的延迟是单 Agent 的几倍，实时对话场景不适合。④成本敏感的场景慎用——每多一个 Agent 多一份 Token 成本，大批量任务用多 Agent 成本会爆炸。判断标准：只有当单 Agent 因"上下文太长"或"需要不同专业视角"干不好时，才考虑多 Agent。其他原因（模型不够强、prompt 没写好）应该先优化单 Agent。
