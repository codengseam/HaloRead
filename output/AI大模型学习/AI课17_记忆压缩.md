# 第 17 章：记忆压缩：长对话怎么不丢重点

> 前置知识：学完第 15 章（上下文五大组成）即可
> 学完你能：①理解长对话为什么会"失忆" ②掌握 3 种记忆压缩方法 ③在工具里实现简单的记忆管理

## 一、讲清楚：聊到第 50 轮，AI 忘了第 3 轮说的话

你大概遇到过这种情况：和 AI 聊一个项目，从需求拆解聊到技术选型，再聊到接口设计，几十轮下来，你随口问一句"我们最开始定的目标用户是谁来着？"——AI 给你一个全新的答案，和第 3 轮它自己说过的完全对不上。

不是 AI 在敷衍你，也不是模型变笨了。这是上下文窗口（Context Window，模型一次能"看到"的 Token 上限）的硬限制。

第 04 章讲过 Token，第 10 章讲过消息结构。多轮对话的本质是把整个 messages 数组重新喂给模型——这意味着对话越长，累积的 Token 越多。当总 Token 超过窗口上限时，最早的消息会被截断或丢弃，AI 就"忘了"开头说过什么。

这就引出本章的核心问题：**长对话里，怎么让 AI 不丢重点？**

答案是记忆压缩（Memory Compression，把历史对话浓缩成更短、信息密度更高的形式，再放回上下文里）。本章给三种方法：滑动窗口、摘要压缩、关键信息抽取。三种方法可以单独用，也可以组合用。学完之后，你能在 Coze、Claude 这些工具里，或者在 API 调用里，实现一个简单的记忆管理逻辑。

## 二、看案例：一个长项目讨论，前 20 轮的需求被"挤出去"了

来看一个真实场景。

你在用 Claude 讨论一个"个人知识库助手"的项目。对话是这样推进的：

- 第 1-5 轮：定需求。目标用户是"独立开发者"，核心场景是"把零散笔记整理成可检索的知识库"，必须支持 Markdown，不要做云端同步。
- 第 6-15 轮：技术选型。讨论用 Python 还是 Node，最后定 Python；数据库选 SQLite；前端用 Streamlit。
- 第 16-25 轮：接口设计。定义了 add_note、search_note、list_tags 三个核心函数。
- 第 26-35 轮：写代码。开始实现 search_note，讨论分词、索引、相似度算法。
- 第 36 轮：你说"按我们最开始定的需求，把 add_note 先写完"。

结果 AI 写出来的 add_note，加了一堆云端同步的逻辑。

为什么？因为第 3 轮那句"不要做云端同步"早就被挤出上下文窗口了。模型看到的只是第 16 轮之后的对话，它根本不知道你最初说过不要云端同步。

这不是 Claude 的 bug，所有大模型都一样。窗口满了就得丢，丢的就是最早的。

那怎么办？总不能每次都把第 1-5 轮的需求手动复制粘贴一遍。这就是记忆压缩要解决的问题。

## 三、上手步骤：三种记忆压缩方法

### 方法 1：滑动窗口——只保留最近 N 轮

最简单粗暴的方法：只把最近 N 轮对话放进 messages 数组，更早的直接丢掉。

伪代码：

```python
def build_messages(history, system_prompt, current_user_msg, n=10):
    # 只取最近 n 轮（一轮 = 一条 user + 一条 assistant）
    recent = history[-n*2:] if len(history) > n*2 else history
    messages = [{"role": "system", "content": system_prompt}]
    messages.extend(recent)
    messages.append({"role": "user", "content": current_user_msg})
    return messages
```

调用时：

```python
history = load_history()  # 加载全部历史
messages = build_messages(history, system_prompt, "把 add_note 写完", n=10)
response = client.chat.completions.create(model="...", messages=messages)
```

优点：实现简单，Token 消耗可控。

缺点：丢历史。第 3 轮的需求一旦超出窗口，就彻底没了。

适用场景：闲聊、不需要长期记忆的任务。比如让 AI 帮你润色一段文字，前 50 轮的对话根本不重要，留最近 5 轮就够。

### 方法 2：摘要压缩——定期把旧对话总结成摘要

滑动窗口丢历史，那能不能把旧对话"压缩"一下，留下要点？

可以。思路是：每隔 N 轮，把前面的对话让 AI 自己总结成一段摘要，放进 system 消息里。这样既省 Token，又保留了关键信息。

伪代码：

```python
def maybe_summarize(history, system_prompt, every_n=10):
    if len(history) == 0 or len(history) % (every_n*2) != 0:
        return system_prompt  # 还没到压缩点

    # 把最早的 every_n 轮拿出来总结
    to_summarize = history[:every_n*2]
    summary_prompt = "请把下面这段对话总结成 200 字以内的要点，保留所有关键决策、需求、约定：\n"
    for msg in to_summarize:
        summary_prompt += f"{msg['role']}: {msg['content']}\n"

    summary = llm_call([{"role": "user", "content": summary_prompt}])

    # 把摘要追加到 system，删掉已总结的旧消息
    new_system = system_prompt + "\n\n【历史摘要】\n" + summary
    save_history(history[every_n*2:])  # 删掉已总结的部分
    save_system(new_system)
    return new_system
```

实际使用时，system 消息会变成这样：

```
你是一位 Python 专家，协助开发个人知识库助手。

【历史摘要】
- 目标用户：独立开发者
- 核心场景：零散笔记整理成可检索知识库
- 技术栈：Python + SQLite + Streamlit
- 关键约束：必须支持 Markdown，不要云端同步
- 已定义接口：add_note、search_note、list_tags
```

之后每轮对话，模型都能在 system 里看到这段摘要，不会丢重点。

优点：保留关键信息，Token 消耗可控。

缺点：①摘要本身要花一次额外 LLM 调用，有成本；②摘要可能丢细节——比如某轮讨论里定下的"add_note 函数返回 note_id 而不是 bool"这种细节，摘要时可能被省略；③摘要质量依赖模型能力。

适用场景：长项目讨论、客服 Bot、需要保留长期上下文的任务。

**实操要点**：摘要的频率（every_n）很关键。太频繁，额外 LLM 调用太多；太稀疏，旧消息已经超出窗口才压缩，来不及。经验值是 8-15 轮压缩一次，根据对话信息密度调整。

### 方法 3：关键信息抽取——用结构化 JSON 存关键事实

摘要压缩是"自然语言压缩"，还有一种更精确的方法：把关键事实抽成结构化数据（JSON），每轮注入。

思路：维护一个 JSON 文件，存对话中产生的关键事实（决策、约束、变量值）。每轮对话时，把这个 JSON 序列化成字符串，放进 system 或 user 消息里。

伪代码：

```python
import json

# 关键事实库，初始为空
facts = {
    "project": "个人知识库助手",
    "target_user": "独立开发者",
    "tech_stack": ["Python", "SQLite", "Streamlit"],
    "constraints": ["支持 Markdown", "不要云端同步"],
    "interfaces": ["add_note", "search_note", "list_tags"],
    "decisions": []
}

def build_messages_with_facts(history, system_prompt, current_user_msg, facts):
    facts_str = json.dumps(facts, ensure_ascii=False, indent=2)
    full_system = f"{system_prompt}\n\n【项目关键事实】\n{facts_str}"
    messages = [{"role": "system", "content": full_system}]
    messages.extend(history[-10:])  # 最近 5 轮
    messages.append({"role": "user", "content": current_user_msg})
    return messages
```

system 消息里会带上：

```json
{
  "project": "个人知识库助手",
  "target_user": "独立开发者",
  "tech_stack": ["Python", "SQLite", "Streamlit"],
  "constraints": ["支持 Markdown", "不要云端同步"],
  "interfaces": ["add_note", "search_note", "list_tags"],
  "decisions": []
}
```

每次有新决策，更新这个 JSON：

```python
# 比如第 20 轮决定 add_note 返回 note_id
facts["decisions"].append({
    "round": 20,
    "topic": "add_note 返回值",
    "decision": "返回 note_id（int），不是 bool"
})
save_facts(facts)
```

优点：①信息精确，不会因为摘要而丢细节；②结构化数据模型理解得更稳；③可以人工编辑——发现某个事实错了，直接改 JSON。

缺点：①要维护 facts 的更新逻辑（可以手动，也可以让 AI 自动抽取）；②结构设计要花心思，不同任务结构不同。

适用场景：需要精确保留约束和决策的任务——代码项目、合同审阅、产品需求讨论。

**进阶玩法**：让 AI 自己抽取关键事实。每轮对话结束后，额外调一次 LLM，让它从这轮对话里判断"是否有新的关键事实需要记录"，如果有就更新 facts JSON。这就是 MemGPT（一种分层记忆管理架构）的核心思路——把操作系统的"内存分级"思想搬到大模型上，主存放当前对话，外存放压缩后的长期记忆。

### 方法 4：在 Coze 和 Claude 里用现成的记忆功能

如果你不想自己写代码，主流工具都提供了记忆管理能力。

**Coze 的"记忆"功能**：

Coze（字节跳动的 Bot 搭建平台）在 Bot 配置里有"记忆"模块，常见的两类能力：

- **变量记忆**：你定义一些变量（比如"用户姓名""项目名称"），Bot 会自动从对话里抽取并保存。下次对话时这些变量会自动注入上下文。
- **长期记忆**：把对话中的关键信息以结构化条目形式存起来，后续按需检索注入。

配置入口：Coze 编辑器 → 顶部"记忆"标签 → 添加变量或记忆条目。具体能力以 Coze 官方文档为准（访问日期 2026-06-23）。

**Claude 的"项目"功能**：

Claude 网页版有 Projects（项目）功能。一个 Project 里可以：

- 上传项目相关的资料文件（代码、文档、需求）。
- 写一段 Project Instructions（项目指令，相当于这个 Project 专属的 system）。
- 在这个 Project 下的所有对话共享上述上下文。

适合长周期项目：把需求文档、技术决策、接口定义都放进 Project 的资料里，每次新对话都自动带上，不用重复粘贴。

注意：Project 的资料也有大小上限，且资料多了同样会受 Lost in the Middle（第 18 章会讲）影响——模型对长上下文中间部分的关注度会下降。所以资料要精选，不要把所有历史对话都塞进去。

## 四、验收标准：让 30 轮对话不丢失前 5 轮的关键信息

理论讲完了，怎么验证你真的掌握了？

设计一个最小验收实验：

1. **准备**：写一份 system prompt，里面只包含基本人设（"你是一位 Python 工程师"），不包含任何项目细节。
2. **前 5 轮**：和 AI 讨论一个虚构项目，定下 5 条关键约束。比如：
   - 项目名：note-cli
   - 语言：Python 3.11
   - 数据库：SQLite
   - 不要外部依赖
   - 命令行界面
3. **中间 20 轮**：聊技术细节、写代码、调试，故意把对话拉长。
4. **第 30 轮**：问 AI"我们这个项目最开始定的 5 条约束是什么？"

**验收标准**：

- 不做记忆压缩：AI 大概率只能答出 1-2 条，甚至一条都答不出。
- 用滑动窗口（n=10）：和不做压缩一样，前 5 轮早被丢了。
- 用摘要压缩：AI 应该能答出 4-5 条（取决于摘要质量）。
- 用关键信息抽取（JSON）：AI 应该能 100% 答出全部 5 条。

如果你的实现能让 AI 在第 30 轮准确答出全部 5 条约束，这一章就算过关。

## 五、悟本质：记忆压缩的本质是"用信息密度换信息长度"

回头看这三种方法，本质上都在做同一件事：**用信息密度换信息长度**。

上下文窗口是固定的——不管模型是 8K、32K 还是 128K，总有一个上限。在这个上限内，你能塞进去的"信息量"取决于信息密度。

- 原始对话：信息密度低。第 3 轮那句"不要云端同步"在 500 字的讨论里只占 10 个字，密度极低。
- 摘要压缩：信息密度提高。500 字压成 50 字摘要，关键信息都在。
- 关键信息抽取：信息密度最高。50 字摘要再压成一行 JSON `"constraints": ["不要云端同步"]`，纯信息，零冗余。

这个洞察可以迁移到很多地方：

- **写 prompt 也是记忆压缩**：你写 system prompt 时，把散乱的需求整理成结构化清单，本质上就是在提高信息密度。
- **RAG 的本质也是压缩**：从 100 万字的资料库里检索出 3 段最相关的，放进上下文——这是把"全量资料"压缩成"最相关片段"。
- **知识管理也是压缩**：读书笔记、会议纪要、项目文档，都是把零散信息压缩成高密度结构化产物。

反过来想：为什么有时候 AI 回答得不好？很多时候不是模型不行，是你给的信息密度太低——一大段话里只有一两句关键信息，剩下全是铺垫。模型要在低密度信息里"找重点"，自然容易出错。

所以记忆压缩不只是一个技术技巧，更是一种思维方式：**和 AI 沟通时，永远在问自己——我给的信息密度够高吗？**

## 六、结语

记忆压缩的本质，是用信息密度换信息长度——窗口有限，密度无限。
