# 第 26 章：Loop Engineering：让 AI 自己迭代自己

> 前置知识：学完第 25 章（Harness Engineering）即可
> 学完你能：①理解 Loop Engineering 的核心思想 ②看懂 AI 自我迭代的几种模式 ③在工具里识别 Loop 设计

## 一、讲清楚：从"一次生成"到"反复迭代"

上一章讲 Harness——套在模型外面的工程系统。一个设计良好的 Harness 能让模型在单次推理里表现不错。但有些任务，单次推理再强也搞不定，必须让模型"反复迭代"。

举个具体场景。你让 AI 写一段代码："实现一个函数，读取一个大型 CSV 文件，按某列分组求和，输出结果到新文件。"

如果只让模型"一次生成"，它会写出一段代码，但这段代码几乎一定有问题——可能没处理空值、可能内存爆了、可能列名拼错、可能输出格式不对。你拿去跑，报错，再贴回来让模型改，再跑，再报错……这个"生成→跑→改→再跑"的循环，本来是你手动在做。

Loop Engineering（循环工程，让 AI 在一个闭环里反复迭代、每轮根据反馈改进的设计实践），就是把这个循环自动化——让 AI 自己跑代码、自己看报错、自己改、再跑，直到代码能跑通。

这听起来简单，但它是当前最强 AI 应用的核心模式。Cursor 的 Composer 能自己改代码、自己跑测试、自己修 bug；Trae 的 Agent 会循环执行直到任务完成；Claude 的 sub-agent 能在复杂任务里反复迭代。这些产品之所以强，不只是因为模型强、Harness 好，还因为它们让 AI 进入了"自我迭代"的循环。

本章解决三个问题：Loop Engineering 到底是什么、它有哪几种典型模式、设计一个 Loop 要注意什么。学完之后，你看 AI 产品的执行日志，能一眼识别出"它在跑哪种 Loop、为什么这么设计"。

## 二、看案例：Cursor Composer 怎么自己修 bug

我们看一个真实的 Loop 场景——Cursor 的 Composer 修 bug。

你在 Cursor 里打开一个 TypeScript 项目，按 Cmd+I 唤起 Composer，输入："修复 src/utils/parser.ts 里的 parseDate 函数，它对 '2026-06-23' 这种格式解析报错。"

Composer 不是"一次生成"代码就完事，它会进入一个循环：

**第 1 轮**：

- 读 `src/utils/parser.ts`，看到 `parseDate` 函数当前实现。
- 读相关测试文件，看测试用例覆盖了哪些格式。
- 生成修改：把 `Date.parse(dateStr)` 换成手动解析 `YYYY-MM-DD`。
- 写入文件。
- 跑测试（`npm test`）。
- 观察结果：3 个测试通过，1 个失败——失败的是"解析 '2026/06/23' 应返回 null"，但新代码把它也解析了。

**第 2 轮**：

- 看到测试失败信息。
- 思考：原来 `parseDate` 只应处理 `YYYY-MM-DD`，其他格式应返回 null。
- 生成修改：加一个格式校验，不匹配 `YYYY-MM-DD` 的直接返回 null。
- 写入文件。
- 跑测试。
- 观察结果：4 个测试全过。

**第 3 轮**：

- 看到测试全过。
- 思考：任务完成了，可以结束。
- 调用 finish，把修改总结反馈给用户。

这就是一个典型的 Loop——**生成代码 → 跑测试 → 看结果 → 不合格则改 → 再跑**，循环直到测试通过。整个过程中，模型推理了 3 次，每次都基于上一轮的反馈改进。如果没有 Loop，模型只能"一次生成"，那段代码大概率过不了测试。

注意几个细节：

- **每轮都有明确的反馈**：测试通过/失败是客观信号，不是模型自己"觉得对不对"。
- **每轮的上下文在累积**：第 2 轮模型看到了第 1 轮的代码和测试报错，第 3 轮看到了第 2 轮的代码和测试通过。这些反馈都拼在上下文里。
- **有明确的终止条件**：测试全过 + 模型判断"任务完成"。如果没有终止条件，Loop 可能永远跑下去。
- **有最大迭代次数兜底**：即使测试一直过不了，跑够 N 轮也会强制停，避免无限循环。

这个案例展示了 Loop Engineering 的核心价值——**把"一次性预测"变成"渐进式逼近"**。模型不需要一次写对，只要每轮能改进一点，几轮下来就能收敛到正确结果。

## 三、拆原理：三种 Loop 模式

Loop Engineering 不是只有一种 Loop。根据"反馈从哪来、Loop 在干什么"，可以分成三种典型模式：生成-评估 Loop、执行-反馈 Loop、规划-执行 Loop。

### 1. 生成-评估 Loop（Generate-Evaluate Loop）

**结构**：AI 生成内容 → 评估器打分 → 不合格则改进 → 再生成 → 再评估……

**反馈来源**：评估器（Evaluator）。评估器可以是另一个 LLM、一段规则代码、一个打分模型，甚至人工审核。

**典型场景**：

- 写文案：AI 生成广告文案 → 评估器打分（吸引力、合规性、品牌一致性）→ 不合格则改 → 再生成。
- 写代码：AI 生成代码 → 跑单元测试 → 测试不过则改 → 再生成。
- 翻译：AI 生成译文 → 评估器打分（流畅度、准确性）→ 不合格则改 → 再生成。

**关键设计**：

- **评估器要可靠**：评估器不准，Loop 就会越改越差。比如让一个弱模型评估强模型的输出，弱模型可能把对的改成错的。
- **反馈要具体**：评估器不能只说"不合格"，要说"哪里不合格、怎么改"。模糊的反馈让模型无所适从。
- **终止条件**：评估分数超过阈值，或达到最大迭代次数。

生成-评估 Loop 的核心是"有一个客观的评估标准"。如果评估标准本身模糊（比如"写得好不好"），Loop 效果会大打折扣。

### 2. 执行-反馈 Loop（Act-Observe Loop）

**结构**：AI 执行动作 → 观察结果 → 失败则调整 → 再执行……

**反馈来源**：真实世界的执行结果。代码跑出来的报错、API 返回的数据、文件操作的成功失败。

**典型场景**：

- 修 bug：AI 改代码 → 跑编译 → 编译报错 → 看报错改 → 再跑。
- 调 API：AI 写 API 调用 → 跑 → 返回 401 → 改认证 → 再跑。
- 操作数据库：AI 写 SQL → 跑 → 报错"列不存在" → 改 SQL → 再跑。

**和生成-评估 Loop 的区别**：

- 生成-评估 Loop 的反馈来自"评估器"（可能是另一个 AI 或规则）。
- 执行-反馈 Loop 的反馈来自"真实世界"（代码跑出来的客观结果）。

执行-反馈 Loop 的反馈更可靠——编译报错就是报错，不会骗你。但它也有局限：有些任务没有"可执行的反馈"（比如写文案，没有客观的"跑一下"）。

**关键设计**：

- **反馈要可解析**：执行结果可能是长报错日志、嵌套 JSON、HTML。要预处理成模型能理解的形式。
- **失败要可恢复**：不是所有失败都能让模型自己改——比如权限不足、网络断了，模型改代码也解决不了。要有降级策略。
- **最大迭代次数**：避免无限循环。

第 20 章讲的 ReAct 范式，本质上就是执行-反馈 Loop 的一种实现——Thought（决定做什么）→ Action（执行）→ Observation（观察结果）→ 再 Thought。ReAct 是 Loop Engineering 在 Agent 领域的经典应用。

### 3. 规划-执行 Loop（Plan-Execute Loop）

**结构**：AI 拆解任务为子步骤 → 逐步执行 → 遇阻则重新规划 → 继续执行……

**反馈来源**：执行过程中的障碍。某个子步骤失败、某个假设被推翻、某个中间结果和预期不符。

**典型场景**：

- 做项目：AI 规划"先建数据库、再写 API、再写前端" → 执行中发现数据库 schema 设计有问题 → 重新规划 → 调整后续步骤。
- 写研究报告：AI 规划"先查文献、再整理观点、再写初稿" → 查文献时发现某个关键论文找不到 → 重新规划 → 调整研究方向。
- 部署系统：AI 规划"先配环境、再跑服务、再测连通性" → 配环境失败 → 重新规划 → 换方案。

**和前两种 Loop 的区别**：

- 生成-评估 Loop 关注"输出质量"（生成的内容好不好）。
- 执行-反馈 Loop 关注"动作结果"（执行的动作成不成功）。
- 规划-执行 Loop 关注"整体路径"（任务拆解对不对、要不要重新规划）。

规划-执行 Loop 处理的是更复杂的任务——任务太大、步骤太多、不确定性太高，单步的执行-反馈 Loop 不够用，需要在更高层面动态调整。

**关键设计**：

- **规划要可拆解**：任务要能拆成相对独立的子步骤，否则没法逐步执行。
- **执行要可观测**：每个子步骤的执行结果要能反馈给规划器，否则规划器不知道要不要重新规划。
- **重新规划要有触发条件**：不是每次小失败都要重新规划（那样太频繁），而是"重大障碍"才触发——比如某个子步骤彻底走不通、某个关键假设被推翻。

### 4. 三种 Loop 的关系

三种 Loop 不是互斥的，而是可以组合。一个复杂的 AI 应用可能同时用三种：

- **外层是规划-执行 Loop**：把大任务拆成子步骤。
- **每个子步骤内部是执行-反馈 Loop**：执行具体动作，根据反馈调整。
- **某些子步骤还套生成-评估 Loop**：生成内容后用评估器打分，不合格则改。

比如 Cursor 的 Composer 改一个复杂功能：先规划"要改哪几个文件、按什么顺序"（规划-执行 Loop），每个文件改完跑测试看通过没（执行-反馈 Loop），如果代码风格不符合项目规范则改（生成-评估 Loop）。三种 Loop 嵌套使用。

### 5. Loop 的三个关键设计

不管哪种 Loop，都有三个关键设计点：终止条件、反馈质量、最大迭代次数。

**终止条件**：Loop 什么时候停？这是最关键的设计。常见终止条件：

- **目标达成**：测试通过、评估分数超阈值、任务完成标记。
- **最大迭代次数**：跑够 N 轮强制停。
- **无明显改进**：连续几轮反馈相同、分数不再提升，说明收敛了。
- **用户中断**：用户主动喊停。

没有明确终止条件的 Loop，要么无限跑下去（烧钱），要么不知道什么时候算完成（用户体验差）。

**反馈质量**：反馈是 Loop 的"方向盘"。反馈质量差，Loop 越跑越偏。好反馈要：

- **客观**：来自真实世界（代码报错、测试结果）比来自主观判断（另一个 AI 的"我觉得"）更可靠。
- **具体**：指出"哪里错了、怎么改"，而不是"不合格"。
- **可解析**：模型能看懂、能基于它生成下一步。

**最大迭代次数**：兜底机制。即使终止条件没触发，跑够 N 轮也要停。N 设多大？看任务——简单任务 3-5 轮，复杂任务 10-20 轮。超过 N 还没完成，要么任务本身太难（需要拆分），要么 Loop 设计有问题（反馈不够好、模型不够强）。

### 6. Trae 的 LoopAgent 思维

Trae（一个 AI IDE 产品）的 LoopAgent 是 Loop Engineering 思维的一个具体实现。它的核心思路是：把开发任务交给一个 Agent，让它在 Loop 里反复迭代，直到任务完成。

LoopAgent 的工作方式大致是：

- 接收一个开发任务（比如"加一个用户登录功能"）。
- 规划：拆解任务为子步骤（建数据库表、写 API、写前端、写测试）。
- 执行：逐步执行每个子步骤——读相关文件、改代码、跑测试。
- 反馈：每步执行后看结果——测试过没过、编译报错没、功能对不对。
- 调整：遇阻则重新规划或修改实现。
- 终止：所有子步骤完成、测试通过、任务达成。

这个过程中，Agent 在一个 Loop 里反复迭代——不是"一次生成所有代码"，而是"生成一点、执行一点、看反馈、再生成"。这正是 Loop Engineering 的精髓。

### 7. Claude 的 sub-agent 循环

Claude（Anthropic 的 AI 助手）在处理复杂任务时，会用 sub-agent（子代理）循环。它的思路是：

- 主 Agent 接收任务，判断"这个任务需要分几步、每步交给谁"。
- 主 Agent 派活给 sub-agent——每个 sub-agent 专注于一个子任务。
- sub-agent 在自己的 Loop 里迭代（生成-评估、执行-反馈）。
- sub-agent 完成后把结果交回主 Agent。
- 主 Agent 看结果，决定下一步——继续派活、调整计划、或结束。

这是一种"分层 Loop"——主 Agent 跑规划-执行 Loop，每个 sub-agent 跑自己的执行-反馈 Loop。分层的好处是：每个 Loop 的上下文不会太长（sub-agent 只看自己那部分），整体能处理更复杂的任务。

## 四、避坑指南：Loop 的三个常见坑

### 1. 让 Loop 无限循环

Loop 最怕的是停不下来。典型表现：

- Agent 反复调同一个工具、同样的参数，永远不调用 finish。
- 评估器一直打"不合格"，模型一直改，但越改越差或原地打转。
- 规划-执行 Loop 里，每次重新规划都和上次差不多，陷入死循环。

避坑方法：

- **必设最大迭代次数**：这是兜底，没有例外。简单任务 3-5 轮，复杂任务 10-20 轮。
- **检测重复**：如果连续几轮的输出或动作高度相似，强制中断。
- **检测收敛**：如果连续几轮分数不再提升，说明收敛了，停。
- **在 system prompt 里强调"何时该停"**：让模型知道"信息够了就 finish"、"测试过了就结束"。
- **用更强的模型**：弱模型更容易陷入循环，强模型的"停止判断"更准。

### 2. 忽视反馈质量

反馈是 Loop 的方向盘，反馈差 Loop 就跑偏。典型表现：

- 评估器是另一个弱 LLM，把对的改成错的。
- 反馈太模糊（"不合格"），模型不知道怎么改。
- 反馈太长（整个报错日志原样塞进去），关键信息被淹没。
- 反馈不可信（工具返回错误数据），模型基于错反馈改。

避坑方法：

- **优先用客观反馈**：能用代码跑、能用规则判断的，不要用 LLM 评估。客观反馈比主观判断可靠。
- **反馈要具体**：指出"哪里错了、怎么改"，而不是"不合格"。
- **反馈要预处理**：长报错日志要提取关键信息（哪一行、什么错），不要原样塞进去。
- **反馈要可校验**：如果反馈来自外部工具，工具本身要做数据校验，避免把脏数据喂给模型。

### 3. 在简单任务上用复杂 Loop

Loop 不是万能的，简单任务用复杂 Loop 是浪费。典型表现：

- 写一句问候语，套个生成-评估 Loop，让评估器打分，跑 5 轮。
- 改一个 typo，套个规划-执行 Loop，先规划再执行。
- 翻译一个短语，套三层 Loop，主 Agent 派 sub-agent，sub-agent 再跑 Loop。

避坑方法：

- **判断任务复杂度**：任务能不能"一次生成"搞定？如果能，就不要用 Loop。
- **Loop 的复杂度匹配任务**：简单任务用单层 Loop（生成-评估或执行-反馈），复杂任务才用多层 Loop（规划-执行 + sub-agent）。
- **算清成本**：Loop 每轮都要调模型，Token 费用和延迟会倍增。一个 5 轮 Loop 的成本是单次生成的 5 倍以上。简单任务用 Loop 是花 5 倍钱买不到明显提升。

什么时候该用 Loop？一个简单判断：**任务有"客观反馈"且"一次生成大概率不对"**。写代码（有测试反馈）、调 API（有执行结果）、修 bug（有报错信息）——这些适合 Loop。写一句问候语、翻译一个短语、回答一个常识问题——这些不适合。

## 五、悟本质：把一次性预测变成渐进式逼近

讲了这么多，Loop Engineering 的本质可以用一句话概括：**把一次性预测变成渐进式逼近——像人做事一样，做完检查、检查完改**。

人解决复杂问题的方式就是这样。你写一段代码，不会一次写对——写完跑一下，报错了，看报错改，再跑，再报错，再改，直到跑通。你写一篇文章，不会一次写好——写完读一遍，发现逻辑不顺，改，再读，再改，直到满意。你做一个决策，不会一次想清楚——先试一下，看结果，调整方向，再试，直到找到对的路。

人不是"一次性想清楚所有事"，而是"边做边改、边改边逼近"。Loop Engineering 让 AI 也具备了这种能力。

这个本质有几个推论：

**推论 1：Loop 让弱模型干强模型的活。**

一个弱模型一次生成可能写不对代码，但如果给它一个执行-反馈 Loop——跑测试、看报错、改、再跑——几轮下来它可能也能把代码改对。Loop 把"一次性的高难度预测"拆成"多次的低难度改进"，每次只需要"基于反馈改一点"，这对模型能力的要求低得多。这是为什么有些 AI 应用看起来"很聪明"，其实用的模型并不最强——Loop 把弱模型的能力放大了。

**推论 2：反馈比模型更重要。**

Loop 的质量，主要取决于反馈质量，而不是模型质量。一个强模型配上差反馈（模糊、不可信、不可解析），Loop 跑不好。一个弱模型配上好反馈（客观、具体、可解析），Loop 反而能跑出不错的结果。所以设计 Loop 时，优先把反馈做对——能用客观反馈就不要用主观评估，能具体就不要模糊，能预处理就不要原样塞。

**推论 3：Loop 的成本是"多次推理"。**

Loop 不是免费的。每轮都要调模型，Token 费用和延迟会倍增。一个 5 轮 Loop 的成本是单次生成的 5 倍以上，延迟也是 5 倍以上。所以 Loop 要用在"值得"的任务上——任务复杂、一次生成搞不定、有客观反馈。简单任务用 Loop 是浪费钱和时间。

**推论 4：Loop 让 AI 从"工具"变成"协作者"。**

单次生成的 AI 是"工具"——你给输入，它给输出，用完即走。Loop 的 AI 是"协作者"——你给任务，它自己迭代，遇阻调整，直到完成。前者像计算器，后者像一个初级员工。Loop Engineering 的终极方向，就是让 AI 从"工具"变成"协作者"——能自己干活、自己检查、自己改进。这也是为什么 Cursor、Trae 这些 AI IDE 这么强调 Loop——它们不是在做一个"代码生成工具"，而是在做一个"AI 程序员"。

理解了这个本质，你再看任何 AI 产品，都会多一个视角：它是不是在跑 Loop？跑的哪种 Loop？反馈从哪来？终止条件是什么？这个视角，是看懂当前最强 AI 应用的关键。

## 六、结语

Loop Engineering 把一次性预测变成渐进式逼近——让 AI 像人一样，做完检查、检查完改，循环直到对。模型不需要一次写对，只要每轮能改进一点。

## 七、自测三问

**问题 1：用大白话讲清生成-评估 Loop、执行-反馈 Loop、规划-执行 Loop 的区别，各举一个适合的场景。**

参考答案要点：①生成-评估 Loop 的反馈来自"评估器"（另一个 AI 或规则），关注"输出质量"。适合写文案、翻译这类没有客观"跑一下"的任务。②执行-反馈 Loop 的反馈来自"真实世界"（代码报错、API 返回），关注"动作结果"。适合修 bug、调 API、操作数据库这类有客观执行结果的任务。③规划-执行 Loop 的反馈来自"执行过程中的障碍"，关注"整体路径"。适合做项目、写研究报告、部署系统这类复杂、步骤多、不确定性高的任务。三者可以组合——外层规划-执行，子步骤内部执行-反馈，某些步骤套生成-评估。

**问题 2：你设计一个执行-反馈 Loop 让 AI 修 bug，发现它跑了 15 轮还没修好，反复改同一个文件。你会怎么排查和修复？**

参考答案要点：①先看每轮的反馈——是不是反馈质量差（报错信息太长没预处理、关键信息被淹没），导致模型不知道怎么改。②看每轮的修改——是不是模型在原地打转（改了又改回去），如果是，加重复检测，连续几轮修改高度相似就强制中断。③看任务难度——是不是 bug 本身太难（涉及多个文件、需要理解大量上下文），如果是，考虑拆成多个子任务，每个子任务跑短 Loop。④兜底措施：必设最大迭代次数（比如 10 轮），达到上限强制停，把当前状态反馈给用户。⑤检查终止条件——是不是终止条件设计有问题（比如只看"测试通过"但测试本身没覆盖这个 bug），导致模型一直改但"测试一直过不了"。⑥考虑换更强的模型——弱模型可能就是改不对，强模型一两轮就能搞定。

**问题 3：什么情况下不该用 Loop？为什么？**

参考答案要点：①简单任务不该用 Loop——能一次生成搞定的（写一句问候语、翻译一个短语、回答常识问题），用 Loop 是浪费。②没有客观反馈的任务不该用复杂 Loop——如果反馈只能靠主观判断（"写得好不好"），Loop 容易越改越差。③对延迟敏感的任务不该用 Loop——Loop 每轮都要调模型，延迟倍增，实时对话场景（用户等回复）不适合多轮 Loop。④成本敏感的场景要慎用——Loop 的 Token 费用是单次的 N 倍，大批量任务用 Loop 成本会爆炸。判断标准：任务有"客观反馈"且"一次生成大概率不对"才适合 Loop，否则用单次生成更划算。
