# 第 15 章：上下文五大组成：指令、历史、知识、工具、记忆

> 前置知识：学完第 10 章（消息结构）即可
> 学完你能：①说清上下文的五大组成部分 ②理解每一部分的作用 ③看懂复杂 AI 应用的上下文结构

## 一、讲清楚：从"写好一句话"到"管好一整段对话"

第 10 章讲过消息结构——system、user、assistant 三种角色，构成一次对话的骨架。如果你只是偶尔用 ChatGPT 聊聊天、写写文案，掌握消息结构就够了。

但只要你开始做"真正的 AI 应用"——比如搭一个客服机器人、做一个法律问答系统、写一个能调用工具的 Agent——你会发现一个新问题：**消息结构只是骨架，骨架里要塞什么东西，才是决定应用好坏的关键**。

举一个对比：

- 提示词工程（Prompt Engineering，通过设计和优化输入提示来引导模型输出期望结果的技术）：核心是"写好一句话"。你琢磨怎么把一个问题问得清楚、怎么给 AI 设定好人设、怎么用 few-shot 引导它。关注的焦点是"当前这一次"的输入。
- 上下文工程（Context Engineering，管理和优化模型每次推理时看到的完整输入信息的技术）：核心是"管好一整段对话"。你要决定模型每次推理时看到哪些指令、哪些历史、哪些知识、哪些工具定义、哪些用户记忆。关注的焦点是"整个输入窗口"的组成。

打个比方：提示词工程像"写好一封求职信"，关注这封信本身怎么写；上下文工程像"准备一场面试"，你要决定带什么简历、准备什么案例、提前查什么公司资料、要不要带作品集——所有这些"带进面试室的东西"，都是上下文。

为什么上下文工程更接近真实应用？因为真实应用里，模型从来不是"只看你这一句话"就回答。它看到的是一个完整的输入窗口，里面有系统设定、有对话历史、有检索来的资料、有可调用的工具、有用户的长期偏好。**这个窗口里放什么、不放什么、按什么顺序放，直接决定了模型的回答质量**。

本章讲清楚五件事：上下文的五大组成部分、每一部分的作用、它们怎么拼接成一个完整的 messages 数组、常见踩坑、以及上下文工程的本质。

## 二、看案例：一个客服机器人的完整上下文

假设你在为一家电商公司搭一个客服机器人，要处理订单查询、退换货、物流追踪这些问题。用户问："我上周买的那个耳机怎么还没到？"

如果你只让模型看到这一句话，它会怎么回答？大概率是："请提供订单号，我帮您查询。"——因为它什么都不知道。

但如果你把完整的上下文喂给它，回答会完全不同。我们来看看这个客服机器人的上下文里都有什么。

### 1. 指令（Instructions）

```
你是一位电商客服助手，负责处理订单、退换货、物流问题。
- 语气友好、专业，先共情再解决问题
- 涉及退款、赔偿，必须先核实订单再操作
- 不知道的信息不要编造，告诉用户"我帮您转人工"
- 当前用户已登录，可以查询其订单
```

这是 system prompt，定义了机器人的身份、能力边界、行为规则。

### 2. 历史（History）

```
user: 我的订单 12345 什么时候发货？
assistant: 您的订单 12345 已于 6 月 15 日发货，预计 6 月 18 日送达。
user: 那个耳机呢？
```

这是当前会话的多轮对话记录。模型看到"那个耳机"，能从历史里推断出指的是订单 12345 里的耳机——这就是历史的"连贯性"作用。

### 3. 知识（Knowledge）

```
[检索到的订单信息]
订单号：12345
商品：蓝牙耳机 × 1
下单时间：2026-06-12
发货时间：2026-06-15
物流单号：SF1234567890
当前状态：运输中，预计 6 月 18 日送达

[检索到的物流轨迹]
6-15 包裹已揽收
6-16 到达上海中转中心
6-17 离开上海，发往北京
```

这是从数据库和物流系统检索来的外部资料。模型本身不知道这个订单的存在，必须靠检索把信息"喂"给它。这就是 RAG（Retrieval-Augmented Generation，检索增强生成，先检索相关资料再让模型生成回答的技术）的核心——详见第 22 章。

### 4. 工具（Tools）

```json
[
  {
    "name": "query_logistics",
    "description": "查询物流单号的实时轨迹",
    "parameters": {
      "type": "object",
      "properties": {
        "tracking_number": {"type": "string"}
      }
    }
  },
  {
    "name": "apply_refund",
    "description": "为指定订单提交退款申请",
    "parameters": {
      "type": "object",
      "properties": {
        "order_id": {"type": "string"},
        "reason": {"type": "string"}
      }
    }
  }
]
```

这是模型可以调用的函数定义。模型看到这些定义后，如果判断"需要查实时物流"，就会生成一个工具调用请求，让外部代码去执行，再把结果返回给它。工具的详细机制见第 20、21 章。

### 5. 记忆（Memory）

```
[用户长期记忆]
- 用户偏好：希望使用简洁回复，不要过多寒暄
- 历史摘要：该用户过去 3 个月共有 2 次退换货记录，均为尺码问题
- 会员等级：黄金会员，享受优先处理
```

这是跨会话的长期记忆。哪怕用户开了一个新对话，这些信息也会被注入上下文，让机器人"记得"这位用户的偏好和历史。记忆通常以摘要形式存储，避免把所有历史原样塞进上下文。

### 6. 把它们拼起来

模型每次推理时，看到的完整 messages 数组大致是这样的（简化版）：

```json
[
  {"role": "system", "content": "你是一位电商客服助手...[指令]"},
  {"role": "system", "content": "[用户记忆] 用户偏好：简洁回复..."},
  {"role": "user", "content": "[知识] 订单 12345 信息：... 物流轨迹：..."},
  {"role": "user", "content": "我的订单 12345 什么时候发货？"},
  {"role": "assistant", "content": "您的订单 12345 已于 6 月 15 日发货..."},
  {"role": "user", "content": "那个耳机呢？"},
  {"role": "tool", "name": "query_logistics", "content": "最新轨迹：6-17 离开上海..."}
]
```

模型看到这一整串，才能给出这样的回答："您的耳机（订单 12345）6 月 15 日已发货，目前离开上海发往北京，预计明天（6 月 18 日）送达。需要我帮您持续追踪吗？"

注意几个细节：

- **指令通常放最前面**，因为 system 权重最高（第 10 章讲过）。
- **知识可以放在 user 消息里**（作为检索结果注入），也可以单独放在 system 里，看具体实现。
- **工具定义通常作为 API 的独立参数传入**（如 OpenAI 的 `tools` 字段），不直接进 messages 数组，但模型推理时会看到。
- **记忆可以放在 system 里**（作为长期设定的注入），也可以放在 user 消息里（作为"参考信息"）。
- **历史按时间顺序排列**，最早的在最前面。

不同框架（LangChain、LlamaIndex、Coze、Dify）对这五部分的拼接方式略有差异，但本质都是这五大组成的组合。

## 三、拆原理：五大组成各自的作用

### 1. 指令（Instructions）：定调

指令包括两部分：

- **system prompt**：长期生效的全局设定，定义 AI 的身份、能力、风格、规则。第 10 章详细讲过。
- **当前任务指令**：用户这一次的具体诉求，通常就是 user 消息的内容。

指令的作用是"定调"——告诉模型"你是谁、要做什么、怎么做"。

类比：指令像一份"工作 brief"。你给设计师派活，第一句话是"我们要做一张电商 banner"（任务），后面跟着"风格偏极简、主色用品牌蓝、目标用户是 25-35 岁女性"（设定）。这份 brief 决定了设计师后续所有工作的方向。

指令的核心原则：**重要规则放 system，具体任务放 user**。第 10 章讲过原因——system 优先级最高、全局生效。

### 2. 历史（History）：连贯

历史是多轮对话的记录，包括之前的 user 消息和 assistant 消息（如果用了工具，还包括 tool 消息）。

历史的作用是"连贯"——让模型能理解上下文，避免每轮都从零开始。

第 10 章讲过：AI 没有真正的记忆，每一轮都是"重新算一遍"。所谓"记得"，是因为历史消息被拼进 messages 数组重新喂给模型。

历史的关键特性：

- **Token 累积**：对话越长，历史越大，费用越高、速度越慢。
- **会被截断**：超过上下文窗口时，最早的消息会被丢弃或压缩。
- **会影响生成**：历史里的 assistant 消息会形成"先例"，影响后续回答。

历史管理的常见策略：

- **滑动窗口**：只保留最近 N 轮对话，超出的丢弃。
- **摘要压缩**：把早期对话总结成一段摘要，替换原始消息。
- **选择性保留**：根据相关性筛选要保留的历史（比如只保留和当前问题相关的轮次）。

### 3. 知识（Knowledge）：补盲

知识是从外部数据源检索来的资料，包括文档片段、数据库记录、API 返回结果等。

知识的作用是"补盲"——模型本身的知识来自训练数据，有截止日期、有覆盖盲区，遇到"私有数据"或"最新信息"时必须靠检索补充。

比如模型不知道你们公司的内部规章、不知道用户具体的订单信息、不知道今天的新闻——这些都要靠 RAG 检索后注入上下文。

知识的形态通常是"文本片段"，比如：

```
[来源：公司退换货政策 v2.3]
7 天无理由退货：商品签收后 7 天内可申请，需保持商品完好。
特殊商品不支持退货：贴身衣物、定制商品、虚拟商品。
```

模型看到这段知识，回答退换货问题时就会依据公司政策，而不是凭训练数据里的"常识"瞎编。

知识的详细检索、压缩、重排机制见第 16 章（RCR 框架）和第 22 章（RAG）。

### 4. 工具（Tools）：扩展

工具是模型可以调用的外部函数，让模型能"做事"而不只是"说话"。

工具的作用是"扩展"——突破模型"只能生成文本"的限制，让它能查数据库、调 API、执行代码、读写文件。

工具定义通常包括三部分：

- **名称**：函数名，模型用它来调用。
- **描述**：这个函数做什么、什么时候用。模型靠描述判断"该不该调这个工具"。
- **参数 schema**：函数接受什么参数、什么类型。模型按这个 schema 生成调用参数。

工具调用的流程：

1. 模型看到 user 问题和工具定义。
2. 模型判断"需要调某个工具"，生成一个工具调用请求（函数名 + 参数）。
3. 外部代码执行这个函数，把结果作为 tool 消息返回。
4. 模型看到 tool 消息，基于结果生成最终回答。

工具的详细机制见第 20、21 章。这里只需要记住：**工具定义会占用上下文空间**，每个工具的描述和参数 schema 都要算 Token。

### 5. 记忆（Memory）：延续

记忆是跨会话的长期信息，包括用户偏好、历史摘要、关键事实等。

记忆的作用是"延续"——让模型在多个会话之间保持对用户的"认识"，而不是每次开新对话都从零开始。

记忆和历史的区别：

- **历史是当前会话内的**：开新对话就清空。
- **记忆是跨会话的**：开新对话依然保留。

记忆通常以"摘要"形式存储，而不是原始对话。比如：

```
[用户画像]
- 姓名：张三
- 偏好：技术阅读，喜欢简洁回复
- 历史：过去 3 个月咨询过 5 次退换货，主要问题是尺码
- 重要事实：黄金会员，2024 年开始使用本服务
```

这段记忆可能是从过去几十次对话里抽取、总结出来的，存储在数据库里。每次新对话开始时，从数据库取出注入上下文。

记忆管理的常见策略：

- **全量注入**：把所有记忆都塞进 system。简单但占空间。
- **按需检索**：根据当前问题检索相关记忆。省空间但实现复杂。
- **分层记忆**：短期记忆（最近几次对话）、中期记忆（最近一个月）、长期记忆（用户画像）。不同层用不同策略。

### 6. 五大组成的拼接顺序

完整的 messages 数组通常按这个顺序拼接：

1. **system 消息**：指令（system prompt + 当前任务规则）。
2. **system 消息（或 user 消息）**：长期记忆（用户画像、偏好）。
3. **user 消息（或 system 消息）**：检索来的知识（RAG 片段）。
4. **历史消息**：按时间顺序排列的 user / assistant / tool 消息。
5. **当前 user 消息**：用户这一次的输入。
6. **工具定义**：作为 API 参数传入，模型推理时可见。

不同框架的拼接顺序略有差异，但有几个通用原则：

- **指令放最前**：system 权重最高，定调要早。
- **记忆和知识放指令之后、历史之前**：作为"参考资料"注入。
- **历史按时间顺序**：保持对话连贯性。
- **当前用户消息放最后**：模型对最近的输入注意力最强。

工具定义的位置比较特殊——它通常不进 messages 数组，而是作为 API 的独立参数（如 OpenAI 的 `tools` 字段、Anthropic 的 `tools` 字段）。但模型推理时会把工具定义和 messages 一起作为输入，所以工具定义也属于"上下文"的一部分。

## 四、避坑指南：上下文工程的常见反模式

### 1. 把所有东西塞进 system

新手最容易犯的错：把指令、知识、记忆、甚至 few-shot 示例全塞进 system 消息。

后果：

- system 太长，注意力衰减（第 04 章讲过），中间的规则被忽略。
- 知识频繁变化（每次检索结果不同），但 system 通常是静态的，混在一起难以管理。
- 调试困难——出问题时不知道是哪部分出了错。

正确做法：

- **system 只放"长期设定"**：身份、能力边界、风格、规则。
- **知识放 user 消息**：作为"参考信息"注入，每次检索结果不同时方便替换。
- **记忆可以放 system 或单独的 user 消息**：根据是否频繁变化决定。
- **few-shot 示例放历史**：用 user/assistant 对话形式，而不是塞 system。

### 2. 忽视历史消息的累积成本

很多人搭机器人时，把所有历史都拼进上下文，对话进行到第 20 轮，Token 数已经爆了，费用飙升、速度变慢、还可能触发窗口截断。

正确做法：

- 设定滑动窗口（如保留最近 5-10 轮）。
- 早期对话做摘要压缩。
- 监控每次请求的 Token 数，超阈值时触发压缩。

### 3. 让工具定义占满上下文

有些应用给模型配了几十个甚至上百个工具，每个工具的描述和参数 schema 都很长。结果工具定义本身就占了上下文的一半，留给历史和知识的空间所剩无几。

正确做法：

- **工具分层**：常用工具常驻上下文，非常用工具按需检索（先让模型决定"需要哪类工具"，再注入对应工具定义）。
- **描述精简**：每个工具的 description 控制在一两句话，参数 schema 只保留必要字段。
- **定期清理**：不再使用的工具及时从配置里移除。

### 4. 知识片段过多导致"信息淹没"

RAG 检索时贪多，一次塞 20 段文档片段进上下文，结果模型被信息淹没，反而回答不准。

正确做法：

- 控制注入片段数量（通常 3-5 段为宜）。
- 用重排（Reranking）筛选最相关的片段（详见第 16 章）。
- 片段之间用清晰分隔符隔开，并标注来源。

### 5. 记忆和知识混淆

把"用户的长期偏好"（记忆）和"本次检索到的资料"（知识）混在一起管理，导致记忆被频繁覆盖、知识被错误地长期保留。

正确做法：

- 记忆和知识分开存储、分开注入。
- 记忆更新频率低（按周/月），知识每次检索都变。
- 在上下文里用清晰标签区分（如 `[用户记忆]` vs `[检索知识]`）。

### 6. 忽视上下文顺序

把当前用户问题放在历史前面，或者把知识放在最后——顺序乱了，模型的注意力分配也会乱。

正确做法：遵循"指令 → 记忆 → 知识 → 历史 → 当前问题"的顺序，让模型先看规则、再看参考、最后看问题。

## 五、悟本质：在有限的窗口里，放最有价值的信息

讲了这么多，上下文工程的本质可以用一句话概括：**在有限的窗口里，放最有价值的信息**。

把上下文窗口想象成一个行李箱：

- 行李箱容量有限（Token 上限）。
- 你要带的东西很多（指令、历史、知识、工具、记忆）。
- 每样东西都有重量（Token 数）。
- 每样东西的"价值"不同（对当前问题的相关性）。

上下文工程就是"装箱"——在容量限制下，选择放什么、不放什么、按什么顺序放，让整个行李箱的"总价值"最大化。

这个本质有几个推论：

**推论 1：不是越多越好。**

塞满上下文不等于信息丰富，反而可能让模型注意力分散、关键信息被淹没。少而精，比多而全更有效。

**推论 2：价值是动态的。**

同样一段历史，在当前问题下可能高度相关，换一个问题可能就无关。上下文工程要根据当前问题动态调整内容，而不是静态地"全部塞进去"。

**推论 3：顺序影响价值。**

模型对上下文不同位置的注意力不同（第 04 章讲过注意力衰减）。同样一段信息，放在前面和放在后面，效果可能差很多。

**推论 4：管理比写更重要。**

提示词工程关注"怎么写好一段话"，上下文工程关注"怎么管好一整段对话"。后者更像系统工程——涉及存储、检索、压缩、调度，是真实 AI 应用的核心。

理解了这个本质，你再看 LangChain、LlamaIndex、Coze、Dify 这些框架，会发现它们做的都是同一件事：帮你管理上下文的五大组成，让模型每次推理时看到的输入"刚好够用、刚好相关、刚好有序"。

## 六、结语

上下文不是"对话记录"，而是模型每次推理时看到的整个世界——指令定调、历史连贯、知识补盲、工具扩展、记忆延续，五者各司其职，共同决定 AI 的回答形状。

## 七、自测三问

**问题 1：为什么不能把所有东西（指令、知识、记忆、few-shot 示例）都塞进 system 消息？**

参考答案要点：①system 太长会导致注意力衰减，中间的规则容易被忽略；②知识是动态的（每次检索结果不同），而 system 通常是静态的，混在一起难以管理；③few-shot 示例放 system 不如放历史（用 user/assistant 对话形式），后者利用了 assistant 消息的"先例"作用，模型学到的模式更稳；④调试困难，出问题时难以定位是哪部分出了错。正确做法是分层：system 放长期设定，知识放 user 消息，记忆按更新频率决定放 system 还是 user，few-shot 放历史。

**问题 2：历史和记忆有什么区别？为什么需要分开管理？**

参考答案要点：①历史是当前会话内的多轮对话记录，开新对话就清空；记忆是跨会话的长期信息（用户偏好、历史摘要），开新对话依然保留。②历史通常是原始消息，Token 占用大；记忆通常是摘要形式，Token 占用小。③历史按时间顺序排列，记忆按相关性或更新频率管理。④分开管理的原因：历史频繁变化（每轮都加新消息），记忆更新频率低（按周/月）；混在一起会导致记忆被频繁覆盖、历史被错误地长期保留。

**问题 3：如果一个客服机器人的工具定义占了上下文的 60%，你会怎么优化？**

参考答案要点：①工具分层——常用工具常驻上下文，非常用工具按需检索（先让模型决定"需要哪类工具"，再注入对应工具定义）；②描述精简——每个工具的 description 控制在一两句话，参数 schema 只保留必要字段，去掉冗余说明；③定期清理——不再使用的工具及时从配置里移除；④监控 Token 占用——给工具定义设上限（如不超过上下文的 20%），超限时触发精简或分层。核心思路是"按需注入"而非"全量暴露"。
