# 算法课·21｜堆与优先队列：不完全排序反而更快

> 前置知识：学完第 08 章（堆排序），理解堆的数组表示、上浮/下沉、Floyd 建堆
> 学完你能：①讲清堆作为"部分排序数据结构"的独特定位，与堆排序章区分 ②手撕最小堆 + Top-K 跟踪器 ③理解优先队列在 Dijkstra/A*/任务调度里的角色 ④用双堆求流式数据中位数 ⑤掌握 heapq 库的接口与内部原理 ⑥用对数器对比手写堆与 heapq，并拿下"前 K 高频""合并 K 链表""数据流中位数"三道高频题

## 一、问题与思路：Top-K 为什么不用排序而用堆

### 1. 一个生活场景：热搜榜每秒数千条新数据

微博热搜榜维护着 Top 50 的话题，每秒有数千条新互动数据涌入。如果每来一条数据就把全量互动数排序再取前 50，100 万条数据要重排 100 万次——每次 $O(n\log n)$，CPU 直接被打爆。聪明的做法是只维护一个大小为 50 的小顶堆，堆顶永远是"当前第 50 名"的门槛，新数据比门槛大就替换并调整，每次只花 $O(\log 50) \approx O(1)$。

这就是堆作为数据结构的真正用武之地——**动态维护"部分有序"**，只关心最值或前 K，不关心全局顺序。

### 2. 反直觉：不完全排序反而更快

第 08 章讲过堆排序的排序流程，本章换个视角：堆首先是数据结构，其次才是排序工具。一个反直觉的事实是——**当你只需要前 K 个元素时，把整个数组排好反而比"只维护前 K"慢几个数量级**。

对比 100 万数据求 Top 10：

- 全排序：$O(n\log n) = 10^6 \times 20 \approx 2 \times 10^7$ 次操作
- 大小 K 的小顶堆：$O(n\log k) = 10^6 \times \log_2 10 \approx 3.3 \times 10^6$ 次操作

差距约 6 倍。如果 K 更小（如 Top 3），差距更大。原因在于：**排序付出了"全量排序"的成本，但你只用了前 K 个结果**——后面 $n - K$ 个元素的排序工作全是浪费。堆的"部分有序"特性，让它只付出"维护前 K"的成本，不多不少。

### 3. 与堆排序章的边界划分

第 08 章讲的是"堆如何用作通用排序工具"——建最大堆、反复取顶放末尾，最终得到完整有序数组。本章不再重复排序流程，而是聚焦堆作为**数据结构本身**的能力：

- **堆排序章**：堆是工具，目标是"全排序"
- **本章**：堆是主角，目标是"动态维护最值/前 K/优先级"

两者的代码底层都是 `sift_up`/`sift_down`，但应用场景完全不同。如果你只想排好一个数组，用 `sorted()`；如果你要持续地"取最值、插新值、维护 Top K"，堆是唯一最优解。

### 4. 优先队列：堆的"职业身份"

普通队列遵循 FIFO（First In First Out，先进先出），谁先来谁先走。但很多场景需要"谁优先级高谁先走"——这就是优先队列（Priority Queue）。

| 场景 | "优先级"是什么 |
|---|---|
| 操作系统进程调度 | 进程的 nice 值 |
| Dijkstra 最短路 | 节点到起点的当前距离 |
| A* 寻路 | $f = g + h$（已走代价 + 启发式估计） |
| 任务调度器 | 任务的优先级标签 |
| 急诊分诊 | 病情严重程度 |

堆是优先队列的标准实现——`push` 入队 $O(\log n)$，`pop` 出队（取最优先）$O(\log n)$，`peek` 查看队首 $O(1)$。Python 的 `heapq`、Java 的 `PriorityQueue`、C++ 的 `std::priority_queue`、Go 的 `container/heap`，底层都是堆。

## 二、原理与实现：手撕最小堆 + heapq 原理

### 1. 手撕最小堆（与堆排序章的最大堆区分）

堆排序章讲的是最大堆（取顶放末尾得升序）。本章手撕**最小堆**——因为 Top-K（求前 K 大）用的是最小堆：堆顶是"当前 Top K 中最小"的那个，作为门槛。

```python
class MinHeap:
    """最小堆：堆顶永远是最小值。"""
    def __init__(self):
        self.heap = []

    def push(self, val):
        """插入：放到末尾，向上调整。"""
        self.heap.append(val)
        self._sift_up(len(self.heap) - 1)

    def _sift_up(self, i):
        """最小堆上浮：子比父小就交换。"""
        while i > 0:
            parent = (i - 1) // 2
            if self.heap[i] >= self.heap[parent]:
                break
            self.heap[i], self.heap[parent] = self.heap[parent], self.heap[i]
            i = parent

    def pop(self):
        """取顶：把末尾填到堆顶，向下调整。"""
        if not self.heap:
            raise IndexError("pop from empty heap")
        top = self.heap[0]
        last = self.heap.pop()              # 末尾元素弹出
        if self.heap:                       # 还有元素，把末尾填到堆顶再下沉
            self.heap[0] = last
            self._sift_down(0, len(self.heap))
        return top

    def _sift_down(self, i, n):
        """最小堆下沉：父比某个子大就和较小子交换。"""
        while True:
            left = 2 * i + 1
            right = 2 * i + 2
            smallest = i
            if left < n and self.heap[left] < self.heap[smallest]:
                smallest = left
            if right < n and self.heap[right] < self.heap[smallest]:
                smallest = right
            if smallest == i:
                break
            self.heap[i], self.heap[smallest] = self.heap[smallest], self.heap[i]
            i = smallest

    def peek(self):
        return self.heap[0] if self.heap else None

    def __len__(self):
        return len(self.heap)
```

**逐行讲透**（最小堆版本，与第 08 章最大堆的关键差异）：

- **`if self.heap[i] >= self.heap[parent]: break`**：最小堆的堆序性是"父 ≤ 子"，所以子 ≥ 父就该停。注意是 `>=` 不是 `>`——相等也停，避免无意义交换
- **`last = self.heap.pop()` + `self.heap[0] = last`**：取顶的标准三步——存顶、弹末尾、末尾填顶。**不能直接 `pop(0)`**，那是 $O(n)$；用末尾填顶 + 下沉是 $O(\log n)$
- **`if self.heap:` 守卫**：弹末尾后如果堆空了（原本只有 1 个元素），就不用填顶也不用下沉——这个边界漏了会越界
- **`smallest = i` + 两次比较**：最小堆下沉是"三选最小"框架，和最大堆的"三选最大"对称。`<` 严格比较，相等不换，尽量保持稳定性（虽然堆本身不稳定）
- **`left < n` 边界**：`n` 是当前堆大小，不是数组长度。这个判断不能漏——少了会越界访问已不属于堆的区段

### 2. heapq 库的原理与接口

Python 标准库 `heapq` 就是最小堆的封装，内部就是一个普通 list + `sift_up`/`sift_down`。掌握它的接口能避免重复造轮子：

| 函数 | 作用 | 复杂度 |
|---|---|---|
| `heapq.heappush(heap, item)` | 入堆 | $O(\log n)$ |
| `heapq.heappop(heap)` | 弹堆顶 | $O(\log n)$ |
| `heapq.heapify(list)` | 原地建堆（Floyd 法） | $O(n)$ |
| `heapq.heapreplace(heap, item)` | 弹顶 + 入新（原子操作） | $O(\log n)$ |
| `heapq.heappushpop(heap, item)` | 入新 + 弹顶（原子操作） | $O(\log n)$ |
| `heapq.nlargest(k, iterable)` | 取前 K 大 | $O(n\log k)$ |
| `heapq.nsmallest(k, iterable)` | 取前 K 小 | $O(n\log k)$ |

**几个容易踩的细节**：

- **`heapreplace` vs `heappushpop`**：前者先弹后入（堆空会报错），后者先入后弹（堆空不报错，直接返回新元素）。Top-K 流式更新用 `heapreplace` 更直观——"弹掉门槛，放入新值"
- **`heapify` 是 $O(n)$ 不是 $O(n\log n)$**：第 08 章讲过 Floyd 建堆法的证明，这里直接受益
- **`nlargest`/`nsmallest` 内部优化**：当 $k$ 接近 $n$ 时，会退化为先排序再切片（这时排序更快）；当 $k \ll n$ 时，才用大小 K 的堆。库的实现会自动选优

**最大堆怎么办**？`heapq` 只有最小堆，求最大堆的常规技巧是"存负数"：

```python
import heapq
# 想用最大堆，存 -val
max_heap = []
heapq.heappush(max_heap, -5)
heapq.heappush(max_heap, -3)
heapq.heappush(max_heap, -8)
# 堆顶是 -8（最小），取负得 8（最大）
print(-heapq.heappop(max_heap))   # 8
```

双堆求中位数时，前半部分就要用"存负数"模拟最大堆。

### 3. Top-K 的两种实现对比

**实现一：全排序后切片**（朴素做法）

```python
def topk_sort(arr, k):
    return sorted(arr, reverse=True)[:k]   # O(n log n)
```

**实现二：大小 K 的最小堆**（堆做法）

```python
import heapq
def topk_heap(arr, k):
    if k >= len(arr):
        return sorted(arr, reverse=True)
    heap = arr[:k]
    heapq.heapify(heap)                    # O(k)
    for x in arr[k:]:
        if x > heap[0]:                    # 比门槛大
            heapq.heapreplace(heap, x)     # 弹门槛，入新值，O(log k)
    return sorted(heap, reverse=True)      # 最终 Top K 排序输出
```

**复杂度对比**：

| 做法 | 时间 | 空间 |
|---|---|---|
| 全排序切片 | $O(n\log n)$ | $O(n)$（sorted 拷贝） |
| 大小 K 堆 | $O(n\log k)$ | $O(k)$ |

**关键观察**：当 $k \ll n$（如 100 万数据求 Top 10），$\log k \approx 3.3$，$\log n \approx 20$，堆做法快约 6 倍；空间从 $O(n)$ 降到 $O(k)$，对流式数据是硬需求——你不可能把一个无限流全装进内存再排序。

**为什么不直接用 `heapq.nlargest`**？生产里就用它。但面试手撕时，面试官要看你能写出 `heapify + heapreplace` 的循环结构，证明你懂堆的内部机制，而不只是会调库。

### 4. 双堆求流式数据中位数

中位数（Median）是"一半比它小、一半比它大"的中间值。流式数据求中位数是堆的经典应用——单堆做不了，要**两个堆**：

- **大顶堆 `left`**：存较小的一半，堆顶是这半的**最大值**
- **小顶堆 `right`**：存较大的一半，堆顶是这半的**最小值**
- **平衡条件**：`len(left) == len(right)` 或 `len(left) == len(right) + 1`

中位数就恒在两堆顶之间：元素总数奇数时是 `left` 堆顶，偶数时是两堆顶平均。

```python
import heapq
class MedianFinder:
    def __init__(self):
        self.left = []    # 大顶堆（存负数模拟）
        self.right = []   # 小顶堆

    def add(self, num):
        # 默认先入 left
        heapq.heappush(self.left, -num)
        # 把 left 堆顶挪到 right，保证 left 的最大 ≤ right 的最小
        heapq.heappush(self.right, -heapq.heappop(self.left))
        # 平衡大小：right 不能比 left 大
        if len(self.right) > len(self.left):
            heapq.heappush(self.left, -heapq.heappop(self.right))

    def median(self):
        if len(self.left) > len(self.right):
            return -self.left[0]
        return (-self.left[0] + self.right[0]) / 2
```

**逐行讲透**（双堆法的关键三步）：

- **先入 `left` 再挪到 `right`**：这步看似多余，实际是"保证新元素先经过 left 的最大值筛选"。等价于"先放到较小半，再把较小半的最大挪到较大半"——如果新值属于较大半，它会被 `left` 堆顶顶出来进入 `right`
- **平衡条件 `len(right) > len(left)` 时回挪**：保证 `left` 永远等于或多一个。这样奇数个时中位数就是 `left` 堆顶，无需判断总数奇偶
- **取负模拟大顶堆**：`heapq` 只有最小堆，存 `-num` 后，最小堆的堆顶 `-left[0]` 就是原数的最大值。**这个取负技巧是双堆法的标配**，必须熟练

**复杂度**：每次 `add` 是 $O(\log n)$（两次 push + 两次 pop，每次 $O(\log n)$），`median` 是 $O(1)$（直接读堆顶）。比"每次重排求中位数" $O(n\log n)$ 快几个数量级。

## 三、实践与面试：手撕结构 + 对数器 + 高频题

### 1. 手撕结构：Top-K 跟踪器

面试常考的"实时维护 Top K"——把上面的零散代码封装成一个可复用的类：

```python
import heapq
class TopKTracker:
    """实时维护数据流中的 Top K 大元素。"""
    def __init__(self, k):
        self.k = k
        self.heap = []    # 大小 K 的最小堆，堆顶是 Top K 中最小

    def add(self, num):
        if len(self.heap) < self.k:
            heapq.heappush(self.heap, num)
        elif num > self.heap[0]:
            heapq.heapreplace(self.heap, num)   # 替换门槛

    def topk(self):
        return sorted(self.heap, reverse=True)  # 输出时排一下
```

**面试时怎么讲**（按"技术的水有多深"八层面，挑重点）：

1. **问题层**：流式数据求 Top K，不能全排序（每秒新数据要重排，灾难）
2. **原理层**：维护大小 K 的最小堆，堆顶是"当前第 K 名"的门槛；新数据大于门槛就替换并下沉调整
3. **实现层**：`add` 用 `heappush`（堆未满）或 `heapreplace`（堆满且新值大于门槛），$O(\log k)$
4. **优劣层**：每次 $O(\log k)$，总共 $O(n\log k)$；空间 $O(k)$，对流式数据是硬需求
5. **演进层**：朴素做法（全排序 $O(n\log n)$）→ 部分排序（大小 K 堆 $O(n\log k)$）→ 库优化（`heapq.nlargest` 自动选优）

**面试加分点**：主动提"为什么用最小堆而不是最大堆"——求 Top K 大要用最小堆，因为堆顶要当"门槛"（第 K 名），新值只要和门槛比就行；如果用最大堆，堆顶是当前最大，反而没法当门槛。

### 2. 优先队列的四个工程应用

**应用一：Dijkstra 最短路**

Dijkstra 算法每次要"从未确定节点中选距离最小的"。朴素做法是线性扫描 $O(n)$，总 $O(n^2)$；用优先队列（最小堆）优化到 $O((n + m)\log n)$：

```python
import heapq
def dijkstra(graph, start):
    n = len(graph)
    dist = [float('inf')] * n
    dist[start] = 0
    pq = [(0, start)]    # (距离, 节点)，最小堆按距离排序
    while pq:
        d, u = heapq.heappop(pq)
        if d > dist[u]:    # 过期记录跳过
            continue
        for v, w in graph[u]:
            if dist[u] + w < dist[v]:
                dist[v] = dist[u] + w
                heapq.heappush(pq, (dist[v], v))
    return dist
```

**关键细节**：堆里存 `(距离, 节点)` 元组，`heapq` 按元组第一项比较——这就是"距离优先"的优先队列。**过期记录跳过**（`if d > dist[u]`）是 Dijkstra 的标配优化：一个节点可能被多次入堆，只处理第一次弹出的最小距离记录。

**应用二：A\* 寻路**

A* 是 Dijkstra 的"启发式增强版"——优先级从"已走距离 $g$"升级为"$f = g + h$"（$h$ 是到终点的估计距离）。优先队列同样按 $f$ 排序，每次取 $f$ 最小的扩展。**堆在 A\* 里的角色和 Dijkstra 完全一样**，只是优先级的定义不同。

**应用三：任务调度**

操作系统进程调度、定时任务系统（如 Python `sched`、Java `ScheduledThreadPoolExecutor`）都用优先队列——任务按"下次执行时间"或"优先级"排序，堆顶是下一个要执行的任务。

**应用四：合并 K 个有序链表**

K 个有序链表合并成一个，每次要"从 K 个头里选最小的"。用大小 K 的堆：

```python
import heapq
def merge_k_lists(lists):
    # lists 是 K 个有序链表的头节点
    pq = []
    for i, node in enumerate(lists):
        if node:
            heapq.heappush(pq, (node.val, i, node))   # (值, 序号, 节点)
    dummy = ListNode()
    tail = dummy
    while pq:
        val, i, node = heapq.heappop(pq)
        tail.next = node
        tail = tail.next
        if node.next:
            heapq.heappush(pq, (node.next.val, i, node.next))
    return dummy.next
```

**为什么堆里要存序号 `i`**：当两个节点值相等时，`heapq` 会比较元组第二项。如果第二项是节点对象，Python 会报错（节点无 `<` 比较）。存序号 `i` 是"打破平手"的标准技巧——既保证比较可进行，又稳定（同序号链表的节点按入堆顺序出堆）。

总复杂度 $O(N\log K)$，$N$ 是所有节点总数，$K$ 是链表数。朴素做法"每次线性扫 K 个头选最小"是 $O(NK)$，堆优化快 $\log K$ 倍。

### 3. 对数器：对比手写堆与 heapq

写完手撕的 `MinHeap`，怎么验证它对？用 `heapq` 当标准答案对比——`heapq` 是经过千万级生产验证的库，绝对正确。

```python
import random, heapq
# 复用前面的 MinHeap 类定义

def checker(test_times=5000, max_n=200, max_val=10000):
    """对比手写 MinHeap 与 heapq 的行为。"""
    for _ in range(test_times):
        n = random.randint(0, max_n)
        ops = []
        my_heap = MinHeap()
        std_heap = []
        for _ in range(n):
            if random.random() < 0.5 or not std_heap:
                val = random.randint(0, max_val)
                ops.append(('push', val))
                my_heap.push(val)
                heapq.heappush(std_heap, val)
            else:
                ops.append(('pop',))
                a = my_heap.pop()
                b = heapq.heappop(std_heap)
                if a != b:
                    print(f"出错！操作序列: {ops}, 手写: {a}, 标准: {b}")
                    return False
        # 全部 pop 对比
        while std_heap:
            a = my_heap.pop()
            b = heapq.heappop(std_heap)
            if a != b:
                print(f"清空时出错！手写: {a}, 标准: {b}")
                return False
    print("对数器验证通过！")
    return True

checker()
```

**对数器的两个设计要点**：

- **混合 push/pop 操作**：不是先全部 push 再全部 pop，而是随机穿插——能覆盖"堆在动态变化中的状态"，比静态测试更严格
- **保留操作序列**：出错时打印 `ops`，方便定位是哪一步导致分歧。调试手写堆时，"知道哪一步错"比"知道错了"重要 10 倍

**对高手的启发**：对数器思想不止用于堆——任何"有标准答案可对照"的数据结构都能用对数器。写完红黑树？对数器对比 `std::map`。写完哈希表？对数器对比 `dict`。这是"工程化验证"的思维，比"理论上推导正确"更可靠。

### 4. 教科书做法 vs 生产做法

| 场景 | 教科书做法 | 生产做法 | 原因 |
|---|---|---|---|
| Top-K | 手写小顶堆 | `heapq.nlargest/nsmallest` | 库已优化，自动选排序或堆 |
| 优先队列 | 手写 `MinHeap` 类 | `heapq` 或 `queue.PriorityQueue` | 库线程安全（`PriorityQueue`） |
| 流式中位数 | 手写双堆 | 两个 `heapq` | 取负模拟最大堆是标准技巧 |
| Dijkstra 优化 | 手写堆存 `(dist, node)` | `heapq` 直接存元组 | 元组按第一项比较，正好是距离 |
| 合并 K 链表 | 手写堆存节点 | `heapq` 存 `(val, i, node)` | 序号 `i` 打破平手 |

**生产里几乎不手写堆**——`heapq` 足够快、足够稳。但面试必须手撕，因为面试官要看你懂不懂堆的内部机制，而不是会不会调库。

### 5. 三个真实工程坑

**坑一：求 Top K 大用最大堆。**

```python
# 错误：求 Top 10 大，用最大堆
import heapq
max_heap = [-x for x in data]
heapq.heapify(max_heap)
top10 = [-heapq.heappop(max_heap) for _ in range(10)]   # 对，但每次都要全堆取负
```

症状：能算出结果，但 `heapify` 时要全量取负，且堆大小是 $n$ 而不是 $k$。根因：求 Top K 大应该用**最小堆**——堆顶当门槛，新值只和门槛比。修复：用最小堆维护大小 K，详见前面 `TopKTracker`。

**坑二：Dijkstra 不跳过过期记录。**

```python
# 错误：不判断 d > dist[u] 直接处理
while pq:
    d, u = heapq.heappop(pq)
    for v, w in graph[u]:        # 没跳过过期记录
        if dist[u] + w < dist[v]:
            ...
```

症状：算法仍然正确，但慢几倍——同一个节点可能被多次入堆（每次发现更短距离），不跳过会重复处理。根因：堆里可能有"已被更短距离覆盖"的旧记录。修复：弹出时判断 `if d > dist[u]: continue` 跳过过期记录。

**坑三：双堆求中位数没平衡大小。**

```python
# 错误：只保证 left 堆顶 ≤ right 堆顶，没平衡大小
def add_bad(self, num):
    heapq.heappush(self.left, -num)
    if self.left and self.right and -self.left[0] > self.right[0]:
        heapq.heappush(self.right, -heapq.heappop(self.left))
    # 忘了平衡大小，left 可能无限膨胀
```

症状：中位数算错。根因：只保证"left 最大 ≤ right 最小"不够，还要保证"两堆大小差 ≤ 1"——否则中位数不在堆顶之间。修复：每次 add 后检查大小，把多的那堆堆顶挪到少的那堆。

### 6. 面试高频题：三道必背

**题一：前 K 个高频元素**（LeetCode 347）

给一个数组，返回出现频率前 K 高的元素。

```python
import heapq
from collections import Counter
def top_k_frequent(nums, k):
    freq = Counter(nums)
    # 用 (频率, 元素) 的最小堆，维护大小 K
    heap = []
    for num, cnt in freq.items():
        heapq.heappush(heap, (cnt, num))
        if len(heap) > k:
            heapq.heappop(heap)
    return [num for cnt, num in heap]
```

**讲解要点**：①先用 `Counter` 统计频率 $O(n)$；②维护大小 K 的最小堆，堆顶是"第 K 高频率"，$O(n\log k)$。**关键**：堆里存 `(频率, 元素)`，频率在前——`heapq` 按第一项比较，正好按频率排序。

**题二：合并 K 个有序链表**（LeetCode 23）

前面"应用四"已给完整代码。**面试时怎么讲**：①朴素"两两合并"是 $O(KN)$；②堆优化 $O(N\log K)$；③堆里存 `(val, i, node)`，序号 `i` 是为了打破平手——同值节点比较时不会报错。

**题三：数据流中位数**（LeetCode 295）

前面"双堆法"已给完整代码。**面试时怎么讲**：①单堆做不了，要双堆——大顶堆存较小半、小顶堆存较大半；②平衡条件 `len(left) == len(right)` 或 `len(left) == len(right) + 1`；③取负模拟大顶堆；④每次 add $O(\log n)$，median $O(1)$。

## 四、速查与自测

### 速查表：堆作为数据结构的核心操作

| 操作 | 复杂度 | 说明 |
|---|---|---|
| `push`（插入） | $O(\log n)$ | 末尾插入 + 上浮 |
| `pop`（取顶） | $O(\log n)$ | 取顶 + 末尾填顶 + 下沉 |
| `peek`（查顶） | $O(1)$ | 直接读 `heap[0]` |
| `heapify`（建堆） | $O(n)$ | Floyd 法，从 `n//2-1` 倒着下沉 |
| Top-K 维护 | $O(n\log k)$ | 大小 K 的堆 + 单次 $O(\log k)$ 调整 |
| 双堆中位数 | add $O(\log n)$ / median $O(1)$ | 两堆顶之间恒是中位数 |

**heapq 接口速查**：

| 函数 | 用途 | 注意 |
|---|---|---|
| `heappush(h, x)` | 入堆 | 最小堆 |
| `heappop(h)` | 弹堆顶 | 堆空报 `IndexError` |
| `heapify(lst)` | 原地建堆 | $O(n)$，原地修改 |
| `heapreplace(h, x)` | 弹顶 + 入新 | 先弹后入，堆空报错 |
| `heappushpop(h, x)` | 入新 + 弹顶 | 先入后弹，堆空不报错 |
| `nlargest(k, it)` | 取前 K 大 | $k \ll n$ 用堆，$k \approx n$ 用排序 |
| `nsmallest(k, it)` | 取前 K 小 | 同上 |

### 自测三问

**问题一：** 求 Top K 大为什么用最小堆而不是最大堆？

**参考答案：** 求 Top K 大，要找的是"前 K 名"——堆顶需要当"门槛"（第 K 名），新值只要和门槛比就行。最小堆的堆顶是当前堆中最小值，正好是"Top K 中最差的那一个"，作为门槛天然合适：新值大于门槛就替换，否则丢弃。如果用最大堆，堆顶是当前最大值，没法当门槛——你不知道新值是否进得了 Top K，因为门槛应该是第 K 名而不是第 1 名。**关键洞察**：求 Top K 大用最小堆，求 Top K 小用最大堆——堆顶永远是"门槛"，方向反过来。

**问题二：** Dijkstra 用优先队列优化时，为什么要判断 `if d > dist[u]: continue`？

**参考答案：** Dijkstra 中一个节点可能被多次入堆——每次发现更短距离就 push 一次。堆里可能同时存在多个 `(距离, 节点)` 记录指向同一节点，只有第一次弹出的是"最短距离"，后续的都是"过期记录"（已被更短距离覆盖）。如果不跳过，会重复处理同一节点，浪费时间——算法仍正确（因为 `dist[u] + w < dist[v]` 会过滤掉过期记录的松弛），但慢几倍。跳过的判断 `if d > dist[u]` 表示"弹出的距离比已记录的大，是过期记录，丢弃"。这是 Dijkstra 的标配优化，避免重复处理造成常数因子膨胀。

**问题三：** 双堆求流式中位数时，为什么"先入 left 再挪到 right"？

**参考答案：** 直接判断"新值属于左半还是右半"很麻烦——要和两堆顶比较，逻辑复杂且易错。标准技巧是"先入 left 再挪到 right"：①先把新值放入 `left`（大顶堆，存较小半）；②把 `left` 堆顶（这半的最大值）挪到 `right`——这一步保证了"left 的最大 ≤ right 的最小"，因为新加入的元素如果属于较大半，它会被 `left` 堆顶顶出来进入 `right`；③平衡大小——如果 `right` 比 `left` 多，把 `right` 堆顶挪回 `left`，保证 `left` 永远等于或多一个。这样最终 `left` 堆顶（较小半的最大值）就是中位数（奇数个）或中位数左半（偶数个取平均）。**这个"先入再挪"的技巧把"判断新值归属"简化为"机械的两步移动"**，是双堆法的标准写法。

### 算法思想 × 生活迁移

堆作为数据结构，核心思想是"**只维护最值，不全排序**"——动态场景下只付出"维护最值"的成本。

**迁移一：待办事项按优先级处理。** 早上列了 20 项待办，紧急程度各不相同。如果按"列出顺序"做（FIFO 队列），可能先做不重要的；如果每次重排所有待办，$O(n\log n)$ 太慢。优先队列做法——堆顶永远是最紧急的，做完一项 pop 一项 $O(\log n)$；新来一项紧急任务 push 进去 $O(\log n)$。这是 OS 进程调度、任务调度器的底层思路。

**迁移二：维护"前 N 个最好的客户/候选人"。** 客户管理里，你不可能把所有人排好序——人太多且动态变化。维护一个"Top 10 重要客户"的小顶堆，堆顶是"当前第 10 重要"的门槛，新接触一个客户评估后比门槛强就替换。这是 CRM 系统的简化模型——只关心 Top K，不关心全局排名。

**迁移三：实时监控中位数而非平均数。** 监控系统指标（响应时间、QPS）时，平均数会被极端值拉偏，中位数更抗异常。数据是流式的，不可能每次重排求中位数——用双堆，每次新数据 $O(\log n)$ 调整，中位数恒在堆顶之间 $O(1)$ 读取。这是 Prometheus、Grafana 等监控系统计算 P50（中位数）的常见思路。

**为什么这些迁移成立：** 堆成立的前提是"只需要最值或部分有序，不需要全排序"。这个前提在待办管理、Top K 维护、中位数监控里都成立——都是"动态场景、只需要最值或 Top K"。迁移成立的关键是"识别场景是动态的、只需要部分结果"——如果是静态全排序场景，用 `sorted()` 更直接；如果是动态部分有序场景，堆是唯一最优解。
