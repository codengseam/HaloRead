# 算法课·16｜递归与分治：从俄罗斯套娃到分而治之

> 前置知识：本专栏开篇（Polyya 四步法）；建议先有"函数调用栈"的直觉
> 学完你能：①讲清递归不是"自己调用自己"而是"问题同构自相似" ②默写递归三件套并逐行讲透 ③用 Master Theorem 和递归树分析任意分治复杂度 ④判断何时用递归何时用迭代 ⑤把"分而治之"思想迁移到工程和生活

## 一、问题与思路：递归不是"自己调用自己"

先抛一个反直觉的事实：一个函数在执行到一半时调用自己，听起来像在制造死循环，但它偏偏能正常返回。俄罗斯套娃就是这种结构——打开最大的那层，里面是一个一模一样、只是更小的套娃，再打开又是更小的，直到最里面那一层是实心的，打不开了，整个过程才开始反向收尾。

### 1. 生活场景：俄罗斯套娃

俄罗斯套娃（Matryoshka）是理解递归最好的实物。它有三个特征，恰好对应递归要做的三件事：

- **每一层结构和上一层完全相同**：都是"一个空心娃娃，里面装一个更小的"。你不需要为每一层发明新规则，同一个"打开"动作能套用到任意一层。这对应递归里"子问题和原问题同构"。
- **存在一个最小的、实心的核心**：最里面那层娃娃是实心的，打不开。这是整个过程的"终点"——没有它，你会无限往下找更小的娃娃。这对应递归里的"终止条件"（base case）。
- **从外往里打开，从里往外合上**：你先一层层打开到核心，再一层层把外层盖回去。打开是"递"，合上是"归"。函数调用栈正是这个结构——压栈是递，弹栈是归。

**用套娃理解递归的关键认知**：你不是在"让函数调用自己"，你是在"用同一个开套娃的规则，处理一层比一层小的套娃"。每次调用的对象都不同（更小的套娃），只是规则相同。这就是递归的本质——**问题的同构自相似**，而不是"自己调用自己"这个表面动作。

### 2. 递归：把问题归约到同构子问题

很多人对递归的第一印象是"自己调用自己"，这是个有害的简化。它把注意力引向了"调用动作"，却掩盖了真正起作用的东西。

来看阶乘：`fact(n) = n * fact(n-1)`。如果只盯着"自己调用自己"，你会困惑——`fact` 还没算完，怎么就能再调 `fact`？换一个视角：`fact(n)` 这个问题，和 `fact(n-1)` 这个问题**形状完全相同**（都是"算一个数的阶乘"），只是规模小了一点。你把"算 n 的阶乘"这个大问题，归约（reduce）成了"算 n-1 的阶乘"这个小问题，再乘上 n。小问题继续往下拆，拆到 `fact(0)=1`（实心核心，终止条件），整个链条就能收尾。

**所以递归做的是这件事：把一个问题，归约成一个同构的、规模更小的子问题，直到触底。** "自己调用自己"只是这个归约过程的代码表现，不是它真正的工作机理。理解这一点，你才不会把递归当成魔法，而是当成一种"问题归约"的思维方式。

### 3. 分治：子问题独立的递归

递归是一种通用的"归约"机制，可以表达很多种分解方式。分治（Divide and Conquer，分而治之）是递归的一种特定用法——它要求分解出的子问题**相互独立**。

"独立"是什么意思？看两个对比例子：

- **归并排序（分治）**：把数组从中间切开，左半边排序、右半边排序。左半边怎么排，完全不影响右半边——两者数据不重叠、互不依赖。两个子问题独立，可以分别递归求解，最后合并。这就是分治的本质——子问题独立。
- **斐波那契（不是分治，是重叠子问题）**：`fib(n) = fib(n-1) + fib(n-2)`。看起来也是"分成两个子问题"，但 `fib(n-1)` 内部又调了 `fib(n-2)`，两个子问题大量重叠。子问题不独立，所以它不是分治，是"带重叠的递归"——这正是动态规划要解决的场景。

**分治三步**：分解（Divide，把问题切成独立子问题）→ 解决（Conquer，递归求解每个子问题）→ 合并（Combine，把子问题的解拼成原问题的解）。这三步里，"独立"是分治能成立的前提，"合并"是分治区别于普通递归的关键——普通递归可能不合并（如阶乘只是乘一下），分治一定要有一个把子解拼起来的步骤。

### 4. 历史地位：分治是算法设计的支柱

分治不是某个具体算法，是一种**算法设计范式（paradigm）**。CLRS（《算法导论》）把分治放在第 4 章单独讲，因为它是后续大量经典算法的根基：

- 排序：归并排序（$O(n\log n)$）、快速排序（平均 $O(n\log n)$）都是分治。
- 选择：快速选择（quickselect，平均 $O(n)$ 找第 K 大）。
- 几何：最近点对问题（$O(n\log n)$）。
- 矩阵：Strassen 矩阵乘法（$O(n^{2.81})$ 突破朴素 $O(n^3)$）。
- 变换：快速傅里叶变换 FFT（$O(n\log n)$）。

这些算法表面毫无关系，底层却共享同一个"分解→解决→合并"的骨架。学会分治，你拿到的是一把能开多把锁的钥匙，而不是一个孤立算法。

## 二、原理与实现：递归三件套与 Master Theorem

### 1. 递归三件套模板：base case + 递归关系 + 返回值组装

写任何递归，都可以套用"三件套"框架。这是左神反复强调的写法纪律：

1. **终止条件（base case）**：什么时候不再递归？这是"实心核心"。漏掉它，递归就变成无限压栈直到溢出。
2. **递归关系（recursive relation）**：怎么把原问题归约成同构的、更小的子问题？这是"开套娃的动作"。
3. **返回值组装（return value assembly）**：拿到子问题的解后，怎么加工成原问题的解？这是"把外层盖回去"。

以归并排序为例，逐行讲透这三件套：

```python
def merge_sort(nums):
    # ① base case：长度 <= 1，已经有序，直接返回
    if len(nums) <= 1:
        return nums

    # ② 递归关系：从中间切开，左右两半各自排序
    mid = len(nums) // 2
    left = merge_sort(nums[:mid])      # 子问题：排左半
    right = merge_sort(nums[mid:])     # 子问题：排右半

    # ③ 返回值组装：合并两个有序数组
    return merge(left, right)

def merge(a, b):
    """合并两个有序数组，O(n)。"""
    result = []
    i = j = 0
    while i < len(a) and j < len(b):
        if a[i] <= b[j]:
            result.append(a[i]); i += 1
        else:
            result.append(b[j]); j += 1
    result.extend(a[i:])
    result.extend(b[j:])
    return result
```

**逐行讲透**：

- **`if len(nums) <= 1: return nums`**：base case。长度 0 或 1 的数组天然有序，是递归的"实心核心"。这一行绝不能漏——没有它，`merge_sort` 会对空数组或单元素数组继续切分，无限递归。判断"什么规模算触底"是写递归的第一步。
- **`mid = len(nums) // 2`**：分解点。从正中间切，保证左右两半规模都约为 $n/2$，这是后面能用 Master Theorem 算出 $O(n\log n)$ 的关键。如果切偏（比如总切成 1 和 n-1），复杂度会退化到 $O(n^2)$——快速排序最坏情况就是这么来的。
- **`left = merge_sort(nums[:mid])`**：把"排左半"当成一个同构子问题递归求解。注意传进去的是 `nums[:mid]`（一个更小的新数组），不是原数组——每次调用的对象确实不同、确实更小。
- **`right = merge_sort(nums[mid:])`**：右半同理。左右两个子问题数据不重叠，互相独立，这是分治成立的前提。
- **`return merge(left, right)`**：返回值组装。两个子问题各自给出一个"有序数组"，但原问题要的是"整体有序"，所以需要 `merge` 把两段拼起来。这一步是分治区别于普通递归的关键——`merge` 的 $O(n)$ 合并成本，正是归并排序 $O(n\log n)$ 中那个 $\log n$ 之外另一个 $n$ 的来源。
- **`merge` 函数**：双指针扫描两个有序数组，每次取较小者放入结果。$O(n)$ 时间、$O(n)$ 空间。这是"合并"步骤的标准实现。

**三件套检查纪律**：写完任何递归，回头逐条核对——base case 写了吗？递归关系保证子问题更小吗（否则不会触底）？返回值组装正确吗？三个问题答得清楚，递归就不会错。

### 2. 递归树分析法：把递归摊开成树

分析递归复杂度，最直观的工具是**递归树（recursion tree）**。把每次递归调用画成一个节点，子调用画成它的子节点，整棵递归树就摊开了所有工作量。

以归并排序 `T(n) = 2T(n/2) + O(n)` 为例：

```
                n              ← 第0层，合并成本 O(n)
              /    \
            n/2    n/2         ← 第1层，2 个节点，每个 O(n/2)，合计 O(n)
           /  \   /  \
        n/4 n/4 n/4 n/4       ← 第2层，4 个节点，每个 O(n/4)，合计 O(n)
            ...
        每层规模减半，节点数翻倍，每层合计 O(n)
        树高 log n 层（n → n/2 → ... → 1，要 log n 次除以 2）
```

**两层观察**：

- **每层总成本是 $O(n)$**：第 0 层 1 个节点成本 $n$，第 1 层 2 个节点每个 $n/2$ 合计 $n$，第 2 层 4 个节点每个 $n/4$ 合计 $n$……每层都是 $n$，因为节点数翻倍和节点规模减半正好抵消。
- **树高 $O(\log n)$**：每次规模减半，从 $n$ 减到 1 要 $\log_2 n$ 次。

总成本 = 每层成本 × 树高 = $O(n) \times O(\log n) = O(n\log n)$。这就是归并排序复杂度的来源——递归树把"递归"这个抽象过程，变成了"数树上有多少节点、每层多少工作量"的具体算术。

**对比一个"子问题不均匀"的例子**：快速排序最坏情况 $T(n) = T(n-1) + O(n)$（pivot 总选到最值，每次只切掉一个元素）。递归树长成一条链：

```
      n           ← 第0层，成本 O(n)
       \
       n-1        ← 第1层，成本 O(n-1)
         \
         n-2      ← 第2层，成本 O(n-2)
           \
           ...
           1      ← 第 n 层，成本 O(1)
```

这棵树退化成链表，树高从 $\log n$ 变成了 $n$。每层成本从 $n$ 递减到 1，总成本 $n + (n-1) + \dots + 1 = O(n^2)$。同样是递归树，归并那棵是"胖而矮"（每层 $n$、高 $\log n$），快排最坏这棵是"瘦而高"（每层递减、高 $n$）——形状决定复杂度。这也解释了为什么 Master Theorem 不覆盖这种情况：它的前提是子问题规模均匀（都是 $n/b$），而 $T(n-1)$ 的子问题规模是 $n-1$，不均匀，定理套不上。

递归树的好处是**直观**：你看得见每一层在干什么、总共有几层、树是胖是瘦。它的局限是**不精确**——当每层成本不是规整的 $O(n)$、或子问题大小不均匀时，求和要靠等差/等比数列技巧，容易算错。这时就需要 Master Theorem 把常见情况公式化。

### 3. Master Theorem：分治复杂度的公式

**主定理（Master Theorem）** 给出了一类分治递推的封闭解。形如：

$$T(n) = aT(n/b) + f(n), \quad a \geq 1, \ b > 1$$

其中 $a$ 是子问题个数，$n/b$ 是子问题规模，$f(n)$ 是分解与合并的成本。令 $c = \log_b a$（递归树每层节点数翻几倍的对数），比较 $f(n)$ 和 $n^c$：

| 情况 | 条件 | 结论 | 直觉 |
|---|---|---|---|
| 情况一 | $f(n) = O(n^{c-\varepsilon})$（合并成本远小于叶子层） | $T(n) = \Theta(n^c)$ | 工作量集中在递归树底层叶子，叶子有 $a^{\log_b n} = n^c$ 个 |
| 情况二 | $f(n) = \Theta(n^c)$（合并成本和叶子层相当） | $T(n) = \Theta(n^c \log n)$ | 每层成本相当，共 $\log n$ 层 |
| 情况三 | $f(n) = \Omega(n^{c+\varepsilon})$ 且满足正则条件（合并成本远大于叶子层） | $T(n) = \Theta(f(n))$ | 工作量集中在根节点那一层 |

**三个经典对号入座**：

- **归并排序** $T(n) = 2T(n/2) + O(n)$：$a=2, b=2, c=\log_2 2=1$，$f(n)=n=\Theta(n^1)$ → 情况二 → $O(n\log n)$。
- **二分查找** $T(n) = T(n/2) + O(1)$：$a=1, b=2, c=\log_2 1=0$，$f(n)=1=\Theta(n^0)$ → 情况二 → $O(\log n)$。
- **Strassen 矩阵乘法** $T(n) = 7T(n/2) + O(n^2)$：$a=7, b=2, c=\log_2 7 \approx 2.81$，$f(n)=n^2=O(n^{2.81-\varepsilon})$ → 情况一 → $O(n^{2.81})$，突破朴素 $O(n^3)$。

**Master Theorem 的价值**：它把"画递归树慢慢数"变成"代三个数对号入座"。面试时被问"你这个分治算法复杂度多少"，你能立刻说出 $a, b, c$ 并对号情况二，比现场画树快得多、也稳得多。

**它的边界**：Master Theorem 只适用于"子问题规模均匀（都是 $n/b$）"的分治。快速排序最坏情况 $T(n) = T(n-1) + O(n)$（每次只切掉一个元素）就不适用——子问题规模不均匀。这种情况得用递归树或代入法具体分析。

### 4. 经典案例：归并、快排、最近点对

三个案例展示分治的同一骨架如何落在不同问题上：

**归并排序**（已在上文逐行讲过）：分解=对半切，解决=递归排序两半，合并=双指针归并。$T(n)=2T(n/2)+O(n)=O(n\log n)$。特点是稳定、复杂度严格保证，但需要 $O(n)$ 额外空间。

**快速排序**：分解=选一个 pivot 把数组划成"小于 pivot"和"大于 pivot"两段，解决=递归排两段，合并=无需合并（划分时已就位）。平均 $T(n)=2T(n/2)+O(n)=O(n\log n)$，但最坏 $T(n)=T(n-1)+O(n)=O(n^2)$（pivot 总选到最值）。注意快排的"合并"成本是 0，工作量全在"分解"那一步的划分上——这是它和归并的关键差别。

**最近点对问题**：平面上 $n$ 个点，找距离最近的两个点。朴素做法两两比较 $O(n^2)$。分治做法：按 x 坐标排序后从中间切一刀，最近点对要么在左半、要么在右半、要么跨越中线。前两种递归求解，关键是第三种——跨越中线的点对。利用一个巧妙观察：只有距中线水平距离小于当前最小距离 $d$ 的点才可能更新答案，把这些点按 y 排序后，每个点只需和其后 7 个点比较。合并成本 $O(n)$，总复杂度 $T(n)=2T(n/2)+O(n)=O(n\log n)$。

**为什么是 7？** 这是最近点对最精巧的一步。设中线为竖线，取距中线水平距离 $< d$ 的点组成"中线条带"（宽 $2d$）。对条带里的点按 y 排序后，任取一点 $p$，能与 $p$ 形成更近点对的点 $q$ 必须落在以 $p$ 为底、高 $d$ 的 $2d \times d$ 矩形内。把这个矩形切成 8 个边长 $d/2$ 的小方格——每个小方格的对角线是 $d/\sqrt{2} < d$，而左右两半各自内部的点对距离都 $\ge d$（因为 $d$ 是两半各自的最小距离），所以每个小方格最多容纳 1 个点，整个矩形最多 8 个点（含 $p$）。也就是说 $p$ 之后最多还有 7 个点需要比较，再往后必然超过 $d$。每个点 $O(1)$，$n$ 个点合计 $O(n)$。鸽巢原理把"看起来要两两比"的合并步骤，压成了线性。

**三个案例的共同骨架**：分解 → 递归解决 → 合并。差别只在每一步的具体动作。这就是"范式"的力量——同一套框架，换不同的分解/合并策略，就能解决排序、选择、几何等完全不同领域的问题。

### 5. 栈溢出与深度限制：递归的代价

递归不是免费的。每次递归调用都会在调用栈（call stack）上压一帧，存参数、局部变量、返回地址。栈空间有限（Python 默认递归深度上限约 1000，C++ 默认栈大小几 MB），递归太深就会**栈溢出（stack overflow）**。

```python
import sys
print(sys.getrecursionlimit())   # 默认 1000

def sum_to(n):
    if n == 0:
        return 0
    return n + sum_to(n - 1)

sum_to(1000)   # RecursionError: maximum recursion depth exceeded
```

`sum_to(1000)` 在多数 Python 环境直接报错。这不是算法错，是递归太深。

**两种应对**：

1. **改成迭代**：用循环+显式栈替代递归调用。任何递归都能改写成迭代（因为递归本质是"用系统栈存状态"，迭代就是"用自己维护的栈存状态"）。`sum_to` 的迭代版就是一个 `for` 循环。
2. **尾递归优化（tail recursion optimization）**：如果递归调用是函数的最后一步动作（尾调用），编译器理论上可以复用当前栈帧，不压新栈。`sum_to` 改成尾递归形式：

```python
def sum_to_tail(n, acc=0):
    if n == 0:
        return acc
    return sum_to_tail(n - 1, acc + n)   # 递归调用是最后动作，acc 累加结果
```

**但要注意**：Python **不做**尾递归优化（Guido 明确反对，认为调试栈追踪更重要）。所以即使写成尾递归，Python 仍会栈溢出。C/C++（开启 `-O2`）、Scheme、Scala 等语言才支持。在 Python 里，深度大的递归只能改迭代。

**递归转迭代的通用手法：显式栈。** 简单的线性递归（如 `sum_to`）转成 `for` 循环即可；但树形递归（如遍历二叉树）不能简单循环，得用一个显式栈模拟调用栈。以二叉树前序遍历为例：

```python
# 递归版
def preorder_rec(root):
    if root is None:
        return []
    return [root.val] + preorder_rec(root.left) + preorder_rec(root.right)

# 迭代版：用显式栈模拟
def preorder_iter(root):
    if root is None:
        return []
    stack = [root]
    result = []
    while stack:
        node = stack.pop()
        result.append(node.val)
        if node.right:           # 先压右再压左，弹出时左先处理
            stack.append(node.right)
        if node.left:
            stack.append(node.left)
    return result
```

递归版靠系统栈存"还没处理的节点"，迭代版自己用一个 `stack` 列表干同样的事。两者逻辑等价，但迭代版的栈分配在堆上（不受调用栈深度限制），能处理几十万节点的树。**这就是"任何递归都能改迭代"的具体含义——把隐式的调用栈换成显式的数据结构栈**，状态从"系统替你管"变成"你自己管"。

## 三、实践与面试：手撕模板、性能对比、面试题

### 1. 面试手撕：递归三件套模板

面试中遇到递归题，先口述三件套再写代码，比直接闷头写稳得多。模板：

```python
def recursive_solve(problem):
    # ① base case
    if is_base_case(problem):
        return base_answer(problem)
    # ② 递归关系：拆成同构子问题
    sub_problems = decompose(problem)
    sub_answers = [recursive_solve(sp) for sp in sub_problems]
    # ③ 返回值组装
    return combine(sub_answers)
```

**面试时怎么讲**（按"技术的水有多深"框架）：

1. **问题层**：这题能不能归约成同构子问题？子问题独立吗（决定能不能用分治）？
2. **原理层**：base case 是什么？递归关系是什么？返回值怎么组装？
3. **复杂度层**：$a, b, f(n)$ 各是多少？Master Theorem 哪种情况？最坏如何？
4. **工程层**：递归深度多大？会不会栈溢出？要不要改迭代或加记忆化？

**面试加分点**：主动说"我先确认子问题是否独立——独立用分治，重叠用记忆化/DP"。这一句能立刻体现你分得清"分治"和"动态规划"的边界。

### 2. 递归 vs 迭代性能对比

递归优雅，但有函数调用开销和栈空间成本。用斐波那契数列做对比——它最能暴露递归的两种坑（重复计算、栈深度）：

```python
import time
import sys
sys.setrecursionlimit(100000)

# 版本一：朴素递归（指数级，重复计算严重）
def fib_rec(n):
    if n < 2:
        return n
    return fib_rec(n - 1) + fib_rec(n - 2)

# 版本二：递归 + 记忆化（O(n)，消除重复计算）
from functools import lru_cache
@lru_cache(maxsize=None)
def fib_memo(n):
    if n < 2:
        return n
    return fib_memo(n - 1) + fib_memo(n - 2)

# 版本三：迭代（O(n)，无栈开销）
def fib_iter(n):
    if n < 2:
        return n
    a, b = 0, 1
    for _ in range(n - 1):
        a, b = b, a + b
    return b

# 性能对比
for n in [30, 40]:
    t = time.perf_counter()
    fib_rec(n)
    print(f"朴素递归  fib({n}): {time.perf_counter()-t:.3f}s")

for n in [40, 1000, 10000]:
    t = time.perf_counter()
    fib_memo(n)
    print(f"记忆化递归 fib({n}): {time.perf_counter()-t:.6f}s")

    t = time.perf_counter()
    fib_iter(n)
    print(f"迭代      fib({n}): {time.perf_counter()-t:.6f}s")
```

**典型结果**（具体数值因机器而异，但量级关系稳定）：

| 调用 | 朴素递归 fib(30) | 朴素递归 fib(40) | 记忆化 fib(10000) | 迭代 fib(10000) |
|---|---|---|---|---|
| 耗时 | ~0.3s | ~30s（指数爆炸） | ~5ms | ~2ms |
| 是否栈溢出 | 否 | 否 | 是（n=10000 超 1000 上限） | 否 |

**三个结论**：

1. **朴素递归的指数爆炸来自重叠子问题**：`fib(40)` 朴素递归要算约 $2^{40}$ 次（实际约 $3\times10^8$ 次），记忆化后只算 40 次。这印证了"斐波那契不是分治"——子问题重叠，必须用记忆化或 DP。
2. **递归的栈深度是硬约束**：记忆化虽然 $O(n)$ 时间，但 `fib(10000)` 会递归 10000 层，Python 直接 `RecursionError`。迭代版没有这个问题——它用循环，不压栈。
3. **同等时间复杂度下，迭代通常更快**：记忆化和迭代都是 $O(n)$，但迭代省去了函数调用开销（压栈、弹栈、参数传递），常数更小。所以 `fib(10000)` 迭代约 2ms，记忆化约 5ms。

**选型纪律**：

- 逻辑天然递归（树遍历、分治）、深度可控 → 用递归，代码清晰。
- 深度可能很大（> 几千）→ 改迭代，避免栈溢出。
- 子问题重叠 → 加记忆化或直接上 DP，别用朴素递归。

### 3. 教科书做法 vs 生产做法

| 场景 | 教科书做法 | 生产做法 | 原因 |
|---|---|---|---|
| 排序 | 手写归并（递归） | `sorted()`（Timsort，迭代+归并混合） | Timsort 对真实数据（部分有序）更优，且无递归栈风险 |
| 树遍历 | 手写递归前/中/后序 | 框架已封装（如 ORM 的树查询） | 库已优化，深度由数据决定时库会处理 |
| 大规模递归计算 | 手写递归 | 改迭代或显式栈 | 避免栈溢出，可控内存 |
| 需要并行 | 递归分治 | 任务并行框架 | 分治的独立子问题可分到多核 |

**生产里递归不是默认选项**：能迭代就迭代，递归只用在"逻辑确实递归、深度确实可控"的场景。但理解递归是理解分治、DP、回溯的前提——这些范式的思想都用递归表达，不学递归就接不上后续。

### 4. 三个真实工程坑

**坑一：递归处理大树，栈溢出。**

```python
def tree_sum(node):
    if node is None:
        return 0
    return node.val + tree_sum(node.left) + tree_sum(node.right)

# 一棵退化成链表的树（10 万节点），tree_sum 直接栈溢出
```

症状：`RecursionError`。根因：树退化成链表时递归深度等于节点数。修复：改用显式栈的迭代版（用栈模拟中序遍历），或对超大树用 Morris 遍历（$O(1)$ 空间）。

**坑二：朴素递归算斐波那契，指数级慢。**

```python
# 同事写的"简洁"版本
def fib(n):
    return n if n < 2 else fib(n-1) + fib(n-2)
# n=40 要跑几十秒，n=50 直接卡死
```

症状：小数据正常，数据稍大就极慢。根因：重叠子问题导致 $O(2^n)$ 重复计算。修复：加 `lru_cache` 记忆化，或改迭代。教训：看到"递归里同一参数被多次计算"，立刻想到记忆化。

**坑三：以为写成尾递归 Python 就能优化。**

```python
def fact_tail(n, acc=1):
    if n == 0:
        return acc
    return fact_tail(n - 1, acc * n)   # 尾递归形式
# fact_tail(1000) 在 Python 仍栈溢出
```

症状：改了尾递归还是溢出。根因：Python 不做尾递归优化（语言设计决策，非 bug）。修复：Python 里深度大的递归一律改迭代，别指望尾递归。C++/Scheme 才吃这一套。

### 5. 面试高频题

**面试题一**："手写归并排序，分析复杂度，说明和快排的区别。"

满分回答：默写上面的 `merge_sort`，口述三件套（base case 是 `len<=1`，递归关系是对半切，组装是 `merge`）。复杂度用 Master Theorem：$a=2, b=2, f(n)=O(n)$，情况二，$O(n\log n)$。和快排的区别：归并的合并成本在 `merge`（$O(n)$ 额外空间），快排的合并成本是 0（划分时已就位）；归并最坏也是 $O(n\log n)$，快排最坏 $O(n^2)$（pivot 退化）；归并稳定，快排不稳定。

**面试题二**："斐波那契第 N 项，要求 $O(\log n)$。"

加分回答：朴素 $O(2^n)$，记忆化/迭代 $O(n)$，但还能更快——用矩阵快速幂。$\begin{pmatrix} F_{n+1} \\ F_n \end{pmatrix} = \begin{pmatrix} 1 & 1 \\ 1 & 0 \end{pmatrix}^n \begin{pmatrix} 1 \\ 0 \end{pmatrix}$，矩阵幂用快速幂（分治：$x^n = (x^{n/2})^2$），$O(\log n)$。这题考的是"分治思想能优化到多低"——不只是排序，连线性递推都能用分治加速。

**面试题三**："什么时候用递归，什么时候用迭代？"

回答框架：①逻辑天然递归（树、分治、回溯）且深度可控 → 递归，代码清晰易维护；②深度可能很大 → 迭代，避免栈溢出；③子问题重叠 → 记忆化或 DP，别用朴素递归；④需要尾递归优化 → 确认语言支持（Python 不支持）。判断标准是"递归深度是否可控"和"子问题是否重叠"。

## 四、速查与自测

### 速查表：递归与分治核心概念

| 概念 | 一句话 | 要点 |
|---|---|---|
| 递归 | 问题同构自相似 | 不是"自己调用自己"，是"归约到更小的同构子问题" |
| 分治 | 子问题独立的递归 | 三步：分解/解决/合并；独立是前提 |
| 递归三件套 | base case + 递归关系 + 返回值组装 | 写任何递归先口述这三件 |
| Master Theorem | $T(n)=aT(n/b)+f(n)$ | 三情况对号入座，看 $f(n)$ vs $n^{\log_b a}$ |
| 递归树 | 把递归摊成树数工作量 | 直观但不精确，复杂情况用主定理 |

**Master Theorem 速查**：

| 算法 | 递推 | $a$ | $b$ | $c=\log_b a$ | $f(n)$ | 情况 | 结果 |
|---|---|---|---|---|---|---|---|
| 归并排序 | $2T(n/2)+O(n)$ | 2 | 2 | 1 | $n$ | 二 | $O(n\log n)$ |
| 二分查找 | $T(n/2)+O(1)$ | 1 | 2 | 0 | $1$ | 二 | $O(\log n)$ |
| Strassen | $7T(n/2)+O(n^2)$ | 7 | 2 | 2.81 | $n^2$ | 一 | $O(n^{2.81})$ |
| 快排平均 | $2T(n/2)+O(n)$ | 2 | 2 | 1 | $n$ | 二 | $O(n\log n)$ |
| 最近点对 | $2T(n/2)+O(n)$ | 2 | 2 | 1 | $n$ | 二 | $O(n\log n)$ |

### 自测三问

**问题一：** 为什么说递归不是"自己调用自己"，而是"把问题归约到同构子问题"？请用俄罗斯套娃和阶乘说明。

**参考答案：** "自己调用自己"只描述了调用动作，没说出递归为什么能 work。俄罗斯套娃的每一次"打开"处理的是不同对象（更小的套娃），只是规则相同——这对应递归里"子问题和原问题同构但规模更小"。阶乘 `fact(n)=n*fact(n-1)` 把"算 n 的阶乘"归约成"算 n-1 的阶乘"，每次调用的对象都不同、都更小，直到 `fact(0)=1` 触底。所以递归的本质是"把问题归约到同构的更小子问题"，"自己调用自己"只是这个归约的代码表现。

**问题二：** 分治和带记忆化的递归（动态规划）有什么区别？斐波那契为什么不是分治？

**参考答案：** 分治要求子问题相互独立——归并排序的左右两半数据不重叠、互不依赖，可以分别求解再合并。斐波那契 `fib(n)=fib(n-1)+fib(n-2)` 看起来也是分成两个子问题，但 `fib(n-1)` 内部又调用 `fib(n-2)`，两个子问题大量重叠——子问题不独立，所以不是分治。重叠导致朴素递归 $O(2^n)$ 重复计算，必须用记忆化或 DP 消除重叠。判断标准就是"子问题是否独立"：独立→分治，重叠→DP。

**问题三：** 用 Master Theorem 分析 $T(n)=4T(n/2)+O(n)$，说明它属于哪种情况、复杂度是多少。

**参考答案：** $a=4, b=2, c=\log_2 4=2$，$f(n)=n=O(n^{2-\varepsilon})$（$n$ 远小于 $n^2$）。这是情况一——合并成本远小于叶子层。结论 $T(n)=\Theta(n^c)=\Theta(n^2)$。直觉：递归树有 $4^{\log_2 n}=n^2$ 个叶子，合并成本每层 $O(n)$ 共 $O(n\log n)$，叶子成本 $O(n^2)$ 主导，所以总量 $O(n^2)$。这个例子说明：子问题个数多（$a=4$）会让叶子层成本暴涨，即使合并很便宜，总成本仍由叶子决定。

### 算法思想 × 生活迁移

分治的思想是"把大问题切成独立的同构小问题，分别解决再合并"。这个骨架在算法之外同样成立。

**迁移一：用分治读一本厚书。** 一本 800 页的技术书，别从头读到尾（那是 $O(n)$ 顺序扫，容易读到后面忘前面）。先看目录把全书切成几大块（分解），每块独立理解（解决），再把各块的主线串成全书脉络（合并）。因为各章相对独立，分块阅读不会互相干扰——这正是分治"子问题独立"的前提。

**迁移二：用分治组织一场大型会议。** 200 人的年会，一个人统筹是 $O(n)$ 痛点。按模块切：节目组、后勤组、宣传组（分解，各模块工作不重叠=独立），各组分头推进（解决），最后汇总成整体流程（合并）。能分治的前提是模块独立——如果节目依赖后勤、后勤依赖宣传，就退化成有依赖的调度，得用拓扑排序而非分治。

**迁移三：用分治排查线上故障。** 全链路慢，别逐个接口查。先用二分思维把链路切成两段，定位慢的在哪段（分解），对慢的那段再切（递归解决），直到定位到具体接口（base case）。这其实就是二分查找——分治的最简形式。前提是"慢的根源能被切分定位"，如果故障是全局性的（如数据库整体宕机），切分就没用。

**为什么这些迁移成立：** 分治成立的前提是"问题能切成独立的同构子问题"。读书、办会、排障都满足——各部分相对独立、结构和整体相似。迁移失败的情况是子问题不独立（有依赖）或不可同构（各部分性质完全不同）。判断"能不能用分治"，先问"子问题独立吗、和原问题同构吗"。
