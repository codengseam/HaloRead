# 算法课·07｜归并排序：分治的另一种姿态，稳定的保证

> 前置知识：学完第 06 章（快速排序），理解分治与"分区 vs 合并"的对称性
> 学完你能：①默写归并排序的合并算法 ②讲清归并为什么稳定、为什么最坏 $O(n\log n)$ ③理解归并的"难在合"与快排的"难在分"对称 ④知道归并在外部排序、链表排序、Timsort 里的核心地位

## 一、问题与思路：分治的另一种切入——"难在合"

### 1. 一个生活场景：合并两叠已排好的试卷

期末考完，两个课代表各抱一叠按学号排好的试卷来办公室，要把两叠合成一叠，仍按学号有序。

**笨办法（插入式）**：把第二叠的试卷一张张拿出来，往第一叠里插。每插一张都要从前往后找位置，最坏要翻遍整个第一叠。两叠各 50 张，最坏要 2500 次比较。

**聪明办法（归并式）**：两叠都从头开始，各看最上面一张，学号小的先拿出来放到结果叠里，被拿走的那叠再翻到下一张。每比较一次必然产出一张试卷，两叠共 100 张，最多 99 次比较就合完。

**为什么比插入快那么多？** 插入是"每次拿一张去和已排好的全比对"，归并是"利用两叠都已有序这个前提，每次比较直接产出一张"。前提越强，效率越高——这就是归并的核心思想：**先让子问题各自有序，再线性合并**。扑克牌玩家整理手牌时也是这个动作：把两摞已排好的牌头对头比较，小的先落到结果摞。

### 2. 归并与快排：分治的两面

上一章讲快排时说，分治有两种姿态：

- **快排：难在分**。分区是 $O(n)$ 的辛苦活，分完之后两边独立排好就结束，不需要合。
- **归并：难在合**。拆分是 $O(1)$ 的轻松活（直接切两半），但排完两半后要把两个有序数组合并成一个有序数组，合并是 $O(n)$ 的辛苦活。

为什么会有这两种姿态？因为"分"和"合"是对称的——你可以选择"分的时候辛苦"（快排），也可以选择"合的时候辛苦"（归并）。两种都能达到 $O(n\log n)$，但特性不同：

| 维度 | 快排（难在分） | 归并（难在合） |
|---|---|---|
| 分的代价 | $O(n)$（分区） | $O(1)$（切两半） |
| 合的代价 | $O(0)$（不需要合） | $O(n)$（合并有序数组） |
| 空间 | $O(\log n)$（原地） | $O(n)$（辅助数组） |
| 稳定 | 否 | 是 |
| 最坏 | $O(n^2)$ | $O(n\log n)$ |

**核心思想**：归并的价值不在"快"（快排平均更快），而在两个快排给不了的东西——**稳定**和**最坏 $O(n\log n)$ 保证**。这两个特性让归并成为"需要稳定"或"需要最坏保证"场景的唯一选择。

### 3. 归并的历史：比快排更早的分治

归并排序的思想其实比快排更早。1945 年，John von Neumann 在 EDVAC 报告里就描述了归并排序——这可能是第一个 $O(n\log n)$ 的排序算法。von Neumann 的动机是：早期计算机内存小，排序要分批读入内存、排好、再合并，归并天然适合这种"外部排序"场景。

所以归并的核心思想还有一个维度：**它是为"数据放不进内存"的场景设计的**。这个场景今天仍然存在——大数据排序、数据库外部排序、MapReduce 的 sort 阶段，底层都是归并。归并是"外部排序之王"。

### 4. 合并的本质：批量消除跨区间逆序对

归并的合并步骤是 $O(n)$ 的，但它"批量消除"了什么？

考虑合并两个有序数组 A 和 B（长度各 n/2）。A 内部无逆序对（已有序），B 内部无逆序对（已有序），但 A 和 B 之间存在逆序对——A 中比 B 某元素大的，构成逆序对。

合并时，`A[i] > B[j]` 意味着 `A[i..]` 都大于 `B[j]`，一次比较的结果让 `A[i]` 一次性"跳过"了 B 中剩余的所有元素——相当于一次操作消除了 `len(B)-j` 个逆序对（A[i] 和 B[j..] 的所有逆序对）。

**这就是归并能突破 $O(n^2)$ 的本质**：合并时，每次比较的结果批量消除跨区间逆序对。和快排的分区"批量消除跨区间逆序对"对称——快排在分的时候消除，归并在合的时候消除。殊途同归。

## 二、原理与实现：合并是归并的灵魂

### 1. 归并排序的实现

```python
def merge_sort(nums):
    if len(nums) <= 1:
        return nums
    mid = len(nums) // 2
    left = merge_sort(nums[:mid])   # 排左半
    right = merge_sort(nums[mid:])  # 排右半
    return merge(left, right)       # 合并两个有序数组

def merge(a, b):
    result, i, j = [], 0, 0
    while i < len(a) and j < len(b):
        if a[i] <= b[j]:            # 注意 <=，相等时左半优先，保证稳定
            result.append(a[i]); i += 1
        else:
            result.append(b[j]); j += 1
    result.extend(a[i:])            # 处理剩余
    result.extend(b[j:])
    return result
```

**逐行讲透**（左神风格，每一行都不能含糊）：

- **`if len(nums) <= 1: return nums`**：递归基线。`len == 0` 是空数组天然有序，`len == 1` 是单元素天然有序。这行不写会无限递归切到空数组，栈溢出。
- **`mid = len(nums) // 2`**：从中间切。归并永远平衡切分，这是它最坏仍 $O(n\log n)$ 的根本——递归树是完美平衡二叉树，深度恒为 $\lceil \log n \rceil$。
- **`left = merge_sort(nums[:mid])`**：递归排左半。注意切片产生新数组，这里先按教学版写，生产版要避免重复分配（见第三章坑一）。
- **`right = merge_sort(nums[mid:])`**：递归排右半。左右两半独立，互不依赖——这就是"分"的轻松。
- **`return merge(left, right)`**：合并两个已有序的半边。所有辛苦都集中在这一个 `merge` 里——这就是"难在合"。
- **`result, i, j = [], 0, 0`**：result 是输出数组，i 指向 a 当前待选，j 指向 b 当前待选。双指针从 0 开始。
- **`while i < len(a) and j < len(b):`**：两边的当前元素都还有，才能比较。任一边耗尽就退出循环，剩下的另一边直接 extend。
- **`if a[i] <= b[j]:`**：**用 `<=` 而非 `<`，这是归并能稳定的根本**。相等时取左半 a[i]，保证左半元素（原排在右半前）合后仍在前面。写成 `<` 就丢稳定性（见第三章坑二）。
- **`result.append(a[i]); i += 1`**：a[i] 较小（或相等时优先），产出 a[i]，i 后移。
- **`else: result.append(b[j]); j += 1`**：b[j] 较小，产出 b[j]，j 后移。
- **`result.extend(a[i:])`**：a 还有剩余，整体追加。退出 while 时必有一边已耗尽，另一边剩余元素本就有序，直接追加即可。
- **`result.extend(b[j:])`**：b 还有剩余，同上。

**i 和 j 的语义是关键**：始终指向"各自数组里下一个待选元素"。任一边耗尽，另一边剩余直接追加——因为剩余部分本就有序，无需再比较。

**用一个例子走一遍**。合并 `A = [1, 3, 5, 7]` 和 `B = [2, 4, 6, 8]`：

```
i=0, j=0: A[0]=1 <= B[0]=2, C=[1], i=1
i=1, j=0: A[1]=3 >  B[0]=2, C=[1,2], j=1
i=1, j=1: A[1]=3 <= B[1]=4, C=[1,2,3], i=2
i=2, j=1: A[2]=5 >  B[1]=4, C=[1,2,3,4], j=2
... 以此类推
```

每次比较，较小的进 C，对应指针后移。这就是合并。它的时间是 $O(|A| + |B|)$——每个元素进 C 一次。

**合并的稳定性**：关键在 `a[i] <= b[j]` 用 `<=` 而非 `<`。相等时取左半的，保证左半元素（原排在右半前）合后仍在前面。如果用 `<`，相等时取右半，稳定性就丢了。这是归并能稳定的根本原因——**合并时左半优先**。

### 2. 复杂度分析

**时间复杂度**：

$$T(n) = 2T(n/2) + O(n)$$

主定理：$T(n) = O(n\log n)$。无论数据如何分布，每次都切两半（不像快排可能切偏），所以**最好、平均、最坏都是 $O(n\log n)$**。

**空间复杂度**：$O(n)$。合并时需要辅助数组存放结果。注意：递归栈是 $O(\log n)$，但辅助数组是 $O(n)$，所以总 $O(n)$。

**为什么归并最坏也是 $O(n\log n)$**？因为它不依赖 pivot 选择——每次都从中间切，递归树是完美的平衡二叉树，深度恒为 $\log n$。这是归并对比快排的核心优势：**没有最坏情况陷阱**。

### 3. 五指标评价

| 指标 | 归并 | 说明 |
|---|---|---|
| 最好 | $O(n\log n)$ | 不自适应，已有序也要走完递归 |
| 平均 | $O(n\log n)$ | 总是平衡切分 |
| 最坏 | $O(n\log n)$ | 没有 $O(n^2)$ 陷阱 |
| 空间 | $O(n)$ | 辅助数组，是归并的"代价" |
| 稳定 | 是 | 合并时左半优先 |
| 自适应 | 否 | 不识别已有序 |
| 缓存 | 中 | 合并时双数组跳跃访问 |

### 4. 归并的"自适应版"：自然归并

标准归并不自适应——已有序数据也要走完 $\log n$ 层递归。**自然归并**（Natural Merge）是改进版：扫描数组找已有序的"run"（递增或递减子序列），然后合并相邻 run。

```
原始数组：[3, 1, 4, 1, 5, 9, 2, 6]
找 run：  [3], [1, 4], [1, 5, 9], [2, 6]
合并：    [1, 3, 4], [1, 2, 5, 6, 9]
再合并：  [1, 1, 2, 3, 4, 5, 6, 9]
```

对基本有序数据，自然归并的 run 很长，合并次数少，接近 $O(n)$。Timsort 就是"自然归并 + 插入排序"的混合——这是 Python/Java 默认排序的底层。

### 5. 归并 vs 快排：何时选谁

| 场景 | 归并 | 快排 |
|---|---|---|
| 需要稳定 | ✅ | ❌ |
| 需要最坏保证 | ✅ | ❌ |
| 内存敏感 | ❌（要 $O(n)$ 辅助） | ✅ |
| 通用排序（追求平均速度） | ❌（常数大） | ✅ |
| 链表排序 | ✅（链表合并 $O(1)$ 空间） | ❌（链表分区难） |
| 外部排序（数据 > 内存） | ✅ | ❌ |
| 大量重复元素 | 中 | 三路快排更好 |

**链表排序是归并的"主场"**。链表不支持随机访问，快排的分区（要交换）很难高效实现；但链表的合并只需改指针，$O(1)$ 空间。所以链表排序几乎都用归并。

## 三、实践与面试：归并的生产地位

### 1. 面试手撕模板（预分配辅助数组版）

面试中如果被要求手写归并，用"预分配辅助数组"版——既展示你知道"标准教学版每次切片会内存翻倍"的工程坑，又是最容易写对的生产级模板：

```python
def merge_sort(nums):
    """归并排序 - 预分配辅助数组版，面试手撕模板。"""
    aux = nums[:]   # 预分配 O(n) 辅助数组，避免递归里反复切片
    _merge_sort(nums, aux, 0, len(nums))
    return nums

def _merge_sort(nums, aux, lo, hi):
    if hi - lo <= 1:        # 单元素或空区间，天然有序
        return
    mid = (lo + hi) // 2
    _merge_sort(nums, aux, lo, mid)   # 排左半
    _merge_sort(nums, aux, mid, hi)   # 排右半
    _merge(nums, aux, lo, mid, hi)    # 合并

def _merge(nums, aux, lo, mid, hi):
    for k in range(lo, hi):           # 先把 [lo, hi) 拷到 aux
        aux[k] = nums[k]
    i, j = lo, mid
    for k in range(lo, hi):
        if i >= mid:                  # 左半耗尽，取右半
            nums[k] = aux[j]; j += 1
        elif j >= hi:                 # 右半耗尽，取左半
            nums[k] = aux[i]; i += 1
        elif aux[i] <= aux[j]:        # 相等取左半，保证稳定
            nums[k] = aux[i]; i += 1
        else:
            nums[k] = aux[j]; j += 1
    return nums
```

**面试时怎么讲**（按"技术的水有多深"八层面，挑重点讲）：

1. **问题层**（核心思想）：排序本质是消除逆序对。归并用分治——先让两半各自有序（子问题独立），再线性合并，合并时每次比较批量消除跨区间逆序对，突破 $O(n^2)$。
2. **原理层**：从中间切两半 → 递归排左右 → 双指针合并（`<=` 保稳定，剩余直接追加）。递归式 $T(n)=2T(n/2)+O(n)$，主定理 $O(n\log n)$。
3. **优劣层**：稳定、最坏 $O(n\log n)$、链表 $O(1)$ 空间、外部排序之王；但要 $O(n)$ 辅助空间、常数比快排大、不自适应。
4. **演进层**：von Neumann 1945 原始版 → 自然归并（找 run 自适应）→ Timsort（自然归并+插入排序，Python/Java 默认）→ 多路归并（外部排序、MapReduce sort）。

**面试加分点**：主动提"教学版每次 `nums[:mid]` 切片会内存翻倍，生产版预分配一个 aux 数组在原数组和 aux 之间切换"，并说出"归并是稳定排序里最坏 $O(n\log n)$ 的通用方案"——这能立刻区分你和"只会背模板"的候选人。再补一句"逆序对计数是归并的经典应用，把 merge 里 `aux[i] > aux[j]` 那一行加一个累加就行"，直接留下深度印象。

### 2. 对数器：左神风格的验证方法

左神反复强调：**写完排序不要靠肉眼检查，要用对数器验证**。对数器的思路是——生成随机数据，用你的归并和标准库排序（绝对正确）对比，跑足够多次，只要有一次不一致就说明你的归并有 bug。

```python
import random

def merge_sort(nums):
    """你的归并实现。"""
    aux = nums[:]
    _merge_sort(nums, aux, 0, len(nums))
    return nums

def _merge_sort(nums, aux, lo, hi):
    if hi - lo <= 1:
        return
    mid = (lo + hi) // 2
    _merge_sort(nums, aux, lo, mid)
    _merge_sort(nums, aux, mid, hi)
    _merge(nums, aux, lo, mid, hi)

def _merge(nums, aux, lo, mid, hi):
    for k in range(lo, hi):
        aux[k] = nums[k]
    i, j = lo, mid
    for k in range(lo, hi):
        if i >= mid:
            nums[k] = aux[j]; j += 1
        elif j >= hi:
            nums[k] = aux[i]; i += 1
        elif aux[i] <= aux[j]:
            nums[k] = aux[i]; i += 1
        else:
            nums[k] = aux[j]; j += 1

def checker(test_times=5000, max_n=100, max_val=1000):
    """对数器：跑 test_times 次，每次随机数据对比。"""
    for _ in range(test_times):
        n = random.randint(0, max_n)         # 随机长度（含 0 和 1）
        arr1 = [random.randint(0, max_val) for _ in range(n)]
        arr2 = arr1[:]                        # 深拷贝
        merge_sort(arr1)                      # 你的排序
        arr2.sort()                           # 标准库（绝对正确）
        if arr1 != arr2:
            print(f"出错！输入: {arr2}, 你的结果: {arr1}")
            return False
    print("对数器验证通过！")
    return True

checker()
```

**对数器的价值**：

- **覆盖边界**：随机长度包含 0 和 1（空数组和单元素数组），随机值包含重复值，能覆盖手写时容易遗漏的边界
- **不依赖人工**：跑 5000 次只要几秒，比手动构造测试用例全面得多
- **稳定性也能验**：把 `arr2.sort()` 换成"带原始下标的稳定排序"对比，还能验证你的归并是否真的稳定

**对高手的启发**：对数器思想不止用于排序——任何"有标准答案可对照"的算法都能用对数器。写完红黑树？对数器对比 `std::map`。写完哈希表？对数器对比 `dict`。这是"工程化验证"的思维，比"理论上推导正确"更可靠。

### 3. 教科书做法 vs 生产做法

| 场景 | 教科书做法 | 生产做法 | 原因 |
|---|---|---|---|
| 通用排序 | 手写归并 | `sorted()`（Timsort） | Timsort 是归并 + 插入的混合，更优 |
| 链表排序 | 手写归并 | `list.sort()` 内部或归并 | 链表归并 $O(1)$ 空间 |
| 外部排序 | 手写多路归并 | Spark/MapReduce sort | 分布式归并是工程问题 |
| 需要稳定的对象排序 | 手写归并 | `sorted()`（Timsort 稳定） | Timsort 已稳定 |
| 大数据 Top-K | 不用归并 | 堆 | 归并不适合 Top-K |

**归并在生产里"无处不在但隐身"**：Timsort 底层是归并、Java 对象排序底层是归并、数据库外部排序底层是归并、MapReduce sort 阶段底层是归并。你不直接写归并，但归并在所有这些场景里默默工作。

### 4. 三个真实工程坑

**坑一：归并的辅助数组导致内存翻倍。**

```python
# 错误：每次递归都新建辅助数组
def merge_sort_bad(nums):
    if len(nums) <= 1: return nums
    mid = len(nums) // 2
    left = merge_sort_bad(nums[:mid])    # 切片产生新数组
    right = merge_sort_bad(nums[mid:])   # 切片产生新数组
    return merge(left, right)            # merge 又产生新数组
```

症状：100 万数据排序，内存占用 4-5 倍。根因：每次递归都 `nums[:mid]` 切片，产生大量临时数组。修复：预分配一个 $O(n)$ 辅助数组，递归时在原数组和辅助数组之间切换（见上面手撕模板）。这样总空间是 $O(n)$（一个辅助数组），不是 $O(n \log n)$（每层都新建）。

**坑二：合并时 `<=` 写成 `<`，丢稳定性。**

```python
# 错误：用 < 导致相等时取右半，不稳定
def merge_unstable(a, b):
    result, i, j = [], 0, 0
    while i < len(a) and j < len(b):
        if a[i] < b[j]:   # 错！相等时取了 b，不稳定
            result.append(a[i]); i += 1
        else:
            result.append(b[j]); j += 1
    ...
```

症状：对象排序丢稳定性。根因：相等时取了右半，左半元素跑到右半后面。修复：永远用 `<=`，相等时取左半。

**坑三：用归并排小数据，在小数据上不如插入排序。**

```python
# 在 n=20 的小数组上用归并
small_data = list(range(20))
merge_sort(small_data)   # 慢

# 插入排序
insertion_sort(small_data)   # 快得多
```

症状：小数据上归并比插入慢几倍。根因：归并的递归开销（函数调用、辅助数组操作）在小数据上不划算。修复：小数据切到插入排序，这是 Timsort 的核心策略——n < 64 用插入，n >= 64 用归并。

### 5. 高手向：外部排序、Timsort、逆序对计数

归并的"高手向"有三个必须掌握的应用——它们都是"归并的合并步骤换个场景"的产物。

**应用一：外部归并排序（数据放不进内存）。**

```python
# 大文件排序：文件 100GB，内存只有 1GB
# 思路：分块读入内存排序，写回临时文件，再多路归并

import heapq

def external_sort(input_file, output_file, chunk_size=10**8):
    """外部归并排序：分块 + 多路归并。"""
    # 阶段 1：分块排序
    chunks = []
    with open(input_file) as f:
        chunk_idx = 0
        while True:
            chunk = []
            for _ in range(chunk_size):
                line = f.readline()
                if not line: break
                chunk.append(int(line))
            if not chunk: break
            chunk.sort()   # 内存内排序
            chunk_file = f"chunk_{chunk_idx}.tmp"
            with open(chunk_file, 'w') as cf:
                for x in chunk:
                    cf.write(f"{x}\n")
            chunks.append(chunk_file)
            chunk_idx += 1

    # 阶段 2：多路归并
    files = [open(c) for c in chunks]
    # 用堆做 K 路归并，每次取所有文件当前行的最小值
    heap = []
    for idx, f in enumerate(files):
        line = f.readline()
        if line:
            heapq.heappush(heap, (int(line), idx))

    with open(output_file, 'w') as out:
        while heap:
            val, idx = heapq.heappop(heap)
            out.write(f"{val}\n")
            line = files[idx].readline()
            if line:
                heapq.heappush(heap, (int(line), idx))

    for f in files:
        f.close()
```

这是归并的"主场"——数据放不进内存时，归并是唯一可行的方案。Spark、Hadoop 的 sort 阶段、数据库的 external sort，底层都是这个思路：**分块排序 + 多路归并**。快排不行（要随机访问整个数组），堆排也不行（要全局建堆），只有归并的"分批 + 合并"结构适合外部排序。

**应用二：Timsort = 自然归并 + 插入排序。**

Python 的 `sorted()` 和 `list.sort()`、Java 对象的 `Arrays.sort` 用的都是 Timsort（Tim Peters 2002）。它的核心是"按数据特征动态切换"：

1. **找 run**：扫描数组找已有序的子序列（递增或递减），递减的翻转成递增。最小 run 长度 32-64。
2. **小 run 用插入排序补齐**：run 太短就用插入排序补到最小长度（小数据插入更快）。
3. **合并 run**：用归并合并相邻 run，维护一个 run 栈，按"长度平衡"策略决定何时合并。

Timsort 的精妙在于"**利用数据本来的有序性**"——已有序数据几乎不动（找 run 是 $O(n)$），无序数据走归并（$O(n\log n)$）。这是"自然归并 + 插入排序"的混合，对真实数据（往往局部有序）极其高效。这也是为什么生产排序几乎不用纯归并——Timsort 是它的工程进化版。

**应用三：逆序对计数（归并的经典衍生题）。**

逆序对：数组中 `i < j` 但 `a[i] > a[j]` 的对数。暴力法 $O(n^2)$，归并可 $O(n\log n)$。

**关键洞察**：归并合并时，`aux[i] > aux[j]`（左半当前元素大于右半当前元素），意味着左半从 i 到 mid 的所有元素都大于 aux[j]（因为左半已有序）——一次性找出 `mid - i` 个逆序对。

```python
def count_inversions(nums):
    """逆序对计数：归并排序的衍生应用。"""
    aux = nums[:]
    return _count(nums, aux, 0, len(nums))

def _count(nums, aux, lo, hi):
    if hi - lo <= 1:
        return 0
    mid = (lo + hi) // 2
    cnt = _count(nums, aux, lo, mid) + _count(nums, aux, mid, hi)
    # 合并并统计跨区间逆序对
    for k in range(lo, hi):
        aux[k] = nums[k]
    i, j = lo, mid
    for k in range(lo, hi):
        if i >= mid:
            nums[k] = aux[j]; j += 1
        elif j >= hi:
            nums[k] = aux[i]; i += 1
        elif aux[i] <= aux[j]:
            nums[k] = aux[i]; i += 1
        else:
            nums[k] = aux[j]; j += 1
            cnt += mid - i    # 左半 [i, mid) 全部 > aux[j]，贡献 mid-i 个逆序对
    return cnt
```

**这一行 `cnt += mid - i` 是核心**——把 merge 里"左半当前比右半当前大"这一事件，从"产出 1 个结果元素"升级为"统计 mid-i 个逆序对"。归并的结构（已有序、双指针、跨区间比较）天然适合统计跨区间逆序对。

**面试常考**：LeetCode 上的"数组中的逆序对"（剑指 Offer 51）、"计算右侧小于当前元素的个数"（315）都是这个套路。会了归并，这些题就是"在 merge 里加一行计数"。

### 6. 面试高频题

**面试题一**："手写归并排序，说明时间复杂度，如何优化。"

**满分回答模板**：用上面的预分配辅助数组版，边写边讲：

1. "我先预分配一个 aux 数组，避免递归里反复切片导致内存翻倍。"
2. "递归：从中间切两半，左右各自归并，最后合并。基线是区间长度 <= 1。"
3. "合并：双指针扫描，`aux[i] <= aux[j]` 时取左半（保证稳定），剩余直接追加。"
4. "时间复杂度：最好、平均、最坏都是 $O(n\log n)$——总是平衡切分，递归深度 $\log n$。空间 $O(n)$ 辅助数组。"
5. "稳定——合并时左半优先。"
6. "优化方向：①预分配 aux ②小数据切插入排序（Timsort 策略）③自然归并（找 run 自适应）④链表版 $O(1)$ 空间。"

**面试官追问"归并和快排怎么选"**：

- 需要稳定 → 归并
- 需要最坏 $O(n\log n)$ → 归并（快排最坏 $O(n^2)$）
- 内存受限 → 快排（原地，归并要 $O(n)$ 辅助空间）
- 链表排序 → 归并（链表合并 $O(1)$ 空间，快排分区难）
- 外部排序（数据 > 内存）→ 归并（唯一可行）
- 通用场景 → 快排（缓存友好、常数小），但生产用 Timsort 或 introsort

**面试官追问"归并为什么稳定，关键在哪一行"**：关键在合并时的 `if aux[i] <= aux[j]`——用 `<=` 而非 `<`。相等时取左半的 aux[i]，保证左半元素（原排在右半前）合后仍在前面。如果写成 `<`，相等时会进入 else 取右半，左半元素跑到右半后面，稳定性丢失。所有稳定的归并实现都靠这个 `<=`。

## 四、速查与自测

### 速查表：归并排序五指标

| 指标 | 归并 | 说明 |
|---|---|---|
| 最好 | $O(n\log n)$ | 不自适应 |
| 平均 | $O(n\log n)$ | 总是平衡切分 |
| 最坏 | $O(n\log n)$ | **无 $O(n^2)$ 陷阱** |
| 空间 | $O(n)$ | 辅助数组（链表版 $O(1)$） |
| 稳定 | 是 | 合并时左半优先 |
| 自适应 | 否（自然归并可自适应） | 标准版不识别有序 |
| 缓存 | 中 | 合并时双数组访问 |

### 自测三问

**问题一：** 归并排序和快排都是分治，都是 $O(n\log n)$。它们的本质区别是什么？请用"难在分 vs 难在合"来解释，并说明这导致了哪些特性差异。

**参考答案：** 快排"难在分"——分区是 $O(n)$ 的辛苦活，要把小于 pivot 的挪到左边；分完后两边独立排好就结束，不需要合。归并"难在合"——拆分是 $O(1)$（直接切两半），但排完两半后要合并两个有序数组，合并是 $O(n)$。这个差别导致：①空间——快排原地（$O(\log n)$ 栈），归并要 $O(n)$ 辅助数组。②稳定——快排分区跨距离交换不稳定，归并合并时左半优先稳定。③最坏——快排可能切偏退化 $O(n^2)$，归并总是平衡切分最坏 $O(n\log n)$。两者殊途同归到 $O(n\log n)$，但姿态不同，特性不同。

**问题二：** 归并排序稳定，关键在哪一行代码？写错会怎样？

**参考答案：** 关键在合并时的 `if a[i] <= b[j]`——用 `<=` 而非 `<`。相等时取左半的 a[i]，保证左半元素（原排在右半前）合后仍在前面。如果写成 `<`，相等时会进入 else 取右半 b[j]，左半元素跑到右半后面，稳定性丢失。这一行代码是归并能稳定的根本——**左半优先**。所有稳定的归并实现都靠这个 `<=`。

**问题三：** 100GB 数据排序，内存只有 1GB，怎么排？为什么归并是唯一可行方案？

**参考答案：** 外部归并排序。①分块：每次读 1GB 进内存，用快速排序（或 Timsort）排好，写回临时文件。100GB 分成 100 个临时文件。②多路归并：用堆做 100 路归并，每次从 100 个文件各读一行，取最小值写输出。归并是唯一可行方案的原因：①分治天然适合"分批处理"——分块独立排序，不依赖全局信息。②合并是有序流的合并，$O(n)$ 时间且只需常量内存（每路一个 buffer）。快排不行——它要随机访问整个数组，100GB 数据放不进内存无法随机访问。堆排也不行——堆要全局建堆。只有归并的"分批 + 合并"结构适合外部排序。这是归并的"主场"。

### 算法思想 × 生活迁移

归并的核心思想是"**分批处理 + 合并**"——把大问题拆成可独立处理的小块，处理完再合并。

**迁移一：合并多个团队的工作成果。** 10 个团队并行做一个大项目，每个团队独立完成自己那部分（分块排序），最后把 10 份成果合并成最终产品（多路归并）。关键前提是"接口规范统一"——就像归并要求各块已有序，合并才能高效。如果各团队输出格式不统一，合并阶段要大量转换，成本爆炸。**"先规范接口再并行，最后归并"是大型协作的高效模式。**

**迁移二：整理分散的信息源。** 你有 5 个笔记 App 里的内容要合并成一份知识库。归并式做法：先把每个 App 的内容各自整理成有序结构（分块排序），再按主题合并（多路归并）。不要试图"一次性把所有内容混在一起整理"——那是 $O(n^2)$ 的两两比较。归并式"先各自有序再合并"高效得多。

**迁移三：Timsort 式"按规模切换策略"。** 不要执着于"一个方法解决所有问题"。小问题用简单方法（小数据用插入排序），中等问题用归并方法（数据量上来用归并），局部有序的问题利用已有结构（自然归并找 run）。项目管理也类似：小任务直接做（插入），常规任务走分治流程（归并），发现已有半成品就顺势利用（自然归并）。**"按数据/问题特征动态切换策略"比"坚持单一最优"更鲁棒。**

**为什么这些迁移成立：** 归并成立的前提是"问题可拆成独立子问题，且子问题的结果能高效合并"。这个前提在团队协作、信息整合里都成立——只要子问题独立、合并接口规范，归并式"分批 + 合并"就高效。迁移成立的关键是"子问题独立性"和"合并接口规范"——如果子问题之间耦合深（合并时要解决大量冲突），归并就退化成 $O(n^2)$。**判断问题能否归并式处理，看"子问题是否独立、合并是否低代价"。**
