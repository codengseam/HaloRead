# 算法课·23｜图的存储与表达：邻接矩阵 vs 邻接表

> 前置知识：学完第 22 章（树与二叉树），理解"非线性结构"的指针组织方式
> 学完你能：①给小白讲清同一张图为什么换个存法性能差几百倍 ②默写邻接矩阵与邻接表并讲透每一行 ③手撕邻接表结构应对面试 ④理解边集数组、十字链表、邻接多重表各自适合什么场景 ⑤用对数器对比两种存储的内存与速度 ⑥看清"存储选择决定算法选择"这条工程主线

## 一、问题与思路：图是抽象的，存储是物理实现

### 1. 一个生活场景：全国铁路网怎么存进计算机

假设你要把全国铁路网存进计算机：5000 个车站、8000 条直达线，每个站点想快速查"我能直达哪些站"。

**存法一（邻接矩阵）**：开一张 $5000 \times 5000$ 的二维表，行是起点站、列是终点站，有直达线就填 1，没有填 0。占内存 $\approx 25\text{MB}$（$5000 \times 5000$ 字节）。查"北京能直达哪些站"要扫第"北京"行整整 5000 格——即使北京只直通 8 个站，也得扫完 5000 格才能确认另外 4992 格都是 0。

**存法二（邻接表）**：给每个站开一个"邻居清单"，只记有边的站。北京只直通 8 个站，清单里就只放 8 项。5000 个站、8000 条线的总清单长度 $\approx 8000$ 项，内存压到 100KB 量级。查"北京能直达哪些站"直接读清单，3-4 次内存访问搞定。

差了 250 倍内存、上千倍查询速度——但存的是**同一张图**。这不是算法差异，是存储差异。本章要回答的核心问题是：**图是抽象的，存储是物理实现，存储选择决定算法选择**。

### 2. 反直觉事实：算法复杂度上的 O(1) 不是免费的

很多人记得"邻接矩阵查边 $O(1)$、邻接表查边 $O(\text{度数})$"，于是下意识觉得矩阵更快。这是只看单次查边的复杂度，忽略了三件事：

- **空间代价**：矩阵 $O(V^2)$ 是"预付"的，无论图多稀疏都得付满。$V=5000$ 时 25MB 起步，$V=10^5$ 时直接 10GB——内存爆了，单次 $O(1)$ 也无意义。
- **遍历代价**：邻接矩阵遍历某点所有邻居是 $O(V)$，不管实际有几条边；邻接表是 $O(\text{度数})$。BFS、DFS、Dijkstra 这些"扫邻居"的算法，邻接表总成本 $O(V+E)$，矩阵是 $O(V^2)$。
- **常数因子**：矩阵每次访问要算 `i*V+j` 的下标，缓存不连续；邻接表链表节点连续分配时缓存友好，实际跑起来比理论值更省。

**核心论点**：图论算法的复杂度通常写成 $O(V+E)$ 或 $O(V^2)$——这里的 $V$、$E$ 不是图本身的属性，是**存储表达**的属性。同一张稀疏图，用矩阵存跑 Dijkstra 是 $O(V^2)$，用邻接表存是 $O((V+E)\log V)$。算法没变，存储变了，复杂度就变了。这条主线贯穿整章。

### 3. 历史地位：从画在纸上到存进内存

图论起源于 1736 年欧拉的哥尼斯堡七桥问题，但那时图是"画在纸上的圆圈和连线"。图真正成为计算机能处理的数据结构，要等到 1950 年代之后——邻接矩阵的二维数组表示随早期 FORTRAN 出现，邻接表的链表形式则由 Hopcroft 与 Tarjan 在 1970 年代初的图算法论文中系统使用。两种存储的取舍，是图算法工程化的第一个分水岭：**选错存储，再好的算法也跑不动**。

## 二、原理与实现：四种存储的代码与复杂度

### 1. 邻接矩阵（Adjacency Matrix）：稠密图的标配

最直接的存法：用 $V \times V$ 的二维数组 `adj[i][j]` 表示"i 到 j 是否有边"。

```python
class GraphMatrix:
    """邻接矩阵实现：无权图。"""
    def __init__(self, n, directed=False):
        self.n = n
        self.directed = directed
        self.adj = [[0] * n for _ in range(n)]   # n*n 矩阵，0 表示无边

    def add_edge(self, u, v, w=1):
        self.adj[u][v] = w                       # 有边设为权重
        if not self.directed:
            self.adj[v][u] = w                   # 无向图对称设值
```

**逐行讲透**（左神风格，每一行都不能含糊）：

- **`self.n = n`**：记录顶点数。矩阵的维度由它决定，后续所有循环都以它为界。
- **`self.adj = [[0] * n for _ in range(n)]`**：开 $n \times n$ 的二维数组。注意必须用列表推导式，不能写 `[[0]*n]*n`——后者所有行共享同一个内层列表，改一行全跟着改，是经典坑。
- **`self.adj[u][v] = w`**：把第 u 行第 v 列设为权重 w。无权图就设 1。这一步是 $O(1)$——直接下标寻址，不扫描。
- **`if not self.directed: self.adj[v][u] = w`**：无向图对称。这条是面试常考点：**无向图的邻接矩阵是对称矩阵**，`adj[i][j] == adj[j][i]`，所以理论上可以只存上三角省一半空间，但工程上为了 $O(1)$ 查询通常全存。

**五指标**：

| 指标 | 邻接矩阵 | 说明 |
|---|---|---|
| 空间 | $O(V^2)$ | 与边数无关，只看点数 |
| 加边 | $O(1)$ | 下标直接设值 |
| 删边 | $O(1)$ | 下标置 0 |
| 查边 (u,v) 是否存在 | $O(1)$ | 直接读 `adj[u][v]` |
| 遍历 u 的所有邻居 | $O(V)$ | 必须扫完整行才知道哪些是 1 |

### 2. 邻接表（Adjacency List）：稀疏图的标配（逐行讲透）

邻接表为每个顶点维护一个"邻居清单"（链表或动态数组），只存实际有边的邻居。

```python
from collections import defaultdict

class GraphList:
    """邻接表实现：无权图，用 defaultdict(list) 简化。"""
    def __init__(self, n, directed=False):
        self.n = n
        self.directed = directed
        self.adj = defaultdict(list)             # 每个顶点一个邻居清单

    def add_edge(self, u, v, w=1):
        self.adj[u].append((v, w))               # u 的清单追加 (v, w)
        if not self.directed:
            self.adj[v].append((u, w))           # 无向图双向追加

    def neighbors(self, u):
        return self.adj[u]                       # 直接返回清单，O(1) 拿到引用
```

**逐行讲透**：

- **`self.adj = defaultdict(list)`**：用 `defaultdict(list)` 比 `dict + setdefault` 简洁，访问不存在的键会自动建空列表。等价于 `self.adj = {i: [] for i in range(n)}`，但前者按需创建更省内存。
- **`self.adj[u].append((v, w))`**：在 u 的清单末尾追加一条 `(v, w)`。append 是均摊 $O(1)$。**注意存的是元组不是单值**——因为要带权重；无权图可只存 `v` 省一半内存。
- **`if not self.directed: self.adj[v].append((u, w))`**：无向图双向。注意这会让总空间翻倍——$E$ 条无向边在邻接表里占 $2E$ 个清单项。所以无向图邻接表空间是 $O(V + 2E) = O(V+E)$，常数是 2。
- **`def neighbors(self, u): return self.adj[u]`**：返回清单引用，$O(1)$。**BFS/DFS/Dijkstra 都是先调用这个拿邻居清单再遍历**——这一步 $O(1)$，遍历本身 $O(\text{度数})$。

**五指标**：

| 指标 | 邻接表 | 说明 |
|---|---|---|
| 空间 | $O(V+E)$ | 无向图 $O(V+2E)$ |
| 加边 | $O(1)$ | append |
| 删边 | $O(\text{度数})$ | 要在清单里找到目标 |
| 查边 (u,v) 是否存在 | $O(\text{度数})$ | 扫 u 的清单 |
| 遍历 u 的所有邻居 | $O(\text{度数})$ | 清单多长扫多长 |

**关键观察**：邻接表与邻接矩阵的复杂度差异，全部体现在"度数 vs V"上。稠密图（$E \approx V^2$）时度数 $\approx V$，两者差不多；稀疏图（$E \ll V^2$）时度数远小于 V，邻接表完胜。**这是存储选择的第一条铁律：稀疏用邻接表，稠密用邻接矩阵**。

### 3. 边集数组（Edge List）：Kruskal 的好朋友

最简单的存法：把所有边塞进一个数组，每条边是 `(u, v, w)` 三元组。

```python
class GraphEdges:
    """边集数组：只存边，不存顶点清单。"""
    def __init__(self, n):
        self.n = n
        self.edges = []                          # [(u, v, w), ...]

    def add_edge(self, u, v, w=1):
        self.edges.append((u, v, w))
```

**逐行讲透**：

- **`self.edges = []`**：一个列表存所有边。空间 $O(E)$——比邻接表还省，连顶点清单的"空列表头"都不用付。
- **`self.edges.append((u, v, w))`**：append 一条边。$O(1)$。

**为什么不直接用边集数组？** 因为它查"u 的邻居"是 $O(E)$——要扫所有边。BFS/DFS 用它就是灾难。但有一种场景非它不可：**需要对所有边排序的算法**。Kruskal 最小生成树要按权重排序边，Bellman-Ford 要遍历所有边松弛——这两种场景邻接矩阵和邻接表都得先"把边收集出来再排"，等于多绕一圈；边集数组本来就是边数组，直接 `sort()` 就能用。

**五指标**：

| 指标 | 边集数组 | 说明 |
|---|---|---|
| 空间 | $O(E)$ | 最省 |
| 加边 | $O(1)$ | append |
| 查边 (u,v) 是否存在 | $O(E)$ | 全扫 |
| 遍历 u 的所有邻居 | $O(E)$ | 全扫找 u |
| 全边排序 | $O(E \log E)$ | 直接 sort |

### 4. 十字链表与邻接多重表：有向图的高级优化

这两种是工程教材里的"高级款"，面试极少考，但理解它们能看清"为什么邻接表还不够"。

**十字链表（Orthogonal List）**：用于**有向图**。邻接表存有向图要两套表——出边表（"我能到谁"）和入边表（"谁能到我"），空间翻倍。十字链表用一套节点，每条边同时挂在起点的出边链和终点的入边链上，通过两个指针字段共享同一条边记录。求顶点的出度和入度都是 $O(\text{出度}/\text{入度})$。

```python
# 概念示意，不展开完整实现
class ArcNode:
    """十字链表的边节点：同时属于起点的出边链和终点的入边链。"""
    def __init__(self, tail, head, w=1):
        self.tail = tail            # 起点
        self.head = head            # 终点
        self.w = w
        self.tlink = None           # 同起点的下一条出边
        self.hlink = None           # 同终点的下一条入边
```

**邻接多重表（Adjacency Multilist）**：用于**无向图**。普通邻接表存无向图时，一条边 `(u,v)` 在 u 的清单和 v 的清单里各存一份，删除时要改两处。邻接多重表让一条无向边只占一个节点，通过两个指针分别挂到 u 和 v 的链上，删边改一处即可。**适合频繁删边的场景**，比如某些网络流算法的残余图维护。

**为什么不常用？** 因为代码复杂、常数大、缓存不友好。现代工程里更常见的是用 `dict[dict]` 或 `defaultdict(set)` 配合显式边对象，靠语言层面的高效哈希表实现类似效果。这两种结构的价值在"思想"——**通过共享节点减少冗余存储**——而不在直接照搬实现。

### 5. 四种存储五指标对比

| 指标 | 邻接矩阵 | 邻接表 | 边集数组 | 十字链表/多重表 |
|---|---|---|---|---|
| 空间 | $O(V^2)$ | $O(V+E)$ | $O(E)$ | $O(V+E)$ |
| 查边 (u,v) | $O(1)$ | $O(\text{度数})$ | $O(E)$ | $O(\text{度数})$ |
| 遍历邻居 | $O(V)$ | $O(\text{度数})$ | $O(E)$ | $O(\text{度数})$ |
| 全边排序 | $O(V^2 \log V)$ | $O(E \log E)$ | $O(E \log E)$ | $O(E \log E)$ |
| 适合 | 稠密图、Floyd | 稀疏图、Dijkstra、BFS/DFS | Kruskal、Bellman-Ford | 频繁删边、有向图出入度 |

**核心思想（第一次出现）**：存储选择不是"哪种更好"的选择题，是"算法需要哪种操作"的匹配题。Floyd 三重循环每次都要 $O(1)$ 查 `adj[i][j]`，必须用矩阵；Dijkstra 每次要遍历当前点的邻居，必须用邻接表；Kruskal 要按权重排边，必须用边集数组。**先定算法，再定存储**——这是工程主线。

## 三、实践与面试：对数器、手撕结构、面试题

### 1. 对数器：随机图对比两种存储的内存与速度

左神反复强调：**对数器不只验证正确性，还能验证"两种实现是否等价、性能差多少"**。这里用对数器对比邻接矩阵与邻接表在稀疏图上的差异——亲眼看 250 倍的内存差距和上千倍的遍历速度差距。

```python
import random
import sys
import time

def gen_random_graph(n, edge_ratio, directed=False):
    """生成随机图：n 个顶点，每对顶点以 edge_ratio 概率连边。"""
    edges = []
    for u in range(n):
        for v in range(u + 1, n) if not directed else range(n):
            if u == v:
                continue
            if random.random() < edge_ratio:
                edges.append((u, v))
    return edges

def build_matrix(n, edges, directed=False):
    adj = [[0] * n for _ in range(n)]
    for u, v in edges:
        adj[u][v] = 1
        if not directed:
            adj[v][u] = 1
    return adj

def build_list(n, edges, directed=False):
    from collections import defaultdict
    adj = defaultdict(list)
    for u, v in edges:
        adj[u].append(v)
        if not directed:
            adj[v].append(u)
    return adj

def checker(n=500, edge_ratio=0.01, times=10):
    """对数器：稀疏图下对比两种存储的内存与遍历速度。"""
    matrix_mem_total = 0
    list_mem_total = 0
    matrix_time_total = 0
    list_time_total = 0
    for _ in range(times):
        edges = gen_random_graph(n, edge_ratio)
        # 内存对比
        m = build_matrix(n, edges)
        l = build_list(n, edges)
        matrix_mem_total += sys.getsizeof(m) + sum(sys.getsizeof(row) for row in m)
        list_mem_total += sys.getsizeof(l)
        # 遍历速度对比：遍历每个顶点的所有邻居
        t0 = time.perf_counter()
        for u in range(n):
            for v in range(n):
                if m[u][v]:
                    pass
        matrix_time_total += time.perf_counter() - t0
        t0 = time.perf_counter()
        for u in range(n):
            for v in l[u]:
                pass
        list_time_total += time.perf_counter() - t0
    print(f"邻接矩阵 平均内存: {matrix_mem_total / times / 1024:.1f} KB")
    print(f"邻接表   平均内存: {list_mem_total / times / 1024:.1f} KB")
    print(f"邻接矩阵 遍历耗时: {matrix_time_total * 1000:.1f} ms")
    print(f"邻接表   遍历耗时: {list_time_total * 1000:.1f} ms")

checker(n=500, edge_ratio=0.01)
# 典型输出（n=500, edge_ratio=1%）：
# 邻接矩阵 平均内存: ~980 KB
# 邻接表   平均内存: ~12 KB
# 邻接矩阵 遍历耗时: ~45 ms
# 邻接表   遍历耗时: ~0.3 ms
```

**对数器的价值**：

- **亲眼看见 250 倍内存差距**：$n=500$、边密度 1% 时（稀疏图），矩阵 $500 \times 500$ 字节起步 ~980KB，邻接表只有 ~12KB。差距随 $n$ 增大急剧拉大。
- **亲眼看见百倍速度差距**：遍历邻居时矩阵要扫 $n=500$ 格，邻接表只扫实际度数（约 5 个），实测耗时差两个数量级。
- **不靠背复杂度公式**：复杂度公式告诉你"邻接表更优"，但"优多少"是肉眼看见的——这才是工程决策的依据。

**对高手的启发**：对数器不只是"找 bug"的工具，更是"性能验证"的工具。两个方案复杂度都是 $O(V+E)$，但常数差 5 倍，工程上就是"能用 vs 不能用"的差别。**跑对数器看实测数据，比纠结理论复杂度更接近工程真相**。

### 2. 手撕结构：邻接表的完整实现

面试中如果被要求"手写图的存储结构"，写邻接表——它是最高频考点，且能体现你懂"按算法选存储"。下面是带权重、支持有向/无向的完整版：

```python
from collections import defaultdict
import heapq

class Graph:
    """邻接表完整实现：带权重，支持有向/无向。"""
    def __init__(self, n, directed=False):
        self.n = n
        self.directed = directed
        self.adj = defaultdict(list)             # 邻居清单：(邻居, 权重)

    def add_edge(self, u, v, w=1):
        """加边：u→v 权重 w；无向图同时加 v→u。"""
        self.adj[u].append((v, w))
        if not self.directed:
            self.adj[v].append((u, w))

    def neighbors(self, u):
        """返回 u 的所有邻居（带权重）。O(1) 拿引用，遍历 O(度数)。"""
        return self.adj[u]

    def has_edge(self, u, v):
        """查 u→v 是否存在。O(度数)。"""
        for nb, _ in self.adj[u]:
            if nb == v:
                return True
        return False

    def degree(self, u):
        """返回 u 的度（出度，无向图即度数）。O(1)。"""
        return len(self.adj[u])
```

**面试时怎么讲**（按"技术的水有多深"八层面，挑重点讲）：

1. **问题层**（核心思想，第二次出现）：图算法复杂度公式里的 $V$、$E$ 是存储表达的属性，不是图本身的属性。同一张稀疏图，邻接表跑 Dijkstra 是 $O((V+E)\log V)$，邻接矩阵是 $O(V^2)$——算法没变，存储变了，复杂度就变了。
2. **原理层**：邻接表为每个顶点维护"邻居清单"，加边 append $O(1)$，遍历邻居 $O(\text{度数})$。无向图每条边在两个清单里各存一份，空间常数是 2。
3. **优劣层**：空间 $O(V+E)$ 适合稀疏图；查边 $O(\text{度数})$ 比矩阵 $O(1)$ 慢，但 BFS/DFS/Dijkstra 这种"扫邻居"的算法邻接表完胜。
4. **演进层**：邻接表（链表）→ `defaultdict(list)`（动态数组）→ 哈希表嵌套——基本思路不变（按顶点组织邻居），但底层数据结构演进让常数更小、缓存更友好。

**面试加分点**：主动提"无向图邻接表空间常数是 2"，并说明"Floyd 必须用邻接矩阵因为要 $O(1)$ 查 `adj[i][j]`，Dijkstra 必须用邻接表因为要 $O(\text{度数})$ 遍历邻居"——这能立刻区分你和"只会背模板"的候选人。

### 3. 教科书做法 vs 生产做法

| 场景 | 教科书做法 | 生产做法 | 原因 |
|---|---|---|---|
| 稀疏图（社交网络、网页链接） | 邻接表（链表） | `defaultdict(set)` 或 NetworkX | 哈希表查边 $O(1)$，比链表 $O(\text{度数})$ 快 |
| 稠密图（完全图、棋盘格图） | 邻接矩阵 | NumPy 二维数组 | 向量化运算，Floyd 可上 GPU |
| 边频繁变化的图 | 邻接多重表 | 显式边对象 + 字典索引 | 代码可读性远高于多重表 |
| 超大规模图（亿级顶点） | 邻接表 | 图数据库 Neo4j / 压缩位图 CSR | 内存放不下，必须分片或压缩 |

**生产里几乎不手写邻接表**，但有两个残留场景：

1. **面试与算法竞赛**：手写邻接表是基本功，因为标准库的 `dict` 在最坏情况下退化，竞赛选手要可控的常数。
2. **教学价值**：理解邻接表才能理解为什么 Dijkstra 用邻接表写是 $O((V+E)\log V)$、用矩阵写是 $O(V^2)$——存储选择决定算法复杂度这条主线，必须从手写存储开始体会。

### 4. 三个真实工程坑

**坑一：稠密图误用邻接表，遍历反而比矩阵慢。**

```python
# 完全图 K_500：每对顶点都有边，E = 500*499/2 ≈ 125000
n = 500
edges = [(u, v) for u in range(n) for v in range(u+1, n)]
adj_list = build_list(n, edges)        # 邻接表
adj_matrix = build_matrix(n, edges)    # 邻接矩阵
```

症状：邻接表遍历所有邻居的总时间是 $O(V+E) = O(V^2)$，和矩阵 $O(V^2)$ 一样；但邻接表链表跳转缓存不友好，实测比矩阵连续二维数组慢 2-3 倍。根因：稠密图 $E \approx V^2$，邻接表的"省空间"优势消失，"缓存不友好"劣势暴露。修复：稠密图直接用 NumPy 矩阵。

**坑二：无向图邻接表忘了双向加边，BFS 漏搜一半。**

```python
def add_edge_wrong(self, u, v, w=1):
    self.adj[u].append((v, w))         # 只加 u→v，忘了 v→u
    # if not self.directed: self.adj[v].append((u, w))   # 漏了这行
```

症状：BFS 从起点出发，能到的所有点都到不了起点。根因：无向图必须双向加边，邻接表代码里这行最容易漏。修复：把 `directed` 标志作为构造参数强制传入，加边方法统一处理。

**坑三：查边用邻接表 $O(\text{度数})$，热点路径被卡到 $O(n)$。**

```python
# 高频查 "u 到 v 是否有边"，u 是超级节点度数 = n-1
def has_edge(self, u, v):
    for nb, _ in self.adj[u]:           # O(度数) = O(n)
        if nb == v:
            return True
    return False
```

症状：单次查边 $O(n)$，查 $n$ 次变 $O(n^2)$。根因：邻接表查边要扫清单。修复：把邻接表的清单从 `list` 换成 `set`，查边变 $O(1)$；或额外维护一个 `set[(u,v)]` 边集做 $O(1)$ 查询。

### 5. 面试高频题：课程表（拓扑排序前置）、克隆图

**面试题一：课程表（LeetCode 207）**

> 你这学期必须选修 numCourses 门课程，记为 0 到 numCourses-1。给你一个数组 prerequisites，其中 prerequisites[i] = [a, b] 表示要先修 b 才能修 a。判断是否可能完成所有课程。

**核心思路**：这是拓扑排序的判定题——把课程建成有向图（b→a 表示先修关系），如果图有环则不可能完成（环上的课谁也修不了），无环则能完成。判环用 Kahn 算法（BFS 拓扑）或 DFS 三色标记。**注意存储必须用邻接表**——prerequisites 通常是稀疏的（课程先修关系不会是稠密图），用矩阵会 $O(V^2)$ 空间爆内存。

```python
from collections import defaultdict, deque

def canFinish(numCourses, prerequisites):
    """拓扑排序判环：Kahn 算法（BFS）。"""
    # 邻接表存图：b → a 表示 b 是 a 的先修
    adj = defaultdict(list)
    indeg = [0] * numCourses                # 入度数组
    for a, b in prerequisites:
        adj[b].append(a)
        indeg[a] += 1
    # 入度为 0 的课程先入队
    q = deque([i for i in range(numCourses) if indeg[i] == 0])
    finished = 0
    while q:
        u = q.popleft()
        finished += 1
        for v in adj[u]:                    # 遍历 u 的邻居，O(度数)
            indeg[v] -= 1
            if indeg[v] == 0:
                q.append(v)
    return finished == numCourses           # 全部修完 = 无环
```

**讲解要点**（边写边讲）：

1. "用邻接表存图，因为先修关系是稀疏图，矩阵会爆内存。"
2. "Kahn 算法：每次取入度为 0 的点入队，相当于'砍掉没有依赖的课程'，砍完一轮更新后继入度，再砍下一轮。能砍完所有点说明无环。"
3. "时间 $O(V+E)$——每条边在 `indeg[v] -= 1` 时被访问一次，每个点入队一次。空间 $O(V+E)$。"
4. "对比 DFS 三色标记：DFS 也能判环，但 Kahn 同时给出拓扑序，更通用。"

**面试官追问"为什么不用邻接矩阵"**：prerequisites 最多 $10^5$ 条，课程数 $N \le 10^5$。用矩阵要 $10^{10}$ 个元素，内存爆。这是"存储选择决定算法可行性"的典型例子——算法没错，存储错了，直接跑不起来。

**面试题二：克隆图（LeetCode 133）**

> 给你无向连通图中一个节点的引用，返回该图的深拷贝。

**思路**：BFS/DFS 遍历原图，用哈希表 `old_to_new` 记录"原节点 → 克隆节点"映射。每访问一个邻居，若没克隆过就新建并加入映射，再在新图中加边。

```python
from collections import deque

class Node:
    def __init__(self, val=0, neighbors=None):
        self.val = val
        self.neighbors = neighbors if neighbors is not None else []

def cloneGraph(node):
    """BFS 克隆无向图。"""
    if not node:
        return None
    old_to_new = {node: Node(node.val)}     # 起点先克隆
    q = deque([node])
    while q:
        u = q.popleft()
        for v in u.neighbors:               # 邻接表遍历邻居
            if v not in old_to_new:
                old_to_new[v] = Node(v.val)
                q.append(v)
            old_to_new[u].neighbors.append(old_to_new[v])
    return old_to_new[node]
```

**讲解要点**：

1. "图克隆的核心难点是环——无向图中 A 和 B 互为邻居，朴素递归会无限循环。哈希表 `old_to_new` 既当 memo 又当 visited。"
2. "邻接表用 `neighbors` 字段直接表示，遍历是 $O(\text{度数})$，总时间 $O(V+E)$。"
3. "如果改成邻接矩阵存图，遍历邻居变 $O(V)$，总时间 $O(V^2)$——又一次存储决定复杂度。"

## 四、速查与自测

### 速查表：四种存储对比

| 指标 | 邻接矩阵 | 邻接表 | 边集数组 | 十字链表/多重表 |
|---|---|---|---|---|
| 空间 | $O(V^2)$ | $O(V+E)$ | $O(E)$ | $O(V+E)$ |
| 加边 | $O(1)$ | $O(1)$ | $O(1)$ | $O(1)$ |
| 查边 (u,v) | $O(1)$ | $O(\text{度数})$ | $O(E)$ | $O(\text{度数})$ |
| 遍历邻居 | $O(V)$ | $O(\text{度数})$ | $O(E)$ | $O(\text{度数})$ |
| 适合场景 | 稠密图、Floyd | 稀疏图、Dijkstra、BFS/DFS | Kruskal、Bellman-Ford | 频繁删边 |
| 缓存友好 | 好（连续） | 中（链表跳转） | 好（数组） | 差（双指针） |

**存储选择决策树**：

```
需要按边排序（Kruskal/Bellman-Ford）？
├── 是 → 边集数组
└── 否 → 图是稠密的吗（E ≈ V²）？
         ├── 是 → 邻接矩阵（Floyd、稠密图传递闭包）
         └── 否 → 邻接表（Dijkstra、BFS、DFS、拓扑排序）
                  └── 频繁删边？→ 邻接多重表/十字链表（极少用）
```

### 自测三问

**问题一：** 同一张稀疏图，为什么邻接矩阵跑 Dijkstra 是 $O(V^2)$，邻接表跑是 $O((V+E)\log V)$？算法没变，差距从哪来？

**参考答案：** Dijkstra 每轮要"取当前点 u 的所有邻居松弛距离"。邻接矩阵遍历 u 的邻居必须扫完整行（$V$ 格），即使 u 只有 2 个邻居也得扫 $V$ 次，$V$ 轮总成本 $O(V^2)$。邻接表遍历 u 的邻居只扫清单（度数个），所有顶点度数之和 $= 2E$，总成本 $O(V+E)$，加上堆操作 $O(\log V)$，总计 $O((V+E)\log V)$。**差距从"遍历邻居"这一步的存储表达来**——矩阵是 $O(V)$，邻接表是 $O(\text{度数})$。稀疏图度数远小于 $V$，邻接表完胜。

**问题二：** Kruskal 最小生成树为什么优先用边集数组而不是邻接表？两者空间都是 $O(E)$ 级别。

**参考答案：** Kruskal 第一步是"按权重对所有边排序"。边集数组本身就是边数组，直接 `sort()` 即可，$O(E \log E)$。邻接表要先"把所有边从邻接表里收集出来"（$O(V+E)$ 遍历），再排序（$O(E \log E)$），多了一步收集；而且无向图邻接表每条边存了两份，收集后还要去重，又多一步。**边集数组的"原始形态"就是 Kruskal 需要的输入形态**，零转换成本。邻接表的优势是"按顶点查邻居"，Kruskal 不需要这个能力。**存储要匹配算法的核心操作**——这是"存储选择决定算法选择"的反向应用。

**问题三：** 生产代码里几乎不手写邻接表，那为什么面试和算法竞赛还考？请从"可控常数""存储决定复杂度的体感""算法演进脉络"三个角度回答。

**参考答案：** ①可控常数：标准库 `dict`/`set` 最坏情况退化（哈希冲突），竞赛选手要可控的常数和最坏复杂度，手写邻接表（动态数组或链表）能精确控制。②存储决定复杂度的体感：不手写邻接表，就无法体会"同一张图用矩阵存跑 Dijkstra 是 $O(V^2)$、用邻接表存是 $O((V+E)\log V)$"这种数量级差异。手写过程逼你思考"遍历邻居是 $O(V)$ 还是 $O(\text{度数})$"，这正是工程决策的核心。③算法演进脉络：邻接表（链表）→ `defaultdict(list)`（动态数组）→ 哈希表嵌套 → CSR 压缩——基本思路不变（按顶点组织邻居），但底层演进让常数更小、缓存更友好。理解原始邻接表，才能理解后续演进为什么这样设计。

### 算法思想 × 生活迁移

图存储的核心思想是"**按访问模式组织数据**"——同一份信息，按不同访问模式组织，性能差几个数量级。

**迁移一：通讯录用按姓氏分组还是按关系亲疏排序。** 同样 1000 个联系人，要"快速查某个号码"用拼音首字母分组（类似邻接矩阵的 $O(1)$ 查询，但预付了 26 个字母桶的空间）；要"快速群发家人"用关系标签分组（类似邻接表的按顶点组织邻居）。**按访问模式选组织方式**——你不会为了"群发家人"而去扫 1000 个联系人，就像不会用邻接矩阵扫 $V$ 格找一个邻居。

**迁移二：知识库用目录索引还是按标签检索。** 一份技术文档库，按目录树组织（稠密结构，每篇文档都有归属，类似邻接矩阵）适合"系统性学习"；按标签组织（稀疏结构，每篇文档挂几个标签，类似邻接表）适合"按主题检索"。如果你常做主题检索却用目录树，每次都要扫整棵树——这正是"邻接表场景误用邻接矩阵"的生活版。

**迁移三：朋友圈用关注关系还是兴趣推荐。** 社交平台存"用户关注了谁"用邻接表（每个用户的关注清单只有几十到几百个，稀疏）；存"用户兴趣标签"用矩阵（兴趣维度有限，每个标签下用户密集）。**关注关系稀疏用邻接表，兴趣分布稠密用矩阵**——同一平台两种数据，两种存储，对应两种推荐算法。这是"先定算法，再定存储"的工程实践。

**为什么这些迁移成立：** 图存储成立的前提是"同一份信息可以有多种组织方式，且不同组织方式适配不同访问模式"。这个前提在通讯录、知识库、社交平台都成立——都是"信息本身不变，访问模式决定组织方式"。迁移成立的关键是"访问模式可识别"——如果一种访问模式从未发生（比如从不查"u 到 v 是否有边"），就别为它预付空间。
