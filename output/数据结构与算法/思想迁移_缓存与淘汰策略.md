# 算法课·36｜缓存与淘汰策略：从 LRU 到工作记忆

> 前置知识：学完第 16 章（哈希表进阶），懂 O(1) 查找的底层是哈希 + 数组
> 学完你能：①讲清 LRU/LFU/FIFO/ARC 四种淘汰策略的差异与适用场景 ②识别缓存抖动并知道怎么破 ③理解人脑工作记忆的容量限制和遗忘曲线是缓存淘汰的生物版 ④把"复习策略"翻译成缓存淘汰算法来优化自己的学习

## 一、问题与思路：缓存为什么必须淘汰，以及这为什么和你的脑子有关

### 1. 一个生活场景：书桌与书架

你打开浏览器第二次访问一个新闻网站，加载比第一次快得多——这是缓存。但同样是"第二次访问"，你想得起昨晚吃了什么，却想不起上周三中午吃了什么——这也是缓存。这两个看似无关的现象，背后跑的是同一套算法：缓存淘汰（cache eviction）。

先把场景具象化。你的书桌只能放下 7 本书，但你有 500 本书堆在书架上。你在写论文，需要不断翻阅不同的书。

- **书架** = 磁盘/主存：容量大（500 本），但取一本要起身走过去（慢）
- **书桌** = 缓存（cache）：容量小（7 本），但伸手就到（快）
- **问题**：第 8 本要用的书来了，书桌满了，拿走哪一本？

这个"拿走哪一本"的决定，就是**缓存淘汰策略**。选错了，你刚拿走的书马上又要用，得再起身去书架取——这就是缓存未命中（cache miss），代价昂贵。选对了，常用的书常驻书桌，少跑腿——这就是缓存命中（cache hit），省时省力。

计算机系统里这个问题无处不在：CPU 的 L1/L2/L3 缓存存的是内存里最热的指令和数据；浏览器缓存存的是最近访问过的网页资源；Redis 缓存存的是数据库里最热的查询结果。**所有缓存的物理现实都是同一个：容量远小于总数据量，必须淘汰。**

而且缓存是**多级嵌套**的，从快到慢、从小到大层层堆叠：

```
CPU 寄存器（B 级，ns）→ L1（KB，ns）→ L2（百 KB，十 ns）
→ L3（MB，十 ns）→ 内存（GB，百 ns）→ SSD（百 GB，μs）
→ 机械盘（TB，ms）→ 网络/远端（PB，十 ms+）
```

每一级都是下一级的缓存。L1 是 L2 的缓存，L2 是内存的缓存，内存是磁盘的缓存，磁盘是远端存储的缓存。**整个计算机系统就是一座层层叠加的"缓存金字塔"**，每一层都面对同样的矛盾：容量有限，必须决定保留什么、淘汰什么。这个矛盾从 CPU 内部一直贯穿到分布式存储——同一个算法问题，换了无数个名字重复出现。人脑也是这个结构：感觉记忆（毫秒级）→ 工作记忆（秒级）→ 长时记忆（年级）——同样是多级缓存。

### 2. 反直觉问题：遗忘不是 bug，是 feature

这里有个反直觉的事：**缓存"忘掉"数据，和你的大脑"忘掉"事情，不是缺陷，是策略。**

想象一个不会遗忘的大脑——你记得十年来每天中午吃过什么、每次排队时前后站着谁、每个陌生人的脸。这些信息涌进工作记忆，你会被淹没：真正要用的信息（今晚的约会、刚才老板交代的任务）被海量无用信息挤出去。神经科学里这叫"前摄干扰"（proactive interference）——旧记忆干扰新记忆。

所以遗忘是大脑的缓存淘汰。问题不是"要不要忘"，而是"忘掉哪些"。记不住昨晚吃了什么但记得十年前的婚礼——你的大脑在用什么淘汰策略？答案后面揭晓，先看计算机怎么解。

### 3. 缓存的根本矛盾与评价指标

缓存的根本矛盾：**容量有限 vs 访问无限**。一个 1GB 的 Redis 缓存，可能面对 100GB 的数据库；一个 32KB 的 L1 缓存，面对的是 GB 级的内存。容量永远不够装下所有数据，所以必须做选择。

衡量选择好坏的核心指标是**缓存命中率（hit rate）**：

$$\text{命中率} = \frac{\text{命中次数}}{\text{总访问次数}}$$

命中率越高，说明淘汰策略越"猜得准"——淘汰掉的都是真的不再用的，留下来的都被反复命中。命中率低意味着频繁换入换出，缓存形同虚设。

有个相关的反面现象叫**缓存抖动（thrashing）**：缓存容量略小于工作集（working set，即一段时间内活跃访问的数据集合），导致刚淘汰的马上又被访问，命中率断崖式下跌。后面会讲这是怎么发生的、怎么破。

### 4. 一个关键观察：淘汰策略的"预测"框架

所有淘汰策略都在做同一件事——**预测**：预测哪些数据未来不会再被访问，把它们淘汰掉。预测准，命中率高；预测错，命中率低。

但未来不可知（没有水晶球），所以所有策略都基于一个假设：**过去的行为能在一定程度上预测未来**。不同策略的差别，就是"用什么维度的过去"来预测：

- FIFO：用"进入缓存的顺序"——先进来的先走
- LRU：用"最近访问的时间"——最久没碰的先走
- LFU：用"访问的总次数"——碰得最少的先走
- ARC：用"上面几种的组合"——动态调整

**这个"用过去预测未来"的框架，正是后面迁移到人脑的桥梁**：人脑的遗忘曲线，预测的也是"哪些记忆未来用得到"。把这条线想通，你就明白为什么计算机的缓存策略能直接解释你的记忆力好坏。

## 二、原理与实现：四种淘汰策略的取舍

### 1. FIFO：最朴素的先进先出

FIFO（First In First Out）最简单：维护一个队列，新数据进队尾，满时淘汰队头。先进入缓存的先被淘汰，不管它后来有没有被访问过。

```python
from collections import OrderedDict

class FIFOCache:
    def __init__(self, capacity):
        self.cap = capacity
        self.od = OrderedDict()
    def get(self, key):
        if key not in self.od:
            return -1
        return self.od[key]              # FIFO 不调整顺序
    def put(self, key, val):
        if key in self.od:
            self.od[key] = val
            return
        if len(self.od) >= self.cap:
            self.od.popitem(last=False)  # 淘汰队头（最先进来的）
        self.od[key] = val
```

FIFO 的优点是公平、实现简单。缺点很明显：**它对"访问模式"完全无视**。一个被频繁访问的热点数据，只要它进缓存早，照样被淘汰——这违背了局部性原理。

那 FIFO 什么时候反而合适？当**访问模式本身就没有时间局部性**时——比如日志归档、消息队列的消费位点、流式处理的滑动窗口。这些场景里"先来先走"恰恰是对的：数据按时间顺序产生、按时间顺序消费，老的数据天然不会再被访问。这时上 LRU 反而是浪费：维护链表的代价换不回任何额外命中。**没有一种淘汰策略是绝对最优的，关键是访问模式与策略假设是否匹配**——这条原则贯穿所有缓存选型。

> **局部性原理（principle of locality）**：程序在一段时间内倾向于反复访问同一批数据（时间局部性），以及访问相邻地址的数据（空间局部性）。这是所有缓存策略的根基——没有局部性，缓存根本不成立。

### 2. LRU：最近最少使用

LRU（Least Recently Used）的假设最贴合局部性原理：**最近被访问过的数据，未来更可能再被访问；最久没被碰过的，最该淘汰。**

LRU 的经典实现是**哈希表 + 双向链表**，两个操作都 O(1)：

```python
class Node:
    def __init__(self, key=0, val=0):
        self.key, self.val = key, val
        self.prev = self.next = None

class LRUCache:
    def __init__(self, capacity):
        self.cap = capacity
        self.map = {}                    # key -> Node，O(1) 查找
        self.head = Node()               # 哨兵头（最近用过的）
        self.tail = Node()               # 哨兵尾（最久未用的）
        self.head.next = self.tail
        self.tail.prev = self.head
    def _remove(self, n):                # 从链表摘除节点
        n.prev.next, n.next.prev = n.next, n.prev
    def _add_head(self, n):              # 插到头部（标记为最近用过）
        n.next = self.head.next
        n.prev = self.head
        self.head.next.prev = n
        self.head.next = n
    def get(self, key):
        if key not in self.map:
            return -1
        n = self.map[key]
        self._remove(n); self._add_head(n)   # 访问即"刷新"到头部
        return n.val
    def put(self, key, val):
        if key in self.map:
            self.map[key].val = val
            self._remove(self.map[key]); self._add_head(self.map[key])
            return
        if len(self.map) >= self.cap:
            lru = self.tail.prev         # 链表尾部 = 最久未用
            self._remove(lru); del self.map[lru.key]
        n = Node(key, val)
        self.map[key] = n; self._add_head(n)
```

数据结构组合的分工很清晰：**哈希表负责 O(1) 定位**（key→节点），**双向链表负责 O(1) 调整顺序**（摘除+插入只需改四个指针）。两者合一，get 和 put 都是 O(1)。这是"用空间换时间"和"用结构换效率"的双重体现。

LRU 的"刷新"语义很关键：每次 get 都把节点挪到头部，相当于"刚用过，续命"。这和人脑的"复习一次就记得更久"高度对应——后面迁移篇会展开。

### 3. LFU：最不经常使用

LFU（Least Frequently Used）换了个维度：不看你"多久没碰"，看你"总共碰了几次"。访问次数最少的先淘汰。

LFU 的实现比 LRU 复杂，需要维护"频率→节点链表"的映射。难点在于**频率相同的节点之间怎么排序**——通常按时间，即同频率内再用 LRU。Redis 4.0 之后在 `maxmemory-policy allkeys-lfu` 里用的就是 LFU 的近似实现。

LFU 解决了 LRU 的一个问题：**偶尔被访问一次的冷数据不会因为"刚碰过"就续命**——它要看累计频率。但它有自己的坑——**老数据霸占**：一个历史高频、但现在已经不再用的数据，因为累计频率高，长期霸占缓存不被淘汰。这是后面"反例与边界"要讲的。

### 4. ARC：自适应缓存

前面三种策略各有适用场景，但**真实负载是变化的**——上午的访问模式可能偏 LRU 友好，下午偏 LFU 友好。固定策略顾此失彼。

ARC（Adaptive Replacement Cache，2003 年 Megiddo 和 Modha 在 FAST 会议上提出）的思路是**同时维护两个链表**：一个 LRU（T1，最近用过一次的）一个 LFU 的近似（T2，最近用过两次及以上的），并动态调整两者的容量分配。当发现 T1 命中多，就给 T1 多分容量；T2 命中多，就给 T2 多分。

更具体地，ARC 还维护了两个"影子链表" B1、B2，分别记录"刚从 T1 淘汰出去的"和"刚从 T2 淘汰出去的"。当一次访问命中 B1（说明 T1 淘汰错了，那数据本该留 LRU 侧），ARC 就把容量配额往 T1 倾斜；命中 B2 则往 T2 倾斜。**它用"淘汰后的反悔信号"来校准未来的淘汰决策**——这是一种带反馈的学习机制，而不是一次性写死的策略。

ARC 的价值不在某个固定公式，而在它的思想：**淘汰策略本身应该是自适应的**。这个思想后面迁移到人脑时会有呼应——人的"复习策略"也是动态调整的，越熟悉的内容复习间隔越长。

### 5. 缓存命中率与抖动：为什么会断崖式下跌

缓存抖动发生在**缓存容量略小于工作集**时。工作集是一段时间内活跃访问的数据集合。举个例子：

- 缓存容量 100，工作集 105（多 5 个）
- 工作集是循环访问的（A、B、C……循环）
- 每次访问某个数据时，它刚刚被淘汰出去——因为 105 > 100，循环到它时它一定不在缓存

这时命中率趋近 0，缓存不仅没帮忙，反而增加了换入换出的开销。这就是抖动（thrashing）。

破抖动的两条路：

1. **加大缓存**：让容量 ≥ 工作集。但缓存贵，不可能无限加。
2. **识别并缩小工作集**：把"扫描式一次性访问"和"真正的热点"区分开，让一次性访问绕过缓存（bypass），不挤占热点位置。这也是 ARC 等自适应策略的用武之地。

**工作集理论**：1968 年 Peter Denning 提出，程序的"工作集"是它在某段时间窗口 $\Delta$ 内访问的页面集合。只要缓存能装下工作集，命中率就高；一旦工作集超过缓存，抖动就来了。这个理论给了缓存调优一个可操作的标准——不是盲目加容量，而是先量出工作集有多大，再让缓存跟上。后面迁移到人脑时，你会发现"一段时间内能同时处理几件事"就是大脑的工作集，超过 7±2 就开始"抖动"（丢三落四）。

### 6. 写策略：除了"读"还有"写"

前面讲的淘汰策略都在回答"读 miss 时换掉谁"，但缓存还有一半场景是"写"。数据被改了，缓存和后端存储怎么同步？两种主流写策略：

- **写穿透（write-through）**：同时写缓存和后端存储。简单、强一致，但写延迟高（每次写都要等磁盘）。
- **写回（write-back）**：只写缓存，标记为"脏"（dirty），等该数据被淘汰时再写回后端。写延迟低，但崩溃时可能丢数据，且一致性弱。

写策略和淘汰策略是耦合的：写回模式下，淘汰一个脏数据要先写回磁盘再淘汰，开销比淘汰干净数据大。所以一些实现会优先淘汰干净数据，把脏数据留到不得不写回时——这又是一种"预测"：预测脏数据马上还会被改，提前写回是浪费。

这条放在迁移篇有个伏笔：人脑的"记忆固化"（从工作记忆转入长时记忆）主要发生在睡眠时，而不是学习当下——这恰恰是写回策略的生物版：白天只写"缓存"（工作记忆），夜里批量写回"磁盘"（长时记忆），省白天的"写延迟"。

### 7. 四策略横向对比

| 策略 | 淘汰依据 | 实现 | 优点 | 缺点 | 典型场景 |
|---|---|---|---|---|---|
| FIFO | 进入顺序 | 队列 | 简单公平 | 无视访问模式 | 朴素缓存、日志 |
| LRU | 最近访问时间 | 哈希+双向链表 | 贴合局部性 | 扫描式访问会失效 | 通用缓存（Redis/Guava） |
| LFU | 访问总次数 | 频率桶+链表 | 抓长期热点 | 老数据霸占 | 偏稳定热点的场景 |
| ARC | LRU+LFU 自适应 | 双链表+动态分配 | 适应变化负载 | 实现复杂、开销略大 | 负载多变的存储层 |

## 三、实践与面试：反例、边界与跨域映射

> 本章是迁移章，重点不在手撕代码，而在把"淘汰策略"这套思维迁移到人脑认知。先讲清每种策略的失效边界，再展开跨域映射——后者是本章的核心。

### 1. 反例与边界：每种策略什么时候会失效

**LRU 的反例：扫描式访问（one-shot scan）。**

假设缓存容量 3，访问序列 `A B C D A B C D ...`（循环扫描 4 个元素）。LRU 每次都会淘汰"最久未用"，但因为是循环，每次访问的元素恰好是刚被淘汰的那个：

```
访问 A：装入 A，缓存=[A]
访问 B：装入 B，缓存=[A,B]
访问 C：装入 C，缓存=[A,B,C]
访问 D：满，淘汰 A，装入 D，缓存=[B,C,D]
访问 A：满，淘汰 B，装入 A，缓存=[C,D,A]   ← A 刚被淘汰又回来
访问 B：满，淘汰 C，装入 B，缓存=[D,A,B]   ← B 刚被淘汰又回来
……命中率 0%
```

LRU 在循环扫描下命中率归零，这就是缓存抖动的典型形态。真实场景：数据库全表扫描、日志归档读取——这些是一次性遍历，不该污染缓存。工程解法是 **scan-resistant** 设计（如 ARC、CLOCK 算法变种），让一次性访问不轻易挤占热点。

**LFU 的反例：老数据霸占（stale popularity）。**

某新闻网站，一篇爆款文章一周内被访问 10 万次，缓存里频率记到 10 万。一周后热度消退，再没人看，但它的频率仍是 10 万——远超任何新文章。新文章进不来，缓存被一堆过气爆款占满。

工程解法是**频率衰减**（aging）：定期把所有频率乘以一个衰减系数（如 0.99），让老数据的频率随时间递减。Redis 的 LFU 实现就带衰减机制。这个"衰减"思想，正是遗忘曲线的算法版——后面会呼应。

**FIFO 的反例：热点被淘汰与 Belady 异常。**

FIFO 完全不看访问，所以最热的元素只要进得早也会被淘汰。更反直觉的是 **Belady 异常**：在某些访问序列下，**增大缓存容量反而降低命中率**——这反直觉到成了操作系统教材的经典例子。LRU 和 LFU 不会出现 Belady 异常，这是它们相对 FIFO 的理论优势之一（属于"栈算法"类，满足包含性）。

**边界：缓存容量接近工作集时的临界行为。**

当缓存容量远小于工作集（如容量 10、工作集 1000），命中率高低的差异不大——反正装不下。当缓存容量远大于工作集（如容量 1000、工作集 10），命中率都接近 100%——反正装得下。**策略之间的差异，集中在"容量略大于或略小于工作集"的临界区**。这个临界区是工程调优的主战场，也是抖动高发区。理解这一点，你就知道为什么生产环境调缓存大小经常是"差一点就天差地别"。

### 2. 思想迁移一：人脑工作记忆是 7±2 的缓存

1956 年，心理学家 George Miller 在《心理学评论》（*Psychological Review*）发表《神奇的数字 7±2》，提出人类**工作记忆（working memory）**的容量大约是 7±2 个信息单元（chunk）。

工作记忆是什么？就是你此刻"脑子里能同时 hold 住的东西"。读一个电话号码、做一道心算、听完一句话理解它的意思——都靠工作记忆。它的特征和计算机缓存高度同构：

| 维度 | 计算机缓存 | 人脑工作记忆 |
|---|---|---|
| 容量 | 小（KB~GB） | 7±2 个 chunk |
| 速度 | 快（ns~ms） | 快（几百 ms 提取） |
| 上游 | 内存/磁盘 | 长时记忆 |
| 功能 | 暂存热数据 | 暂存当前任务相关信息 |
| 痛点 | 容量不足须淘汰 | 容量不足须遗忘 |

7±2 不是巧合，是进化的缓存设计。大脑皮层能耗极高，不可能把所有记忆都放进"快速通道"，必须有淘汰。**打电话时你默念的那 11 位号码，就是工作记忆里的几个 chunk——通话结束它就被淘汰了，腾地方给下一个任务。** 如果不淘汰，下一个任务的号码和这个混在一起，你拨哪个？

**chunking：扩容缓存的唯一办法。** 7±2 限制的是"chunk 数"而不是"比特数"。一个 chunk 可以是一个数字，也可以是一组有意义的组合——比如"139"对你来说是一个 chunk（手机号前三位），对不懂中文的外国人可能是三个 chunk。专家记忆棋盘位置远超 7 个，因为他们把"一组棋子"压缩成一个 chunk。这对应工程上的**缓存行（cache line）**：CPU 不是按字节缓存，而是按"一行"（通常 64 字节）缓存，用空间换命中。chunking 和缓存行是同一个思想——**用"成组"来扩容逻辑上的缓存容量，绕过物理容量的硬限制**。

这条迁移的落点是：**你记不住东西，常常不是脑子差，是工作记忆容量被低效占满**。把不重要的从"脑子里的缓存"清出去（写下来、交给工具），腾出容量给真正要思考的事——这和给 Redis 调容量是同一件事。而训练自己把零散信息组织成大 chunk（比如把一串数字编成口诀、把知识点串成体系），就等于在不动硬件的前提下给缓存"扩容"。

### 3. 思想迁移二：遗忘曲线是缓存淘汰的生物版

1885 年，德国心理学家 Hermann Ebbinghaus 在《记忆》(*Über das Gedächtnis*) 一书中发表**遗忘曲线（forgetting curve）**：人脑对新学信息的记忆留存率随时间呈指数衰减——20 分钟后约 58%，1 小时后约 44%，1 天后约 33%，之后趋于平缓。

把这个曲线和 LRU 的淘汰逻辑放在一起：

- **LRU**：最久没被访问的淘汰。时间越久没访问，越可能被淘汰。
- **遗忘曲线**：最久没被回忆的记忆衰减最厉害。时间越久没复习，越可能遗忘。

两者几乎是同一个算法的不同实现。LRU 用"链表位置"标记新旧，遗忘曲线用"神经突触强度"标记新旧——底层都是"时间衰减"。

更精确地说，遗忘曲线对应的是**带时间衰减的 LRU**：不只有"是否访问过"的二元标记，还有"多久以前访问的"的连续衰减。这和前面讲 LFU 的频率衰减（aging）是同一个思想——**用时间衰减防止老数据霸占**。人脑不需要专门写一个 aging 函数，突触的可塑性天然就带衰减。

这条迁移的意义在于：**遗忘不是大脑的缺陷，是缓存淘汰策略的生物实现**。一个不遗忘的大脑，等于一个不淘汰的缓存——会被"过气热点"占满，新信息进不来。Ebbinghaus 的曲线不是在描述"大脑有多差"，而是在描述"大脑的淘汰策略有多精细"。

### 4. 思想迁移三：复习策略就是缓存淘汰策略

这是最实用的一条迁移。既然遗忘 = 缓存淘汰，那么**对抗遗忘的复习，就是刷新缓存命中**。

回想 LRU 的"刷新"语义：每次 get 都把节点挪到链表头部，相当于"刚用过，续命"。人脑的复习正是如此——**每次复习都把那条记忆的"遗忘衰减计时器"清零**，让它在工作记忆/长时记忆里续命。

于是各种复习方法都能对应到缓存策略：

- **临考前突击复习**：相当于一次性把所有数据塞进缓存，考试一过立刻全淘汰。短期有效，长期命中率极低——这正是 FIFO 的行为，先进先出不留痕迹。
- **每天固定时间复习所有内容**：相当于固定频率刷新所有缓存项。比突击好，但对已经很熟的内容也复习，浪费"缓存带宽"——这是不带区分的 LFU，频率一刀切。
- **间隔重复（spaced repetition）**：根据每条记忆的遗忘程度动态调整复习间隔，越熟越久才复习、越生越快复习。这就是 Anki、SuperMemo 背后的算法。

间隔重复的算法核心（SM-2 算法，SuperMemo 1987）可以翻译成缓存语言：

```
每次"回忆"（get）成功：
  → 该记忆的"质量分"提升（类似 LFU 频率 +1）
  → 下次复习间隔变长（类似降低刷新频率，省带宽）
每次"回忆"失败：
  → 质量分下降
  → 下次复习间隔缩短（重新频繁刷新）
```

这相当于一个**带反馈的自适应缓存策略**——和 ARC 的"动态调整两个链表容量"是同一层思想：策略本身随负载变化。**会学习的人，就是给自己大脑跑了一个高效的 ARC 缓存。**

### 5. 反向迁移：从生物启发改进算法

迁移不是单向的。人脑的遗忘机制也给算法设计提供启发：

- **睡眠的"重放与巩固"**：人在睡眠时海马体会重放白天的记忆片段，强化重要的、弱化无用的。这对应缓存的**后台预热与清理**——业务低峰期做缓存整理，淘汰冷数据、预热热点。工程上很多系统（如数据库的后台 vacuum、缓存的预热脚本）正是这个套路：把昂贵的整理工作挪到低峰期，白天只做"快路径"读写。
- **情绪标记增强记忆**：带强烈情绪的事件记得更牢（闪光灯记忆）。这对应缓存的**优先级标记**——重要数据不被轻易淘汰，工程上叫"钉住"（pin）某些 key。Redis 的 `OBJECT HELP` 里有 pin 机制，CDN 也有"永久缓存"的层级，都是给某些数据加"情绪标记"。
- **间隔效应（spacing effect）**：分散学习比集中学习效果好。这暗示**缓存的写入策略**：一次性大量写入会互相挤占（缓存立刻满、立刻开始淘汰），分批写入能让每条数据稳定"固化"。这也解释了为什么"批量导入"场景下缓存命中率会短暂跳水——一股脑灌进去，淘汰策略还没适应工作集就先丢了一批。

**一个真实的算法启发案例**：CLOCK 算法（及其变种 CLOCK-PRO）被 Linux 内核用作页面替换算法，它的"访问位"轮转扫描机制，和神经科学里的"突触增强/抑制"有结构上的相似——都是用一个粗粒度的"近期是否活跃"标记来近似 LRU，避免 LRU 那种精确但昂贵的链表维护。生物用突触强度做近似标记，操作系统用访问位做近似标记，都是"用粗粒度近似换实现成本"的取舍。

这些不是牵强附会——它们都指向同一个底层问题：**有限容量下的信息保留取舍**。计算机用算法解，大脑用神经可塑性解，殊途同归。把这条想通，你看任何"取舍"问题都会多一个维度：这个东西的"缓存"在哪、"容量"多大、"淘汰策略"是什么。

### 6. 双向迁移的总结：算法与认知互为镜像

把本章的迁移路径收个口：

- **正向**（算法→认知）：LRU 解释遗忘曲线，LFU 解释用进废退，ARC 解释自适应复习，缓存抖动解释"丢三落四"。
- **反向**（认知→算法）：睡眠巩固启发写回与后台整理，情绪标记启发优先级 pin，突触衰减启发频率 aging，chunking 启发缓存行。

为什么这种双向映射成立？因为"有限容量 + 局部性访问"是物理世界和生物世界共享的约束。只要这个约束在，"用过去预测未来做淘汰"就是必经之路，无论载体是硅片还是神经元。这也是为什么《算法之美》(*Algorithms to Live By*) 这本书能把缓存策略写成一章——它不是比喻，是同构。

理解了这层同构，你回头看第 16 章的哈希表、本章的 LRU，就不再是两个孤立的考点，而是"有限资源下信息存取"这个统一问题的不同切面。**算法课的终点，不是会写更多代码，而是看世界多了一个"资源-存取-淘汰"的透镜。**

## 四、速查与自测

### 速查表：四策略一图掌握

> 一张表收齐"淘汰谁、复杂度、反例、对应人脑现象"四个维度。复习时先盖住右两列自测，再对照。

| 策略 | 淘汰谁 | 复杂度 | 反例 | 对应人脑现象 |
|---|---|---|---|---|
| FIFO | 最先进来的 | O(1) | Belady 异常、热点被淘汰 | 短期记忆自然淡出 |
| LRU | 最久没用的 | O(1) | 扫描式访问抖动 | 遗忘曲线（时间衰减） |
| LFU | 用得最少的 | O(log n) 或 O(1) | 老数据霸占 | "用进废退"的神经可塑性 |
| ARC | 动态选 LRU/LFU | O(1) | 实现复杂 | 自适应复习（间隔重复） |

**关键指标速查**：

- 缓存命中率 = 命中次数 / 总访问次数
- 抖动高发区：缓存容量略小于工作集
- LRU 实现：哈希表（O(1) 定位）+ 双向链表（O(1) 调整顺序）
- LFU 防老数据霸占：频率衰减（aging）
- Belady 异常：FIFO 特有，增大缓存反而降低命中率

### 自测三问

> 下面四题（含一道写策略题）覆盖反例、迁移、双向映射三个层次。建议先合上书自己答，再对照参考答案。

**问题一：** 为什么 LRU 在循环扫描访问下命中率会归零？这和缓存抖动是什么关系？

**参考答案：** 循环扫描（如 `A B C D A B C D...`）的工作集大小若略大于缓存容量，每次访问的元素恰好是上一轮被淘汰的那个——它"最久未用"所以被淘汰，但循环到它时它又是最该用的。LRU 的"最近访问预测未来"假设在循环扫描下失效。这就是缓存抖动的典型形态：容量略小于工作集，导致刚淘汰的马上又被访问，命中率断崖式下跌。工程解法是用 scan-resistant 策略（ARC、CLOCK）让一次性访问不挤占热点，或让全表扫描绕过缓存（bypass）。

**问题二：** 遗忘曲线和 LRU 的对应关系是什么？为什么说"遗忘是 feature 不是 bug"？

**参考答案：** LRU 淘汰最久未访问的数据，遗忘曲线让最久未复习的记忆衰减最厉害——两者底层都是"时间衰减"。说遗忘是 feature 不是 bug，是因为工作记忆容量有限（7±2），如果不遗忘，过气信息会霸占"缓存"挤掉当前任务需要的信息（前摄干扰）。一个不遗忘的大脑等于一个不淘汰的缓存，会被冷数据占满，新信息进不来。Ebbinghaus 曲线描述的是大脑淘汰策略的精细程度，不是大脑的缺陷。

**问题三：** 间隔重复（spaced repetition）算法在缓存语言里怎么描述？它和 ARC 有什么共性？

**参考答案：** 间隔重复把每条记忆当一个缓存项：回忆成功→质量分提升、复习间隔拉长（类似降低刷新频率省带宽）；回忆失败→质量分下降、复习间隔缩短（重新频繁刷新）。这是一个带反馈的自适应缓存。和 ARC 的共性在于"自适应"——ARC 根据 LRU/LFU 两个链表各自的命中情况动态调整容量分配，间隔重复根据每条记忆的回忆成败动态调整复习间隔。两者都是"策略本身随负载变化"，而不是用一个固定策略硬套所有数据。会学习的人，相当于给自己的工作记忆跑了一个高效的 ARC。

**问题四：** 为什么"睡眠巩固记忆"可以看作缓存的写回（write-back）策略？相比"学完立刻背熟"（写穿透）它省了什么？

**参考答案：** 写穿透是边学边把每条信息固化到长时记忆，强一致但写延迟高——白天学习时要同时承担"理解"和"固化"两份开销，吞吐量低。写回是白天只写工作记忆（缓存），标记为待固化，夜里睡眠时海马体批量重放、统一写入长时记忆——把昂贵的"写磁盘"挪到低峰期，白天只走快路径。省的是白天的"写延迟"和认知带宽。代价是崩溃风险（睡眠剥夺会导致缓存里的内容没写回就丢失，第二天记不住）。这也解释了为什么熬夜学习效果差——不是时间不够，是写回没完成，缓存被新信息覆盖了。

### 算法思想 × 生活迁移

缓存淘汰的核心思想是"**有限容量下，用过去预测未来，决定保留什么、淘汰什么**"。

**迁移一：信息流的"注意力缓存"。** 你每天刷的信息（朋友圈、新闻、推文）远超工作记忆容量。注意力就是缓存的分配机制——你"关注"谁，就是把谁放进注意力缓存常驻；取关谁，就是主动淘汰。无限关注等于无限放大缓存、最终什么都记不住。定期清理关注列表，相当于手动触发缓存淘汰，给真正重要的信息腾出容量。

**迁移二：知识体系的"长时记忆 = 持久化存储"。** 工作记忆是缓存，长时记忆是磁盘。新知识先进入工作记忆（缓存），经过复习（刷新命中）逐步固化到长时记忆（持久化）。从不复习的知识会从缓存淘汰，但没进入长时记忆，等于"丢了"——这就是为什么"看过≠学会"。学习的关键不是"看过多少"，而是"有多少从缓存固化到了磁盘"。

**迁移三：团队知识的"文档缓存"。** 团队 wiki 是持久化存储，但每个人脑子里只有一小部分是"工作记忆缓存"。新人入职要查文档（cache miss，慢），熟手张口就来（cache hit，快）。文档没人维护就会"过期"，相当于缓存数据陈旧——读出来是错的。定期 review 文档就是缓存的"一致性校验"（consistency check）。

**迁移四：用工具"卸载"工作记忆。** 你的笔记本、待办清单、搜索引擎，本质是把工作记忆里的内容"卸载"到外部持久化存储。凡是"我不必此刻记住但要确保能查到"的，都该卸载——电话号码、会议时间、API 参数。这腾出的工作记忆容量，留给真正需要"在线思考"的部分（推理、决策、创造）。GTD（Getting Things Done）方法论的内核就这一句：**大脑是用来思考的，不是用来记事的**——翻译成缓存语言，就是"缓存用来命中热数据，不是用来当仓库"。聪明的做法是主动制造 cache miss 给外部工具（查一次比记错一次便宜），而不是硬塞缓存导致抖动。

**为什么这些迁移成立：** 缓存淘汰成立的前提是"容量有限 + 访问有局部性"。注意力、学习、团队知识都满足这两个前提——容量都有限（时间精力有限），访问都有局部性（反复用同一批知识/工具/人）。只要这两个前提成立，"淘汰策略"的思维就适用。迁移失效的场景是"容量无限"或"访问完全随机"——前者不需要淘汰，后者淘汰策略也帮不上忙。

**一个迁移失效的例子**：有人试图用 LRU 管理自己的"人脉缓存"——定期联系最近互动过的人。但人脉的访问模式不是 LRU 友好的：一个十年没联系的老同学，可能在你创业时突然成为关键人（访问完全不可预测，没有时间局部性）。对人脉这种"长期不联系但一旦联系价值极高"的场景，LRU 会把关键弱联系全淘汰掉。这时更接近 LFU（按累计互动深度）或干脆不淘汰（全量保留，靠外部工具检索）。**迁移不是万能公式，先验证"容量有限 + 局部性"两个前提是否成立，再决定用哪种淘汰策略。**
