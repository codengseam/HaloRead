# 算法课·22｜一致性哈希：分布式下的哈希与虚拟节点

> 前置知识：学完第 10 章（哈希表与布隆过滤器），尤其要懂"`hash(key) % N` 把键落到 N 个桶"这件事，以及 Bloom Filter 的"用误差换空间"哲学
> 学完你能：①给纯小白讲清"为什么加一台缓存机器会引发雪崩" ②默写一致性哈希环并讲透每一行 ③手撕虚拟节点版应对面试 ④用对数器对比朴素取模与一致性哈希的迁移率 ⑤讲清 Redis Cluster / Cassandra / DynamoDB 为何都选一致性哈希 ⑥理解"分布式下的哈希"和"单机哈希"在归属规则上的关键差异

## 一、问题与思路：朴素取模的痛点与一致性哈希的破局

### 1. 一个反直觉场景：加一台机器，九成缓存失效

假设你维护一个缓存集群，10 台机器，用 `hash(key) % 10` 把数据分散到 10 台，命中率 95%。某天大促流量上涨，你加了 1 台机器变成 11 台——`hash(key) % 11` 对几乎所有 key 算出的下标都变了，**约 90.9% 的缓存数据要重新分配**。

更糟的是：迁移期间这些 key 全部 miss，请求穿透到数据库，数据库瞬间被打挂——这就是著名的"缓存雪崩"。**反直觉的地方在于**：加机器本来是为了扛更多流量，结果反而把系统搞挂了。

问题出在哪？`hash(key) % N` 这个函数对 $N$ 极度敏感——$N$ 变 1，几乎每个 key 的映射结果都变。我们需要一种"对 $N$ 不敏感"的哈希方案：加机器只影响一小段数据，减机器也只影响一小段。这就是一致性哈希要解决的问题。

### 2. 朴素取模的两个痛点

**痛点一：增删节点迁移量极大。** $N \to N+1$ 时，朴素取模的迁移率是 $\frac{N}{N+1}$，接近 1——几乎所有数据都要动。10 台变 11 台约 90.9% 迁移，100 台变 101 台约 99.0% 迁移。机器越多，加一台的相对影响越小，但绝对迁移量仍接近全量。

**痛点二：节点编号硬绑定。** 朴素取模把节点编号（0 到 $N-1$）和哈希值硬绑死。节点 5 故障下线变成 9 台，原本落在节点 5 的数据不能简单地"交给节点 6"——因为 `hash(key) % 9` 会让所有节点重新洗牌，不是只动节点 5 的数据。换言之，**节点故障的代价被放大成了全集群洗牌**。

**核心思想**：一致性哈希把"节点编号"和"哈希值"解耦——不再用 `hash(key) % N`，而是把节点和键都映射到同一个环上，让"相邻关系"决定归属。加减节点只影响环上一小段，其他段不变。

### 3. 历史地位：从 MIT 论文到分布式基础设施

1997 年，MIT 的 David Karger 等人在 *Consistent Hashing and Random Trees* 中提出一致性哈希，原始目的是解决"分布式缓存加减节点雪崩"。此后这个思想渗透到几乎所有分布式存储：

- **1997** Karger 等人提出一致性哈希（MIT LCS TR）[25]
- **2003** Amazon Dynamo 用一致性哈希做数据分片（内部系统，论文 2007 SOSP 发表）
- **2007** Last.fm 的 Richard Jones 开源 Ketama——一致性哈希的工程实现，成为 Memcached 的事实标准
- **2008** Cassandra 用一致性哈希做分片，1.2 版引入 vnode（虚拟节点）
- **2010** Redis Cluster 引入哈希槽（16384 槽），是一致性哈希的工程变体
- **2014** Google Megastore / Spanner 等系统借鉴一致性哈希思想做分片

**这条历史线的关键洞察**：一致性哈希把"单机哈希表"的 $O(1)$ 定位思想，扩展到"多机分布式"场景——单机哈希解决"一个桶里找键"，一致性哈希解决"一群机器里找键归属"。理解这条线，就理解了"哈希思想在分布式下的工程延伸"，也是为什么本专栏在哈希表章节之外单开一章讲它。

## 二、原理与实现：从哈希环到虚拟节点

### 1. 一致性哈希环：0~2³² 的环

把哈希空间看成一个环，范围 $[0, 2^{32})$。把节点和键都用同一个哈希函数映射到这个环上：

- **节点映射**：`hash(node_name) % 2^32`，得到节点在环上的位置
- **键映射**：`hash(key) % 2^32`，得到键在环上的位置
- **归属规则**：键顺时针找到的**第一个节点**就是它的归属

```
        0
       / \
   nodeA   nodeB
     |       |
    key1    key2
       \ /
      2^32
```

走一个具体例子：3 个节点 `nodeA`、`nodeB`、`nodeC`，哈希后落在环上的位置（用 0~1000 简化示意）：

```
nodeA → 100
nodeB → 500
nodeC → 900
```

插入 4 个 key：

```
key1 → 50   → 顺时针找 → nodeA（100）
key2 → 300  → 顺时针找 → nodeB（500）
key3 → 700  → 顺时针找 → nodeC（900）
key4 → 950  → 顺时针找 → nodeA（100，绕回环首）
```

**关键观察**：键的归属只取决于"环上最近的下游节点"。加减节点时，只有"新节点上游到新节点"这一段数据的归属变化，其他全部不变。

### 2. 加减节点的迁移量：从接近全量降到约 1/N

**加节点 nodeD（落在 400）**：只有 key2（300）的归属从 nodeB 变成 nodeD——因为 nodeD 插在 key2 和 nodeB 之间。其他 key 不动。迁移量约 $1/(N+1)$。

**减节点 nodeB（500）**：原本属于 nodeB 的 key2（300）顺时针找下一个节点 → nodeD（400）。其他 key 不动。迁移量约 $1/N$。

对比朴素取模的迁移率：

| 操作 | 朴素取模迁移率 | 一致性哈希迁移率 |
|---|---|---|
| 10 → 11 节点 | 90.9% | ~9.1% |
| 50 → 51 节点 | 98.0% | ~2.0% |
| 100 → 101 节点 | 99.0% | ~1.0% |
| 减 1 节点（10 台） | 90.9% | ~10.0% |

**这就是一致性哈希的价值**：把"加减节点的迁移率"从 $O(1)$（接近全量）降到 $O(1/N)$（只动一段）。机器越多，加一台的相对影响越小，这正是大集群下必须用一致性哈希的硬理由。

### 3. 朴素一致性哈希的致命问题：数据倾斜

朴素一致性哈希有个问题——节点在环上是"随机"分布的，可能挤在一起也可能有大片空白。3 个节点恰好都落在 100、110、120，那 90% 的环空间无节点，所有 key 都归第一个节点——严重的**数据倾斜**。

```
环：0 ----------------- 1000
节点：   100 110 120
         ↑↑↑ 全挤在一起，900 多空间无节点
```

**根因**：节点数太少时，哈希函数的"均匀性"在统计上不成立——大数定律需要 $N$ 足够大。3 个点的均匀分布概率极低，绝大多数情况是倾斜的。

### 4. 虚拟节点：用"放大样本"解决倾斜

解法是**虚拟节点（Virtual Node, vnode）**——每台物理机在环上放多个"虚拟副本"，比如每台 150 个。3 台物理机 = 450 个虚拟节点，统计上分布就均匀了。

```
nodeA → vnode: nodeA#1, nodeA#2, ..., nodeA#150
nodeB → vnode: nodeB#1, nodeB#2, ..., nodeB#150
nodeC → vnode: nodeC#1, nodeC#2, ..., nodeC#150
```

键顺时针找到的第一个**虚拟节点**，其归属的物理机就是键的归属。

**为什么虚拟节点能解决倾斜**？因为大数定律——$N$ 越大，样本分布越接近均匀。3 个点倾斜概率高，450 个点倾斜概率极低。每个物理机的虚拟节点在环上各处散布，统计上各物理机占的环空间接近 $1/3$。

**虚拟节点数的经验值**：Memcached / Ketama 默认每台 160 个，Cassandra 默认每台 256 个。太少倾斜重，太多内存开销大（要存 vnode 到物理机的映射）。150-256 是工程上的甜蜜点。

### 5. 手写一致性哈希（带虚拟节点）

```python
import hashlib
import bisect

class ConsistentHash:
    def __init__(self, replicas=150):
        self.replicas = replicas               # 每台物理机的虚拟节点数
        self.ring = {}                          # hash值 -> 物理机名
        self.sorted_keys = []                   # 排序后的hash值列表（用于二分查找）

    def _hash(self, key):
        # MD5 取前 8 字节，雪崩性好且确定性（跨机器一致）
        h = hashlib.md5(str(key).encode()).hexdigest()
        return int(h[:8], 16) % (2 ** 32)

    def add_node(self, node):
        # 为每台物理机生成 replicas 个虚拟节点
        for i in range(self.replicas):
            vkey = f"{node}#{i}"                # 虚拟节点名：nodeA#0, nodeA#1, ...
            h = self._hash(vkey)
            self.ring[h] = node
            bisect.insort(self.sorted_keys, h)  # 维护有序，O(n) 插入

    def remove_node(self, node):
        for i in range(self.replicas):
            vkey = f"{node}#{i}"
            h = self._hash(vkey)
            if h in self.ring:
                del self.ring[h]
                idx = bisect.bisect_left(self.sorted_keys, h)
                self.sorted_keys.pop(idx)

    def get_node(self, key):
        if not self.sorted_keys:
            return None
        h = self._hash(key)
        # 顺时针找第一个 > h 的虚拟节点（二分 O(log n)）
        idx = bisect.bisect_right(self.sorted_keys, h)
        if idx == len(self.sorted_keys):        # 绕回环首
            idx = 0
        return self.ring[self.sorted_keys[idx]]
```

**逐行讲透**（左神风格，每一行都不能含糊）：

- **`self.replicas = 150`**：每台物理机的虚拟节点数。150 是 Ketama 经验值——足够均匀，内存开销可接受（10 台物理机 = 1500 个 vnode 映射，几十 KB）。
- **`self.ring = {}`**：哈希值到物理机名的映射。注意 key 是哈希值（int），value 是物理机名（str）。一个物理机会有 150 个 key 指向它。
- **`self.sorted_keys = []`**：所有哈希值的排序列表。**关键**：查找"顺时针下一个节点"本质是"找第一个大于 h 的哈希值"，必须有序才能二分。
- **`hashlib.md5(...).hexdigest()[:8]`**：用 MD5 前 8 字节（32 bit）作为哈希。为什么不用 Python 内置 `hash`？因为 `hash` 对字符串每次进程启动结果不同（哈希随机化），分布式场景需要跨机器一致——MD5 是确定性的。
- **`for i in range(self.replicas)`**：为每台物理机生成 150 个虚拟节点。虚拟节点名用 `node#i` 区分，保证它们的哈希值各不相同。
- **`bisect.insort(self.sorted_keys, h)`**：插入并保持有序。`bisect.insort` 是 $O(n)$ 插入（数组移动），但只在加减节点时调用，不在查询路径上，可接受。
- **`bisect.bisect_right(self.sorted_keys, h)`**：二分查找第一个大于 h 的位置（顺时针下一个）。$O(\log n)$，这是查询的性能命根。
- **`if idx == len(...): idx = 0`**：h 落在最后一个节点之后，绕回环首。这就是"环"的体现——首尾相连。

**走一个例子**：3 个节点 `A/B/C`，replicas=2（简化）：

```
add_node("A"): ring[hash("A#0")]=A, ring[hash("A#1")]=A
add_node("B"): ring[hash("B#0")]=B, ring[hash("B#1")]=B
add_node("C"): ring[hash("C#0")]=C, ring[hash("C#1")]=C
sorted_keys = [hash值排序后的列表]

get_node("user:1001"):
  h = hash("user:1001")
  二分找第一个 > h 的虚拟节点 → 假设落在 A#1 的位置
  返回 ring[hash("A#1")] = "A"
```

**关键观察**：查询只需 1 次哈希 + 1 次二分 $O(\log (N \cdot R))$，与物理机数无关——这就是分布式下的快速定位（严格是 $O(\log (NR))$，但 $NR$ 是常数级，实际接近 $O(1)$）。

### 6. 五指标评价

| 指标 | 一致性哈希（带虚拟节点） | 说明 |
|---|---|---|
| 查询 | $O(\log (N \cdot R))$ | 二分查找，$N$ 物理机、$R$ 虚拟节点/机 |
| 加/减节点 | $O(R \log (N \cdot R))$ | 插入/删除 $R$ 个 vnode，每个 $O(\log)$ 二分 |
| 迁移量 | $O(1/N)$ | 加减 1 节点只动 $1/N$ 数据 |
| 空间 | $O(N \cdot R)$ | 存 vnode → 物理机映射 |
| 均匀性 | 依赖 $R$ | $R$ 越大越均匀，$R=150$ 时倾斜可忽略 |
| 单调性 | 满足 | 加节点不改变既有键的相对归属 |

**单调性（Monotonicity）** 是一致性哈希的关键性质：当节点集合扩大时，原有键的归属要么不变，要么落到新加入的节点——绝不会从原有节点 A 跳到原有节点 B。朴素取模不满足单调性（加节点后所有键重新洗牌），这是两者最关键的差异。

## 三、实践与面试：手撕模板、对数器、面试题

### 1. 面试手撕模板（按"技术的水有多深"八层讲）

面试中被要求"设计分布式缓存"或"实现一致性哈希"，下面的模板够用：

```python
import hashlib
import bisect

class ConsistentHashRing:
    def __init__(self, replicas=150):
        self.replicas = replicas
        self.ring = {}
        self.sorted_hashes = []

    def _hash(self, key):
        digest = hashlib.md5(str(key).encode()).hexdigest()
        return int(digest[:8], 16)

    def _add_virtual_node(self, vnode, node):
        h = self._hash(vnode)
        if h not in self.ring:                    # 哈希碰撞时跳过（极少）
            self.ring[h] = node
            bisect.insort(self.sorted_hashes, h)

    def add_node(self, node):
        for i in range(self.replicas):
            self._add_virtual_node(f"{node}#{i}", node)

    def remove_node(self, node):
        for i in range(self.replicas):
            h = self._hash(f"{node}#{i}")
            if h in self.ring:
                del self.ring[h]
                idx = bisect.bisect_left(self.sorted_hashes, h)
                self.sorted_hashes.pop(idx)

    def get_node(self, key):
        if not self.sorted_hashes:
            return None
        h = self._hash(key)
        idx = bisect.bisect_right(self.sorted_hashes, h)
        if idx >= len(self.sorted_hashes):
            idx = 0
        return self.ring[self.sorted_hashes[idx]]
```

**面试时怎么讲**（八层面挑重点）：

1. **问题层**（核心思想）：朴素取模 `hash(key) % N` 对 $N$ 敏感，加减节点迁移率接近 100%。一致性哈希把节点和键都映射到 $[0, 2^{32})$ 环上，键顺时针找第一个节点，加减只影响一段。
2. **原理层**：节点用 `hash(node#i)` 在环上放 $R$ 个虚拟节点（$R=150$），键用 `hash(key)` 落环，二分找顺时针下一个 vnode。虚拟节点解决"节点少时分布不均"的倾斜问题。
3. **优劣层**：迁移率从 $O(1)$ 降到 $O(1/N)$；但查询从 $O(1)$ 升到 $O(\log (NR))$，空间从 $O(N)$ 升到 $O(NR)$——用空间和时间换迁移效率。
4. **演进层**：Karger 1997 原始论文（无虚拟节点）→ Ketama 2007（虚拟节点工程化）→ Cassandra（vnode 默认 256）→ Redis Cluster（16384 哈希槽变体）。
5. **坑位层**：①虚拟节点数太少倾斜重 ②MD5 跨机器一致性（不能用 Python `hash`）③哈希碰撞处理（极罕见，跳过即可）。
6. **场景层**：分布式缓存（Memcached / Redis）、分布式存储（Cassandra / DynamoDB）、负载均衡（带会话保持的 L7 路由）。
7. **替代层**：Redis Cluster 用"哈希槽"——固定 16384 槽手动分配到节点，加减节点时迁移槽位。这是"预先划分环段"的工程变体，避免运行时二分。
8. **追问层**：虚拟节点数怎么选？答：Memcached 150、Cassandra 256，经验上 100-300 都可；太少倾斜，太多内存开销。

**面试加分点**：主动提"单调性"——一致性哈希满足"加节点不改变既有键的相对归属"，而朴素取模不满足。这能立刻区分你和"只会背环结构"的候选人。

### 2. 对数器：对比朴素取模与一致性哈希的迁移率

左神反复强调对数器——这里我们用它对比两种方案的迁移率，量化一致性哈希的优势。

```python
import random

def naive_mod_migration(keys, old_n, new_n):
    """朴素取模：N -> N+1 时的迁移率。"""
    moved = 0
    for k in keys:
        if hash(k) % old_n != hash(k) % new_n:
            moved += 1
    return moved / len(keys)

def consistent_hash_migration(keys, ch, new_node):
    """一致性哈希：加一个节点的迁移率。"""
    before = {k: ch.get_node(k) for k in keys}
    ch.add_node(new_node)
    after = {k: ch.get_node(k) for k in keys}
    ch.remove_node(new_node)                      # 还原，便于重复实验
    moved = sum(1 for k in keys if before[k] != after[k])
    return moved / len(keys)

def migration_checker(test_times=100, n_keys=10000):
    """对数器：跑 test_times 次，对比两种方案的迁移率。"""
    for t in range(test_times):
        n_old = random.randint(5, 50)
        n_new = n_old + 1
        keys = [f"key_{i}_{t}" for i in range(n_keys)]

        # 朴素取模迁移率
        m_naive = naive_mod_migration(keys, n_old, n_new)

        # 一致性哈希迁移率
        ch = ConsistentHashRing(replicas=150)
        for i in range(n_old):
            ch.add_node(f"node_{i}")
        m_consistent = consistent_hash_migration(
            keys, ch, f"node_{n_old}")

        # 一致性哈希迁移率应远低于朴素取模
        # 理论值：朴素 ~n_old/(n_old+1)，一致性 ~1/(n_old+1)
        if m_consistent >= m_naive * 0.5:         # 一致性应 < 朴素的 50%
            print(f"异常！N={n_old} 朴素={m_naive:.3f} 一致性={m_consistent:.3f}")
            return False
    print(f"对数器通过：{test_times} 轮，一致性哈希迁移率显著低于朴素取模")
    return True

migration_checker()
```

**典型输出**：

```
N=10  朴素=0.909  一致性=0.095   （一致性低约 10 倍）
N=20  朴素=0.952  一致性=0.049
N=50  朴素=0.980  一致性=0.020
```

**对数器的价值**：把"一致性哈希迁移少"从口号变成可量化的数字。$N$ 越大，朴素取模迁移率越接近 100%，一致性哈希迁移率越接近 0——差距随 $N$ 增大而放大。这就是大集群下必须用一致性哈希的硬理由。**左神原话**："写完一个数据结构或算法，第一件事就是写对数器，跑够量再说自己写对了。"——这里我们把对数器从"验证正确性"扩展到"对比方案优劣"，是同一种思路的延伸。

### 3. 教科书做法 vs 生产做法

| 场景 | 教科书做法 | 生产做法 | 原因 |
|---|---|---|---|
| 分布式缓存路由 | 手写一致性哈希环 | Ketama / hashring 库 | 工程实现要考虑副本、权重、故障转移 |
| 大规模 KV 存储 | 一致性哈希 + 虚拟节点 | Cassandra 内置 vnode | 生产需要副本放置策略（RF=3 跨机架） |
| Redis 分片 | 一致性哈希 | Redis Cluster 哈希槽（16384 槽） | 哈希槽手动管理更可控，迁移可量化 |
| 负载均衡（带会话） | 一致性哈希 | Nginx `upstream consistent_hash` | Nginx 已内置，无需手写 |
| 数据库分库分表 | hash(key) % N | ShardingSphere 一致性哈希分片 | 中间件已实现，关注分片键设计 |

**生产里几乎不手写一致性哈希**，但理解它的价值在于：

1. **读懂分布式中间件源码**：Cassandra、DynamoDB、Memcached 的分片逻辑都基于一致性哈希，不懂就读不懂它们的副本放置、故障恢复机制。
2. **设计分片方案时的权衡**：什么时候选一致性哈希、什么时候选哈希槽、什么时候选范围分片——不懂一致性哈希就做不了这个决策。
3. **面试高频考点**："设计分布式缓存""设计短链系统""设计 L7 负载均衡"都涉及一致性哈希，是系统设计面试的常驻题。

### 4. 三个真实工程坑

**坑一：虚拟节点数太少，数据严重倾斜。**

```python
ch = ConsistentHashRing(replicas=3)             # 太少
for nd in ["A", "B", "C"]:
    ch.add_node(nd)
# 跑 10000 个 key，统计各节点占比
# 可能出现 A=70%, B=5%, C=25% 的极端倾斜
```

症状：某节点 CPU 100%、其他空闲。根因：$R=3$ 时 3 个物理机只有 9 个 vnode，大数定律不成立，环空间分布严重不均。修复：$R$ 提到 150-256，倾斜降到 ±5% 以内。

**坑二：用 Python 内置 `hash` 当哈希函数，跨机器结果不一致。**

```python
# 错误写法
def _hash(self, key):
    return hash(key) % (2 ** 32)                # Python hash 有随机化！
```

症状：A 机器算 key 归 nodeX，B 机器算同一 key 归 nodeY——请求路由混乱。根因：Python 3.3+ 默认开启哈希随机化（防 HashDoS，详见第 10 章），每个进程 `hash("abc")` 结果不同。修复：用确定性哈希 MD5 / SHA1 / MurmurHash。

**坑三：节点故障下线时，未做"优雅迁移"导致雪崩。**

```python
# 节点 B 故障，直接 remove_node
ch.remove_node("B")
# B 上的所有 key 瞬间全部转到下一个节点 C
# C 承受 2 倍流量，可能也被打挂
```

症状：一台故障引发连锁雪崩。根因：`remove_node` 后，故障节点的全部流量瞬间涌向下一个节点。修复：①副本机制（每个 key 写 3 副本到不同节点，故障时从副本读）②"优雅下线"——先把故障节点的 key 逐步迁移到下游，再正式下线 ③熔断降级，避免 C 也被打挂。

### 5. 面试高频题：设计分布式缓存

**面试题**："设计一个支持动态扩容的分布式缓存系统。"

**满分回答框架**：

1. **需求澄清**：QPS？数据量？强一致还是最终一致？假设 10M QPS、100TB 数据、最终一致。
2. **分片方案**：用一致性哈希 + 虚拟节点（$R=150$）。理由：扩容时迁移量 $O(1/N)$，避免雪崩。
3. **数据结构**：每台机器维护哈希环的 vnode → 物理机映射，查询时二分 $O(\log (NR))$ 定位。
4. **副本机制**：每个 key 写 3 副本，顺时针找下 3 个不同物理机。故障时从副本读，保证可用性。
5. **扩容流程**：①新节点加入环 ②迁移"新节点上游到新节点"这一段的 key ③迁移完成后正式接流量。
6. **故障恢复**：心跳检测节点下线 → 标记为 dead → 副本提升为主 → 后台补齐副本数。

**面试官追问"为什么不用朴素取模"**：朴素取模加机器迁移率接近 100%，迁移期间全 miss 引发雪崩。一致性哈希迁移率约 $1/N$，10 台加 1 台只动 10%，可控。

**面试官追问"虚拟节点数怎么选"**：太少倾斜（$R=3$ 可能 70/5/25 倾斜），太多内存开销（$R=10000$ 时 10 台机 10 万 vnode 映射，几十 MB）。经验值 150-256，倾斜控制在 ±5%。

**面试官追问"Redis Cluster 为什么用哈希槽而不是一致性哈希"**：哈希槽（16384 槽）是"预先划分环段"的工程变体——固定槽数手动分配到节点，加减节点时迁移整个槽位。优点：①迁移可量化（槽数明确）②无需运行时二分（直接 `slot = hash(key) % 16384`）③便于运维（槽位迁移有明确进度）。缺点：扩容时槽位重新分配需人工介入，不如一致性哈希自动。

### 6. 高手向：Redis Cluster / Cassandra / DynamoDB 的应用

**Redis Cluster：哈希槽变体。**

Redis Cluster 不用纯一致性哈希，而是 16384 个固定哈希槽：`slot = CRC16(key) % 16384`，每个节点负责一部分槽。加减节点时把槽位从一个节点迁移到另一个。**为什么是 16384？** Redis 作者 antirez 解释：①槽数足够多保证均匀 ②槽数足够少保证 gossip 协议消息小（每节点用 2KB bitmap 表示持有的槽）③CRC16 简单快。这是"一致性哈希思想 + 工程权衡"的典范——保留"加减只动一段"的核心，用固定槽位换可运维性。

**Cassandra：vnode 默认 256。**

Cassandra 1.2 引入 vnode（虚拟节点），每台物理机默认 256 个 vnode 随机分布在环上。加减节点时自动 rebalance，无需人工分配 token。**关键设计**：副本放置用 `NetworkTopologyStrategy`——每个 key 的副本落在顺时针接下来的 $N$ 个不同 rack（机架），保证机架级故障不丢数据。这是把一致性哈希 + 副本策略 + 拓扑感知三者结合的工程典范。

**DynamoDB（Amazon）：一致性哈希 + 向量时钟。**

Amazon 2007 年的 Dynamo 论文（SOSP）首次把一致性哈希 + 最终一致性 + 向量时钟组合成生产级 KV 存储。一致性哈希做分片，向量时钟解决多副本冲突，sloppy quorum 保证高可用。这套设计影响了后续 Cassandra、Riak、Voldemort 等 NoSQL 系统。**核心洞察**：一致性哈希是"分布式存储三件套"（分片 + 副本 + 拓扑）的第一块基石，没有它就没有现代 NoSQL。

## 四、速查与自测

### 速查表：一致性哈希 vs 朴素取模

| 维度 | 朴素取模 `hash%N` | 一致性哈希（带 vnode） |
|---|---|---|
| 查询复杂度 | $O(1)$ | $O(\log (NR))$ |
| 加减节点迁移率 | 接近 100% | 约 $1/N$ |
| 数据均匀性 | 完全均匀（依赖哈希函数） | 统计均匀（依赖 $R$ 足够大） |
| 单调性 | 不满足 | 满足 |
| 空间开销 | $O(1)$ | $O(NR)$ |
| 实现复杂度 | 极简 | 中等 |
| 适用场景 | 节点数固定的单层分片 | 节点动态增删的分布式系统 |

**主流系统选型**：

| 系统 | 分片方案 | 虚拟节点 / 槽数 |
|---|---|---|
| Memcached | Ketama 一致性哈希 | 150-160 vnode/机 |
| Redis Cluster | 哈希槽 | 16384 槽 |
| Cassandra | 一致性哈希 + vnode | 256 vnode/机 |
| Amazon DynamoDB | 一致性哈希 | 256 vnode/机 |
| MongoDB Sharding | 范围分片（非哈希） | - |

### 自测三问

**问题一：** 一致性哈希为什么能把加减节点的迁移率从接近 100% 降到约 1/N？请用"环上相邻关系"解释。

**参考答案：** 朴素取模 `hash(key) % N` 中，键的归属由"哈希值和 $N$ 共同决定"，$N$ 变 1 几乎所有键的归属都变。一致性哈希中，键的归属只由"环上顺时针下一个节点"决定——加减节点只改变"新节点上游到新节点"这一段的归属，其他段不变。具体地：加节点 D 插在 A 和 B 之间，原本属于 B 的、落在 D 上游那段（A 到 D）的键改归 D，其他键归属不变；这段占环空间约 $1/(N+1)$，所以迁移率约 $1/(N+1) \approx 1/N$。**核心洞察**：一致性哈希把"归属依赖全局 $N$"改成"归属依赖局部相邻关系"——这是迁移率骤降的根本原因。

**问题二：** 虚拟节点为什么能解决数据倾斜？数量怎么选？太少太多各有什么问题？

**参考答案：** 朴素一致性哈希节点少时（如 3 台），哈希函数的均匀性在统计上不成立——3 个点大概率挤在一起或有大片空白，导致某节点占 70% 环空间。虚拟节点让每台物理机在环上放 $R$ 个副本，$N \cdot R$ 个点的大数定律成立，分布均匀。数量选择：太少（$R < 50$）倾斜重，$R=3$ 可能 70/5/25 分布；太多（$R > 1000$）内存开销大，10 台机 1 万 vnode 映射几十 MB，且加减节点插入 $R$ 次的开销累积。经验值 150-256：Memcached 150、Cassandra 256，倾斜控制在 ±5% 以内。**关键**：虚拟节点不是"消除倾斜"而是"用样本量把倾斜降到可接受"——这是统计思想在工程上的应用。

**问题三：** Redis Cluster 用哈希槽（16384 槽）而不是纯一致性哈希，请分析这个工程权衡的得失。

**参考答案：** 哈希槽是"预先划分环段"的一致性哈希变体——`slot = CRC16(key) % 16384`，每个节点持有若干槽，加减节点时迁移整个槽。**得**：①迁移可量化（明确知道哪些槽要动）②无需运行时二分（直接取模定位槽，再查表找节点）③运维友好（槽位迁移有明确进度，可暂停可回滚）④gossip 协议消息小（每节点 2KB bitmap 表示持槽）。**失**：①扩容时槽位重新分配需人工或工具介入，不如纯一致性哈希自动 ②槽数固定 16384，集群规模超 1000 节点时每节点平均 16 槽，分配粒度变粗。**核心洞察**：哈希槽保留了"加减只动一段"的一致性哈希核心，用"固定槽位 + 手动分配"换"可运维性"——这是工程权衡的典范，没有银弹，只有按场景选自动性 vs 可控性。

### 算法思想 × 生活迁移

一致性哈希的核心思想是"**用相邻关系而非全局编号决定归属**"——加减成员只影响局部，不引发全局洗牌。

**迁移一：餐厅排班加减服务员。** 朴素取模式：10 个服务员按"工号 % 10"分桌——加 1 人变成"工号 % 11"，所有桌的责任人全变，重新交接一团乱。一致性哈希式：把桌号和服务员都映射到时间环上，每桌归顺时针下一个服务员——加 1 人只影响"新服务员上游那段桌"，其他桌责任人不变。**关键洞察**：当"成员会变动"时，归属规则要基于"相对位置"而非"绝对编号"——这是分布式系统鲁棒性的根本。

**迁移二：项目交接的"上下游"思维。** 朴素取模式交接：人一走，所有任务重新分配，每个人都要重新熟悉新任务。一致性哈希式交接：离职人的任务只转给"顺时针下一个"同事，其他人的任务不动。**迁移启示**：组织设计里，"按上下游交接"比"全局重新分配"鲁棒得多——成员变动是常态，归属规则要为变动而设计。

**迁移三：CDN 边缘节点的请求路由。** 用户请求归哪个 CDN 节点？朴素取模：`hash(user_ip) % N`，加节点全乱。一致性哈希：用户 IP 和 CDN 节点都映射到环上，请求归顺时针最近节点——加节点只影响一小段用户的归属，缓存命中率不崩。**现实案例**：Akamai、Cloudflare 的边缘路由都基于一致性哈希思想，保证"加边缘节点不洗牌全局缓存"。

**为什么这些迁移成立：** 一致性哈希成立的前提是"成员集合会变动，且变动时希望最小化扰动"。这个前提在排班、交接、CDN 路由里都成立——都是"成员动态增减，归属要平稳过渡"。迁移成立的关键是"归属基于相邻关系而非绝对编号"——只要场景满足这个前提，一致性哈希思想就适用。
