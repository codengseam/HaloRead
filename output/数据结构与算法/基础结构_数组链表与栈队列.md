# 算法课·19｜数组与链表，栈与队列：连续离散与两种工作模式

> 前置知识：具备基本编程基础，能看懂 Python 语法和指针概念
> 学完你能：①用一句话讲清数组与链表的物理差异 ②默写链表反转、检测环、找中点、合并有序四件套 ③手撕 LRU 缓存和单调队列 ④解释为什么"理论上更快的链表"在工程上常被数组碾压 ⑤用对数器对比手写链表与 Python list

## 一、问题与思路：连续离散与两种工作模式

### 1. 一个生活场景：电影院选座 vs 老式火车车厢

设想你去电影院选座——一排 20 个座位连在一起，编号 1 到 20，找 7 号座位直接走过去坐下，但要在 7 号位置插一个人，7 到 20 号全部往后挪一格。再设想老式火车，每节车厢独立，车厢之间靠挂钩连起来——找第 7 节车厢要从车头一节一节数过去，但要在中间插一节车厢，解开挂钩挂上新的就行。

这是两种座位安排，对应数据结构里的两种基础形态：数组与链表。差别不在功能，在物理约束——座位连在一起还是离散分布。这个约束决定了两种结构各自的强项和弱项。

**电影院选座（数组式）**：座位连排，找位 O(1)，插人 O(n)——后面全挪。
**老式火车车厢（链表式）**：车厢离散，找位 O(n)，插车厢 O(1)——只改挂钩。

这就是数组与链表的差别：**数组随机访问 O(1) 但插入 O(n)，链表插入 O(1) 但访问 O(n)**。

### 2. 一个反直觉问题：为什么理论更优的链表常被数组碾压

教科书会告诉你：链表插入 O(1)，数组插入 O(n)。那为什么生产环境里 `ArrayList`、`vector`、Python `list` 几乎都是数组实现，链表反而少见？

答案在 CPU 缓存。数组在内存中是连续的，CPU 读取 `arr[i]` 时会把整个缓存行（cache line，通常 64 字节）预取进 L1 缓存，后续读 `arr[i+1]`、`arr[i+2]` 直接命中缓存，速度比从主存读快 10-100 倍。链表节点散落在堆（heap）各处，每次访问都是一次 cache miss，CPU 要去主存捞数据，慢一个数量级。

这就是为什么 $O(1)$ 的链表插入，在小数据量上跑不过 $O(n)$ 的数组插入——常数项被缓存放大了几十倍。**复杂度只描述增长趋势，不描述常数项，更不描述缓存友好性**。这个反直觉的事实是本章最重要的一条工程认知。

### 3. 栈与队列：受限的线性表，两种工作模式

数组链表是"全能线性表"——两端都可读写。栈和队列是"受限的线性表"——只允许在一端操作。

- **栈（Stack）= LIFO（Last In First Out，后进先出）**：只允许在栈顶 push/pop。最近放入的最先取出——"最近相关"。
- **队列（Queue）= FIFO（First In First Out，先进先出）**：一端入队（enqueue），另一端出队（dequeue）。最早放入的最先取出——"公平等待"。

这两种"受限"不是缺点，是刻意的工程约束。栈天然对应"递归调用栈"和"括号匹配"——最近打开的括号最先闭合；队列天然对应"BFS 层序遍历"和"任务调度"——先到先服务。**LIFO 和 FIFO 是两种工作模式，对应两类不同性质的问题**。

### 4. 为什么这四种结构放一起讲

数组和链表是底座（物理层），栈和队列是模式（逻辑层）。栈和队列既可以用数组实现，也可以用链表实现——选择哪种底座取决于场景：

| 结构 | 常见实现底座 | 选择依据 |
|---|---|---|
| 栈 | 数组（Python `list` 的 `append/pop`） | 栈只操作一端，数组缓存友好 |
| 队列 | 链表 或 环形数组 | 队头出队频繁，数组头删 O(n)，所以要么用链表要么用环形数组 |

后文按"底座（数组链表）→ 模式（栈队列）→ 经典操作 → 工程取舍"展开。

## 二、原理与实现：从内存模型到经典操作

### 1. 数组的内存模型：连续分配带来随机访问

数组在内存里是**一段连续的内存块**。声明 `int arr[100]`，操作系统分配 400 字节（假设 int 4 字节）连续空间，`arr[0]` 在基地址 `base`，`arr[i]` 在 `base + 4*i`。

```
地址：  base   base+4  base+8  base+12 ...
内容：  arr[0] arr[1] arr[2] arr[3]  ...
       └────────连续──────────────┘
```

**随机访问 O(1) 的来源**：CPU 算地址 `base + i*4` 是一次乘加，O(1)。不需要遍历，直接定位。

**插入 O(n) 的来源**：在位置 k 插入元素，k 到 n-1 的所有元素要后移一格，最坏 n-k 次。最坏 O(n)。

**删除 O(n) 同理**：删 k 位置元素，后面元素前移。

**CPU 缓存友好**：连续内存被 CPU 以"缓存行"（cache line，64 字节）为单位预取。遍历数组时，读完 `arr[0]`，CPU 顺便把 `arr[1]` 到 `arr[15]`（16 个 int）都拉进 L1 缓存。后续 15 次访问全命中缓存，每次约 1 纳秒；不命中要去主存捞，每次约 100 纳秒。差距是 100 倍量级。

### 2. 链表的内存模型：节点离散，指针串联

链表节点散落在堆内存各处，每个节点存"数据 + 下一个节点地址"。头节点（head）是入口。

```
head ──→ [data|next]   [data|next]   [data|next]   [data|None]
            ↑              ↑              ↑              ↑
         地址 0x100     地址 0x450     地址 0x230     地址 0x780
         （节点离散分布，靠 next 指针串联）
```

**随机访问 O(n) 的来源**：要找第 k 个节点，必须从 head 开始，沿 next 指针走 k 步。无法像数组那样直接算地址。

**插入 O(1) 的来源**：在节点 p 之后插入新节点 x，只需 `x.next = p.next; p.next = x`——两次指针赋值，O(1)。不需要移动其他节点。

**删除 O(1) 的来源**：删节点 p 的下一个节点 q，`p.next = q.next`——一次指针赋值，O(1)。注意是"删下一个节点"O(1)，删"指定节点本身"需要先找到前驱，仍是 O(n)。

**CPU 缓存不友好**：节点散落，每次 `p = p.next` 都可能 cache miss。这就是 §1.2 反直觉问题的根因——链表的 O(1) 操作每次都付 cache miss 的代价，数组的 O(n) 操作却在缓存里飞。

### 3. 数组 vs 链表：复杂度对比表

| 操作 | 数组 | 链表 | 说明 |
|---|---|---|---|
| 随机访问 `arr[i]` | O(1) | O(n) | 数组算地址；链表从头走 |
| 头部插入 | O(n) | O(1) | 数组要全后移；链表改头指针 |
| 尾部插入 | 摊还 O(1) | O(1)（带尾指针）/ O(n)（无尾指针） | 数组 push_back 摊还 O(1) |
| 头部删除 | O(n) | O(1) | 数组全前移；链表改头指针 |
| 中间插入/删除 | O(n) | O(1)（已知前驱节点） | 链表插入本身 O(1)，但找前驱 O(n) |
| 内存 | 紧凑，缓存友好 | 每节点多存指针，缓存不友好 | 链表每个节点多 8 字节（64 位指针） |

**记忆口诀**：数组"查快改慢"，链表"查慢改快"——前提是"改"的位置已知。如果还要先查到位置，链表的"改快"就被"查慢"吃掉了。

### 4. 链表经典操作之一：反转链表（逐行讲透）

反转链表是面试高频题，也是理解链表指针操作的入门题。题目：给定单链表头节点 head，反转整个链表，返回新的头节点。

```python
class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

def reverse_list(head):
    """迭代法反转链表。"""
    prev = None                # 前驱节点，初始为 None（反转后链表的尾指向 None）
    cur = head                 # 当前节点，从头开始
    while cur is not None:     # 走到尾节点为止
        nxt = cur.next         # 暂存下一个节点（因为下一步要改 cur.next）
        cur.next = prev        # 反转指针：cur 指向前驱
        prev = cur             # prev 前进一步
        cur = nxt              # cur 前进一步
    return prev                # 循环结束时 prev 是原链表尾，即新链表头
```

**逐行讲透**（左神风格，每一行都不能含糊）：

- **`prev = None`**：反转后链表的最后一个节点要指向 None（原 head 变成新 tail）。prev 初始为 None，正是为了让原 head 反转后指向 None。
- **`cur = head`**：从原链表头开始遍历。
- **`nxt = cur.next`**：这步至关重要。下一行要执行 `cur.next = prev`，会覆盖 `cur.next`，导致丢失原链表的后续节点。先暂存到 nxt，遍历才能继续。
- **`cur.next = prev`**：反转的核心动作。把 cur 的指针从"指向下一个"改成"指向前一个"。
- **`prev = cur` / `cur = nxt`**：双指针同步前进。prev 始终是 cur 在原链表中的前驱，也始终是 cur 在新链表中的后继。
- **`return prev`**：循环退出时 `cur is None`，prev 是最后一个非空节点，即原链表的尾节点，也是新链表的头节点。

**用例子走一遍**。链表 `1 -> 2 -> 3 -> None`：

```
初始：prev=None, cur=1
第1轮：nxt=2, 1.next=None, prev=1, cur=2   链表：None<-1  2->3->None
第2轮：nxt=3, 2.next=1,  prev=2, cur=3   链表：None<-1<-2  3->None
第3轮：nxt=None, 3.next=2, prev=3, cur=None  链表：None<-1<-2<-3
循环结束（cur is None），返回 prev=3
```

最终链表：`3 -> 2 -> 1 -> None`，反转完成。

### 5. 链表经典操作之二：环形链表检测（快慢指针）

判断链表是否有环。经典解法是 Floyd 快慢指针——慢指针走一步，快指针走两步，若有环必相遇。

```python
def has_cycle(head):
    """快慢指针检测环。"""
    slow = head
    fast = head
    while fast is not None and fast.next is not None:
        slow = slow.next           # 慢指针走 1 步
        fast = fast.next.next      # 快指针走 2 步
        if slow is fast:           # 相遇，有环
            return True
    return False                   # 快指针走到 None，无环
```

**为什么快慢指针一定能相遇？** 设环外长度 a，环长 b。慢指针进环后，快指针每轮比慢指针多走 1 步，相当于快指针在环里以"每轮追 1 步"的速度追赶慢指针。环长 b，最多 b 轮必追上。所以若有环，必在 O(n) 内相遇。

**为什么不能让快指针走 3 步？** 走 3 步在某些环长（如偶数环）上可能"跳过"慢指针永不相遇。走 2 步保证每轮追 1 步，必相遇。这是"步长选 2"的数学依据。

### 6. 链表经典操作之三：找中点（快慢指针）

找链表中点，也是快慢指针：慢指针走 1 步，快指针走 2 步，快指针到尾时慢指针正好在中点。

```python
def find_middle(head):
    """快慢指针找中点。"""
    slow = head
    fast = head
    while fast is not None and fast.next is not None:
        slow = slow.next
        fast = fast.next.next
    return slow    # 偶数长度时返回右中点
```

链表长度 n 时，快指针走 2n 步到尾，慢指针走 n 步——正好到中点。这个操作是归并排序链表版的基础：找中点 → 切两半 → 各自排序 → 合并。

### 7. 链表经典操作之四：合并两个有序链表

合并两个升序链表，是"归并排序链表版"的核心子过程，也是面试常考题。

```python
def merge_two_lists(l1, l2):
    """合并两个有序链表，返回合并后的头节点。"""
    dummy = ListNode(-1)         # 哑节点，简化头节点处理
    tail = dummy                 # tail 始终指向已合并部分的尾
    while l1 is not None and l2 is not None:
        if l1.val <= l2.val:
            tail.next = l1
            l1 = l1.next
        else:
            tail.next = l2
            l2 = l2.next
        tail = tail.next
    # 一方已空，把另一方剩余部分接上
    tail.next = l1 if l1 is not None else l2
    return dummy.next
```

**逐行讲透**：

- **`dummy = ListNode(-1)`**：哑节点（dummy node）。不存真实数据，作用是避免处理"头节点为空"的特殊情况。最终返回 `dummy.next` 即真实头节点。
- **`tail`**：始终指向已合并部分的尾节点。新节点接到 `tail.next`，然后 `tail` 前进。
- **`if l1.val <= l2.val`**：取较小者接上。注意用 `<=` 不用 `<`——保证稳定性（相等时取 l1 的，保持原相对顺序）。
- **`tail.next = l1 if l1 is not None else l2`**：循环结束时一方必为空，另一方剩余节点直接接上（已有序，无需再处理）。

**dummy 节点的价值**：不用 dummy 的话，第一次接节点要单独判断"头是否为空"，代码会多一个 if 分支。dummy 是链表题的通用技巧——只要涉及"构建新链表且头节点待定"，就用 dummy。

### 8. 栈：LIFO 的实现与典型应用

栈只允许在栈顶操作。Python 里用 `list` 的 `append` 和 `pop` 就能模拟栈（均摊 O(1)）。

```python
stack = []
stack.append(1)    # push
stack.append(2)
stack.pop()        # 2，pop 是 O(1)
stack[-1]          # peek 栈顶，O(1)
```

**典型应用：括号匹配**。给一串括号字符串，判断是否合法（每个开括号都有对应闭括号，且嵌套正确）。

```python
def is_valid_parens(s):
    """有效括号：栈匹配。"""
    pair = {')': '(', ']': '[', '}': '{'}
    stack = []
    for ch in s:
        if ch in '([{':
            stack.append(ch)        # 开括号入栈
        elif ch in ')]}':
            if not stack or stack[-1] != pair[ch]:
                return False        # 栈空或不匹配
            stack.pop()             # 匹配，弹出
    return not stack                # 栈空才合法
```

**为什么用栈？** 因为括号是"最近打开最先闭合"——`([{}])` 里 `{` 最后打开但最先闭合，正是 LIFO。栈和括号匹配是天然契合——这就是"工作模式匹配问题性质"。

**典型应用二：表达式求值**。中缀表达式 `1 + 2 * 3` 转 后缀表达式 `1 2 3 * +` 用栈；求值也用栈。运算符优先级靠栈维护。

**典型应用三：函数调用栈**。递归的底层就是栈——每次函数调用压栈，return 弹栈。递归爆栈就是调用栈深度超限。

### 9. 队列：FIFO 的实现与典型应用

队列一端入队，另一端出队。Python 的 `collections.deque` 是双端队列，两端操作均 O(1)。

```python
from collections import deque
q = deque()
q.append(1)         # 入队（右端）
q.append(2)
q.popleft()         # 1，出队（左端），O(1)
```

**为什么不用 `list`？** `list` 的 `pop(0)` 是 O(n)——删头要把后面元素全前移。`deque` 是双向链表/环形数组实现，两端都 O(1)。

**典型应用：BFS 层序遍历**。树的层序遍历、图的最短路径（无权图）都用队列。BFS 的核心是"先访问的节点先扩展"，正是 FIFO。

```python
from collections import deque

def bfs_level_order(root):
    """二叉树层序遍历。"""
    if root is None:
        return []
    q = deque([root])
    result = []
    while q:
        node = q.popleft()        # 出队
        result.append(node.val)
        if node.left:
            q.append(node.left)   # 入队
        if node.right:
            q.append(node.right)
    return result
```

**典型应用二：任务调度**。操作系统的就绪队列、消息队列（RabbitMQ、Kafka）都是 FIFO——保证公平性，先到的任务先处理。

## 三、实践与面试：手撕结构、对数器、高频题

### 1. 手撕结构一：链表反转（迭代 + 递归两种写法）

迭代版见 §二.4，这里讲递归版。递归版的思路：先反转 head.next 之后的链表，再把 head 接到反转后链表的尾。

```python
def reverse_list_recursive(head):
    """递归反转链表。"""
    if head is None or head.next is None:
        return head                       # base case：空或单节点
    new_head = reverse_list_recursive(head.next)   # 反转后面的
    head.next.next = head                 # head.next 现在是反转后链表的尾，让它指回 head
    head.next = None                      # head 变成新尾，指向 None
    return new_head
```

**逐行讲透**：

- **`if head is None or head.next is None`**：base case。空链表或单节点，不需要反转，直接返回。
- **`new_head = reverse_list_recursive(head.next)`**：递归反转 `head.next` 开始的链表。递归返回的是反转后的头节点（即原链表的尾节点）。
- **`head.next.next = head`**：这一行最绕。递归调用前，`head.next` 是 head 的下一个节点。递归调用后，`head.next` 仍然是同一个节点，但现在它是反转后链表的尾节点（因为原 head 还没接进去）。让它的 next 指回 head，就把 head 接到了反转后链表的尾。
- **`head.next = None`**：head 现在是新链表的尾，next 必须指向 None，否则成环。
- **`return new_head`**：一路向上传递反转后的头节点。

**迭代 vs 递归的取舍**：迭代 O(n) 时间 O(1) 空间；递归 O(n) 时间 O(n) 空间（递归栈）。生产用迭代，面试时递归写法展示"递归思维"——能讲清"递归信任"（trust the recursion）就过关。

### 2. 手撕结构二：环形链表检测 + 找环入口

检测环见 §二.5，这里进阶：找到环的入口节点。

```python
def detect_cycle_entry(head):
    """找到环的入口节点，无环返回 None。"""
    slow = head
    fast = head
    # 阶段一：快慢指针找相遇点
    while fast is not None and fast.next is not None:
        slow = slow.next
        fast = fast.next.next
        if slow is fast:
            # 阶段二：从头出发的指针和相遇点出发的指针，会在环入口相遇
            ptr = head
            while ptr is not slow:
                ptr = ptr.next
                slow = slow.next
            return ptr
    return None
```

**阶段二的数学证明**：设环外长度 a，环长 b，相遇点距环入口 c。慢指针走了 `a + c`，快指针走了 `a + c + n*b`（n 是快指针多绕的圈数）。快指针速度是慢指针 2 倍：`2(a+c) = a + c + n*b`，化简得 `a = n*b - c`。也就是说，从 head 走 a 步 = 从相遇点走 `n*b - c` 步 = 绕环 n 圈再退 c 步 = 正好到环入口。所以两个指针一个从 head、一个从相遇点，同步走，必在环入口相遇。

这个证明是面试加分点——能讲清数学，区分你和"只会背模板"的候选人。

### 3. 对数器：对比手写链表与 Python list

左神风格：写完链表操作，用对数器验证。链表没有标准库可直接对照，但可以**对比手写链表与 Python list 的行为**——把同样的操作分别在链表和 list 上跑，结果应一致。

```python
import random

class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

def list_to_linked(lst):
    """Python list 转链表。"""
    dummy = ListNode()
    tail = dummy
    for v in lst:
        tail.next = ListNode(v)
        tail = tail.next
    return dummy.next

def linked_to_list(head):
    """链表转 Python list。"""
    result = []
    while head:
        result.append(head.val)
        head = head.next
    return result

def reverse_list(head):
    """手写链表反转。"""
    prev = None
    cur = head
    while cur:
        nxt = cur.next
        cur.next = prev
        prev = cur
        cur = nxt
    return prev

def checker(test_times=5000, max_n=100, max_val=1000):
    """对数器：对比手写链表反转与 list 反转。"""
    for _ in range(test_times):
        n = random.randint(0, max_n)
        lst = [random.randint(0, max_val) for _ in range(n)]
        # 手写链表反转
        head = list_to_linked(lst)
        reversed_head = reverse_list(head)
        result_linked = linked_to_list(reversed_head)
        # Python list 反转（绝对正确）
        result_list = lst[::-1]
        if result_linked != result_list:
            print(f"出错！输入: {lst}, 链表结果: {result_linked}, list 结果: {result_list}")
            return False
    print("对数器验证通过！")
    return True

checker()
```

**对数器的价值**：

- **覆盖边界**：随机长度包含 0 和 1（空链表和单节点），是手写最容易出错的地方。
- **不依赖人工**：跑 5000 次几秒内完成，比手动构造用例全面得多。
- **通用思路**：任何手写数据结构都能这样验证——手写栈对比 Python list 的 append/pop，手写队列对比 deque。这是工程化验证思维，比"理论上推导正确"更可靠。

### 4. 面试高频题一：LRU 缓存（链表 + 哈希）

LRU（Least Recently Used，最近最少使用）缓存是面试超高频题。要求：`get(key)` 和 `put(key, value)` 均 O(1)，容量超限时淘汰最久未使用的。

**思路**：哈希表 O(1) 查找 + 双向链表 O(1) 调整顺序。

- 哈希表存 `key -> 链表节点`，O(1) 找到节点。
- 双向链表按"最近使用顺序"排列，头是最近用的，尾是最久未用的。
- `get`/`put` 时把节点移到链表头；`put` 超容量时删链表尾。

```python
class DLinkedNode:
    def __init__(self, key=0, value=0):
        self.key = key
        self.value = value
        self.prev = None
        self.next = None

class LRUCache:
    def __init__(self, capacity: int):
        self.capacity = capacity
        self.cache = {}                          # key -> DLinkedNode
        # 哑节点，省去头尾判空
        self.head = DLinkedNode()                # 哑头
        self.tail = DLinkedNode()                # 哑尾
        self.head.next = self.tail
        self.tail.prev = self.head

    def _remove(self, node):
        """从链表中删除节点。"""
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_to_head(self, node):
        """把节点插到头（head 之后）。"""
        node.next = self.head.next
        node.prev = self.head
        self.head.next.prev = node
        self.head.next = node

    def _move_to_head(self, node):
        """节点移到头：先删再加。"""
        self._remove(node)
        self._add_to_head(node)

    def get(self, key: int) -> int:
        if key not in self.cache:
            return -1
        node = self.cache[key]
        self._move_to_head(node)                 # 访问过，移到头
        return node.value

    def put(self, key: int, value: int) -> None:
        if key in self.cache:
            node = self.cache[key]
            node.value = value
            self._move_to_head(node)
        else:
            node = DLinkedNode(key, value)
            self.cache[key] = node
            self._add_to_head(node)
            if len(self.cache) > self.capacity:
                # 淘汰尾节点（tail.prev 是真实尾）
                removed = self.tail.prev
                self._remove(removed)
                del self.cache[removed.key]
```

**为什么用双向链表不用单向？** 因为删除节点要知道前驱。单向链表删节点要 O(n) 找前驱；双向链表节点自带 prev 指针，O(1)。

**为什么用哑头哑尾？** 避免"链表为空"或"操作真实头/尾"时的判空分支。`head` 和 `tail` 永远存在，真实节点都在 `head` 和 `tail` 之间。这是链表题的通用技巧——只要频繁操作头尾，就用哑节点。

**LRU 的工程意义**：Redis 的 `maxmemory-policy=allkeys-lru`、MySQL Buffer Pool 的 LRU 变体、操作系统页面置换——都是 LRU 思想。理解这题，就理解了一大类缓存淘汰策略。

### 5. 面试高频题二：滑动窗口最大值（单调队列）

给数组 nums 和窗口大小 k，窗口从左滑到右，返回每个窗口的最大值。要求 O(n)。

**朴素法**：每个窗口扫一遍找最大，O(n*k)。k 接近 n 时退化 $O(n^2)$。

**单调队列法**：维护一个**双端队列**，存"可能成为最大值"的元素下标，队列从队头到队尾递减。

```python
from collections import deque

def max_sliding_window(nums, k):
    """单调队列：队头始终是当前窗口最大值的下标。"""
    dq = deque()                          # 存下标，对应值单调递减
    result = []
    for i, num in enumerate(nums):
        # 1. 队头出窗口，弹出
        while dq and dq[0] <= i - k:
            dq.popleft()
        # 2. 队尾比当前小的，弹出（它们再也不会成为最大值）
        while dq and nums[dq[-1]] <= num:
            dq.pop()
        # 3. 当前下标入队
        dq.append(i)
        # 4. 窗口形成后，队头就是最大值
        if i >= k - 1:
            result.append(nums[dq[0]])
    return result
```

**逐行讲透**：

- **`while dq and dq[0] <= i - k`**：队头下标如果超出窗口左边界（`i - k + 1`），弹掉。注意 `dq[0] <= i - k` 等价于 `dq[0] < i - k + 1`。
- **`while dq and nums[dq[-1]] <= num`**：当前元素 num 比队尾大，队尾再也没机会当最大值（num 既比它大又比它晚出窗口），弹掉。这一步保证队列单调递减。
- **`dq.append(i)`**：当前下标入队。
- **`if i >= k - 1`**：前 k-1 个元素时窗口还没形成，不输出。从下标 k-1 开始输出队头。

**为什么是 O(n)？** 每个元素最多入队一次、出队一次，均摊 O(1)。

**单调队列的本质**：用 O(1) 维护"滑动窗口内的最大值"。它能推广到"滑动窗口最小值"（队列改单调递增）、"滑动窗口第 k 大"等。这是面试加分点——能讲"单调队列维护的是候选集，淘汰的是再无机会的元素"，区分你和"只会背模板"的候选人。

### 6. 教科书做法 vs 生产做法

| 场景 | 教科书做法 | 生产做法 | 原因 |
|---|---|---|---|
| 动态数组 | 手写 `ArrayList` | Python `list` / C++ `vector` | 标准库已优化，扩容策略成熟 |
| 链表 | 手写单/双向链表 | 极少手写 | 数组缓存友好，链表生产用得少 |
| 栈 | 手写数组栈 | Python `list.append/pop` | 标准库即数组栈，O(1) |
| 队列 | 手写链表队列 | `collections.deque` | deque 双端 O(1)，已优化 |
| LRU 缓存 | 手写双向链表+哈希 | Python 3.7+ 用 `OrderedDict` | `OrderedDict.move_to_end` + `popitem(last=False)` 直接实现 |

**生产里手写链表几乎绝迹**，原因就是 §二.2 讲的缓存不友好。但两个场景仍要懂：

1. **面试**：链表题是高频考点，反转/检测环/找中点/合并是基本功。
2. **理解 LRU/跳表/图邻接表**：这些高级数据结构以链表为底座，不懂链表无法理解。

### 7. 三个真实工程坑

**坑一：用 Python `list` 的 `pop(0)` 当队列。**

```python
q = [1, 2, 3]
q.pop(0)   # O(n)，后面元素全前移
```

症状：队列规模大时性能急剧下降。根因：`list` 是数组，`pop(0)` 要把 1 到 n-1 元素全前移。修复：用 `collections.deque`，`popleft()` 是 O(1)。

**坑二：链表反转忘记暂存 next，导致死循环或断链。**

```python
# 错误写法
cur.next = prev
cur = cur.next   # 此时 cur.next 已经被改成 prev，死循环！
```

症状：链表反转后死循环或链表断裂。根因：改 `cur.next` 前没暂存原 next。修复：先 `nxt = cur.next`，再改 `cur.next`，最后 `cur = nxt`。

**坑三：LRU 用单向链表，删除节点 O(n)。**

```python
# 错误：单向链表删节点要找前驱
def remove(node):
    prev = head
    while prev.next is not node:   # O(n) 找前驱
        prev = prev.next
    prev.next = node.next
```

症状：LRU 的 get/put 退化到 O(n)，不满足 O(1) 要求。根因：单向链表无 prev 指针。修复：用双向链表，节点自带 prev，删除 O(1)。

## 四、速查与自测

### 速查表：四种线性结构复杂度对比

| 操作 | 数组 | 链表 | 栈（数组实现） | 队列（deque） |
|---|---|---|---|---|
| 随机访问 | O(1) | O(n) | O(n)（非栈顶） | O(n)（非队头尾） |
| 头部插入 | O(n) | O(1) | — | O(1)（入队） |
| 头部删除 | O(n) | O(1) | — | O(1)（出队） |
| 尾部插入 | 摊还 O(1) | O(1)（带尾指针） | O(1)（push） | — |
| 尾部删除 | O(1) | O(n)（无尾指针前驱）/ O(1)（双向带尾指针） | O(1)（pop） | — |
| 中间插入（已知位置） | O(n) | O(1) | — | — |
| 缓存友好性 | 好 | 差 | 好（数组实现） | 中（deque） |

**链表四件套速查**：

| 操作 | 算法 | 时间复杂度 | 关键技巧 |
|---|---|---|---|
| 反转链表 | 迭代/递归 | O(n) | 双指针 prev/cur，暂存 next |
| 检测环 | Floyd 快慢指针 | O(n) | 慢 1 步快 2 步，相遇即有环 |
| 找环入口 | 快慢指针+双指针 | O(n) | 相遇后从头出发，与慢指针同步走 |
| 找中点 | 快慢指针 | O(n) | 慢 1 步快 2 步，快到尾慢在中点 |
| 合并有序 | 双指针+dummy | O(n+m) | dummy 节点简化头处理 |

### 自测三问

**问题一：** 为什么数组随机访问 O(1) 而链表 O(n)？请从内存模型解释。

**参考答案：** 数组在内存中是一段连续空间，元素地址可通过 `base + i * sizeof(elem)` 直接计算，CPU 一次乘加定位，O(1)。链表节点散落在堆各处，每个节点只存"下一个节点地址"，找第 k 个节点必须从 head 沿 next 指针走 k 步，O(n)。这个差异的根源是物理约束——连续分配 vs 离散分配。连带效应是 CPU 缓存友好性：连续内存能被缓存行预取，离散内存每次访问都可能 cache miss。

**问题二：** 反转链表时为什么要先 `nxt = cur.next` 再 `cur.next = prev`？直接 `cur.next = prev; cur = cur.next` 会怎样？

**参考答案：** `cur.next = prev` 覆盖了 cur 原本的 next 指针，原链表的后续节点地址就丢了。如果不先暂存到 nxt，执行 `cur = cur.next` 时 cur.next 已经是 prev，cur 会跳回 prev，陷入死循环或断链。先 `nxt = cur.next` 暂存，改完 `cur.next = prev` 后 `cur = nxt` 才能继续往后走。这是链表指针操作的核心技巧——**改指针前先暂存会被覆盖的值**。

**问题三：** LRU 缓存为什么用双向链表不用单向？为什么用哑头哑尾？

**参考答案：** 双向链表每个节点带 prev 指针，删除节点时 O(1) 找到前驱（直接 `node.prev`），满足 LRU 的 O(1) 要求。单向链表删节点要遍历找前驱，O(n)，破坏 LRU 性能。哑头哑尾的作用是省去"链表为空"或"操作真实头尾"时的判空分支——`head` 和 `tail` 永远存在，真实节点都在两者之间，所有插入/删除操作写一套代码即可，不用特判边界。这是链表题的通用技巧：只要频繁操作头尾，就用哑节点。

### 算法思想 × 生活迁移

这章的核心思想是"**物理约束决定能力边界，受限接口催生工作模式**"——数组链表因物理约束各有所长，栈队列因受限接口各自适配不同问题。

**迁移一：书架 vs 抽屉——收纳方式的取舍。** 书架（数组式）每本书固定位置，找书 O(1)，但要在中间插一本书得把后面的全挪一遍。抽屉（链表式）东西随便放，找东西得翻一遍 O(n)，但加东西往里塞就行。日常收纳的取舍和数据结构一样——看你更在乎"找得快"还是"加得快"。

**迁移二：单线程调用栈 vs 多线程任务队列——两种工作模式。** 单线程调用栈是 LIFO——最后调用的函数最先返回，对应"最近相关"。多线程任务队列是 FIFO——先提交的任务先执行，对应"公平等待"。两种工作模式的选择不是技术问题，是问题本身的性质——递归天然 LIFO，调度天然 FIFO。

**迁移三：API 设计中的"受限接口"。** 栈和队列是"受限的线性表"——只暴露一端操作。这种"刻意受限"在 API 设计中常见：只读集合（不允许增删）、不可变对象（不允许修改）、Builder 模式（分步构造）。受限不是缺陷，是工程约束——把"能做什么"限定在"该做什么"之内，能避免误用。栈只让你 push/pop，你就不会误用成"中间插入"。

**为什么这些迁移成立：** 这些迁移成立的前提是"物理约束或接口约束真实存在且影响行为"。书架 vs 抽屉的物理约束真实影响找东西的效率；单线程 vs 多线程的执行模式真实决定任务顺序；API 受限真实影响使用方式。如果约束是人为虚设的（比如本来该用队列却强行用栈），迁移就不成立——约束要匹配问题性质，这是选择数据结构的基本依据。
