# 算法课·02｜二分查找：为什么"折半"能省下 99% 的工作

> 前置知识：学完第 01 章（为什么要学算法）即可，最好懂一点 Python
> 学完你能：①讲清二分查找的本质是"单调性 + 折半"而非"有序数组查找" ②默写出不会溢出、不会死循环的二分 ③识别二分的三个变体（精确查找/左边界/右边界）④把二分思想用到非查找场景

## 一、问题与思路：字典查词的痛苦，逼出了折半直觉

人类为什么会想到二分？

不是为了应付面试，是因为真实世界里有种痛：**在一堆有序的东西里找某一个，从头翻到尾太慢了。**

最直观的例子是查字典。一本《新华字典》收词约 1.1 万页，假设你要查"算法"两个字。最笨的办法是从第 1 页一页页翻——平均要翻 5500 页。没人这么干。

所有人都会直觉地做一件事：**翻到中间，看一眼。** 如果中间这一页是"法"字开头，你要找的"算法"在"法"之前，那就把后半本扔掉，在前半本里继续翻中间。再翻中间，再扔一半。每翻一次，剩下的页数砍一半。

一本 1.1 万页的字典，按这个办法，最多翻几次？算一下：$2^{13} = 8192$，$2^{14} = 16384$。所以最多翻 14 次。从 5500 页砍到 14 次，省下了 99.7% 的工作。

这就是二分查找（Binary Search）的起源——**不是某个天才拍脑袋发明的，是所有人在面对"有序集合里找东西"时都会直觉想到的策略。** 算法书上把它形式化、精确化了而已。

这里有一个关键的本质：二分能成立的前提，不是"数组有序"，而是**"每一步都能排除一半的可能性"**。字典能折半，是因为字典按拼音排序——你看到中间页，就知道目标在前半还是后半。这种"看了中间就能判断方向"的性质，叫**单调性**（Monotonicity）。

记住这一点。很多人把二分理解成"在有序数组里找数"，这是第一层的理解。第二层的理解是：**任何场景，只要满足"单调性"——即能根据中间点的结果判断答案在左半还是右半——都能用二分。** 数组有序只是单调性最常见的一种。后面会看到，二分答案、浮点二分、二分求根，都是同一种思想的不同外衣。

## 二、原理与实现：从直觉到不会写错的代码

### 1. 最朴素的二分：精确查找

先写最基础的版本：在一个升序数组里找一个 target，找到返回下标，找不到返回 -1。

```python
def binary_search(nums, target):
    left, right = 0, len(nums) - 1   # 闭区间 [left, right]
    while left <= right:              # 区间非空就继续
        mid = left + (right - left) // 2   # 防溢出，见下文
        if nums[mid] == target:
            return mid
        elif nums[mid] < target:
            left = mid + 1            # target 在右半，收缩左边界
        else:
            right = mid - 1           # target 在左半，收缩右边界
    return -1
```

这段代码看着简单，但每一行都藏着坑。**逐行讲透**（左神风格，每一行都不能含糊）：

- **`left, right = 0, len(nums) - 1`**：初始化闭区间 `[left, right]`，左右端点都可能是答案。这是"闭区间写法"的根基——后面所有边界都由它决定。如果数组为空，`len(nums) - 1 = -1`，`left = 0 > right = -1`，循环不进入，直接返回 -1，正确。
- **`while left <= right`**：区间非空就继续。用 `<=` 不是 `<`，因为当 `left == right` 时区间里还有一个元素没检查，必须进入循环。用 `<` 会漏掉这个元素。
- **`mid = left + (right - left) // 2`**：防溢出版本求中点。不用 `(left + right) // 2`，因为在 Java/C++ 里 `left + right` 可能超过 `int` 上限（比如 `left = 2^30`，`right = 2^30`，和是 `2^31`，溢出成负数，`mid` 为负导致数组越界）。`left + (right - left) // 2` 永远不溢出。这是 Java 标准库 `Arrays.binarySearch` 在 2006 年之前的真实 bug，被 Google 的 Joshua Bloch 公开指出。**写二分时永远用防溢出版本，是肌肉记忆。**
- **`if nums[mid] == target: return mid`**：命中直接返回下标。
- **`elif nums[mid] < target: left = mid + 1`**：mid 位置的值偏小，target 在右半，收缩左边界到 `mid + 1`。用 `mid + 1` 不是 `mid`，因为 `nums[mid]` 已经检查过且不等于 target，mid 这个位置可以排除。写 `left = mid` 会死循环——当 `left + 1 == right` 时，`mid = left`，永远跳不出来。
- **`else: right = mid - 1`**：mid 位置的值偏大，target 在左半，收缩右边界。同理用 `mid - 1` 排除 mid。
- **`return -1`**：循环正常结束（`left > right`，区间空）说明没找到，返回 -1。

### 2. 复杂度：为什么是 O(log n)

每轮循环，区间长度砍半。n 个元素，砍多少次剩 1 个？

$$n \to n/2 \to n/4 \to \dots \to 1$$

砍 k 次后剩 $n / 2^k$ 个。令 $n / 2^k = 1$，得 $k = \log_2 n$。所以时间复杂度是 $O(\log n)$，空间复杂度 $O(1)$（只用了几个变量）。

这个 $O(\log n)$ 有多快？10 亿元素，$\log_2(10^9) \approx 30$。30 次操作就能在 10 亿个有序数据里找到目标。这就是二分的威力——它不是把工作"加速"，是把工作"砍掉 99%"。

### 3. 二分的三个变体：左边界、右边界、精确查找

上面写的是"精确查找"——找到就返回。但工程里更常见的两个变体是：

- **找左边界**：数组里有重复元素，找第一个 ≥ target 的位置（lower_bound）
- **找右边界**：找最后一个 ≤ target 的位置（upper_bound 的变体）

这两个变体是二分真正容易写错的地方。LeetCode 上二分题 70% 的通过率低，不是题难，是边界写不对。我给一个**统一模板**，三个变体只改一行：

```python
def binary_search_variant(nums, target):
    left, right = 0, len(nums) - 1
    while left <= right:
        mid = left + (right - left) // 2
        if nums[mid] < target:        # 注意是 <，不是 <=
            left = mid + 1
        else:
            right = mid - 1
    # 循环结束时，left 就是第一个 >= target 的位置
    return left
```

这个模板找的是"第一个 ≥ target 的下标"（即 C++ 的 `lower_bound`）。它有三个特点：

1. `if nums[mid] < target` 用严格小于，等于时不收缩左边界，而是收缩右边界——这样 left 会停在第一个 ≥ target 的位置。
2. 循环结束后，`left` 就是答案（无需在循环里 return）。
3. 如果所有元素都 < target，`left` 会等于 `len(nums)`（越界，表示"没找到"）。

为什么这个模板好？因为它**只有一处分支判断**（`<` 还是 `<=` 决定找左边界还是精确查找），不会死循环，不需要记忆多种写法。左神（左程云）在课上反复强调："二分不要背多种模板，背一个能覆盖所有变体的模板。" 这是经验之谈。

### 4. 二分的本质：单调性，不是有序

回到本质。二分能成立的根本条件是单调性——"看中间能判断方向"。有序数组只是单调性的一种。来看几个非"有序数组查找"的二分场景：

**场景一：二分答案。** 题：把数组分成 K 段，使各段和的最大值最小。这个"最大值"在什么范围？最小是数组最大元素，最大是数组总和。答案在这个区间里有单调性——"最大值上限越大，需要的段数越少"。所以可以二分这个上限，对每个 mid 算需要几段，判断是否 ≤ K。这是二分思想从"查找值"扩展到"搜索答案空间"。

**场景二：浮点二分。** 求 $\sqrt{2}$ 的近似值。答案在 `[0, 2]` 之间，`mid` 平方后大于 2 就往左，小于 2 就往右。精度到 $10^{-6}$ 就停。浮点二分和整数二分本质一样，只是终止条件从"区间空"变成"精度够"。

**场景三：求函数零点。** 只要函数在区间内单调且变号，就能二分求零点。这是数值分析里二分法（Bisection Method）的源头。

这三个场景的共同点：**都能用"中间点的结果判断方向"来排除一半搜索空间。** 这才是二分的本质。把这个本质内化，你看到任何"答案在一个范围内、且有方向性判断"的问题，都会直觉想到二分。

### 5. 高手向：三分法——二分思想求极值

二分求"零点"（$f(x) = 0$），三分求"极值"（$f'(x) = 0$）。如果函数 $f$ 在区间 $[l, r]$ 上**严格单峰**（先增后减，或先减后增），可以用三分法（Ternary Search）求极值点。

```python
def ternary_search_max(f, left, right, eps=1e-7):
    """三分法求单峰函数 f 在 [left, right] 上的极大值点。"""
    while right - left > eps:
        m1 = left + (right - left) / 3
        m2 = right - (right - left) / 3
        if f(m1) < f(m2):
            left = m1       # 极大值在 [m1, right]
        else:
            right = m2      # 极大值在 [left, m2]
    return (left + right) / 2
```

**逐行讲透**：

- **`m1 = left + (right - left) / 3`**：取左三等分点。
- **`m2 = right - (right - left) / 3`**：取右三等分点。`m1 < m2`，把区间三等分。
- **`if f(m1) < f(m2): left = m1`**：若 `f(m1) < f(m2)`，说明极大值点不可能在 `[left, m1]`（否则 `f` 应该在 `[m1, m2]` 递减，与 `f(m1) < f(m2)` 矛盾），所以极值在 `[m1, right]`，收缩左边界到 `m1`。
- **`else: right = m2`**：对称情况，极值在 `[left, m2]`。
- **`return (left + right) / 2`**：区间足够小，取中点作为近似。

**复杂度**：每轮区间缩减 $2/3$，所以迭代次数是 $\log_{1.5}((r-l)/\text{eps})$，比二分的 $\log_2$ 略慢（同样区间三分要更多次迭代），但能求极值——这是二分做不到的。

**三分法的陷阱**：

1. **平台区失效**。如果 $f$ 在某段是常数（平台），`f(m1) == f(m2)` 时无法判断方向。严格单峰要求没有平台，工程上遇到平台要改用黄金分割法（0.618 法）或退化为离散扫描。
2. **离散三分**。求离散单峰序列（先增后减的数组）的峰值下标时，三分法同样适用，但要用整数版本：`m1 = l + (r - l) // 3`，循环条件 `while r - l > 2`，最后在剩余 2-3 个元素里取最大。LeetCode 162（寻找峰值）的"严格递增后严格递减"变体就是经典应用。
3. **与二分的关系**。二分和三分本质同源——都是"用区间内的探针信息判断极值/零点在哪一侧"。二分用 1 个探针（中点），三分用 2 个探针（三等分点）。如果函数可导，求极值可以转化为"求 $f'(x)$ 的零点"，这时三分退化为二分——但前提是你能算导数。

**对高手的启发**：看到"求极值"问题，先判断函数是否单峰。单峰用三分，多峰要先用爬山/模拟退火，凸函数可以用黄金分割法或牛顿法。三分法是"无导数优化"里最朴素的一种，理解它能帮你建立"探针法优化"的直觉。

## 三、实践与面试：教科书手写 vs 生产调库

### 1. 教科书做法 vs 生产做法

| 场景 | 教科书做法 | 生产做法 | 何时需要手写 |
|---|---|---|---|
| 有序数组查精确值 | 手写 `binary_search` | `bisect.bisect_left` 后判断 | 几乎不需要，除非是面试 |
| 找第一个 ≥ target | 手写左边界二分 | `bisect.bisect_left(nums, target)` | 库够用 |
| 找最后一个 ≤ target | 手写右边界二分 | `bisect.bisect_right(nums, target) - 1` | 库够用 |
| 二分答案（非查找） | 必须手写 | 无现成库 | **必须手写**——库帮不了你 |

生产里 90% 的二分场景，Python 标准库 `bisect` 都能解决。`bisect` 模块就是 C 实现的二分，比你手写快且不会写错边界。用法：

```python
import bisect

nums = [1, 2, 2, 2, 3, 5, 7]
target = 2

# 第一个 >= 2 的下标
idx_left = bisect.bisect_left(nums, target)   # 返回 1

# 第一个 > 2 的下标（即最后一个 <= 2 的下标 + 1）
idx_right = bisect.bisect_right(nums, target)  # 返回 4

# 2 在 nums 里的范围是 [idx_left, idx_right)
# 最后一个 2 的下标是 idx_right - 1 = 3
```

记住这个规律：`bisect_left` 返回"插入点"（第一个 ≥ target 的位置），`bisect_right` 返回"最后一个 ≤ target 的位置 + 1"。两者配合能定位任何边界。

### 2. 三个真实工程坑

**坑一：在没排序的数组上用二分。**

```python
nums = [3, 1, 4, 1, 5, 9, 2, 6]
idx = bisect.bisect_left(nums, 4)   # 返回 4，但 nums[4]=5 不是 4
```

症状：返回的下标指向的元素不是 target。根因：二分依赖单调性，数组无序时单调性不成立。修复：先 `nums.sort()`，但注意排序是 $O(n\log n)$，如果只查一次还不如线性扫描。**二分适用于"一次排序、多次查询"的场景。**

**坑二：浮点二分的精度与死循环。**

```python
# 求 sqrt(2)，错误写法
left, right = 0.0, 2.0
while left < right:          # 错！浮点数永远 left < right，死循环
    mid = (left + right) / 2
    if mid * mid < 2:
        left = mid
    else:
        right = mid
```

症状：程序卡死。根因：浮点数有精度，`left` 和 `right` 可能永远不相等。修复：用精度作为终止条件。

```python
left, right = 0.0, 2.0
eps = 1e-7
while right - left > eps:
    mid = (left + right) / 2
    if mid * mid < 2:
        left = mid
    else:
        right = mid
return left
```

**坑三：二分答案时的边界判断写反。**

二分答案题里，最常见错误是"check 函数返回 True 时往哪边收缩"搞反。规律：**如果 check(mid) 为 True 表示 mid 可行，且要求"最小的可行值"，则往左收缩（`right = mid`）；要求"最大的可行值"，则往右收缩（`left = mid`）。** 写反了要么答案错，要么死循环。建议每道二分答案题，先在草稿上画清楚"可行域在哪边"。

### 3. 对数器：左神风格的验证方法

左神（左程云）反复强调：**写完二分不要靠肉眼检查，要用对数器验证**。对数器的思路是——生成随机数据，用你的二分和标准库（绝对正确）对比，跑足够多次，只要有一次不一致就说明你的二分有 bug。

下面这个对数器验证前面第 3 小节的"统一模板"（找第一个 ≥ target），对比 Python 标准库 `bisect.bisect_left`：

```python
import random
import bisect

def lower_bound(nums, target):
    """手写二分：找第一个 >= target 的下标（统一模板）。"""
    left, right = 0, len(nums) - 1
    while left <= right:
        mid = left + (right - left) // 2
        if nums[mid] < target:
            left = mid + 1
        else:
            right = mid - 1
    return left   # 第一个 >= target 的下标，全小于则返回 len(nums)

def checker(test_times=5000, max_n=200, max_val=1000):
    """对数器：跑 5000 次，每次随机数据，对比 bisect.bisect_left。"""
    for _ in range(test_times):
        n = random.randint(0, max_n)              # 随机长度（含 0 和 1）
        arr = [random.randint(0, max_val) for _ in range(n)]
        arr.sort()                                 # 二分前提：有序
        # target 故意取数组外的值，覆盖"全小于"和"全大于"两种越界情况
        target = random.randint(-5, max_val + 5)
        mine = lower_bound(arr, target)
        std = bisect.bisect_left(arr, target)
        if mine != std:
            print(f"出错！输入: arr={arr}, target={target}, 我的: {mine}, 标准库: {std}")
            return False
    print("对数器验证通过！5000 次随机数据全部一致。")
    return True

checker()
```

**对数器的价值**：

- **覆盖边界**：随机长度包含 0 和 1（空数组和单元素数组），target 取值范围故意超出数组元素范围，能覆盖手写时容易遗漏的"全小于 target"（应返回 0）和"全大于 target"（应返回 `len(arr)`）两种越界情况。
- **对比标准库**：`bisect.bisect_left` 是 C 实现的绝对正确版本，作为基准，不容辩驳。
- **不依赖人工**：跑 5000 次只要零点几秒，比手动构造测试用例全面得多。

**左神原话**："对数器是算法工程师的基本功。写完一个算法，第一件事就是写对数器，而不是跑几个例子看看对不对。"

**对高手的启发**：对数器思想不止用于排序——任何"有标准答案可对照"的算法都能用对数器。写完二分？对数器对比 `bisect`。写完红黑树？对数器对比 `std::map`。写完哈希表？对数器对比 `dict`。这是"工程化验证"的思维，比"理论上推导正确"更可靠。

### 4. 面试手撕模板：按"技术的水有多深"八层面讲

面试官让你"手写二分查找"，不要只写代码就走。按"技术的水有多深"八层面逐层展开，能立刻区分你和"只会背模板"的候选人：

```python
def lower_bound(nums, target):
    """面试手撕模板：找第一个 >= target 的下标（统一模板）。"""
    left, right = 0, len(nums) - 1
    while left <= right:
        mid = left + (right - left) // 2
        if nums[mid] < target:
            left = mid + 1
        else:
            right = mid - 1
    return left
```

**八层面讲解**（边写边讲，挑重点展开）：

1. **问题层**（本质）：在有序集合里定位目标，朴素线性扫描 $O(n)$，二分利用单调性每步砍一半，做到 $O(\log n)$。本质不是"有序数组查找"，是"每步排除一半可能性"。
2. **原理层**：闭区间 `[left, right]`，每轮取中点 `mid`，根据 `nums[mid]` 与 `target` 的关系判断目标在左半还是右半，收缩区间到一半。
3. **优劣层**：时间 $O(\log n)$、空间 $O(1)$、不递归（不会栈溢出）；但要求**有序**（排序成本 $O(n\log n)$）和**随机访问**（链表上二分退化为 $O(n)$，要用跳表）。适用于"一次排序、多次查询"。
4. **演进层**：二分查找最早 1946 年出现在文献中，但**正确实现直到 1962 年才出现**——中间 16 年里几乎所有教科书版本都有边界 bug。2006 年 Joshua Bloch 又指出 Java 标准库的 `mid` 溢出 bug。一个 10 行的算法，正确实现用了 60 年。
5. **实现层**：`while left <= right`（闭区间用 `<=`），`mid = left + (right - left) // 2`（防溢出），`left = mid + 1` / `right = mid - 1`（必须排除 mid，否则死循环）。这三处是二分所有 bug 的高发区。
6. **边界层**：统一模板找"第一个 ≥ target"（lower_bound）。改 `<` 为 `<=` 可退化为精确查找；找"最后一个 ≤ target"用 `bisect_right - 1`。空数组、target 小于所有元素（返回 0）、target 大于所有元素（返回 `len(nums)`）三种边界都要对。
7. **工程层**：生产查有序数组用 `bisect`（C 实现，快且不会写错边界）；二分答案、浮点二分、自定义谓词必须手写——库帮不了你。原则：**库能解决的不手写，库解决不了的必须会手写。**
8. **迁移层**：二分思想在工程里到处都是——`git bisect` 定位引入 bug 的 commit、APM 工具的链路追踪定位慢接口、灰度发布按 1%→10%→50%→100% 定位问题流量。本质都是"用中点信息判断方向"。

**面试加分点**：主动提"二分 1946 年提出、1962 年才有正确实现"，能立刻展示你知道这段历史；主动说"本质是单调性不是有序"，能展示你理解第二层。这两点足够区分你和背模板的候选人。

**面试官追问"二分答案怎么写"**：先画可行域——`check(mid)` 为 True 表示 mid 可行，求"最小可行值"往左收缩（`right = mid`），求"最大可行值"往右收缩（`left = mid`）。注意二分答案的循环条件通常是 `while left < right`（不是 `<=`），且收缩用 `mid` 不是 `mid ± 1`（因为 mid 可能就是答案，不能排除）。

### 5. 一个生产级例子：日志时间范围查询

```python
import bisect

class LogIndex:
    """对按时间排序的日志，支持快速时间范围查询。"""
    def __init__(self, logs):
        # logs 是 [(timestamp, content), ...]，假设已按 timestamp 升序
        self.times = [t for t, _ in logs]
        self.logs = logs

    def range_query(self, start, end):
        """返回 timestamp 在 [start, end) 内的所有日志。"""
        # 第一个 >= start 的下标
        lo = bisect.bisect_left(self.times, start)
        # 第一个 >= end 的下标
        hi = bisect.bisect_left(self.times, end)
        return self.logs[lo:hi]

# 使用
logs = [(1, "a"), (3, "b"), (5, "c"), (7, "d"), (9, "e")]
idx = LogIndex(logs)
print(idx.range_query(3, 8))  # [(3,'b'), (5,'c'), (7,'d')]
```

这个例子体现了二分的生产价值：**一次构建索引 $O(n)$（或排序 $O(n\log n)$），之后每次查询 $O(\log n + k)$（k 是结果数）。** 比线性扫描快几个数量级。Elasticsearch、数据库索引，本质都是这种"预排序 + 二分定位"的思路。

## 四、速查与自测

### 速查表：二分查找复杂度与模板

| 操作 | 时间复杂度 | 空间复杂度 |
|---|---|---|
| 精确查找 | $O(\log n)$ | $O(1)$ |
| 左边界（lower_bound） | $O(\log n)$ | $O(1)$ |
| 右边界（upper_bound） | $O(\log n)$ | $O(1)$ |
| 二分答案（check 函数 O(f)） | $O(f \cdot \log(\text{值域}))$ | $O(1)$ |
| 浮点二分（精度 eps） | $O(\log(\text{值域}/\text{eps}))$ | $O(1)$ |
| 三分法（单峰求极值） | $O(\log_{1.5}(\text{值域}/\text{eps}))$ | $O(1)$ |

**统一模板（找第一个 ≥ target）：**

```python
left, right = 0, len(nums) - 1
while left <= right:
    mid = left + (right - left) // 2
    if nums[mid] < target:
        left = mid + 1
    else:
        right = mid - 1
return left   # 第一个 >= target 的下标，全小于则返回 len(nums)
```

### 自测三问

**问题一：** 为什么 `mid = left + (right - left) // 2` 比 `(left + right) // 2` 安全？在 Python 里这俩等价，那为什么还要用前者？

**参考答案：** 在 Python 里两者等价（Python 整数不溢出）。但在 Java/C++ 等语言里，`left + right` 当两个数都接近 `INT_MAX` 时会溢出成负数，导致 `mid` 为负，数组越界。这是 Java 标准库 2006 年之前的真实 bug。用 `left + (right - left) // 2` 永远不溢出。写代码时养成习惯用防溢出版本，是为跨语言安全和肌肉记忆——面试时用 Python 写，工作时可能用 Java/Go，统一用防溢出版本不会错。

**问题二：** 二分查找的本质是"单调性"，不是"有序数组"。请用"二分答案"的例子说明这一点。具体来说：把数组 `[7,2,5,10,8]` 分成 2 段，使各段和的最大值最小。这个"最大值"为什么能二分？

**参考答案：** 这题数组无序，但"答案"（即各段和的最大值）具有单调性。设上限为 M：如果各段和都不超过 M 就能分成 2 段，那么 M 越大越容易满足（需要的段数越少）。M 的范围是 `[max(nums), sum(nums)]` = `[10, 32]`。对每个 mid，贪心地从左到右累加，超过 mid 就开新段，算需要几段。如果段数 ≤ 2，说明 mid 可行，往左找更小的；如果段数 > 2，说明 mid 太小，往右找。这里"单调性"体现在：mid 越大，需要的段数越少。看到"答案在一个范围 + 有方向性判断"，就是二分答案的信号。

**问题三：** 生产里查有序数组，你会用 `bisect` 还是手写二分？什么场景必须手写？

**参考答案：** 查有序数组用 `bisect`，因为它是 C 实现，快且不会写错边界。必须手写的场景：二分答案（check 函数是业务逻辑，库没有）、浮点二分求根、自定义单调性判断（比如"找第一个满足某谓词的位置"，谓词不是简单的大小比较）、三分法求极值。原则是：**库能解决的不手写，库解决不了的必须会手写。** 学手写二分不是为了替代库，是为了在库帮不了时还能自己上。

### 算法思想 × 生活迁移

二分查找的核心思想是：**每一步排除一半可能性。** 这个思想在生活里有很多用武之地。

**迁移一：debug 时的"二分定位法"。** 线上 bug，不知道是哪段代码引入的。最笨的办法是逐行查。聪明的办法是 git bisect——用二分法在提交历史里找引入 bug 的那一次 commit。先测中间那次 commit，如果没问题，bug 在后半段；有问题，bug 在前半段。1000 次提交，最多测 10 次（$\log_2 1000 \approx 10$）就能定位。Git 内置了 `git bisect` 命令，背后就是二分。这不是巧合——版本历史的"单调性"在于"bug 一旦引入就不会自己消失"，所以能二分。

**迁移二：决策中的"排除法"。** 买房子看了 20 套，挑花眼。与其一套套重新评估，不如先按某个维度排序（比如价格），定一个心理预期（比如预算 300 万），用二分思路：先看第 10 套（中位数价位），如果第 10 套满足核心需求且价格可接受，说明前 10 套里可能有更好的；如果不满足，说明这个价位不够，往后找。这种"用中点判断方向"的思路，能避免对每个选项都做完整评估——省下的精力用于对候选区做精细决策。

**迁移三：长文档检索的"中点跳转"。** 在 1000 页的 PDF 里找某一段，不要从第 1 页翻起。先跳到第 500 页看一眼页眉/目录，判断目标在前半还是后半，再跳到 250 或 750。这就是电子书阅读器都支持"跳到百分比位置"和"目录跳转"的原因——本质是支持你做二分定位。纸质时代查字典、查百科全书用的都是这个直觉。有趣的是，很多人在长文档里找东西还是 Ctrl+F 从头搜（线性扫描，$O(n)$），忘了"先跳中间判断方向"能省 99% 的工作。这也是为什么"目录""索引""章节标题"这些看似古老的设计至今不可替代——它们是手动二分的"中点探针"。

**为什么这些迁移成立：** 二分成立的条件是"单调性 + 可判断方向"。git bisect 成立，是因为"bug 引入后持续存在"（单调性）+ "某次 commit 有没有 bug 可测"（可判断方向）。买房排除法成立，是因为"价格与满意度有单调关系"（近似）+ "看了中位数能判断方向"。长文档检索成立，是因为"页码与内容顺序单调对应"（页码递增即内容递进）+ "看一眼中点能判断方向"。迁移成立的关键是结构同构——不是所有决策都能二分，但只要满足这两个条件，二分就是最优策略。
