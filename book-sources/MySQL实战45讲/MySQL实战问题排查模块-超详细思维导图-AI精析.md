## 二、🔢 自增 ID 不连续问题（第 39 讲）

### 2.1 现象观察

#### 2.1.1 典型案例
```sql
CREATE TABLE users (
  id INT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(50)
);

INSERT INTO users(name) VALUES('Tom');    -- id = 1 ✅
INSERT INTO users(name) VALUES('Jerry');  -- id = 2 ✅
INSERT INTO users(name) VALUES('Bob');    -- id = 5 ❌ 为什么跳到了 5？
```

#### 2.1.2 业务影响评估
- **一般情况下**：不影响业务（ID 只是标识符，不需要连续）
- **特殊情况需要注意**：
  - 如果业务依赖 ID 的连续性（如订单号），应使用**自定义生成规则**
  - 如果 ID 接近上限（BIGINT UNSIGNED 最大值约 18.4 亿亿），需要关注浪费率

### 2.2 原因一：事务回滚（最主要的原因）⭐⭐⭐

#### 2.2.1 现象描述
- **核心机制**：自增 ID 一旦分配就**不会回收**
- **设计原则**："宁可不连续，也要保证唯一性"

#### 2.2.2 执行过程演示
```
事务 A 开始
  ↓
INSERT INTO users(name) VALUES('Alice');
  ↓
InnoDB 分配 id=3，自增值变为 4
  ↓
ROLLBACK;  -- 事务回滚
  ↓
但是 id=3 已经被分配且不会回收 ❌
  ↓
事务 B 开始
  ↓
INSERT INTO users(name) VALUES('Bob');
  ↓
分配 id=4（跳过了 3）❌
```

#### 2.2.3 为什么这样设计？
##### 原因 1：避免并发冲突
- **场景分析**：
  ```
  事务A申请id=2，事务B申请id=3
  若A回滚后回退自增值
  后续事务C会申请到id=2
  导致主键冲突，破坏数据一致性
  ```

##### 原因 2：性能代价太大
- **如果允许回退**：
  - 方案 A：每次申请前检查 id 是否存在（查询开销大）
  - 方案 B：扩大锁范围（严重影响并发性能）
- **结论**：MySQL 选择"不回退"策略，用空洞换取性能

### 2.3 原因二：批量插入预分配

#### 2.3.1 触发场景
- **语句类型**：
  - `INSERT ... SELECT`（从其他表插入）
  - `REPLACE ... SELECT`
  - `LOAD DATA INFILE`（批量导入数据）

#### 2.3.2 预分配策略（翻倍分配）
- **分配规则**：
  | 申请次数 | 分配数量 | 累计分配 |
  |---------|---------|---------|
  | 第 1 次 | 1 个 | 1 |
  | 第 2 次 | 2 个 | 3 |
  | 第 3 次 | 4 个 | 7 |
  | 第 4 次 | 8 个 | 15 |
  | ... | 翻倍 | ... |

#### 2.3.3 浪费示例
```sql
-- 插入 4 行数据
INSERT INTO t2(c,d) SELECT c,d FROM t;

-- 申请过程：
-- 第1次：申请 id=1（使用 1 个）
-- 第2次：申请 id=2,3（使用 2 个）
-- 第3次：申请 id=4,5,6,7（使用 1 个，浪费 3 个）
-- 实际使用：id=1,2,3,4
-- 浪费：id=5,6,7（3 个 ID 空洞）
```

#### 2.3.4 设计原因：空间换时间
- **为什么这样设计？**
  - 批量插入无法预知需要多少 id
  - 逐个申请：10 万行需要申请 10 万次，性能极差
  - 翻倍预分配：10 万行只需申请约 17 次（2^17=131072）
- **浪费程度可控**：最多浪费约 50%（最后一批可能只用了一部分）

### 2.4 原因三：唯一键冲突导致的 ID 浪费

#### 2.4.1 执行流程
```
1. InnoDB 获取当前自增值（如 2）
   ↓
2. 将自增值修改为下一个值（3）
   ↓
3. 执行插入操作
   ↓
4. 发生唯一键冲突，语句失败
   ↓
5. 自增值已改为 3，不会回退 ❌
   ↓
6. 下次插入得到 id=3，产生空洞
```

#### 2.4.2 示例代码
```sql
-- 表中已有 (1, 1)
INSERT INTO t VALUES(null, 1, 1);  -- c=1 冲突，失败
-- 自增值从 2 变为 3，但 id=2 未插入成功

INSERT INTO t VALUES(null, 2, 2);  -- 成功，得到 id=3
-- 下次插入会得到 id=4（跳过了 2）
```

### 2.5 自增值存储机制的版本差异

#### 2.5.1 MySQL 5.7 及之前版本
- **存储位置**：内存中
- **重启行为**：重启后自增值 = max(id) + 1
- **问题**：重启后可能丢失部分已分配但未使用的自增值

#### 2.5.2 MySQL 8.0+ 版本
- **存储位置**：持久化到 redo log
- **重启行为**：重启后恢复重启前的自增值
- **改进**：解决了重启后自增值重置的问题

### 2.6 解决方案与最佳实践

#### 2.6.1 接受现实（推荐）
- **适用场景**：大多数业务场景
- **理由**：ID 不连续是可以接受的，只要保证唯一性和递增性即可
- **优势**：无需额外开发成本

#### 2.6.2 使用 UUID（完全避免自增问题）
- **优点**：全局唯一，无连续性依赖
- **缺点**：
  - 存储空间大（36 字符 vs 8 字节）
  - 无序导致页分裂，影响插入性能
  - 作为主键时聚簇索引效率低

#### 2.6.3 使用雪花算法（Snowflake）
- **适用场景**：分布式环境下生成有序唯一 ID
- **优点**：
  - 时间有序（趋势递增）
  - 全局唯一
  - 高性能（本地生成，无网络开销）
- **组成结构**：1 位符号 + 41 位时间戳 + 10 位机器 ID + 12 位序列号

#### 2.6.4 监控 ID 使用率
- **定期检查**：
  ```sql
  -- 查看当前自增计数器的值
  SHOW TABLE STATUS LIKE 'users'\G
  -- 关注 Auto_increment 字段
  ```
- **告警阈值**：当使用率超过 80% 时发出告警

### 2.7 自增锁优化策略

#### 2.7.1 innodb_autoinc_lock_mode 参数
| 模式值 | 行为 | 适用场景 | 性能 |
|-------|------|---------|------|
| **0** | 语句级锁（MySQL 5.0 策略） | 兼容旧版本 | 低 |
| **1** | 普通 insert 立即释放，批量 insert 语句级锁 | 默认值 | 中 |
| **2** | 所有 insert 立即释放锁 | 高并发场景 | 高 |

#### 2.7.2 生产环境推荐配置
```sql
-- 推荐配置组合
SET GLOBAL innodb_autoinc_lock_mode = 2;  -- 立即释放锁
SET GLOBAL binlog_format = row;           -- 记录实际数据值
```

#### 2.7.3 为什么这样配置安全？
- **lock_mode=2**：所有 insert 申请到 id 后立即释放锁，提升并发
- **binlog_format=row**：记录实际数据值（包括 id 值），不依赖自增逻辑
- **主备一致保障**：避免 statement 格式下主备执行顺序不同导致的数据不一致

## 四、⚠️ 大事务的危害与避免（第 03、07、26 讲综合）

### 4.1 什么是大事务？

#### 4.1.1 定义
- **定义**：运行时间长、影响行数多、占用资源多的交易
- **本质特征**：占用数据库资源的时间超过业务合理范围

#### 4.1.2 判断标准
| 维度 | 阈值 | 说明 |
|------|------|------|
| **执行时间** | > 5 秒 | 事务持续时间过长 |
| **影响行数** | > 1000 行 | 单次操作影响数据量大 |
| **undo log 占用** | > 10MB | 产生大量回滚日志 |
| **业务角度** | > 60秒 | 超过业务容忍的响应时间 |

### 4.2 危害一：锁占用时间长 🔒

#### 4.2.1 现象描述
```sql
-- 事务 A（大事务）
BEGIN;
UPDATE orders SET status = 'processed'
WHERE create_time < '2023-01-01';  -- 影响行数：10 万行
-- 这个 UPDATE 会持有这些行的 X 锁直到事务结束
-- 假设事务 A 运行了 10 分钟...

-- 事务 B 尝试更新其中某一行
UPDATE orders SET status = 'urgent' WHERE id = 12345;
-- 阻塞！因为这一行被事务 A 锁住了
-- 如果事务 A 运行了 10 分钟，事务 B 就要等 10 分钟 ❌
COMMIT;
```

#### 4.2.2 两阶段锁协议（核心机制）
- **核心原则**：
  > 在 InnoDB 事务中，行锁是在需要的时候才加上的，但并不是不需要了就立刻释放，而是要等到**事务结束时**才释放。

- **锁的生命周期**：
  ```
  语句开始执行
      ↓
  需要修改某行 → 加 X 锁
      ↓
  继续执行其他语句...
      ↓
  （锁一直持有，不会中途释放）
      ↓
  COMMIT 或 ROLLBACK
      ↓
  所有锁一次性释放
  ```

#### 4.2.3 后果分析
- **系统吞吐量下降**：大量事务在等待锁
- **用户体验变差**：响应时间显著增加
- **连锁反应**：连接池耗尽，应用雪崩

### 4.3 危害二：回滚耗时长 ⏱️

#### 4.3.1 现象描述
```
大事务执行了一半发现错误，需要 ROLLBACK
    ↓
回滚过程需要逐行撤销所有修改
    ↓
如果修改了 10 万行，回滚可能也需要几分钟
    ↓
回滚期间同样持有锁，加剧阻塞问题 ❌
```

#### 4.3.2 回滚的性能特点
- **逐行撤销**：不能批量回滚，必须按 undo log 逐行恢复
- **IO 密集**：大量的磁盘读写操作
- **锁持有**：回滚期间仍然持有所有行锁
- **不可中断**：一旦开始回滚，无法被 kill（必须完成）

#### 4.3.3 极端案例
- **案例描述**：数据 20GB，undo log 膨胀到 200GB
- **回滚时间**：数小时甚至更长
- **最终结果**：可能需要重建整个数据库

### 4.4 危害三：undo log 膨胀 📈

#### 4.4.1 undo log 的作用回顾
- **作用 1**：事务回滚（恢复到修改前的状态）
- **作用 2**：MVCC 多版本并发控制（提供历史版本数据）

#### 4.4.2 undo log 的生命周期
```
数据更新时 → 创建 undo log 记录
    ↓
事务活跃期间 → undo log 必须保留（用于可能的回滚）
    ↓
事务提交后 → undo log 不能立即删除
    ↓
原因：可能有其他事务的 read-view 需要读到旧版本
    ↓
只有当没有事务需要这些 undo log 时 → 才能 purged（清理）
```

#### 4.4.3 长事务导致 undo log 无法清理
- **根本原因**：
  - 长事务持有老的 read-view
  - 所有用到的 undo log 记录必须保留
  - 导致 undo log 大量累积

- **后果链**：
  ```
  长事务运行中（比如 1 小时）
      ↓
  产生大量 undo log（假设 500MB）
      ↓
  这 1 小时内的所有 undo log 都不能清理
      ↓
  undo 表空间快速增长
      ↓
  磁盘空间不足
      ↓
  影响数据库正常运行
      ↓
  极端情况：undo 段膨胀到 200GB（数据才 20GB）
  ```

#### 4.4.4 监控 undo log 使用情况
```sql
-- 查看 InnoDB 的 undo log 状态
SHOW ENGINE INNODB STATUS\G
-- 关注 History list length 字段

-- 查看 undo 表空间大小
SELECT
  file_name,
  total_extents,
  free_extents,
  (total_extents - free_extents) * extent_size / 1024 / 1024 AS used_mb
FROM information_schema.INNODB_TABLESPACES
WHERE space_type = 'Undo';
```

### 4.5 危害四：主备延迟增大 🔄

#### 4.5.1 影响机制
```
主库：
  大事务执行时间长（比如 30 分钟）
  但可以与其他事务并发执行 ✅
      ↓
备库（单线程重放）：
  必须完整重放大事务的所有操作
  这 30 分钟内阻塞后续所有事务的重放 ❌
      ↓
seconds_behind_master 持续增大
      ↓
  延迟累积到小时级别
```

#### 4.5.2 并发能力的差异
- **主库**：多线程并发执行，大事务只占用一个线程
- **备库（传统复制）**：单线程 SQL Thread 重放，大事务阻塞全部
- **备库（并行复制）**：多线程重放，但仍受限于事务依赖关系

#### 4.5.3 延迟的影响
- **读请求路由失败**：无法将读请求路由到备库
- **主库压力增大**：所有读写都压到主库
- **故障恢复风险**：主库故障时备库数据严重滞后

### 4.6 如何避免大事务？💡

#### 4.6.1 方法一：拆分事务（最有效）⭐⭐⭐

##### 思想
- **核心策略**：把大事务拆成多个小事务
- **目标**：每个小事务的执行时间控制在秒级

##### 实现示例
```sql
-- ❌ 大事务（一次性更新 100 万行）
BEGIN;
UPDATE orders SET status = 'archived'
WHERE create_time < '2023-01-01';  -- 100 万行
COMMIT;
-- 问题：锁持有时间长，undo log 膨胀，主备延迟

-- ✅ 拆分为小事务（每次更新 1 万行）
DELIMITER //

CREATE PROCEDURE batch_update_orders()
BEGIN
  DECLARE done INT DEFAULT FALSE;
  DECLARE min_id INT;
  DECLARE max_id INT;

  DECLARE cur CURSOR FOR
    SELECT MIN(id), MAX(id)
    FROM orders
    WHERE create_time < '2023-01-01'
    GROUP BY FLOOR((id - 1) / 10000);

  DECLARE CONTINUE HANDLER FOR NOT FOUND SET done = TRUE;

  OPEN cur;

  read_loop: LOOP
    FETCH cur INTO min_id, max_id;
    IF done THEN
      LEAVE read_loop;
    END IF;

    BEGIN;
    UPDATE orders SET status = 'archived'
    WHERE id BETWEEN min_id AND max_id;
    COMMIT;
  END LOOP;

  CLOSE cur;
END //

DELIMITER ;

CALL batch_update_orders();
```

##### 拆分粒度建议
| 数据量 | 建议批次大小 | 每批耗时 |
|-------|------------|---------|
| < 1 万行 | 不拆分 | < 1 秒 |
| 1-10 万行 | 每 1000 行一批 | < 1 秒 |
| 10-100 万行 | 每 1 万行一批 | 1-3 秒 |
| > 100 万行 | 每 1 万行一批 | 1-3 秒 |

#### 4.6.2 方法二：设置超时时间

##### MySQL 5.7.8+ 的 MAX_EXECUTION_TIME
```sql
-- 设置单个查询的最大执行时间（毫秒）
SET SESSION MAX_EXECUTION_TIME = 5000;  -- 5 秒超时

-- 超过 5 秒的查询会被自动终止
-- 返回错误：Query execution was interrupted (max_execution_time exceeded)
```

##### 全局配置
```ini
[mysqld]
# 设置默认的最大执行时间（毫秒），0 表示不限制
max_execution_time = 5000
```

##### 注意事项
- **只对 SELECT 有效**：对 INSERT/UPDATE/DELETE 无效
- **事务级别控制**：需要在事务内部设置

#### 4.6.3 方法三：监控和预警

##### 查询正在执行的长事务
```sql
SELECT
  trx_id,
  trx_state,
  trx_started,
  TIMESTAMPDIFF(SECOND, trx_started, NOW()) AS duration_seconds,
  trx_mysql_thread_id,
  trx_query,
  trx_tables_locked,
  trx_lock_structs
FROM information_schema.INNODB_TRX
WHERE TIMESTAMPDIFF(SECOND, trx_started, NOW()) > 10  -- 超过 10 秒
ORDER BY duration_seconds DESC;
```

##### 监控指标建议
| 指标 | 告警阈值 | 说明 |
|------|---------|------|
| 事务持续时间 | > 60 秒 | Warning |
| 事务持续时间 | > 300 秒（5 分钟） | Critical |
| undo log 大小 | > 1GB | Warning |
| undo log 大小 | > 5GB | Critical |
| 锁等待次数 | > 100/分钟 | Warning |

##### 自动 kill 长事务脚本（示例）
```bash
#!/bin/bash
# 自动 kill 超过 300 秒的事务

THRESHOLD=300  -- 5 分钟阈值

mysql -u monitor -p'password' -e "
  SELECT CONCAT('KILL QUERY ', trx_mysql_thread_id, ';')
  FROM information_schema.INNODB_TRX
  WHERE TIMESTAMPDIFF(SECOND, trx_started, NOW()) > $THRESHOLD;
" | mysql -u admin -p'password'
```

#### 4.6.4 方法四：优化事务设计

##### 显式启动事务
```sql
-- ✅ 推荐：显式控制事务边界
BEGIN;
-- 业务操作 1
-- 业务操作 2
COMMIT;

-- ❌ 避免：使用 set autocommit=0
SET autocommit = 0;
-- 如果忘记 commit，事务会一直运行
```

##### 将热点数据操作放在事务最后
- **依据两阶段锁协议**：行锁要等到事务结束才释放
- **优化思路**：缩短热点数据的锁持有时间

- **案例**：电影票交易业务
  ```
  建议顺序：
  1. 记录交易日志（insert）         ← 先执行
  2. 扣除顾客 A 账户余额（update）   ← 再执行
  3. 增加影院 B 账户余额（update）   ← 最后执行（热点数据）
  
  影院账户是热点数据，放在最后可最大程度减少锁持有时间
  ```

##### 避免在事务中进行远程调用
```sql
-- ❌ 错误：事务中包含 HTTP 调用
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;
-- 调用第三方支付接口（可能耗时数秒）
SELECT http_post('https://payment.api/pay', ...);
UPDATE orders SET status = 'paid' WHERE order_id = 123;
COMMIT;
-- 问题：事务持有了 accounts 表的锁，却在等待网络响应

-- ✅ 正确：先完成外部调用，再启动事务
-- 调用第三方支付接口
result := http_post('https://payment.api/pay', ...);

-- 支付成功后再启动事务
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;
UPDATE orders SET status = 'paid' WHERE order_id = 123;
COMMIT;
```

### 4.7 大事务危害总结

| 危害类型 | 影响范围 | 严重程度 | 解决方案 |
|---------|---------|---------|---------|

### 6.2 核心知识点关联图

```
误删数据恢复
├── 依赖：binlog（ROW 格式）
├── 相关：延迟复制备库
└── 工具：Flashback / binlog2sql

大事务危害
├── 涉及：两阶段锁协议
├── 影响：undo log 膨胀
├── 影响：主备延迟
└── 解决：拆分事务

Kill 语句
├── 机制：埋点 + 信号
├── 场景：回滚不可中断
└── 预防：避免大事务

自增 ID 问题
├── 原因：事务回滚不退
├── 原因：批量预分配
├── 原因：唯一键冲突
└── 预防：使用 BIGINT
```
