# 为什么表数据删掉一半，表文件大小不变？

## 思考问题
1. 为什么DELETE命令删除数据后，磁盘文件大小不会改变？这与MySQL的存储机制有什么关系？
2. 如果你的表文件已经很大，想要收缩表空间，应该用什么方法？这个过程中需要注意什么风险？



### 🔹 数据删除的底层机制

#### 记录级删除
**流程**：删除R4记录 → 标记为删除 → 后续插入可复用该位置

**关键限制**：
- 记录复用仅限于符合范围条件的数据
- 例如：删除ID=400的记录后，只能复用于插入ID在300-600之间的记录
- **磁盘文件大小不变**

#### 数据页级删除
**流程**：删除整个数据页的所有记录 → 标记页为可复用 → 可复用到任何位置

**关键区别**：
- 页复用比记录复用更灵活，可以复用到任何位置
- 但**磁盘文件大小仍然不变**

**思考**：为什么MySQL不直接释放空数据页的磁盘空间？这涉及文件系统的特性，频繁申请/释放磁盘空间会带来性能开销。



### 🔹 重建表的两种方式

#### 方式一：传统DDL（MySQL 5.5及之前）
**流程**：
```
创建临时表B → 按主键顺序从表A读取数据 → 插入到表B → 用表B替换表A
```

**关键限制**：
- ❌ 整个过程需要MDL写锁，阻塞所有DML操作
- ❌ 非Online，对业务影响大
- ❌ 大表操作耗时长，风险高

#### 方式二：Online DDL（MySQL 5.6+）
**流程**：
```
1. 建立临时文件，扫描表A主键所有数据页
2. 生成B+树存储到临时文件
3. 记录所有对表A的操作到row log
4. 将row log应用到临时文件
5. 用临时文件替换表A的数据文件
```

**关键优势**：
- ✅ MDL写锁在拷贝数据前退化为读锁
- ✅ 允许增删改操作，对业务影响小
- ✅ 锁时间短，可认为是Online的

**命令**：
```sql
ALTER TABLE t ENGINE=InnoDB;
-- 等价于
ALTER TABLE t ENGINE=InnoDB, ALGORITHM=inplace;
```

**实践要点**：
- 大表重建消耗IO和CPU资源，需在业务低峰期执行
- 推荐使用GitHub开源的gh-ost工具，更安全



## 次要信息速览（80%快速带过）

### 【表结构存储】
- MySQL 8.0前：表结构存在.frm文件
- MySQL 8.0+：表结构可存在系统数据表
- 表结构占用空间小，主要讨论表数据

### 【共享表空间】
- `innodb_file_per_table=OFF`：表数据放在系统共享表空间
- DROP TABLE后空间不会回收
- 不推荐使用，建议设置为ON

### 【truncate vs delete】
- TRUNCATE = DROP + CREATE，会释放表空间
- DELETE只标记删除，不释放空间

### 【其他重建命令】
- `ANALYZE TABLE t`：重新统计索引信息，不修改数据，加MDL读锁
- `OPTIMIZE TABLE t` = recreate + analyze

### 【课后思考题答案】
**问题**：重建表后，空间反而变大（1TB → 1.01TB）？

**原因**：
1. 原表已经很紧凑，页填充率>90%
2. 重建时按90%满的比例重新整理页数据（预留10%给UPDATE）
3. 结果：文件反而变大

```
DELETE不释放空间 → 标记可复用 → 产生空洞
空洞来源 → 删除 + 插入（页分裂） + 更新

Online DDL → 允许DML + MDL锁退化 + row log机制
```
