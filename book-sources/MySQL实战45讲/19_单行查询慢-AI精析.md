# 为什么我只查一行的语句，也执行这么慢？

## 核心知识清单（20%重点）

### 🔹 问题定位方法论

**第一步：排除数据库整体压力**
- CPU占用率高或ioutil高 → 所有语句都可能变慢（非本文讨论范围）
- 数据库压力正常 → 进入具体排查流程

**第二步：执行 `show processlist` 查看线程状态**
- 根据状态信息定位具体问题类型
- 不同状态对应不同的锁等待场景



### 🔹 场景二：等flush

**核心概念**：
- `flush tables t with read lock;` → 关闭指定表t
- `flush tables with read lock;` → 关闭所有打开的表

**典型状态**：`Waiting for table flush`

**复现场景**：
```
Session A: select * from t where id=1; 
           -- 假设每行sleep(1)，长时间持有表t的"打开"状态
Session B: flush tables t;  -- 等待Session A释放表
Session C: select * from t where id=1;  -- 被flush命令阻塞
```

**关键要点**：
- flush命令本身执行很快，除非被其他线程阻塞
- 形成阻塞链：长查询 → flush命令 → 新查询
- 排查方法：`show processlist` 查看哪个线程持有表



### 🔹 场景四：一致性读慢（undo log回滚）

**核心概念**：
- **当前读**：读取最新已提交数据，如 `LOCK IN SHARE MODE`、`FOR UPDATE`
- **一致性读**：基于MVCC，读取事务开始时的数据快照
- **undo log**：回滚日志，记录数据修改前的值

**典型现象**：
```
-- 语句1：扫描1行，执行800ms
SELECT * FROM t WHERE id=1;

-- 语句2：扫描1行，执行0.2ms
SELECT * FROM t WHERE id=1 LOCK IN SHARE MODE;
```

**复现场景**：
```
Session A: START TRANSACTION WITH CONSISTENT SNAPSHOT;
Session B: UPDATE t SET c=c+1 WHERE id=1;  -- 执行100万次
           -- 生成100万个undo log
Session A: SELECT * FROM t WHERE id=1;  -- 需要回滚100万次undo log
```

**关键原理**：
- 一致性读需要从最新值开始，依次执行undo log回滚到事务开始时的状态
- undo log数量越多，回滚时间越长
- 当前读直接读取最新值，无需回滚

**性能影响**：
- 【次要：undo log记录的是"把2改成1"的操作逻辑，非简单的数值减1】
- 长事务未提交期间，其他事务的更新会持续积累undo log



## 思考问答

### 问题1：为什么一致性读比当前读慢这么多？

**答案要点**：
- 一致性读需要执行undo log回滚，回滚次数取决于事务开始后的修改次数
- 当前读直接读取最新值，无需回滚
- 本例中Session B执行了100万次更新，Session A的一致性读需要回滚100万次

**实践启示**：
- 长事务会积累大量undo log，影响其他事务的一致性读性能
- 需要及时提交或回滚长事务



### 问题3：无索引字段查询如何加锁？

**核心结论**：
- **RC隔离级别**：锁住所有扫描到的行（无GAP锁）
- **RR隔离级别**：锁住所有行 + 所有间隙（GAP锁）
- 锁释放时机：事务提交或回滚后

**验证方法**：
```sql
-- Session 1
BEGIN;
SELECT * FROM t WHERE c=5 FOR UPDATE;

-- Session 2（RC模式：成功，RR模式：等待）
INSERT INTO t VALUES(100001, 8);

-- Session 3（RC/RR模式：都等待）
DELETE FROM t WHERE c=10;
```
