# 核心知识清单(20%)

## 一、加锁规则体系

### 🔹 两个原则(必记)

**原则1:加锁基本单位是next-key lock**
- **定义**:前开后闭区间,如(5,10]表示5到10之间的间隙+10这一行
- **应用场景**:所有加锁操作的起点判断
- **常见错误**:误以为只锁行,忽略间隙锁的存在

**原则2:查找过程中访问到的对象才会加锁**
- **定义**:只有扫描过程中碰到的记录才会被加锁
- **关键点**:覆盖索引可能只锁二级索引,不锁主键索引
- **实践指导**:查询字段选择会影响锁范围

### 🔹 两个优化(关键)

**优化1:唯一索引等值查询退化为行锁**
- **触发条件**:唯一索引+等值查询
- **效果**:next-key lock → 行锁
- **示例**:`select * from t where id=10 for update`只锁id=10这一行

**优化2:等值查询向右遍历退化为间隙锁**
- **触发条件**:等值查询+向右遍历+最后值不满足条件
- **效果**:next-key lock → 间隙锁
- **示例**:`select * from t where c=5 lock in share mode`在索引c上加(0,5]和间隙锁(5,10)

### 🔹 一个Bug(注意)

**唯一索引范围查询会多锁一个值**
- **现象**:范围查询会扫描到第一个不满足条件的值
- **示例**:`select * from t where id>10 and id<=15 for update`会多锁(15,20]
- **影响**:可能导致不必要的锁等待

## 二、关键案例深度解析

### 🔹 案例二:非唯一索引等值锁(核心)

**场景**:`select * from t where c=5 lock in share mode`

**加锁过程**:
1. 根据原则1,加next-key lock(0,5]
2. 向右遍历到c=10(不满足条件),根据优化2退化为间隙锁(5,10)
3. **关键**:覆盖索引不访问主键,主键索引无锁

**实践指导**:
- `lock in share mode`只锁覆盖索引
- `for update`会同时锁主键索引
- 需要锁主键时,查询字段加入索引外字段(如`select d`)

### 🔹 案例七:limit语句加锁(重要)

**场景对比**:
- 无limit:`delete from t where c=10` → 锁范围大
- 有limit:`delete from t where c=10 limit 2` → 锁范围小

**原因**:limit限制扫描行数,提前终止遍历

**实践指导**:删除数据时尽量加limit,减小锁范围

### 🔹 案例八:死锁原理(关键)

**核心认知**:next-key lock = 间隙锁 + 行锁(分两步执行)

**死锁过程**:
1. Session A:加间隙锁(5,10)成功 + 行锁c=10成功
2. Session B:加间隙锁(5,10)成功 → 行锁c=10等待
3. Session A:插入(8,8,8) → 被Session B的间隙锁阻塞 → 死锁

**思考**:为什么Session B的间隙锁能加成功?因为间隙锁之间不互斥

## 三、锁的本质认知

### 🔹 锁是加在索引上的
- 主键索引、二级索引分别加锁
- 覆盖索引优化可能导致主键索引无锁
- 不同索引上的锁相互独立

### 🔹 间隙锁的作用
- **目的**:防止幻读(防止插入新记录)
- **特性**:间隙锁之间不互斥,只阻塞插入操作
- **范围**:有间隙的地方就有间隙锁

### 🔹 隔离级别的影响
- **可重复读**:间隙锁生效
- **读提交**:间隙锁失效,只保留行锁
- **实践**:读提交锁范围更小,但可能幻读
