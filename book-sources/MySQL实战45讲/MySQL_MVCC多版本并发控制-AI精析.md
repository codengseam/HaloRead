# MySQL MVCC 多版本并发控制 - 思维导图

## 🧠 一、MVCC 概述

### 📌 1.1 什么是 MVCC？

- **全称**：Multi-Version Concurrency Control（**多版本并发控制**）
- **核心思想**：
  - 通过维护数据的**多个历史版本**，让读写操作互不冲突
  - 类似 Git 版本控制的思想 🔀
  - 同一条记录在系统中可存在**多个版本**
- **目标**：🎯 **提高数据库的并发性能**

### 📌 1.2 MVCC 解决的核心问题

#### ✅ 1.2.1 **读写不冲突**
- 读操作（SELECT）可以**不被写操作阻塞** 📖
- 写操作（UPDATE/DELETE）可以**不被读操作阻塞** ✏️
- 类比 Java 中 **CopyOnWriteArrayList** 的思想 ☕
- **本质**：读操作访问的是数据的**快照（Snapshot）**，而非最新数据

#### ✅ 1.2.2 **实现快照读（Consistent Read）**
- 在 **RR 级别**下，同一事务内多次读取结果**一致** ✓
- 无需加锁就能实现**可重复读**
- 避免了传统的"读写锁竞争"

### ⚠️ 1.3 MVCC 的局限性

- ❌ 只支持 **RC（Read Committed）** 和 **RR（Repeatable Read）** 两种隔离级别
- **RU（Read Uncommitted）** 太危险不需要 MVCC
- **Serializable（串行化）** 直接用锁，不需要 MVCC
- ⚠️ 只能解决**部分幻读问题**（需配合**间隙锁**才完整解决）



## 🔍 三、可见性判断算法（MVCC 的灵魂）⚡

### 📌 3.1 **五步判断规则**（核心算法）

当一个事务 T 要读取某行数据 R（R.trx_id = X）时：

```
规则 1️⃣：如果 X == creator_trx_id（T 自己修改的）
  → ✅ 可见（自己能看到自己的修改）

规则 2️⃣：如果 X < m_up_limit_id（X 在 ReadView 创建前已提交）
  → ✅ 可见（已经提交的事务应该可见）

规则 3️⃣：如果 X >= m_low_limit_id（X 在 ReadView 创建后才启动）
  → ❌ 不可见（未来的事务不应该可见）

规则 4️⃣：如果 X 在 m_ids 中（X 是活跃未提交的事务）
  → ❌ 不可见（未提交的事务不应该可见）

规则 5️⃣：以上都不满足（X 已提交但在 m_up_limit_id 和 m_low_limit_id 之间）
  → 通过 DB_ROLL_PTR 找到 undo log 中的旧版本
  → 对旧版本重新执行上述判断
  → 直到找到可见版本或到达链尾
```

### 📌 3.2 **可见性判断流程图** 🔄

```
开始读取行 R (R.trx_id = X)
         ↓
    ┌────────────────────┐
    │ R.trx_id == 当前   │ ──Yes──→ ✅ 可见，返回该行
    │ 事务ID？           │
    └────────┬───────────┘
             ↓ No
    ┌────────────────────┐
    │ R.trx_id < up_     │ ──Yes──→ ✅ 可见，返回该行
    │ limit_id？         │
    └────────┬───────────┘
             ↓ No
    ┌────────────────────┐
    │ R.trx_id < low_    │ ──Yes──→ ❌ 不可见
    │ limit_id？         │
    └────────┬───────────┘
             ↓ No
    ┌────────────────────┐
    │ R.trx_id 在 m_ids  │ ──Yes──→ ❌ 不可见
    │ 中？               │
    └────────┬───────────┘
             ↓ No
    ┌────────────────────┐
    │ 通过 roll_ptr 找   │
    │ 到前一版本         │
    └────────┬───────────┘
             ↓
    对新版本重复上述判断
```

### 📌 3.3 **简化记忆口诀** 💡

1. **自己改的** → 总是可见 ✅
2. **老事务改的**（视图前提交） → 可见 ✅
3. **新事务改的**（视图后启动） → 不可见 ❌
4. **正在跑的事务改的** → 不可见 ❌
5. **都不匹配** → 回溯旧版本再判断 🔄

## 🔐 五、MVCC 与锁的关系（协同工作）🤝

### 📌 5.1 分工明确

| 机制 | 负责范围 | 工作方式 | 适用场景 |
|------|---------|---------|---------|
| **MVCC** | **读操作**（普通 SELECT） | 无锁并发，基于快照 | 快照读（一致性读） |
| **锁** | **写操作**和**当前读** | 加锁互斥，保证原子性 | UPDATE/DELETE/INSERT、SELECT FOR UPDATE |

### 📌 5.2 两者如何协同工作

#### 🔸 5.2.1 读操作流程（MVCC 负责）
1. 事务启动时生成 ReadView（RR 级别）或在每次查询时生成（RC 级别）
2. 从数据页读取行的最新版本
3. 根据**可见性算法**判断该版本是否可见
4. 如果不可见，通过 **roll_ptr 回溯 undo log** 找到历史版本
5. 返回可见的历史版本数据（**不加任何锁**）🔓

#### 🔸 5.2.2 写操作流程（锁负责）
1. 使用**当前读**读取最新已提交的数据（加锁）🔒
2. 对数据进行修改
3. 生成新版本并写入数据页
4. 将旧版本拷贝到 undo log
5. 提交事务后释放锁

#### 🔸 5.2.3 协同效果
- 读操作通过 MVCC 看到 **consistent snapshot**（一致性快照）
- 写操作通过锁保证数据修改的**原子性和隔离性**
- 两者配合完美实现事务的**隔离性**要求
- 达到**高并发 + 强一致性**的平衡 ⚖️

### 📌 5.3 一致性读 vs 当前读（重要区分）⚡

| 对比维度 | **一致性读（快照读）** | **当前读（实时读）** |
|---------|---------------------|---------------------|
| **定义** | 基于事务启动时的视图，按照可见性规则读取历史版本 | 读取数据的最新已提交版本，并加锁 |
| **触发方式** | 普通 `SELECT` 语句 | `UPDATE` / `DELETE` / `SELECT ... FOR UPDATE` / `SELECT ... LOCK IN SHARE MODE` |
| **是否加锁** | ❌ 不加锁 🔓 | ✅ 加行锁 🔒 |
| **读取版本** | 根据可见性规则判断（可能是历史版本） | 始终读取**最新版本** |
| **使用机制** | **MVCC** | **锁 + MVCC** |
| **性能** | 高（无锁等待） | 可能存在锁等待 |
| **应用场景** | 可重复读隔离级别下的普通查询 | 更新操作必须使用当前读 |

💡 **关键规则**：更新操作必须使用当前读，不能在历史版本上更新！

## 🎯 七、MVCC 实战应用场景

### 📌 7.1 适用场景
- ✅ **高并发读多写少**的业务系统
- ✅ 需要**可重复读**保证的场景（如对账、报表统计）
- ✅ **OLTP（在线事务处理）** 系统
- ✅ 数据校验和数据一致性检查

### 📌 7.2 不适用场景
- ❌ **写密集型**业务（频繁更新导致版本链过长）
- ❌ 需要实时读到最新数据的场景（应使用 RC 或当前读）
- ❌ **批量数据处理**（可能产生大量 undo log）
- ❌ 对存储空间敏感的系统（长事务导致 undo log 膨胀）

### 📌 7.3 性能优化建议
- 合理选择**隔离级别**（根据业务需求选择 RC 或 RR）
- 避免**长事务**（及时提交，减少 undo log 积压）
- 对于**热点数据**的频繁更新，考虑缓存层卸载压力
- 定期监控 **undo log** 空间使用情况



## 💡 九、面试高频问题速查

### Q1：什么是 MVCC？它解决了什么问题？
**答案要点**：
- Multi-Version Concurrency Control（多版本并发控制）
- 通过维护数据的**多个版本**实现读写不冲突
- 解决**读写阻塞**问题，提高并发性能
- 实现**快照读**，避免加锁

### Q2：MVCC 的实现原理是什么？
**答案要点**：
- **三大支柱**：隐藏字段 + Undo Log 版本链 + ReadView
- **隐藏字段**：DB_TRX_ID、DB_ROLL_PTR、DB_ROW_ID
- **版本链**：通过 undo log 形成链表结构
- **ReadView**：判断数据可见性的核心数据结构

### Q3：ReadView 的可见性判断规则是什么？
**答案要点**：
- 规则 1：自己修改的 → 可见
- 规则 2：视图前提交的 → 可见
- 规则 3：视图后启动的 → 不可见
- 规则 4：活跃事务修改的 → 不可见
- 规则 5：以上都不满足 → 回溯旧版本

### Q4：RC 和 RR 隔离级别的区别是什么？
**答案要点**：
- **RC**：每次 SELECT 都生成新 ReadView → 会出现不可重复读
- **RR**：只在第一次 SELECT 时生成 ReadView → 实现可重复读
- 这是两者最核心的区别！

### Q5：为什么 UPDATE 必须使用当前读？
**答案要点**：
- 如果允许在历史版本上更新会导致**丢失更新**
- 多个事务基于不同版本更新会产生**数据不一致**
- 必须在最新版本基础上更新才能保证**事务隔离性**

### Q6：长事务有什么危害？如何避免？
**答案要点**：
- **三大危害**：回滚日志膨胀、锁资源占用、连接池污染
- **避免方法**：显式 begin/commit、监控事务时长、拆分长事务、合理选择隔离级别
