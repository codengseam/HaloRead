# 核心知识清单（20%关键项）

## 🔹 过期读问题的本质

**定义**：客户端执行完更新事务后立即查询从库，可能读到事务更新前的状态。

**根本原因**：
- 主库执行完成 → binlog发送 → 从库接收 → 从库执行
- 客户端收到提交确认时，从库可能还未收到或执行该事务的binlog

**思考**：为什么主备延迟无法100%避免？即使优化了网络和硬件，哪些场景仍会导致延迟？



## 🔹 等主库位点方案（精确解决过期读）

**核心命令**：
```sql
SELECT master_pos_wait(file, pos[, timeout]);
```

**执行流程**：
1. 事务更新后，执行`SHOW MASTER STATUS`获取File和Position
2. 选定从库执行查询
3. 在从库执行`SELECT master_pos_wait(File, Position, 1)`
4. 返回值≥0 → 在从库查询；否则 → 到主库查询

**返回值含义**：
- 正整数M：从开始执行到应用完位点，执行了M个事务
- NULL：备库同步线程异常
- -1：等待超时
- 0：已执行过该位置

**关键优势**：只等待特定事务，避免过度等待

**思考**：如果所有从库延迟都超过1秒，查询压力会集中到哪？如何设计限流策略？



## 🔹 强制走主库方案（最实用）

**分类策略**：
- 必须拿到最新结果的请求 → 强制发到主库
- 可接受旧数据的请求 → 发到从库

**典型场景**：
- 卖家发布商品后查看 → 走主库
- 买家浏览商品列表 → 走从库

**局限性**：金融类业务所有查询都不能过期读 → 放弃读写分离

**思考**：如何识别哪些查询必须走主库？代码层面如何优雅实现分类逻辑？



# 思考问答

1. **场景题**：你的系统采用等GTID方案，主库突然执行大表DDL（耗时10分钟），会发生什么？如何避免？
   
   **提示**：DDL期间主备延迟持续增大 → 等GTID超时 → 流量全打到主库 → 主库压力骤增
