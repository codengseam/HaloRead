# 日志系统:一条SQL更新语句是如何执行的？

## 核心概念

### 1. WAL技术(Write-Ahead Logging)

**定义**:先写日志,再写磁盘的核心思想。

**类比理解**:酒店掌柜的粉板与账本
- 粉板(redo log):快速记录当前操作,提升效率
- 账本(磁盘数据):系统空闲时批量更新

**核心价值**:将随机写转换为顺序写,大幅提升更新性能。



### 🔹 binlog(归档日志)

#### 核心特性
- **所属层级**:MySQL Server层实现,所有引擎可用
- **日志类型**:逻辑日志,记录语句的原始逻辑(如"给ID=2的c字段加1")
- **写入方式**:追加写,不覆盖历史日志
- **核心用途**:数据恢复、主从复制

#### binlog格式
| 格式 | 说明 | 适用场景 |
|------|------|----------|
| statement | 记录SQL语句 | 日志量小,但时间函数可能导致主从不一致 |
| row | 记录更新前后的行数据 | 数据一致性强,推荐使用 |
| mixed | 混合模式 | 自动选择 |

#### 关键参数
```sql
sync_binlog = 1
```
**含义**:每次事务的binlog都持久化到磁盘
**建议**:设置为1,保证异常重启后binlog不丢失



## 次要信息速览(80%快速带过)

### 【更新语句执行流程】
- 连接器建立连接
- 分析器解析语句
- 优化器选择索引
- 执行器执行更新
- 清空查询缓存(不建议使用查询缓存)

### 【redo log与binlog的三大区别】
1. 所属层级不同:引擎层 vs Server层
2. 日志类型不同:物理日志 vs 逻辑日志
3. 写入方式不同:循环写 vs 追加写

### 【数据恢复流程】
- 找最近全量备份 → 恢复到临时库
- 从备份时间点开始 → 重放binlog到目标时刻
- 从临时库导出数据 → 恢复到线上库

### 【备份周期选择】
- 一天一备:恢复速度快,binlog依赖少
- 一周一备:存储成本低,但恢复时间长,binlog依赖多



## 实践要点

### 生产环境配置建议

```sql
-- 保证redo log持久化
innodb_flush_log_at_trx_commit = 1

-- 保证binlog持久化
sync_binlog = 1

-- binlog格式推荐
binlog_format = ROW
```

### 监控指标

- redo log使用率:write pos与checkpoint距离
- binlog大小与增长速度
- 备份完整性:定期验证备份可恢复性
