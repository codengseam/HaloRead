# MySQL主备一致性的核心机制

> **思考问题**：为什么MySQL几乎所有高可用架构都依赖binlog？如果binlog格式选择不当，会导致什么严重后果？

## 核心知识清单（20%重点）

### 1. 主备复制的基本原理

**定义**：主库将更新操作记录到binlog，备库通过IO线程拉取binlog并写入relay log，SQL线程解析relay log并执行，从而保持与主库数据一致。

**关键流程**：
```
主库A执行更新 → 写入binlog
                ↓
备库B的io_thread连接主库 → 拉取binlog
                ↓
写入本地relay log（中转日志）
                ↓
sql_thread读取relay log → 解析并执行
```

**应用场景**：
- 一主一备架构：备库设置为只读（readonly），防止误操作和双写
- 双M架构：A、B互为主备，切换时无需修改主备关系

**常见错误及规避**：
- ❌ 错误：认为备库设置readonly后无法同步更新
- ✅ 正确：readonly对超级权限用户无效，同步线程拥有超级权限

**实操练习**（30秒落地）：
在备库执行 `SHOW PROCESSLIST`，识别 `system user` 线程（即io_thread和sql_thread），观察其状态是否为 `Waiting for master to send event`。



### 3. 循环复制问题与server_id机制

**问题场景**：双M架构下，A→B→A形成循环复制。

**解决方案**：server_id机制

**核心逻辑**：
1. 每个MySQL实例必须有唯一的server_id
2. binlog记录执行该事务的server_id
3. 备库重放binlog时，生成相同server_id的新binlog
4. 收到binlog时，若server_id与自己相同，直接丢弃

**执行流程示例**：
```
节点A执行事务 → binlog记录server_id=A
        ↓
节点B执行 → 生成binlog的server_id仍为A
        ↓
传回节点A → A判断server_id=A（与自己相同）→ 丢弃
```

**思考问题**：如果同步过程中修改了server_id，会发生什么？（答案：两个节点都认为不是自己的binlog，导致循环执行）
