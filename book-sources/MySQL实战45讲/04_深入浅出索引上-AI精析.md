# 核心知识清单(20%)

## 1. 索引的本质

**定义**:索引是数据库表的"目录",核心目标是提高数据查询效率。

**类比理解**:就像一本500页的书,通过目录可以快速定位知识点,而不是逐页翻找。



### 2.2 有序数组索引

**核心特点**:
- 数据按key递增顺序存储
- 查询用二分法,时间复杂度O(log(N))

**适用场景**:
- ✅ **等值查询性能优秀**
- ✅ **范围查询性能优秀**
- ❌ **插入/更新成本极高**(需挪动后续所有记录)

**典型应用**:静态存储引擎(如2017年某城市人口信息,数据不再修改)



## 3. InnoDB的B+树索引模型

### 3.1 索引组织表

**定义**:InnoDB中,表根据主键顺序以索引形式存放,称为**索引组织表**。

**核心机制**:
- 每个索引对应一棵B+树
- 数据存储在B+树中



## 4. 索引维护:页分裂与合并

### 4.1 页分裂

**触发场景**:插入新数据时,目标数据页已满

**影响**:
- 性能下降:需申请新数据页,挪动部分数据
- 空间利用率降低:原本一页的数据分到两页,利用率约降低50%

### 4.2 页合并

**触发场景**:相邻两页因删除数据,利用率很低

**过程**:分裂的逆过程



### 5.2 何时用业务字段做主键?

**适用场景**(KV场景):
- 只有一个索引
- 该索引必须是唯一索引

**优势**:
- 没有其他索引,不用担心二级索引叶子节点大小问题
- 直接用业务字段做主键,避免回表(不用搜索两棵树)

**典型例子**:用户表用身份证号做主键,且只有这一个唯一索引



# 思考问答

## 1. 深度理解题

**问题**:如果表T有两个索引(主键id和普通索引k),执行以下两条SQL的性能差异是什么?为什么?

```sql
-- SQL 1
SELECT * FROM T WHERE id = 500;

-- SQL 2
SELECT * FROM T WHERE k = 5;
```

**思考路径**:
- SQL 1:直接搜索主键索引树,叶子节点包含整行数据,无需回表
- SQL 2:先搜索k索引树→得到id=500→再搜索主键索引树(回表)
- **SQL 1更快,因为只需扫描一棵树**



## 3. 原理深挖题

**问题**:为什么InnoDB选择B+树而不是B树作为索引结构?

**提示**:
- B+树非叶子节点只存储key,不存储数据
- B树每个节点都存储key和data
- B+树叶子节点用指针连接,形成有序链表

**思考路径**:
- B+树非叶子节点不存数据,单页可存更多key→树更矮→磁盘访问次数更少
- B+树叶子节点有序链表→范围查询更高效(只需遍历链表)
