# 核心知识清单（20%重点）

## 🔹 内部临时表的本质与使用场景

### 核心概念
内部临时表是MySQL在SQL语句执行过程中,用于存储中间结果的内存结构。与sort buffer(有序数组)、join buffer(无序数组)不同,临时表是**二维表结构**,支持主键索引约束。

### 使用时机判断
**核心原则**:如果执行逻辑需要用到二维表特性,就会优先考虑使用临时表。

| 数据结构 | 特性 | 使用场景 |
|---------|------|---------|
| join_buffer | 无序数组 | join连接 |
| sort_buffer | 有序数组 | 排序操作 |
| 临时表 | 二维表+主键索引 | union去重、group by分组统计 |

**思考**:为什么union需要临时表而union all不需要?→union需要去重,利用临时表主键唯一性约束实现;union all直接返回结果,无需去重。



## 🔹 GROUP BY执行流程与临时表机制

### 标准执行流程
```
1. 创建内存临时表(字段m, c; 主键m)
2. 扫描表t1索引a,计算id%10
   - 临时表无m值 → 插入(m, 1)
   - 临时表有m值 → c值+1
3. 对临时表按m排序
4. 返回结果
```

### 内存临时表 vs 磁盘临时表

| 对比项 | 内存临时表 | 磁盘临时表 |
|-------|-----------|-----------|
| 存储引擎 | Memory引擎 | InnoDB(默认) |
| 大小限制 | tmp_table_size(默认16M) | 磁盘空间 |
| 性能 | 快 | 慢 |
| 数据结构 | 堆表(无序) | B+树(有序) |

**关键参数**:
- `tmp_table_size`:控制内存临时表最大值
- 超过限制 → 自动转为磁盘临时表

**思考**:为什么内存临时表order by null时0在最后,磁盘临时表0在最前?→内存临时表是堆表(插入顺序),磁盘临时表是InnoDB(主键有序)。



### 优化方法二:SQL_BIG_RESULT提示

**原理**:告诉优化器数据量大,直接使用排序算法,跳过临时表。

**执行流程**:
```
1. 初始化sort_buffer(字段m)
2. 扫描表t1,将id%100存入sort_buffer
3. 对sort_buffer排序(可能用磁盘临时文件)
4. 遍历有序数组,统计不同值出现次数
```

**语法**:
```sql
SELECT SQL_BIG_RESULT id%100 AS m, COUNT(*) AS c FROM t1 GROUP BY m;
```

**适用场景**:
- 数据量大,不适合创建索引
- 明确知道结果集很大

**错误规避**:误用SQL_BIG_RESULT在小数据集 → 反而降低性能(排序开销)。



## 🔹 GROUP BY优化决策树

```
需要group by?
├─ 分组字段可建索引?
│  ├─ 是 → 创建索引 → explain验证无Using temporary
│  └─ 否 → 判断数据量
│           ├─ 小(tmp_table_size内) → 默认内存临时表
│           ├─ 中 → 调大tmp_table_size
│           └─ 大 → 使用SQL_BIG_RESULT
└─ 需要排序结果?
   ├─ 是 → 默认行为
   └─ 否 → 添加ORDER BY NULL
```
