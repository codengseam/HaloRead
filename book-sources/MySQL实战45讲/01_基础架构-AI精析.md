# 核心知识清单(20%重点)

## 一、MySQL双层架构体系

### 1. Server层(服务层)
**定义**: MySQL的核心服务层,负责SQL语句的解析、优化和执行,所有跨存储引擎的功能都在此实现。

**关键组件**:
- 连接器: 连接管理 + 权限验证
- 查询缓存: 已废弃(MySQL 8.0删除)
- 分析器: 词法分析 + 语法分析
- 优化器: 执行计划选择
- 执行器: 调用引擎接口执行

**应用场景**: 所有SQL语句都必须经过Server层的完整处理流程。

**常见错误**: 混淆Server层和存储引擎层的职责边界 → Server层负责"做什么"和"怎么做",存储引擎负责"怎么存"。

> 思考:为什么MySQL要将Server层和存储引擎层分离?这种架构设计带来了什么好处?

### 2. 存储引擎层
**定义**: 负责数据的存储和提取,采用插件式架构,支持多种存储引擎。

**关键特性**:
- 插件式架构: InnoDB、MyISAM、Memory等
- 默认引擎: InnoDB(MySQL 5.5.5+)
- 不同引擎: 数据存取方式不同,支持功能不同

**实操**: 建表时指定引擎
```sql
CREATE TABLE T (...) ENGINE=InnoDB;  -- 默认
CREATE TABLE T (...) ENGINE=Memory;  -- 内存引擎
```



## 三、关键知识点总结

### 1. 权限验证的两个时机
- **连接建立时**: 连接器读取权限并缓存
- **执行前**: 执行器检查表级权限
- **特殊情况**: 查询缓存命中时在返回结果前验证,优化器前有precheck

### 2. 长连接内存管理
**问题根源**: MySQL执行过程中临时内存管理在连接对象中,连接断开才释放。

**解决方案对比**:
| 方案 | 优点 | 缺点 | 适用版本 |
|------|------|------|----------|
| 定期断开重连 | 简单可靠 | 需要重新权限验证 | 所有版本 |
| mysql_reset_connection | 无需重连,快速 | 仅重置资源 | MySQL 5.7+ |

### 3. 查询缓存为何被废弃
- 更新操作导致缓存大面积失效
- 维护缓存成本 > 缓存命中收益
- 现代数据库更依赖Buffer Pool机制
