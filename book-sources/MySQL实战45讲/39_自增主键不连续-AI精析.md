# 自增主键不连续的核心解析

## 核心知识清单（20%重点）

### 🔹 自增值存储机制

**InnoDB引擎的自增值存储策略**：
- **MySQL 5.7及之前**：保存在内存中，重启后自增值 = max(id) + 1
- **MySQL 8.0+**：持久化到redo log，重启后恢复重启前的值

**关键差异**：
- MyISAM：自增值保存在数据文件中（天然持久化）
- InnoDB：早期版本重启会丢失自增值，8.0才解决

> **思考**：为什么InnoDB直到8.0才实现自增值持久化？提示：考虑redo log的设计初衷



#### 原因二：事务回滚

**现象**：
```sql
BEGIN;
INSERT INTO t VALUES(null, 2, 2);  -- 申请到id=2
ROLLBACK;  -- 回滚，但自增值已变为3
-- 下次插入得到id=3
```

**为什么不能回退自增值？**
- **并发场景问题**：事务A申请id=2，事务B申请id=3，若A回滚后回退自增值，后续事务会申请到id=2，导致主键冲突
- **性能代价**：若要回退，需每次申请前检查id是否存在，或扩大锁范围，都会严重影响性能

> **思考**：如果允许自增值回退，在高并发场景下会出现什么问题？



### 🔹 自增锁优化策略

#### innodb_autoinc_lock_mode参数

| 值 | 行为 | 适用场景 |
|---|---|---|
| 0 | 语句级锁（MySQL 5.0策略） | 兼容旧版本 |
| 1 | 普通insert立即释放，批量insert语句级锁 | 默认值 |
| 2 | 所有insert立即释放锁 | 高并发场景 |

#### 为什么默认值不是2？

**数据一致性问题**：
- `binlog_format=statement`时，批量插入生成的id可能不连续
- 主库和备库执行顺序不同，导致数据不一致

**解决方案**：
```sql
-- 生产环境推荐配置
innodb_autoinc_lock_mode = 2
binlog_format = row
```

**为什么这样配置安全？**
- `lock_mode=2`：立即释放锁，提升并发
- `binlog_format=row`：记录实际数据值，不依赖自增逻辑，避免主备不一致

> **思考**：为什么普通insert（多值）不受批量插入锁的限制？提示：可以预先计算需要的id数量



## 实践要点

### ✅ 生产环境配置建议
```sql
-- 查看当前配置
SHOW VARIABLES LIKE 'innodb_autoinc_lock_mode';
SHOW VARIABLES LIKE 'binlog_format';

-- 推荐配置
SET GLOBAL innodb_autoinc_lock_mode = 2;
SET GLOBAL binlog_format = 'row';
```

### ✅ 业务设计注意事项
1. **不要依赖自增主键的连续性**：业务逻辑不应假设id连续
2. **监控自增值空洞**：大量空洞可能影响性能（id值过大）
3. **批量插入优化**：使用`binlog_format=row`配合`lock_mode=2`

### ✅ 故障排查
- **自增值异常**：检查是否有大量唯一键冲突或事务回滚
- **重启后自增值变化**：MySQL 5.7及以下版本正常现象，升级到8.0解决
