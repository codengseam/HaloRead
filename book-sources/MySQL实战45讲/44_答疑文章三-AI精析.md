# 核心知识清单(20%重点)

## 1. join的写法陷阱(核心)

### 1.1 left join的驱动表选择

**关键结论**:使用left join时,左边的表不一定是驱动表

**核心机制**:
- 优化器会根据查询语义进行改写
- 如果where条件中包含被驱动表的字段判断,优化器会将left join改写为join
- 改写后重新选择驱动表以利用索引

**实践示例**:
```sql
-- Q1: 标准left join语义(返回6行)
select * from a left join b on(a.f1=b.f1) and (a.f2=b.f2);

-- Q2: 被优化器改写为join(返回4行)
select * from a left join b on(a.f1=b.f1) where (a.f2=b.f2);
```

**执行差异**:
- Q1: 表a为驱动表,使用BNL算法
- Q2: 表b为驱动表,使用NLJ算法(因为表a.f1有索引)

**思考**:为什么where条件会导致left join语义改变?

### 1.2 on与where的本质区别

**核心规则**:
- **left join**: 条件写在on中保留左表所有行,写在where中会过滤掉NULL值行
- **join**: on和where无区别,优化器会统一改写

**关键知识点**:
- MySQL中NULL与任何值的等值判断结果都是NULL
- `where a.f2=b.f2`会过滤掉b.f2为NULL的行
- 这导致left join语义退化为join语义

**实践要点**:
```sql
-- 需要left join语义时,必须将条件写在on中
select * from a left join b on(a.f1=b.f1) and (a.f2=b.f2);

-- 如果where中包含被驱动表字段,直接写join更清晰
select * from a join b on(a.f1=b.f1) where (a.f2=b.f2);
```

**常见错误**:混淆left join语义,将过滤条件写在where中导致结果不符合预期



## 3. distinct与group by性能等价(核心)

### 3.1 语义等价性

**关键结论**:不需要聚合函数时,distinct和group by语义相同,性能相同

**示例对比**:
```sql
-- 语义等价,性能相同
select a from t group by a order by null;
select distinct a from t;
```

### 3.2 执行流程

**统一执行流程**:
1. 创建临时表,字段a上有唯一索引
2. 遍历表t,插入临时表:
   - 唯一键冲突则跳过
   - 否则插入成功
3. 返回临时表作为结果集

**实践要点**:
- 标准group by应包含聚合函数
- 仅去重时,distinct语义更清晰
- 性能无差异,选择可读性更好的写法

**思考**:为什么MySQL没有为distinct设计专门的优化算法?
