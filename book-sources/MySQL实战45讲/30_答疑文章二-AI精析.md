# 核心知识清单(20%)

## 🔹 加锁规则体系

### 两个基本原则
- **原则1**: 加锁基本单位是next-key lock(前开后闭区间)
- **原则2**: 查找过程中访问到的对象才会加锁

### 两个优化策略
- **优化1**: 唯一索引等值查询 → next-key lock退化为行锁
- **优化2**: 索引等值查询向右遍历时,最后一个值不满足条件 → next-key lock退化为间隙锁

### 一个Bug
- **唯一索引范围查询**: 会访问到不满足条件的第一个值为止

**思考**: 为什么优化2只适用于"最后一个值不满足条件"的情况?中间值不满足条件会怎样?



## 🔹 间隙的本质定义

### 核心观点
**"间隙"由该间隙右边的记录定义**

#### 实例验证
```
原状态: id=5, 10, 15
间隙: (5,10), (10,15)

执行delete id=10后:
间隙: (5,15)  // 由id=15定义
```

**思考**: 空表的间隙是什么?由谁定义?



## 🔹 死锁分析方法

### show engine innodb status解读要点

#### LATEST DETECTED DEADLOCK结构
1. **TRANSACTION (1)**: 第一个事务信息
   - `WAITING FOR THIS LOCK TO BE GRANTED`: 等待的锁
   - `lock mode S waiting`: 等待读锁
   - `Record lock`: 记录锁类型

2. **TRANSACTION (2)**: 第二个事务信息
   - `HOLDS THE LOCK(S)`: 持有的锁
   - `WAITING FOR THIS LOCK TO BE GRANTED`: 等待的锁

3. **WE ROLL BACK TRANSACTION (1)**: 回滚决策
   - 回滚成本更小的事务

**实践要点**: 死锁现场信息不完备,需结合业务逻辑推导完整锁状态
