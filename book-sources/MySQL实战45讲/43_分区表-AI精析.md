# 要不要使用分区表？- AI精析

## 一、核心概念：分区表的双重身份

### 1.1 分区表的本质定义

分区表在MySQL中具有**双重身份**：

- **对Server层**：这是**1个表**
- **对引擎层**：这是**多个表**（每个分区对应一个独立的.ibd文件）

这个双重身份是理解分区表所有行为的关键。

### 1.2 实际案例验证

通过间隙锁实验验证引擎层行为：
- 普通表：间隙锁会锁住记录间的间隙
- 分区表：间隙锁只在**单个分区内生效**，跨分区不受影响

**关键结论**：引擎层的锁机制以分区为单位，不同分区被视为独立表。



## 三、分区表的三大核心问题

### 3.1 问题一：首次访问需打开所有分区

**现象**：
- 第一次访问分区表时，MySQL需访问所有分区
- 分区过多（如1000+）时可能触发文件打开上限错误

**影响范围**：
- MyISAM：受`open_files_limit`限制，易报错
- InnoDB：使用本地分区策略，相对安全

**实践建议**：
- 控制分区数量（避免按天分区且预创建多年）
- 单表/单分区1000万行已是小表，无需过度分区

### 3.2 问题二：共用MDL锁

**现象**：
- Server层认为这是同一张表
- 所有分区共用同一个MDL锁
- DDL操作影响范围更大

**对比手动分表**：
- 手动分表：truncate一个分表不影响其他分表查询
- 分区表：DDL操作会锁住整个分区表

**关键影响**：
```sql
-- Session A: 持有整个表的MDL锁
-- Session B: 即使只操作单个分区，也会被阻塞
ALTER TABLE t DROP PARTITION p_2017;
```

### 3.3 问题三：分区键选择

**查询优化依赖**：
- WHERE条件包含分区键：只访问必要分区
- WHERE条件不含分区键：访问所有分区

**示例对比**：
```sql
-- 只访问p_2019分区
WHERE ftime = '2018-4-1'

-- 需访问p_2019和p_others两个分区
WHERE ftime >= '2018-4-1'

-- 访问所有分区
WHERE c = 1  -- 未使用分区键
```



## 五、最佳实践指南

### 5.1 分区粒度控制

**核心原则**：
- 单表/单分区1000万行已是小表（现代硬件能力）
- 分区并非越细越好
- 避免按天分区且预创建多年

### 5.2 分区预留策略

**推荐做法**：
- 按月分区：每年年底创建下一年度的12个分区
- 按年分区：提前1-2年创建
- 及时清理无数据的历史分区

### 5.3 主键设计要求

**MySQL约束**：分区表的主键必须包含分区字段

**设计示例**：
```sql
-- 原表定义
PARTITION BY RANGE (YEAR(ftime))

-- 主键必须包含ftime字段
PRIMARY KEY (id, ftime)
-- 或
PRIMARY KEY (ftime, id)
```

**设计理由**：
- 确保主键在全局唯一
- 保证数据分布符合分区规则



## 七、关键决策点

### 7.1 适合使用分区表的情况

- 单机环境
- 历史数据需定期清理
- 团队无成熟分库分表中间件
- 分区数量可控（< 100个）

### 7.2 不适合使用分区表的情况

- 需要跨多个分区频繁查询
- 分区数量过多（如按天分区多年）
- 团队已有成熟分库分表中间件
- 需要独立DDL操作各分区



## 九、延伸思考

### 思考题
如果需要在分区表t中增加自增主键id，应该如何定义主键？为什么？

**答案要点**：
```sql
-- 方案1：ftime在前
PRIMARY KEY (ftime, id)

-- 方案2：id在前，ftime在后
PRIMARY KEY (id, ftime)
```

**设计考量**：
- 必须包含分区字段ftime
- 方案1：利于按时间范围查询，但id非递增
- 方案2：id自增，但需确保ftime字段在主键中
