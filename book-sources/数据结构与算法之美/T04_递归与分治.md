# 核心观点

递归是**"将问题分解为子问题"的编程技巧**，核心在于找到**递推公式和终止条件**；分治是**"分解→解决→合并"的算法思想**，典型应用包括归并排序、快速排序和MapReduce。理解递归的三个条件、常见陷阱（堆栈溢出、重复计算）以及与迭代的关系，能帮你写出简洁高效的代码，并为后续学习动态规划、回溯算法打下基础。

## 核心概念解析（20% 重点深度解析）

### 1. 递归的三要素与思维方法

- **定义**：函数直接或间接调用自身来解决问题的编程技巧
- **为什么重要**：
  - ⭐ **DFS、二叉树遍历、回溯、分治的基础**
  - 代码表达力极强，几行代码就能解决复杂问题（如汉诺塔、斐波那契）
  - 面试高频考点，但也是最容易出错的地方

- **关键机制——递归三要素**：

  **① 问题的解可以分解为几个子问题的解**
  ```
  子问题 = 数据规模更小的同类问题

  示例：
  - 走台阶问题：f(n) = f(n-1) + f(n-2)
    （n阶台阶的走法 = 先走1阶后剩下的走法 + 先走2阶后剩下的走法）
  - 查找最终推荐人：findRoot(userId) = findRoot(referrerId)
  - 二叉树深度：depth(root) = max(depth(left), depth(right)) + 1
  ```

  **② 问题与子问题的求解思路完全相同**
  ```
  只是数据规模不同，逻辑一致

  示例（走台阶）：
  f(5) 的思路：分解为 f(4) 和 f(3)
  f(4) 的思路：分解为 f(3) 和 f(2)   ← 与f(5)完全相同的模式！
  f(3) 的思路：分解为 f(2) 和 f(1)   ← 同上
  ...
  ```

  **③ 存在递归终止条件（Base Case）**
  ```
  必须有明确的停止点，否则无限递归导致栈溢出

  示例（走台阶）：
  f(1) = 1  （只有1种走法：走1步）
  f(2) = 2  （2种走法：走2次1步 或 直接走2步）

  ⚠️ 终止条件必须足够且正确，否则：
  - 缺少 → StackOverflowError（死循环）
  - 错误 → 结果不对（如只写f(1)=1但没处理f(2)）
  ```

- **编写递归代码的正确姿势**：

  ❌ **错误思维**：试图用人脑模拟整个"递→归"过程（容易绕进去）
  ✅ **正确思维**：
  ```
  步骤1：写出递推公式（数学建模）
     f(n) = f(n-1) + f(n-2), 其中 f(1)=1, f(2)=2

  步骤2：确定终止条件（边界情况）
     if (n == 1) return 1;
     if (n == 2) return 2;

  步骤3：假设子问题已解决，思考如何组合
     return f(n-1) + f(n-2);  // 不需要想f(n-1)内部怎么算！

  核心原则：屏蔽递归细节，只关注当前层与下一层的关系
  ```

  **经典案例：三行代码找最终推荐人**
  ```java
  long findRootReferrerId(long actorId) {
      Long referrerId = select referrer_id from table where actor_id = actorId;
      if (referrerId == null) return actorId;  // 终止条件：没有推荐人
      return findRootReferrerId(referrerId);    // 递归调用
  }
  // 简洁优雅，但要注意：可能栈溢出 + 可能有环（A→B→C→A）
  ```

- **时间/空间复杂度**：
  - 时间：取决于问题本身（如斐波那契O(2ⁿ)，二分查找O(log n)）
  - 空间：**O(递归深度)**（每次调用都要在栈中保存现场）
- **实际应用**：
  - **树形结构遍历**：文件系统目录遍历、DOM树操作、组织架构查询
  - **分治算法**：归并排序、快速排序、大整数乘法（Karatsuba算法）
  - **回溯算法**：N皇后、全排列、正则表达式匹配

### 2. 递归的陷阱与解决方案

- **定义**：使用递归时常见的三类问题及其应对策略
- **为什么重要**：
  - 面试官常追问："你的递归代码有什么问题？如何优化？"
  - 实际项目中不当使用递归会导致生产事故（StackOverflowError、性能瓶颈）
  - 掌握这些才能写出**健壮的递归代码**

- **关键机制——三大陷阱及解法**：

  **陷阱①：堆栈溢出（Stack Overflow）**

  原因：递归层次太深，超出系统栈空间限制（通常1MB~8MB）
  ```
  示例：f(19999) 在默认JVM配置下会抛出StackOverflowError

  解决方案：
  方案A：限制递归深度（适用于深度可控的场景）
    int depth = 0;
    int f(int n) {
        if (++depth > 1000) throw new RuntimeException("Too deep");
        if (n == 1) return 1;
        return f(n-1) + 1;
    }

  方案B：改用迭代（循环替代递归）
    int f(int n) {
        int ret = 1;
        for (int i = 2; i <= n; i++) ret += 1;
        return ret;
    }

  方案C：尾递归优化（部分语言/编译器支持）
    // Scala/Haskell等函数式语言可自动优化为循环
    def f(n: Int, acc: Int = 1): Int = {
        if (n == 1) acc else f(n-1, acc+1)
    }
  ```

  **陷阱②：重复计算（Overlap Subproblem）**

  原因：同一子问题被多次计算（如斐波那契f(5)=f(4)+f(3)，而f(4)又要算f(3)）
  ```
  斐波那契的重复计算问题：
  f(5)
   ├── f(4)
   │    ├── f(3)
   │    │    ├── f(2) ← 计算1次
   │    │    └── f(1)
   │    └── f(2) ← 又计算1次！（重复！）
   └── f(3)
        ├── f(2) ← 第3次计算！！！
        └── f(1)

  时间复杂度：O(2ⁿ)（指数级爆炸！）

  解决方案：记忆化搜索（Memoization）
  Map<Integer, Integer> memo = new HashMap<>();

  int f(int n) {
      if (n == 1 || n == 2) return n;
      if (memo.containsKey(n)) return memo.get(n);  // 查缓存
      int ret = f(n-1) + f(n-2);
      memo.put(n, ret);  // 存结果
      return ret;
  }
  // 优化后时间复杂度：O(n)（每个f(k)只算一次）
  ```

  **陷阱③：函数调用开销**

  原因：每次递归调用都有压栈、跳转、保存现场等开销
  ```
  影响：
  - 时间：频繁的函数调用比循环慢（虽然现代编译器会优化）
  - 空间：每层调用占用栈帧空间（局部变量、返回地址等）

  解决方案：根据场景选择
  - 简单线性递归（如求和）→ 用循环更高效
  - 树形结构递归（如DFS）→ 保持递归，代码更清晰
  - 性能敏感场景 → 改为迭代 + 显式栈
  ```
- **时间/空间复杂度对比**：

| 问题 | 朴素递归 | 记忆化 | 迭代 |
|------|----------|--------|------|
| 斐波那契 | O(2ⁿ) ❌ | O(n) ✅ | O(n) ✅ |
| 二叉树深度 | O(n) ✅ | O(n) ✅ | O(n) ✅（需用栈）|
| 走台阶 | O(2ⁿ) ❌ | O(n) ✅ | O(n) ✅ |
| 汉诺塔 | O(2ⁿ) ✅（无法优化）| O(2ⁿ) | O(2ⁿ)（需模拟递归）|

- **实际应用**：
  - **数据库查询**：递归CTE（Common Table Expression）查询组织架构树
  - **前端渲染**：React/Vue的虚拟DOM diff算法（递归对比树节点）
  - **编译器**：语法分析（递归下降解析器）
- **关联知识**：[[T14_动态规划]]（递归+记忆化=自顶向下DP）

### 3. 分治算法（Divide and Conquer）

- **定义**：将复杂问题**分解**为多个独立的子问题，分别**求解**后**合并**结果的算法范式
- **为什么重要**：
  - ⭐ **MapReduce、分布式计算的理论基础**
  - 归并排序、快速排序、大数乘法的底层思想
  - 与动态规划的区别：**子问题独立 vs 子问题重叠**

- **关键机制——分治三步曲**：

  **步骤1：分解（Divide）**
  ```
  将原问题分解为若干个规模较小的子问题
  - 归并排序：将数组分成两半
  - 快速排序：选择pivot分区
  - 大数乘法：拆分为高位×高位 + 高位×低位 + 低位×高位 + 低位×低位
  ```

  **步骤2：解决（Conquer）**
  ```
  递归地解决子问题（如果子问题仍大则继续分解）
  - 当子问题规模足够小时直接求解（Base Case）
  - 归并排序：当数组长度为1时已有序
  - 快速排序：当区间只有一个元素时无需排序
  ```

  **步骤3：合并（Combine）**
  ```
  将子问题的解合并成原问题的解
  - 归并排序：合并两个有序数组（O(n)）
  - 快速排序：不需要显式合并（原地分区即可）
  - 大数乘法：将四部分结果相加并处理进位
  ```

  **经典案例：归并排序（Merge Sort）**
  ```
  递推公式：T(n) = 2T(n/2) + O(n)
  终止条件：n = 1 时返回

  过程示例：[8, 3, 5, 2, 9, 1, 6, 4]
  分解：
  [8,3,5,2] [9,1,6,4]
  [8,3] [5,2] [9,1] [6,4]
  [8] [3] [5] [2] [9] [1] [6] [4]

  合并（两两归并）：
  [3,8] [2,5] [1,9] [4,6]
  [2,3,5,8] [1,4,6,9]
  [1,2,3,4,5,6,8,9]  ✅

  时间复杂度：O(n log n)（稳定）
  空间复杂度：O(n)（需要辅助数组）
  稳定性：✅ 稳定排序（相等元素相对位置不变）
  ```

  **主定理（Master Theorem）——快速分析分治复杂度**
  ```
  对于 T(n) = aT(n/b) + f(n)：
  - a: 子问题个数
  - b: 子问题规模缩小倍数
  - f(n): 分解和合并的工作量

  三种情况：
  1. f(n) = O(n^c) where c < log_b(a) → T(n) = O(n^{log_b(a})
     例：归并排序 T(n)=2T(n/2)+O(n) → O(n log n)

  2. f(n) = Θ(n^{log_b(a)}) → T(n) = O(n^{log_b(a)} * log n)

  3. f(n) = Ω(n^c) where c > log_b(a) → T(n) = O(f(n))
     例：快排最坏 T(n)=T(n-1)+O(n) → O(n²)
  ```
- **时间/空间复杂度**：
  - 取决于具体算法（归并排序O(n log n)，Strassen矩阵乘法O(n^2.81)）
  - 通常比暴力法优一个数量级
- **实际应用**：

  **① MapReduce中的分治思想**
  ```
  大数据处理框架MapReduce = 分治思想的工业级实现

  流程：
  1. Map阶段（分解）：
     将大数据集切分为多个split，分发到不同机器并行处理
     例：统计词频 → 每个 mapper 处理一部分文档

  2. Shuffle阶段（中间结果汇总）：
     按 key 分组，将相同 key 的数据发送到同一个 reducer

  3. Reduce阶段（合并）：
     每个 reducer 合并处理分配到的数据
     例：对每个单词的局部计数求和得到全局计数

  对比分治算法：
  - 分解：数据切分
  - 解决：Map任务并行执行
  - 合并：Reduce任务聚合结果

  优势：分布式计算，处理TB/PB级数据
  案例：Hadoop MapReduce、Spark、Flink
  ```

  **② 快速排序（Quick Sort）**
  ```
  分治策略：
  1. 选pivot（通常选最后一个元素或随机选）
  2. 分区：小于pivot放左边，大于放右边
  3. 递归排序左右两部分

  平均时间：O(n log n)
  最坏时间：O(n²)（已排好序且总是选最差pivot）
  优化：随机化pivot + 小数组改用插入排序

  vs 归并排序：
  - 快排：原地排序（空间O(log n)），但不稳定
  - 归并：需要额外空间O(n)，但稳定
  - 实际中快排更快（缓存友好性更好）
  ```
- **关联知识**：[[T05_排序算法]], [[T14_动态规划]]（区别：重叠子问题vs独立子问题）

## 次要信息速览（80% 快速带过）

- 【递归 vs 迭代的取舍】：
  - **递归优势**：代码简洁、符合问题定义（树形结构）、易于证明正确性
  - **迭代优势**：性能更高（无函数调用开销）、无栈溢出风险、空间效率好
  - **选择原则**：问题天然递归（树/图）→ 递归；线性问题 → 迭代；深度可控 → 递归；深度不可控 → 迭代

- 【递归转迭代的通用方法】：
  - 方法1：手动维护栈（模拟系统栈的行为）
  - 方法2：找出状态转移方程改为循环（如斐波那契用两个变量滚动）
  - 方法3：尾递归优化（支持的语言可自动转换）

- 【分治 vs 动态规划的核心区别】：
  - **分治**：子问题**独立**，没有重叠（如归并排序左右两边互不影响）
  - **动态规划**：子问题**重叠**，需要保存中间结果避免重复计算（如斐波那契）
  - 判断标准：画递归树看是否有重复子问题

- 【调试递归代码的技巧】：
  - 打印日志：每层递归打印参数和返回值
  - 条件断点：设置特定深度的断点
  - 小规模测试：先用n=1,2,3验证正确性再测试大规模
  - 可视化：用工具画出递归调用树（如Python Tutor）

# 实践要点

- ✅ **先写递推公式再写代码**：这是编写正确递归代码的最有效方法
- ✅ **明确终止条件**：确保所有路径都能到达base case，防止无限递归
- ✅ **考虑边界输入**：n=0, n=1, 负数、null等特殊情况的处理
- ✅ **小规模测试先行**：先用简单例子手工验证逻辑再运行大规模测试
- ❌ **试图人脑展开整个递归过程**：这会导致思维混乱，应该相信递归的正确性
- ❌ **忽略递归深度限制**：生产环境要评估最大深度，必要时改用迭代
- ❌ **忘记记忆化优化**：对于有重叠子问题的递归，一定要加缓存（HashMap/数组）
