### 1. Redis 数据结构深度剖析

- **定义**：Redis 的五种核心数据类型（String、List、Hash、Set、ZSET）在底层使用了多种不同的数据结构实现，根据数据量和特征自动选择最优方案
- **为什么重要**：
  - ⭐ **后端开发必备知识**：Redis 是最广泛使用的缓存中间件
  - **"数据结构选择影响性能"的最佳教学案例**

- **关键机制——五种类型的底层实现**：

  **① String：SDS（Simple Dynamic String）**
  ```
  为什么不用 C 语言原生 char[]？

  C 字符串的问题：
  1. 获取长度需要 O(n)（遍历到 '\0'）
  2. 不是二进制安全的（不能包含 '\0'）
  3. 缓冲区溢出风险（strcat 不检查边界）
  4. 修改字符串可能触发 n 次内存重分配

  SDS 的结构：
  struct sdshdr {
      int len;           // 已使用长度（O(1)获取！）
      int free;          // 剩余可用空间
      char buf[];        // 实际存储字符的数组（柔性数组）
  }

  SDS 的优势：
  ✅ O(1) 获取长度
  ✅ 二进制安全（用 len 而非 '\0' 判断结尾）
  ✅ 杜绝缓冲区溢出（API 会检查空间）
  ✅ 空间预分配（减少内存重分配次数）

  空间预分配策略：
  - 修改后长度 < 1MB：分配 2 倍大小（len * 2）
  - 修改后长度 ≥ 1MB：分配 len + 1MB

  编码优化（节省内存）：
  - int: 存储 ≤ 2^63-1 的整数（8字节），不转成字符串
  - embstr: ≤ 39字节的字符串，SDS头和数据连续分配
  - raw: > 39字节的字符串，SDS头和数据分开分配
  ```

  **② Hash：dict + ziplist 双实现**
  ```
  实现一：压缩列表（ziplist）—— 小数据量时使用
  ────────────────────────────────────────
  结构：一块连续内存，紧凑存储多个键值对

  触发条件（可配置）：
  - hash-max-ziplist-entries（默认512）：元素数量阈值
  - hash-max-ziplist-value（默认64）：单元素最大字节

  优点：内存极其紧凑，CPU 缓存友好
  缺点：增删改查都是 O(n)，数据量大时性能下降


  实现二：字典（dict）—— 大数据量时自动转换
  ────────────────────────────────────
  底层：哈希表 + 链地址法解决冲突

  struct dict {
      dictht ht[2];      // 两个哈希表（用于渐进式 rehash）
      int rehashidx;     // rehash 进度，-1 表示未进行
  };

  渐进式 rehash（Redis 的精髓设计）：
  问题：一次性 rehash 100万 key 会导致长时间阻塞！

  Redis 的解决方案：
  1. 维护两个哈希表：ht[0]（旧）和 ht[1]（新）
  2. 将 rehash 分散到每次增删改查操作中
  3. 每次操作顺便迁移几个桶
  4. 定时任务也会帮忙迁移
  5. 全部迁移完成后，ht[1] 变为 ht[0]

  效果：将一次性的 O(n) 阻塞分散到 O(1) 的多次操作中
  用户几乎感知不到 rehash 的发生！✅
  ```

  **③ ZSET：跳表 + 哈希表的组合**
  ```
  为什么 ZSET 用跳表而非红黑树/B+树？（T15已详述）

  Redis ZSET 的完整结构：
  struct zset {
      dict *dict;         // member → score 的映射（O(1)查找）
      zskiplist *zsl;     // 按 score 排序的跳表（范围查询）
  };

  双结构的协同：
  - ZADD key member score:
    1. dict 检查 member 是否存在（O(1)）
    2. 如果存在，更新 score 并在跳表中调整位置
    3. 如果不存在，同时插入 dict 和跳表

  - ZRANGE key start stop（范围查询）:
    直接在跳表中顺序遍历即可！

  - ZSCORE key member（查询某成员分数）:
    直接查 dict 即可（O(1)）！

  这种"哈希表 + 有序结构"的组合模式在工程中非常常见：
  - MySQL InnoDB：B+树索引 + 自适应哈希索引
  - Elasticsearch：倒排索引 + doc values 列存
  ```

### 2. 搜索引擎背后的数据结构和算法

- **定义**：现代搜索引擎是一个复杂的系统工程，其核心技术包括信息采集、分词、建立索引、排序等多个环节
- **为什么重要**：
  - ⭐ **理解互联网基础设施的窗口**：Google、百度、Elasticsearch 的原理
  - **多种数据结构和算法的综合应用范例**

- **关键机制——搜索引擎的核心组件**：

  **① 倒排索引（Inverted Index）**
  ```
  正排索引（Forward Index）：文档ID → 文档内容
  例：Doc1 → "谷歌地图是一款优秀的产品"
      Doc2 → "谷歌搜索是全球最大的搜索引擎"

  倒排索引（Inverted Index）：关键词 → 包含该词的文档列表
  例："谷歌" → [Doc1, Doc2]
      "地图" → [Doc1]
      "搜索" → [Doc2]
      "产品" → [Doc1]
      "引擎" → [Doc2]


  倒排索引的数据结构：

  Term Dictionary（词典）：
  - 存储所有经过分词后的关键词
  - 需要支持快速查找 → 用 Trie 树 或 B+ 树 / FST（有限状态转换器）

  Posting List（倒排表）：
  - 存储包含该词的所有文档 ID
  - 文档 ID 用差分编码 + 变长整数压缩（Frame-of-Reference）
  - 同时存储词频（TF）、位置信息等

  示例：
  Term: "算法"
  Posting List: [doc_id: 1024, tf: 3, positions: [0, 25, 100]]
              [doc_id: 2049, tf: 1, positions: [50]]
              [doc_id: 3100, tf: 5, positions: [10, 30, 60, 90, 120]]

  查询处理示例：
  用户搜索："数据结构 算法"

  Step 1: 分别查找两个词的倒排表
  "数据结构" → [Doc5, Doc12, Doc23, ...]
  "算法"       → [Doc5, Doc18, Doc23, Doc30, ...]

  Step 2: 求交集（合并两个有序列表）
  结果：[Doc5, Doc23, ...] （同时包含两个词的文档）

  Step 3: 对结果排序（见下文 TF-IDF 和 PageRank）
  ```

  **② TF-IDF 排序算法**
  ```
  目标：衡量一个词对文档的重要程度

  TF（Term Frequency，词频）：
  TF(t, d) = 词 t 在文档 d 中出现的次数 / 文档 d 的总词数

  IDF（Inverse Document Frequency，逆文档频率）：
  IDF(t) = log(总文档数 / 包含词 t 的文档数)

  TF-IDF = TF × IDF

  直觉解释：
  - 一个词在某文档中出现很多次 → TF 高 → 重要 ⬆️
  - 一个词在很多文档中都出现 → IDF 低 → 不重要 ⬇️（如"的"、"是"）
  - 一个词只在少数文档中出现 → IDF 高 → 区分度高 ⬆️

  示例：
  文档总数 N = 10000

  词 "算法" 在 Doc5 中出现 3 次，Doc5 总词数 500
  TF("算法", Doc5) = 3/500 = 0.006

  "算法" 出现在 50 个文档中
  IDF("算法") = log(10000/50) = log(200) ≈ 5.3

  TF-IDF = 0.006 × 5.3 ≈ 0.032

  词 "的" 在 Doc5 中出现 20 次
  TF("的", Doc5) = 20/500 = 0.04

  "的" 出现在 9800 个文档中
  IDF("的") = log(10000/9800) ≈ 0.02

  TF-IDF = 0.04 × 0.02 ≈ 0.0008

  结论："算法"的 TF-IDF 远高于 "的"，所以排名更靠前 ✅
  ```

  **③ PageRank：链接分析算法**
  ```
  核心思想：被更多高质量页面链接的页面更重要

  类比：学术论文的引用次数——被越多好论文引用，
        这篇论文就越权威

  PageRank 值的计算（简化版）：

  PR(A) = (1-d)/n + d × (PR(T1)/C(T1) + PR(T2)/C(T2) + ...)

  其中：
  - PR(A)：页面 A 的 PageRank 值
  - d：阻尼系数（通常 0.85），表示用户继续点击的概率
  - n：网页总数
  - T1, T2,...：指向 A 的页面
  - C(Ti)：页面 Ti 的出链数量

  计算方法：幂迭代法（Power Iteration）
  1. 初始：所有页面的 PR = 1/n
  2. 迭代：根据公式重新计算每个页面的 PR
  3. 收敛：当变化量 < 阈值时停止（通常需要 50-100 次迭代）

  本质：这是一个图上的随机游走（Random Walk）问题
  可以用矩阵运算加速（稀疏矩阵 × 向量）
  ```

### 3. Disruptor：高性能队列的黑科技

- **定义**：LMAX 公司开发的高性能线程间消息队列，能够在单线程中实现每秒处理 600 万订单的性能
- **为什么重要**：
  - ⭐ **高性能并发编程的经典案例**：理解 CPU 缓存、无锁编程的绝佳教材
  - **金融交易系统的核心组件**

- **关键机制——Disruptor 的核心优化**：

  **① 环形数组缓冲区（Ring Buffer）**
  ```
  传统队列的问题：
  - 链表节点动态分配 → GC 压力大
  - 节点不连续 → CPU 缓存未命中率高

  Disruptor 的方案：预分配的环形数组

  图示：
  +---+---+---+---+---+---+---+---+
  | 0 | 1 | 2 | 3 | 4 | 5 | 6 | 7 |
  +---+---+---+---+---+---+---+---+
                ↑           ↑
            sequence   cursor

  特点：
  - 数组大小固定为 2^n（方便位运算取模）
  - 所有节点预分配（无 GC！）
  - 内存连续（缓存友好！）
  - 环形复用（覆盖旧数据）

  序列号（Sequence）机制：
  - 每个 Event 有一个唯一的递增序列号
  - 生产者写入前申请序列号
  - 消费者消费后更新消费进度
  - 通过 CAS 操作保证并发安全
  ```

  **② 解决伪共享（False Sharing）**
  ```
  什么是伪共享？

  CPU 缓存行（Cache Line）通常 64 字节
  当两个变量在同一缓存行中时：
  - Core1 修改变量A → 整个缓存行失效
  - Core2 即使只读变量B 也需要重新加载
  → 性能下降！

  图示（伪共享）：
  Cache Line: [var_A (8B)][padding][var_B (8B)]
                    ↑ Core1 写入导致整行失效
                                          ↑ Core2 受牵连

  Disruptor 的解决方案：填充（Padding）

  @Contended  // Java 8+ 注解，自动添加填充
  class Sequence {
      long volatile value;
      // 自动填充到 128 字节（避免与其他变量共享缓存行）
  }

  手动填充（Java 8 之前）：
  class Sequence {
      long p1, p2, p3, p4, p5, p6, p7;   // 填充！
      long volatile value;
      long p9, p10, p11, p12, p13, p14, p15; // 填充！
  }

  效果：每个 Sequence 独占一个缓存行，互不干扰 ✅
  ```

  **③ 无锁设计（Lock-Free）**
  ```
  传统阻塞队列的问题：
  - ReentrantLock：内核态切换开销大（~1μs）
  - synchronized：轻量级锁升级为重量级锁

  Disruptor 的 CAS 方案：
  - 生产者：CAS 更新 cursor（生产位置）
  - 消费者：CAS 更新 sequence（消费位置）
  - 无需加锁！无需上下文切换！

  多生产者协调：
  - 使用 CAS 在环形数组中抢占槽位
  - 抢到槽位后才写入数据
  - 保证数据不覆盖（依赖消费者进度）

  性能对比（大致）：
  ArrayBlockingQueue: ~500万 ops/s
  Disruptor:          ~2500万 ops/s
  提升：约 5 倍！
  ```

### 4. 微服务接口鉴权与限流

- **定义**：保护微服务免受恶意请求冲击的关键设施，包括身份认证（鉴权）和流量控制（限流）两个方面
- **为什么重要**：
  - ⭐ **微服务架构的基础设施**：没有限流的系统就像没有堤坝的河流
  - **多种算法的实际应用场景对比**

- **关键机制——四种限流算法**：

  **① 固定窗口计数器（Fixed Window Counter）**
  ```
  思想：将时间划分为固定大小的窗口，统计每个窗口内的请求数

  实现：
  int windowStart = currentTime / WINDOW_SIZE * WINDOW_SIZE;
  if (windowStart != currentWindowStart) {
      count = 0;                          // 新窗口，重置计数
      currentWindowStart = windowStart;
  }
  count++;
  if (count > LIMIT) return REJECTED;

  问题：临界突发（Critical Burst）
  图示：
  时间轴：|----窗口1----||----窗口2----|
  请求：  ██████████████░░██████████████
                      ↑
               窗口1的最后时刻来了100个请求（刚好没超限）
               窗口2的开始时刻又来了100个请求
               在这个临界点附近，短时间内涌入了200个请求！
  ```

  **② 滑动窗口日志（Sliding Window Log）**
  ```
  思想：记录每个请求的时间戳，统计当前时间窗口内的请求数

  实现：
  Queue<Long> requestTimes = new LinkedList<>();

  boolean allowRequest() {
      long now = System.currentTimeMillis();
      long windowStart = now - WINDOW_SIZE;

      // 移除超出窗口的旧记录
      while (!requestTimes.isEmpty()
             && requestTimes.peek() < windowStart) {
          requestTimes.poll();
      }

      if (requestTimes.size() >= LIMIT) {
          return false;                     // 超限了
      }

      requestTimes.offer(now);              // 记录本次请求
      return true;
  }

  优点：解决了临界突发问题
  缺点：存储所有请求时间戳，空间 O(n)
  ```

  **③ 滑动窗口计数器（Sliding Window Counter）⭐推荐**
  ```
  思想：结合固定窗口的空间效率和滑动窗口的精度

  实现：维护多个子窗口的计数器

  int[] counters = new int[NUM_SUB_WINDOWS];
  int totalCount = 0;

  boolean allowRequest() {
      long now = System.currentTimeMillis();
      int currentIndex = (int)(now % WINDOW_SIZE / SUB_WINDOW_SIZE);

      // 检查是否进入新的子窗口
      if (currentIndex != lastSubWindowIndex) {
          // 减去最老的子窗口计数
          totalCount -= counters[(currentIndex + 1) % NUM_SUB_WINDOWS];
          counters[(currentIndex + 1) % NUM_SUB_WINDOWS] = 0;
          lastSubWindowIndex = currentIndex;
      }

      totalCount++;
      counters[currentIndex]++;
      return totalCount <= LIMIT;
  }

  精度：介于固定窗口和滑动窗口日志之间
  空间：O(子窗口数)，通常远小于滑动窗口日志
  ```

  **④ 令牌桶（Token Bucket）与漏桶（Leaky Bucket）**
  ```
  令牌桶算法：

  原理：
  - 以固定速率向桶中放入令牌
  - 每个请求消耗一个令牌
  - 桶满时多余的令牌丢弃
  - 桶空时请求被拒绝（或等待）

  参数：
  - rate：令牌生成速率（如 1000 tokens/sec）
  - capacity：桶容量（允许的最大突发量）

  特点：
  ✅ 允许一定程度的突发（桶中积累的令牌）
  ✅ 平均速率受限（由 rate 决定）
  ✅ 平滑限流（适合对外暴露的 API）

  适用场景：API 网关限流、云服务商配额管理


  漏桶算法：

  原理：
  - 请求先进入桶中排队
  - 以固定速率从桶底"漏水"（处理请求）
  - 桶满时新请求被丢弃

  特点：
  ✅ 强制匀速输出（严格限制处理速率）
  ✅ 无法突发（即使桶中有余量也按固定速率处理）

  适用场景：保护下游脆弱服务（如数据库、第三方 API）


  选择指南：
  场景                  推荐算法
  ──────────────────────────────────────
  一般 API 限流         令牌桶（允许突发）
  保护下游服务          漏桶（强制匀速）
  精确控制 QPS          滑动窗口计数器
  简单实现              固定窗口（容忍临界突发的场景）
  ```

### 5. 短网址系统设计

- **定义**：将长 URL 映射为短 URL 的服务，广泛应用于微博、微信、营销链接等场景
- **为什么重要**：
  - ⭐ **系统设计面试的高频题**：涵盖哈希、编码、分布式等多个知识点
  - **"如何生成全局唯一标识"的经典案例**

- **关键机制——两种主流方案**：

  **① 方案A：自增 ID + Base62 编码**
  ```
  生成流程：

  Step 1: 数据库自增生成唯一 ID
  INSERT INTO url_mapping (long_url) VALUES ('https://very-long-url...');
  → 返回 auto_increment_id = 12345678

  Step 2: 将 ID 转换为 Base62 编码
  字符集：0-9 (10) + a-z (26) + A-Z (26) = 62 个字符

  12345678 转 Base62：
  12345678 ÷ 62 = 199121 余 16 → 'g'
  199121 ÷ 62 = 3211 余 39 → 'G'
  3211 ÷ 62 = 51 余 49 → 'x'
  51 ÷ 62 = 0 余 51 → 'z'

  从下往上读：zgxg
  短 URL：http://short.url/zgxg

  优势：
  ✅ 绝对唯一（数据库主键保证）
  ✅ 短 URL 长度可控（7位 Base62 可表示 62^7 ≈ 3.5万亿个）
  ✅ ID 单调递增，容易按时间排序

  劣势：
  ❌ ID 可预测（安全隐患，可遍历所有短链）
  ❌ 数据库成为瓶颈（高并发写入）
  ❌ 分布式扩展困难（需要全局 ID 生成器）


  改进：使用 Snowflake 算法生成分布式唯一 ID
  - 1位符号 + 41位时间戳 + 10位机器ID + 12位序列号
  - 同一毫秒同一机器可生成 4096 个 ID
  - 整体有序（按时间趋势递增）
  ```

  **② 方案B：MurmurHash / MD5 截断**
  ```
  生成流程：

  Step 1: 对长 URL 计算 Hash
  hash = MurmurHash3('https://very-long-url...')
  → 得到一个 128 位的十六进制串
  → 例如：a1b2c3d4e5f6...

  Step 2: 取前 N 位作为短码
  取前 7 个字符：a1b2c3d
  短 URL：http://short.url/a1b2c3d

  处理冲突：
  如果 a1b2c3d 已经存在且对应不同的长 URL：
  方案1：在原 URL 后加随机后缀重新 hash
  方案2：使用线性探测法（a1b2c3e, a1b2c3f, ...）

  优势：
  ✅ 不可预测（哈希的单向性）
  ✅ 无需中心化 ID 生成器
  ✅ 生成速度快（纯计算）

  劣势：
  ❌ 可能存在碰撞（概率极低但理论上存在）
  ❌ 相同长 URL 每次生成的短码不同（除非做去重）


  工程选型建议：
  - 内部工具/低安全要求：自增ID + Base62（简单可靠）
  - 公开服务/高安全要求：Hash 方案（防遍历）
  - 超大规模：两种方案结合（分片 + 各片独立生成）
  ```

# 思考问答

1. **Redis 的渐进式 rehash 过程中，如果在 rehash 进行期间有大量写入操作，会发生什么？rehash 何时完成？**
   - 提示：写入会同时写新旧两个哈希表；定时任务也会加速 rehash；最终全部迁移完毕

2. **如果让你设计一个支持模糊匹配（如 "data*" 匹配 "data structure"）的搜索引擎倒排索引，你会如何改进？**
   - 提示：通配符词典、N-gram 索引、Percolator 倒排

3. **Disruptor 的环形数组大小为什么要设置为 2 的幂？如果不是 2 的幂会有什么后果？**
   - 提示：可以用位运算代替取模运算（index & (size-1)）；否则需要取模运算，稍慢

4. **令牌桶和漏桶算法分别适用于什么场景？能否在一个系统中同时使用两者？如何组合？**
   - 提示：可以两级限流——入口用令牌桶（允许突发均匀化），出口用漏桶（保护下游匀速消费）

5. **短网址系统中，如何防止恶意用户利用短链接进行钓鱼攻击或传播非法内容？有哪些技术手段和管理策略？**
   - 提示：URL 安全检测（黑名单/ML分类）、访问频率限制、举报机制、过期策略、https only
