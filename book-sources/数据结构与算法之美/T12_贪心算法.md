# 核心观点

贪心算法是**"每一步都做出当前最优选择，寄希望于达到全局最优"**的算法策略。从Huffman编码的"频率越高的字符编码越短"的直觉设计，到活动选择问题的"结束时间越早越好"的排序策略，体现了**"局部最优能否推导全局最优"是贪心可行性的核心判据**。理解贪心选择性质、最优子结构、无后效性这三大特征，能帮你区分贪心与动态规划（DP考虑所有子问题）、回溯（穷举+剪枝）的边界。掌握贪心的适用场景（如区间调度、Huffman、最小生成树）和局限（如0-1背包），能在面试和工程中快速判断是否应该使用贪心策略。

## 核心概念解析（20% 重点深度解析）

### 1. 贪心算法的核心思想与特征

- **定义**：在对问题求解时，总是做出在当前看来是最好的选择，不回溯
- **为什么重要**：
  - ⭐ **最直观的算法策略**：代码简洁、效率高、易于理解
  - **区分 DP/回溯/贪心的关键能力**：面试高频考点

- **关键机制——贪心的三大特征 vs 其他算法**：

  **① 三大特征详解**
  ```
  特征1：贪心选择性质（Greedy Choice Property）
  ────────────────────────────────────────
  定义：通过一系列局部最优的选择，可以得出全局最优解

  直观解释：
  - 每一步做选择时，只考虑当前状态下的最优选项
  - 不需要考虑未来的影响（这是与DP的关键区别）
  - 一旦做出选择，就不再改变

  数学表达：
  如果问题的一个全局最优解包含子问题的局部最优解，
  则该问题具有贪心选择性质

  示例验证（活动选择问题）：
  假设活动按结束时间排序：a1(1,4), a2(3,5), a3(0,6), a4(5,7), a5(3,9), a6(5,9), a7(6,10), a8(8,11), a9(8,12), a10(2,14), a11(12,16)

  贪心策略：每次选结束最早且与已选活动不冲突的活动
  第1步：选 a1 (1,4) ← 结束最早
  第2步：跳过 a2,a3（与a1冲突），选 a4 (5,7)
  第3步：跳过 a5,a6，选 a7 (6,10)
  第4步：选 a9 (8,12)？等等，a9开始时间8 < a7结束时间10 → 冲突！
         实际上应该选 a8 (8,11)... 让我重新算

  正确过程：
  选 a1(1,4) → 下一个可选的是 a4(5,7) → a7(6,10)? 不对，a7开始6<7
  实际：a1(1,4) → a4(5,7) → a8(8,11) → a11(12,16)

  结果：{a1, a4, a8, a11} 共4个活动 ← 这就是最优解！✅


  特征2：最优子结构（Optimal Substructure）
  ────────────────────────────────────────
  定义：问题的最优解包含其子问题的最优解

  与DP的最优子结构相同，但区别在于：
  - 贪心：每步只做一个选择（局部最优）
  - DP：需要综合考虑所有可能的选择

  特征3：无后效性（No Aftereffect）
  ────────────────────────────────────────
  定义：一旦某个状态确定，之后的过程不会影响之前的状态

  类比：
  - 贪心像"走一步看一步"，但每步都是确定的
  - 后面的选择不会让前面的选择变得"错误"
  ```

  **② 贪心 vs DP vs 回溯 对比**
  ```
  以「0-1背包问题」为例（物品重量w，价值v，背包容量W）

  问题：4个物品 {w:[2,3,4,5], v:[3,4,5,6]}, W=5

  【贪心策略】（尝试多种贪心规则）

  规则A：价值优先（先拿价值最高的）
  选物品4(v=6,w=5) → 背包满
  总价值 = 6

  规则B：性价比优先（v/w 最大优先）
  性价比：[1.5, 1.33, 1.25, 1.2]
  选物品1(v=3,w=2) → 还剩容量3
  选物品2(v=4,w=3) → 背包满
  总价值 = 3 + 4 = 7 ✅ 这个恰好是最优！

  但如果数据换成：{w:[2,3,5], v:[2,4,6]}, W=5
  性价比：[1, 1.33, 1.2]
  选物品2(v=4,w=3) → 还剩容量2
  选物品1(v=2,w=2) → 背包满
  总价值 = 6
  最优解：选物品3(v=6,w=5) → 总价值 = 6（相同）
  
  再换：{w:[1,2,4], v:[10,20,30]}, W=4
  性价比：[10, 10, 7.5]
  选物品1或2都可以...
  贪心选物品1(v=10,w=1)→剩3→选物品2(v=20,w=2)→总价值=30
  最优解：选物品3(v=30,w=4)→总价值=30（相同）
  
  反例：{w:[3,3,5], v:[5,5,8]}, W=6
  性价比：[1.67, 1.67, 1.6]
  贪心选物品1或2→再选另一个→总价值=10
  最优解也是10... 
  
  经典反例：{w:[1,3,4,5], v:[3,4,5,6]}, W=7
  性价比：[3, 1.33, 1.25, 1.2]
  贪心：选1(w=1,v=3)→剩6→选2(w=3,v=4)→剩3→无法再选
  总价值=7
  最优：选2(w=3,v=4)+选3(w=4,v=5)=9 或 选1+选4(w=5,v=6)=9
  
  结论：贪心在0-1背包上不一定能得到最优解！❌


  【DP解法】（保证最优）
  状态定义：dp[i][j] = 前 i 个物品，容量 j 时的最大价值
  状态转移：dp[i][j] = max(dp[i-1][j], dp[i-1][j-wi]+vi)
  结果：通过填表得到精确最优解

  【回溯解法】（穷举所有可能）
  每个物品选或不选，共 2^n 种组合
  加剪枝优化（超重时提前返回）
  结果：一定找到最优解，但效率较低

  总结对比：

  维度        贪心          动态规划        回溯
  ──────────────────────────────────────────────
  选择策略    局部最优      全局最优(穷举子问题)  穷举+剪枝
  时间复杂度  通常 O(nlogn)  O(n×W)或O(n²)   O(2^n)
  是否最优    不一定        ✅ 一定          ✅ 一定
  适用条件    需要证明      有重叠子问题      解空间可枚举
  典型应用    活动选择      背包/编辑距离     N皇后/排列
  ```

### 2. Huffman 编码：贪心构造最优前缀码

- **定义**：一种基于字符出现频率的可变长度编码方案，频率高的字符用短编码，频率低的用长编码
- **为什么重要**：
  - ⭐ **贪心算法的经典应用**：同时展示了贪心正确性证明
  - **实际应用广泛**：ZIP压缩、JPEG、MP3等都用到变长编码思想

- **关键机制——Huffman 编码的完整构建过程**：

  **① 问题背景**
  ```
  定长编码的问题：
  ASCII码：每个字符8位（256个字符）
  "aabacabad" = 9个字符 × 8位 = 72位

  观察：'a' 出现5次，'b' 出现2次，'c' 出现1次，'d' 出现1次
  如果让 'a' 用更短的编码，就能节省空间！

  变长编码的要求——前缀码性质：
  任何一个字符的编码不能是另一个字符编码的前缀
  否则解码时会歧义！
  ```

  **② 构建步骤（贪心过程）**
  ```
  示例：字符集 {a:5, b:2, c:1, d:1}

  Step 1: 统计频率，构建节点
  a(5)  b(2)  c(1)  d(1)

  Step 2: 将所有节点放入优先队列（最小堆）

  Step 3: 循环合并最小的两个节点

  第1次合并：取出 c(1) 和 d(1)，合并为新节点 cd(2)
  队列中：a(5), b(2), cd(2)

  第2次合并：取出 b(2) 和 cd(2)，合并为 bcd(4)
  队列中：a(5), bcd(4)

  第3次合并：取出 bcd(4) 和 a(5)，合并为 root(9)
  队列中：root(9) → 完成！

  构建的Huffman树：

          root(9)
         /       \
       a(5)    bcd(4)
              /     \
            b(2)   cd(2)
                  /    \
                c(1)  d(1)

  Step 4: 生成编码（左0右1）
  a: 0           (1位)
  b: 10          (2位)
  c: 110         (3位)
  d: 111         (3位)

  编码结果："aabacabad"
  = a-a-b-a-c-a-b-a-d
  = 0-0-10-0-110-0-10-0-111
  = 001001100100111 （共18位）

  对比：
  原始定长编码：72位
  Huffman编码：18位
  压缩率：75%！✅
  ```

  **③ 为什么 Huffman 编码是最优前缀码？**
  ```
  证明思路（贪心正确性证明）：

  引理1：两个频率最低的字符，其编码长度必然最长
  - 否则可以将它们与短编码交换，得到更好的编码

  引理2：两个频率最低的字符，可以作为一个整体（合并节点）处理
  - 合并后的新问题与原问题等价

  引理3（交换论证法 Exchange Argument）：
  - 假设存在比 Huffman 更优的前缀码 T*
  - 通过交换操作，可以将 T* 逐步变为 Huffman 树
  - 每次交换不会增加编码总长度
  - 因此 Huffman 的总长度 ≤ T* 的总长度
  - 即 Huffman 是最优的 ✅

  关键洞察：
  Huffman 的贪心策略——每次合并频率最小的两个节点——
  本质上是在构建一棵"加权路径长度最小"的二叉树。
  这与最优二叉树（Optimal Binary Tree）问题是等价的。

  WPL（加权路径长度）= Σ(频率_i × 编码长度_i)
  上例：5×1 + 2×2 + 1×3 + 1×3 = 5 + 4 + 3 + 3 = 15（比特）
  注意：实际编码长度 = WPL + 字符数（每个字符需要一个分隔符，
        或者使用前缀码不需要分隔符，所以直接等于WPL）
  ```

### 3. 经典贪心问题详解

- **定义**：一类可以用贪心策略高效解决的经典问题集合
- **为什么重要**：
  - ⭐ **面试高频题型**：区间调度、跳跃游戏、分糖果等反复出现
  - **训练贪心直觉的最佳素材**

- **关键机制——四大经典问题**：

  **① 活动选择问题（Interval Scheduling / 区间调度）**
  ```
  问题描述：
  有 n 个活动，每个活动有开始时间 s_i 和结束时间 f_i
  选择尽可能多的互不冲突的活动

  输入示例：
  活动:  a1(1,4)  a2(3,5)  a3(0,6)  a4(5,7)  a5(3,9)
        a6(5,9)  a7(6,10) a8(8,11) a9(8,12) a10(2,14) a11(12,16)

  贪心策略：按结束时间排序，每次选结束最早且兼容的活动

  为什么按结束时间排序是最优的？
  反证法：
  假设最优解的第一个活动不是结束最早的
  设最优解第一个活动为 a_k（不是最早结束的 a_j）
  由于 a_j 比 a_k 早结束
  将 a_k 替换为 a_j 后，剩下的可用时间更多
  所以替换后的解不会比原解差 ✅

  代码实现：
  int activitySelection(int[][] activities) {
      // 按结束时间排序
      Arrays.sort(activities, (a, b) -> a[1] - b[1]);

      int count = 1;                        // 至少选一个
      int lastEnd = activities[0][1];        // 上一个活动的结束时间

      for (int i = 1; i < activities.length; i++) {
          if (activities[i][0] >= lastEnd) { // 不冲突
              count++;
              lastEnd = activities[i][1];
          }
      }
      return count;
  }

  时间复杂度：O(n log n)（排序主导）
  空间复杂度：O(1)

  变种问题：
  - 区间覆盖问题：用最少区间覆盖整个范围
  - 加权区间调度：每个活动有权值，求最大权值（这个不能用简单贪心！需用DP）
  ```

  **② 分糖果问题（Assign Cookies / 贪心匹配）**
  ```
  问题描述：
  每个孩子有一个胃口值 g[i]，每块饼干有一个尺寸 s[j]
  每块饼干只能分配给一个孩子
  目标：满足尽可能多的孩子（饼干尺寸 ≥ 孩子胃口才算满足）

  贪心策略：
  1. 将孩子按胃口值从小到大排序
  2. 将饼干按尺寸从小到大排序
  3. 用最小的饼干去满足能满足的最小胃口的孩子

  为什么这样是最优的？
  - 大饼干给大胃口的孩子不会"浪费"（小饼干也满足不了大胃口）
  - 小饼干给小胃口的孩子是"刚好够用"
  - 这种配对方式最大化了利用率

  代码实现：
  int findContentChildren(int[] g, int[] s) {
      Arrays.sort(g);
      Arrays.sort(s);

      int child = 0, cookie = 0;
      while (child < g.length && cookie < s.length) {
          if (s[cookie] >= g[child]) {
              child++;                       // 满足了这个孩子
          }
          cookie++;                          // 这块饼干用了（不管有没有满足）
      }
      return child;
  }

  时间复杂度：O(n log n + m log m)
  空间复杂度：O(1)
  ```

  **③ 跳跃游戏（Jump Game）**
  ```
  问题描述 I（LeetCode 55 - 判断能否到达终点）：
  给定非负整数数组 nums，初始位置在第一个元素
  每个元素代表你在该位置能跳跃的最大长度
  判断是否能到达最后一个位置

  示例：nums = [2, 3, 1, 1, 4]
  位置0可以跳1或2步 → 到位置1或2
  从位置1可以跳最多3步 → 可以到位置4（终点）✅

  贪心策略：维护当前能到达的最远位置

  boolean canJump(int[] nums) {
      int maxReach = 0;                      // 当前能到达的最远位置
      for (int i = 0; i < nums.length; i++) {
          if (i > maxReach) return false;    // 到不了这里
          maxReach = Math.max(maxReach, i + nums[i]);
      }
      return true;
  }

  为什么贪心是对的？
  - 在位置 i 时，如果能到达 i（即 i ≤ maxReach）
  - 那么从 i 可以到达的范围扩展到 i + nums[i]
  - 取所有可达位置的并集，就是最终能到达的范围
  - 如果最终 maxReach ≥ 最后一个下标，说明能到达

  问题描述 II（LeetCode 45 - 最少跳跃次数）：
  求到达终点的最少跳跃次数

  int jump(int[] nums) {
      int steps = 0;                         // 跳跃次数
      int currentEnd = 0;                    // 当前这一跳能到的边界
      int farthest = 0;                      // 下一跳能到的最远位置

      for (int i = 0; i < nums.length - 1; i++) {
          farthest = Math.max(farthest, i + nums[i]);
          if (i == currentEnd) {             // 到达了当前跳跃边界
              steps++;                       // 必须跳一次
              currentEnd = farthest;          // 更新边界
          }
      }
      return steps;
  }

  时间复杂度：O(n)
  空间复杂度：O(1)
  ```

  **④ 钱币找零问题（Coin Change - 贪心适用版）**
  ```
  问题描述：
  使用最少数量的硬币凑出目标金额
  硬币面额：[1, 5, 10, 25, 100]

  贪心策略：每次使用不超过剩余金额的最大面额硬币

  为什么这种情况下贪心是对的？
  条件：硬币面额是"规范"的（Canonical Coin System）
  - 每种较大面额都能被较小面额"整除式表示"
  - 如美国货币体系：25 = 10+10+5, 10 = 5+5 等

  反例（贪心不适用的币值系统）：
  硬币面额：[1, 3, 4]，目标金额：6
  贪心：选4 → 剩余2 → 选1+1 → 共3枚
  最优：选3+3 → 共2枚 ❌ 贪心失败！

  代码实现（适用于规范货币体系）：
  int coinChange(int[] coins, int amount) {
      Arrays.sort(coins);                    // 排序
      int count = 0;

      for (int i = coins.length - 1; i >= 0; i--) {
          count += amount / coins[i];        // 使用该面额的数量
          amount %= coins[i];                // 剩余金额
          if (amount == 0) break;
      }
      return count;
  }

  如何判断贪心是否适用？
  - 对于一般情况，需要数学证明或反例验证
  - 无法简单判断时，应使用 DP（完全背包问题）
  ```

## 次要信息速览（80% 快速带过）

- 【贪心算法的适用场景总结】：
  ```
  ✅ 适合贪心的问题类型：
  1. 区间调度类：活动选择、会议室安排
  2. 构造类：Huffman编码、最小生成树（Prim/Kruskal）
  3. 匹配类：分糖果、任务分配
  4. 单调决策类：跳跃游戏、股票买卖（一次交易）

  ❌ 不适合贪心的问题类型：
  1. 0-1 背包问题（需用 DP）
  2. 加权区间调度（需用 DP）
  3. 一般的钱币找零（需用 DP，除非币值系统特殊）
  4. 旅行商问题TSP（NP-hard，需用近似算法）
  ```

- 【贪心正确性证明方法】：
  ```
  方法1：交换论证法（Exchange Argument）
  - 假设存在最优解不同于贪心解
  - 通过一系列不使结果变差的交换操作
  - 将最优解转化为贪心解
  - 说明贪心解也是最优的

  方法2：归纳法（Induction / 数学归纳法）
  - 证明基础情况成立
  - 假设对规模为 n-1 的问题贪心最优
  - 证明对规模为 n 的问题贪心也最优

  方法3：反证法（Contradiction）
  - 假设贪心解不是最优的
  - 导出矛盾
  ```

  | Dijkstra | 优先队列选最近点 | O((V+E)logV) |

# 思考问答

1. **对于活动选择问题，如果要求选择的"活动总时长最长"而非"活动数量最多"，贪心策略还适用吗？如果不适用，应该如何解决？**
   - 提示：按结束时间的贪心不一定得到最长总时长，需要用DP

2. **请举例说明一个看起来可以用贪心但实际上不行的问题，并分析为什么贪心会失败。**
   - 提示：0-1背包、某些钱币找零、加权区间调度

3. **Huffman编码中，如果有n个不同的字符，构建出的Huffman树有多少个节点？编码的最大长度是多少？**
   - 提示：内部节点数=n-1，总节点数=2n-1；最大长度取决于频率分布

4. **Dijkstra算法是贪心算法吗？它满足贪心选择性质吗？请证明或反驳。**
   - 提示：Dijkstra确实是一种贪心策略，每次选择距离最近的未访问节点

5. **如何判断一个问题是否适合用贪心算法？有没有通用的判断流程？**
   - 提示：尝试贪心 → 找反例 → 若找不到反例 → 尝试证明（交换论证/归纳法）

# 实践要点

- ✅ **掌握经典贪心问题的标准解法**：活动选择、分糖果、跳跃游戏、Huffman编码
- ✅ **学会证明贪心正确性**：至少掌握交换论证法和反证法的基本思路
- ✅ **能够区分贪心和DP的适用边界**：知道什么时候贪心够用，什么时候必须用DP
- ✅ **理解贪心的局限性**：不要在所有优化问题上都试图用贪心
- ❌ **未经证明就直接使用贪心**：可能导致错误答案（尤其在竞赛和面试中）
- ❌ **混淆"贪心适用"和"贪心能得到正确答案"**：有些问题贪心只能得到近似解
- ❌ **忽视排序在贪心中的作用**：很多贪心问题的关键在于"按什么标准排序"
