# 核心观点

堆是**"基于完全二叉树的高效优先级队列实现"**，核心价值在于用O(log n)的时间复杂度支持动态获取最大/最小值。从堆排序的"理论上优秀但实际不如快排"的启示，到TopK问题的"小顶堆维护K个最大元素"的经典解法，再到利用双堆求中位数的精妙设计，体现了**"数据结构选择要结合实际场景和硬件特性"**的工程智慧。理解堆的数组存储、上下堆化操作、建堆的O(n)时间复杂度证明，能帮你在优先级调度、海量数据统计、实时流处理等场景中设计出高效的解决方案。

## 核心概念解析（20% 重点深度解析）

### 1. 为什么堆排序没有快速排序快？

- **定义**：虽然堆排序和快速排序的时间复杂度都是O(n log n)，但实际运行时快速排序通常更快
- **为什么重要**：
  - ⭐ **打破"时间复杂度相同则性能相同"的误区**
  - 理解CPU缓存、内存访问模式对性能的影响
  - 学会从理论和实践两个角度评估算法

- **关键机制——两大性能差距原因**：

  **原因①：数据访问方式不友好（缓存未命中率高）**
  ```
  快速排序的数据访问模式：
  - 顺序访问（局部性好）
  - 分区时遍历数组的一段连续区间
  - CPU预取机制能有效工作

  示例（快速排序分区）：
  数组：[5, 3, 8, 1, 2, 9, 4]
  pivot = 5
  遍历顺序：0→1→2→3→4→5→6 （连续访问！）
  CPU缓存行可以一次加载多个元素，命中率极高

  堆排序的数据访问模式：
  - 跳跃式访问（局部性差）
  - 堆化时需要比较父节点和子节点，它们在数组中不相邻

  示例（大顶堆化，下标从0开始）：
  堆顶节点在位置0
  左子节点在位置1 (2*0+1)
  右子节点在位置2 (2*0+2)
  左子节点的左子节点在位置3 (2*1+1)
  访问路径：0 → 1 → 3 → 7 ... （跳跃式！）

  实际影响：
  对于100万个元素的数组：
  - 快速排序：大部分数据访问命中L1/L2缓存
  - 堆排序：频繁触发缓存未命中（Cache Miss），每次都要从主存读取

  性能差距量化（大致）：
  - 缓存命中：~1个CPU周期
  - 缓存未命中：~100-200个CPU周期（取决于内存延迟）
  - 堆排序可能比快排慢2-3倍（即使理论复杂度相同）
  ```

  **原因②：数据交换次数更多**
  ```
  排序的本质：消除逆序对（Inversion）

  有序度的定义：
  - 有序对：如果 i < j 且 a[i] <= a[j]，称为有序对
  - 逆序对：如果 i < j 且 a[i] > a[j]，称为逆序对
  - 完全有序数组：有序度 = n(n-1)/2，逆序度 = 0
  - 完全逆序数组：有序度 = 0，逆序度 = n(n-1)/2

  快速排序的交换策略：
  - 每次交换都至少消除一个逆序对
  - 总交换次数 ≤ 初始逆序度
  - 对于基本有序的数组，交换次数很少

  堆排序的问题：
  第一步建堆会破坏原有的有序性！

  示例（已有序数组 [1, 2, 3, 4, 5]）：
  建堆前：[1, 2, 3, 4, 5]（完全有序，逆序度=0）
  建堆后（大顶堆）：[5, 4, 3, 1, 2]（变得无序了！）

  后果：
  - 即使输入数据已经有序，堆排序也要进行大量无谓的交换
  - 而快速排序对于有序数据的分区几乎不需要交换

  实验验证（可自行测试）：
  int swapCount = 0;
  // 在每次swap()时计数
  // 结果：堆排序的swapCount通常比快排多20%-50%
  ```

  **总结对比表**

  | 维度 | 快速排序 | 堆排序 |
  |------|----------|--------|
  **平均时间复杂度** | O(n log n) | O(n log n) |
  **最坏时间复杂度** | O(n²)（可通过优化避免） | O(n log n) ⭐更稳定 |
  **空间复杂度** | O(log n)（递归栈） | O(1) ⭐原地排序 |
  **稳定性** | 不稳定 | 不稳定 |
  **缓存友好性** | ⭐⭐⭐⭐⭐ 极好 | ⭐⭐ 较差 |
  **已有序数据表现** | ⭐⭐⭐⭐⭐ 很快（优化后） | ⭐ 较慢（仍需建堆） |
  **实际运行速度** | ⭐⭐⭐⭐⭐ 通常更快 | ⭐⭐⭐ 较慢 |

  **工程选择建议**：
  ```
  选择快速排序的场景：
  ✅ 通用排序需求（如Arrays.sort()、C++ std::sort）
  ✅ 数据量适中（百万级别以内）
  ✅ 对平均性能要求高
  ✅ 可以接受最坏情况下的退化（或使用随机化/三数取中优化）

  选择堆排序的场景：
  ✅ 需要保证最坏情况性能（如实时系统）
  ✅ 内存极度受限（原地排序，O(1)额外空间）
  ✅ 只需要部分排序（如TopK问题）
  ✅ 数据流式到达，需要动态维护极值

  实际应用：
  - Java Arrays.sort()：基本类型用双轴快排，对象类型用TimSort（归并+插入混合）
  - C++ std::sort()：内省排序（Introsort）= 快排 + 堆排序（防止退化）
  - Python sorted()：Timsort（自适应归并排序）
  ```

### 2. 堆的核心操作与实现

- **定义**：堆是一种特殊的完全二叉树，满足每个节点的值都≥（大顶堆）或≤（小顶堆）其子节点的值
- **为什么重要**：
  - ⭐ **优先级队列的标准实现**：Java PriorityQueue、Python heapq
  - **TopK问题的最优解之一**：时间O(n log K)，空间O(K)
  - **理解完全二叉树的数组表示**：为后续学习B+树等打下基础

- **关键机制——堆的存储与基础操作**：

  **① 数组存储结构（从下标0开始）**
  ```
  大顶堆示例（10个元素）：
  数组：[90, 80, 70, 60, 50, 40, 30, 20, 10, 5]

  可视化（完全二叉树形式）：
              90(0)            ← 堆顶（最大值）
            /    \
         80(1)   70(2)        ← 第二层
        / \      / \
     60(3)50(4)40(5)30(6)     ← 第三层
     /  \
  20(7)10(8)                    ← 第四层
  （5(9)是最后一个叶子节点）

  寻址公式（i从0开始）：
  parent(i) = (i - 1) / 2
  leftChild(i) = 2 * i + 1
  rightChild(i) = 2 * i + 2

  验证：
  - 节点80(1)的父节点 = (1-1)/2 = 0 → 90 ✅
  - 节点60(3)的父节点 = (3-1)/2 = 1 → 80 ✅
  - 节点80(1)的左子节点 = 2*1+1 = 3 → 60 ✅
  - 节点80(1)的右子节点 = 2*1+2 = 4 → 50 ✅
  ```

  **② 从上往下堆化（Heapify Down）—— 删除堆顶时使用**
  ```
  目的：将不符合堆性质的节点下沉到正确位置

  适用场景：
  - 删除堆顶元素后，用末尾元素替换堆顶，然后下沉
  - 建堆过程中，对非叶子节点依次下沉

  算法步骤（以大顶堆为例）：
  heapify(array, n, i):  // 对第i个节点进行下沉
      while (true):
          maxPos = i                          // 假设当前位置最大
          left = 2 * i + 1                    // 左子节点
          right = 2 * i + 2                   // 右子节点

          if (left < n && array[left] > array[maxPos]):
              maxPos = left                    // 左子节点更大
          if (right < n && array[right] > array[maxPos]):
              maxPos = right                   // 右子节点更大

          if (maxPos == i):
              break                           // 已经满足堆性质，停止
          else:
              swap(array, i, maxPos)           // 与更大的子节点交换
              i = maxPos                      // 继续向下检查

  时间复杂度：O(log n)（树的高度）
  空间复杂度：O(1)

  图解示例（小顶堆化，删除最小值后）：
  初始堆（小顶堆）：
        10
       /  \
     20    30
    / \   /
   40 50 60

  Step 1: 删除堆顶10，用末尾元素60替换
        60
       /  \
     20    30
    / \   /
   40 50

  Step 2: 60 > 子节点(20,30)，与最小的20交换
        20
       /  \
     60    30
    / \   /
   40 50

  Step 3: 60 > 子节点(40,50)，与最小的40交换
        20
       /  \
     40    30
    / \   /
   60 50

  完成！新堆满足小顶堆性质 ✅
  ```

  **③ 从下往上堆化（Heapify Up）—— 插入元素时使用**
  ```
  目的：将新插入的元素上浮到正确位置

  适用场景：
  - 向堆中插入新元素时，放在末尾然后上浮

  算法步骤（以大顶堆为例）：
  insert(data):
      if (堆满) return
      array[count] = data               // 放到末尾
      i = count
      count++

      while (i > 0 && array[i] > array[(i-1)/2]):  // 比较父节点
          swap(array, i, (i-1)/2)        // 比父节点大，交换
          i = (i-1)/2                    // 继续向上检查

  时间复杂度：O(log n)
  空间复杂度：O(1)

  图解示例（大顶堆，插入85）：
  初始堆（大顶堆）：
        90
       /  \
     80    70
    / \   /
   60 50 40

  Step 1: 插入85到末尾
        90
       /  \
     80    70
    / \   /
   60 50 40
  /
 85

  Step 2: 85 > 父节点40，上浮
        90
       /  \
     80    70
    / \   /
   60 50 85
  /

  Step 3: 85 > 父节点70，继续上浮

 40

  Step 4: 85 < 父节点80，停止

  **④ 建堆操作（Build Heap）—— O(n)时间复杂度的证明** ⭐重点
  ```
  方法：从最后一个非叶子节点开始，从右往左、从下往上依次下沉

  为什么从 n/2 开始？
  - 对于完全二叉树，下标 ≥ n/2 的节点都是叶子节点
  - 叶子节点没有子节点，无需堆化
  - 最后一个非叶子节点就是 n/2 - 1（下标从0开始）

  代码实现：
  buildHeap(array, n):
      for (i = n/2 - 1; i >= 0; i--):  // 从最后一个非叶子节点开始
          heapify(array, n, i)           // 依次下沉

  时间复杂度分析（为什么是O(n)而不是O(n log n)？）：

  思路：计算每个节点堆化的代价之和

  假设树的高度为h（根节点高度为0，叶子节点高度为h）
  - 高度为0的节点（根节点）：1个，最多下沉h次
  - 高度为1的节点：2个，最多下沉h-1次
  - 高度为2的节点：4个，最多下沉h-2次
  - ...
  - 高度为h的节点（叶子节点）：2^h个，不下沉

  总代价 S = Σ (每层节点数 × 该层最大下沉次数)
          = 1×h + 2×(h-1) + 4×(h-2) + ... + 2^h×0

  使用错位相减法：
  S = h + 2(h-1) + 4(h-2) + ... + 2^(h-1)×1
  2S = 2h + 4(h-1) + 8(h-2) + ... + 2^h×1

  2S - S = h + 2 + 4 + 8 + ... + 2^(h-1)
  S = h + 2^h - 2

  因为 2^h ≈ n（节点总数），h = log₂n
  所以 S = O(n) + O(log n) = O(n)  ✅

  直观理解：
  - 大多数节点都在底层（叶子层或接近叶子的层）
  - 底层的节点高度低，下沉次数少（0次或1-2次）
  - 只有少数顶层节点需要多次下沉
  - 所以总代价是线性的O(n)，而非O(n log n)

  对比：逐个插入建堆
  - 每次插入调用insert()，时间O(log n)
  - 插入n个元素，总时间O(n log n)
  - 所以buildHeap()比逐个插入更高效！
  ```

### 3. TopK问题的堆解法

- **定义**：从海量数据中快速找出最大（或最小）的K个元素
- **为什么重要**：
  - ⭐ **大数据面试必考题**：搜索引擎热词统计、商品销量排行、用户活跃度排名
  - **体现"空间换时间"的思想**：用O(K)的空间处理n≫K的数据
  - **工程实用技能**：日志分析、推荐系统、监控系统

- **关键机制——两种堆的选择策略**：

  **问题分类**
  ```
  类型A：求最大的前K个元素（TopK Maximum）
  例：找出点击量最高的10篇新闻

  类型B：求最小的前K个元素（TopK Minimum）
  例：找出响应时间最快的10个接口

  类型C：动态数据流中的TopK
  例：实时统计当前最热门的10个搜索词
  ```

  **经典解法：堆 + 散列表（静态数据）**
  ```
  问题：从10亿个搜索关键词中找出Top 10最热门的

  Step 1: 统计词频（使用散列表）
  HashMap<String, Integer> wordCount = new HashMap<>();
  for (String keyword : keywords) {
      wordCount.put(keyword, wordCount.getOrDefault(keyword, 0) + 1);
  }
  // 时间：O(n)，空间：O(m)，m=不重复的关键词数

  Step 2: 用小顶堆维护Top 10
  PriorityQueue<Map.Entry<String, Integer>> minHeap =
      new PriorityQueue<>(11, Comparator.comparingInt(Map.Entry::getValue));

  for (Map.Entry<String, Integer> entry : wordCount.entrySet()) {
      if (minHeap.size() < 10) {
          minHeap.offer(entry);           // 堆没满，直接加入
      } else if (entry.getValue() > minHeap.peek().getValue()) {
          minHeap.poll();                 // 比堆顶大，淘汰堆顶
          minHeap.offer(entry);           // 加入新元素
      }
      // 否则：比堆顶小，直接丢弃
  }

  // 最终堆中的10个元素就是Top 10
  // 时间：O(m log K) = O(m log 10) ≈ O(m)
  // 空间：O(K) = O(10)  ← 非常节省！

  关键技巧：
  - 求最大的K个 → 用小顶堆（堆顶是当前第K大的，比它小的直接淘汰）
  - 求最小的K个 → 用大顶堆（堆顶是当前第K小的，比它大的直接淘汰）
  ```

  **优化：分片处理（内存不足时）**
  ```
  问题：10亿关键词，内存只有1GB，无法一次性加载

  解决方案：Hash分片 + 各片分别求TopK + 合并

  Step 1: Hash分片
  创建10个文件：file_0, file_1, ..., file_9
  for (String keyword : allKeywords) {
      int fileIndex = hash(keyword) % 10;
      writeToFile(fileIndex, keyword);
  }
  // 相同的关键词一定会被分到同一个文件

  Step 2: 各片独立求Top 10
  List<List<Result>> topKLists = new ArrayList<>();
  for (int i = 0; i < 10; i++) {
      List<Result> top10 = findTopKFromFile(file_i, 10);
      topKLists.add(top10);
  }

  Step 3: 合并10个Top 10，得到最终的Top 10
  Result finalTop10 = mergeTopK(topKLists, 10);

  总时间：O(n) + 10 × O(m/10 log 10) + O(100 log 10) ≈ O(n)
  总空间：O(m/10)（每次只处理一个分片）✅
  ```

  **动态数据流的TopK维护**
  ```
  场景：数据持续流入，随时查询当前的TopK

  数据结构：堆（大小固定为K）+ 散列表（用于更新已有元素的频率）

  操作流程：

  addElement(newElement):
      if (hashMap contains newElement):
          update frequency in hashMap
          update position in heap (tricky! 需要特殊处理)
      else:
          if (heap.size() < K):
              heap.offer(newElement)
              hashMap.put(newElement, freq)
          else if (newElement.freq > heap.peek().freq):
              heap.poll()
              heap.offer(newElement)
              hashMap.put(newElement, freq)

  getTopK():
      return heap.toArray()

  注意事项：
  - 如果元素频率会变化（不是单调递增），需要能够更新堆中的元素
  - Java PriorityQueue不支持高效的update操作（需要remove+add，O(log n)）
  - 更高级的实现可以用索引堆（Indexed Heap）或斐波那契堆
  ```

  **与其他方法的对比**
  ```
  方法                时间复杂度    空间复杂度    适用场景
  ──────────────────────────────────────────────────────
  全排序             O(n log n)   O(n)         n较小（<百万）
  快速选择(QuickSelect) O(n)       O(1)        只需一次查询
  堆（本方法）        O(n log K)   O(K)        ⭐n很大，K很小，多次查询
  计数排序/Bucket     O(n + range) O(range)    数据范围有限
  MapReduce          O(n/m)       分布式       超大规模（TB级）

  为什么堆方法最适合TopK？
  1. 空间效率极高：只需要O(K)空间，K通常很小（10/100/1000）
  2. 支持动态更新：数据流式到达时可以增量维护
  3. 实现简单：语言标准库都有现成的堆/优先级队列实现
  4. 性能足够好：log K ≈ log 1000 ≈ 10，非常快
  ```

### 4. 优先级队列的实际应用

- **定义**：一种特殊的队列，出队顺序由元素的优先级决定，而非入队顺序
- **为什么重要**：
  - ⭐ **任务调度系统的核心**：操作系统进程调度、线程池任务队列
  - **图算法的基础**：Dijkstra最短路径、Prim最小生成树
  - ** Huffman编码的关键**：构建最优前缀码

- **关键机制——两大经典应用场景**：

  **应用①：高性能定时器**
  ```
  传统方案的问题：
  while (true):
      sleep(1秒)                    // 每1秒轮询一次
      for (task in taskList):
          if (task.isDue()):        // 遍历所有任务
              task.execute()

  问题：
  ❌ 浪费CPU：大部分时间无事可做
  ❌ 响应不及时：任务可能在两次轮询之间到期
  ❌ 扩展性差：任务越多，遍历越慢

  基于堆的优化方案：
  PriorityQueue<Task> timerQueue = new PriorityQueue<>(
      Comparator.comparingLong(Task::getExecuteTime)
  );

  void addTask(Task task) {
      timerQueue.offer(task);
  }

  void startTimer() {
      while (true) {
          if (timerQueue.isEmpty()) {
              sleep(∞);                         // 无任务，永久等待
          } else {
              Task nextTask = timerQueue.peek();
              long waitTime = nextTask.getExecuteTime() - currentTimeMillis();
              if (waitTime > 0) {
                  sleep(waitTime);               // 精确等待到下一个任务
              }
              Task task = timerQueue.poll();     // 取出执行
              task.execute();

              if (task.isRepeated()) {           // 定时重复任务
                  task.setNextExecuteTime();     // 设置下次执行时间
                  timerQueue.offer(task);        // 重新加入队列
              }
          }
      }
  }

  优势：
  ✅ 精确定时：精确到毫秒级（受限于系统调度精度）
  ✅ 高效：每次只需查看堆顶，O(1)查找 + O(log n)插入/删除
  ✅ 支持动态添加：新任务可以在任何时候加入
  ✅ 低功耗：没有任务时可以休眠，不浪费CPU

  实际应用：
  - Java Timer/ScheduledExecutorService
  - Linux crontab（简化版）
  - 游戏引擎的事件系统
  - 网络框架的超时管理（Netty HashedWheelTimer）
  ```

  **应用②：合并K个有序文件/数组**
  ```
  问题：有100个有序的小文件，合并成一个有序的大文件

  暴力方案：
  1. 将100个文件全部读入内存
  2. 用归并排序的merge函数两两合并
  问题：内存占用大，I/O次数多

  基于堆的优化方案（多路归并）：
  PriorityQueue<FileReader> minHeap = new PriorityQueue<>(
      101,
      Comparator.comparingInt(FileReader::getCurrentValue)
  );

  // 初始化：从每个文件读取第一个元素
  for (FileReader reader : readers) {
      if (reader.hasNext()) {
          reader.readNext();
          minHeap.offer(reader);
      }
  }

  // 合并过程
  Writer outputWriter = new Writer("merged_output.txt");
  while (!minHeap.isEmpty()) {
      FileReader minReader = minHeap.poll();       // 取出最小值
      outputWriter.write(minReader.getCurrentValue());  // 写入输出

      if (minReader.hasNext()) {
          minReader.readNext();                     // 从该文件读取下一个
          minHeap.offer(minReader);                  // 重新加入堆
      }
  }

  时间复杂度：
  - 初始化：O(K)（K=文件数）
  - 合并：O(N log K)（N=总元素数，每次堆操作O(log K)）
  - 总计：O(N log K)

  对比：
  - 两个文件归并：O(N)（普通二路归并）
  - K个文件归并：O(N log K)（多路归并+堆）
  - 当K=100时，log K ≈ 7，开销很小！

  实际应用：
  - 数据库的外部排序（External Sort）
  - MapReduce的Reduce阶段（Shuffle后的数据合并）
  - 搜索引擎的倒排索引合并
  - LeetCode 23: Merge k Sorted Lists（经典面试题）

  **应用③：Dijkstra最短路径算法**
  ```
  核心思想：贪心 + 优先级队列

  伪代码：
  dijkstra(graph, source):
      dist[] = {∞, ∞, ..., ∞}        // 到各点的距离初始化为无穷大
      dist[source] = 0                // 到自身距离为0

      PriorityQueue<Node> pq = new PriorityQueue<>(
          Comparator.comparingInt(n -> dist[n.id])
      )
      pq.offer(source)                // 将起点加入队列

      visited[] = {false, ..., false}

      while (!pq.isEmpty()):
          u = pq.poll()              // 取出距离最小的未访问节点
          if (visited[u]): continue  // 已访问过，跳过
          visited[u] = true

          for (v in u.neighbors):     // 松弛操作（Relaxation）
              alt = dist[u] + weight(u, v)
              if (alt < dist[v]):
                  dist[v] = alt
                  pq.offer(v)         // 更新距离后重新入队

      return dist[]                   // 返回到所有点的最短距离

  时间复杂度：
  - 朴素实现（数组）：O(V²)（V=顶点数）
  - 二叉堆优化：O((V + E) log V)（E=边数）
  - 斐波那契堆优化：O(E + V log V)（理论最优）

  为什么必须用堆？
  - 每次需要取出当前距离最小的节点
  - 用堆可以在O(log V)时间内完成
  - 如果用数组扫描，需要O(V)时间
  - 当V很大时（如地图导航，V=数百万），堆的优势明显
  ```

### 5. 双堆技巧：求中位数与百分位数

- **定义**：使用一个大顶堆和一个小顶堆来动态维护数据集的中位数或其他百分位数
- **为什么重要**：
  - ⭐ **面试高频题**："如何从数据流中获取中位数？"
  - **实时统计分析的核心技术**：99%响应时间、P95/P99延迟
  - **体现"平衡思维"**：两个堆的大小关系决定了中位数的位置

- **关键机制——双堆协作原理**：

  **数据分配规则**
  ```
  大顶堆（maxHeap）：存储较小的那一半数据
  小顶堆（minHeap）：存储较大的那一半数据

  不变式（Invariant）：
  1. maxHeap的所有元素 ≤ minHeap的所有元素
  2. 两个堆的大小相差不超过1
  3. 如果总元素数为偶数：两个堆大小相等
     如果总元素数为奇数：maxHeap多一个元素（或minHeap，看约定）

  中位数的获取：
  - 元素数为奇数：maxHeap.peek()（或minHeap.peek()）
  - 元素数为偶数：(maxHeap.peek() + minHeap.peek()) / 2.0
  ```

  **插入操作的完整流程**
  ```
  addNum(num):
      if (maxHeap.isEmpty() || num <= maxHeap.peek()):
          maxHeap.offer(num)           // 加入大顶堆
      else:
          minHeap.offer(num)           // 加入小顶堆

      // 平衡两个堆的大小
      if (maxHeap.size() > minHeap.size() + 1):
          minHeap.offer(maxHeap.poll())  // 从大顶堆移动一个到小顶堆
      else if (minHeap.size() > maxHeap.size()):
          maxHeap.offer(minHeap.poll())  // 从小顶堆移动一个到大顶堆

  findMedian():
      if (maxHeap.size() == minHeap.size()):
          return (maxHeap.peek() + minHeap.peek()) / 2.0
      else:
          return (double) maxHeap.peek()

  时间复杂度：
  - addNum(): O(log n)（堆操作）
  - findMedian(): O(1)（直接返回堆顶）
  - 空间复杂度：O(n)

  图示示例（逐步插入 [5, 3, 1, 2, 4]）：

  Insert 5:
  maxHeap: [5]        minHeap: []
  median = 5

  Insert 3 (3 ≤ 5, 进入maxHeap):
  maxHeap: [5, 3]      minHeap: []      → 平衡：移动5到minHeap
  maxHeap: [3]         minHeap: [5]
  median = (3 + 5) / 2 = 4

  Insert 1 (1 ≤ 3, 进入maxHeap):
  maxHeap: [3, 1]      minHeap: [5]
  median = (3 + 5) / 2 = 4

  Insert 2 (2 ≤ 3, 进入maxHeap):
  maxHeap: [3, 2, 1]   minHeap: [5]     → 平衡：移动3到minHeap
  maxHeap: [2, 1]      minHeap: [5, 3]
  median = (2 + 3) / 2 = 2.5

  Insert 4 (4 > 2, 进入minHeap):
  maxHeap: [2, 1]      minHeap: [3, 4, 5]  → 平衡：移动3到maxHeap
  maxHeap: [3, 2, 1]   minHeap: [4, 5]
  median = 3
  ```

  **扩展：求任意百分位数**
  ```
  求99%响应时间（P99）：
  - maxHeap存储前99%的数据（较小的部分）
  - minHeap存储后1%的数据（较大的部分）
  - 保持 maxHeap.size() : minHeap.size() = 99 : 1
  - P99 = maxHeap.peek()（maxHeap的最大值就是99%的分位点）

  求P95延迟：
  - 类似地，保持比例 95 : 5
  - P95 = maxHeap.peek()

  实际应用场景：
  - API网关的性能监控（P99/P95/P50响应时间）
  - 广告系统的竞价排名（展示前1%的高价广告）
  - 金融系统的风险控制（监控异常交易额）
  ```

## 次要信息速览（80% 快速带过）

- 【堆的类型与变种】：
  - **大顶堆（Max Heap）**：父节点 ≥ 子节点，堆顶是最大值
  - **小顶堆（Min Heap）**：父节点 ≤ 子节点，堆顶是最小值
  - **二项堆（Binomial Heap）**：支持高效合并（O(log n)），用于高级算法
  - **斐波那契堆（Fibonacci Heap）**：均摊O(1)的decrease-key操作，理论最优但实现复杂
  - **d-ary堆（d叉堆）**：每个节点有d个子节点，减少高度，适合缓存

- 【堆排序的完整流程】：
  ```
  步骤1：建堆（Build Heap）
         从最后一个非叶子节点开始，依次下沉
         时间：O(n)

  步骤2：排序（Sort）
         循环n-1次：
           1. 交换堆顶和末尾元素（最大值放到最终位置）
           2. 堆大小减1
           3. 对新的堆顶进行下沉操作
         时间：O(n log n)

  总时间：O(n) + O(n log n) = O(n log n)
  空间：O(1)（原地排序）
  稳定性：不稳定（远距离交换可能改变相等元素的相对顺序）
  ```

- 【Java PriorityQueue的使用注意事项】：
  - 默认是小顶堆（自然排序），要大顶堆需传入 `Collections.reverseOrder()`
  - 不支持 `O(log n)` 的更新操作（只能 `remove` + `add`，实际是 `O(n)`）
  - 不是线程安全的，并发场景用 `PriorityBlockingQueue`
  - 容量无限（自动扩容），但可以指定初始容量优化性能

- 【Python heapq模块简介】：
  ```python
  import heapq

  # 默认是小顶堆
  heap = []
  heapq.heappush(heap, 5)
  heapq.heappush(heap, 3)
  heapq.heappush(heap, 8)
  print(heapq.heappop(heap))  # 输出 3（最小值）

  # 大顶堆技巧：存储负数
  max_heap = []
  heapq.heappush(max_heap, -5)  # 存储 -5
  print(-heapq.heappop(max_heap))  # 输出 5（最大值）

  # 建堆：O(n)
  arr = [5, 3, 8, 1, 2]
  heapq.heapify(arr)  # 原地建堆
  ```

- 【堆的应用场景汇总】：
  - **TopK问题**：热搜榜、销量排行、成绩排名
  - **优先级队列**：任务调度、事件系统、Dijkstra算法
  - **堆排序**：内存受限时的排序选择
  - **中位数/百分位数**：实时数据分析、性能监控
  - **多路归并**：外部排序、MapReduce Reduce阶段
  - **Huffman编码**：构建最优前缀码树

# 思考问答

1. **如果堆中某个非堆顶元素的值发生了变化（增大或减小），如何高效地更新堆？普通的PriorityQueue能做到吗？**
   - 提示：考虑索引堆（Indexed Heap）、 decrease-key 操作、或者简单的 remove+re-insert

2. **给定一个包含10亿个整数的文件，如何找出出现频率最高的前100个数？（假设内存只有1GB）**
   - 提示：Hash分片 + 各片统计词频 + 小顶堆维护Top100 + 合并结果

3. **为什么建堆的时间复杂度是O(n)而不是O(n log n)？请用自己的话解释这个证明过程的直觉。**
   - 提示：考虑大多数节点都在底层，底层节点的下沉代价很小

4. **如果让你设计一个支持以下三个操作的数据结构：insert(val)、getMedian()、removeMedian()，你会怎么做？时间复杂度分别是多少？**
   - 提示：双堆 + 延迟删除（Lazy Deletion）标记，或使用两个TreeMap

5. **在Dijkstra算法中，如果用数组代替优先级队列，时间复杂度会变成多少？什么情况下这种"退化"是可以接受的？**
   - 提示：O(V²)，当图是稠密图（E ≈ V²）时，两者相当
